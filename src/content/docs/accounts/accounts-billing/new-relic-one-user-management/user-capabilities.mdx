In the user management space, a **capability** is a granular thing that you can do with New Relic that we've made available for addition to a custom role. These capabilities are also attached to our pre-build [standard roles](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts#roles). Examples of capabilities are: the capability to view APM app settings, or modify alert conditions, or manage data retention settings. 

To learn what capabilities a role has, go to the user management UI and view a specific role. The location of this UI depends on what [user model](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models) a user is on: 

* Newer user model: from [one.newrelic.com](https://one.newrelic.com), click **[account dropdown](/docs/new-relic-solutions/get-started/glossary/#account-dropdown) &gt; Administration &gt; Access management &gt; Roles**.  
* Original user model: from [one.newrelic.com](https://one.newrelic.com), click **[account dropdown](/docs/new-relic-solutions/get-started/glossary/#account-dropdown) &gt; Account settings &gt; Users and roles**.  

## What are capabilities?  [#capabilities-roles]

The capabilities we expose for selection are just a subset of all the things a user can do in New Relic. There are many capabilities that we don't make available for selection and that are available to all users, based on their user type, like the ability to see and use various UI pages. And some roles have hidden capabilities that aren't made available: for example, the **Organization manager** role can manage and configure organization-level settings, but those aren't available for addition to a custom role. The capabilities we make available for selection are those we think organizations would find value in being able to enable or disable for custom roles. 

Some other important points for understanding and using capabilities: 

* **Capabilities can change.** The capabilities we expose in the UI are subject to change and reorganization. The capabilities in this doc were last updated June, 2022.    
* **Some capabilities have overlapping functionality.** That is why selecting some capability checkboxes will automatically check or uncheck other boxes.
* **Capabilities don't affect querying of data.** Most capabilities apply to New Relic UI and API experiences and not to querying data. For example, if your capabilities restrict you from accessing the APM UI, you can still query APM-related data using the query builder if you have access to that account. If you require more firm data boundaries for some projects or users, one way to do this is by [adding accounts](/docs/accounts/accounts-billing/account-structure/add-accounts).

To learn more about the main ways user permissions are controlled, see [User management concepts](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts/#user-type-groups-relation). 

## Our pre-built roles [#pre-built-roles]

Our pre-built roles have various groupings of capabilities. How our pre-built roles work is different depending on which of our [user models](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models) you're on: 

* Our newer user model: [learn about roles](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts). 
* Our original user model: [learn about roles](/docs/accounts/original-accounts-billing/original-users-roles/users-roles-original-user-model).
