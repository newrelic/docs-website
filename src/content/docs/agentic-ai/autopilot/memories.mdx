Telemetry tells you what's broken, but resolving complex incidents often relies on context that live data can't provide. This operational knowledge—past failure patterns, effective workarounds, and unique infrastructure quirks—is typically lost across fragmented daily interactions.

<DNT>Autopilot</DNT> Memory systematically extracts these operational lessons from your interactions with <DNT>Autopilot</DNT> and turns them into a persistent knowledge base you can access from the Memory Management UI.

## Opt in [#opt-in]

<DNT>Autopilot</DNT> Memory requires an Advanced Compute SKU, and you manage access through [Feature control manager](/docs/accounts/accounts-billing/new-relic-one-user-management/feature-control-manager/). <DNT>Autopilot</DNT> Memory is disabled by default. To opt in, an Organization Manager or Authentication Domain Manager must explicitly enable the feature in <DNT>Feature control manager</DNT>, under Administration. This helps ensure your organization retains full control over data capture and sharing.

## How memory works [#how-it-works]

If enabled, <DNT>Autopilot</DNT> Memory continuously extracts knowledge from your interactions with the agent to deepen its context and improve response accuracy. Understanding how <DNT>Autopilot</DNT> scopes information during an investigation helps you understand how it adapts to your environment.

<DNT>Autopilot</DNT> uses distinct levels of contextual awareness to balance immediate conversational needs with persistent, cross-session knowledge in long-term memory:

* **Active thread context**: High-speed, low-latency retrieval scoped to your current chat session. This immediate context supports quick follow-ups, helping <DNT>Autopilot</DNT> maintain continuity in your investigation without pulling in unrelated historical data.
* **Long-term memory**: Persistent, cross-session knowledge that lets <DNT>Autopilot</DNT> systematically learn your preferences and your unique environment. Instead of explaining your setup in every chat session, <DNT>Autopilot</DNT> automatically retrieves historical insights, preferences, and past feedback. For example, if a previous investigation flagged a database spike as a safe nightly job, <DNT>Autopilot</DNT> applies that context to future analyses. Long-term memory has three scopes:
  * **User scope**: Context and preferences saved exclusively for your individual interactions with <DNT>Autopilot</DNT>.
  * **Account scope**: Shared operational knowledge accessible to all authorized users within a specific New Relic account.
  * **Organization scope**: Global operational knowledge accessible to all authorized users across your entire organization.

You can view and manage these scoped memories in the [Memory Management UI](#memory-management-ui).

## Capture and retrieve memories [#capture-retrieval]

Memory data flows in and out of long-term storage through the following mechanisms:

* **Automatic capture**: Once an Organization Manager enables Memory in <DNT>Feature control manager</DNT>, <DNT>Autopilot</DNT> continuously processes your chat interactions behind the scenes. This automated pipeline ensures <DNT>Autopilot</DNT> retains high-quality context without requiring manual effort.
* **Fact and relation extraction**: <DNT>Autopilot</DNT> analyzes your prompts and its own responses to extract durable operational details, preferences, and architectural context. <DNT>Autopilot</DNT> automatically categorizes and stores these insights as structured facts and relations, ready for fast, semantic retrieval during future investigations.
* **Manual promotion**: Use the <DNT>**Save Response**</DNT> button in the chat panel to manually save a valuable insight to memory. Where you can save it depends on your [RBAC permissions](#rbac):
  * **Organization Managers** can promote an insight to Organization, Account, or User memory. They also have full authority to view and manually delete memories at the Organization and Account scopes—in addition to their own User-scoped memories—to help keep long-term memory accurate.
  * **All-Product Admins** can promote an insight to Account or User memory. They also have full authority to view and manually delete memories at the Account scope, in addition to their own User-scoped memories.
  * **All other roles** can only save responses to their own private User memory.
* **Automated learning and feedback**: <DNT>Autopilot</DNT> adapts its logic over time based on your corrections, feedback, and context. For example, providing highly specific feedback—such as "Update this NRQL filter to account for the auth-service memory leak"—creates high-quality memory entries.
* **Intelligent retrieval**: Rather than flooding prompts with unnecessary data, <DNT>Autopilot</DNT> uses vector similarity search to rank and retrieve only the most relevant historical lessons. It scopes memory to the active thread for immediate follow-ups, but can instantly recall past sessions when you reference historical events.

## Delete memories [#delete-memories]

Automated background jobs handle [data retention](#data-retention), but your environment can change—sometimes rapidly. If a saved memory becomes outdated (for example, an architectural change makes an old workaround obsolete) or was captured incorrectly, authorized users can manually delete specific long-term memory entries directly from the Memory Management UI. This helps ensure long-term memory always reflects your current operational reality.

## Manage memories in the UI [#memory-management-ui]

<Callout variant="tip">
  To use this feature, an Organization Manager or Authentication Domain Manager must first enable it in [Feature control manager](/docs/accounts/accounts-billing/new-relic-one-user-management/feature-control-manager/), under Administration.
</Callout>

The centralized Memory Management dashboard gives you full visibility and control over what <DNT>Autopilot</DNT> learns. Once the feature is enabled, you can view, audit, and manage all memory entries here, based on your [RBAC](#rbac) permissions.

* **Access**: Open the dashboard from the left navigation menu, under <DNT>**Knowledge > Memories**</DNT>.
* **Scope filtering**: Toggle between Organization, Account, and User tabs to view scoped memory.
* **Search memories**: Use the search bar for text search, or apply filters to find specific context.
* **Data hierarchy**: <DNT>Autopilot</DNT> groups memories by conversation thread. Expand a thread to see granular details, including the facts and relations captured, along with other metadata. Use this to find and manage outdated entries.

## Role-based access control (RBAC) [#rbac]

Your assigned role governs your access to the Memory Management UI:

* **Organization Managers** can view and fully manage (create and delete) Organization, Account, and User memory.
* **All-Product Admins** can view Organization memory, and fully manage (create and delete) both Account and User memory.
* **All other roles** can view shared Organization and Account memory, and fully manage (create and delete) their own private User memory.

If the default permissions don't meet your organization's needs, see [Create custom roles for Autopilot Memories](#custom-roles).

## Data retention [#data-retention]

Memory storage adheres to your existing compliance requirements. All long-term memory—whether scoped to a user, account, or your entire organization—follows the New Relic AI data retention policy you configure in the [Retention UI](/docs/data-apis/manage-data/manage-data-retention/).

* **Default retention**: The base retention period for long-term memory is **90 days**.
* **Customization**: You can change this duration (up to a maximum of 365 days) by going to <DNT>**Administration > Data Retention**</DNT>, selecting the Organization-scoped tab, and adjusting the policy for the Long-Term Memory source.
* **Automated purging**: Once memory entries exceed their configured retention limit, automated background jobs delete the expired data—no manual intervention is needed to maintain compliance and purge stale context.

## Create custom roles for Autopilot Memories [#custom-roles]

<DNT>Autopilot</DNT> Memory lets <DNT>Autopilot</DNT> retain context across conversations and sessions. Controlling who can read, create, update, or delete that stored context helps you protect sensitive session data while still letting the right people manage it.

New Relic provides default access to Memory through these standard roles:

#### Account-scoped access

    <tr>

    <tr>
      <td>Delete memory</td>

    </tr>
  </tbody>
</table>

### Create a custom role via NerdGraph [#nerdgraph-steps]

To create a custom role via the NerdGraph API, follow these steps:

<Steps>
  <Step>
    #### Find the required permission IDs for Memory

    Use the `customerAdministration` query to retrieve a list of capabilities, their permission IDs, and related information. Filter by feature to get Memory permissions.

    **Input parameters**

    <table>
      <thead>
        <tr>
          <th>Parameter</th>
          <th>Data type</th>
          <th>Is it required?</th>
          <th>Description</th>
        </tr>
      </thead>
      <tbody>
        <tr>
          <td>`eq`</td>
          <td>String</td>
          <td>Yes</td>
          <td>Set the value to `Memory` (account-scoped) or `Memory For Organization` (organization-scoped) to retrieve the matching permissions.</td>
        </tr>
      </tbody>
    </table>

    **Sample request — account-scoped**

    ```graphql
    {
      customerAdministration {
        permissions(filter: { feature: { eq: "Memory" } }) {
          items {
            feature
            category
            id
            product
          }
        }
      }
    }
    ```

    **Sample response**

    ```json
    {
      "data": {
        "customerAdministration": {
          "permissions": {
            "items": [
              {
                "category": "READ",
                "feature": "Memory",
                "id": "xxxxx",
                "product": "New Relic AI"
              },
              {
                "category": "MODIFY",
                "feature": "Memory",
                "id": "xxxxxx",
                "product": "New Relic AI"
              },
              {
                "category": "DELETE",
                "feature": "Memory",
                "id": "xxxxxx",
                "product": "New Relic AI"
              }
            ]
          }
        }
      }
    }
    ```

    Run the same query with `feature: { eq: "Memory For Organization" }` to retrieve the organization-scoped permission IDs (READ, MODIFY, DELETE for Memory For Organization).

    From the response, copy the permission IDs where `feature` is `Memory` or `Memory For Organization`, matching the access level you want to grant. You'll need these IDs in the next step.
  </Step>

  <Step>
    #### Retrieve your organization ID

    Retrieve your organization ID, which you'll use in subsequent mutations. For an account-scoped role, also retrieve the account ID.

    **Sample request**

    ```graphql
    {
      actor {
        organization {
          id
        }
        accounts {
          id
          name
        }
      }
    }
    ```

    **Sample response**

    ```json
    {
      "data": {
        "actor": {
          "organization": {
            "id": "YOUR_ORGANIZATION_ID"
          },
          "accounts": [
            {
              "id": "YOUR_ACCOUNT_ID",
              "name": "Your Account Name"
            }
          ]
        }
      }
    }
    ```

    Copy the ID(s) you need from the response. You'll use them to create the custom role.
  </Step>

  <Step>
    #### Create the custom role

    Use the `customRoleCreate` mutation to create your custom role for Memory management.

    **Input parameters**

    <table>
      <thead>
        <tr>
          <th>Parameter</th>
          <th>Data type</th>
          <th>Is it required?</th>
          <th>Description</th>
        </tr>
      </thead>
      <tbody>
        <tr>
          <td>`id`</td>
          <td>String</td>
          <td>Yes</td>
          <td>The organization ID (organization-scoped) or account ID (account-scoped) from the previous step.</td>
        </tr>
        <tr>
          <td>`type`</td>
          <td>String</td>
          <td>Yes</td>
          <td>Set to `organization` for an organization-scoped role, or `account` for an account-scoped role.</td>
        </tr>
        <tr>
          <td>`name`</td>
          <td>String</td>
          <td>Yes</td>
          <td>The display name for the custom role.</td>
        </tr>
        <tr>
          <td>`permissionIds`</td>
          <td>Array of Integers</td>
          <td>Yes</td>
          <td>The Memory permission IDs identified in the previous step.</td>
        </tr>
        <tr>
          <td>`scope`</td>
          <td>String</td>
          <td>Yes</td>
          <td>Set to `organization` or `account`, matching `type`.</td>
        </tr>
      </tbody>
    </table>

    **Sample request — account-scoped role**

    ```graphql
    mutation {
      customRoleCreate(
        container: { id: "YOUR_ACCOUNT_ID", type: "account" }
        name: "Memory manager"
        permissionIds: [xxxxx, xxxxx]
        scope: "account"
      ) {
        id
      }
    }
    ```

    **Sample request — organization-scoped role**

    ```graphql
    mutation {
      customRoleCreate(
        container: { id: "YOUR_ORGANIZATION_ID", type: "organization" }
        name: "Memory manager (org-wide)"
        permissionIds: [xxxxx, xxxxx]
        scope: "organization"
      ) {
        id
      }
    }
    ```

    **Sample response**

    ```json
    {
      "data": {
        "customRoleCreate": {
          "id": 9999999
        }
      }
    }
    ```

    Save the returned role ID—you'll need it to assign this role to a user group.
  </Step>

  <Step>
    #### Add the custom role to a user group

    After creating the custom role, assign it to a user group.

    **Retrieve group IDs**

    Use the `customerAdministration` query to get a list of available user groups.

    **Input parameters**

    <table>
      <thead>
        <tr>
          <th>Parameter</th>
          <th>Data type</th>
          <th>Is it required?</th>
          <th>Description</th>
        </tr>
      </thead>
      <tbody>
        <tr>
          <td>`id`</td>
          <td>String</td>
          <td>Yes</td>
          <td>The organization ID from the previous step.</td>
        </tr>
      </tbody>
    </table>

    **Sample request**

    ```graphql
    {
      customerAdministration {
        groups(filter: { organizationId: { eq: "YOUR_ORGANIZATION_ID" } }) {
          nextCursor
          items {
            id
            name
            users {
              items {
                id
                email
              }
            }
          }
        }
      }
    }
    ```

    From the response, copy the group ID for the group you want to assign the Memory role to.

    **Assign the role to the group**

    Use the `authorizationManagementGrantAccess` mutation to assign the custom role to a user group.

    **Input parameters**

    <table>
      <thead>
        <tr>
          <th>Parameter</th>
          <th>Data type</th>
          <th>Is it required?</th>
          <th>Description</th>
        </tr>
      </thead>
      <tbody>
        <tr>
          <td>`roleId`</td>
          <td>String</td>
          <td>Yes</td>
          <td>The custom role ID from the previous step.</td>
        </tr>
        <tr>
          <td>`groupId`</td>
          <td>String</td>
          <td>Yes</td>
          <td>The group ID retrieved above.</td>
        </tr>
      </tbody>
    </table>

    **Sample request — account-scoped role**

    ```graphql
    mutation {
      authorizationManagementGrantAccess(
        grantAccessOptions: {
          accountAccessGrants: { accountId: "YOUR_ACCOUNT_ID", roleId: "YOUR_ROLE_ID" }
        }
        groupId: "YOUR_GROUP_ID"
      ) {
        roles {
          id
          name
          accountId
          roleId
          groupId
          displayName
        }
      }
    }
    ```

    **Sample request — organization-scoped role**

    ```graphql
    mutation {
      authorizationManagementGrantAccess(
        grantAccessOptions: {
          organizationAccessGrants: { roleId: "YOUR_ROLE_ID" }
        }
        groupId: "YOUR_GROUP_ID"
      ) {
        roles {
          id
          name
          organizationId
          roleId
          groupId
          displayName
        }
      }
    }
    ```

    **Sample response**

    ```json
    {
      "data": {
        "authorizationManagementGrantAccess": {
          "roles": [
            {
              "displayName": "Memory manager",
              "groupId": null,
              "id": "99999999",
              "name": "memory_manager",
              "organizationId": "YOUR_ORGANIZATION_ID",
              "roleId": 99999
            }
          ]
        }
      }
    }
    ```
  </Step>
</Steps>
