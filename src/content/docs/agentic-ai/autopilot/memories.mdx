Telemetry tells you what is broken, but resolving complex incidents often relies on context that live data cannot provide. This vital context exists as operational or tribal knowledge—past failure patterns, effective workarounds, and unique infrastructure quirks—that is typically lost across fragmented daily interactions.
Autopilot memory systematically extracts these operational lessons from your interactions with the Autopilot and transforms them into a persistent knowledge base accessible in the memory management portion of the New Relic UI.

## Opt in [#opt-in]

<DNT>Autopilot</DNT> memory requires an Advanced Compute SKU and is disabled by default. To opt in, an Organization Manager or Authentication Domain Manager must explicitly enable the feature via the [Feature control manager](/docs/accounts/accounts-billing/new-relic-one-user-management/feature-control-manager/), under Administration. This ensures your organization retains full control over data capture and sharing.

## Role-based access control (RBAC) [#rbac]

Access to the Memory Management UI is governed by your assigned role:

* **Organization Managers**: Can view and manage (create and delete) Organization, Account, and User Memory.
* **All Product Admins**: Can view Organization Memory, and manage (create and delete) both Account and User Memory.
* **All other roles**: Can view shared Organization and Account Memory, and manage (create and delete) their own private User Memory.

If you need to set up custom permissions for managing <DNT>Autopilot</DNT> memories, see [Create custom roles for Autopilot Memories](#custom-roles).

## How memory works [#how-it-works]

If enabled, <DNT>Autopilot</DNT> memory continuously extracts knowledge from agent-user interactions to deepen <DNT>Autopilot</DNT> context and understanding to help improve response accuracy.

To provide the best answers, <DNT>Autopilot</DNT> relies on two types of memory—combining the immediate context of your current chat with historical knowledge saved from past sessions:

* **Active thread context**: Active chat sessions are temporarily retained for immediate context for quick follow-ups, allowing <DNT>Autopilot</DNT> to help maintain the continuity of your current investigation.
* **Long-Term Memory**: This memory is a persistent cross-session knowledge that allows <DNT>Autopilot</DNT> to have historical context around your preferences and your unique environment. Instead of explaining your setup to the agent in every chat session, <DNT>Autopilot</DNT> can automatically retrieve historical insights, user preferences, and past feedback in long-term memory. For example, if a previous investigation determined that a recurring database spike is expected behavior caused by a scheduled nightly backup, <DNT>Autopilot</DNT> can apply that operational context to future analyses. Long-Term Memory is categorized into three distinct scopes:
  * **User Scope**: Context and preferences saved exclusively for your individual interactions with the AI.
  * **Account Scope**: Shared operational knowledge accessible to all authorized users within a specific New Relic account.
  * **Organization Scope**: Global operational knowledge accessible to all authorized users across your entire New Relic organization.

You can view and manage these distinct scoped memories in the [Memory Management UI](#memory-management-ui).

## Autopilot memory capture and retrieval [#capture-retrieval]

Data flows into and out of the long term memory through the following mechanisms:

* **Automatic Memory**: Once an Organization Manager enables memory via the Feature Control Manager, the system continuously processes your chat interactions behind the scenes. This automated pipeline helps <DNT>Autopilot</DNT> to retain incident context without requiring manual effort:
  * **Fact and Relation Extraction**: The system analyzes prompts and the responses to intelligently extract durable operational details, user preferences, and architectural context. These insights are automatically categorized and stored as structured facts and relations, ready for fast, semantic retrieval during future investigations.
* **Manual Promotion (Save Response button in Chat Panel)**: You can also use the Save Response button in the chat panel to manually add what you may consider to be a valuable insight to the <DNT>Autopilot</DNT> memory. Your options for where to save this context depend on your [RBAC permissions](#rbac):
  * **Organization Managers**: Can choose to promote the insight to the Organization, Account, or User memory.
  (Note: Apart from the ability to view and delete their own User-scoped memories, Organization Managers have full authority to view and manually delete memories at the Organization and Account scopes to keep the long-term memory accurate)
  * **All Product Admins**: Can choose to promote the insight to the Account or User memory.
  (Note: Apart from the ability to view and delete their own User-scoped memories, All Product Admins have full authority to view and manually delete memories at the Account scope).
  * **All other roles**: Can save the response exclusively to their own private User memory.
* **Feedback**: <DNT>Autopilot</DNT> has the ability to adapt logic for your use over time based on the corrections, feedback, and context you provide. For example, when refining agent logic, providing highly specific feedback (e.g., "Update this NRQL filter to account for the auth-service memory leak") creates high-quality entries in Memories.
* **Intelligent Retrieval**: Rather than flooding prompts with unnecessary data, <DNT>Autopilot</DNT> dynamically selects the context. It uses vector similarity search to rank and retrieve only the most relevant historical lessons. It isolates memory to the active thread for immediate follow-ups, but can recall past sessions when you reference historical events.

## Targeted memory deletion [#delete-memories]

While automated background jobs handle [data retention](#data-retention), we recognize that your environment can change, sometimes rapidly. If a previously saved context becomes outdated (e.g., an architectural change makes an old workaround obsolete) or was captured incorrectly, authorized users can manually delete specific long term memory entries directly from the Memory Management UI. This helps to ensure the persistent long term memory base reflects your current operational reality.

## Autopilot memory management UI [#memory-management-ui]

The centralized Memory Management dashboard gives you visibility and control over what <DNT>Autopilot</DNT> can use for context and understanding your environment. Once the feature is enabled, you can view, audit, and manage all memory entries here based on the [RBAC](#rbac) permissions.

* **Access**: Go to <DNT>**[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > Knowledge > Memories**</DNT>.
* **Scope filtering**: Toggle between Organization, Account, and User tabs to view scoped memory.
* **Search memories**: Use the search bar for text search, or apply filters to find specific context.
* **Data hierarchy**: <DNT>Autopilot</DNT> groups memories by conversation thread. Expand a thread to see granular details, including the facts and relations captured, along with other metadata. Use this to find and manage outdated entries.

## Data retention [#data-retention]

All Long-Term Memory—whether scoped to a user, account, or the entire organization—is governed by what you have chosen for New Relic AI Data Retention policy in the [Retention UI](/docs/data-apis/manage-data/manage-data-retention/).

* **Default Retention**: The base retention period for Long-Term Memory is 90 days.
* **Customization**: You can modify this duration (up to a maximum of 365 days) by navigating to Administration > Data Retention, selecting the Organization-scoped tab, and adjusting the policy for the Long Term Memory source.

## Create custom roles for Autopilot Memories [#custom-roles]

If the default RBAC permissions don't meet your organization's needs, you can create a custom role for Memory using the New Relic NerdGraph API.

For the full walkthrough—including default access and permission tables, plus the step-by-step API calls to create the role and assign it to a user group—see [Create custom roles for Autopilot Memories](/docs/apis/nerdgraph/examples/nerdgraph-memories-custom-role-tutorial/) in the API documentation.
