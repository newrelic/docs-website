## What is the New Relic SRE Agent? [#what-is-sre-agent]

The New Relic SRE Agent is an intelligent agent designed to be your on-call teammate. By integrating directly with your telemetry data and established workflows, it moves beyond simple alerting to provide active investigation, diagnosis, and remediation.

Here are some examples of how the SRE Agent supports your operations:

1. **Alert triage and validation**: The agent acts as a first-line filter for incoming notifications. It evaluates the context of an alert against historical baselines and current system health to determine if an event is a high-priority anomaly or an expected fluctuation, reducing alert fatigue for on-call engineers.
2. **Real-time entity health**: The agent provides an instant, high-level pulse check on any part of your stack. By correlating golden signal metrics with underlying infrastructure performance, it delivers a narrative summary of health that goes deeper than a simple green or red status.
3. **Change intelligence**: Deployments are often the catalyst for instability. The SRE Agent reviews deployment markers and compares pre- and post-change telemetry to identify performance regressions, new error patterns, or latency spikes introduced by recent updates.
4. **Deep incident analysis**: When system degradation occurs, the agent performs automated root-cause discovery. It traverses distributed traces, analyzes log patterns, and inspects resource metrics to help identify potential technical failures or downstream dependencies that may be contributing to an outage.
5. **Automated remediation**: Beyond diagnosis, the SRE Agent suggests specific technical interventions—such as rolling back a problematic deployment or adjusting resource allocations.

## Prerequisites [#prerequisites]

Access to the SRE Agent is available for customers on the **Advanced Compute** or **Advanced Compute Add-on** SKU.

Because the SRE Agent operates at the organization level, you must be an Organization Administrator and Authentication Domain Manager to complete setup. Your user role must include the following organization-level capabilities:

* **SRE Agent** - Modify, Delete, Other (Ask, Add to Automation)
* Permissions to manage system identities (create, read, update, and delete)

<Callout variant="tip">
  If you don't have these permissions, you'll see the message: "This looks like a job for an admin." Contact your organization administrator to enable the agent for you.
</Callout>

## Set up the SRE Agent [#setup]

The onboarding workflow handles the full setup automatically, including provisioning a system identity to represent the agent, assigning required permissions, and creating an alert destination you can add to notification flows.

<Callout variant="important">
  **What is a system identity?** A system identity is a non-human identity that allows the agent to authenticate directly to other services without requiring a user account. System identities are scoped to a single New Relic organization.
</Callout>

To onboard the SRE Agent:

1. Open the **Ask AI** panel and navigate to the **Configure** tab (the gear icon).
2. Under **Agents**, select **SRE Agent**.
3. In the **Agent name** text field, provide a recognizable name. The name you choose will be automatically prepended with `SRE Agent -` and applied to the system identity and associated access management components.
4. Use the dropdown to select the specific **account(s)** you want the agent to analyze.

   <Callout variant="important">
     Adding more accounts allows the agent to better identify connections and surface insights across your stack. However, the agent's analysis will be visible to any users with SRE Agent interaction access, regardless of their direct access to those accounts.
   </Callout>

5. Review the **Permissions** summary. Upon initial setup, the SRE Agent is granted the standard read-only role on the selected accounts. Admin users can modify these grants later, but reducing access below the standard read-only role will limit the telemetry the agent can access, resulting in incomplete analysis, missed anomalies, or degraded root-cause identification.
6. Click **Add Agent**. The system will provision the system identity, assign the necessary read-only grants, and create a cross-account "SRE Agent" alert destination.

<Callout variant="tip">
  Creating the alert destination does not automatically assign any alert conditions or policies to the SRE Agent. See [Set up the SRE Agent with alerts](#alerts) to configure that.
</Callout>

## Set up the SRE Agent with alerts [#alerts]

Connect the SRE Agent to your alert workflows so that SRE Agent insights are automatically attached to every fired alert or issue.

To validate performance, we recommend a phased approach: start with a small subset of alert conditions or policies, confirm the agent is working well for your environment, then expand more broadly.

To set up the SRE Agent with alerts:

1. Go to **Issues and Activity &rarr; Notification Workflows &rarr; Add Workflow**.
2. In your workflow configuration, under the **Notify** section, select **AI Agent**.
3. Choose the SRE Agent destination created during onboarding and click **Save message**.

Once configured, when you open a recently fired alert or issue, you'll automatically see the SRE Agent summary for that item.

## How the SRE Agent analyzes your environment [#how-it-works]

The SRE Agent acts as an intelligent layer between your complex system data and actionable insights. When you ask the agent a question, it follows three primary analysis pillars:

### 1. Comprehensive context gathering [#context-gathering]

The agent begins by mapping your architecture. It identifies relevant entities (services, hosts, or containers), explores their relationships and dependencies, and checks for recent change events or deployments that might have triggered a shift in performance.

### 2. Deep telemetry investigation [#telemetry-investigation]

Once context is established, the agent dives into the data:

* **Metrics and trends**: Analyzes throughput, latency, and error rates while forecasting future system behavior.
* **Logs and errors**: Sifts through distributed traces, error groups, and log patterns to isolate the location and cause of an anomaly.
* **Health and alerts**: Reviews active incidents, alert policies, and security vulnerabilities to understand the current blast radius.

### 3. Automated root cause and impact analysis [#root-cause-analysis]

The agent performs causal analysis—correlating performance dips with specific transaction patterns, infrastructure bottlenecks (such as Kafka lag), and end-user impact. It can also convert natural language into complex NRQL queries to surface data points that manual investigation might miss.

## Usage tips [#usage-tips]

* The SRE Agent is optimized for live analysis and searches events, issues, and entities from the **last 24 hours** by default. To analyze older data, specify a time range. You can use `SINCE <time> UNTIL <time>` phrasing (similar to NRQL) to target a specific window.
* Use the agent to convert natural language into NRQL queries, then run them directly in the query builder.
* In longer conversations, avoid ambiguous pronoun references for more accurate results.
