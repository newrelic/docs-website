Before [installing our infrastructure agent](/docs/infrastructure/install-infrastructure-agent/get-started/install-infrastructure-agent-new-relic), make sure your system and any on-host integrations you configure meet the requirements.

## Processor architectures [#processors]

The infrastructure agent supports only x86 processor architectures:

* **Linux**: 64-bit architecture (also requires 64-bit package manager and dependencies)
* **Windows**: both 32 and 64-bit architectures
* **ARM**: [AWS Graviton 2 processor](https://aws.amazon.com/ec2/graviton/) (using the [tarball setup instructions](https://docs.newrelic.com/docs/infrastructure/install-configure-manage-infrastructure/linux-installation/install-infrastructure-linux-using-package-manager#custom-setups-tarball)). Support is limited to infrastructure monitoring. Integrations, such as log forwarding, may not work as expected.

## Unique hostname [#hostname]

The infrastructure agent uses the hostname to uniquely identify each server. To avoid inaccurate metrics from combining multiple servers under a single hostname, make sure that each monitored server has a unique hostname.

You can use the optional [`display_name`](/docs/infrastructure/new-relic-infrastructure/configuration/configure-infrastructure-agent#display_name) setting to override the default hostname.

<Callout variant="important">
  Servers named `localhost` are not reported: it's a default name and inherently non-unique.
</Callout>

## Permissions

The infrastructure agent requires these permissions:

* **Linux**: By default, [the agent runs and installs as root](https://docs.newrelic.com/docs/infrastructure/new-relic-infrastructure/installation/install-infrastructure-agent-linux#agent-mode-intro). You can also select [privileged](https://docs.newrelic.com/docs/infrastructure/new-relic-infrastructure/installation/install-infrastructure-agent-linux#install-privileged) or [unprivileged](https://docs.newrelic.com/docs/infrastructure/new-relic-infrastructure/installation/install-infrastructure-agent-linux#install-unprivileged) run modes.
* **Windows**: The agent must be installed from an Administrator account and requires Administrator privileges to run.

## Libraries

For [agent versions 1.1.19 or higher](/docs/release-notes/infrastructure-release-notes/infrastructure-agent-release-notes), you need the `libcap` library in order to install Infrastructure. It's available in the official repositories of your distribution.

## Network access [#network]

In order to report data to New Relic, our infrastructure agent must have outbound access to [certain domains and ports](/docs/apm/new-relic-apm/getting-started/networks#infrastructure). If your system needs a proxy to connect to these domains, use the [`proxy`](/docs/infrastructure/new-relic-infrastructure/configuration/configure-infrastructure-agent#proxy) setting.

## Container software [#container]

The infrastructure agent [instruments Docker containers](/docs/infrastructure-docker-instrumentation) when installed on the host server. We support Docker versions 1.12 or higher.

## CPU, memory, and disk usage [#resources]

The infrastructure agent is fairly lightweight. For typical CPU, memory, and disk usage, see our page on [agent performance overhead](/docs/infrastructure/new-relic-infrastructure/getting-started/infrastructure-agent-performance-overhead). For more information on supported file systems, see [Storage sample attributes](https://docs.newrelic.com/docs/infrastructure/new-relic-infrastructure/data-instrumentation/default-infrastructure-attributes-events#storagesample-attributes).
