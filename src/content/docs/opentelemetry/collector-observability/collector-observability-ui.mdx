Once you've configured your collector with internal telemetry, you can monitor it using New Relic's dedicated APM UI experience for OpenTelemetry collectors.

## Accessing collector monitoring [#access]

Your OpenTelemetry collectors appear as service entities in the APM section of New Relic:

1. From the New Relic UI, go to **APM & Services**
2. Search for or filter by your collector name (configured via `INTERNAL_TELEMETRY_SERVICE_NAME`)
3. Click on the collector entity to open its monitoring dashboard

<Callout variant="tip">
You can also find collectors by filtering for entities with the tag `newrelic.service.type: otel_collector`.
</Callout>

## Understanding the UI experience [#ui-experience]

The collector observability UI is a tailored APM experience designed specifically for monitoring OpenTelemetry collectors. It differs from traditional APM service monitoring because collectors are stream-processing systems, not request-response applications.

### What's included

The UI provides insights into:
- Collector health and performance metrics
- Throughput and latency indicators
- Resource utilization (CPU, memory)
- Infrastructure relationships
- Component-level metrics

### What's excluded

Traditional APM features that don't apply to collectors are hidden:
- **Apdex score**: Not applicable to stream processing
- **Error rate**: Collectors don't have traditional errors; they have data drops and refusals
- **Transaction traces**: Collectors don't process transactions
- **Service maps**: Collectors are data processors, not service endpoints

## Summary page [#summary-page]

The Summary page provides a high-level overview of your collector's health and performance.

<Callout variant="important">
The Summary page is billable during preview as measured in Core CCU. See [Understanding data costs](/docs/opentelemetry/collector-observability/collector-telemetry-overhead#billing-preview) for details.
</Callout>

### Key metrics on the Summary page

#### Throughput metrics

Monitor the volume of data flowing through your collector:

**Spans received and sent**
- Shows the rate of spans being received by the collector
- Compares received vs. successfully sent spans
- Helps identify data loss or processing delays

**Metrics received and sent**
- Displays metric data points processed per second
- Useful for understanding collector load

**Logs received and sent**
- Tracks log event processing rate
- Indicates collector health for log pipelines

<Callout variant="tip">
If received counts significantly exceed sent counts, investigate for dropped data or exporter issues.
</Callout>

#### Performance indicators

**Processing latency**
- Time spent processing data through the pipeline
- Broken down by component (receivers, processors, exporters)

**Queue depth**
- Current number of items waiting in exporter queues
- High queue depth may indicate bottlenecks or downstream issues

#### Resource utilization

**Memory usage**
- Current memory consumption
- Memory growth over time (watch for leaks)

**CPU usage**
- Processor utilization percentage
- Spikes may indicate inefficient processing

### Golden signals for collectors

The UI adapts the concept of golden signals for collectors:

1. **Throughput**: Data processed per second (spans/metrics/logs)
2. **Latency**: Processing time through the pipeline
3. **Errors**: Dropped, refused, or failed data
4. **Saturation**: Queue depth and resource utilization

## Infrastructure relationships [#infrastructure]

When properly configured, the Summary page can display infrastructure metrics for the host or container running your collector.

### How relationships work

Infrastructure relationships are established through resource attributes:
- **Host relationships**: Collector telemetry includes host identifiers
- **Container relationships**: Collector telemetry includes container/pod information

### Viewing infrastructure data

When a relationship exists, you'll see:
- Host or container name
- CPU, memory, and disk metrics from the infrastructure
- Links to the infrastructure entity

<Callout variant="important">
Infrastructure relationships require:
1. Your infrastructure to be instrumented (host or Kubernetes monitoring)
2. Proper resource attributes configured on collector telemetry
3. Time for relationship synthesis (up to 70 minutes)

See [configuration examples](/docs/opentelemetry/collector-observability/collector-observability-examples) for setup instructions.
</Callout>

### Missing infrastructure data

If infrastructure relationships aren't appearing:

1. **Verify infrastructure monitoring**: Ensure the host/container is reporting to New Relic
2. **Check resource attributes**: Collector telemetry must include matching identifiers
3. **Wait for synthesis**: Relationships can take up to 70 minutes to form
4. **Review configuration**: See [examples](/docs/opentelemetry/collector-observability/collector-observability-examples#infra-relationships) for proper setup

## Process page [#process-page]

<Callout variant="tip">
The Process page is not billable during preview.
</Callout>

The Process page provides detailed system-level metrics:

### Available metrics

**Process information**
- Uptime
- Process ID
- Runtime version (Go version for the collector)

**Memory metrics**
- RSS (Resident Set Size)
- Heap allocations
- GC (Garbage Collection) statistics

**CPU metrics**
- CPU seconds consumed
- CPU utilization percentage
- Thread count

### When to use the Process page

The Process page is particularly useful for:
- **Memory leak investigation**: Watch for steady memory growth
- **Performance tuning**: Understand CPU patterns
- **Capacity planning**: Track resource trends over time
- **Troubleshooting crashes**: Review metrics before restart events

## HTTP/RPC page [#http-rpc-page]

<Callout variant="tip">
The HTTP/RPC page is not billable during preview.
</Callout>

The HTTP/RPC page shows network communication metrics for HTTP and gRPC connections:

### HTTP metrics

**Request rates**
- HTTP requests per second
- Broken down by endpoint and method

**Response times**
- Request duration histograms
- Percentile breakdowns (p50, p95, p99)

**Error rates**
- HTTP error responses
- Connection failures

### RPC metrics

**gRPC calls**
- RPC call rates
- Call duration
- RPC errors

### Use cases

The HTTP/RPC page helps with:
- **OTLP receiver monitoring**: Track incoming data from agents
- **Exporter health**: Monitor outgoing connections to New Relic
- **Network issues**: Identify connectivity problems
- **Rate limiting**: Detect throttling or backpressure

## Querying collector metrics [#querying]

You can also query collector telemetry directly using NRQL:

### Find all collectors

```sql
FROM Metric
SELECT uniques(entity.name)
WHERE newrelic.service.type = 'otel_collector'
AND instrumentation.provider = 'opentelemetry'
SINCE 1 hour ago
```

### Receiver metrics

```sql
FROM Metric
SELECT rate(sum(otelcol.receiver.accepted_spans), 1 second) AS 'Spans/sec'
WHERE newrelic.service.type = 'otel_collector'
FACET service.name, receiver
TIMESERIES
```

### Exporter metrics

```sql
FROM Metric
SELECT
  rate(sum(otelcol.exporter.sent_spans), 1 second) AS 'Sent',
  rate(sum(otelcol.exporter.send_failed_spans), 1 second) AS 'Failed'
WHERE newrelic.service.type = 'otel_collector'
FACET service.name, exporter
TIMESERIES
```

### Queue depth

```sql
FROM Metric
SELECT latest(otelcol.exporter.queue_size) AS 'Queue Items'
WHERE newrelic.service.type = 'otel_collector'
FACET service.name, exporter
TIMESERIES
```

### Memory usage

```sql
FROM Metric
SELECT latest(otelcol.process.runtime.heap_alloc_bytes) / 1e9 AS 'Heap GB'
WHERE newrelic.service.type = 'otel_collector'
FACET service.name
TIMESERIES
```

### Dropped data

```sql
FROM Metric
SELECT
  rate(sum(otelcol.processor.dropped_spans), 1 second) AS 'Spans/sec',
  rate(sum(otelcol.processor.dropped_metric_points), 1 second) AS 'Metrics/sec',
  rate(sum(otelcol.processor.dropped_log_records), 1 second) AS 'Logs/sec'
WHERE newrelic.service.type = 'otel_collector'
FACET service.name
TIMESERIES
```

## Creating custom dashboards [#custom-dashboards]

While the built-in UI provides valuable insights, you may want to create custom dashboards for:

- Monitoring multiple collectors at once
- Tracking custom component metrics
- Combining collector metrics with other data
- Creating team-specific views

### Example dashboard queries

**Collector health overview**

```sql
SELECT
  latest(otelcol.process.uptime) / 3600 AS 'Uptime (hours)',
  latest(otelcol.process.runtime.heap_alloc_bytes) / 1e9 AS 'Memory (GB)',
  rate(sum(otelcol.receiver.accepted_spans), 1 second) AS 'Throughput (spans/s)'
FROM Metric
WHERE newrelic.service.type = 'otel_collector'
FACET service.name
```

**Pipeline efficiency**

```sql
SELECT
  rate(sum(otelcol.receiver.accepted_spans), 1 second) AS 'Received',
  rate(sum(otelcol.exporter.sent_spans), 1 second) AS 'Sent',
  rate(sum(otelcol.processor.dropped_spans), 1 second) AS 'Dropped'
FROM Metric
WHERE newrelic.service.type = 'otel_collector'
AND service.name = 'YOUR_COLLECTOR_NAME'
TIMESERIES
```

**Component breakdown**

```sql
SELECT rate(sum(otelcol.receiver.accepted_spans), 1 second) AS 'Spans/sec'
FROM Metric
WHERE newrelic.service.type = 'otel_collector'
FACET receiver, service.name
TIMESERIES
```

## Setting up alerts [#alerts]

Create alerts to be notified of collector issues:

### High queue depth

Alert when exporter queues are filling up:

```sql
FROM Metric
SELECT latest(otelcol.exporter.queue_size)
WHERE newrelic.service.type = 'otel_collector'
FACET service.name, exporter
```

Threshold: Alert when queue size &gt; 80% of capacity

### Data drops

Alert when data is being dropped:

```sql
FROM Metric
SELECT rate(sum(otelcol.processor.dropped_spans), 1 second)
WHERE newrelic.service.type = 'otel_collector'
FACET service.name
```

Threshold: Alert when dropped spans &gt; 0 for more than 5 minutes

### High memory usage

Alert on memory growth:

```sql
FROM Metric
SELECT latest(otelcol.process.runtime.heap_alloc_bytes) / 1e9
WHERE newrelic.service.type = 'otel_collector'
FACET service.name
```

Threshold: Alert when memory &gt; expected baseline + 50%

### Collector down

Alert when collector stops reporting:

```sql
FROM Metric
SELECT count(*)
WHERE newrelic.service.type = 'otel_collector'
FACET service.name
```

Threshold: Alert when no data received for 5 minutes

## Toggling the UI experience [#toggling]

You can enable or disable the dedicated collector UI by managing the entity tag:

### Enable the UI

Add the tag `newrelic.service.type: otel_collector`:
1. Go to your collector entity in APM
2. Click the tag icon
3. Add `newrelic.service.type` with value `otel_collector`

### Disable the UI

Remove the tag:
1. Go to your collector entity in APM
2. Click the tag icon
3. Remove the `newrelic.service.type: otel_collector` tag

The entity will then display in the standard APM UI instead of the collector-specific UI.
