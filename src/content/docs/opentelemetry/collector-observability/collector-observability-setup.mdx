This guide walks you through enabling internal telemetry for your OpenTelemetry collector and configuring it to send monitoring data to New Relic.

## Prerequisites [#prerequisites]

Before you begin, ensure you have:

- **NRDOT Collector version 1.10.0 or later**: This feature requires a recent version of the New Relic distribution of the OpenTelemetry collector
- **New Relic license key**: Your [ingest license key](/docs/apis/intro-apis/new-relic-api-keys/#license-key) for authentication
- **Configuration file access**: Ability to modify and mount configuration files to your collector
- **Network access**: Your collector must be able to reach New Relic's OTLP endpoints

<Callout variant="tip">
You can use a different license key for internal telemetry than the one your collector uses to send regular data. This can help you track collector observability costs separately.
</Callout>

## Step 1: Download the internal telemetry configuration [#download-config]

The collector does not emit [internal telemetry](https://opentelemetry.io/docs/collector/internal-telemetry/) by default, so you need to enable it with a configuration file.

Download the pre-built internal telemetry configuration:

```bash
curl -L https://raw.githubusercontent.com/newrelic/nrdot-collector-releases/refs/tags/1.10.0/examples/internal-telemetry-config.yaml \
  --silent --output internal-telemetry-config.yaml
```

This configuration file includes:
- Internal metrics collection
- Log sampling configuration
- The `newrelic.service.type: otel_collector` resource attribute (for UI opt-in)
- OTLP exporter for sending data to New Relic

<Callout variant="important">
For reliability in production, we recommend downloading and storing this configuration file rather than referencing it directly via URL. While the collector can load configurations from URLs, this creates a dependency on external network availability during collector startup.
</Callout>

## Step 3: Run collector with merged configuration [#run-collector]

The collector supports merging multiple configuration files. Add the internal telemetry configuration as a second `--config` argument to your existing collector startup command.

### Docker example

```bash
docker run \
  -e INTERNAL_TELEMETRY_NEW_RELIC_LICENSE_KEY='YOUR_LICENSE_KEY' \
  -e NEW_RELIC_LICENSE_KEY='YOUR_LICENSE_KEY' \
  -e INTERNAL_TELEMETRY_SERVICE_NAME='demo-collector' \
  -v './internal-telemetry-config.yaml:/etc/nrdot-collector/config-internal.yaml' \
  newrelic/nrdot-collector:1.10.0 \
  --config=/etc/nrdot-collector/config.yaml \
  --config='/etc/nrdot-collector/config-internal.yaml'
```

### Docker Compose example

```yaml
services:
  otel-collector:
    image: newrelic/nrdot-collector:1.10.0
    environment:
      - INTERNAL_TELEMETRY_NEW_RELIC_LICENSE_KEY=${INTERNAL_TELEMETRY_NEW_RELIC_LICENSE_KEY}
      - NEW_RELIC_LICENSE_KEY=${NEW_RELIC_LICENSE_KEY}
      - INTERNAL_TELEMETRY_SERVICE_NAME=demo-collector
    volumes:
      - ./config.yaml:/etc/nrdot-collector/config.yaml
      - ./internal-telemetry-config.yaml:/etc/nrdot-collector/config-internal.yaml
    command:
      - "--config=/etc/nrdot-collector/config.yaml"
      - "--config=/etc/nrdot-collector/config-internal.yaml"
```

### Kubernetes example

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: otel-collector
spec:
  template:
    spec:
      containers:
      - name: otel-collector
        image: newrelic/nrdot-collector:1.10.0
        env:
        - name: INTERNAL_TELEMETRY_NEW_RELIC_LICENSE_KEY
          valueFrom:
            secretKeyRef:
              name: newrelic-secret
              key: license-key
        - name: INTERNAL_TELEMETRY_SERVICE_NAME
          value: "prod-k8s-collector"
        - name: INTERNAL_TELEMETRY_OTLP_ENDPOINT
          value: "https://otlp.nr-data.net"
        args:
        - "--config=/etc/otel/config.yaml"
        - "--config=/etc/otel/config-internal.yaml"
        volumeMounts:
        - name: config
          mountPath: /etc/otel/config.yaml
          subPath: config.yaml
        - name: internal-config
          mountPath: /etc/otel/config-internal.yaml
          subPath: internal-telemetry-config.yaml
      volumes:
      - name: config
        configMap:
          name: otel-collector-config
      - name: internal-config
        configMap:
          name: otel-collector-internal-config
```

<Callout variant="important">
**Configuration merge order matters!** The collector uses a "last one wins" strategy when merging configurations. If you have existing configuration under the `service::telemetry` node, place the internal telemetry config last to ensure it takes precedence:

```bash
--config=/etc/nrdot-collector/config.yaml \
--config='/etc/nrdot-collector/config-internal.yaml'
```

Certain parts of the configuration (lists, leaf nodes) cannot be merged and will be overwritten by the last `--config` argument.
</Callout>

### Alternative: URL-based configuration (not recommended)

For testing purposes only, you can reference the configuration directly via URL:

```bash
docker run \
  -e INTERNAL_TELEMETRY_NEW_RELIC_LICENSE_KEY='YOUR_LICENSE_KEY' \
  -e NEW_RELIC_LICENSE_KEY='YOUR_LICENSE_KEY' \
  -e INTERNAL_TELEMETRY_SERVICE_NAME='demo-collector' \
  newrelic/nrdot-collector:1.10.0 \
  --config=/etc/nrdot-collector/config.yaml \
  --config='https://raw.githubusercontent.com/newrelic/nrdot-collector-releases/refs/tags/1.10.0/examples/internal-telemetry-config.yaml'
```

<Callout variant="caution">
This approach is not recommended for production as it creates a network dependency during collector startup. If the URL is unavailable, your collector will fail to start.
</Callout>

## Step 4: Add entity tag for UI opt-in [#add-tag]

To see your collector in the dedicated APM UI experience, you need to add the entity tag `newrelic.service.type: otel_collector`. Choose one of the following methods:

### Option 1: Via APM UI (recommended)

1. Navigate to **APM & Services** in New Relic
2. Find your collector entity (named by `INTERNAL_TELEMETRY_SERVICE_NAME`)
3. Click on the tag icon at the top of the page
4. Add the tag: `newrelic.service.type` with value `otel_collector`

This method allows you to easily toggle the UI experience on or off by adding or removing the tag.

### Option 2: Via configuration argument

Add the tag as a resource attribute using a configuration argument:

```bash
docker run \
  -e INTERNAL_TELEMETRY_NEW_RELIC_LICENSE_KEY='YOUR_LICENSE_KEY' \
  -e NEW_RELIC_LICENSE_KEY='YOUR_LICENSE_KEY' \
  newrelic/nrdot-collector:1.10.0 \
  --config=/etc/nrdot-collector/config.yaml \
  --config=/etc/nrdot-collector/config-internal.yaml \
  --config='yaml:service::telemetry::resource::newrelic.service.type: otel_collector'
```

This adds the attribute as a resource attribute, and New Relic automatically creates the entity tag on ingest.

<Callout variant="tip">
If you remove this configuration option later, it will take approximately one day for the tag to expire.
</Callout>

### Option 3: Use the example configuration

The example internal telemetry configuration file already includes this resource attribute, so if you're using it without modifications, no additional action is needed.

## Step 5: Verify setup [#verify]

After completing the setup, verify that collector observability is working:

### Check entity creation

1. Go to **APM & Services** in New Relic
2. Search for your collector by the name you specified in `INTERNAL_TELEMETRY_SERVICE_NAME`
3. Verify the entity appears and has the `newrelic.service.type: otel_collector` tag

### Check data ingest

Run the following NRQL query to verify metrics are being ingested:

```sql
FROM Metric
SELECT count(*)
WHERE entity.name = 'YOUR_COLLECTOR_NAME'
AND instrumentation.provider = 'opentelemetry'
SINCE 10 minutes ago
```

Replace `YOUR_COLLECTOR_NAME` with your `INTERNAL_TELEMETRY_SERVICE_NAME`.

### Check collector logs

Review your collector logs for any errors related to internal telemetry:

```bash
docker logs <container-id> | grep -i telemetry
```

Look for successful OTLP exports and absence of authentication errors.

## Customization [#customization]

The internal telemetry configuration exposes several environment variables for customization:

### Metrics detail level

```bash
INTERNAL_TELEMETRY_METRIC_LEVEL=detailed  # Options: none, basic, normal, detailed
```

### Log sampling

```bash
INTERNAL_TELEMETRY_LOG_LEVEL=info          # Options: debug, info, warn, error
INTERNAL_TELEMETRY_LOG_SAMPLING=enabled    # Enable/disable log sampling
```

### Trace configuration (experimental)

```bash
INTERNAL_TELEMETRY_TRACE_LEVEL=info        # Enable traces
INTERNAL_TELEMETRY_TRACE_SAMPLE_RATIO=0.001 # Sample 0.1% of traces
```

<Callout variant="caution">
Traces are disabled by default and considered experimental. See [Understanding data overhead](/docs/opentelemetry/collector-observability/collector-telemetry-overhead#traces) before enabling traces.
</Callout>

For the complete list of customization options, refer to the [configuration file](https://github.com/newrelic/nrdot-collector-releases/blob/main/examples/internal-telemetry-config.yaml).

## Troubleshooting [#troubleshooting]

### Collector entity not appearing

If your collector entity doesn't appear in APM:

1. **Verify data is being sent**: Run the NRQL query in the [verification step](#verify)
2. **Check license key**: Ensure `INTERNAL_TELEMETRY_NEW_RELIC_LICENSE_KEY` is correct
3. **Verify endpoint**: Confirm you're using the correct OTLP endpoint for your region
4. **Check collector logs**: Look for OTLP exporter errors
5. **Wait for entity synthesis**: New entities may take a few minutes to appear

### Tag not being applied

If the `newrelic.service.type` tag isn't showing:

1. **Manual application**: Try adding the tag via the UI (Option 1)
2. **Check resource attributes**: Verify the resource attribute is in your telemetry data:
   ```sql
   FROM Metric
   SELECT uniques(newrelic.service.type)
   WHERE entity.name = 'YOUR_COLLECTOR_NAME'
   ```
3. **Configuration order**: Ensure internal telemetry config is last in merge order

### Configuration merge conflicts

If you're experiencing unexpected behavior:

1. **Review merge order**: Ensure `--config` arguments are in the correct order
2. **Check for conflicts**: Look for overlapping configuration nodes
3. **Test in isolation**: Try running with only the internal telemetry config to verify it works
4. **Validate YAML**: Ensure your configuration files are valid YAML

### Missing metrics

If some metrics aren't appearing:

1. **Check metric level**: Ensure `INTERNAL_TELEMETRY_METRIC_LEVEL` is set appropriately
2. **Verify components**: Only active components emit metrics
3. **Review documentation**: Check if metrics are part of [standard internal telemetry](https://opentelemetry.io/docs/collector/internal-telemetry/)

### Infrastructure relationships not forming

If your collector isn't showing infrastructure relationships:

1. **Wait for synthesis**: Relationships can take up to 70 minutes to form
2. **Verify infrastructure monitoring**: Ensure your host/container is instrumented
3. **Check resource attributes**: See [examples](/docs/opentelemetry/collector-observability/collector-observability-examples) for proper configuration

## Current limitations [#limitations]

<Callout variant="important">
Collector observability is in public preview. The following limitations apply:
</Callout>

### Telemetry stability

- **Schema changes**: Collector telemetry schema is not stable during preview. Metric names and structures may change.
- **Version support**: Only the most recent telemetry version is supported during preview
  - Supported: `http.client.request.duration` (collector >= v0.128.0, NRDOT >= v1.10.0)
  - Not supported: `http.request.duration` (older naming)
- **Backwards compatibility**: Will be provided starting at General Availability

### Export format

- **OTLP only**: UI expects OTLP exporter format
- **Prometheus not supported**:
  - ✅ `http.client.request.duration` (OTLP format)
  - ❌ `http_client_request_duration` (Prometheus format)

### Custom component metrics

- **Standard metrics supported**: Metrics from [internal telemetry documentation](https://opentelemetry.io/docs/collector/internal-telemetry/)
- **Custom metrics**: Limited support for custom metrics from contrib components
- **Workaround**: Create custom dashboards for non-standard metrics

### APM features

The following APM features are hidden as they don't apply to the collector's stream-processing nature:

- Apdex score
- Traditional error rate
- Transaction traces
- `apm.%` metrics

### Infrastructure integration

- **Container golden metrics**: Partial support; some infrastructure panel columns may be unpopulated
- **Relationship synthesis**: May take up to 70 minutes to appear
