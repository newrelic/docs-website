Set up Microsoft SQL Server monitoring using the NRDOT collector on Windows environments to monitor AWS RDS for SQL Server instances.

Based on your RDS SQL Server security configuration, you can choose from three authentication methods to connect the NRDOT Collector to your RDS instance:

### Prerequisites [#prerequisites-sql]

Ensure your environment meets our [compatibility and prerequisites](/docs/opentelemetry/database/mssql/compatibility).

<Step>

### Set up the NRDOT Collector [#setup-sql]

Install the NRDOT Collector on an EC2 Windows instance or a local Windows machine that can connect to your RDS SQL Server instance.

<Callout variant="important">
  For RDS monitoring, install the NRDOT Collector on an EC2 instance in the same VPC as your RDS instance for optimal network connectivity and lower latency.
</Callout>

1. Download the installer by pasting this URL into your browser:

   ```powershell
   https://download.newrelic.com/database-monitoring/nrdot-collector/nrdot-collector-windows-amd64-latest.msi
   ```

2. Locate your downloaded file and double-click it to install the NRDOT Collector.

3. Ensure your security group allows inbound connections from the NRDOT Collector host to your RDS instance on port 1433 (or your custom port).

</Step>

<Step>

### Configure database user [#configure-user-sql]

Connect to your RDS SQL Server instance using SQL Server Management Studio (SSMS) or sqlcmd with your admin credentials.

Run the following script to create the `newrelic` monitoring user with appropriate permissions:

```sql
USE [master];
GO
CREATE LOGIN [newrelic] WITH PASSWORD = '<PASSWORD>';
GO
-- Instance-level permissions
GRANT VIEW SERVER STATE TO [newrelic];
GRANT VIEW ANY DEFINITION TO [newrelic];
GRANT VIEW ANY DATABASE TO [newrelic];
GO
-- Grant read access privileges to all user databases
DECLARE @name SYSNAME;
DECLARE db_cursor CURSOR READ_ONLY FORWARD_ONLY FOR
SELECT [name]
FROM [master].[sys].[databases]
WHERE [name] NOT IN ('master', 'msdb', 'tempdb', 'model', 'rdsadmin', 'distribution')
AND [state] = 0; -- Only online databases
OPEN db_cursor;
FETCH NEXT FROM db_cursor INTO @name;
WHILE @@FETCH_STATUS = 0
BEGIN
  BEGIN TRY
    PRINT 'Processing database: ' + @name;
    EXEC('USE [' + @name + '];
    IF NOT EXISTS (SELECT 1 FROM sys.database_principals WHERE name = ''newrelic'')
    BEGIN
      CREATE USER [newrelic] FOR LOGIN [newrelic];
    END;
    GRANT VIEW DATABASE STATE TO [newrelic];
    GRANT VIEW DEFINITION TO [newrelic];
    ALTER ROLE db_datareader ADD MEMBER [newrelic];');
    PRINT 'Success: ' + @name;
  END TRY
  BEGIN CATCH
    PRINT 'Error on ' + @name + ': ' + ERROR_MESSAGE();
  END CATCH
  FETCH NEXT FROM db_cursor INTO @name;
END
CLOSE db_cursor;
DEALLOCATE db_cursor;
GO
```

<Callout variant="tip">
  **For RDS:** Use your RDS master username (or a user with `rds_superuser` role) to execute these commands. The script automatically excludes the `rdsadmin` database which is reserved by AWS.
</Callout>

</Step>

<Step>

### Restart NRDOT collector [#restart-sql]

After updating your configuration, restart the NRDOT Collector service on your EC2 or local Windows machine:

```powershell
net stop nrdot-collector
net start nrdot-collector
```

<Callout variant="tip">
  Always restart the NRDOT collector service after making configuration changes to ensure the new settings take effect.
</Callout>

</Step>

<Step>

### (Optional) Set up APM-database correlation [#correlation-sql]

To correlate your application performance with database operations, you can set up database service identification. This feature allows you to see exactly which applications are generating specific database workloads. For more information, refer to [set up database service identification](/docs/opentelemetry/database/database-service-identification/) to get APM-database correlation in New Relic.

<Callout variant="important">
  To view database performance data in APM, both entities must be in the same New Relic account. If the entities are in different New Relic accounts, you must have access to both accounts to view the data.
</Callout>

</Step>

<Step>

### (Optional) Configure secret management [#secret-management-sql]

The secret management feature allows you to securely manage sensitive information, such as database credentials. This helps to enhance the security of your monitoring setup by avoiding hardcoding sensitive data in configuration files. For more information, refer to [secret management](/docs/opentelemetry/database/database-service-identification/#secret-management).

</Step>

<Step>

### Find and use your data [#find-sql]

Once your data is being collected, you can access comprehensive SQL Server database monitoring through New Relic's UI.

To find your SQL Server database entity in New Relic:

1. Go to <DNT>**[https://one.newrelic.com](https://one.newrelic.com) > All Capabilities > Databases**</DNT>.
2. Set the search criteria as <DNT>`instrumentation.provider = opentelemetry`</DNT>.
3. Select your SQL Server database from the list of entities.

</Step>

<Step>

### Prerequisites [#prerequisites-domain]

Ensure your environment meets our [compatibility and prerequisites](/docs/opentelemetry/database/mssql/compatibility).

**Additional requirements for Windows Domain Authentication with RDS:**
- Windows domain environment with Active Directory
- Domain-joined Windows EC2 instance or local machine 
- RDS SQL Server configured to accept Windows Authentication
- Domain user account with appropriate SQL Server permissions
- Security group configured to allow connections on SQL Server port (default 1433)

</Step>

<Step>

### Enable your Microsoft SQL Server [#enable-sql-server-domain]

Connect to your RDS SQL Server instance using SQL Server Management Studio (SSMS) or sqlcmd with your admin credentials.

Run the following script to create a Windows domain login and grant necessary permissions:

```sql
USE master;
CREATE LOGIN [DomainName\DomainUser] FROM WINDOWS;
GRANT CONNECT SQL TO [DomainName\DomainUser];
GRANT VIEW SERVER STATE TO [DomainName\DomainUser];
GRANT VIEW ANY DEFINITION TO [DomainName\DomainUser];
GO
```

<Callout variant="important">
  Replace `DomainName\DomainUser` with your actual domain and username (e.g., `CORP\SQLMonitor`).
</Callout>

</Step>

<Step>

### Grant READ access privileges [#grant-permissions-domain]

Use the following statements to grant READ access privileges to the domain user across all databases:

```sql
DECLARE @name SYSNAME
DECLARE db_cursor CURSOR
READ_ONLY FORWARD_ONLY
FOR
SELECT NAME
FROM master.sys.databases
WHERE NAME NOT IN ('master','msdb','tempdb','model','rdsadmin','distribution')
OPEN db_cursor
FETCH NEXT FROM db_cursor INTO @name 
WHILE @@FETCH_STATUS = 0
BEGIN
    EXECUTE('USE "' + @name + '"; CREATE USER [DomainName\DomainUser] FOR LOGIN [DomainName\DomainUser];' );
    FETCH NEXT FROM db_cursor INTO @name
END
CLOSE db_cursor
DEALLOCATE db_cursor
GO
```

<Callout variant="tip">
  **For RDS:** The script automatically excludes the `rdsadmin` database which is reserved by AWS. Ensure you replace `DomainName\DomainUser` with your actual domain credentials.
</Callout>

</Step>

<Step>

### Set up the NRDOT Collector [#setup-domain]

Install the NRDOT Collector on a domain-joined EC2 Windows instance or a local Windows machine that is part of the same domain and can connect to your RDS SQL Server instance.

1. Download the installer by pasting this URL into your browser:

   ```powershell
   https://download.newrelic.com/database-monitoring/nrdot-collector/nrdot-collector-windows-amd64-latest.msi
   ```

2. Locate your downloaded file and double-click it to install the NRDOT Collector.

3. **Change Service Logon to Domain Account (Using PowerShell):**

   Configure the NRDOT Collector service to run as your domain user using PowerShell:

   ```powershell
   # Configure service to use domain account
   # The space after obj= and password= is required syntax for sc.exe
   sc.exe config "nrdot-collector" obj= "OTEL\Admin" password= "<password>"
   
   # Restart the service
   Restart-Service nrdot-collector
   ```

   Verify the service identity:

   ```powershell
   Get-WmiObject Win32_Service -Filter "Name='nrdot-collector'" | Select Name, State, StartName
   # Expected: StartName = OTEL\Admin
   ```

   <Callout variant="important">
     Replace `OTEL\Admin` with your actual domain and username (e.g., `CORP\SQLMonitor`), and `<password>` with the domain user's password.
   </Callout>

4. Ensure your security group allows inbound connections from the NRDOT Collector host to your RDS instance on port 1433 (or your custom port).

</Step>

<Step>

### Configure NRDOT Collector [#configure-domain]

Update your NRDOT Collector configuration file at `C:\Program Files\NRDOT Collector Host\config.yaml`.

<Callout variant="tip">
  **RDS Endpoint Format:** Your RDS endpoint will be in the format: `your-db-instance.xxxxxxxxxxxx.region.rds.amazonaws.com`. Find this in your AWS RDS Console under "Connectivity & security".
</Callout>

<Callout variant="important">
  Windows Domain Authentication uses integrated security (Windows authentication), which uses the same configuration approach as gMSA authentication. Simply update the `datasource` server value to your RDS endpoint. The NRDOT service will authenticate using the domain account you configured in Step 3.
</Callout>

Use the configuration below with your RDS endpoint:

```yaml
# NRDOT Collector Configuration for RDS SQL Server - Windows Domain Auth
receivers:
  nrsqlserver:
    collection_interval: 15s
    # Windows integrated security - no username/password needed
    datasource: "server=<YOUR-RDS-ENDPOINT>;integrated security=true;encrypt=true;TrustServerCertificate=true;"

    # Enable query performance monitoring
    events:
      db.server.query_sample:
        enabled: true
      db.server.top_query:
        enabled: true

    # Top query collection
    top_query_collection:
      lookback_time: 15s
      max_query_sample_count: 1000
      top_query_count: 20
      collection_interval: 15s

    # Query sample collection
    query_sample_collection:
      max_rows_per_query: 20

    # APM correlation
    collect_full_query_text: true
    allowed_comment_keys:
      - nr_service_guid

    # Core metrics (uptime, CPU count only)
    metrics:
      sqlserver.computer.uptime:
        enabled: true
      sqlserver.cpu.count:
        enabled: true

processors:
  batch:
    timeout: 10s
    send_batch_size: 1024

exporters:
  otlphttp:
    endpoint: "https://otlp.nr-data.net"
    headers:
      api-key: "YOUR_LICENSE_KEY"
    compression: gzip

  debug:
    verbosity: detailed

service:
  telemetry:
    metrics:
      level: none

  pipelines:
    metrics:
      receivers: [nrsqlserver]
      processors: [batch]
      exporters: [otlphttp, debug]

    logs:
      receivers: [nrsqlserver]
      processors: [batch]
      exporters: [otlphttp, debug]
```

Example with RDS endpoint:
```yaml
datasource: "server=mydb-instance.c9akciq32.us-east-1.rds.amazonaws.com;integrated security=true;encrypt=true;TrustServerCertificate=true;"
```

</Step>

<Step>

### Prerequisites [#prerequisites-gmsa]

Ensure your environment meets our [compatibility and prerequisites](/docs/opentelemetry/database/mssql/compatibility).

**Additional requirements for gMSA (Group Managed Service Account) with RDS:**
- Windows Server 2012 or later
- Active Directory environment with gMSA support
- Domain controller with Key Distribution Service (KDS) root key configured
- gMSA account created and configured for SQL Server access
- Domain-joined Windows EC2 instance authorized to use the gMSA account
- Security group configured to allow connections on SQL Server port (default 1433)

</Step>

<Step>

### Enable your Microsoft SQL Server [#enable-sql-server-gmsa]

Connect to your RDS SQL Server instance and run the following SQL commands to create a login for the gMSA account and grant necessary permissions:

```sql
USE master;
CREATE LOGIN [DomainName\gMSA] FROM WINDOWS;
GRANT CONNECT SQL TO [DomainName\gMSA];
GRANT VIEW SERVER STATE TO [DomainName\gMSA];
GRANT VIEW ANY DEFINITION TO [DomainName\gMSA];
```

<Callout variant="important">
  Replace `DomainName\gMSA` with your actual domain name and gMSA account name (e.g., `CORP\SQL-gMSA$`). Note that gMSA accounts typically end with a `$` symbol.
</Callout>

</Step>

<Step>

### Set up the NRDOT Collector [#setup-gmsa]

Install the NRDOT Collector on an EC2 Windows instance that is domain-joined and authorized to use the gMSA account.

<Callout variant="important">
  For RDS with gMSA, the NRDOT Collector **must** be installed on a domain-joined Windows EC2 instance that has permissions to use the gMSA account. Ensure your security group allows connections to RDS on port 1433.
</Callout>

1. Download the installer by pasting this URL into your browser:

   ```powershell
   https://download.newrelic.com/database-monitoring/nrdot-collector/nrdot-collector-windows-amd64-latest.msi
   ```

2. Locate your downloaded file and double-click it to install the NRDOT Collector.

</Step>

<Step>

### Configure NRDOT Collector Service to Run as gMSA [#configure-service-gmsa]

After installing the NRDOT Collector, configure the service to run as your gMSA account using PowerShell:

```powershell
# Stop the NRDOT Collector service
Stop-Service nrdot-collector

# Configure service to use gMSA account
# Use cmd /c to avoid PowerShell treating $ as a variable
cmd /c 'sc config nrdot-collector obj= "OTEL\sqlotel$" password= ""'

# Verify the configuration
Get-WmiObject Win32_Service -Filter "Name='nrdot-collector'" | Select Name, StartName
```

<Callout variant="important">
  Replace `OTEL\sqlotel$` with your actual domain and gMSA account name (e.g., `CORP\SQL-gMSA$`). Note the empty password - gMSA accounts don't require passwords.
</Callout>

</Step>
