## What metrics are collected? [#metrics]

OpenTelemetry Kafka monitoring collects comprehensive metrics through multiple receivers:

### [Kafka metrics receiver](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/receiver/kafkametricsreceiver)
Collects metrics by connecting to Kafka's bootstrap port:
* **Broker metrics**: Number of active brokers in the cluster
* **Topic metrics**: Partition count, replication factor, and minimum in-sync replicas per topic
* **Partition metrics**: Current and oldest offsets, replica counts, and in-sync replica status
* **Consumer group metrics**: Consumer lag, offset positions, lag aggregations, and group membership

### [JMX receiver](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/receiver/jmxreceiver)
Collects detailed broker metrics through JMX:

**Default Kafka target system metrics:**
* **Message metrics**: Total messages received by brokers
* **Request metrics**: Request counts, failure rates, processing times (average, 50th/99th percentile), and queue sizes
* **Network I/O**: Bytes in and out per broker
* **Partition health**: Partition counts, offline partitions, and under-replicated partitions
* **ISR operations**: In-sync replica shrink and expand operations
* **Replication**: Maximum lag between follower and leader replicas
* **Controller metrics**: Active controller count, leader election rates, and unclean election rates
* **Purgatory**: Number of requests waiting in purgatory
* **Log flush**: Log flush counts and timing percentiles

**Custom JMX metrics** (from additional configuration):
* **Per-topic metrics**: Message counts and byte rates per individual topic
* **Cluster-level**: Total topic count, partition count, fenced brokers, and preferred leader imbalance
* **Broker-specific**: Partitions under minimum ISR, broker uptime, and leader partition count
* **JVM metrics**: Garbage collection, heap memory usage, thread counts, CPU utilization, file descriptors, class loading, and memory pool details

For a complete list with metric names, see [Kafka metrics reference](/docs/opentelemetry/integrations/kafka/metrics-reference).

## How it works [#how-it-works]

The OpenTelemetry Collector uses two complementary receivers to collect comprehensive Kafka metrics:

* **Kafka metrics receiver** - Connects to Kafka's bootstrap port to collect cluster and topic-level metrics
* **JMX receiver** - Connects to JMX port to collect broker-specific metrics and JVM metrics

The collector then processes and exports this data to New Relic via the OTLP exporter. Each deployment guide provides complete configuration examples with specific prerequisites tailored to your environment (self-hosted or Kubernetes).

## Instrument producer and consumer applications [#instrument-apps]

Optionally instrument your Kafka producer and consumer applications using the [OpenTelemetry Java Agent](https://opentelemetry.io/docs/zero-code/java/agent/getting-started/) to collect:

* Request latencies
* Throughput metrics
* Error rates
* Distributed traces

This provides end-to-end visibility from producers through brokers to consumers.

The Java Agent provides [out-of-the-box Kafka instrumentation](https://opentelemetry.io/docs/zero-code/java/spring-boot-starter/out-of-the-box-instrumentation/) with zero code changes. For advanced configuration, see the [Kafka instrumentation documentation](https://github.com/open-telemetry/opentelemetry-java-instrumentation/tree/main/instrumentation/kafka).

* Go to <DNT>**[one.newrelic.com](https://one.newrelic.com) > All Capabilities > On Host Integrations**</DNT>
* View Kafka broker, cluster, and topics entity types

### 5. Query with NRQL

You can also query Kafka metrics using NRQL:

```sql
FROM Metric SELECT *
WHERE kafka.cluster.name = 'production-cluster' AND instrumentation.provider='opentelemetry'
```
