Amazon ECS monitoring provides real-time visibility into your containerized workloads running on AWS Elastic Container Service to ensure reliable performance and prevent resource bottlenecks in distributed systems. Using OpenTelemetry Collector, you get comprehensive monitoring through a flexible, vendor-neutral approach that works across both ECS on EC2 and ECS Fargate launch types.

## Collector requirements [#collector-requirements]

ECS monitoring requires the **OpenTelemetry Collector Contrib** distribution, which includes the AWS ECS Container Metrics receiver. This receiver is not currently available in the NRDOT Collector distribution.

* **OpenTelemetry Collector Contrib**: The community distribution that includes the `awsecscontainermetricsreceiver`. For more information, see the [OpenTelemetry Collector Contrib GitHub repository](https://github.com/open-telemetry/opentelemetry-collector-releases/tree/main/distributions/otelcol-contrib).

<Callout variant="important">
  The AWS ECS Container Metrics receiver (`awsecscontainermetricsreceiver`) is only available in the OpenTelemetry Collector Contrib distribution. NRDOT Collector does not currently support this receiver.
</Callout>

## Why ECS monitoring?
- Prevent outages - Get alerts for container failures, resource exhaustion, and service health issues before they cause downtime
- Optimize performance - Identify CPU/memory bottlenecks, I/O constraints, and resource allocation issues that affect application speed
- Plan capacity - Track resource usage, task scaling patterns, and cluster utilization to scale proactively
- Ensure service availability - Monitor task health, service deployment status, and container lifecycle events

### Common use case

Whether you're running microservices on ECS Fargate for serverless containers, or managing containerized applications on ECS with EC2 for cost optimization, ECS monitoring helps you catch issues before they impact your business. Get alerted when CPU utilization spikes threatening application performance, when memory usage approaches limits risking task termination, or when container health checks fail indicating service degradation. This monitoring is essential for web applications, API services, and any containerized workload where performance and availability directly impact user experience.

## How it works [#how-it-works]

The OpenTelemetry Collector runs as a sidecar container in your ECS task definitions to collect comprehensive metrics:

**Data collection:**
- **[AWS ECS Container Metrics receiver](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/receiver/awsecscontainermetricsreceiver)** connects to ECS Task Metadata Endpoint v4 to collect container CPU, memory, network, and storage metrics
- **[Host Metrics receiver](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/receiver/hostmetricsreceiver)** collects system-level metrics including CPU, memory, filesystem, disk, and network utilization (available on EC2, limited on Fargate)

**Data processing:**
- **Resource detection processor** automatically discovers ECS metadata including cluster name, service name, task ID, and launch type
- **Metrics transform processors** normalize container metrics for better compatibility with monitoring dashboards
- **Filter processors** exclude irrelevant metrics to optimize data volume and costs

**Data flow:**
1. Collector gathers container metrics from ECS Task Metadata API and host metrics from system interfaces
2. Data is enriched with ECS resource attributes and filtered to remove unnecessary metrics
3. Metrics are batched and exported to New Relic via OTLP endpoint

## What you'll monitor [#what-monitor]

This integration provides comprehensive visibility into your ECS infrastructure:

**Container metrics:**
- CPU utilization and usage per container
- Memory utilization and usage per container
- Network I/O (bytes sent/received, packets, errors)
- Storage I/O (read/write operations and bytes)

**Host metrics (EC2 launch type):**
- System CPU utilization across all cores
- Memory usage and available memory
- Filesystem utilization and available space
- Disk I/O operations, timing, and throughput
- Network interface statistics and connections
- System load averages and process counts

**ECS resource attributes:**
- Cluster name and service name
- Task ID and task definition family
- Launch type (EC2 or Fargate)
- Container names and image information
- AWS region and availability zone
