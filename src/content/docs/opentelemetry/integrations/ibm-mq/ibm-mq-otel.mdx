Monitor your IBM MQ queue managers in New Relic by deploying the **NRDOT collector** — New Relic's supported, production-ready distribution of the [OpenTelemetry Collector](https://opentelemetry.io/docs/collector/). The collector scrapes the Prometheus `/metrics` endpoint that the [mq-metric-samples](https://github.com/ibm-messaging/mq-metric-samples) exporter — the IBM MQ team's Prometheus metrics exporter for IBM MQ — publishes for each queue manager, reshapes those metrics so they map cleanly onto New Relic's IBM MQ entity model, and forwards them to New Relic over OTLP/HTTP (the OpenTelemetry-native wire protocol).

The payoff: each queue manager becomes a `IBMMQ_MANAGER` entity in New Relic, with its queues as child `IBMMQ_QUEUE` entities — ready for dashboards, alerts, golden-signal views, and entity relationships. This is a pure OpenTelemetry path: it needs no New Relic Infrastructure agent and no on-host integration package. If you already run the OpenTelemetry Collector elsewhere in your estate, IBM MQ monitoring becomes one more pipeline in the same tool you already operate.

This guide covers two deployment shapes of the same pipeline:

* **Linux host** — the collector runs under systemd and scrapes a fixed set of exporter endpoints through **static** Prometheus receivers (one per queue manager). Best when your queue managers live on VMs, bare metal, or EC2 and the set of queue managers changes rarely.
* **Kubernetes** — the collector runs in-cluster as a Deployment and **auto-discovers** annotated queue manager pods through Prometheus `kubernetes_sd`. Best when your queue managers run as pods and the fleet scales up and down.

Both shapes run the **same processing chain** and produce the **identical metric shape and entities** — they differ only in how the collector *finds* the exporters (hardcoded endpoints on Linux, pod auto-discovery on Kubernetes). Pick the platform that matches where your collector will run; the data in New Relic looks the same either way.

<Callout variant="important">
  This guide assumes your IBM MQ broker and the [mq-metric-samples](https://github.com/ibm-messaging/mq-metric-samples) exporter (the IBM MQ team's Prometheus metrics exporter, which runs its `mq_prometheus` program) are **already running and exposing `/metrics`** for each queue manager. It documents only the New Relic OpenTelemetry bridge — the collector configuration, environment variables, and verification queries needed to pipe that data into New Relic.
</Callout>

## Architecture [#architecture-overview]

The data flows from each IBM MQ queue manager, through the mq-metric-samples exporter, into the NRDOT collector, and on to New Relic:

```
IBM MQ Queue Manager(s)        mq-metric-samples exporter         NRDOT / OTel collector            New Relic
  QM1 / QM2 / QM3      ──PCF──▶  (mq_prometheus)       ──HTTP──▶  prometheus receiver(s)  ──OTLP/HTTP──▶  one IBMMQ_MANAGER
  (broker)                       exposes /metrics                 → filter / transform /                  entity per QM
                                                                  cumulativetodelta → otlphttp           (target.name:qmgr)
```

Reading that flow left to right:

1. **Queue manager (broker).** Each IBM MQ queue manager exposes administrative statistics over PCF (Programmable Command Format). This is the source of truth — queue depths, channel status, connection counts, and queue-manager health all originate here.
2. **mq-metric-samples exporter.** The exporter (one process per queue manager) polls the queue manager over PCF on an interval and translates the results into Prometheus text-format metrics on an HTTP `/metrics` endpoint. This component is **required** and assumed to be already running — see [Before you begin](#prerequisites).
3. **NRDOT collector.** The collector scrapes each exporter's `/metrics`, runs the metrics through a processing chain (described below), and exports them. This is the only component this guide configures.
4. **New Relic.** Metrics arrive over OTLP/HTTP and are synthesized into IBM MQ entities (described in [The IBM MQ entity model](#entity-model)).

Inside the collector, each queue manager gets its own metrics pipeline with an identical, **order-sensitive** chain of processors:

```
prometheus → filter/ibmmq-overhead → filter/ibmmq-queues → resourcedetection
           → transform/ibmmq-cleanup → memory_limiter → cumulativetodelta → batch → otlphttp
```

Each stage has a job, and the order matters:

* **`prometheus`** — scrapes one exporter endpoint and attaches the entity-critical identity labels — `targetName` and the optional `clusterName` — to every datapoint. (These are renamed downstream to the dotted OTel form `target.name` / `cluster.name` by `transform/ibmmq-cleanup`, because Prometheus label names can't contain dots.)
* **`filter/ibmmq-overhead`** — drops the exporter's own self-metrics (`go_*`, `process_*`, `promhttp_*`, `scrape_*`), which carry no IBM MQ signal and would only add ingest cost.
* **`filter/ibmmq-queues`** — cardinality control: drops noisy `SYSTEM.*` and `AMQ.*` internal queues so you don't pay to store thousands of low-value time series.
* **`resourcedetection`** — discovers the host's identity (cloud, EC2, system) and stamps it on every metric, which populates the host and cloud widgets on New Relic's IBM MQ dashboards.
* **`transform/ibmmq-cleanup`** — the most entity-critical step: strips OpenTelemetry's auto-added `service.*` attributes that would otherwise route IBM MQ metrics onto the *collector's* own entity instead of an `IBMMQ_MANAGER` entity, and renames the injected identity labels to their OTel dotted form (`targetName` → `target.name`, `clusterName` → `cluster.name`) — the names New Relic's IBM MQ entity synthesis keys on. It must run *after* `resourcedetection`, which is why the order is load-bearing.
* **`memory_limiter`** — applies back-pressure so the collector sheds load instead of being OOM-killed under a metric spike.
* **`cumulativetodelta`** — converts Prometheus's cumulative counters to the delta temporality New Relic expects; without it, counter metrics render as flat lines.
* **`batch`** — bundles datapoints to reduce the number of OTLP HTTP requests.
* **`otlphttp`** — ships the finished metrics to your New Relic OTLP endpoint, authenticated with your license key.

The full annotated configuration for each processor lives in the platform-specific guides linked below; this overview is enough to understand *why* the chain is shaped the way it is.

### The IBM MQ entity model [#entity-model]

New Relic turns the incoming metrics into **entities** — the durable, navigable objects you see in the UI — using *entity synthesis* rules that key off three things in the data:

* the **underscore metric names** (`ibmmq_qmgr_status`, `ibmmq_queue_depth`, and so on),
* the raw **`qmgr`** and **`queue`** labels, and
* the **`target.name`** identity attribute the collector attaches (injected as the `targetName` scrape label, then renamed to `target.name`).

From those, New Relic synthesizes:

| Entity type | Composite key | What it represents |
|---|---|---|
| `IBMMQ_MANAGER` | `target.name:qmgr` | One queue manager (for example `prod-mq-01:QM1`) |
| `IBMMQ_QUEUE` | `target.name:qmgr:queue` | One queue within a queue manager |

Because the key starts with `target.name`, that value must be **stable for the life of the deployment** — it's effectively the entity's identity. The `cluster.name` tag rides along (useful for grouping queue managers that belong to the same MQ cluster) but is *not* part of the key, so you can leave it unset or change it later without orphaning entities.

<Callout variant="caution">
  **Do not rename or "dot" the metric names or the `qmgr` / `queue` labels.** New Relic's IBM MQ entity synthesis depends on the raw Prometheus shape: underscore metric names (`ibmmq_*`) and the raw `qmgr` / `queue` labels. Renaming `qmgr` to something like `ibmmq.queue_manager`, or dotting the metric names, **breaks entity synthesis** and your `IBMMQ_MANAGER` dashboards go blank. The two *identity* attributes are the deliberate exception: the `targetName` / `clusterName` scrape labels are renamed to the dotted OTel form `target.name` / `cluster.name` by `transform/ibmmq-cleanup`, because that is what synthesis keys on. Each queue manager synthesizes into one `IBMMQ_MANAGER` entity keyed as `target.name:qmgr`.
</Callout>

## Before you begin [#prerequisites]

This guide documents only the New Relic OpenTelemetry bridge — the collector and its configuration. The two upstream components (the queue manager and the exporter) and your New Relic account must already be in place. Confirm all three before you start:

* **A New Relic account and a license key.** You need a [New Relic account](https://newrelic.com/signup) and your <InlinePopover type="licenseKey"/>. The license key authenticates the collector's OTLP export; it's the single credential the collector needs to send data. Treat it as a secret — on Linux it lives in a root-owned env file, and on Kubernetes in a `Secret` (both covered in the platform guides). Note which New Relic region your account is in (US or EU): the OTLP endpoint differs, and sending to the wrong region will silently drop your data.

* **A running IBM MQ broker with a monitoring channel.** You need one or more queue managers reachable by a user and server-connection channel that has PCF query rights. The exporter uses this channel to poll administrative statistics; if its credentials are missing the right authority, the exporter connects but returns no queue-manager data. A least-privilege monitoring channel (rather than an admin channel) is the recommended production setup.

* **The mq-metric-samples exporter, already running (required).** [mq-metric-samples](https://github.com/ibm-messaging/mq-metric-samples) is the **official** IBM MQ metrics exporter, built and maintained by the IBM MQ team. It connects to a queue manager over PCF, polls its administrative statistics on an interval, and republishes them as Prometheus metrics on an HTTP `/metrics` endpoint by way of its `mq_prometheus` program. It is the **required** source of IBM MQ telemetry for this integration: the NRDOT collector is only a *scraper* — it can forward to New Relic exactly what the exporter publishes and nothing more — so a healthy exporter, one instance per queue manager, is a hard prerequisite.

  <Callout variant="important">
    Standing up and configuring mq-metric-samples is **outside the scope of this guide**. Follow the official [mq-metric-samples documentation](https://github.com/ibm-messaging/mq-metric-samples) from the IBM MQ team to install the exporter, point it at each queue manager, and expose `/metrics`. Return here once each exporter is live.
  </Callout>

  Verify each exporter is publishing IBM MQ metrics before you touch the collector:

  ```bash
  curl -s http://localhost:9157/metrics | grep '^ibmmq_qmgr_status'
  ```

  You should see one `ibmmq_qmgr_status{qmgr="..."}` line per queue manager with the value `2` (running). If the endpoint doesn't respond, the exporter isn't up; if it responds but `ibmmq_qmgr_status` is absent, the exporter can't reach the queue manager — check its channel name, connection name, credentials, and channel authority records against the mq-metric-samples documentation. The collector picks up automatically once `/metrics` is live.

### New Relic region endpoints [#region-endpoints]

The collector's OTLP endpoint must match your account's region. Sending to the wrong region **silently drops** your data — no error surfaces to the collector:

| Region | `NEW_RELIC_OTLP_ENDPOINT` |
|---|---|
| US (default) | `https://otlp.nr-data.net:4318` |

| FedRAMP | See [New Relic OTLP documentation](/docs/opentelemetry/best-practices/opentelemetry-otlp) for the current FedRAMP ingest endpoint |

### Set up on Linux [#linux]

Run the NRDOT (New Relic OpenTelemetry) collector on your Linux host from a single, standalone configuration file. The collector scrapes each queue manager's mq-metric-samples exporter with one static Prometheus receiver per queue manager, normalizes the metrics, and ships them to New Relic over OTLP/HTTP — all driven by environment variables (`NEW_RELIC_LICENSE_KEY`, `TARGET_NAME`, and one `IBMMQ_QM<N>_ENDPOINT` per queue manager).

At a glance, the Linux path is:

1. Install the OpenTelemetry collector on the host.
2. Save the standalone IBM MQ collector configuration locally.
3. Set the required environment variables (license key, a stable `TARGET_NAME`, and one exporter endpoint per queue manager).
4. Run the collector under systemd and confirm metrics arrive in New Relic.

The complete, copy-paste Linux setup — install commands, the full collector configuration, the environment-variable reference, the systemd unit, and verification queries — lives in the dedicated configuration guide: **[Configure OpenTelemetry for IBM MQ on Linux](/docs/opentelemetry/integrations/ibm-mq/ibm-mq-linux-config)**.

<Callout variant="caution">
  `TARGET_NAME` (emitted as the `target.name` attribute) is the first segment of every `IBMMQ_MANAGER` and `IBMMQ_QUEUE` entity GUID (`target.name:qmgr`). On Linux it defaults to the host's short hostname — choose a stable value once and never change it, because changing it mints brand-new entities and orphans the old ones.
</Callout>

### Set up on Kubernetes [#kubernetes]

Run a single collector in-cluster that **auto-discovers** every annotated IBM MQ exporter pod through the Prometheus `kubernetes_sd` (Kubernetes service discovery) mechanism — which finds pods by their labels and annotations instead of from a static endpoint list. One collector Deployment serves any number of queue managers, and adding a queue manager needs zero collector changes.

At a glance, the Kubernetes path is:

1. Create a Kubernetes Secret holding your New Relic license key and OTLP endpoint.
2. Define the collector's Helm values (the auto-discovery receiver, the processor pipeline, and the OTLP exporter).
3. Install the collector with the `open-telemetry/opentelemetry-collector` Helm chart.
4. Verify the collector pod is `1/1` and metrics arrive in New Relic.

The complete, copy-paste Kubernetes setup — the New Relic credentials secret, the full Helm `values.yaml`, the install command, the pod-annotation contract, and verification queries — lives in the dedicated configuration guide: **[Configure OpenTelemetry for IBM MQ on Kubernetes](/docs/opentelemetry/integrations/ibm-mq/ibm-mq-kubernetes-config)**.

<Callout variant="caution">
  Keep the `target.name` value stable. It is the first segment of every `IBMMQ_MANAGER` and `IBMMQ_QUEUE` entity GUID (`target.name:qmgr`); changing it after deploy mints new entities and orphans the old ones.
</Callout>

## Find your data [#find-data]

After a few minutes, IBM MQ metrics appear in New Relic. The pipeline preserves the raw Prometheus shape — underscore metric names (`ibmmq_*`) and the raw `qmgr` and `queue` labels — and stamps a `target.name` identity onto every metric. New Relic uses these to synthesize one `IBMMQ_MANAGER` entity per queue manager, keyed as `target.name:qmgr`.

<Callout variant="tip">
  Query metrics by their underscore names and raw labels — `ibmmq_qmgr_status`, `ibmmq_queue_depth`, `qmgr`, `queue`. The metric names are **not** dotted and the `qmgr` label is **not** renamed, so NRQL written against `ibmmq.*` or `ibmmq.queue_manager.name` will not match.
</Callout>

Confirm data is flowing — every IBM MQ metric:

```sql
FROM Metric SELECT count(*) WHERE metricName LIKE 'ibmmq_%' SINCE 30 minutes ago
```

Count the queue managers reporting (each becomes one `IBMMQ_MANAGER` entity):

```sql
FROM Metric SELECT uniqueCount(qmgr) WHERE metricName LIKE 'ibmmq_%' FACET target.name SINCE 30 minutes ago
```

Queue manager status (`2` = running), one row per queue manager:

```sql
FROM Metric SELECT latest(ibmmq_qmgr_status) FACET qmgr SINCE 30 minutes ago
```

Current depth of every monitored queue:

```sql
FROM Metric SELECT latest(ibmmq_queue_depth) FACET qmgr, queue SINCE 30 minutes ago
```

Dead-letter queue depth across the fleet (a common alert signal):

```sql
FROM Metric SELECT latest(ibmmq_queue_depth) WHERE queue = 'SYSTEM.DEAD.LETTER.QUEUE' FACET qmgr SINCE 30 minutes ago
```

To browse entities, go to **All entities** in the New Relic UI and filter for your `IBMMQ_MANAGER` and `IBMMQ_QUEUE` entities, or open a queue manager directly from any of the queries above.

## Metrics reference [#metrics]

The mq-metric-samples exporter publishes a large catalog of IBM MQ metrics — several hundred `ibmmq_*` time series per queue manager once MQI statistics are enabled — and the NRDOT collector forwards them to New Relic unchanged. All metrics keep the raw Prometheus shape: **underscore names** (`ibmmq_*`) and the raw **`qmgr`**, **`queue`**, **`channel`**, and **`topic`** labels (plus the `target.name` identity and optional `cluster.name` tag the collector attaches).

<Callout variant="important">
  **Enable MQI statistics on each queue manager, or the throughput and timing metrics stay empty.** Queue-manager health, queue depth, and channel status report out of the box, but the MQI activity metrics (the [queue throughput](#metrics-queue-throughput) and [queue manager MQI activity](#metrics-qmgr-mqi) tables — `ibmmq_queue_mqput_count`, `ibmmq_queue_average_queue_time_seconds`, and so on) require statistics collection to be **on**, which is **off by default**. Enable it once per queue manager with MQSC — for example `ALTER QMGR STATMQI(ON) STATQ(ON)`, then `ALTER QLOCAL(...) STATQ(ON)` for any queue you want per-queue stats on (or rely on the queue-manager `STATQ` default). Without this, the exporter is healthy and the entity appears, but those panels render blank. See the [mq-metric-samples documentation](https://github.com/ibm-messaging/mq-metric-samples) for the exact statistics each metric depends on.
</Callout>

The tables below group the metrics you'll reach for most often by area. They are a representative selection, not the full catalog. To list every metric actually reporting in your account, run:

```sql
FROM Metric SELECT uniques(metricName, 2000) WHERE metricName LIKE 'ibmmq_%' SINCE 1 day ago
```

<Callout variant="tip">
  Status metrics report numeric **enum** values, not `1`/`0` — see the [status-code reference](#status-codes) below before alerting on them. Counters are converted to delta temporality by the collector; use `sum()` / `rate()` over a window rather than `latest()` when charting them.
</Callout>

### MQ status-code reference [#status-codes]

IBM MQ status metrics report numeric enum values, **not** `1`/`0` — and the healthy ("running") value differs by metric (`2` for queue-manager-level status, `3` for channel status). Alert on *inequality* to the running value (for example, `ibmmq_qmgr_status != 2`):

| Metric | Healthy value | Meaning |
|---|---|---|
| `ibmmq_qmgr_status` | `2` | Queue manager is running |
| `ibmmq_qmgr_command_server_status` | `2` | Command server is running |
| `ibmmq_qmgr_channel_initiator_status` | `2` | Channel initiator is running |
| `ibmmq_channel_status` | `3` | Channel is running |

Any other value (or a missing data point) means the component is not in its running state.

## Troubleshooting [#troubleshooting]

<CollapserGroup>
  <Collapser
    id="metrics-but-no-entity"
    title="Metrics show up in NRDB, but no IBMMQ_MANAGER entity appears"
  >
    This is almost always caused by altering the entity-critical metric shape. Entity synthesis keys on the raw underscore metrics (`ibmmq_*`), the raw `qmgr` label, and a `target.name` attribute. Check that:

    * The metric names are **not** dotted (you should see `ibmmq_queue_depth`, not `ibmmq.queue.depth`).
    * The `qmgr` label is **present and not renamed** (not `ibmmq.queue_manager.name`).
    * `target.name` is set on the metrics (and the camelCase `targetName` is **not** — if `targetName` is still present, `transform/ibmmq-cleanup` isn't running or is misordered):

      ```sql
      FROM Metric SELECT uniques(target.name), uniques(qmgr) WHERE metricName LIKE 'ibmmq_%' SINCE 30 minutes ago
      ```

    The processor order in the collector configuration is load-bearing — `resourcedetection → transform/ibmmq-cleanup`. If you reordered or removed `transform/ibmmq-cleanup`, IBM MQ metrics route onto the collector's own entity instead of an `IBMMQ_MANAGER` entity. Restore the documented pipeline order.
  </Collapser>

  <Collapser
    id="counters-flat"
    title="Counter metrics appear as flat lines"
  >
    New Relic expects delta temporality for counters; Prometheus emits cumulative. The `cumulativetodelta/ibmmq` processor converts them in place. If counters look flat, confirm that processor is present in each metrics pipeline.
  </Collapser>

  <Collapser
    id="new-stale-entities"
    title="Old entities went stale after a redeploy"
  >
    The `target.name` value (set via the `TARGET_NAME` env on Linux, or the `targetName` relabel on Kubernetes) is the first segment of every `IBMMQ_MANAGER` and `IBMMQ_QUEUE` GUID. Changing it after deployment mints brand-new entities and the old ones go stale, breaking dashboards and alerts that point at the old GUIDs. Pick a stable value once and never change it.

    If you've already changed it, restore the original `TARGET_NAME` value and restart the collector. The correctly-named entities resume reporting; the duplicates created under the wrong name stop receiving data and age out of the UI after New Relic's entity reporting window. Rebuild any dashboards or alerts only if they were repointed at the wrong GUIDs in the meantime.
  </Collapser>

  <Collapser
    id="no-metrics-at-all"
    title="No ibmmq_* metrics arrive in New Relic"
  >
    Work the pipeline back to front. First confirm the exporter is publishing — `curl -s http://<exporter-host>:9157/metrics | grep '^ibmmq_qmgr_status'` should return a line per queue manager (if not, the problem is the [mq-metric-samples](https://github.com/ibm-messaging/mq-metric-samples) exporter, not the collector). Then confirm the collector is scraping it (see *Is the collector healthy and scraping?* below). Finally, confirm the OTLP endpoint matches your account [region](#region-endpoints) and the license key is valid — a wrong region silently drops data with no collector-side error.
  </Collapser>

  <Collapser
    id="enable-debug-logging"
    title="Enable collector debug logging"
  >
    Turn up the collector's internal log level to see scrape and export activity.

    On Linux, add to the collector config and restart:

    ```yaml
    service:
      telemetry:
        logs:
          level: "debug"
    ```

    On Kubernetes (Helm), set the same block under `config:` in your `values.yaml` and run `helm upgrade`. You can also temporarily add the `debug` exporter to a pipeline to print datapoints to the collector log before they're sent to New Relic.
  </Collapser>

  <Collapser
    id="check-health"
    title="Is the collector healthy and scraping?"
  >
    The collector configuration enables a health endpoint at `:13133`:

    ```bash
    curl http://127.0.0.1:13133
    ```

    A healthy collector returns `{"status":"Server available", ...}`. The collector's own self-metrics (scrape counts, export successes/failures) are exposed on `:8888` — check `otelcol_exporter_send_failed_metric_points` for export problems and `scrape_*` activity for receiver problems.
  </Collapser>
</CollapserGroup>

## Next steps [#next-steps]

* **Build dashboards and alerts** on the `ibmmq_*` metrics. Useful alert conditions include `ibmmq_qmgr_status != 2` (queue manager not running), `ibmmq_channel_status != 3` (channel not running), and `ibmmq_queue_depth` for `SYSTEM.DEAD.LETTER.QUEUE` or relative to `ibmmq_queue_attribute_max_depth` (a queue filling toward its limit).
* **Correlate IBM MQ with your applications** by instrumenting the apps that produce to and consume from these queues with OpenTelemetry and sending them to the same New Relic account — queue health then sits alongside application performance.
* **Add more queue managers.** On Linux, add a receiver and pipeline plus an `IBMMQ_QM<N>_ENDPOINT` env var; on Kubernetes, just deploy another annotated exporter pod and the collector discovers it with zero config change.
