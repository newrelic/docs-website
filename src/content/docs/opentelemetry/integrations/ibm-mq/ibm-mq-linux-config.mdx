This guide provides the **complete, standalone collector configuration** for shipping IBM MQ queue manager metrics to New Relic from a Linux host: the collector config file, the environment file that feeds it, and the systemd drop-in that loads them. It assumes your IBM MQ broker and the [mq-metric-samples](https://github.com/ibm-messaging/mq-metric-samples) exporter (one process per queue manager) are already running and exposing `/metrics`.

For the architecture and data flow, the IBM MQ entity model, prerequisites, the metrics reference, and troubleshooting, see the [Monitor IBM MQ with OpenTelemetry](/docs/opentelemetry/integrations/ibm-mq/ibm-mq-otel) overview. This page focuses on the configuration files themselves.

<Callout variant="important">
  This guide uses **static scrape targets** — one Prometheus receiver per queue
  manager. This differs from the Kubernetes path, which
  uses pod auto-discovery. The processor chain and OTLP exporter are identical between
  the two, so the metric shape and entity synthesis are the same.
</Callout>

The collector configuration in this guide is **standalone and complete** — you create
a single config file from the YAML provided here. The collector loads only that one
file. There is no base config to merge with.

<Steps>

<Step>

### Install the collector [#install]

Download and install the pre-built package for your Linux distribution. Replace
`<ver>` with the latest release tag from the
[nrdot-collector-releases](https://github.com/newrelic/nrdot-collector-releases/releases)
page (for example, `v0.12.0`).

**Debian / Ubuntu:**

```bash
curl -LO https://github.com/newrelic/nrdot-collector-releases/releases/download/<ver>/nrdot-collector_<ver>_linux_amd64.deb
sudo dpkg -i nrdot-collector_<ver>_linux_amd64.deb
```

**RHEL / Rocky / Amazon Linux:**

```bash
curl -LO https://github.com/newrelic/nrdot-collector-releases/releases/download/<ver>/nrdot-collector_<ver>_linux_amd64.rpm
sudo dnf install ./nrdot-collector_<ver>_linux_amd64.rpm
```

The package installs:

- `/usr/bin/nrdot-collector` — the collector binary
- `/etc/nrdot-collector/config.yaml` — the package's default collector config
- A `nrdot-collector.service` systemd unit (disabled by default)

This guide uses a **single standalone config** for IBM MQ, created in the next step at
`/etc/nrdot-collector/ibmmq-config.yaml`. The systemd drop-in in [Configure the systemd unit](#systemd)
replaces the service's start command so the collector loads **only** the IBM MQ config — the
package's default `config.yaml` is not loaded, so the two never conflict.

</Step>

<Step>

### Create the IBM MQ config file [#create-config]

Create the directory where your config and secrets will live, then create the
config file itself. You will populate the YAML in the next sub-section.

```bash
# Config directory — group-readable by nrdot only (it will hold secrets)
sudo install -d -o root -g nrdot -m 0750 /etc/nrdot-collector

# Create the config file with correct permissions
sudo install -o root -g nrdot -m 0640 /dev/null /etc/nrdot-collector/ibmmq-config.yaml
sudo -u nrdot ${EDITOR:-vi} /etc/nrdot-collector/ibmmq-config.yaml
```

Paste the following complete collector configuration into the file. It defines
three `prometheus/ibmmq-qm*` receivers (one per queue manager), the required
processor chain, an `otlphttp/ibmmq` exporter, and per-QM pipelines. All
deployment-specific values are read from environment variables — you set those in
the next step.

```yaml
extensions:
  health_check:
    endpoint: 127.0.0.1:13133

processors:

  filter/ibmmq-overhead:
    metrics:
      exclude:
        match_type: regexp
        metric_names:
          - "^go_.*"
          - "^process_.*"
          - "^promhttp_.*"
          - "^scrape_.*"

  filter/ibmmq-queues:
    metrics:
      datapoint:
        - 'attributes["queue"] != nil and IsMatch(attributes["queue"], "^SYSTEM\\.(ADMIN\\.|MQSC\\.|DEFAULT\\.|AUTH\\.|CHANNEL\\.|CHLAUTH\\.|CICS\\.|SYNCPOINT\\.|INTERNAL\\.|PENDING\\.|PROTECTION\\.|BROKER\\.|AMQP\\.|DOTNET\\.|REST\\.|RETAINED\\.|SELECTION\\.|DURABLE\\.|HIERARCHY\\.|DDELAY\\.)")'
        - 'attributes["queue"] != nil and IsMatch(attributes["queue"], "^SYSTEM\\.CLUSTER\\.(COMMAND|HISTORY)\\.QUEUE$")'
        - 'attributes["queue"] != nil and IsMatch(attributes["queue"], "^AMQ\\.")'

  resourcedetection:
    detectors: [env, ec2, system]
    system:
      resource_attributes:
        host.name:
          enabled: true
        host.id:
          enabled: true

  transform/ibmmq-cleanup:
    metric_statements:
      - context: resource
        statements:
          - delete_key(attributes, "service.name")
          - delete_key(attributes, "service.instance.id")
          - delete_key(attributes, "server.address")
          - delete_key(attributes, "server.port")
          - delete_key(attributes, "url.scheme")
          - delete_key(attributes, "host.name")
      - context: datapoint
        statements:
          # Rename the injected identity labels to their OTel dotted form — the
          # names New Relic's IBM MQ entity synthesis keys on. qmgr / queue stay raw.
          - set(attributes["target.name"], attributes["targetName"]) where attributes["targetName"] != nil
          - delete_key(attributes, "targetName")
          - set(attributes["cluster.name"], attributes["clusterName"]) where attributes["clusterName"] != nil
          - delete_key(attributes, "clusterName")
          - delete_key(attributes, "instance")
          - delete_key(attributes, "job")

  memory_limiter/ibmmq:
    check_interval: 1s
    limit_mib: 512
    spike_limit_mib: 256

  cumulativetodelta/ibmmq: {}

  batch/ibmmq:
    send_batch_size: 1000
    timeout: 200ms

exporters:

  otlphttp/ibmmq:
    endpoint: ${env:NEW_RELIC_OTLP_ENDPOINT:-https://otlp.nr-data.net:4318}
    headers:
      api-key: ${env:NEW_RELIC_LICENSE_KEY}

receivers:

  prometheus/ibmmq-qm1:
    config:
      scrape_configs:
        - job_name: 'ibmmq-qm1'
          scrape_interval: ${env:IBMMQ_SCRAPE_INTERVAL:-60s}
          scrape_timeout: 15s
          static_configs:
            - targets:
                - "${env:IBMMQ_QM1_ENDPOINT:-localhost:9157}"
              labels:
                targetName: "${env:TARGET_NAME}"
                clusterName: "${env:IBMMQ_CLUSTER_NAME:-}"

  prometheus/ibmmq-qm2:
    config:
      scrape_configs:
        - job_name: 'ibmmq-qm2'
          scrape_interval: ${env:IBMMQ_SCRAPE_INTERVAL:-60s}
          scrape_timeout: 15s
          static_configs:
            - targets:
                - "${env:IBMMQ_QM2_ENDPOINT:-localhost:9158}"
              labels:
                targetName: "${env:TARGET_NAME}"
                clusterName: "${env:IBMMQ_CLUSTER_NAME:-}"

  prometheus/ibmmq-qm3:
    config:
      scrape_configs:
        - job_name: 'ibmmq-qm3'
          scrape_interval: ${env:IBMMQ_SCRAPE_INTERVAL:-60s}
          scrape_timeout: 15s
          static_configs:
            - targets:
                - "${env:IBMMQ_QM3_ENDPOINT:-localhost:9159}"
              labels:
                targetName: "${env:TARGET_NAME}"
                clusterName: "${env:IBMMQ_CLUSTER_NAME:-}"

service:
  pipelines:
    metrics/ibmmq-qm1:
      receivers: [prometheus/ibmmq-qm1]
      processors:
        - filter/ibmmq-overhead
        - filter/ibmmq-queues
        - resourcedetection
        - transform/ibmmq-cleanup
        - memory_limiter/ibmmq
        - cumulativetodelta/ibmmq
        - batch/ibmmq
      exporters: [otlphttp/ibmmq]

    metrics/ibmmq-qm2:
      receivers: [prometheus/ibmmq-qm2]
      processors:
        - filter/ibmmq-overhead
        - filter/ibmmq-queues
        - resourcedetection
        - transform/ibmmq-cleanup
        - memory_limiter/ibmmq
        - cumulativetodelta/ibmmq
        - batch/ibmmq
      exporters: [otlphttp/ibmmq]

    metrics/ibmmq-qm3:
      receivers: [prometheus/ibmmq-qm3]
      processors:
        - filter/ibmmq-overhead
        - filter/ibmmq-queues
        - resourcedetection
        - transform/ibmmq-cleanup
        - memory_limiter/ibmmq
        - cumulativetodelta/ibmmq
        - batch/ibmmq
      exporters: [otlphttp/ibmmq]

  extensions: [health_check]
```

<Callout variant="important">
  **Do not reorder or remove processors.** The chain order is load-bearing for
  entity synthesis. `transform/ibmmq-cleanup` must run after `resourcedetection`.
  Reversing this causes IBM MQ metrics to land on the collector's own entity instead
  of `IBMMQ_MANAGER` entities, and dashboards go blank.
</Callout>

#### Component reference [#component-reference]

The pipeline is the same on Linux and Kubernetes; only the receiver layer differs (static targets here, pod auto-discovery on Kubernetes). Each component has a specific, order-sensitive job:

| Component | Role |
|---|---|
| `health_check` extension | Liveness endpoint on `127.0.0.1:13133`; returns `{"status":"Server available"}` once the collector is ready. |
| `prometheus/ibmmq-qm1..3` receivers | One static Prometheus scraper per queue manager (`scrape_interval` default `60s`, `scrape_timeout` `15s`). Each reads one exporter's `/metrics` and stamps the `targetName` and `clusterName` identity labels (camelCase — Prometheus label names can't contain dots) from `TARGET_NAME` / `IBMMQ_CLUSTER_NAME`. |
| `filter/ibmmq-overhead` | Drops the exporter's own self-metrics (`go_*`, `process_*`, `promhttp_*`, `scrape_*`), which carry no IBM MQ signal, to save ingest. |
| `filter/ibmmq-queues` | Cardinality control: drops internal `SYSTEM.*` and `AMQ.*` queues so only application queues become `IBMMQ_QUEUE` entities. Extend the regexps to exclude your own internal queues. |
| `resourcedetection` | Stamps host and cloud identity (`env`, `ec2`, `system` detectors). `host.id` and the `cloud.*` / EC2 attributes survive downstream to populate the host and cloud widgets and link to the underlying host entity. |
| `transform/ibmmq-cleanup` | **Entity-critical.** Strips the Prometheus receiver's `service.*` / `server.*` / `url.scheme` / `host.name` attributes (which would otherwise route metrics to the collector's own entity), and renames the identity labels to their dotted OTel form (`targetName` → `target.name`, `clusterName` → `cluster.name`) — the names synthesis keys on. `qmgr` / `queue` stay raw. Must run **after** `resourcedetection`. |
| `memory_limiter/ibmmq` | Back-pressure safety valve (`limit_mib: 512`, `spike_limit_mib: 256`) so the collector sheds load instead of being OOM-killed. The systemd `MemoryMax=1G` is the hard ceiling above it. |
| `cumulativetodelta/ibmmq` | Converts Prometheus cumulative counters to the delta temporality New Relic expects; without it, counters render as flat lines. Gauges (`ibmmq_qmgr_status`, `ibmmq_queue_depth`) are unaffected. |
| `batch/ibmmq` | Bundles datapoints (`send_batch_size: 1000`, `timeout: 200ms`) to reduce the number of OTLP HTTP requests. |
| `otlphttp/ibmmq` exporter | POSTs metrics to your New Relic OTLP endpoint (`NEW_RELIC_OTLP_ENDPOINT`), authenticated with the `NEW_RELIC_LICENSE_KEY` ingest key. Retries transient errors with backoff; does not retry `401`/`403` auth failures. |

Each pipeline (`metrics/ibmmq-qm1..3`) wires one receiver through this shared, order-sensitive processor chain to the exporter, so a failure scraping one queue manager does not stall the others. For the full rationale behind each stage, see [Architecture](/docs/opentelemetry/integrations/ibm-mq/ibm-mq-otel#architecture-overview) on the overview.

</Step>

<Step>

### Set environment variables [#set-env-vars]

The config file reads all deployment-specific values from environment variables.
Create a secrets file at `/etc/nrdot-collector/ibmmq-env`:

```bash
# Owner nrdot, mode 0600 — this file holds the license key
sudo install -o nrdot -g nrdot -m 0600 /dev/null /etc/nrdot-collector/ibmmq-env
sudo -u nrdot ${EDITOR:-vi} /etc/nrdot-collector/ibmmq-env
```

Populate the file with `KEY=VALUE` lines. Do not use `export` — systemd's
`EnvironmentFile=` directive does not support shell syntax:

```ini
# /etc/nrdot-collector/ibmmq-env

# --- Required ---

# New Relic ingest license key (40 chars, suffix NRAL)
NEW_RELIC_LICENSE_KEY=<your-license-key>

# Stable host identity — first segment of every IBMMQ entity GUID
TARGET_NAME=<your-hostname>

# mq-metric-samples exporter endpoints, one per queue manager
IBMMQ_QM1_ENDPOINT=localhost:9157
IBMMQ_QM2_ENDPOINT=localhost:9158
IBMMQ_QM3_ENDPOINT=localhost:9159

# --- Optional (shown with defaults) ---

# OTLP endpoint: US prod default shown; EU prod = https://otlp.eu01.nr-data.net:4318
NEW_RELIC_OTLP_ENDPOINT=https://otlp.nr-data.net:4318

# Optional tag grouping for the IBMMQ_MANAGER entity (does not affect the entity GUID).
# Leave unset to omit cluster.name (nullable); set it to group QMs under one cluster tag.
# IBMMQ_CLUSTER_NAME=prod-mq-cluster

# Scrape interval — must be >= the mq-metric-samples pollInterval (default 60s)
IBMMQ_SCRAPE_INTERVAL=60s
```

To find your license key, go to **New Relic UI → top-right profile menu → API keys → License key** (<InlinePopover type="licenseKey"/>).

<Callout variant="caution">
  **`TARGET_NAME` is entity-critical and must never change after first deployment.** It becomes the `target.name` attribute — the first segment of every `IBMMQ_MANAGER` and `IBMMQ_QUEUE` entity GUID (`target.name:qmgr`). Changing it after metrics are flowing mints brand-new entities and orphans the old ones, breaking dashboards and alerts. Choose a stable string once — the short hostname (`$(hostname -s)`, evaluated and hard-coded at provisioning time, since systemd does not expand shell substitutions in `EnvironmentFile=` entries) or the EC2 instance ID — and never change it. See [The IBM MQ entity model](/docs/opentelemetry/integrations/ibm-mq/ibm-mq-otel#entity-model).
</Callout>

**Variable reference:**

| Variable | Required | Default | Description |
|---|---|---|---|
| `NEW_RELIC_LICENSE_KEY` | Yes | — | New Relic ingest license key (40 chars, suffix `NRAL`). Passed as the `api-key` header; account-scoped. An invalid/User key returns `401`, a non-OTLP-enabled account returns `403`. |
| `TARGET_NAME` | Yes | — | Stable host identity; first GUID segment. Never change after first deploy. |
| `IBMMQ_QM1_ENDPOINT` | Yes | `localhost:9157` | `host:port` for QM1's exporter |
| `IBMMQ_QM2_ENDPOINT` | Only with a 2nd QM | `localhost:9158` | `host:port` for QM2's exporter. Required only if you keep the `prometheus/ibmmq-qm2` receiver; for a single queue manager use the [single-QM variant](#single-qm). |
| `IBMMQ_QM3_ENDPOINT` | Only with a 3rd QM | `localhost:9159` | `host:port` for QM3's exporter. Required only if you keep the `prometheus/ibmmq-qm3` receiver. |
| `NEW_RELIC_OTLP_ENDPOINT` | No | `https://otlp.nr-data.net:4318` | US prod default; EU prod = `https://otlp.eu01.nr-data.net:4318`. The wrong region silently drops data. |
| `IBMMQ_CLUSTER_NAME` | No | _(unset → nullable)_ | Display-only tag for entity grouping; safe to change later. |
| `IBMMQ_SCRAPE_INTERVAL` | No | `60s` | Prometheus scrape cadence; must be `>=` the exporter's `pollInterval`. |

</Step>

<Step>

### Configure the systemd unit [#systemd]

Create a systemd drop-in that loads your config and env file. Using a drop-in
(rather than replacing the package's unit) survives package upgrades and keeps the
IBM MQ config isolated:

```bash
sudo mkdir -p /etc/systemd/system/nrdot-collector.service.d
sudo tee /etc/systemd/system/nrdot-collector.service.d/ibmmq.conf > /dev/null << 'EOF'
[Service]
EnvironmentFile=/etc/nrdot-collector/ibmmq-env
ExecStart=
ExecStart=/usr/bin/nrdot-collector --config /etc/nrdot-collector/ibmmq-config.yaml
MemoryMax=1G
LimitNOFILE=65536
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
EOF
```

The empty `ExecStart=` line clears the package's default command before the
second `ExecStart=` sets the new one (a drop-in *appends* to `ExecStart=` rather than
replacing it, so without the clear the collector would start twice). The remaining
directives load the env file, cap memory above the collector's own `memory_limiter`,
raise the file-descriptor limit, and harden the service.

After creating the drop-in, verify systemd can parse it without errors:

```bash
sudo systemd-analyze verify /etc/systemd/system/nrdot-collector.service
# No output means no syntax errors
```

</Step>

<Step>

### Start the collector and verify [#start-verify]

Reload the systemd daemon to pick up the drop-in, then start the collector:

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now nrdot-collector.service
sudo systemctl status nrdot-collector.service --no-pager
```

`systemctl enable --now` enables the unit (so it starts automatically on boot) and
starts it immediately. The `status` output should show `Active: active (running)`. If
it shows `Active: failed`, run `journalctl -u nrdot-collector.service -n 200 --no-pager` —
startup failures almost always produce a descriptive error in the first few lines (a
common one is a bad license key, which logs `401`/`403` from the OTLP exporter on first flush).

Confirm the collector's health endpoint responds:

```bash
curl -sS http://127.0.0.1:13133
# Expected: {"status":"Server available", ...}
```

Then confirm IBM MQ metrics are arriving in New Relic: wait one scrape interval (default
`60s`) after the collector reports ready, then follow [Find your data](/docs/opentelemetry/integrations/ibm-mq/ibm-mq-otel#find-data)
on the overview, which has the verification NRQL and the [MQ status-code reference](/docs/opentelemetry/integrations/ibm-mq/ibm-mq-otel#status-codes).

</Step>

<Collapser id="add-fourth-qm" title="Add a fourth (or Nth) queue manager">

The static Linux pipeline requires a config edit for every new queue manager. To
add `QM4` exposing metrics on port `9160`:

1. Confirm the mq-metric-samples exporter for `QM4` is running and reachable:

   ```bash
   curl -sS http://localhost:9160/metrics | grep -E '^ibmmq_qmgr_status' | head -1
   ```

2. Add `IBMMQ_QM4_ENDPOINT` to the env file at `/etc/nrdot-collector/ibmmq-env`:

   ```ini
   IBMMQ_QM4_ENDPOINT=localhost:9160
   ```

3. Open `/etc/nrdot-collector/ibmmq-config.yaml` and add a fourth receiver block inside
   `receivers:`:

   ```yaml
   prometheus/ibmmq-qm4:
     config:
       scrape_configs:
         - job_name: 'ibmmq-qm4'
           scrape_interval: ${env:IBMMQ_SCRAPE_INTERVAL:-60s}
           scrape_timeout: 15s
           static_configs:
             - targets:
                 - "${env:IBMMQ_QM4_ENDPOINT:-localhost:9160}"
               labels:
                 targetName: "${env:TARGET_NAME}"
                 clusterName: "${env:IBMMQ_CLUSTER_NAME:-}"
   ```

4. Add the matching pipeline inside `service.pipelines:`:

   ```yaml
   metrics/ibmmq-qm4:
     receivers: [prometheus/ibmmq-qm4]
     processors:
       - filter/ibmmq-overhead
       - filter/ibmmq-queues
       - resourcedetection
       - transform/ibmmq-cleanup
       - memory_limiter/ibmmq
       - cumulativetodelta/ibmmq
       - batch/ibmmq
     exporters: [otlphttp/ibmmq]
   ```

5. Restart the collector:

   ```bash
   sudo systemctl restart nrdot-collector.service
   ```

To **remove** a queue manager, disable its exporter service, delete its receiver
and pipeline blocks from `ibmmq-config.yaml`, remove its `IBMMQ_QMN_ENDPOINT`
from the env file, and restart the collector. The entity ages out per New Relic's
reporting window.

<Collapser id="tls-basic-auth" title="TLS and basic auth for the exporter scrape">

If your mq-metric-samples exporter is configured to require HTTPS or basic
authentication, add the relevant fields to each receiver's `scrape_configs`
block. The following shows both options for `prometheus/ibmmq-qm1` — apply the
same pattern to `qm2` and `qm3`:

```yaml
prometheus/ibmmq-qm1:
  config:
    scrape_configs:
      - job_name: 'ibmmq-qm1'
        scrape_interval: ${env:IBMMQ_SCRAPE_INTERVAL:-60s}
        scrape_timeout: 15s
        scheme: https
        tls_config:
          ca_file: /etc/nrdot-collector/ca.pem
          insecure_skip_verify: false
        basic_auth:
          username: ${env:IBMMQ_EXPORTER_USER}
          password: ${env:IBMMQ_EXPORTER_PASSWORD}
        static_configs:
          - targets:
              - "${env:IBMMQ_QM1_ENDPOINT:-localhost:9157}"
            labels:
              targetName: "${env:TARGET_NAME}"
              clusterName: "${env:IBMMQ_CLUSTER_NAME:-}"
```

Add the corresponding variables to `/etc/nrdot-collector/ibmmq-env`:

```ini
IBMMQ_EXPORTER_USER=mqmonitor
IBMMQ_EXPORTER_PASSWORD=<your-exporter-password>
```

</Collapser>

<Collapser id="remote-exporter" title="Remote exporter (collector and queue manager on separate hosts)">

When the mq-metric-samples exporter runs on a different host from the collector,
change the `IBMMQ_QMN_ENDPOINT` values in the env file to point at the remote
host:

```ini
IBMMQ_QM1_ENDPOINT=mq-host-1.example.com:9157
IBMMQ_QM2_ENDPOINT=mq-host-1.example.com:9158
IBMMQ_QM3_ENDPOINT=mq-host-2.example.com:9159
```

Firewall the exporter ports so only the collector's IP can reach them. On
Ubuntu / UFW:

```bash
sudo ufw allow from <collector-ip> to any port 9157 proto tcp
sudo ufw allow from <collector-ip> to any port 9158 proto tcp
sudo ufw allow from <collector-ip> to any port 9159 proto tcp
```

On RHEL / firewalld, create a dedicated zone scoped to the collector source IP
and the three ports.

If the collector is behind an egress proxy, add `HTTPS_PROXY` to the env file
and include `NO_PROXY=localhost,127.0.0.1` to prevent the proxy from
intercepting the local scrape requests:

```ini
HTTPS_PROXY=http://proxy.example.com:3128
NO_PROXY=localhost,127.0.0.1
```

</Collapser>

<Collapser id="eu-region" title="EU region">

Change `NEW_RELIC_OTLP_ENDPOINT` in the env file to the EU ingest endpoint:

```ini
NEW_RELIC_OTLP_ENDPOINT=https://otlp.eu01.nr-data.net:4318
```

No other changes are required.
