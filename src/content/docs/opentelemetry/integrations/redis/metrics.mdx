This reference guide provides a comprehensive overview of all metrics collected by the Redis OpenTelemetry integration, helping you understand what data is available for monitoring your Redis instance health and performance.

## Key metrics at a glance

Monitor your Redis instance health and performance with these essential metrics:

<table>
  <thead>
    <tr>
      <th style={{ width: "200px" }}>Metric category</th>
      <th>What it measures</th>
      <th style={{ width: "120px" }}>Priority</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>**Memory utilization**</td>
      <td>`redis.memory.used`, `redis.maxmemory` — Current memory consumption vs configured limit</td>
      <td>Critical</td>
    </tr>
    <tr>
      <td>**Cache hit ratio**</td>
      <td>`redis.keyspace.hits`, `redis.keyspace.misses` — Ratio of successful key lookups</td>
      <td>Critical</td>
    </tr>
    <tr>
      <td>**Key evictions**</td>
      <td>`redis.keys.evicted` — Keys removed due to memory pressure</td>
      <td>Critical</td>
    </tr>
    <tr>
      <td>**Commands/sec**</td>
      <td>`redis.commands`, `redis.commands.processed` — Throughput of commands being executed</td>
      <td>Important</td>
    </tr>
    <tr>
      <td>**Connected clients**</td>
      <td>`redis.clients.connected`, `redis.clients.blocked` — Active and waiting connections</td>
      <td>Important</td>
    </tr>
    <tr>
      <td>**Fragmentation ratio**</td>
      <td>`redis.memory.fragmentation_ratio` — Ratio of OS-allocated to Redis-allocated memory</td>
      <td>Monitoring</td>
    </tr>
    <tr>
      <td>**Cluster health**</td>
      <td>`redis.cluster.state`, `redis.cluster.slots_ok`, `redis.cluster.known_nodes` — Overall cluster status and slot coverage (Cluster mode only)</td>
      <td>Critical</td>
    </tr>
  </tbody>
</table>

## Cluster metrics

These metrics are only available when monitoring Redis in Cluster mode. They require enabling cluster metrics in the Redis receiver (all cluster metrics are disabled by default) or using the Prometheus receiver approach with `redis_exporter --is-cluster`.

<table>
  <thead>
    <tr>
      <th style={{ width: "260px" }}>Metric</th>
      <th>Description</th>
      <th style={{ width: "100px" }}>Type</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>`redis.cluster.state`</td>
      <td>Overall cluster health status. 1 = OK (all slots served), 0 = FAIL (one or more slots unavailable).</td>
      <td>Gauge</td>
    </tr>
    <tr>
      <td>`redis.cluster.slots_assigned`</td>
      <td>Number of hash slots mapped to nodes. Must be 16384 for a fully operational cluster.</td>
      <td>Gauge</td>
    </tr>
    <tr>
      <td>`redis.cluster.slots_ok`</td>
      <td>Number of slots actively serving requests without issues.</td>
      <td>Gauge</td>
    </tr>
    <tr>
      <td>`redis.cluster.slots_pfail`</td>
      <td>Number of slots in PFAIL (probable failure) state — a node is suspected unreachable but not yet confirmed.</td>
      <td>Gauge</td>
    </tr>
    <tr>
      <td>`redis.cluster.slots_fail`</td>
      <td>Number of slots in confirmed FAIL state — NOT serving traffic.</td>
      <td>Gauge</td>
    </tr>
    <tr>
      <td>`redis.cluster.known_nodes`</td>
      <td>Total number of nodes (masters + replicas) visible in the cluster.</td>
      <td>Gauge</td>
    </tr>
    <tr>
      <td>`redis.cluster.node.count`</td>
      <td>Number of master nodes currently serving hash slots.</td>
      <td>Gauge</td>
    </tr>
    <tr>
      <td>`redis.cluster.stats_messages_sent`</td>
      <td>Total cluster bus messages sent (ping, pong, meet, fail). Steady traffic indicates healthy gossip protocol.</td>
      <td>Cumulative</td>
    </tr>
    <tr>
      <td>`redis.cluster.stats_messages_received`</td>
      <td>Total cluster bus messages received from peer nodes. A sudden drop indicates possible network partition.</td>
      <td>Cumulative</td>
    </tr>
  </tbody>
</table>

<Callout variant="tip">
All nodes in a Redis Cluster report identical cluster metrics (`CLUSTER INFO` returns the same data from any node). You can enable cluster monitoring on all nodes for resilience or on a single node to minimize redundant data. For the Prometheus receiver approach, use `redis_exporter --is-cluster` which automatically collects these metrics.
</Callout>
