Redis is a high-performance, in-memory data structure store used as a database, cache, message broker, and streaming engine. With OpenTelemetry and New Relic, you gain deep visibility into Redis performance, memory utilization, cache effectiveness, and connection health across your entire infrastructure.

This integration leverages the OpenTelemetry Collector to collect metrics and logs from your Redis instances and send them to New Relic for analysis, alerting, and correlation with the rest of your stack. You can deploy it using either the [NRDOT collector](/docs/opentelemetry/nrdot/nrdot-collector/) or the [OTel Collector Contrib](https://github.com/open-telemetry/opentelemetry-collector-contrib).

<Callout variant="tip">
  This documentation covers monitoring Redis using OpenTelemetry. If you're looking for the traditional New Relic Redis on-host integration (nri-redis), see [Redis monitoring integration](/docs/infrastructure/host-integrations/host-integrations-list/redis/redis-integration/).

  The OpenTelemetry approach provides vendor-neutral instrumentation, unified collection pipelines, and native correlation with other OTel-instrumented services.
</Callout>

## Why choose OpenTelemetry for Redis monitoring? [#why-otel]

Monitoring Redis with OpenTelemetry offers several advantages over traditional proprietary agents:

1. **Unified collection pipeline**: Use a single OpenTelemetry Collector to gather metrics from Redis alongside all your other infrastructure components, applications, and services. No need for separate agents per technology.

2. **Vendor neutrality**: OpenTelemetry is an open standard backed by the CNCF. Your instrumentation investment remains portable across observability backends, avoiding vendor lock-in.

3. **High performance**: The Redis receiver is designed for minimal overhead, collecting metrics through Redis's INFO command without impacting your instance's throughput or latency.

4. **Future-proof**: As OpenTelemetry evolves, you automatically gain access to new capabilities, semantic conventions, and community-contributed enhancements without changing your monitoring stack.

5. **Native New Relic integration**: New Relic provides first-class support for OpenTelemetry data, with full platform capabilities including alerting, dashboards, and service maps.


## Identifier patterns [#identifier-patterns]

The integration supports two entity identification patterns for your Redis instances:

| Pattern | Identifier | Best for |
|---------|-----------|----------|
| **Composite** (default) | `server.address:server.port` | Standard deployments with unique host:port per instance |
| **Custom** (`redis.instance.id`) | User-defined string | When you want human-readable names, or when endpoint is `localhost` |

**Rule of thumb:** If your collector runs on the same host as Redis and connects via `localhost:6379`, use the custom pattern to give your entity a meaningful name. If you connect via a real hostname/IP (e.g., `redis-prod-01:6379`), the composite pattern works automatically.
