After shipping code, use the APM summary page as a quick health check. The goal is to catch regressions early — before customers report them.

To open the APM summary page, go to <DNT>**[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > APM & services**</DNT> and select your application.

<Callout variant="important">
  The **Last deployment** tile only populates if you've set up [deployment markers](/docs/apm/apm-ui-pages/events/record-deployments/). If the tile is empty, you're flying blind after every release. Setting up markers is one of the highest-value instrumentation steps you can take.
</Callout>

## Check the deployment tile [#deployment-tile]

<img
  title="deployment summary tile"
  alt="A screenshot showing the deployment summary tile"
  src="/images/apm_screenshot-crop_deployment-summary-tile.webp"
/>

The **Last deployment** tile automatically compares your service's behavior before and after your most recent deployment marker — no manual setup required. It shows the delta in error rate and response time at a glance.

The comparison window grows proportionally with time since deploy:

| Time since deployment | Comparison window |
|---|---|
| 45 minutes | 45 min before vs. 45 min after |
| 2 hours | 2 hours before vs. 2 hours after |
| 3+ hours | Settles into a standard 3-hour window |

## Look for correlated changes across charts [#correlated-changes]

A regression typically shows up as a correlated shift across two charts at the same point in time. Deployment markers appear as gray pins on each chart.

**What to look for:**

* **Error rate rising** in the **Errors** chart at the deployment pin
* **Response time increasing** in **Web transactions** — especially in the database (yellow) or external (green) segment if the deployment touched those code paths

If both metrics worsen at the same pin, you have a strong signal the deployment introduced a regression.

| Where the spike is | What it likely means |
|---|---|
| Database segment (yellow) | A slow or missing query introduced by the change |
| External segment (green) | The deployment changed how your service calls a downstream API |
| App code segment (blue) | A logic change introduced latency in your own code |
| Error rate only, no response time change | A new code path is throwing exceptions but returning fast (common with unhandled edge cases) |

## Watch your error budget [#error-budget]

If you have [service levels configured](/docs/service-level-management/intro-slm/), the **Service levels** tile shows whether the deployment is burning error budget faster than expected.

| Tile state | What to do |
|---|---|
| **On track** | Service is performing within SLO. Continue monitoring. |
| **At risk** | Budget burning faster than expected. Watch closely and be ready to roll back. |
| **Non-compliant** | SLO is breached. Start a rollback conversation now. |
