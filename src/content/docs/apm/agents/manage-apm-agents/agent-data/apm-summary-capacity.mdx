The APM summary page isn't just for incidents. It's where you go when you want to answer the question: "Can my service handle more load, and what breaks first if it can't?"

Use this page before a planned traffic event (product launch, seasonal peak, marketing campaign), after noticing sustained growth in throughput, or when your team is deciding whether to scale horizontally or optimize first.

To open the APM summary page, go to <DNT>**[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > APM & services**</DNT> and select your application.

## Check how response time behaves under increasing load [#response-time-under-load]

<img
  title="web transactions apm summary"
  alt="A screenshot depicting web transactions in the APM summary page"
  src="/images/apm_screenshot-crop_web-transactions-in-APM.webp"
/>

The relationship between throughput and response time is the single most important signal for capacity planning. Open the time picker to a longer window (7 days or 30 days) and look at both charts together.

| What you observe | What it means | Action |
|---|---|---|
| Throughput grows, response time stays flat | Healthy headroom — service scales linearly | Set a throughput alert just above your observed peak as an early warning |
| Throughput grows, response time climbs slowly | Approaching a bottleneck — likely database connections or thread pool saturation | Investigate the yellow (database) segment first; check connection pool limits |
| Throughput grows, response time climbs sharply | Near or at capacity — the service can't absorb more load without degradation | Scale horizontally now; investigate code-level bottlenecks in parallel |
| Throughput is flat, response time slowly climbs over days | Resource leak or data growth issue — not a traffic problem | Check memory trends in the Infrastructure section; look for growing query times in Databases |

<Callout variant="tip">
  Use the **Compare with** option in the time picker to overlay your current week against the same period last month. A widening gap between the two lines tells you whether your growth rate is accelerating.
</Callout>

## Assess host-level headroom [#host-headroom]

<img
  title="Infrastructure summary page"
  alt="A screenshot depicting the infrastructure section of the apm summary page."
  src="/images/apm_screenshot-crop_infrastructure-on-apm-summary-page.webp"
/>

The **Infrastructure** section shows CPU and memory utilization per host alongside application metrics. This is where you find out whether you have spare capacity or whether hosts are already near their limits during normal operation.

**What to do:**
1. Scroll to the **Infrastructure** section on the summary page.
2. Look at the CPU % and memory % columns across all hosts during your current peak hours.
3. Apply this rule of thumb:

| CPU at peak | Memory at peak | Capacity verdict |
|---|---|---|
| Below 50% | Below 60% | Comfortable headroom — service can likely absorb 2x load |
| 50–70% | 60–75% | Moderate — handle a planned 2x event with caution; have rollback ready |
| 70–85% | 75–85% | Tight — a traffic spike will likely cause degradation; scale before the event |
| Above 85% | Above 85% | At capacity now — scale immediately; don't wait for an event |

4. Click any host row to open the full infrastructure detail page for that host and see historical CPU and memory trends.

<Callout variant="tip">
  If all hosts show similar utilization, horizontal scaling (adding more instances) directly improves capacity. If one host is consistently higher than the rest, investigate that host for configuration drift or uneven load balancer distribution first.
</Callout>

## Identify your bottleneck layer [#bottleneck-layer]

Before deciding to add infrastructure, identify which layer is the actual constraint. Scaling the wrong layer wastes resources.

**What to do:**
1. Go to **Transactions** and sort by **Most time consuming**.
2. Click your highest-traffic, slowest transaction and look at the trace breakdown.
3. Match the bottleneck to the right scaling action:

| Bottleneck layer | What you see in APM | Scaling action |
|---|---|---|
| Application tier | High CPU on all hosts; blue segment dominant in Web transactions | Add more app instances (horizontal scale) |
| Database | Yellow segment dominant; slow queries in Databases page | Optimize queries first; add read replicas if query optimization is insufficient |
| External dependencies | Green segment dominant; high latency in External services | You can't scale this yourself — contact the dependency owner; add caching or circuit breakers |
| Memory pressure | Memory at peak >80% on all hosts; response time spikes under load | Increase instance memory or fix memory leaks in your application |

## Check Apdex stability at peak load [#apdex-at-peak]

<img
  title="apdex in APM"
  alt="A screenshot depicting a sample Apdex score in APM"
  src="/images/apm_screenshot-crop_apdex-in-apm.webp"
/>

Your Apdex score shows whether users actually feel the capacity constraints — not just whether resources are saturated. A service can have 70% CPU and still maintain a healthy Apdex if response times remain within the threshold.

**What to do:**
1. Set the time picker to cover your last peak period.
2. Compare Apdex at peak throughput versus Apdex during off-peak.
3. If Apdex drops significantly at peak, users are feeling the load — scaling is justified even if infrastructure metrics look moderate.
4. If Apdex stays stable at peak, your service handles the load well — optimize before scaling.

## Set up capacity alerts [#capacity-alerts]

Don't wait until a peak event to discover you're out of headroom. Alert on the leading indicators.

**Recommended alert conditions:**

| What to alert on | NRQL | Threshold guidance |
|---|---|---|
| Throughput approaching capacity | `SELECT rate(count(*), 1 minute) FROM Transaction WHERE appName = 'your-app'` | 80% of your observed maximum safe throughput |
| Response time degradation under load | `SELECT average(duration) FROM Transaction WHERE appName = 'your-app'` | 1.5x your normal peak response time |
| Host CPU saturation | Use the Infrastructure alert condition builder | 80% CPU sustained for 5+ minutes |
| Apdex degrading at peak | `SELECT apdex(duration, t:0.5) FROM Transaction WHERE appName = 'your-app'` | Below 0.7 |

See [Create your first NRQL alert condition](/docs/alerts-applied-intelligence/new-relic-alerts/get-started/your-first-nrql-condition/) for setup instructions.
