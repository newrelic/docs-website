The incident is resolved. Now you need to understand what actually happened — when it started, what caused it, which users were affected, and what you'd need to catch it earlier next time.

The APM summary page is where you reconstruct the timeline. Everything you need — error spikes, deployment markers, infrastructure changes, log patterns, and distributed tracing — is in one place, and you can scroll back in time to the exact window of the incident.

To open the APM summary page, go to <DNT>**[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > APM & services**</DNT> and select your application.

## Step 1: Set the time window to the incident period [#set-time-window]

Before doing anything else, use the time picker to focus on the incident window. Set the start time to 30–60 minutes before the incident began (to establish a clean pre-incident baseline) and the end time to 30 minutes after resolution.

<Callout variant="tip">
  If you know the exact start time from an alert notification, use **Custom** time range in the time picker and enter the precise timestamps. This gives you a clean view of exactly what changed and when.
</Callout>

## Step 2: Establish when the incident actually started [#incident-start]

The time an alert fired is not always when the incident started — the problem may have been degrading for minutes or hours before crossing an alert threshold.

**What to do:**
1. Look at the **Errors** chart and **Apdex** chart together. Find the exact point where either metric began to deviate from the pre-incident baseline — that's your true start time.
2. Note the gap between the true start time and when your alert fired. A large gap means your alert threshold is set too conservatively for this type of failure.
3. Check the **Last deployment** tile. If a deployment marker sits at or just before the true start time, the deployment is your primary suspect.

## Step 3: Identify the root cause layer [#root-cause]

<CollapserGroup>
  <Collapser
    id="post-code-change"
    title="Was it a code change?"
  >
    **Signals:** Deployment marker in the charts at or before the incident start time; error rate or response time changed abruptly (not gradually).

    **What to do:**
    1. Click the deployment marker pin on the **Errors** or **Web transactions** chart to see which version was deployed.
    2. Go to **Errors inbox** and filter to the incident time window. Look for new error types that didn't exist before the deployment.
    3. Go to **Transactions** and compare the response time of key endpoints before and after the deployment marker. A single transaction with a sharp increase points to the specific code path that regressed.
  </Collapser>

  <Collapser
    id="post-dependency"
    title="Was it a dependency?"
  >
    **Signals:** No deployment at the start time; error rate or latency increased gradually; the green (external) or yellow (database) segment in Web transactions grew.

    **What to do:**
    1. Check **Distributed tracing insights** for downstream services showing degradation at the same start time as your incident.
    2. Go to **External services** and look for a call with increased latency or error rate during the incident window.
    3. Go to **Databases** and look for queries that slowed down — a database host issue often presents as a gradual response time increase across many transactions.

    If a dependency caused the incident, note the exact service and time range for the postmortem. Even if you didn't own the root cause, document the blast radius on your service.
  </Collapser>

  <Collapser
    id="post-infrastructure"
    title="Was it infrastructure?"
  >
    **Signals:** Problem affected some hosts but not others; CPU or memory spiked on the Infrastructure section; response time increased without a corresponding throughput increase.

    **What to do:**
    1. Scroll to the **Infrastructure** section and look at the per-host table during the incident window. Identify which hosts showed degraded metrics and when.
    2. Click the affected host to open the infrastructure detail page. Look for CPU, memory, or disk events that align with the incident start time.
    3. Check whether the affected hosts were running a different agent version or configuration — partial deployments and config drift are common causes of single-host incidents.
  </Collapser>
</CollapserGroup>

## Step 4: Measure the blast radius [#blast-radius]

Understanding how many users were affected — and for how long — is essential for communicating impact in a postmortem.

**What to do:**
1. Go to the **Errors** chart and use the dropdown to switch to **Impacted users** view. Note the count during the incident window versus normal baseline.
2. Check the **Service levels** tile. If service levels are configured, the tile shows exactly how much error budget was consumed during the incident — a precise, stakeholder-readable measure of impact.
3. Note the duration: time from true start (when the metric deviated) to resolution (when it returned to baseline). This is different from the alert-to-resolve duration — use the metric deviation window for the postmortem.

| Metric | Where to find it | What to record |
|---|---|---|
| Incident start time | Errors or Apdex chart — first deviation from baseline | Timestamp |
| Incident end time | Same charts — return to baseline | Timestamp |
| Users affected | Errors chart → Impacted users view | Count at peak |
| Error budget consumed | Service levels tile | Percentage |
| Hosts affected | Infrastructure section | Count and names |
| Root cause layer | Web transactions segments + Distributed tracing | Code / dependency / infrastructure |

## Step 5: Find the log evidence [#log-evidence]

<img
  title="logs in apm"
  alt="A screenshot showing logs in apm"
  src="/images/apm_screenshot-crop_logs-in-apm.webp"
/>

Log patterns often contain the most specific evidence of root cause — the exact exception message, the specific query that failed, or the configuration value that was wrong.

**What to do:**
1. Select **Log patterns (top 10)** in the Logs section.
2. Look for patterns that appeared during the incident window but are absent in the pre-incident baseline. New patterns that emerged at the start of the incident are strong evidence.
3. Click **Logs** to open the full logs UI. Filter to the incident time window and search for the error message or stack trace identified in Errors inbox.
4. Copy the relevant log lines for the postmortem document.

## Step 6: Identify what would have caught it earlier [#early-detection]

A postmortem isn't complete without a prevention recommendation.

**What to do:**
1. Compare the true incident start time against the time the alert fired. If there's a gap, consider tightening the alert threshold or adding a faster-firing condition.
2. Look at the pre-incident baseline in the Apdex and error charts. Was there a subtle early signal — a small uptick in errors or a slight Apdex dip — that preceded the main incident? If so, an alert on that leading indicator would have given earlier warning.
3. Check whether a deployment marker sits at the start time. If yes, [deployment markers](/docs/apm/apm-ui-pages/events/record-deployments/) were already set up — consider adding a post-deploy alert condition that fires faster than your standard thresholds.

<Callout variant="tip">
  The most common gap: alerts fire on averages, but the incident started in a single transaction or a subset of users. Adding an alert on **p99 response time** or **error count by transaction name** catches localized failures that averages hide.
</Callout>
