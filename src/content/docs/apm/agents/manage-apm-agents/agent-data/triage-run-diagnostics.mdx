Once you've installed a New Relic <InlinePopover type="apm"/> agent, the APM summary page is your operational hub. It's where you go to understand whether your service is healthy, investigate degradation, and verify that a recent deployment didn't introduce problems.

To open it, go to <DNT>**[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > APM & services**</DNT> and select your application.

## What's on this page [#page-overview]

The APM summary page is divided into sections that answer different questions. Here's what each one tells you:

**Summary tiles (top row)**

| Tile | What it shows | When to focus on it |
|---|---|---|
| **Issues** | Open alert incidents for this service | First thing to check when an alert fires |
| **Last deployment** | Before/after comparison for your most recent release | After every deployment; when an incident starts shortly after a release |
| **Service levels** | SLO compliance status and error budget | Weekly health reviews; after a deployment; during an active incident |
| **Vulnerabilities** | Known security vulnerabilities in your dependencies | Security reviews; when unusual behavior appears with no performance cause |

**Key metric charts**

| Chart | What it shows | When to focus on it |
|---|---|---|
| **Apdex** | User satisfaction score (0–1) based on response time | First signal that users are experiencing problems |
| **Web transactions time** | Response time broken into app code (blue), database (yellow), and external calls (green) | Identifying which layer is slow |
| **Throughput** | Requests per minute | Detecting traffic spikes, drops, and growth trends |
| **Error rate** | Percentage of requests that resulted in an error | Catching code failures and dependency errors |

**Lower sections**

| Section | What it shows | When to focus on it |
|---|---|---|
| **Distributed tracing insights** | Upstream and downstream services with correlated changes | Ruling out whether a dependency caused your incident |
| **Infrastructure** | Per-host CPU, memory, response time, and error rate | Determining if a problem is isolated to one host or fleet-wide |
| **Logs** | Log volume and top patterns from your application | Finding specific error messages and exception details |
