You've just joined a team, been assigned a new service, or taken on-call for something you've never worked with. Before you can triage an incident or make a safe code change, you need a mental model of how the service behaves. The APM summary page is where to build that model — it shows you what the service does, what it depends on, and what normal looks like, without needing to read the entire codebase first.

To open the APM summary page, go to <DNT>**[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > APM & services**</DNT> and select your application.

## Understand what the service does [#what-it-does]

Start with the top of the summary page before looking at any charts.

**What to note:**

* **Service name and language:** Shown in the header. The agent language (Java, Python, Ruby, etc.) tells you what kind of runtime you're dealing with.
* **Throughput:** How many requests per minute does this service handle? A service handling 10 rpm is very different operationally from one handling 10,000 rpm.
* **Apdex score:** Is this service performing well right now? If you're inheriting a service with an Apdex below 0.7, there may be pre-existing problems to be aware of before you take on-call.
* **Error rate:** What's the baseline error rate? Some services intentionally have non-zero error rates (expected 404s, for example) — knowing the baseline prevents false alarms when you're on-call.

## Learn what normal looks like [#what-is-normal]

Set the time picker to **Last 7 days** and observe the charts across a full week. This gives you a representative sample of the service's natural rhythm.

**Questions to answer:**

| Question | Where to look |
|---|---|
| When is peak traffic? | Throughput chart — find the daily high points |
| Does traffic differ significantly by day of week? | Throughput chart — compare weekday vs. weekend patterns |
| What does a healthy Apdex look like for this service? | Apdex chart — note the range during normal operation |
| What's the normal error rate range? | Errors chart — note the baseline and whether it spikes predictably |
| How much of response time is database vs. external vs. app code? | Web transactions chart — note the proportion of each color segment |

Write these down. They become your personal reference when you're on-call and something looks wrong — you'll know immediately whether what you're seeing is normal or an anomaly.

## Map the dependencies [#map-dependencies]

A service doesn't exist in isolation. Understanding what it calls and what calls it is essential before you're responsible for it.

**What to do:**
1. Scroll to the **Distributed tracing insights** section. This shows both upstream services (callers) and downstream services (dependencies) that your service communicates with.
2. Note the name and health of each dependency. Are any of them showing elevated latency or errors right now? If yes, those are known issues to be aware of.
3. Go to **[APM > your app > External services](/docs/apm/apm-ui-pages/monitoring/external-services/external-services-intro/)** for a complete list of external HTTP calls with their average latency and error rates.
4. Go to **[APM > your app > Service map](/docs/understand-dependencies/understand-system-dependencies/service-maps/service-maps/)** for a visual representation of the entire dependency graph — upstream callers, downstream services, and databases.

**Key questions to answer:**
- Which downstream services are critical path? (If they go down, does this service break?)
- Which upstream services send the most traffic to this service?
- Are there any dependencies that are already showing degraded health?

Scroll to the **Infrastructure** section and look at the per-host table.

**What to note:**

| What you see | What it tells you |
|---|---|
| Number of hosts | Scale of the service — is this a single instance or a fleet? |
| CPU and memory levels | Is the service resource-constrained or running with headroom? |
| Hosts with different response times | Uneven load distribution, or a host that's drifted from the rest |
| Agent version consistency | If hosts are running different agent versions, a partial deployment may be in progress |

## Check recent deployment history [#deployment-history]

Understanding what's changed recently helps you understand the current state of the service and any in-progress work.

**What to do:**
1. Click the **Last deployment** tile. This shows the most recent deployment marker.
2. Go to **[APM > your app > Events > Deployments](/docs/apm/apm-ui-pages/events/record-deployments/)** for a full deployment history.
3. Look for deployments in the last 2–4 weeks and check whether any correlated with a change in error rate or response time.

If the service has no deployment markers, that's important context: the team isn't tracking deployments in New Relic, which means the deployment tile and post-deploy comparison features won't work until markers are set up.

## Set up your on-call baseline [#oncall-baseline]

Before your first on-call shift, document the normal operating values you found above. When something looks wrong at 2 AM, you need these numbers accessible without having to remember them.

**Minimum baseline to document:**

```
Service: [name]
Normal throughput range: [X–Y rpm] (peak: [Z rpm])
Normal Apdex range: [0.XX–0.XX]
Normal error rate: [X%]
Web transactions breakdown: [XX% DB / XX% external / XX% app code]
Key dependencies: [list upstream and downstream services]
Hosts: [count], typical CPU: [X%], typical memory: [X%]
Last deployment: [date]
```

<Callout variant="tip">
  Save this in your team's runbook or incident response wiki. The goal is that anyone on-call — not just you — can quickly orient themselves using the same reference.
</Callout>

## Know what to watch during your first on-call [#first-oncall]

Now that you have a baseline, you know what "wrong" looks like:

* Apdex drops below your documented normal range → users are feeling it
* Error rate rises above your documented baseline → something broke
* Throughput drops outside of expected off-peak hours → an upstream caller may be down
* A single host deviating from the rest in the Infrastructure section → isolated failure

For each of these, use the [triage workflow](/docs/apm/agents/manage-apm-agents/agent-data/apm-summary-triage) to investigate.
