The global `newrelic.ini` file applies one configuration to every PHP app on the server. Per-directory INI settings let you **override specific settings per app** — without touching the global file.

<Callout variant="tip">
  **Not sure if you need this?** If you run a single PHP application on your server, the global `newrelic.ini` is all you need. Come here only if you have multiple apps on one server, need different settings per directory, or want to disable the agent for a specific path.
</Callout>

## Apache [#perdir-apache]

### Method 1: VirtualHost in `httpd.conf` [#apache-virtualhost]

Override settings inside each `<VirtualHost>` block using `php_value` for strings and numbers, or `php_flag` for booleans. This is the recommended approach for Apache — all overrides live in one file.

**Step 1:** Find your PHP module name:

```bash
apachectl -t -D DUMP_MODULES | grep php
```

Common names: `php_module`, `php5_module`, `mod_php5`. Replace `PHP_MODULE` in the examples below with your actual module name.

**Step 2:** Add per-app settings to each `<VirtualHost>` block:

```apacheconf
# Global default in newrelic.ini: newrelic.appname = "My Server"

<VirtualHost 192.168.42.43>
  ServerName www.app1.com
  DocumentRoot "/path/to/app1/"
  <IfModule PHP_MODULE>
    php_value newrelic.appname "App One"
  </IfModule>
</VirtualHost>

<VirtualHost 192.168.42.44>
  ServerName www.app2.com
  DocumentRoot "/path/to/app2/"
  <IfModule PHP_MODULE>
    php_value newrelic.appname "App Two"
  </IfModule>
</VirtualHost>

<VirtualHost 192.168.42.45>
  ServerName www.app3.com
  DocumentRoot "/path/to/app3/"
  <IfModule PHP_MODULE>
    php_flag newrelic.enabled off
  </IfModule>
</VirtualHost>
```

**Syntax rules:**
- Strings and numbers: `php_value newrelic.SETTING "value"`
- Booleans: `php_flag newrelic.SETTING on` or `off`
- Restart Apache after changes: `sudo apachectl restart`

---

### Method 2: `.htaccess` file [#apache-htaccess]

Place a `.htaccess` file in any directory to override settings for that directory and all its subdirectories. Same `php_value` / `php_flag` syntax as `httpd.conf`.

**Example:** A server with three sections that should report separately:

```
/data/webroot/          ← uses global newrelic.ini appname
/data/webroot/blog/     ← .htaccess sets appname = "My Blog"
/data/webroot/shop/     ← .htaccess sets appname = "Shopping Cart"
```

`/data/webroot/blog/.htaccess`:

```apacheconf
php_value newrelic.appname "My Blog"
```

`/data/webroot/shop/.htaccess`:

```apacheconf
php_value newrelic.appname "Shopping Cart"
```

Any path **without** a `.htaccess` file uses the global app name from `newrelic.ini`.

<Callout variant="important">
  The `.htaccess` file must be in the **top-level directory** of the application. It applies to everything inside that directory and its subdirectories.
</Callout>

**Advantage over `httpd.conf`:** No Apache restart required — changes take effect immediately on the next request.

For more on `.htaccess` setup and security, see the [Apache HTTP server tutorial](http://httpd.apache.org/docs/2.2/howto/htaccess.html).

## PHP-FPM [#php-fpm_per-dir]

PHP-FPM runs PHP as a separate process. It does not read Apache's `php_value` directives, so you need FPM-specific methods.

### Method 1: Pool configuration [#fpm-pool]

Each app gets its own pool in `php-fpm.conf` (or a separate `.conf` file in `php-fpm.d/`). Each pool has a dedicated socket and its own settings:

```ini
[app1]
listen = /tmp/pool-app1.sock
php_value[newrelic.appname] = "My App 1"

[app2]
listen = /tmp/pool-app2.sock
php_value[newrelic.appname] = "My App 2"

[app3]
listen = /tmp/pool-app3.sock
php_flag[newrelic.enabled] = off
```

**Syntax rules:**
- Strings and numbers: `php_value[newrelic.SETTING] = "value"`
- Booleans: `php_flag[newrelic.SETTING] = on` or `off`

Then point your web server at the correct pool socket for each app. Restart PHP-FPM after changes: `sudo systemctl restart php-fpm`

<Callout variant="caution">
  Each pool requires its own set of worker processes. If you have many apps, this approach uses significantly more memory. For large numbers of apps, consider the `.user.ini` method below.
</Callout>

---

### Method 2: `.user.ini` file [#fpm-user-ini]

Place a `.user.ini` file in the root directory of your app. PHP-FPM reads it automatically every 5 minutes — **no restart needed**.

```ini
newrelic.appname = "My App Name"
```

**Use `.user.ini` when:**
- You have many apps and don't want to create a separate FPM pool for each
- You're using NGINX and `fastcgi_param PHP_VALUE` values aren't reaching PHP (a common NGINX/FPM communication gap)
- PHP is running with `suPHP` and `.htaccess` doesn't work

<Callout variant="important">
  PHP-FPM caches `.user.ini` files. Changes take effect within 5 minutes by default (controlled by `user_ini.cache_ttl` in `php.ini`). You don't need to restart PHP-FPM.
</Callout>

For more on `.user.ini`, see the [PHP user.ini documentation](http://php.net/manual/en/configuration.file.per-user.php).

## NGINX [#PHP-FPM_nginix]

NGINX passes settings to PHP-FPM via `fastcgi_param PHP_VALUE` inside location blocks:

```nginx
location /blog {
  fastcgi_param PHP_VALUE "newrelic.appname=My Blog";
  include fastcgi_params;
  fastcgi_pass unix:/tmp/php-fpm.sock;
}

location /shop {
  fastcgi_param PHP_VALUE "newrelic.appname=Shopping Cart";
  include fastcgi_params;
  fastcgi_pass unix:/tmp/php-fpm.sock;
}

location /admin {
  fastcgi_param PHP_VALUE "newrelic.enabled=false";
  include fastcgi_params;
  fastcgi_pass unix:/tmp/php-fpm.sock;
}
```

Reload NGINX after changes: `sudo nginx -s reload`

<Callout variant="important">
  Requires PHP 5.3.3 or higher.
</Callout>

**If `fastcgi_param` values aren't reaching PHP:** This is a known NGINX/FPM communication issue. Use the [`.user.ini` method](#fpm-user-ini) as a fallback — place a `.user.ini` file in the app's root directory with the settings you need.

## API option: set app name from code [#api-calls]

If you can't edit server config files (managed hosting, third-party platforms), set the app name directly in PHP code:

```php
newrelic_set_appname("My App Name");
```

Call this **as early as possible** in your bootstrap or `index.php` — before any transactions start. Read the [call and location behavior guidelines](https://docs.newrelic.com/docs/apm/agents/php-agent/php-agent-api/newrelic_set_appname/#call-and-location-behavior) carefully, as timing affects which transactions get attributed to which app name.

For other settings you can change via the API, see the [PHP agent API reference](/docs/php/the-php-api).

## Roll-up application names [#perdir-rollup]

A semicolon-separated list of app names lets one app report to **multiple New Relic entities at once**. This is useful when you want both a per-app view and an aggregate view across all apps on a server:

```ini
newrelic.appname = "Virtual Host 1;All Virtual Hosts"
```

- **"Virtual Host 1"** shows data for that specific app only
- **"All Virtual Hosts"** aggregates data from every app using this as a secondary name

You can define up to three names total (one primary + two secondary). The primary name is always listed first.
