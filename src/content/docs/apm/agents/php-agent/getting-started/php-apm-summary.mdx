After installing the PHP agent, the APM summary page is your operational hub. PHP apps have patterns that look different from other languages — transaction names that depend on your framework, a response time breakdown dominated by database queries, and background jobs that track separately from web traffic.

## What you'll see in APM [#what-youll-see]

<Callout variant="important">
  Some sections of the APM summary page require additional configuration to populate. Distributed tracing, logs in context, and infrastructure data won't appear unless enabled in your agent config. See [PHP agent configuration](/docs/apm/agents/php-agent/configuration/php-agent-configuration) to verify your setup.
</Callout>

<Callout variant="tip">
  If your app isn't showing up yet, run `php -r "phpinfo();" | grep -i newrelic` to confirm the extension is loaded, then see [No data appears (PHP)](/docs/agents/php-agent/troubleshooting/no-data-appears-php).
</Callout>

### Transaction naming [#transaction-naming]

Transaction names are usually the first thing that looks unexpected after a fresh install. What you see depends on whether the PHP agent detected your framework:

| Situation | What you see in APM |
|---|---|
| Framework detected and instrumented | Rosrc/content/docs/apm/agents/php-agent/getting-started/php-apm-summary.mdxute names: `GET /products/{id}`, `POST /checkout` |
| No framework or framework not detected | File names: `index.php`, `wp-blog-header.php` |
| CLI or background jobs | Non-web transaction names, separate from web metrics |

Seeing file names instead of route names is a configuration signal, not a data problem. See [PHP frameworks: Integrating support](/docs/apm/agents/php-agent/frameworks-libraries/php-frameworks-integrating-support-new-relic/) to enable framework instrumentation for your stack.

### Database segment dominance [#db-dominance]

PHP web frameworks (Laravel, WordPress, Symfony, Magento) generate many database queries per request — often 20 to 60 on a typical page load, more on complex pages. The yellow (database) segment in the **Web transactions** chart will often be the largest segment. **This is expected, not a problem.**

What is a problem: the yellow segment growing significantly above your established baseline, or individual query durations increasing over time.

### Web vs. non-web transactions [#web-vs-non-web]

The PHP agent tracks both HTTP requests and CLI executions — cron jobs, queue workers, and Artisan or console commands. Non-web transactions appear separately and do not affect your web response time or web error rate.

### PHP fatal errors vs. exceptions [#errors]

PHP has two distinct error types with different signatures in APM:

| Error type | How it appears in APM |
|---|---|
| Exceptions (caught or uncaught) | Error rate spike — visible in Errors inbox with a stack trace |
| PHP fatal errors | Throughput drop — the process terminated before responding, so no error is recorded; the request disappears |

If throughput drops suddenly with no corresponding error rate increase, check your PHP error log for fatal errors.

## Framework-specific patterns [#framework-patterns]

<CollapserGroup>
  <Collapser
    id="laravel"
    title="Laravel"
  >
    Routes appear as `GET /api/users` or `POST /orders`. If you see `index.php` instead, verify your Laravel version is supported in [PHP agent compatibility requirements](/docs/apm/agents/php-agent/getting-started/php-agent-compatibility-requirements/).

    **Eloquent N+1 queries**: High query count per transaction (50+) with individually fast queries. In the **Databases** page, look for the same query executing dozens of times per request. Fix with eager loading (`with()`).

    **Queue workers**: Laravel queue workers, including Horizon, run as non-web transactions. They appear under **Transactions > Non-web** — not in your web traffic metrics. Monitor non-web throughput and error rate separately.

    **Laravel Octane**: Octane's persistent worker mode (Swoole or RoadRunner) shares memory between requests. Watch the memory trend in the **Infrastructure** section over time rather than expecting per-request resets.
  </Collapser>

  <Collapser
    id="wordpress"
    title="WordPress"
  >
    Transactions are named by page template — `template-archive.php`, `single.php`, `wp-login.php`. Admin requests (`/wp-admin/`) appear as separate transactions and typically show higher response times than frontend requests.

    **High query count is expected**: A typical WordPress frontend page makes 20–40 database queries. A page with many active plugins or complex WP_Query calls can exceed 100. What to watch: query count increasing over time on the same page, or specific queries slowing as the database grows.

    **Plugin conflicts**: If error rate spikes without a code change, a plugin update is a common cause. The Errors inbox shows the plugin file path in the stack trace. The **Distributed tracing insights** section may also reveal unexpected external calls introduced by a plugin.

    **WP-CLI**: WP-CLI commands and scheduled `wp cron event run` calls appear as non-web transactions. Errors during off-peak cron runs can hide in non-web metrics if you're only watching the web error rate.
  </Collapser>

  <Collapser
    id="symfony"
    title="Symfony"
  >
    Routes appear as controller actions — `App\Controller\ProductController::index`.

    **Doctrine N+1 queries**: High query count with fast individual execution in the **Databases** page indicates N+1, not a slow database. Fix with `JOIN FETCH` or second-level cache.

    **Console commands**: Symfony console commands run as non-web transactions. Check **Transactions > Non-web** for command execution times and error rates.

    **Messenger component**: Each dispatched message is handled as an individual transaction. Monitor non-web throughput to track processing rate and detect backed-up queues.
  </Collapser>

  <Collapser
    id="magento"
    title="Magento / Adobe Commerce"
  >
    **Very high query counts are normal**: A Magento product page typically executes 100–300 database queries. The yellow segment will dominate the **Web transactions** chart. Investigate only when the count increases significantly above your baseline or when specific query durations rise.

    **Checkout performance**: Checkout (`/checkout/`) transactions are your most latency-sensitive. Watch their response time independently from catalog pages. A checkout slowdown that doesn't affect catalog pages typically points to payment gateway latency (green segment) or order write contention in the database.

    **Cron and indexers**: Magento cron jobs and indexers run as non-web transactions. Reindex operations are expensive and can increase database query times on the web tier during the same window. Correlate non-web spikes with web slowdowns when investigating response time increases.

    **Full-page cache**: When FPC is warmed, most catalog requests bypass the PHP agent entirely. APM throughput reflects only uncached or partially cached requests. A drop in APM throughput after cache warming is expected, not a sign of traffic loss.
  </Collapser> 
</CollapserGroup>
