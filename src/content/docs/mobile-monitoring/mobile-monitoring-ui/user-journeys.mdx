Mobile User Journeys visualizes the complete sequence of user interactions preceding a crash, giving you an aggregate view of how users navigated your app before something went wrong. Instead of analyzing individual crash reports in isolation, you can spot recurring patterns across sessions, correlate specific interaction sequences with crash triggers, and resolve issues faster with full context around every event.

## Why use Mobile User Journeys? [#why-use]

Mobile User Journeys helps you move beyond reactive crash analysis by giving you a clear picture of the user experience before a crash occurred:

- **Instantly visualize crash patterns**: See all user journeys leading up to crashes across sessions, making it easy to identify common interaction sequences and root causes.
- **Analyze crashes in context**: With a single click, examine any user action leading up to a crash, including app version, device, OS, and carrier, to surface issues specific to certain configurations.
- **Trace individual user journeys**: Pinpoint the exact user action that triggered an error by drilling into the event trail of users who experienced crashes, enabling faster and more accurate root-cause analysis.
- **Identify friction points proactively**: Recognize patterns in user behavior that consistently precede errors, so you can address underlying issues before they affect more users.

## How to access User Journeys [#access]

Mobile User Journeys is currently available within the crash analysis experience:

1. Go to **[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > Mobile**.
2. Select your mobile application.
3. In the left navigation menu, click **Crash analysis**.
4. Select a crash group to open the crash details.
5. Click the **User journeys** tab to see the interaction flow leading up to that crash.

## How to use Mobile User Journeys [#how-to-use]

### The interaction flow visualization [#visualization]

The User Journeys visualization shows the sequence of user interactions that occurred before a crash, displayed as a flow diagram. Each node represents a screen view or user action, and the connecting paths show how users moved through the app before the crash occurred.

The diagram is built from interaction events captured by the mobile agent, including screen views, taps, and other tracked user actions. This gives you an aggregate view across all sessions that experienced the selected crash.

### Journey types [#journey-types]

You can view user journeys from three perspectives:

#### Journeys starting at session start

Shows all interactions from the beginning of the session up to the crash. This is the default view and is useful for understanding the full context of what a user was doing before the crash occurred.

#### Journeys containing a specific screen

Filters to show only journeys that passed through a particular screen. Use this to investigate whether a specific screen or interaction is consistently present in the path leading to a crash.

#### Journeys ending at the crash

Shows the interactions that immediately preceded the crash event. This view is useful for narrowing down the exact sequence of actions that triggered the error.

### Path and event details [#path-details]

Clicking on any path or event node opens a details panel showing:

- The sessions that followed that specific path
- Session attributes such as app version, device model, OS version, and carrier
- Links to related crash reports for those sessions

There is also a **Profiles** tab in the details panel that shows the distribution of attribute values across sessions that followed a given path. Use this to identify whether a crash pattern is concentrated in a specific app version, device type, or OS version.

## How to customize the visualization [#customize]

### Filter by session attributes [#filter-attributes]

Use the filter bar to narrow the journeys shown to sessions that match specific criteria. For example, you can filter to only show journeys from users on a particular OS version or app version to isolate platform-specific issues.

### Events before and after [#events-before-after]

By default, the visualization shows the interactions immediately before the crash. You can adjust how many events are shown before the crash point to get a broader or more focused view of the interaction sequence.

### Paths per event [#paths-per-event]

This setting controls how many paths are displayed after each event node. Reducing the number focuses the view on the most common paths, while increasing it gives a more complete picture of all possible sequences.
