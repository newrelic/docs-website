If your environment routes outbound traffic through a corporate proxy, you can configure the New Relic eBPF agent to send OTLP telemetry through that proxy. This page explains how to configure proxy support on Linux hosts and Kubernetes clusters, the proxy server requirements, and how to troubleshoot common issues.

<Callout variant="important">

The eBPF agent supports HTTP CONNECT proxies only. The `https://` proxy scheme (a TLS-encrypted connection to the proxy itself) is not supported by gRPC's HTTP CONNECT implementation. Your telemetry data still travels TLS-encrypted **end-to-end** between the agent and the New Relic OTLP endpoint — the proxy only sees an encrypted tunnel and never has access to your data. The `http://` scheme refers to the connection between the agent and the proxy, not to your data in transit.

</Callout>

## How proxy support works [#how-it-works]

The eBPF agent uses gRPC's built-in HTTP proxy support. When a proxy is configured, the agent:

1. Connects to the proxy server.
2. Sends an `HTTP CONNECT` request to open a tunnel to the New Relic OTLP endpoint.
3. Performs a TLS handshake with the OTLP endpoint *through* that tunnel.
4. Adds a `Proxy-Authorization: Basic <base64>` header if you provided credentials.

```
┌─────────────┐  HTTP CONNECT   ┌─────────────┐  TLS / gRPC   ┌──────────────────┐
│ eBPF Agent  │ ────────────────│    Proxy    │───────────────│  OTLP Endpoint   │
└─────────────┘                 └─────────────┘               │ otlp.nr-data.net │
                                                              └──────────────────┘
```

## Configure a proxy on Linux hosts [#linux]

You can configure proxy settings on a Linux host using either environment variables or the `newrelic-ebpf-agent.yaml` configuration file.

### Configuration precedence [#linux-precedence]

When more than one source defines a proxy, the agent applies them in this order (highest priority first):

1. `NEW_RELIC_PROXY_URL` — complete proxy URL.
2. `NEW_RELIC_PROXY_*` — individual proxy settings (host, port, scheme, user, password).
3. `HTTPS_PROXY` — checked before `HTTP_PROXY`.
4. `HTTP_PROXY` — fallback when no other source is set.

### Option 1: Environment variables [#linux-env-vars]

The agent recognizes the following New Relic-specific environment variables:

<table>
    <thead>
        <tr>
            <th>Variable</th>
            <th>Description</th>
            <th>Example value</th>
        </tr>
    </thead>
    <tbody>
        <tr>
            <td>`NEW_RELIC_PROXY_HOST`</td>
            <td>Proxy hostname. If empty, no proxy is used.</td>
            <td>`proxy.example.com`</td>
        </tr>
        <tr>
            <td>`NEW_RELIC_PROXY_PORT`</td>
            <td>Proxy port. Required when `NEW_RELIC_PROXY_HOST` is set.</td>
            <td>`8080`</td>
        </tr>
        <tr>
            <td>`NEW_RELIC_PROXY_SCHEME`</td>
            <td>Proxy scheme. Only `http` is supported.</td>
            <td>`http`</td>
        </tr>
        <tr>
            <td>`NEW_RELIC_PROXY_USER`</td>
            <td>Username for Basic Authentication.</td>
            <td>`myuser`</td>
        </tr>
        <tr>
            <td>`NEW_RELIC_PROXY_PASS`</td>
            <td>Password for Basic Authentication.</td>
            <td>`mypassword`</td>
        </tr>
        <tr>
            <td>`NEW_RELIC_PROXY_URL`</td>
            <td>Complete proxy URL. Overrides all other `NEW_RELIC_PROXY_*` variables.</td>
            <td>`http://user:pass@proxy.example.com:8080`</td>
        </tr>
    </tbody>
</table>

If no `NEW_RELIC_PROXY_*` variables are set, the agent falls back to standard `HTTPS_PROXY` and then `HTTP_PROXY`. Both variables use the `http://host:port` format — the scheme refers to the connection to the proxy, not to your telemetry data, which is always sent over TLS to the OTLP endpoint.

**Example — unauthenticated proxy:**

```bash
export NEW_RELIC_PROXY_HOST=proxy.corporate.com
export NEW_RELIC_PROXY_PORT=8080
sudo systemctl restart newrelic-ebpf-agent
```

**Example — authenticated proxy:**

```bash
export NEW_RELIC_PROXY_HOST=proxy.corporate.com
export NEW_RELIC_PROXY_PORT=8080
export NEW_RELIC_PROXY_USER=myuser
export NEW_RELIC_PROXY_PASS=mypassword
sudo systemctl restart newrelic-ebpf-agent
```

**Example — using the URL form:**

```bash
export NEW_RELIC_PROXY_URL=http://myuser:mypass@proxy.corporate.com:8080
sudo systemctl restart newrelic-ebpf-agent
```

### Option 2: Configuration file [#linux-config-file]

Add an `otlpProxy:` block to `/etc/newrelic-ebpf-agent/newrelic-ebpf-agent.yaml`:

```yaml
otlpProxy:
  host: "proxy.corporate.com"
  port: 8080
  scheme: "http"
  user: "myuser"
  password: "mypassword"
```

After saving the file, restrict its permissions and restart the agent:

```bash
sudo chmod 600 /etc/newrelic-ebpf-agent/newrelic-ebpf-agent.yaml
sudo systemctl restart newrelic-ebpf-agent
```

For the full list of `otlpProxy.*` parameters, see [Configuration parameters](/docs/ebpf/linux-installation/#config-params) on the Linux installation page.

## Configure a proxy on Kubernetes [#kubernetes]

For Kubernetes clusters, configure the proxy in your Helm `values.yaml` file. For production deployments, use a Kubernetes secret to store credentials instead of writing them directly into `values.yaml`.

### Option 1: Inline credentials [#k8s-inline]

Add an `otlpProxy:` block to your `values.yaml`:

```yaml
otlpProxy:
  host: "proxy.corporate.com"
  port: 8080
  scheme: "http"
  user: "myuser"
  password: "mypassword"
```

Then apply the change:

```bash
helm upgrade --install nr-ebpf-agent newrelic/nr-ebpf-agent -n newrelic --values values.yaml
```

### Option 2: Kubernetes secret (recommended) [#k8s-secret]

Storing proxy credentials in a Kubernetes secret keeps them out of your `values.yaml` file and out of source control.

1. Create the secret. The keys must be named `proxy-user` and `proxy-password`:

    ```bash
    kubectl create secret generic ebpf-proxy-creds \
      --namespace newrelic \
      --from-literal=proxy-user=myuser \
      --from-literal=proxy-password=mypassword
    ```

2. Reference the secret in your `values.yaml`:

    ```yaml
    otlpProxy:
      host: "proxy.corporate.com"
      port: 8080
      existingSecret: "ebpf-proxy-creds"
    ```

3. Apply the change:

    ```bash
    helm upgrade --install nr-ebpf-agent newrelic/nr-ebpf-agent -n newrelic --values values.yaml
    ```

For the full list of `otlpProxy.*` parameters, see [K8s configuration parameters](/docs/ebpf/k8s-installation/#config-params) on the Kubernetes installation page.

## Proxy server requirements [#proxy-requirements]

Your proxy server must support:

* The HTTP `CONNECT` method (required for TLS tunneling to the OTLP endpoint).
* Tunneling to port `443` (the OTLP endpoint uses HTTPS).
* HTTP Basic Authentication, if you use authenticated proxy access.

## Credential character restrictions [#credential-restrictions]

<Callout variant="caution">

gRPC's proxy URI parser silently lowercases usernames and rejects URLs containing percent-encoded characters in the credential portion. If your username uses uppercase letters or your username or password contains special characters, proxy authentication fails without an obvious error. The eBPF agent validates credentials at startup to surface these problems early instead of letting them fail silently.

</Callout>

When you configure proxy credentials, only use the following characters:

* **Username:** lowercase letters (`a–z`), digits (`0–9`), hyphen (`-`), underscore (`_`), period (`.`), tilde (`~`). Uppercase letters are not allowed.
* **Password:** letters (`a–z`, `A–Z`), digits (`0–9`), hyphen (`-`), underscore (`_`), period (`.`), tilde (`~`).

The following characters are **not allowed** in either the username or the password: `@`, `:`, `/`, `?`, `#`, `[`, `]`, `!`, `$`, `&`, `'`, `(`, `)`, `*`, `+`, `,`, `;`, `=`, `%`, spaces, and other special characters.

If your existing proxy credentials don't meet these restrictions, ask your proxy administrator to create a dedicated proxy account for the eBPF agent that uses a lowercase alphanumeric username and a simple password.

## Troubleshooting [#troubleshooting]

### Verify your proxy configuration [#verify-config]

Check the agent log for proxy-related messages:

```bash
journalctl -u newrelic-ebpf-agent | grep -i proxy
```

Look for one of these messages:

* `Using proxy for OTLP endpoint: <endpoint> via http://<host>:<port>` — proxy is enabled.
* `Proxy authentication enabled (username: <user>)` — Basic Authentication is configured.
* `Direct connection to OTLP endpoint (no proxy configured)` — no proxy is in use.

### Test proxy connectivity [#test-connectivity]

Use `curl` to confirm that your proxy supports the `HTTP CONNECT` method and can reach the New Relic OTLP endpoint:

```bash
# Unauthenticated proxy
curl -v -x http://proxy.example.com:8080 https://otlp.nr-data.net

# Authenticated proxy
curl -v -x http://user:pass@proxy.example.com:8080 https://otlp.nr-data.net
```

A successful response includes:

```
< HTTP/1.1 200 Connection established
* Proxy replied 200 to CONNECT request
* CONNECT phase completed!
```

### Common errors [#common-errors]

<CollapserGroup>

<Collapser
    id="error-400"
    title="HTTP proxy returned response code 400"
>

**Cause:** The proxy doesn't support the HTTP `CONNECT` method.

**Solution:** Use a `CONNECT`-capable proxy. See [Supported proxy servers](#supported-proxies).

</Collapser>

<Collapser
    id="error-https-scheme"
    title="'https' scheme not supported in proxy URI"
>

**Cause:** gRPC does not support `https://` as a proxy scheme. This is a gRPC limitation, not an agent restriction.

**Solution:** Set the proxy scheme to `http` (for example, `NEW_RELIC_PROXY_SCHEME=http` or `otlpProxy.scheme: "http"`). Your telemetry data still travels TLS-encrypted end-to-end through the proxy tunnel — only the connection to the proxy itself is plaintext.

</Collapser>

<Collapser
    id="error-socket-closed"
    title="Socket closed or connection refused"
>

**Cause:** The proxy server is not running, or the host or port is incorrect.

**Solution:** Verify that the proxy is reachable using the [`curl` test command](#test-connectivity).

</Collapser>

<Collapser
    id="error-connect-aborted"
    title="Proxy CONNECT aborted"
>

**Cause:** The proxy rejected the `CONNECT` request because of a firewall rule, ACL, or unsupported destination.

**Solution:** Check your proxy server logs and confirm that `CONNECT` requests to port `443` are permitted from the agent's source IP.

</Collapser>

<Collapser
    id="error-tls-required"
    title="Proxy configuration ignored: gRPC HTTP CONNECT proxy requires TLS"
>

**Cause:** A proxy is configured but TLS is disabled (for example, `TLS_ENABLED=false`). gRPC's HTTP CONNECT proxy implementation only works when the connection to the OTLP endpoint uses TLS.

**Solution:** Make sure TLS is enabled (it is by default). The agent will then use the configured proxy.

</Collapser>

<Collapser
    id="error-credential-chars"
    title="Proxy username contains special or uppercase characters"
>

**Cause:** Your proxy username or password contains characters that gRPC's URI parser doesn't accept. See [Credential character restrictions](#credential-restrictions).

**Solution:** Use a proxy account with a lowercase alphanumeric username and a password that uses only letters, digits, and the allowed special characters.

</Collapser>

</CollapserGroup>

## Security considerations [#security]

* **Telemetry data stays TLS-encrypted end-to-end.** The proxy only sees an encrypted tunnel to the OTLP endpoint; it never has access to your data.
* **Proxy credentials are sent in plaintext to the proxy** because the `https://` proxy scheme isn't supported. Only use proxy authentication on a trusted network between the agent and the proxy. If you need encrypted transport between the agent and proxy, run the proxy on `localhost` or use a VPN.
* **Passwords are never written to agent logs.** The username may appear in log lines such as `Proxy authentication enabled (username: <user>)` for debugging.
* **In Kubernetes, prefer `existingSecret` over inline credentials** to keep proxy passwords out of your `values.yaml` file and source control.
