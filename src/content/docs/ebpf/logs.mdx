The New Relic eBPF agent now includes application log ingestion capabilities, evolving from a specialized monitoring tool into a comprehensive **MELT (Metrics, Events, Logs, Traces)** engine. This advancement furthers our single-agent strategy by eliminating the need for additional log forwarding agents, reducing deployment complexity and improving security posture.

## Why eBPF logs? [#why-ebpf-logs]

Today's observability landscape suffers from **fragmentation and friction**. Managing separate agents for different data signals often diminishes the return on investment. The eBPF logs capability addresses critical operational challenges:

* **Eliminate additional agents**: Reduces resource contention, correlation breakage, and maintenance complexity by consolidating log collection into the existing eBPF agent.
* **Zero-plugin security improvement**: Leverages kernel-level log interception to eliminate third-party dependencies, reducing attack surface and supply chain risks.
* **Unified MELT collection**: Provides single-agent strategy with logs-in-context by default, streamlining engineering lifecycle management.

## Important limitations [#limitations]

<Callout variant="caution">
During Limited Preview, be aware of these important constraints:

* **Billable ingestion**: All logs ingested through eBPF logs will be billable according to standard New Relic log pricing
* **No data obfuscation**: Limited or no obfuscation capabilities are available for sensitive data like credit card numbers, Personal Identifiable Information (PII), or financial data
* **Non-production use recommended**: Due to obfuscation limitations, this feature is recommended for development and testing environments only

Data obfuscation capabilities will be enhanced through New Relic's Pipeline Control Gateway before general availability.
</Callout>

## Configuration [#configuration]

To enable eBPF logs capability, configure the following parameters in your eBPF agent deployment:

<CollapserGroup>

<Collapser
    className="freq-link"
    id="k8s-config"
    title="Kubernetes configuration"
>

Add these parameters to your `values.yaml` file:

```yaml
# Enable log reporting
logDataReporting: true

# Set sampling rate (optional)  
logSamplingRate: 1.0

# Configure log file patterns to match (optional)
logFilePatterns: "/var/log/app/*.log,/app/logs/*.log"

# Specify application names to match (optional)
appNameFilter: "my-app,web-service"
```

</Collapser>

<Collapser
    className="freq-link"
    id="linux-config"
    title="Linux host configuration"
>

Add these parameters to your `newrelic-ebpf-agent.conf` file:

```bash
# Enable log reporting
LOG_DATA_REPORTING="true"

# Set sampling rate (optional)
LOG_SAMPLING_RATE="1.0"

# Configure log file patterns to match (optional)
LOG_FILE_PATTERNS="/var/log/app/*.log,/app/logs/*.log"

# Specify application names to match (optional)
APP_NAME_FILTER="my-app,web-service"
```

</Collapser>

</CollapserGroup>

<Callout variant="tip">
The eBPF logs configuration is more focused compared to comprehensive log forwarding solutions like Fluent Bit. It provides essential log collection capabilities optimized for application observability use cases.
</Callout>

## How it works [#how-it-works]

Unlike traditional log forwarders that "tail" files, the eBPF logs capability uses kernel-level visibility for more efficient log capture:

### System call interception
* **File descriptor monitoring**: Monitors file descriptors (FD) and write-related system calls at the kernel level
* **Source-level capture**: Since nearly all application logs pass through kernel interfaces, eBPF captures log data directly at the source with minimal overhead
* **Zero-config discovery**: Automatically detects application log output without requiring file path configuration

### Entity synthesis and linking
* **Automatic correlation**: Applies existing eBPF entity synthesis rules to link captured logs to specific APM service entities already identified by the agent
* **Logs-in-context activation**: Automatically enables New Relic's logs-in-context capabilities without additional customer configuration
* **Unified entity view**: Presents logs alongside metrics, events, and traces for the same application entities

### OpenTelemetry-first architecture
* **OTel endpoint integration**: Sends logs to New Relic's OpenTelemetry endpoint, maintaining consistency with modern observability standards
* **Efficient filtering**: Provides robust filtering at the source to prevent data bloat and maintain performance
* **Resource integrity**: Balances log ingestion capabilities with the low-footprint promise of the eBPF agent

This kernel-level approach eliminates the need for traditional log forwarding infrastructure while providing comprehensive application log visibility directly correlated with your existing telemetry data.

## Use cases [#use-cases]

eBPF logs is ideal for scenarios where you want to:

### Operational efficiency
* **Simplify log collection architecture**: Reduce from multiple agents to a single comprehensive MELT solution
* **Eliminate agent tax**: Move from managing 2+ agent lifecycles to just one, reducing configuration drift and maintenance hours
* **Accelerate time to first log**: Target time to seeing logs in the UI after installation is less than 5 minutes

### Enhanced security
* **Security hardening**: Consolidate telemetry collection to minimize attack surface and reduce binaries with root/kernel access
* **Supply chain risk reduction**: Eliminate dependencies on third-party log forwarding plugins and their associated CVEs
* **Zero-plugin architecture**: Leverage kernel-level visibility instead of external log processing tools

### Improved observability
* **Logs-in-context by default**: Automatically correlate logs with APM traces and infrastructure metrics without additional configuration
* **Entity-linked logging**: Achieve high log-to-entity match rates through automatic synthesis rules
* **Unified troubleshooting**: Access complete MELT telemetry from a single collection and correlation point

This approach is particularly valuable for organizations managing containerized applications, microservices architectures, or environments where security and operational simplicity are paramount.
