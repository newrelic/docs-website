This guide covers the infrastructure validation, Docker daemon configuration, and image deployment required to support Dual Stack (IPv4 and IPv6) testing in your private location environment using Docker.

## Prerequisites [#prerequisites]

Before you begin, ensure you have the following:

* <DNT>**Docker**</DNT>: Docker must be installed and running on your host instance
* <DNT>**Permissions**</DNT>: You must have sudo or root access to modify system configurations
* <DNT>**Custom images**</DNT>: Dual Stack-enabled runtime images for synthetics job manager:
  * Job manager image (`.tar.gz` or `.tar`)
  * Ping runtime image (`.tar.gz` or `.tar`)
  * Node API runtime image (`.tar.gz` or `.tar`)
  * Node browser runtime image (`.tar.gz` or `.tar`)

## Validate your infrastructure [#validate-infrastructure]

Before proceeding, verify that your host machine and network are configured for Dual Stack networking.

Test IPv6 connectivity by running:

```bash
ping -6 ipv6.google.com
```

<Callout variant="important">
  If the ping fails, ensure your VPC, subnet, and security groups are configured to allow IPv6 traffic before continuing with this installation. Your environment must support both IPv4 and IPv6 for Dual Stack operation.
</Callout>

## Configure Docker for Dual Stack [#configure-docker]

Docker requires explicit configuration to enable Dual Stack support with IPv6 routing and addressing.

1. Open the Docker daemon configuration file:

   ```bash
   sudo vi /etc/docker/daemon.json
   ```

2. Add or update the file with the following configuration:

   ```json
   {
     "ipv6": true,
     "fixed-cidr-v6": "fd00:0:0:1::/64",
     "ip6tables": true
   }
   ```

3. Restart Docker to apply the changes:

   ```bash
   sudo systemctl restart docker
   ```

## Load custom runtime images [#load-runtime-images]

Upload the provided Dual Stack-enabled image files to your host instance, then import them into your local Docker engine.

Load each runtime image using the following commands:

```bash
docker load -i <job-manager-image>.tar.gz
docker load -i <node-api-runtime-image>.tar.gz
docker load -i <node-browser-runtime-image>.tar.gz
docker load -i <ping-runtime-image>.tar
```

Replace the placeholders with your actual image filenames.

## Set up a local registry [#setup-local-registry]

If your environment requires a local registry to distribute these images to the Synthetics job manager, follow these steps:

1. Start a local registry:

   ```bash
   docker run -d --restart=always -p 5000:5000 --name registry registry:2
   ```

2. Verify the registry is active:

   ```bash
   curl -s localhost:5000/v2/_catalog
   ```

3. Tag and push the Dual Stack-enabled runtimes to your local registry:

   ```bash
   # Browser runtime
   docker tag <node-browser-runtime-image>:<tag> localhost:5000/node-browser-runtime:<tag>
   docker push localhost:5000/node-browser-runtime:<tag>

   # API runtime
   docker tag <node-api-runtime-image>:<tag> localhost:5000/node-api-runtime:<tag>
   docker push localhost:5000/node-api-runtime:<tag>

   # Ping runtime
   docker tag <ping-runtime-image>:<tag> localhost:5000/synthetics-ping-runtime:<tag>
   docker push localhost:5000/synthetics-ping-runtime:<tag>
   ```

   Replace the placeholders with your actual image names and tags.

## Launch the Synthetics job manager [#launch-job-manager]

Run the Synthetics job manager container with the Dual Stack-enabled runtimes.

<Callout variant="tip">
  Replace the placeholders with your actual private location key from the New Relic UI and your image names/tags.
</Callout>

```bash
docker run \
  -e PRIVATE_LOCATION_KEY=<YOUR_PRIVATE_LOCATION_KEY> \
  -e 'DESIRED_RUNTIMES=["localhost:5000/synthetics-ping-runtime:<tag>","localhost:5000/node-browser-runtime:<tag>","localhost:5000/node-api-runtime:<tag>"]' \
  -d \
  --restart unless-stopped \
  -v /var/run/docker.sock:/var/run/docker.sock:rw \
  <job-manager-image>:<tag>
```

The Synthetics job manager will now be running with Dual Stack support enabled, allowing your monitors to test both IPv4 and IPv6 endpoints.

## Verify installation [#verify-installation]

To confirm your Synthetics job manager is running correctly:

1. Check the container status:

   ```bash
   docker ps
   ```

2. View the job manager logs:

   ```bash
   docker logs <container_id>
   ```

3. In the New Relic UI, go to <DNT>**[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > Synthetic monitoring > Private locations**</DNT> and verify your private location shows as healthy.
