This technical guide explains how APM agent samplers control which transactions within a given service are captured and sent to New Relic. You'll learn about sampler types like adaptive and trace id ratio based, adaptive sampling targets, and how to configure sampling based on upstream sampling decisions.

**Prerequisites**:
1. For strategic planning across services, see [Optimize tracing costs](/docs/distributed-tracing/core-tracing/cost-optimization-strategies)

## How sampling works [#how-sampling-works]

### Sampling decision flow

1. **Request arrives** at your service
2. **Agent checks sampling context**: Is this a root transaction, or did an upstream service already make a sampling decision?
3. **Agent applies appropriate sampler**: Based on the context and your configuration
4. **Sampling decision is made**: Sample or don't sample
5. **Decision propagates**: The sampling decision is passed to downstream services via distributed trace context headers

### Interaction with partial granularity modes

Sampling and partial granularity work together:

* **Sampling** determines **which transactions** are captured
* **Partial granularity** determines which spans and attributes are recorded

## Sampling contexts [#sampling-contexts]

APM agents use three sampling contexts to handle different trace scenarios. Each context can have its own sampler configuration.

### Root context

**When it applies**: The distributed trace originates from the current service (this is the first service in the distributed trace).

**Use case**: Control sampling for distributed traces that originate the current service.

**Example scenario**: Your API gateway is the entry point for customer requests. Configure root sampling to capture 20% of these customer-initiated traces.

```yaml
distributed_tracing:
  sampler:
    root:
      trace_id_ratio_based:
        ratio: 0.2  # 20% of traces originating here
```

### Remote parent sampled context

**When it applies**: An upstream service decided to sample this distributed trace (the upstream service sent a sampling decision of "sampled" in the distributed trace context).

**Use case**: Control tracing for distributed traces that were sampled by the upstream service

**Common pattern**: If the upstream service sampled the distributed trace, always sample the distribtued trace in the current service

```yaml
distributed_tracing:
  sampler:
    remote_parent_sampled:
      always_on  # Always capture traces sampled by upstream
```

**Alternative pattern**: Capture all distributed traces that the upstream service already decided to sample, but 90% with partial granularity and 10% with full granularity:

```yaml
distributed_tracing:
  sampler:
    remote_parent_sampled:
        trace_id_ratio_based:
            ratio:
                0.1
    partial_granularity:
        enabled:
            true
        remote_parent_sampled:
            trace_id_ratio_based:
                ratio:
                    0.9
```

### Remote parent not sampled context

**When it applies**: An upstream service decided NOT to sample this distributed trace (the upstream service sent a sampling decision of "false" in the trace context).

**Use case**: Control sampling for distributed traces that were NOT sampled by the upstream service.

**Common pattern**: If the upstream service did NOT sample the distributed trace never sample the distributed trace in the current service:

```yaml
distributed_tracing:
  sampler:
    remote_parent_not_sampled:
      always_off  # Don't sample traces that upstream rejected
```

**Alternative pattern**: Sample some distributed traces even if upstream chose to not sample them (use cautiously):

```yaml
distributed_tracing:
  sampler:
    remote_parent_not_sampled:
      trace_id_ratio_based:
        ratio: 0.05  # Sample 5% even if upstream said not to
```

<Callout variant="caution">
  Overriding upstream "not sampled" decisions can create fragmented traces where only some services are present. Use this pattern only when you have a specific need to see activity in a downstream service even when upstream didn't sample it.
</Callout>

## Sampler types [#sampler-types]

APM agents provide four sampler types, each appropriate for different scenarios.

### Adaptive sampler [#adaptive-sampler]

**How it works**: Dynamically adapts to the throughput of the service to spread out sampling over the course of a harvest in order to capture a target number of distributed traces originating from the current service per minute, regardless of traffic volume. Always samples transactions that were sampled by an upstream service and does not count those towards the adaptive sampling target.

**Default behavior**: APM agents default to using the adaptive sampler. Most APM agents have a default adaptive sampling target of 10 transactions per minute. See each APM agent's configuration docs for more details.

**When to use**:
* You want a consistent number of transactions sampled per minute, regardless of traffic spikes or drops, instead of a percentage of transactions per minute.
* You want dynamic adjustment to space out sampling throughout each minute.
* You want to sample all transactions that were sampled by the upstream service and not have those count towards the adaptive sampling target.

**Configuration**:

```yaml
distributed_tracing:
  sampler:
    root: adaptive
    adaptive_sampling_target: 20
```

**How it adapts**:
* Low traffic (10 requests/min): Samples every request to approach target
* Medium traffic (100 requests/min): Samples appropriately every 5th request to hit target
* High traffic (1000 requests/min): Samples appropriately every 50th request to hit target

<Callout variant="tip">
  Note: Transactions that were sampled by the upstream service will always be sampled if remote parent sampled is configured to use the adaptive sampler (which it is by default) and these transactions do not count towards the configured adaptive sampling target.
</Callout>

## Adaptive sampling targets [#adaptive-targets]

Adaptive sampling targets control how many traces per minute the agent attempts to capture. Understanding global vs per-context targets is crucial for predictable behavior.

#### Global adaptive sampling target

The global `adaptive_sampling_target` sets the default target for all contexts that don't have an explicit override.
This target is shared for all contexts that do not have an override.

The adaptive sampling target can be configured from 1 to 120 transactions per minute.

**Configuration**:

```yaml
distributed_tracing:
  sampler:
    adaptive_sampling_target: 10  # Applied to all contexts without explicit override
    remote_parent_sampled:
      always_on
    remote_parent_not_sampled:
      always_off
    root: adaptive
    partial_granularity:
      enabled: true
      root: adaptive
```
**Result**: The agent attempts to capture:
* Approximately 5 transactions per minute that originate from the current service are sampled with full granularity
* Approximately 5 transactions per minute that originate from the current service are sampled with partial granularity

#### Per-context adaptive sampling targets

Each sampling context (`root`, `remote_parent_sampled`, `remote_parent_not_sampled`) can override the global target with its own `sampling_target`.

**Configuration**:

```yaml
distributed_tracing:
  sampler:
    adaptive_sampling_target: 10  # Global default
    remote_parent_sampled:
      always_on
    remote_parent_not_sampled:
      always_off
    root:
      adaptive:
        sampling_target: 5  # Root traces: 5/min
    partial_granularity:
      enabled: true
      root: adaptive
```

**Result**: The agent attempts to capture:
* 5 transactions per minute that originate from the current service are sampled with full granularity
* 10 transactions per minute that originate from the current service are sampled with partial granularity

#### Why increase adaptive sampling targets?

The default 10 traces/minute was designed for transaction tracing, not distributed tracing. For distributed tracing, you often want higher sampling to:

1. **Fill service map gaps**: Rare endpoints might never get sampled at 10/min
2. **Improve trace diversity**: See more transaction types and code paths


### Trace ID ratio based sampler [#trace-id-ratio-based-sampler]

**How it works**: Samples a fixed percentage of traces based on the trace ID. Because the sampling decision is based on trace ID, the same trace will get the same decision across all services using ratio-based sampling with the same ratio.

**When to use**:
* You need a consistent percentage of traffic sampled
* You want deterministic sampling (the same distributed traces are sampled across all services)
* You would like to avoid the behavior of the adaptive sampler where transactions that were sampled by the upstream service are always sampled and do not count towards the adaptive sampling target.

**Configuration**:

```yaml
distributed_tracing:
  sampler:
    root:
      trace_id_ratio_based:
        ratio: 0.1  # 10% of traces
```

**Key parameters**:
* `ratio`: Decimal between 0.0 exclusive and 1.0 inclusive (100%)

**Important characteristic**: The trace ID determines whether a trace is sampled. If multiple services use the same ratio, they'll make consistent decisions for the same trace.

**Example**: Sample 10% of root distributed traces and 50% of upstream-sampled distributed traces:

```yaml
distributed_tracing:
  sampler:
    root:
      trace_id_ratio_based:
        ratio: 0.1
    remote_parent_sampled:
      trace_id_ratio_based:
        ratio: 0.5
```

### Always-on sampler

**How it works**: Samples all transactions matching this context.

**When to use**:
* You need complete trace coverage for a specific context (`root`, `remote_parent_sampled`, `remote_parent_not_sampled`)
* You want to respect or overide the upstream sampled decisions

**Configuration**:

```yaml
distributed_tracing:
  sampler:
    remote_parent_sampled:
      always_on  # Capture all upstream-sampled traces
```

<Callout variant="warning">
  Using `always_on` will capture every transaction for the given context, which can generate significant data volume for high-traffic services. Consider capturing partial granularity instead of full granularity distributed traces for a given section if you configure `always_on`.
</Callout>

### Always-off sampler

**How it works**: Samples none of transactions matching a given context.

**When to use**:
* You want to respect upstream NOT sampled decisions
* You want to disable sampling for a specific context

**Configuration**:

```yaml
distributed_tracing:
  sampler:
    remote_parent_not_sampled:
      always_off  # Don't sample if upstream did NOT sample
```

<Callout variant="tip">
  **Tip**: Using `always_off` may be desirable to use to turn off capturing full granularity and turn on capturing partial granularity distributed traces for a given section.
</Callout>

## Priority and precedence [#priority]

1. When both full and partial granularity are enabled, if full granularity decides not sample a transaction, then the decision falls to partial granularity.
2. If the reservoir is full and some spans need to be dropped, spans sampled as partial granularity or from traces originating from the current service are first.

<Callout variant="important">
  **Implication**: In extremely high-traffic scenarios, partial granularity traces may be dropped if full granularity traces fill the reservoir. This is by design to prioritize more detailed data.
</Callout>
