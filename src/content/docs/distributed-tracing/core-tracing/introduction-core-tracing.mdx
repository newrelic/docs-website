Distributed tracing is powerful for understanding your system, but you might face incomplete trace coverage, fragmented service maps, or an all-or-nothing dilemma where you either capture everything or nothing from each service.

Partial granularity distributed traces provide controlled ingestion, allowing you to capture data that really matters in a large distributed system—the spans and attributes that represent entity connections. APM agents give you precise control over your distributed tracing data. You can instrument all services—including non-business-critical ones—to get a complete picture of your system while managing data volumes effectively. These aren't separate products—they're configuration options you set in your APM agent files.

## Common distributed tracing challenges [#challenges]

Customers tell us they face three main challenges with distributed tracing:

* **Incomplete instrumentation and fragmentation:** When only some services send traces, you get fragmented maps and broken trace paths. Gaps in instrumentation create blind spots where you can't see the complete request flow. Without complete visibility across all services (including non-business-critical supporting services), it's difficult to understand your full system behavior and identify the root cause of issues.

* **All-or-nothing problem:** You're forced to choose: capture everything from a service (high data volume) or capture nothing (no visibility). There's no middle ground that gives you connectivity data without capturing every detail for every service.

* **Data volume concerns:** While instrumenting all services provides the complete picture you need, full distributed tracing data can generate significant data volumes as your system grows. You need a way to get complete visibility while managing data volumes appropriately for your needs.

## How APM agent configuration solves these problems [#solution]

APM agents let you control exactly how much trace data each service sends to New Relic. The primary goal is to eliminate fragmentation by instrumenting all services—including non-business-critical entities—to get a complete picture of your system. You can:

* **Eliminate fragmentation** by instrumenting all services to see the complete request flow across your entire system
* **Get the full picture** using lightweight trace data for non-business-critical supporting services while maintaining complete end-to-end visibility
* **Mix configurations strategically** keeping full detail for business-critical services while using reduced granularity for supporting services

This approach lets you instrument entities that aren't business-critical but are essential for understanding the overall picture of your system. These are configuration settings in your APM agent files (Java, .NET, Node.js, Python, Go, Ruby, PHP). Not separate products, not OpenTelemetry—just APM agent settings.

**Note:** These configurations provide controlled ingestion to customize your needs while maintaining the complete visibility required for effective troubleshooting. Results vary by application.

## Three configuration options [#options]

APM agents provide three sequential configuration levels called **partial granularity distributed traces** options. Each option includes the previous level's reductions plus additional filtering:

### Reduced granularity: Remove in-process spans

Drop internal method-level spans, keeping only entry spans (incoming requests) and exit spans (calls to other services, databases, etc.). Note that AI spans are still captured even with this option enabled.

**What you get:**
* Service-to-service visibility intact
* Database and external API calls visible
* Complete distributed trace paths
* AI monitoring spans preserved

**What you lose:**
* Internal method-level execution detail
* Step-by-step code path within each service

**Typical reduction:** 50-80% fewer spans

**Best for:** Services with deep call stacks (Java, .NET) where you rarely need internal method detail

### Essential granularity: Remove in-process spans + reduce attributes

Everything from reduced granularity, plus strip most attributes from entry and exit spans. Keeps only attributes needed for trace connectivity and service relationships.

**What you get:**
* Everything from reduced granularity
* Service names and relationships
* Duration and timing data
* Error tracking

**What you additionally lose:**
* Detailed span metadata (HTTP headers, query parameters, etc.)
* Most custom attributes

**Important:** If your team relies on specific custom attributes or traced functions for monitoring or troubleshooting, verify these will still be available before enabling this option.

**Typical reduction:** 60-75% total data volume

**Best for:** Supporting services where you need connectivity but not detailed diagnostics

### Compact granularity: Remove in-process spans + reduce attributes + compress duplicates

Everything from essential granularity, plus compress duplicate exit spans. If your service calls the same database or downstream service multiple times in a request, only the first call to the same URL or database gets reported as a span.

**What you get:**
* Everything from essential granularity
* Proof that Service A calls Service B
* Complete service relationship graph

**What you additionally lose:**
* Count of how many times each service was called
* Individual timing for repeated calls

**Typical reduction:** 70-85% total data volume

**Best for:** High-volume services or non-production environments where you need connectivity proof but not call frequency

## When to use partial granularity distributed traces [#when-to-use]

Partial granularity distributed traces target customers who:

1. **Have turned distributed tracing off because it is too expensive** - Use essential or compact granularity to re-enable tracing at lower data volumes

2. **Have gaps in their service map** due to some traces occurring with a very low probability and limits on number of traces per minute - Increase sampling with partial granularity to fill gaps

3. **Only want to capture a complete service map** and don't care about the details of everything that happened within that trace - Use essential or compact granularity

4. **Want to capture both a complete service map AND still have the ability to deep dive** into a trace to root cause a performance issue - Use hybrid sampling (see [Hybrid sampling strategy](#hybrid-sampling))

5. **Want to capture low cost traces until the service encounters an issue** and then start capturing full granularity traces for a short time to debug the issue - Use partial granularity by default (future capability: error-based sampling)

6. **Have gaps in their service map and distributed tracing** due to hitting the max reservoir size and dropping spans - Use compact granularity to reduce span count

## Understanding span types [#span-types]

To understand these options, you need to know about the three types of spans in distributed tracing:

* **Entry spans:** The first span when a request enters your service. This represents the incoming request.
  * Example: An HTTP request arriving at your API service

* **Exit spans:** Spans representing calls from your service to downstream dependencies—databases, external services, other microservices, caches, etc.
  * Examples: Database queries, HTTP calls to other services, cache operations (Redis, Memcached), external API calls

* **In-process spans:** Spans representing internal method calls within your service. These show the detailed execution path inside your code.
  * Examples: Individual function calls, internal service layers, business logic methods, framework operations

Together, all spans within one service form a **transaction trace**. When you connect transaction traces across multiple services, you get a **distributed trace**.

**How the options work:**
* **Reduced granularity** removes in-process spans, keeps entry and exit spans
* **Essential granularity** removes in-process spans + strips attributes from entry and exit spans
* **Compact granularity** removes in-process spans + strips attributes + compresses duplicate exit spans

## What traces look like: Full vs Partial granularity [#trace-comparison]

Understanding how traces appear differently in the UI helps you choose the right configuration. Here's what you'll see in a waterfall view:

### Full granularity distributed trace

When viewing a trace with full distributed tracing, you see:

```
Service A (entry span)                    [============================] 210ms
├─ Exit span → Database query            [====] 30ms
├─ In-process span: validateUser()       [=======] 40ms
│  └─ In-process span: checkPermissions()[===] 20ms
├─ Exit span → Service B call            [===========] 70ms
│  └─ Service B (entry span)             [=========] 65ms
│     ├─ Exit span → Database query      [====] 25ms
│     └─ In-process span: processData()  [===] 20ms
└─ In-process span: formatResponse()     [====] 30ms
```

**What you see:** Complete detail including all internal method calls (in-process spans), all database queries, all service calls, with all span attributes (HTTP headers, query parameters, custom attributes, etc.)

### Reduced granularity trace (same request)

With reduced granularity, the same request looks like:

```
Service A (entry span)                    [============================] 210ms
├─ Exit span → Database query            [====] 30ms
└─ Exit span → Service B call            [===========] 70ms
   └─ Service B (entry span)             [=========] 65ms
      └─ Exit span → Database query      [====] 25ms
```

**What you see:** Entry and exit spans only. In-process spans (`validateUser()`, `checkPermissions()`, `processData()`, `formatResponse()`) are removed. You still see all service-to-service connections and all database calls, with all span attributes intact.

**What's preserved:** Complete service connectivity, database visibility, timing information, all attributes

**What's removed:** Internal method-level execution detail

### Essential granularity trace (same request)

With essential granularity, it looks similar to reduced but with minimal attributes:

```
Service A (entry span)                    [============================] 210ms
├─ Exit span → Database                  [====] 30ms
└─ Exit span → Service B                 [===========] 70ms
   └─ Service B (entry span)             [=========] 65ms
      └─ Exit span → Database            [====] 25ms
```

**What you see:** Same span structure as reduced granularity, but each span contains only essential attributes: `entity.name`, `service.name`, `duration`, `timestamp`, `span.id`, `trace.id`, `http.statusCode`, `error` flag, and `nr.pg` marker.

**What's preserved:** Complete service connectivity, entity relationships, timing, error tracking

**What's removed:** In-process spans, detailed HTTP headers, query parameters, custom attributes (unless explicitly opted in), SQL statements

### Compact granularity trace (same request, if Service A calls the database twice)

If the original trace had two database calls to the same database:

```
Service A (entry span)                    [============================] 210ms
└─ Exit span → Service B                 [===========] 70ms
   └─ Service B (entry span)             [=========] 65ms
```

**What you see:** Only unique entity connections. If Service A called the same database twice, only the first call appears. Duplicate exit spans to the same destination are compressed.

**What's preserved:** Proof that Service A connects to Service B, proof that connections exist

**What's removed:** In-process spans, detailed attributes, duplicate exit spans, count of repeated calls

### Key insight for troubleshooting

Even with essential or compact granularity on non-business-critical services, you can still:
- See the complete request flow across your entire system
- Identify which service is slow (duration is preserved)
- Spot errors (error flags are preserved)
- Understand service dependencies (connectivity is preserved)
- Navigate the Dynamic Flow Map (entity relationships are intact)

You just won't have the detailed "how" for non-business-critical services—which is often acceptable for supporting services that aren't directly involved in customer transactions.

## Hybrid sampling strategy [#hybrid-sampling]

You can enable **both full and partial granularity tracing** on the same application simultaneously. This powerful approach lets you:

* Sample a percentage of traces at full granularity for detailed troubleshooting
* Sample the remaining traces at partial granularity for controlled ingestion coverage
* Achieve 100% sampling coverage with mixed detail levels

**Example:** Configure your service to capture:
* 10% full granularity traces (complete detail for deep analysis)
* 90% partial granularity traces (reduced, essential, or compact)
* Result: 100% trace coverage with controlled data ingestion

This hybrid approach is ideal when you need occasional detailed traces but can't justify full granularity for all requests. See your agent's configuration documentation for how to implement sampling strategies alongside partial granularity options.

## Adaptive sampling configuration [#adaptive-sampling]

Partial granularity distributed traces and adaptive sampling help users ingest data in a controlled manner.

### Enhanced adaptive sampling target configuration

Core tracing allows for configuration of the adaptive sampling target for each `root`, `remote_parent_sampled`, and `remote_parent_not_sampled` section. Previously, the adaptive sampling target could only be configured at the global level.

**The problem this solves:**

When the global adaptive sampling target is shared between root and remote parent, if the number of remote parent traces sampled is greater than or equal to the adaptive sampling target, only remote parent traces will be sampled. This happens because remote parent sampling decisions set the priority to greater than 2 and root sampling decisions set the priority to less than 2.

**The solution:**

Configure an adaptive sampling target under `root`, `remote_parent_sampled`, and `remote_parent_not_sampled` to ensure that remote parent sampling decisions do not cause all root spans to be dropped if the sampled remote parent traces exceed the adaptive sampling target threshold.

**Configuration example:**

```yaml
distributed_tracing:
  sampler:
    adaptive_sampling_target: 10  # Global target
    root:
      trace_id_ratio_based:
        ratio: 0.2  # 20% for important traces
    remote_parent_sampled:
      adaptive:
        sampling_target: 20  # Per-section target
```

**Available sampling options:**

- `adaptive` – use adaptive sampling
- `always_on` – always sample
- `always_off` – never sample
- `trace_id_ratio_based` – use ratio/percent based sampling

**Note:** The sampling target budget of 20 won't be used until there is an upstream sampled trace that does not have a trusted account ID.

## Requirements [#requirements]

To use these configuration options, you need:

* **APM agent monitoring:** These configurations only work with New Relic APM agents (Java, .NET, Node.js, Python, Go, Ruby, PHP). They do not apply to OpenTelemetry instrumentation.
* **Distributed tracing enabled:** Your services must already have distributed tracing enabled.
* **Compatible agent versions:** See configuration docs for minimum agent versions.
* **Configuration file access:** You'll modify your agent's configuration file (typically YAML format).

When services with different configurations communicate, trace context propagates correctly—you can mix configuration levels across your architecture (some services with Option 1, others with full tracing, etc.).

**Important limitations:**

* **Infinite tracing is not supported** for partial granularity distributed traces. Partial granularity traces are only supported in standard distributed tracing.
* **Full granularity always takes precedence** over partial granularity when it comes to sampling and reservoir space.
* **OTEL instrumentation**: When using OTEL bridge, spans generated by OTEL instrumentation will also be subject to the same rules of partial granularity tracing.

**Note:** You can also adjust adaptive sampling rates independently of these options to further control how many traces are captured per minute.
