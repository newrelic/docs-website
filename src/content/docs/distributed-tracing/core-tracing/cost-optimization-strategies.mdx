This guide helps you plan and implement cost-saving distributed trace sampling strategies across your services. You'll learn how to prioritize services, calculate potential impact, and roll out configurations to achieve controlled data ingestion.

## Understand your data ingestion [#understand-costs]

Distributed tracing data volumes come from two main factors:

### Ingest costs

You're charged for the data volume (GB) of span data ingested. Span data is controlled by:
* **Sampling rate**: Controlled by [sampling configuration](/docs/distributed-tracing/core-tracing/configure-sampling), which determines the rate of distributed traces captured
* **Partial granularity mode** (reduced, essential, or compact): Controls both the number of spans and the size of each span through combined span filtering and attribute reduction. See [Configure partial granularity](/docs/distributed-tracing/core-tracing/configure-partial-granularity-tracing) for details.

### Compute (CCU) costs

Querying and visualizing trace data consumes compute units. While span configuration primarily addresses ingest costs, the volume of data to query impacts CCU costs.

### Identify services where cost can be reduced

1. Identify all accounts where distributed tracing data is being reported
2. Note current span data volume per service, in each account:
   ```sql
   SELECT bytecountestimate() / 1e9 as 'Daily GB', count(*) as 'Span Count'
   FROM Span
   FACET entity.name
   LIMIT MAX
   SINCE 1 day ago
   ```

   **Note:** `bytecountestimate()` provides an estimate for planning purposes. For monitoring actual consumption, consider using other metrics that track actual data ingest.

3. Identify services which are reporting a high volume of distributed tracing data, and match against the criteria below to decide which to target for reduction

    Not all services need the same level of tracing detail. Choose your configuration based on how frequently you need detailed trace information, whether you rely on custom attributes, and whether you use AI monitoring (LLM) data:

    ***Sampling***
    * ***Use adaptive sampling to control trace sample rates***
      It's important to capture a good baseline of distributed traces end-to-end across your critical services.
      While the [trace ID ratio based sampling](/docs/distributed-tracing/core-tracing/configure-sampling#trace-id-ratio-based-sampler) guarantees a set percent of transactions will be sampled, the volume of trace data will increase with your throughput.
      [Adaptive sampling](/docs/distributed-tracing/core-tracing/configure-sampling#adaptive-sampler) ensures a specified number of traces per minute from each service, regardless of how throughput changes.

    * ***Use trace ID ratio based sampling to control sample rates based on upstream sampling decisions***
      Use [trace ID ratio based sampling](/docs/distributed-tracing/core-tracing/configure-sampling#trace-id-ratio-based-sampler) to capture a percentage of throughput sampled by upstream services. This may be useful when you cannot limit the volume of traces sampled by upstream services, such as when:
      * An upstream service samples a large portion of its transactions.
      * You're seeing many distributed traces from upstream services, but not from the current service, and you don't want to increase the overall volume of trace data you're capturing.

    ***Distributed tracing granularity***
      ***Use a hybrid approach of full and partial granularity distributed tracing to decrease cost per transaction***
      When a service is:
      * You are satisfied with the rate of distributed trace sampling, but only need some distributed traces with full granularity details for debug.
      * Not able to capture enough distributed traces to see a complete service map due to filling up the span reservoir too quickly.

      ***Use only partial granularity distributed tracing to decrease cost per transaction***
      When a service is:
      * A connecting service where you only need to see service-to-service visibility and can tolerate losing internal function call details.
      * Supporting where you mainly need to see "what calls what" and rarely need detailed diagnostics.

    To determine which partial granularity type to use, see [understanding partial granularity tracing modes](/docs/distributed-tracing/core-tracing/#modes)

    <Callout variant="tip">
      **Decision checklist:** Before choosing a configuration:
      1. How often do you need method-level detail for this service?
      2. Does your team rely on custom attributes for monitoring or troubleshooting this service?
      3. Does this service use AI monitoring to track LLM interactions? (If yes, can't use compact mode)
      4. How critical is this service to your business operations?
    </Callout>

    <Callout variant="warning">
      Dynamic configuration changes during active incidents may not be practical due to restart requirements. It's better to have appropriate configurations in place proactively rather than adjusting them reactively during incidents.
    </Callout>

4. Prioritize configuration rollout starting with supporting services (lowest risk)

## Rollout approach [#rollout]

Rollout configuration changes gradually to minimize risk

After the new configurations are rolled out for a service, you can query the change in span data volume with the following query:
   ```sql
   SELECT bytecountestimate() / 1e9 as 'Daily GB', count(*) as 'Span Count'
   FROM Span
   WHERE entity.name = 'YOUR SERVICE NAME'
   SINCE 1 day ago COMPARE WITH 1 day ago
   ```

<Callout variant="tip">
  Don't rush the rollout. Span configuration is about smart trade-offs, not blind cost-cutting.

  What not to sacrifice:

  * **Error visibility:** Always capture errors, even with Options 2-3
  * **Critical paths:** Keep full DT on customer-facing transactions
  * **Regulatory requirements:** Some industries require detailed audit trails
  * **Active incidents:** Temporarily increase detail when troubleshooting
</Callout>

## Track your ingestion impact [#track]

Monitor the effectiveness of your partial granularity distributed traces implementation:

### Daily ingest tracking

```sql
SELECT bytecountestimate() / 1e9 as 'Daily GB'
FROM Span
SINCE 7 days ago
TIMESERIES 1 day
```

### Span count by configuration

```sql
SELECT count(*) as 'Span Count'
FROM Span
SINCE 7 days ago
TIMESERIES 1 day
```
