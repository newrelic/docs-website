This guide shows you how to configure partial granularity distributed tracing for your APM agents. Partial granularity offers three modes—reduced, essential, and compact—that control how much trace data each service sends while maintaining complete end-to-end traces and Dynamic Flow Maps.

**Prerequisites**:
1. First, review [Control distributed tracing data volume](/docs/distributed-tracing/core-tracing/introduction-core-tracing) to understand the three configuration options and when to use them
2. Understand your service's role in your architecture (business-critical vs supporting service)
3. Have permission to modify agent configuration files or environment variables

## Before you begin [#before-you-begin]

Make sure you have:

* **APM agent monitoring:** Partial granularity works only with New Relic APM agents (Java, Node.js, Python)—not OpenTelemetry
* [Distributed tracing enabled](/docs/distributed-tracing/concepts/quick-start) on your services
* Agent versions that support partial granularity (see [requirements](#requirements) below)
* Permission to modify agent configuration files or environment variables

## Understanding the three modes [#modes]

Partial granularity distributed tracing offers three modes. You choose one mode per service—there's no fine-grained control within each mode:

* **Reduced granularity:** Removes in-process spans, keeps entry and exit spans with all attributes
* **Essential granularity:** Removes in-process spans + reduces attributes to minimal set (connectivity only)
* **Compact granularity:** Removes in-process spans + reduces attributes + compresses duplicate exit spans

Each mode is controlled entirely through configuration—span filtering and attribute reduction are combined and automatic within each mode. You cannot mix features from different modes or customize which attributes are included beyond choosing one of these three modes.

## Enabling full and partial granularity together [#hybrid]

You can enable **both full granularity and partial granularity** on the same service simultaneously. This hybrid approach allows you to:

* Sample a percentage of traces at full granularity for detailed troubleshooting
* Sample the remaining traces at partial granularity for controlled ingestion
* Achieve 100% trace sampling coverage with mixed detail levels

**Example hybrid configuration:**

Configure your service to sample:
* 10% of traces at full granularity (complete detail for deep analysis)
* 90% of traces at partial granularity (reduced, essential, or compact)
* Result: 100% trace coverage with controlled data ingestion

This is configured through your agent's sampler settings alongside partial granularity. See [Sampler configuration examples](/docs/distributed-tracing/core-tracing/sampler-configuration-examples) for detailed hybrid sampling configurations and [Hybrid sampling strategy](/docs/distributed-tracing/core-tracing/introduction-core-tracing#hybrid-sampling) for when to use this approach.

## What data gets captured with each mode [#what-data]

Each partial granularity mode captures different amounts of data. Choose based on your needs:

### Reduced granularity

**Spans captured:** Entry spans (incoming requests) and exit spans (calls to databases, external services, etc.). In-process spans are filtered out.

**Attributes:** All attributes are preserved, including custom attributes.

**Best for:** Services where you need full attribute visibility but don't need internal method-level execution detail.

### Essential granularity

**Spans captured:** Entry spans and exit spans only. In-process spans are filtered out.

**Attributes:** Only minimal attributes for trace connectivity:
* **Trace connectivity:** `trace.id`, `span.id`, `parent.id` to link spans together
* **Service identification:** `entity.name`, `service.name` to identify which service owns each span
* **Timing data:** `timestamp`, `duration` for performance analysis
* **Error tracking:** `error` flag to identify failures
* **HTTP status:** `http.status_code` for request outcome

**Attributes excluded:**
* Custom/user-defined attributes
* Detailed HTTP information (URLs, headers, query parameters)
* Database query details (SQL statements, connection strings)
* Detailed error information (stack traces, error messages)
* Additional performance metadata

**Best for:** Supporting services where you need connectivity but not detailed diagnostics.

<Callout variant="important">
  **Custom and user-defined attributes are automatically dropped** in essential and compact modes. If your team relies on custom attributes for troubleshooting or monitoring, use **reduced granularity** or keep those services at **full granularity**. See [What about custom attributes?](#custom-attributes) for details.
</Callout>

### Compact granularity

**Spans captured:** Entry spans only, plus the first exit span to each unique destination. Duplicate exit spans to the same destination are compressed.

**Attributes:** Same minimal attribute set as essential granularity.

**Best for:** High-volume services or non-production environments where you need proof of connectivity but not call frequency or detailed diagnostics.

**The key principle across all modes:** Maintain complete end-to-end traces and service connectivity while controlling data volume based on your needs.

## Configure partial granularity [#configure]

Choose your agent language to see configuration instructions. You must enable partial granularity and specify which mode (reduced, essential, or compact) you want to use.

<Callout variant="important">
  You can enable **both** full granularity and partial granularity sampling simultaneously. For example, sample 10% at full granularity and 90% at partial granularity for complete coverage with controlled ingestion. See [Hybrid sampling strategy](/docs/distributed-tracing/core-tracing/introduction-core-tracing#hybrid-sampling) for details.
</Callout>

<CollapserGroup>
  <Collapser
    id="java"
    title="Java agent"
  >
    To configure partial granularity for your Java application, add the sampler configuration to your `newrelic.yml` file.

    **Configuration file approach:**

    1. Open your `newrelic.yml` file.
    2. Under the `distributed_tracing` section, add the `sampler.partial_granularity` configuration:

    ```yml
    distributed_tracing:
      enabled: true
      sampler:
        partial_granularity:
          enabled: true
          type: essential  # Options: reduced, essential, compact
    ```

    3. Save the file and restart your Java application.

    **Environment variable approach:**

    ```bash
    export NEW_RELIC_DISTRIBUTED_TRACING_SAMPLER_PARTIAL_GRANULARITY_ENABLED=true
    export NEW_RELIC_DISTRIBUTED_TRACING_SAMPLER_PARTIAL_GRANULARITY_TYPE=essential
    ```

  </Collapser>

  <Collapser
    id="nodejs"
    title="Node.js agent"
  >
    To configure partial granularity for your Node.js application, add the sampler configuration to your `newrelic.js` file.

    **Configuration file approach:**

    1. Open your `newrelic.js` file.
    2. Add the sampler configuration:

    ```js
    'use strict'

    exports.config = {
      distributed_tracing: {
        enabled: true,
        sampler: {
          partial_granularity: {
            enabled: true,
            type: 'essential'  // Options: 'reduced', 'essential', 'compact'
          }
        }
      }
    }
    ```

    3. Save the file and restart your Node.js application.

    **Environment variable approach:**

    ```bash
    export NEW_RELIC_DISTRIBUTED_TRACING_SAMPLER_PARTIAL_GRANULARITY_ENABLED=true
    export NEW_RELIC_DISTRIBUTED_TRACING_SAMPLER_PARTIAL_GRANULARITY_TYPE=essential
    ```

  </Collapser>

  <Collapser

    To configure partial granularity for your Python application, add the sampler configuration to your `newrelic.ini` file.

    **Configuration file approach:**

    1. Open your `newrelic.ini` file.
    2. Add the settings under the `[newrelic]` section:


    3. Save the file and restart your Python application.

    **Environment variable approach:**

    ```bash
    export NEW_RELIC_DISTRIBUTED_TRACING_SAMPLER_PARTIAL_GRANULARITY_ENABLED=true
    export NEW_RELIC_DISTRIBUTED_TRACING_SAMPLER_PARTIAL_GRANULARITY_TYPE=essential
    ```

## What about custom attributes? [#custom-attributes]

Custom attributes behave differently depending on which mode you choose:

* **Reduced granularity:** All attributes are preserved, including custom attributes. This mode only filters spans, not attributes.
* **Essential granularity:** Custom attributes are automatically dropped. Only the minimal attribute set needed for trace connectivity is preserved.
* **Compact granularity:** Custom attributes are automatically dropped. Only the minimal attribute set needed for trace connectivity is preserved.

<Callout variant="important">
  There is **no way to opt in custom attributes** for essential or compact modes. If your team relies on specific custom attributes for troubleshooting or monitoring, you must either:
  - Use **reduced granularity** instead (which keeps all attributes but filters spans)
  - Keep those services at **full granularity**
  - Accept that you won't have access to those custom attributes for troubleshooting
</Callout>

The three modes are fixed—you cannot customize which attributes are included beyond choosing one of the three modes. This is by design to keep the feature simple and predictable.

## Verify partial granularity is working [#verify]

After configuring partial granularity, use supportability metrics to verify it's working correctly.

### Check if partial granularity spans are being created

Use this query to check how many spans are being instrumented vs. kept for each mode. Replace `MODE` with your chosen mode (`Reduced`, `Essential`, or `Compact`):

```sql
SELECT
  count(newrelic.timeslice.value) AS 'Spans Kept',
  filter(count(newrelic.timeslice.value),
    WHERE metricTimesliceName LIKE 'Supportability/DistributedTrace/PartialGranularity/MODE/Span/Instrumented') AS 'Spans Instrumented'
FROM Metric
WHERE metricTimesliceName LIKE 'Supportability/DistributedTrace/PartialGranularity/MODE/%'
AND entity.name = 'YOUR_SERVICE_NAME'
SINCE 30 minutes ago
```

**Example for essential mode:**

```sql
SELECT
  count(newrelic.timeslice.value) AS 'Spans Kept',
  filter(count(newrelic.timeslice.value),
    WHERE metricTimesliceName LIKE 'Supportability/DistributedTrace/PartialGranularity/Essential/Span/Instrumented') AS 'Spans Instrumented'
FROM Metric
WHERE metricTimesliceName LIKE 'Supportability/DistributedTrace/PartialGranularity/Essential/%'
AND entity.name = 'YOUR_SERVICE_NAME'
SINCE 30 minutes ago
```

If partial granularity is working, you should see "Spans Instrumented" > "Spans Kept", indicating that spans are being filtered.

### Verify trace completeness

Ensure Dynamic Flow Maps and trace visualization still work:

1. Go to **[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > APM & services > Service map**.
2. Verify your services with partial granularity appear correctly in the map.
3. Check that service-to-service connections are visible.
4. Click on a service to view distributed tracing and confirm traces are complete end-to-end.

## Troubleshoot issues [#troubleshoot]

### Partial granularity not taking effect

**Problem:** You configured partial granularity but spans aren't being filtered.

**Solutions:**

1. Restart your application after configuration changes.
2. Check agent logs for configuration errors.
3. Verify your agent version supports partial granularity (see [requirements](#requirements)).
4. Confirm syntax matches your agent's configuration format exactly.
5. Verify the `partial_granularity.enabled` is set to `true`.
6. Check that you specified a valid type (`reduced`, `essential`, or `compact`).

### Supportability metrics not showing data

**Problem:** You can't see supportability metrics for partial granularity.

**Solutions:**

1. Wait at least 5-10 minutes after restarting the application for metrics to appear.
2. Verify the service is actively receiving traffic and creating spans.
3. Check that your agent version supports partial granularity.
4. Confirm the configuration syntax is correct for your agent language.

### Ingest volume didn't decrease as expected

**Problem:** Partial granularity is enabled but ingest volume hasn't decreased significantly.

**Solutions:**

1. Check supportability metrics to verify spans are being filtered (see [verification](#verify)).
2. Ensure you're using `essential` or `compact` mode (not just `reduced`).
3. Consider that results vary by application based on span structure.
4. Allow 24-48 hours for ingest metrics to stabilize before evaluating impact.
