This comprehensive guide shows you how to configure partial granularity distributed tracing for your APM agents. Partial granularity offers three modes — reduced, essential, and compact—that control how much span and attribute data each service sends while maintaining complete end-to-end traces and Dynamic Flow Maps. You'll learn about the three partial granularity tracing modes and configuration options, advanced sampling strategies, and how to verify everything is working correctly.

**Prerequisites**:
1. First, review [Control distributed tracing data volume](/docs/distributed-tracing/core-tracing/introduction-core-tracing) to understand the three configuration options and when to use them
2. Understand your service's role in your architecture (business-critical vs supporting service)
3. Have permission to modify agent configuration files or environment variables

## Understanding span types [#span-types]

To understand partial granularity distributed tracing, you need to know about the types of spans in distributed tracing:

### Entry spans

The first span in a process or transaction. This represents the incoming request to your service.

**Example:** An HTTP request arriving at your API service.

### Exit spans

Spans that represent calls from your service to downstream services or external resources.

**Examples:**
* HTTP calls to other microservices
* Database queries (SQL, NoSQL)
* Cache operations (Redis, Memcached)
* External API calls
* Message queue operations

### In-process spans

Spans that represent internal method calls within your service. These show the detailed execution path inside your code.

**Examples:**
* Individual function calls
* Web framework middleware
* Internal service layers
* Business logic methods

### LLM/AI monitoring spans

If your application uses New Relic AI monitoring to track LLM (Large Language Model) interactions, then it reports LLM spans, typically beginning with "Llm/".

## Understanding the three modes [#modes]

Partial granularity distributed tracing offers three modes. You choose one mode per service—there's no fine-grained control within each mode.
**The key principle across all modes:** Maintain complete end-to-end traces and service connectivity while controlling data volume based on your needs.

### Reduced granularity
* Omits in-process spans - keeps entry, exit, and LLM spans with all attributes.

**Best for:** Services where you need full attribute visibility but don't need internal method-level execution detail.

### Essential granularity
* Omits in-process spans - keeps entry, exit, and LLM spans.
* Reduces attributes to only those required for service connections, and tags.

**Best for:** Supporting services where you need connectivity but not detailed diagnostics.

<Callout variant="important">
  **Custom and user-defined attributes are automatically omitted** in essential and compact modes. If your team relies on custom attributes for troubleshooting or monitoring, use **reduced granularity** or keep those services at **full granularity**. See [What about custom attributes?](#custom-attributes) for details.
</Callout>

### Compact granularity
* Omits in-process spans - keeps entry, exit, and LLM spans.
* Reduces attributes to only those required for service connections, and tags.
* Compresses exit spans for the same called URL into a single exit span per URL.
* NOT compatible with AI monitoring — LLM spans are not viewable on the AI dashboard.

**Best for:** High-volume services or non-production environments where you need proof of connectivity but not call frequency or detailed diagnostics.

<Callout variant="important">
  **Compact mode is not compatible with AI monitoring (LLM) data.** If your application uses New Relic AI monitoring to track LLM interactions, you cannot use partial granularity in compact mode. Use reduced or essential granularity instead to maintain AI monitoring functionality.
</Callout>

The mode can be configured via `distributed_tracing.sampler.partial_granularity.type` or the NEW_RELIC_DISTRIBUTED_TRACING_SAMPLER_PARTIAL_GRANULARITY_TYPE environment variable. See each APM agent's configuration documentation for details.

## Enabling full and partial granularity together [#hybrid]

You can enable **both full granularity and partial granularity** on the same service simultaneously. This hybrid approach allows you to:

* Sample a percentage of traces at full granularity for detailed troubleshooting
* Sample the remaining traces at partial granularity for controlled ingestion and overall service health
* Achieve up to 100% trace sampling coverage with mixed detail levels

## Configure partial granularity [#configure]

Choose your agent language to see configuration instructions for partial granularity tracing.
Each of these optoins configure partial granularity distribtued tracing without disabling full granularity tracing. See [sampler configuration examples](/docs/distributed-tracing/core-tracing/sampler-configuration-examples#example-three) for an example running only partial granularity distributed tracing.

<CollapserGroup>
  <Collapser
    id="java"
    title="Java agent"
  >
    To configure partial granularity for your Java application, add the sampler configuration to your `newrelic.yml` file.

    **Configuration file approach:**

    1. Open your `newrelic.yml` file.
    2. Under the `distributed_tracing` section, add the `sampler.partial_granularity` configuration:

    ```yml
    distributed_tracing:
      enabled: true
      sampler:
        partial_granularity:
          enabled: true
          type: essential  # Options: reduced, essential, compact
    ```

    3. Save the file and restart your Java application.

    **Environment variable approach:**

    ```bash
    export NEW_RELIC_DISTRIBUTED_TRACING_SAMPLER_PARTIAL_GRANULARITY_ENABLED=true
    export NEW_RELIC_DISTRIBUTED_TRACING_SAMPLER_PARTIAL_GRANULARITY_TYPE=essential
    ```

  </Collapser>

  <Collapser
    id="nodejs"
    title="Node.js agent"
  >
    To configure partial granularity for your Node.js application, add the sampler configuration to your `newrelic.js` file.

    **Configuration file approach:**

    1. Open your `newrelic.js` file.
    2. Add the sampler configuration:

    ```js
    'use strict'

    exports.config = {
      distributed_tracing: {
        enabled: true,
        sampler: {
          partial_granularity: {
            enabled: true,
            type: 'essential'  // Options: 'reduced', 'essential', 'compact'
          }
        }
      }
    }
    ```

    3. Save the file and restart your Node.js application.

    **Environment variable approach:**

    ```bash
    export NEW_RELIC_DISTRIBUTED_TRACING_SAMPLER_PARTIAL_GRANULARITY_ENABLED=true
    export NEW_RELIC_DISTRIBUTED_TRACING_SAMPLER_PARTIAL_GRANULARITY_TYPE=essential
    ```

## Advanced configuration examples [#advanced-config]

For more control over which distributed traces are captured, you can configure sampling alongside partial granularity.

<Callout variant="tip">
  For deep technical documentation about samplers, including sampler types like adaptive and trace id ratio based, adaptive sampling targets, and sampling based on upstream sampling decisions, see [Configure sampling](/docs/distributed-tracing/core-tracing/configure-sampling). Also see [Sampler Configuration Examples](/docs/distributed-tracing/core-tracing/sampler-configuration-examples) for practical configuration examples.
</Callout>

### Sampling decision contexts

The configuration uses these terms from the OpenTelemetry and Hybrid agent spec:

* **`root`**: Defines how to sample when the trace originates from the current service
* **`remote_parent_sampled`**: Defines how to sample when the trace is already sampled by the upstream service
* **`remote_parent_not_sampled`**: Defines how to sample when the trace is not sampled by the upstream service

**Available sampler types:**

* `adaptive` – Use adaptive sampling with a configurable sampling target
* `always_on` – Always sample
* `always_off` – Never sample
* `trace_id_ratio_based` – Use ratio/percent-based sampling (requires `ratio` parameter)

**Priority system:** If the span reservoir is exceeded, full granularity transactions are kept over partial granularity transactions. Similarly, traces that the upstream service sampled are kept over traces that originated from the current service or were not sampled by the upstream service.

## Special handling for ratio-based sampling [#ratio-handling]

When using `trace_id_ratio_based` sampling in both full and partial granularity configurations, be aware of how the trace ID-based sampler works.

**The challenge:** If full granularity is configured to a ratio of 10% and partial granularity is also configured to a ratio of 10%, only full granularity traces will be captured. This is because the trace ID determines the sampling decision, and the same trace IDs would be selected for both configurations.

**How the agent solves this:**

When a section (`root`, `remote_parent_sampled`, or `remote_parent_not_sampled`) is configured to use ratio-based sampling for both full and partial granularity, the agent automatically adjusts the partial granularity sampler's ratio to be the **sum** of both ratios.

**Example:**
* Full granularity ratio: 10%
* Partial granularity ratio: 10%
* **Agent automatically sets partial granularity ratio to: 20%**

This ensures both full and partial granularity traces are captured as intended without overlap.

## Verify partial granularity is working [#verify]

After configuring partial granularity, use supportability metrics to verify it's working correctly.

### Check if partial granularity spans are being created

Use this query to check how many spans are being instrumented vs. kept for each mode. Replace `MODE` with your chosen mode (`Reduced`, `Essential`, or `Compact`):

```sql
SELECT
  filter(count(newrelic.timeslice.value),
    WHERE metricTimesliceName LIKE 'Supportability/DistributedTrace/PartialGranularity/%/Span/Kept') AS 'Spans Kept',
  filter(count(newrelic.timeslice.value),
    WHERE metricTimesliceName LIKE 'Supportability/DistributedTrace/PartialGranularity/%/Span/Instrumented') AS 'Spans Instrumented'
FROM Metric
WHERE entity.name = 'YOUR_SERVICE_NAME'
SINCE 30 minutes ago
```

If partial granularity is working, you should see "Spans Instrumented" > "Spans Kept", indicating that spans are being filtered.

### Verify trace completeness

Ensure that trace details shows the expected level of detail and connectivity:

1. Go to distribtued tracing for your APM service.
2. Open a distributed trace for this service.
3. Confirm that only the expected spans were reported for that service.
4. Confirm that it's dependencies are correct in the map view above the trace.

## Troubleshooting [#troubleshoot]

### Partial granularity not taking effect

**Problem:** You configured partial granularity but spans aren't being filtered.

**Solutions:**

1. Verify your agent version supports partial granularity.
2. Verify the `partial_granularity.enabled` is set to `true`.
3. Check that you specified a valid type (`reduced`, `essential`, or `compact`).
4. Confirm syntax matches your agent's configuration format exactly.
5. Restart your application after configuration changes.
6. Check agent logs for configuration errors.
