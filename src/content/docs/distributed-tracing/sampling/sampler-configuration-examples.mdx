While the default sampling configuration works for most services, you may need to customize sampling behavior for specific scenarios. This guide provides real-world examples of sampler configurations with copy-paste ready code for different service needs.

**Prerequisites**:
1. Review [cost optimization strategies](/docs/distributed-tracing/core-tracing/cost-optimization-strategies)
2. Review [configure sampling](/docs/distributed-tracing/core-tracing/configure-sampling)
3. Review [configure partial granularity tracing](/docs/distributed-tracing/core-tracing/configure-partial-granularity-tracing)

## Example 1: Fix poor endpoint distribution caused by upstream sampling [#example-upstream]

### Problem

With the default configuration that uses the adaptive sampler, upstream services that sample a trace will cause your service to also sample that trace. This can lead to one endpoint being over-sampled while other endpoints are under-sampled.

**Symptoms:**
- Distributed tracing dashboard shows 90% of traces from one endpoint
- Other important endpoints rarely appear in traces
- Your service's span reservoir fills up too quickly with upstream-sampled traces

### Solution 1: Reduce upstream sampling rate

Sample only 10% of transactions that were already sampled by upstream services, leaving more reservoir space for your service's endpoints that do not have an upstream.

**Java (newrelic.yml):**
```yaml
distributed_tracing:
  enabled: true
  sampler:
    remote_parent_sampled:
      trace_id_ratio_based:
        ratio: 0.1  # Sample 10% of upstream-sampled traces
```

**Node.js (newrelic.js):**
```javascript
distributed_tracing: {
  enabled: true,
  sampler: {
    remote_parent_sampled: {
      trace_id_ratio_based: {
        ratio: 0.1  // Sample 10% of upstream-sampled traces
      }
    }
  }
}
```

**Python (newrelic.ini):**
```ini
distributed_tracing.enabled = true
distributed_tracing.sampler.remote_parent_sampled.trace_id_ratio_based.ratio = 0.1
```

**Result:** Better distribution across all your service's endpoints while still capturing some upstream-initiated traces.

### Solution 2: Use hybrid sampling for upstream traces

Capture 5% of upstream-sampled traces at full granularity and the remaining 95% at partial granularity. This keeps distributed traces complete while preserving reservoir space for distribtued traces that original from your service.

**Java (newrelic.yml):**
```yaml
distributed_tracing:
  enabled: true
  sampler:
    remote_parent_sampled:
      trace_id_ratio_based:
        ratio: 0.05  # Only 5% get full granularity
  partial_granularity:
    enabled: true
    type: essential  # Use "essential" if capturing AI data, "compact" otherwise
    remote_parent_sampled: always_on  # Rest get partial granularity
```

**Node.js (newrelic.js):**
```javascript
distributed_tracing: {
  enabled: true,
  sampler: {
    remote_parent_sampled: {
      trace_id_ratio_based: {
        ratio: 0.05  // Only 5% get full granularity
      }
    }
  },
  partial_granularity: {
    enabled: true,
    type: 'essential',  // Use 'essential' if capturing AI data, 'compact' otherwise
    remote_parent_sampled: 'always_on'  // Rest get partial granularity
  }
}
```

**Python (newrelic.ini):**
```ini
distributed_tracing.enabled = true
distributed_tracing.sampler.remote_parent_sampled.trace_id_ratio_based.ratio = 0.05
distributed_tracing.partial_granularity.enabled = true
distributed_tracing.partial_granularity.type = essential
distributed_tracing.partial_granularity.remote_parent_sampled = always_on
```

**Result:** 100% of distributed traces sampled upstream will be captured, with controlled data volume - 5% full granularity + 95% partial granularity distrihbuted traces.

## Example 2: Capture rarely-called endpoints [#example-rare-endpoints]

### Problem

The default adaptive sampler captures ~10 distributed traces per minute. Due to random probability, frequently-called endpoints dominate your distributed traces while rarely-called endpoints almost never appear.

**Symptoms:**
- Endpoints that are requested infrequently, rarely show up in the distributed traces UI

### Solution: 100% partial granularity sampling

Capture every transaction originating from this service using partial granularity, dramatically increasing the chance of capturing rare endpoints.

**Java (newrelic.yml):**
```yaml
distributed_tracing:
  enabled: true
  sampler:
    full_granularity: 
      enabled: false
    partial_granularity:
      enabled: true
      type: compact  # Use "essential" if you have AI monitoring
      root: always_on
      remote_parent_sampled: always_on
      remote_parent_not_sampled: always_off
```

**Node.js (newrelic.js):**
```javascript
distributed_tracing: {
  enabled: true,
  sampler: {
    full_granularity: {
      enabled: false
    },
    root: 'always_on',
    remote_parent_sampled: 'always_on',
    remote_parent_not_sampled: 'always_off'
  }
}
```

**Python (newrelic.ini):**
```ini
distributed_tracing.enabled = true
distributed_tracing.sampler.full_granularity.enabled = true
distributed_tracing.sampler.partial_granularity.type = compact
distributed_tracing.sampler.partial_granularity.root = always_on
distributed_tracing.sampler.partial_granularity.remote_parent_sampled = always_on
distributed_tracing.sampler.partial_granularity.remote_parent_not_sampled = always_off
```
**Result:** Complete endpoint coverage with manageable data volume. All endpoints appear in the distributed trace UI, not just the busiest ones.

<Callout variant="tip">
  **When to use this:** Supporting services, background job processors, admin APIs, or any service with highly variable endpoint traffic patterns.
</Callout>

## Example 3: Fix trace fragmentation caused by spans being dropped from the span reservoir [#example-three]

### Problem

Services with hundreds of spans per transaction fill the agent's span reservoir quickly. When the reservoir fills, spans are randomly dropped, creating gaps in distributed traces.

**Symptoms:**
- Distributed traces show "missing" or "unknown" parent spans

### Solution: Minimal span collection with partial granularity

Capture only entry/exit spans (service boundaries) rather than all in-process spans, decreasing the number of spans, thereby preventing reservoir overflow.

**Java (newrelic.yml):**
```yaml
distributed_tracing:
  enabled: true
  sampler:
    full_granularity:
      enabled: false  # Disable full granularity by default
  partial_granularity:
    enabled: true
    type: compact  # Use "essential" if you have AI monitoring
    root: adaptive
    remote_parent_sampled: always_on  # Always capture upstream-sampled traces
```

**Node.js (newrelic.js):**
```javascript
distributed_tracing: {
  enabled: true,
  sampler: {
    full_granularity: {
      enabled: false  // Disable full granularity by default
    }
  },
  partial_granularity: {
    enabled: true,
    type: 'compact',  // Use 'essential' if you have AI monitoring
    root: 'adaptive',
    remote_parent_sampled: 'always_on'  // Always capture upstream-sampled traces
  }
}
```

**Python (newrelic.ini):**
```ini
distributed_tracing.enabled = true
distributed_tracing.sampler.full_granularity.enabled = false
distributed_tracing.partial_granularity.enabled = true
distributed_tracing.partial_granularity.type = compact
distributed_tracing.partial_granularity.root.adaptive = true
distributed_tracing.partial_granularity.remote_parent_sampled = always_on
```

**Result:** Complete trace paths with no gaps.

<Callout variant="important">
  **Configuration changes require application restart.** This approach works best for services with predictable span patterns, not for dynamic incident response.
</Callout>

## Example 4: Recommended hybrid agent configuration [#example-hybrid-agent]

### Use case

You're using the [APM Hybrid Agent](/docs/distributed-tracing/concepts/how-new-relic-distributed-tracing-works.mdx#high-fidelity-&-hybrid -environments) to send OTEL data to New Relic, and you want to use OTEL sampling configurations.

### Configuration

Sample distributed traces that originate in this service at 10% based on their trace ID so sampling decisions are consistent across services, always sample when upstream sampled, never sample when upstream didn't sample.

**Java (newrelic.yml):**
```yaml
distributed_tracing:
  enabled: true
  sampler:
    root:
      trace_id_ratio_based:
        ratio: 0.1  # 10% of transactions originating here
    remote_parent_sampled: always_on  # Follow upstream sampling decisions
    remote_parent_not_sampled: always_off  # Don't sample if upstream didn't
```

**Node.js (newrelic.js):**
```javascript
distributed_tracing: {
  enabled: true,
  sampler: {
    root: {
      trace_id_ratio_based: {
        ratio: 0.1  // 10% of transactions originating here
      }
    },
    remote_parent_sampled: 'always_on',  // Follow upstream sampling decisions
    remote_parent_not_sampled: 'always_off'  // Don't sample if upstream didn't
  }
}
```

**Python (newrelic.ini):**
```ini
distributed_tracing.enabled = true
distributed_tracing.sampler.root.trace_id_ratio_based.ratio = 0.1
distributed_tracing.sampler.remote_parent_sampled = always_on
distributed_tracing.sampler.remote_parent_not_sampled = always_off
```

**Result:** Consistent sampling decisions across your distributed system. Distributed traces either fully capture all services or none at all.

## Example 5: Combined approach for microservices architecture [#example-microservices]

### Use case

Large microservices architecture with:
- 5 customer-facing services (need detailed distributed traces with the sample rate independent of traffic rate flucuations)
- 40 internal services (need 100% sampling for internal transactions, but full granularity is only needed for debugging some services)
- Variable traffic patterns across services

### Configuration strategy

**Customer-facing services:**
```yaml
# Full granularity with adaptive sampling
distributed_tracing:
  enabled: true
  sampler:
    root:
      adaptive:
        sampling_target: 20  # 20 traces per minute
    remote_parent_sampled: always_on
    remote_parent_not_sampled: always_off
```

**Internal services (need connectivity only):**
```yaml
# 100% partial granularity to ensure coverage
distributed_tracing:
  enabled: true
  full_granularity:
    enabled: false
  partial_granularity:
    enabled: true
    type: compact # Use essential to capture AI related spans
    root: always_on
    remote_parent_sampled: always_on
    remote_parent_not_sampled: always_off
```

**Internal services (need some full granularity traces for debugging):**
```yaml
# Hybrid approach: 10% full + 90% partial
distributed_tracing:
  enabled: true
  sampler:
    root:
      trace_id_ratio_based:
        ratio: 0.1
    remote_parent_sampled:
      trace_id_ratio_based:
        ratio: 0.1
    remote_parent_not_sampled: always_off
  partial_granularity:
    enabled: true
    type: compact
    root:
      trace_id_ratio_based:
        ratio: 0.9
    remote_parent_sampled: always_on
    remote_parent_not_sampled: always_off
```

**Result:** Complete service map coverage with controlled data volume. Critical services have full detail while supporting services provide connectivity data.
