You can access and use <DNT>Performance Risks inbox</DNT> to proactively identify and resolve application performance issues before they impact your end users. This guide will walk you through how to access the <DNT>Performance Risks inbox</DNT>, update the detection thresholds, and utilize the tool effectively within New Relic MCP server. For more information about what <DNT>Performance Risks inbox</DNT> is and its key benefits, refer to [introduction to <DNT>Performance Risks inbox</DNT>](/docs/errors-inbox/performance-risks/introduction).


## Access the <DNT>Performance Risks inbox</DNT> [#access]

You can access the <DNT>Performance Risks inbox</DNT> at account and entity-level:

* [Access at entity-level](#access-entity)
* [Access at account-level](#access-account)


### Access at entity-level [#access-entity]

You can access the <DNT>Performance Risks inbox</DNT> for a specific entity to view performance issues related to that entity and its associated telemetry data. This allows you to focus on the performance of a particular application or service and understand how it may be impacting your end users.

**To access <DNT>Performance Risks inbox</DNT> for an entity:**

1. Go to <DNT>[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > APM & Services</DNT>.
2. Select the required entity.
3. In the entity navigation pane, click <DNT>**Performance Risks**</DNT> in the <DNT>**Triage**</DNT> section.


### Access at account-level [#access-account]

You can also access the <DNT>Performance Risks inbox</DNT> at the account level to get a holistic view of performance issues across all entities within your account. This allows you to identify common performance problems that may be affecting multiple applications or services and prioritize your optimization efforts accordingly. 

**To access <DNT>Performance Risks inbox</DNT> for an account:**

1. Go to <DNT>[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > Errors Inbox</DNT>.
2. In the entity navigation pane, click <DNT>**Performance Risks**</DNT>.

## View and analyze performance risks [#view-analyze]

Once you access <DNT>Performance Risks inbox</DNT>, you can view detected issues in two ways: the <DNT>**Triage**</DNT> view and the <DNT>**Group performance risks**</DNT> view. Both views display the same underlying data but offer different ways to analyze it.

### Triage [#triage]

The <DNT>**Triage**</DNT> view processes and groups similar performance risks together automatically. Similar queries are computed, stored, and merged into a single performance risk group. For example, multiple queries that share the same pattern — such as variations of an `IN` clause — are grouped into one entry rather than appearing as separate items. This makes it easier to identify and prioritize the most impactful problems without reviewing each occurrence individually.

Use the <DNT>**Triage**</DNT> view when you want a pre-grouped summary of performance issues to quickly identify patterns across your application.

### Group performance risks [#group]

The <DNT>**Group performance risks**</DNT> view displays the raw performance risk data without pre-computed groupings. Normalized query representations (such as comma and question mark placeholders) are shown, and you can filter and group the data based on different attributes, such as:

* The user affected by the performance risk
* The service generating the calls
* The API triggering the queries

Use the <DNT>**Group performance risks**</DNT> view when you need to analyze performance risks based on specific attributes that are relevant to your investigation.

## Set the thresholds [#set-thresholds]

<DNT>Performance Risks inbox</DNT> uses configurable thresholds to automatically detect performance anomalies. You can update these thresholds at both the account and entity levels to ensure that the detection criteria align with your specific performance monitoring needs and priorities. These thresholds determine what constitutes a performance risk and help to ensure that you are alerted to the most relevant and actionable performance issues in your applications.

You can set the Performance Risks thresholds at an account or entity level:

* [Account-level thresholds](#account-thresholds)
* [Entity-level thresholds](#entity-thresholds)


### Set the account-level thresholds [#account-thresholds]

Setting thresholds at the account level allows you to establish default performance risk detection criteria that apply across all entities within the account. This is useful for maintaining consistent monitoring standards and ensuring that all applications are evaluated against the same performance benchmarks.

**To set the thresholds at an account level:**

<CollapserGroup>
    <Collapser
        id="new-account-thresholds"
        title="New account-level thresholds"
    >

1. Go to <DNT>**[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > Errors Inbox > Performance Thresholds**.</DNT>.
2. In the **Accounts** tab, click <DNT>**+ New**</DNT>.
3. From the <DNT>**Account**</DNT> dropdown, select the account you want to create a new threshold for.
4. Set the required limits for each analyzer.
5. Click <DNT>**Submit**</DNT> to save.

</Collapser>

<Collapser
        id="edit-account-thresholds"
        title="Edit existing account-level thresholds"
    >


1. Go to <DNT>**[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > Errors Inbox > Performance Thresholds**.</DNT>.
2. In the **Accounts** tab, click the <DNT>**Edit**</DNT> button next to the account you want to update.
3. Set the required limits for each analyzer.
4. Click <DNT>**Submit**</DNT> to save.

<Callout variant="tip">
    When you modify thresholds, your username will be displayed to track who made the changes.

### Set the entity-level thresholds [#entity-thresholds]

Setting thresholds at the entity level allows you to customize performance risk detection criteria for specific applications or services that may have unique performance characteristics or requirements. This is particularly useful for tailoring monitoring to different types of applications, environments, or business priorities.

**To set the thresholds at an entity level:**

You can set the thresholds for a specific entity in one of the following ways:

<CollapserGroup>
    <Collapser
        id="entity-triage"
        title="From the Performance Risks page"
    >

1. Go to <DNT>**[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > APM & Services**.</DNT>.
2. Select the required entity.
3. In the entity navigation pane, click <DNT>**Performance Risks**</DNT> in the <DNT>**Triage**</DNT> section.
3. Click <DNT>**Adjust performance thresholds**</DNT> in the top right corner.
4. Set the required limits for each analyzer.
4. Click <DNT>**Submit**</DNT> to save.

<Callout variant="tip">
    When you modify thresholds, your username will be displayed to track who made the changes.
</Callout>

</Collapser>

<Collapser
        id="entity-page"
        title="From the Performance Thresholds page"
    >

1. Go to <DNT>**[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > Errors Inbox > Performance Thresholds**.</DNT>.
2. In the <DNT>**Entities**</DNT> tab:
  1. From the <DNT>**Account**</DNT> dropdown, select the account of your entity.
  2. From the <DNT>**Entity**</DNT> dropdown, select the entity you want to update.
3. Set the required limits for each analyzer.
4. Click <DNT>**Submit**</DNT> to save.

<Callout variant="tip">
    When you modify thresholds, your username will be displayed to track who made the changes.
</Callout>

</Collapser>

</CollapserGroup>


## Using the <DNT>Performance Risks inbox</DNT> tool in New Relic MCP server [#using]

You can also access the <DNT>Performance Risks inbox</DNT> tool (`list_performance_risks()`) in [New Relic MCP server](/docs/agentic-ai/mcp/overview/). This tool lists all the Performance Risks detected for a given entity, along with their details such as risk type, severity, and occurrence count. You can use this tool to programmatically retrieve performance risk data.
