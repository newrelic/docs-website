Performance Risks inbox creates an intelligent layer that proactively identifies and groups application performance issues, enabling you to detect and resolve problems before they impact your end users. Building on the successful Errors Inbox pattern, Performance Risks inbox leverages New Relic's rich telemetry data and grouping logic to surface actionable insights through a streamlined workflow designed for efficient performance optimization.

## What is Performance Risks inbox? [#what-is-performance-risk-inbox]

Performance Risks inbox transforms how you approach performance monitoring by shifting from reactive firefighting to proactive issue prevention. Instead of waiting for performance problems to escalate into production incidents, Performance Risks inbox helps to automatically detect performance anomalies that could lead to common service interruptions and surfaces them with the context you need to resolve them quickly.
The solution works by:

* **Automatically detecting performance anomalies**: Built-in analyzers continuously monitor your applications for Performance Risks using configurable thresholds.
* **Intelligently grouping related issues**: Similar performance problems are grouped together to simplify analysis and resolution.

## Coverage and scope [#coverage-scope]

Performance Risks inbox provides comprehensive monitoring across your entire application stack:

* APM services
* Browser applications
* Database operations


## Key benefits [#key-benefits]

Performance Risks inbox represents a shift toward intelligent, proactive performance monitoring that helps you maintain optimal application performance while reducing the operational overhead of manual performance investigation.


* **Proactive issue detection**: Identify performance problems before they escalate into production incidents
* **Reduced Mean Time to Resolution (MTTR)**: Intelligent grouping help you focus on the most critical issues first
* **Improved engineering productivity**: Spend less time on reactive firefighting and more time on innovation
* **Better application reliability**: Address Performance Risks before they impact your end users

## How does Performance Risks inbox work? [#how-it-works]

In the current release, Performance Risks inbox uses the following analyzers to detect the most common performance issues:


<CollapserGroup>
    <Collapser
        id="slow"
        title="Slow SQL analyzer"
    >


Slow database queries are often the root cause of application performance issues and can lead to cascading problems across your entire stack. Our analyzer identifies database queries that are taking longer than expected to execute, which can significantly impact application response times and user experience.

**What it can help to detect:**

* SQL queries exceeding performance thresholds
* Database operations that may be causing application bottlenecks
* Queries with degrading performance over time

</Collapser>

<Collapser
        id="n-plus-one"
        title="N+1 query analyzer"
    >


N+1 queries can exponentially increase database load and response times as your data grows, making this a critical performance issue to address early. This analyzer detects the common N+1 query anti-pattern where an application executes N queries instead of one query. This typically occurs in ORM frameworks when loading related data.

**What it can help to detect:**
* Repeated database queries in loops
* Inefficient data loading patterns
* ORM-generated query sequences that could be optimized

</Collapser>

<Collapser
  id="excessive-db"
  title="Excessive DB analyzer"
  >


Excessive database operations can overwhelm your database servers and create performance bottlenecks that affect your entire application. This analyzer monitors for applications that make an unusually high number of database connections or queries within a given timeframe, indicating potential inefficiencies in data access patterns.

**What it can help to detect:**
* Abnormally high database query volumes
* Inefficient batch processing patterns

</Collapser>

<Collapser
  id="sequential-db"
  title="Sequential DB analyzer"
  >

Sequential database operations often represent missed optimization opportunities that can significantly improve application performance with relatively simple changes. This analyzer helps to identify situations where database operations are being performed sequentially when they could be optimized through parallelization or batching.

**What it can help to detect:**
* Sequential database operations that could be parallelized
* Missed opportunities for batch processing
* Inefficient data access patterns

</Collapser>

<Collapser
  id="large-http-payload"
  title="Large HTTP payload analyzer"
  >


Large HTTP payloads can slow down network transfers, increase bandwidth costs, and negatively impact user experience, especially on mobile devices or slower network connections. This analyzer monitors HTTP requests and responses for payloads that exceed optimal size thresholds, which can impact network performance and user experience.

**What it can help to detect:**
* HTTP requests or responses with large payloads
* API endpoints returning excessive data
* Inefficient data serialization or transfer patterns

</Collapser>

<Collapser
  id="slow-http-analyzer"
  title="Slow HTTP analyzer"
  >


Slow HTTP responses can degrade application responsiveness, reduce throughput, and frustrate users waiting on data; particularly in latency-sensitive workflows or regions with higher network overhead. This analyzer monitors HTTP transactions for response times that exceed optimal latency thresholds, which can signal deeper performance issues across your stack.

**What it can help to detect:**
* HTTP responses with high latency
* API endpoints with consistently slow response times
* Inefficient backend processing, query patterns, or third-party dependencies contributing to slowdowns

</Collapser>

</CollapserGroup>
