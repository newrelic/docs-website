Anomaly detection gives your team flexible monitoring for unusual behavior in your system. You can set up anomaly detection for any part of your system and adjust how sensitive it is. Anomaly detection works with the same reliable technology as regular alerts and has the same fine-tuning options. This means the system can adapt to your data patterns and reduce false alarms.

You can add extra information to your anomaly alerts to give more context and include custom descriptions with helpful instructions for your team. New Relic automatically learns your data patterns over time. You can also set up custom patterns if needed.

## Why use anomaly detection [#why-anomaly-detection]

Think of anomaly detection as a smart security guard for your application. Instead of setting fixed rules like "alert me if CPU goes above 80%," anomaly detection learns what's normal for your system and alerts you when something unusual happens.

### When to choose anomaly detection over static thresholds

<Collapser
  id="use-anomaly-detection"
  title="Use anomaly detection when:"
>

* Your metrics have natural patterns (business hours, seasonal trends, weekly cycles)
* You're monitoring dynamic systems where "normal" changes over time
* You want to catch unexpected issues you didn't think to monitor
* You're tired of constantly adjusting static thresholds as your system grows

</Collapser>

<Collapser
  id="use-static-thresholds"
  title="Use static thresholds when:"
>

* You have hard limits that should never be exceeded (disk space, memory limits)
* You need to meet specific SLA requirements (response time &lt; 2 seconds)
* You're monitoring binary states (service up/down, payment processing failures)
* You have compliance requirements with fixed thresholds

</Collapser>

## How anomaly detection works [#how-it-works]

### The basics in simple terms

1. Learning phase (1-4 weeks)
New Relic watches your data and learns patterns like "Mondays are always busy" or "traffic drops at 6 PM."

2. Prediction
The system predicts what your next data point should be based on historical patterns.

3. Acceptable variation zone (grey bands)
Around the prediction, there's a band of acceptable variation. Think of it like a road with guardrails - staying within the guardrails is normal.

4. Alert triggers
When your actual data goes outside the acceptable zone AND stays there for your specified duration, an alert fires.

### Key concepts explained

Baseline (predicted value): The line New Relic draws showing expected values, like a weather forecast.

Grey prediction band: The "acceptable variation" zone around the baseline. Tighter bands = more sensitive alerts.

Duration: How long the problem must persist before alerting. This prevents false alarms from brief spikes.

Standard deviations: The mathematical way to measure "how unusual" a data point is. Higher numbers = more unusual.

## Configure anomaly sensitivity thresholds [#configure-custom-anomalies]

Anomaly sensitivity thresholds control how easily your alerts trigger. Higher sensitivity catches smaller changes but may create more false alarms. Lower sensitivity only alerts on bigger problems but might miss subtle issues.

You can create anomaly sensitivity thresholds from an [alert condition](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/). Here are some tips for setting anomaly thresholds:

* Set the [anomaly direction](#anomaly-direction) to monitor alert events that happen either above or below the anomaly.
* Set the [seasonality](#seasonality) to specify a known seasonality pattern.
* Use the slider bar to adjust the  <DNT>**Critical**</DNT> sensitivity threshold, represented in the preview chart by the light gray area around the signal. The tighter the band around the signal, the more sensitive it is and the more alert events it will generate.
* You can create a [<DNT>**Warning**</DNT> threshold](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/advanced-techniques/set-thresholds-alert-condition/#threshold-levels) (the darker gray area around the anomaly).

Follow these steps to create an anomaly detection alert condition:

1. Go to <DNT>**[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > Alerts > Alert Conditions**</DNT>.

2. Click <DNT>**+ New alert condition > Use guided mode**</DNT> (or the more advanced Query mode).

3. Go through the guided steps until you get to <DNT>**Set thresholds**</DNT>.

4. Select <DNT>**Anomaly**</DNT>.

   <img
     width="80%;"
     title="Set anomaly thresholds"
     alt="Screenshot showing the anomaly threshold configuration options in the New Relic UI"
     src="/images/alerts_screenshot-crop_try-anomaly-thresholds.webp"
   />

5. Select an option to calculate seasonality for the alert condition. To understand seasonality and how it affects your alert condition, refer [Seasonality](#seasonality).


6. Configure the settings for one or more thresholds. Anomaly detection makes a prediction on what the next data point will be based on prior activity. The threshold value for anomaly detection controls the sensitivity of the alert condition for tolerating how far off the actual value is from the predicted value. The threshold is the number of standard deviations your signal value is away from the value that was predicted. We track the standard deviation between the predicted value and the actual value for the prior 7 days of data.

   To configure the threshold, you'll need to:

   * Set the 'threshold direction' to either upper, lower, or both. This means that we'll only create an alert event if the signal value (the output of the query) is above the predicted value, below the predicted value, or either, respectively.

     This field dictates how many of the data points during a specified time period must be outside of the threshold. The options are <DNT>**for at least**</DNT> and <DNT>**at least once in**</DNT>. Selecting <DNT>**for at least**</DNT> means that ALL of your signal's data points must be outside of the threshold for the specified time period before an alert event is opened. The inverse must be true to close the alert event. The <DNT>**at least once in**</DNT> option simply means that as soon as any of your signal's data points are outside of the threshold, an alert event will open. With this option, the time duration is not relevant for determining when to open an alert event. However, it's relevant for closing alert events. All of your signal's data points must be within the threshold for the specified period of time

   * Set the 'threshold duration'. Think of this as how long the signal value must remain outside the threshold before an alert event is opened.   Conversely, it's also how long a signal must be within the threshold for an alert event to be closed.

     This field answers to the time period mentioned above. It's how long the signal exceeds the threshold being defined. This is the actual threshold duration.

   * Set the 'threshold level'. For custom anomaly detection, this is the number of standard deviations the signal's data point is from the value that we predicted it would be.

7. Add the details of the alert condition and click <DNT>**Save condition**</DNT>.

## Understanding when alerts fire [#understanding-alerts]

**The most common confusion:** Customers expect alerts to fire the moment data touches the grey band, but that's not how it works.

### The timeline of an anomaly alert

Here's what actually happens when anomaly detection evaluates your data:

**Step 1:** Data point arrives and goes outside the grey band → **No alert yet**
**Step 2:** Data stays outside the band → **Still no alert**
**Step 3:** Data has been outside the band for your specified duration → **Alert fires!**
**Step 4:** Data returns inside the band → **Alert stays open**
**Step 5:** Data stays inside the band for the specified duration → **Alert closes**

### Key concepts that cause confusion

**🎯 Baseline (Predicted Value)**
- The **center line** that New Relic draws through your data
- Shows what the system expects based on historical patterns
- Like a weather forecast: "We predict 70°F based on past data"

**📊 Grey Bands (Prediction Bands)**
- The **acceptable variation zone** around the baseline
- **Light grey** = Critical threshold (narrower band, more sensitive)
- **Dark grey** = Warning threshold (wider band, less sensitive)
- Think of it like saying "70°F ±10 degrees is normal"

**⏱️ Duration Settings (The Big Confusion)**
This is where most confusion happens. Duration determines:
- **How long** the problem must persist before alerting
- **NOT how quickly** you'll be notified once a problem starts

### Duration options explained

**"For at least X minutes"**
- ALL data points must be outside the band for X minutes straight
- More conservative - reduces false alarms from brief spikes
- Use when: You want to ignore temporary fluctuations

**"At least once in X minutes"**
- ANY single data point outside the band triggers immediately
- The X minutes only affects when the alert closes
- Use when: You want immediate notification of any anomaly

### Common duration misconceptions

❌ **Wrong:** "Set 5 minutes = I'll get notified within 5 minutes of a problem"
✅ **Correct:** "Set 5 minutes = I'll only get notified if the problem lasts 5 minutes"

❌ **Wrong:** "Lower duration = faster alerts"
✅ **Correct:** "Lower duration = alerts fire sooner, but may increase false alarms"

### Recommended settings by use case

**For critical system monitoring (fast response needed):**
```
Duration: 1-2 minutes, "for at least"
Reasoning: Quick detection while avoiding momentary spikes
```

**For business metrics (reduce noise):**
```
Duration: 10-15 minutes, "for at least"
Reasoning: Focus on sustained issues, ignore brief anomalies
```

**For security/fraud detection:**
```
Duration: 1 minute, "at least once in"
Reasoning: Immediate notification of any suspicious activity
```

## Setting thresholds for multi-signal conditions (faceted queries) [#faceted-queries]

Depending on how you defined your query in step 1, the alert condition may be monitoring many signals, not just one. When working with NRQL, these queries use the [`FACET` clause](/docs/query-your-data/nrql-new-relic-query-language/get-started/nrql-syntax-clauses-functions/#sel-facet). The maximum number of signals that one alert condition can monitor is 20,000. The threshold settings that you specify apply the same to all signals being monitored by this condition. Each signal is individually monitored and evaluated, but the settings apply consistently to all signals.
We'll only show a maximum of 500 signals on the preview chart. But, we don't show the predicted signal and threshold bands when there is more than one signal shown on the chart. To show that information while determining the ideal threshold value, select one of the time series signals from the legend to filter the chart down a single time series.

## Anomaly direction: select upper or lower ranges [#anomaly-direction]

You can choose whether you want the condition to look for behavior that goes above the predicted value ("upper") or that goes below the predicted value ("lower"), or that goes either above or below. You choose these with the prediction direction selector.

Example use cases for this:

* You might use the Upper setting for a data source like error rate, because you generally are only concerned if it goes up, and aren't concerned if it goes down.
* You might use the Lower setting for a data source like throughput, because sudden upward fluctuations are quite common, but a large sudden downswing would indicate a problem.

## Rules governing calculation of the predicted value [#anomaly-rules]

The algorithm for calculating the prediction is mathematically complex. Here are some of the major rules governing its predictive abilities:

* <DNT>**Age of data**</DNT> On initial creation, the prediction is calculated using between 1 to 4 weeks of data, depending on data availability and prediction type. Currently, queries that use the `FACET` clause aren't trained on stored data. After its creation, the algorithm takes into account ongoing data fluctuations over a long time period, although greater weight is given to more recent data. For data that has only existed for a short time, the predicted value will likely fluctuate a good deal and not be very accurate. This is because there isn't enough data to determine its usual values and behavior. The more history the data has, the more accurate the prediction will become.
* <DNT>**Consistency of data**</DNT> For metric values that remain in a consistent range or that trend slowly and steadily, their more predictable behavior means that their sensitivity thresholds will become tighter around the prediction. Data that is more varied and unpredictable will have looser (wider) sensitivity thresholds.
* <DNT>**Regular fluctuations**</DNT> For shorter-than-one-week cyclical fluctuations (such as weekly Wednesday 1pm deployments or nightly reports), the prediction algorithm looks for these cyclical fluctuations and attempts to adjust to them.

## Choose the right seasonality setting [#seasonality]

Seasonality helps anomaly detection understand your business patterns so it can distinguish between expected changes (like lunch rush) and real problems (like system outages).

Ask yourself: **Does my metric follow a predictable schedule?**

#### If NO predictable pattern → Choose "New Relic calculation" (recommended) or "None"
**Use "New Relic calculation" when:**
- You're monitoring complex, mixed environments (microservices, global applications)
- You're unsure about patterns or have multiple different patterns
- You want the system to automatically figure it out

**Use "None" when:**
- You're monitoring error rates, security events, or system failures
- The metric should stay flat and any spike is a problem
- You don't want patterns to influence alerting (every anomaly matters)

#### If YES, predictable patterns → Choose based on your pattern type:

**📅 Daily patterns → Choose "Daily"**
- **When to use:** Business hours applications, internal corporate tools
- **Examples:** HR portals (busy 9-5, quiet nights), employee VPN logins, internal APIs
- **Pattern:** High activity during work hours, low activity nights/weekends

**📊 Weekly patterns → Choose "Weekly"**
- **When to use:** B2B applications, office-based services
- **Examples:** Business software platforms, B2B sales systems, corporate websites
- **Pattern:** Busy weekdays, quiet weekends

**⏰ Hourly patterns → Choose "Hourly"**
- **When to use:** Scheduled jobs, automated processes
- **Examples:** Cron jobs, batch processing, automated backups, scheduled reports
- **Pattern:** Activity spikes at regular hourly intervals

### Real examples by industry

**E-commerce/Retail:** Use "New Relic calculation" - Complex patterns with seasonal shopping, daily peaks, and global customers

**Restaurants:** Use "Daily" - Clear business hours patterns with predictable meal rushes

**B2B Software:** Use "Weekly" - Busy during business week, quiet on weekends

**System Monitoring:** Use "None" - Error rates should stay low regardless of time patterns

### Common mistakes to avoid

❌ **Don't use "Daily" for 24/7 consumer apps** - Global users mean activity at all hours
❌ **Don't use "Weekly" for consumer mobile apps** - Users active throughout the week
❌ **Don't use any seasonality for error metrics** - Errors are always bad
❌ **Don't use "Hourly" unless you have actual hourly jobs** - Will create false expectations

### Available seasonality options

* **New Relic calculation** (default): Automatically determines the best seasonality for each signal
* **Hourly**: Expects patterns that repeat every hour (scheduled jobs, batch processes)
* **Daily**: Expects patterns that repeat every day (business hours applications)
* **Weekly**: Expects patterns that repeat every week (B2B applications, office hours)
* **None**: Ignores all patterns (error rates, security events, critical systems)

<Callout variant="tip">
Monthly and yearly seasonality options aren't supported due to computational limitations and data retention requirements.
</Callout>

## Troubleshooting and common gotchas [#troubleshooting]

### 🚨 "My alerts are too noisy" (Most Common Issue)

**Problem:** Getting alerts every few hours for normal business fluctuations

**Quick fixes:**
1. **Increase duration**: Try 10-15 minutes instead of 2-5 minutes
2. **Decrease sensitivity**: Increase standard deviation threshold (try 3 instead of 2)
3. **Check seasonality**: Make sure it matches your business pattern
4. **Switch to "for at least"**: Instead of "at least once in" to require sustained issues

**Example fix:**
```
Before: 2 standard deviations, 3 minutes, "at least once in"
After: 3 standard deviations, 15 minutes, "for at least"
Result: 90% fewer false alarms
```

### 🔴 "My critical issue didn't trigger an alert"

**Problem:** Real outage or problem didn't create an alert when it should have

**Most likely causes:**
1. **Duration too long**: Issue was resolved before alert could fire
2. **Wrong direction**: Set to "upper" but the problem was a drop in traffic ("lower")
3. **Insufficient data**: New metrics need 2-4 weeks to learn patterns accurately
4. **Standard deviation too high**: Set too conservatively, missing real issues

**Quick fixes:**
- Reduce duration to 1-2 minutes for critical systems
- Check direction setting matches your use case
- For new metrics, use static thresholds until enough data is collected
- Lower standard deviation threshold (try 2 instead of 3)

### 📊 "The predictions look completely wrong"

**Problem:** Baseline doesn't match what you see in reality

**Common causes and solutions:**

**New metric (less than 4 weeks old):**
- **Expected behavior** - predictions will be inaccurate and jumpy
- **Solution:** Wait for more data or use static thresholds temporarily

**Recent system changes:**
- **Cause:** Major deployment, infrastructure change, or business change
- **Solution:** Predictions will adapt within 1-2 weeks to new patterns

**Wrong seasonality setting:**
- **Cause:** Using "Daily" for 24/7 global app, or "Weekly" for consumer mobile app
- **Solution:** Switch to "New Relic calculation" or choose more appropriate pattern

**Data inconsistency:**
- **Cause:** Metric has extremely variable patterns
- **Solution:** Use wider standard deviation thresholds or consider if anomaly detection is right for this metric

### ⚡ "Faceted queries aren't working as expected"

**Problem:** Monitoring multiple servers/services but predictions seem off

**Important limitations to remember:**
- **No historical data**: Faceted queries start learning from scratch (don't use stored data)
- **Same settings for all**: All 20,000 signals use identical threshold settings
- **Preview limitations**: Only shows 500 signals max in chart
- **Individual learning**: Each signal learns independently, which takes time

**Recommendations:**
- Allow 4+ weeks for each signal to establish accurate predictions
- Start with conservative settings (higher standard deviations, longer durations)
- Consider separate conditions for different types of servers/services

### 🔢 "Standard deviations are confusing"

**Translation guide:**

- **1 standard deviation**: Catches about 68% of normal variation (very sensitive, lots of alerts)
- **2 standard deviations**: Catches about 95% of normal variation (balanced, most common)
- **3 standard deviations**: Catches about 99.7% of normal variation (conservative, fewer alerts)

**Practical recommendations:**
- **Start with 2-3** for most use cases
- **Use 1-2** for critical systems where you want early detection
- **Use 3-4** for noisy metrics where you only want obvious anomalies

### 🔧 Quick diagnostic checklist

When anomaly detection isn't working as expected, check these in order:

1. **Age of data**: Is the metric older than 4 weeks?
2. **Seasonality match**: Does the setting match your actual business patterns?
3. **Direction setting**: Are you monitoring the right direction (upper/lower/both)?
4. **Duration vs sensitivity balance**: Too sensitive + too short = noise, too conservative + too long = missed issues
5. **Standard deviation level**: Start with 2, adjust based on results

### 🎯 Success criteria: How to know it's working

**Good anomaly detection should:**
- Alert on genuine issues within your target response time
- Have less than 1-2 false alarms per week per condition
- Catch at least 80% of real problems (validate against known incidents)
- Require minimal tuning after the 4-week learning period

**If you're not meeting these criteria**, revisit your seasonality and duration settings first - these are the most common sources of issues.
