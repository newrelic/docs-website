Profiles define both the SNMP OIDs that `ktranslate` collects and how it processes received SNMP trap messages. Follow the instructions here if you want to modify an existing profile or create a new one.

## Requirements [#requirements]

Whether you're creating or modifying a profile, make sure you have the following:

* A [GitHub account](https://docs.github.com/en/get-started/signing-up-for-github/signing-up-for-a-new-github-account) where you can fork the [kentik/snmp-profiles](https://github.com/kentik/snmp-profiles) repository and create pull requests for your changes.
* An [SNMP walk](/docs/network-performance-monitoring/troubleshooting/snmp-walk/#capturing-snmpwalk) from a network device for polling profiles.
* For SNMP trap messages, you should also have access to the MIB from the vendor that defines the messages.

<Callout variant="important">
  New Relic does not provide support directly for the contents of the [kentik/snmp-profiles](https://github.com/kentik/snmp-profiles). If an existing SNMP profile is returning incorrect data or another breaking behavior, open an issue on the [kentik/snmp-profiles](https://github.com/kentik/snmp-profiles) repository. We are unable to provide timelines for modification or creation of SNMP profiles in the third party for `ktranslate`.

  Depending on the service level associated with your New Relic account, you may be eligible for help from New Relic Global Technical Support. In accordance with our [Global Technical Support Offerings](/docs/licenses/license-information/general-usage-licenses/global-technical-support-offerings/#tstools), we will do our best to determine whether an issue is with New Relic's products or caused by something outside our control and purview. For issues with third party tools that are outside our control, we will use commercially reasonable efforts to contribute enhancements, patches, or other functional developments to the upstream open source project.
</Callout>

    ## Background

    When you create SNMP polling profiles, you'll follow these general steps:

    1. Collect an [SNMP walk](/docs/network-performance-monitoring/troubleshooting/snmp-walk/#capturing-snmpwalk) from a sample device.
    2. Identify which MIBs are supported on the device, based on the results of the walk.
    3. In the supported MIBs, identify what metrics you'd like to collect, such as the following:
       * Scalar OIDs with counter, gauge, or enumerated metrics
       * Tabular OIDs with counter, gauge, or enumerated metrics
       * OIDs in either scalar or tabular format that return string values you can use as tags
    4. Create or edit an SNMP profile.

          The first item to inspect is the output of an SNMP walk. This is considered the "source of truth" for your device, and the results show you what specific metrics your particular device actually supports.

          <Callout variant="tip">
            It is common to find OIDs in a MIB file that look promising, only to find that your device doesn't support them. Support of MIBs and the individual OIDs inside of a MIB file are explicitly controlled by the vendor of the device. There may be configuration options available to add additional MIBs to your device, but you need to consult the vendor's documentation to research this.
          </Callout>

        <TabsPageItem id="snmp-yml">
          Every SNMP profile used by `ktranslate` is a YAML file. You can follow this tutorial by using the existing [apc_ups.yml](https://github.com/kentik/snmp-profiles/blob/main/profiles/kentik_snmp/apc/apc_ups.yml) profile, which consists of these sections:

          <CollapserGroup>
            <Collapser
              id="extends"
              title="'extends' section"
            >
              This section shows inheritance of other profiles. This is used to reduce the need to add individual OIDs from high-level MIBs into every single profile. Common examples can be found in the [\_general](https://github.com/kentik/snmp-profiles/tree/main/profiles/kentik_snmp/_general) directory of the repository.

              The syntax here is a list of file names you wish to reference.

              ```yaml
              extends:
                - system-mib.yml
                - ups-mib.yml
              ```
            </Collapser>

            <Collapser
              id="provider"
              title="'provider' section"
            >
              This section identifies the value of `provider` which will be decorated as an attribute for all payloads sent to New Relic.

              The syntax here is a string value using [kebab-case](https://www.freecodecamp.org/news/snake-case-vs-camel-case-vs-pascal-case-vs-kebab-case-whats-the-difference/#kebab-case).

              ```yaml
              provider: kentik-ups
              ```

              <Callout variant="tip">
                The value of `provider` is critical to [entity synthesis](/docs/new-relic-solutions/new-relic-one/core-concepts/what-entity-new-relic/#entity-synthesis) in New Relic. You can find more details about this in the [documentation on device configuration](/docs/network-performance-monitoring/advanced/advanced-config/#devices). If you're using your own profiles, you'll need to do one of the following:

                * Use an existing value that is already mapped to an entity.
                * Create a pull request to the [newrelic/entity-definitions](https://github.com/newrelic/entity-definitions) repository to either map your provider value to an existing definition or build a new definition.
                * Create your own custom dashboard experience to view your data.
              </Callout>
            </Collapser>

            <Collapser
              id="sysobjectid"
              title="'sysobjectid' section"
            >
              This section identifies the [sysObjectID](https://oidref.com/1.3.6.1.2.1.1.2) that `ktranslate` uses to map devices to this profile during [SNMP discovery](/docs/network-performance-monitoring/troubleshooting/snmp-discovery-unexpected-profile/).

              The syntax here is a list of OIDs. Support is provided for a wildcard match using the `*` symbol to account for large groups of devices without annotating them individually. Additionally, adding a YAML comment in line here will allow `ktranslate` to set the value of `tags.kentik.model` that will be decorated on the payloads sent to New Relic.

              ```yaml
              sysobjectid:
                - 1.3.6.1.4.1.318.1.3.17    # APC SmartUPS 3 Phase
                - 1.3.6.1.4.1.318.1.3.*     # APC UPS
                - 1.3.6.1.4.1.318.*         # APC device
              ```
            </Collapser>

            <Collapser
              id="matches"
              title="'matches' section"
            >
              This section provides an optional second level of filtering to be used with the `sysobjectid` mapping during discovery. In cases where a single SysObjectID is used for various devices, you can further isolate your device with a regular expression matching the contents of the [SysDescr](https://oidref.com/1.3.6.1.2.1.1.1) OID. The expression will match on a forced lower case version of the `SysDescr` string. You can see an example in the [net-snmp.yml](https://github.com/kentik/snmp-profiles/blob/main/profiles/kentik_snmp/_general/net-snmp.yml) profile.

              The syntax here is a dictionary of regular expressions, wrapped in double quotes as the key, with an associated value of the profile file name that should be mapped when a match is made.

              ```yaml
              matches:
                "^pfsense": pf_sense.yml
              ```
            </Collapser>

            <Collapser
              id="bulkwalk"
              title="'no_use_bulkwalkall' section"
            >
              This section provides an option to disable the [snmpbulkwalk](http://www.net-snmp.org/docs/man/snmpbulkwalk.html) operation that `ktranslate` uses as a default collection strategy. Setting this key to `true` will force `ktranslate` to use `GET`\|`GETNEXT`\|`GETTABLE` operations individually, which is useful in some older devices that may not support a bulk operation.

              ```yaml
              no_use_bulkwalkall: true
              ```
            </Collapser>

            <Collapser
              id="metrics"
              title="'metrics' section"
            >
              This section outlines the scalar and tabular OIDs that `ktranslate` will target when using this profile. The syntax for both scalar and tabular OIDs varies slightly and will be described in detail in their respective tabs. At a high level, the syntax here is a list of OIDs with various settings that allow you to fully control their collection.

              ```yaml
              metrics:
                # Scalar OID
                - MIB: $MIB_NAME
                  symbol:
                    OID: $SCALAR_OID
                    name: $OID_NAME_FROM_MIB
                # Tabular OIDs
                - MIB: $MIB_NAME
                  table:
                    OID: $TABLE_OID
                    name: $TABLE_NAME
                  symbols:
                    - OID: $COLUMN_OID
                      name: $TABULAR_OID_NAME_FROM_MIB
              ```
            </Collapser>

            <Collapser
              id="metric-tags"
              title="'metric_tags' section"
            >
              This section outlines attributes that will be decorated to the associated metrics. Dimensional metrics must be of type `count` \| `summary` \| `gauge`. As such, string values are not directly supported as metrics and must be decorated onto an existing metric using the `metric_tag` identifier.

              This identifier can be placed in individual MIB table configurations as well as the global profile settings. Use cases would be the [upsOutletGroupStatusName](https://oidref.com/1.3.6.1.4.1.318.1.1.1.12.1.2.1.2) for an outlet group in the `upsOutletGroupStatusTable` table, or the [upsAdvIdentSerialNumber](https://oidref.com/1.3.6.1.4.1.318.1.1.1.1.2.3) of your UPS device represented as a scalar OID.

              ```
              metrics:
                - MIB: PowerNet-MIB_UPS
                  table:
                    OID: 1.3.6.1.4.1.318.1.1.1.12.1.2
                    name: upsOutletGroupStatusTable
                  symbols:
                    - ...
                  metric_tags:
                    - column:
                        OID: 1.3.6.1.4.1.318.1.1.1.12.1.2.1.2
                        name: upsOutletGroupStatusName
              metric_tags:
                - column:
                    OID: 1.3.6.1.4.1.318.1.1.1.1.2.3.0
                    name: upsAdvIdentSerialNumber
              ```
            </Collapser>

            <Collapser
              id="conversions"
              title="Metric conversions available"
            >
              There are numerous conversions available to manipulate the results from SNMP polling.

              <CollapserGroup>
                <Collapser
                  id="tag"
                  title="tag"
                >
                  ```yaml
                  metrics:
                    ...
                      tag: $YOUR_TAG
                  ```

                  Used to override the `name` attribute and present a friendly name that will be sent with the payload sent to New Relic.
                </Collapser>

                <Collapser
                  id="enum"
                  title="enum"
                >
                  ```yaml
                  metrics:
                    ...
                      enum:
                        up: 1
                        down: 2
                        testing: 3
                  ```

                  Used to handle SNMP enumerations which convert the integer value of a dimensional metric into the enumerated value in an attribute decorated on the dimensional metric (using the same metric name suffix). A common example is the conversion of [kentik.snmp.if_AdminStatus](https://oid-rep.orange-labs.fr/get/1.3.6.1.2.1.2.2.1.7) to the enumerated value of [if_AdminStatus](https://github.com/kentik/snmp-profiles/blob/main/profiles/kentik_snmp/_general/if-mib.yml#L59-L66) as either `up`, `down`, or `testing`.
                </Collapser>

                <Collapser
                  id="conversion-hextoint"
                  title="conversion: hextoint"
                >
                  ```yaml
                  metrics:
                    ...
                      conversion: hextoint:$CURRENT:$DESIRED
                  ```

                  Used to convert hexadecimal values into integer format. Options for <DNT>**current**</DNT>: `LittleEndian` \| `BigEndian`. Options for <DNT>**desired**</DNT>: `uint16` \| `uint32` \| `uint64`
                </Collapser>

                <Collapser
                  id="conversion-hextoip"
                  title="conversion: hextoip"
                >
                  ```yaml
                  metrics:
                    ...
                      conversion: hextoip
                  ```

                  Used to convert hexadecimal values into 4-octet IPv4 strings.
                </Collapser>

                <Collapser
                  id="conversion-hwaddr"
                  title="conversion: hwaddr"
                >
                  ```yaml
                  metrics:
                    ...
                      conversion: hwaddr
                  ```

                  Used to convert hexadecimal values into MAC address strings.
                </Collapser>

                <Collapser
                  id="conversion-powerset-status"
                  title="conversion: powerset_status"
                >
                  ```yaml
                  metrics:
                    ...
                      conversion: powerset_status
                  ```

                  Used for enumeration of the [upsBasicStateOutputState](https://oid-rep.orange-labs.fr/get/1.3.6.1.4.1.318.1.1.1.11.1.1) ASCII string in the `POWERNET-MIB`.
                </Collapser>

                <Collapser
                  id="conversion-regexp"
                  title="conversion: regexp"
                >
                  ```yaml
                  metrics:
                    ...
                      conversion: regexp:$YOUR_REGEX_PATTERN
                  ```

                  Places a regex match on the OID output to capture substrings; needs to be wrapped in quotes and have backslashes escaped.
                </Collapser>

                <Collapser
                  id="conversion-to-one"
                  title="conversion: to_one"
                >
                  ```yaml
                  metrics:
                    ...
                      conversion: to_one
                  ```

                  Used to create a gauge metric with the value of `1` in order to poll non-numeric scalar OIDs that don't have enumeration options. An example is the [tlUpsTestResultsDetail](https://oid-rep.orange-labs.fr/get/1.3.6.1.4.1.850.100.1.7.2) OID which returns a value of the type [DisplayString](https://www.circitor.fr/Mibs/Html/S/SNMPv2-TC.php#DisplayString).
                </Collapser>
              </CollapserGroup>
            </Collapser>
          </CollapserGroup>
        </TabsPageItem>

    ## Background

    The creation of SNMP trap message profiles (also known as alerts, events, informs, or notifications) is quite simple. Generally it only requires you to investigate the vendor's MIB file to understand various trap message options.

          <Callout variant="important">
            Generation of SNMP trap messages by a device is wholly dependent on your configuration of that device.           This snippet contains 2 major sections:

          * <DNT>**Notifications by an agent that implements this MIB module**</DNT>: This is the definition of an actual trap message, identified by a unique OID.
            * [vmwVCNotifications](https://www.circitor.fr/Mibs/Html/V/VMWARE-VC-EVENT-MIB.php#vmwVCNotifications): This is the root notification OID which will cover the three defined types of messages available: [vpxdAlarm](https://www.circitor.fr/Mibs/Html/V/VMWARE-VC-EVENT-MIB.php#vpxdAlarm), [vpxdDiagnostic](https://www.circitor.fr/Mibs/Html/V/VMWARE-VC-EVENT-MIB.php#vpxdDiagnostic), and [vpxdAlarmInfo](https://www.circitor.fr/Mibs/Html/V/VMWARE-VC-EVENT-MIB.php#vpxdAlarmInfo).
          * <DNT>**Notification parameters**</DNT>: Commonly referred to as `variable bindings` or `varbinds`, these are the unique `key:value` pairs of alert data that the SNMP agent can possibly send as decorations on the trap messages.
            * In this MIB, there are eight possible varbindsall referenced as the OIDs ranging from `1.3.6.1.4.1.6876.4.3.301` - `1.3.6.1.4.1.6876.4.3.308`. This will be unique to every vendor and MIB.

              events:

                  OID: 1.3.6.1.4.1.6876.4.3.304.0
                  tag: old_status
                - name: vmwVpxdNewStatus
                  OID: 1.3.6.1.4.1.6876.4.3.305.0
                  tag: new_status

                  OID: 1.3.6.1.4.1.6876.4.3.306.0
                  tag: object_value

                  OID: 1.3.6.1.4.1.6876.4.3.307.0
                  tag: object_name

                  OID: 1.3.6.1.4.1.6876.4.3.308.0

                    unknown: 1
                    host: 2
                    vm: 3

## Use custom profiles privately [#private]

The contents of the [kentik/snmp-profiles](https://github.com/kentik/snmp-profiles) repository are loaded to `ktranslate` by default when you build the container or service and are therefore globally applied to all instances.

To support full customization options so you can maintain your own collection settings and patterns, do the following:

1. Clone the repository.
2. Make changes as you see fit.
3. Load your custom profile directory into your Docker container or Linux service on startup.

The way this is done is by mapping your customized files to the KTranslate instance inside the `etc/ktranslate/profiles/` directory.

Assuming you are in a directory just above `snmp-profiles`, the command to add looks  like this:

```shell
`pwd`/snmp-profiles/profiles:/etc/ktranslate/profiles
```

For Docker, you would add this as a [volume mount](https://docs.docker.com/storage/volumes/#start-a-container-with-a-volume), which would look similar to this:

```shell
docker run -d --name ktranslate-$CONTAINER_SERVICE --restart unless-stopped --pull=always -p 162:1620/udp \
  -v `pwd`/snmp-base.yaml:/snmp-base.yaml \
  -v `pwd`/snmp-profiles/profiles:/etc/ktranslate/profiles
  -e NEW_RELIC_API_KEY=$NR_LICENSE_KEY \
kentik/ktranslate:v2 \
  -snmp /snmp-base.yaml \
  -nr_account_id=$NR_ACCOUNT_ID \
  -metrics=jchf \
  -tee_logs=true \
  -service_name=$CONTAINER_SERVICE \
  -snmp_discovery_on_start=true \
  -snmp_discovery_min=180 \
  nr1.snmp
```

For a Linux service, you will edit the contents of `/etc/ktranslate/profiles` on the local host.

<Callout variant="tip">
  To remove the risk of inconsistent behavior in your environment, be sure to pass in your custom version of the profiles every time you launch a container or service.
</Callout>

## Contribute to the public repository [#public]

As an open source repo, contributions are welcome from anyone to create new profiles or improve the existing profiles that will enhance the monitoring capabilities for all users.

New Relic isn't involved in the process of reviewing profiles on the third-party repo. All requests for changes and updates must be addressed with Kentik's maintainers via Github issues.

Shortly after a profile is merged, CI/CD automation makes the new SNMP profiles available in a new version of the Docker image.

## What kinds of OIDs should I include? [#oids]

In many cases SNMP offers a lot of data, but little provides actionable information. The [sample SNMP walk](#snmp-polling) above has 2,022 rows of results from OIDs, and the [resulting profile](https://github.com/kentik/snmp-profiles/blob/main/profiles/kentik_snmp/apc/apc_ups.yml) collects about 50 total OIDs. Following the basic patterns of <DNT>**"Golden Signals"**</DNT> defined by the [Site Reliability Engineering (SRE) discipline](https://sre.google/sre-book/monitoring-distributed-systems/); focus on collecting data that lets you know if there is anything that would stop the device from being able to perform whatever functions you expect it to perform. Building from that, you should collect measurements that tell you how well it is performing those functions.

**Example 1:**

For a device operating as a VPN concentrator, we would collect high-level system metrics like:

* CPU and memory utilization
* Hardware sensor information to make sure that the device isn't going to shut itself down due to things like fan failures
* OIDs that tell us about the aggregated connections and throughput

**Example 2:**

An OID table that lists all the running processes on a network appliance is an example of data that is available but not helpful.

Collecting all running processes might sound useful, but since this is an appliance, you normally do not have the capability or the need to do anything with the processes that run inside it. Polling and storing tables with hundreds of items that you can't actually do anything with would not be efficient.
