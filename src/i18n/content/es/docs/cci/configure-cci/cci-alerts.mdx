Después de configurar la Inteligencia para costos en la nube, cree políticas de alertas con umbrales para recibir notificaciones proactivas antes de exceder los límites financieros y evitar cargos inesperados. Dentro de un presupuesto, puede establecer múltiples umbrales de alerta basados en el porcentaje de uso del presupuesto.

## Crear una nueva condición de alerta

Una condición de alerta es una consulta en ejecución continua que mide un conjunto determinado de eventos contra un umbral definido y abre un incidente cuando se alcanza el umbral durante una duración especificada.

1. Vaya a <DNT>[**one.newrelic.com**](https://one.newrelic.com)</DNT> &gt; <DNT>**Alerts**</DNT> &gt; <DNT>**Alert Policies**</DNT>.
2. En la página de la lista de políticas, haga clic en <DNT>**+ New alert condition**</DNT>.
3. Para crear alertas desde cero, use <DNT>**Write your own query**</DNT>.

## Establece el comportamiento de tu señal

Puede usar una consulta NRQL para definir las señales que desea que una condición de alerta use como base para su alerta. Para este ejemplo, utilizará esta consulta:

```sql
FROM CloudCost SELECT sum(line_item_unblended_cost) FACET product_region_code
```

Usar esta consulta para su condición de alerta le indica a New Relic que desea conocer el costo no combinado desglosado por código de región del producto.

Para obtener más información sobre el uso de NRQL (el lenguaje de consulta de New Relic), visite nuestra [documentación de NRQL](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/).

## Ejecuta y obtén una vista previa de tu señal [#advanced-signal-settings]

1. Una vez que haya definido su señal, haga clic en <DNT>**Run**</DNT>. Aparecerá un gráfico que mostrará el parámetro que has configurado.

   <Callout variant="tip">
     Para configurar alertas entre cuentas, seleccione un <DNT>**data account**</DNT> de la lista desplegable. Tenga en cuenta que solo puede consultar datos de una cuenta a la vez para las alertas entre cuentas.
   </Callout>

   <img title="Create a new alert condition page with advanced signal settings highlighted" alt="Create a new alert condition page with advanced signal settings highlighted" src="/images/cci-create-new-alert.webp" />

   Para este ejemplo, el gráfico mostrará el costo de uso estimado de los recursos cloud de su empresa, desglosado por código de región del producto. Esto le permite monitorear el costo de sus recursos cloud en diferentes regiones.

2. Haga clic en <DNT>**Next**</DNT> y comience a configurar su condición de alerta.

## Agregar detalles de condición de alerta [#add-details]

Tu condición de alerta está completamente definida y creará un incidente cuando se infrinjan tus umbrales. Nombre esta condición y adjúntela a una política para completar la configuración.

Una política es el sistema de clasificación para sus incidentes. Puede conectar políticas a **[flujos de trabajo](/docs/alerts-applied-intelligence/applied-intelligence/incident-workflows/incident-workflows/)** para definir a dónde desea que New Relic envíe esta información y con qué frecuencia.

        Equilibrar la capacidad de respuesta y la fatiga en su estrategia de alertas es crucial. Exploremos las opciones de política:

        1. **Un problema por política (predeterminado)**:

           * **Pros**: Reduce el ruido y garantiza una acción inmediata.
           * **Contras**: Agrupa todos los incidentes bajo un problema, incluso si son activados por diferentes condiciones. No es ideal para cuestiones de múltiples vistas de página.

        2. **Una incidencia por condición**:

           * **Pros**: crea problemas separados para cada condición, ideal para aislar y abordar problemas específicos de latencia.
           * **Contras**: Puede generar más alertas, lo que potencialmente conduce a la fatiga.

        3. **Un problema para cada incidente**:

           * **Pros**: Proporciona detalles granulares para sistemas externos, pero no es óptimo para el consumo interno debido a una posible sobrecarga.
           * **Contras**: es la opción más ruidosa, y es un desafío realizar un seguimiento de tendencias más amplias y priorizar de manera efectiva.

        Consulte [creación de políticas](/docs/alerts-applied-intelligence/new-relic-alerts/alert-policies/specify-when-alerts-create-incidents/) para obtener más información.

        Un incidente se cierra automáticamente cuando la señal objetivo vuelve a un estado de no infracción durante el período indicado en el umbral de la condición. Este tiempo de espera se llama período de recuperación.

        Cuando un incidente se cierra automáticamente:

        1. La timestamp de cierre tiene una fecha retroactiva al inicio del período de recuperación.
        2. La evaluación se reinicia y se reinicia desde que finalizó el incidente anterior.

        Todas las condiciones tienen una configuración de límite de tiempo de incidente que fuerza automáticamente el cierre de un incidente de larga duración. New Relic establece automáticamente 3 días de forma predeterminada y recomienda que use nuestra configuración predeterminada para su primera alerta.

        Otra forma de cerrar un incidente abierto cuando la señal no devuelve datos es configurando un umbral `loss of signal` . Consulte la sección anterior sobre umbral de señal perdida para obtener más detalles.
