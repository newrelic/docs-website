<Callout variant="important">
  Agent Control y New Relic Control están **disponibles de forma general** para Kubernetes. El soporte para hosts Linux y Windows está en el programa de **vista previa pública**, de conformidad con nuestras [políticas de pre-lanzamiento](/docs/licenses/license-information/referenced-policies/new-relic-pre-release-policy).
</Callout>

## Descripción general

Agente Control proporciona una plataforma única y unificada para gestionar una amplia variedad de agentes New Relic y OpenTelemetry en sus flotas. Al centralizar la gestión, podrá implementar, configurar y actualizar su instrumentación con facilidad.

Si bien Agent Control está diseñado para soportar tanto entornos de Kubernetes como basados en host, el soporte para tipos de agentes específicos varía. La siguiente tabla proporciona una descripción general completa de los agentes que pueden gestionarse mediante Agent Control y su estado de soporte en los distintos entornos.

<Callout variant="important">
  Licencias específicas del agente: Agente Control está diseñado para brindarle una gestión de licencias flexible. Si bien el agente Control en sí requiere un cierto nivel de acceso para funcionar, las licencias que otorga a cada agente se adaptan a sus necesidades específicas. A continuación, puede encontrar un desglose de las licencias necesarias para cada tipo de agente.
</Callout>

## Licencias requeridas por tipo de agente

| Tipo de agente                                  | Licencias clave requeridas                                                                                                                            | Ambiente                               |
| :---------------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------- |
| **Agente de infraestructura New Relic**         | Acceso a nivel de host para el sistema métrico y acceso API Kubernetes para datos del clúster.                                                        | Kubernetes / basado en host            |
| **Collector OpenTelemetry New Relic (NRDOT)**   | Las licencias dependen de receptores y exportadores específicos. A menudo requiere acceso a la API de Kubernetes para el descubrimiento de servicios. | Kubernetes / basado en host            |
| **Fluent Bit**                                  | Acceso de lectura a logs de pod y de contenedores.                                                                                                    | Kubernetes                             |
| **Agente New Relic Prometheus**                 | Licencias para descubrir y acceder al servicio extremo dentro del clúster para scraping métrico.                                                      | Kubernetes                             |
| **Agente eBPF New Relic**                       | Privilegios elevados (por ejemplo, `CAP_SYS_ADMIN`) para cargar programas eBPF en el kernel del host.                                                 | Kubernetes, hosts Linux (experimental) |
| **agente APM (.NET, Java, Node, Python, Ruby)** | *Actualmente no es compatible con Agente Control.*                                                                                                    | N/A                                    |

<Callout variant="caution" title="NRDOT en hosts Windows es experimental">
  El New Relic OpenTelemetry Collector (NRDOT) en Windows está disponible, pero no ha sido probado ni documentado oficialmente por el equipo de NRDOT. La configuración empaquetada predeterminada está diseñada para Linux y puede generar advertencias o errores en Windows (por ejemplo, de las rutas de `filelogreceiver`). No se incluye una configuración predeterminada para Windows; debe proporcionar su propia configuración del recopilador. Utilice NRDOT en Windows solo en entornos no críticos o de pruebas hasta que se lance el soporte completo para Windows.
</Callout>

<Callout variant="caution" title="eBPF en hosts de Linux es experimental">
  El agente eBPF de New Relic requiere dependencias a nivel de kernel (como `linux-headers` correspondiente a la versión del kernel en ejecución) que Agent Control no puede resolver automáticamente en hosts Linux. Si estas dependencias faltan o no coinciden, el despliegue puede fallar sin un error claro. El soporte de eBPF en hosts Linux está disponible para entornos de Kubernetes solo en producción. Utilice eBPF en hosts Linux solo en entornos no críticos o de prueba.

  eBPF **no es compatible** con los hosts de Windows.
</Callout>
