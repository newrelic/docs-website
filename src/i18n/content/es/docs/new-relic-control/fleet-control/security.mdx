<DNT>Fleet Control</DNT> y su componente subyacente, <DNT>Agent Control</DNT>, están diseñados con seguridad multicapa. Esta página ofrece una descripción general de las funciones de seguridad clave que se aplican en todos los entornos compatibles (clústeres de Kubernetes, hosts de Linux y hosts de Windows).

</Callout>

## Conexiones seguras e integridad de la configuración

Para proteger los datos en tránsito y evitar la manipulación, <DNT>Fleet Control</DNT> emplea múltiples capas de seguridad para todas las comunicaciones.

* **Cifrado TLS:** Toda la comunicación entre <DNT>Agent Control</DNT> y el backend de New Relic, incluido el endpoint <DNT>Fleet Control</DNT>, se transmite a través de canales seguros cifrados con TLS.

* **Firma criptográfica:** Además de TLS, todas las configuraciones enviadas desde <DNT>Fleet Control</DNT> a <DNT>Agent Control</DNT> están firmadas criptográficamente por New Relic. <DNT>Agent Control</DNT> verifica esta firma antes de aplicar cualquier configuración, garantizando que solo se ejecuten instrucciones auténticas y sin modificaciones en sus entidades gestionadas.

## Autenticación y autorización

<DNT>Fleet Control</DNT> utiliza distintos mecanismos de autenticación para sus diferentes componentes para garantizar un acceso seguro y adecuado.

* **Autenticación de<DNT>Agent Control</DNT>:** El componente <DNT>Agent Control</DNT> se autentica con el backend de <DNT>Fleet Control</DNT> usando identidades a nivel de sistema, como un par de claves y un token o un ID de cliente y un secreto. Esto garantiza que solo las instancias autorizadas de <DNT>Agent Control</DNT> puedan recibir instrucciones de gestión.

* **Autenticación del agente:** Los agentes individuales desplegados por <DNT>Fleet Control</DNT> (como el agente de infraestructura o el colector OTel) se autentican con los endpoints de ingesta de New Relic utilizando su clave de licencia estándar de New Relic.

* **Permisos de usuario:** Todas las acciones de los usuarios dentro de la UI y la API de <DNT>Fleet Control</DNT> se rigen por el control de acceso basado en roles de New Relic. Para crear o administrar flotas, los usuarios necesitan el rol de **Organization Manager** o un rol personalizado con permisos equivalentes de administración de flotas.

## Configuración segura y manejo de secretos

<DNT>Fleet Control</DNT> y <DNT>Agent Control</DNT> están diseñados para garantizar que los datos sensibles, como contraseñas y claves de API, nunca queden expuestos durante la entrega o ejecución de la configuración.

### Seguridad de la entrega de la configuración

Cuando <DNT>Fleet Control</DNT> entrega configuraciones a <DNT>Agent Control</DNT>:

* **Zero Trust Transport:** Todo el tráfico de red se cifra mediante TLS 1.2+
* **Secretos nunca en texto plano:** Las configuraciones usan marcadores de posición o referencias en lugar de los valores reales de los secretos
* **Resolución en tiempo de ejecución:** Los secretos se resuelven localmente por <DNT>Agent Control</DNT> solo cuando es necesario, nunca se almacenan en archivos de configuración
* **Sin transmisión de red:** Los secretos no atraviesan la red en las cargas útiles de OpAMP; solo se transmiten referencias de marcadores de posición

### Opciones de gestión de secretos

<DNT>Agent Control</DNT> admite múltiples métodos para gestionar datos sensibles de forma segura:

* **Expansión de variables de entorno:** El método principal, que utiliza marcadores de posición estilo shell que se resuelven en tiempo de ejecución a partir de variables de entorno
* **Integración con HashiCorp Vault:** Soporte nativo para recuperar secretos de Vault (disponibilidad general)
* **Kubernetes Secrets:** Integración directa con la API de Kubernetes para la recuperación de secretos

Este enfoque de múltiples capas garantiza que la información confidencial permanezca segura durante todo el ciclo de vida de la configuración, desde la entrega hasta la ejecución.

Para obtener instrucciones detalladas de configuración y ejemplos de implementación, consulta la [sección de gestión de secretos en la configuración de <DNT>Agent Control</DNT> ](/docs/new-relic-control/agent-control/configuration#secrets-management).

## Soporte de proxy

Para entornos con salida de red restringida, <DNT>Agent Control</DNT> ofrece soporte completo para proxies HTTP y HTTPS a nivel del sistema. Al configurar las variables de entorno estándar `HTTP_PROXY` y `HTTPS_PROXY` en sus hosts o dentro de su clúster de Kubernetes, toda la comunicación saliente de <DNT>Agent Control</DNT> se enrutará de forma segura a través del proxy especificado.

## Acceso al repositorio público

Para simplificar la instalación y reducir la fricción de seguridad, todos los componentes necesarios para <DNT>Fleet Control</DNT> están alojados en repositorios públicos.

* **Charts de Helm:** Los charts de Helm para instalar y administrar <DNT>Agent Control</DNT> están alojados en un repositorio público de New Relic en GitHub.

* **Imágenes de contenedor:** Las imágenes de contenedor para <DNT>Agent Control</DNT> y sus agentes administrados están alojadas en Docker Hub.
