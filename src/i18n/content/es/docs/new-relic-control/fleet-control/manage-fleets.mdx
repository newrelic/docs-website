Una vez que haya creado su primera flota usando la [guía de configuración](/docs/new-relic-control/fleet-control/setup), realizará la mayor parte de su trabajo diario desde la interfaz de usuario de <DNT>Fleet Control</DNT>. Si bien estas operaciones también pueden realizarse de forma programática a través de la API, esta página se centra en los flujos de trabajo de la interfaz de usuario para gestionar sus flotas, despliegues y configuraciones.

## Comprensión de los tipos de flotas

<DNT>Fleet Control</DNT> admite tres tipos de flotas, cada una diseñada para un tipo específico de entidad administrada:

* **Flotas de clústeres de Kubernetes:** Gestione clústeres de Kubernetes donde los agentes se despliegan como charts de Helm.
* **Flotas de hosts Linux:** Gestione servidores Linux donde los agentes se ejecutan como procesos basados en host. (Vista previa pública)
* **Flotas de hosts de Windows:** Administre servidores Windows donde los agentes se ejecutan como servicios de Windows o procesos basados en el host. (Vista previa pública)

Cuando crea una flota, selecciona su tipo de entidad administrada. Esta elección tiene implicaciones importantes para la forma en que trabaja con esa flota:

* **Asignación de entidades:** Solo puede agregar entidades gestionadas que coincidan con el tipo de flota. Por ejemplo, una flota de Windows Host solo puede contener hosts de Windows, y una flota de Kubernetes Cluster solo puede contener clústeres de Kubernetes.
* **Filtrado de configuración:** Durante el despliegue, solo se le presentarán las configuraciones que coincidan con el tipo de flota. Si está trabajando con una flota de Windows Host, solo aparecerán configuraciones de Windows Host en el selector de configuración. Este filtrado automático evita errores de configuración y agiliza el proceso de despliegue.
* **Disponibilidad del agente:** La lista de agentes disponibles para desplegar se filtrará según el tipo de flota y lo que sea compatible con ese entorno. Por ejemplo, ciertos agentes pueden estar disponibles solo para entornos de Kubernetes o sistemas operativos de host específicos.

Este diseño asegura la consistencia en todas las entidades administradas de una flota y evita configuraciones erróneas, como intentar implementar accidentalmente una configuración de chart de Helm de Kubernetes en un host de Windows.

## Ordenar datos en tablas

* **Ver recuentos resumidos:** vea el recuento total de su flota y la cantidad de entidades gestionadas no asignadas.

### Licencias de flota (acceso detallado)

<DNT>Fleet Control</DNT> utiliza el acceso detallado (FGA) para permitir a los administradores delegar la gestión de flotas específicas a ciertos usuarios o grupos. Esto es útil cuando desea permitir que un equipo administre sus propias flotas sin otorgarles permisos amplios a nivel de organización.

El método principal para delegar acceso es asignar el rol predefinido <DNT>**Fleet Manager**</DNT> a un grupo de usuarios para una flota específica. Esto otorga a los usuarios de ese grupo control total sobre esa flota, incluida la capacidad de crear despliegues, administrar configuraciones y agregar o eliminar entidades administradas.

#### Requisitos previos para el administrador

Para configurar los permisos de la flota, debe tener los roles <DNT>Organization Manager</DNT> y <DNT>AuthDomain Manager</DNT>.

#### Configurar el flujo de trabajo

El proceso implica dos pasos principales: primero, un administrador crea un grupo de usuarios y segundo, le asigna a ese grupo el rol <DNT>`Fleet Manager`</DNT> para una flota específica.

**Paso 1: Crear un grupo de usuarios**

1. Desde la parte inferior del menú de navegación izquierdo, haga clic en su nombre de usuario y luego seleccione <DNT>**Administration**</DNT>.
2. En la página <DNT>Administration</DNT>, seleccione <DNT>**Access Management**</DNT>.
3. Vaya a la pestaña <DNT>groups</DNT> y haga clic en **Create a new group**.
4. Dale al grupo un nombre significativo (por ejemplo, `k8s-platform-team-fleet-managers`).
5. En la sección <DNT>members</DNT>, agregue los usuarios a los que desea delegar la gestión de flotas.
6. Haga clic en <DNT>**Add user to the group**</DNT>. No necesita asignar ningún rol a nivel de cuenta u organización aquí. Los permisos se otorgarán a nivel de flota.

**Paso 2: Asignar un grupo y un rol a una flota**

Puede asignar licencias al crear una nueva flota o al editar una existente.

**Para asignar licencias a una flota existente:**

1. Desde la página principal de <DNT>fleets</DNT>, busque la flota que desea administrar.

2. Haga clic en los puntos suspensivos (`...`) al final de la fila y seleccione **Edit permissions**.

3. En el modal <DNT>fleet permissions</DNT> :

   * <DNT>**Select a group**</DNT>: Seleccione el grupo de usuarios que creó en el paso anterior.
   * **Seleccione un rol:** Elija el rol <DNT>`Fleet Manager`</DNT>.

4. Haga clic en <DNT>**Save**</DNT>. Los usuarios del grupo seleccionado ahora tienen permisos completos de administración para esta flota.

**Para asignar licencias durante la creación de la flota:**

1. Siga los pasos para [crear una flota en nuestra guía de configuración](/docs/new-relic-control/fleet-control/setup).
2. Antes de hacer clic en **Create Fleet**, expanda la sección **Access management**.
3. Emplee el desplegable para seleccionar el **Group** deseado y el rol de **Fleet Manager**.
4. Haga clic en **Create Fleet**.

## Gestionar una flota específica

Al hacer clic en el nombre de una flota, accederá a su página de gestión detallada.

<Callout title="Importante: Reglas de configuración de la flota" variant="important">
  Las flotas están diseñadas para garantizar la consistencia en todas sus entidades gestionadas. Para lograr esto, se aplican tres reglas importantes:

  * **Consistencia del tipo de entidad administrada:** Cada flota tiene un tipo de entidad administrada específico (Clúster de Kubernetes, Host de Linux o Host de Windows) que se establece cuando se crea la flota. Solo puede asignar entidades administradas que coincidan con este tipo a la flota. Esto garantiza que todas las entidades de la flota sean compatibles con las mismas configuraciones y estrategias de despliegue. Consulte [Comprender los tipos de flotas](#understanding-fleet-types) para obtener más detalles.

  * **Una configuración por tipo de agente:** dentro de una sola flota, solo puede tener una configuración para cada tipo de agente. Por ejemplo, si tiene tres clústeres de Kubernetes en una flota, todos ellos deben usar la misma configuración única para el agente New Relic Infrastructure. La cardinalidad de un tipo de agente para su configuración dentro de una flota es siempre 1:1.

  * **Instrumentación de infraestructura exclusiva:** El sistema impide desplegar tanto el agente de infraestructura de New Relic como la Distribución de New Relic de OpenTelemetry Collector (NRDOT) en la misma flota. Esta validación ayuda a evitar la doble instrumentación accidental de sus entidades gestionadas para el monitoreo de infraestructura.
</Callout>

### La cabecera de la flota: etiqueta y metadatos

En la parte superior de la página, el encabezado muestra el nombre de la flota junto con los botones **Tags** y **Metadatas** en los que se puede hacer clic. Estos proporcionan un acceso rápido a información de identificación importante para la flota.

* **Etiqueta:** Al hacer clic aquí se revela un conjunto completo de datos de pares principales de valor asociados con la flota, como `accountId`, `description` de la flota y la marca de tiempo de creación o actualización.
* **Metadatos:** al hacer clic aquí se muestran los detalles de identificación principales de la flota como una entidad New Relic, como `Entity guid`, `Account ID` y `Managed entities type`.

La página de administración detallada está organizada en tres pestañas principales: **Summary**, **Agents** y **Deployments**.

Esta pestaña proporciona una instantánea del estado actual de la flota, organizada en tres tablas:

* <DNT>**Active deployment table:**</DNT> Muestra el nombre del despliegue más reciente, el número de agentes/configuraciones incluidos y la fecha de la última actualización.

### La pestaña del agente

Esta pestaña proporciona una vista detallada de todos los agentes que se ejecutan en la entidad gestionada dentro de la flota. Puede emplear la barra de búsqueda en la parte superior de esta pestaña para encontrar un agente específico por su nombre o tipo.

Un detalle importante es que el supervisor <DNT>New Relic Agent Control</DNT> también aparece aquí como un tipo de agente. Para cada agente, puede ver su entidad gestionada asociada, el estado de instrumentación, el estado de configuración remota y cuándo se recibió un latido por última vez.

#### Ver configuración efectiva

Una característica clave de la pestaña <DNT>agents</DNT> es la capacidad de ver la *configuración efectiva* de cualquier agente. Haga clic en el nombre de un agente para abrir una vista detallada. Aquí puedes ver metadatos como la versión del chart de Helm, pero, lo más importante, puedes ver la configuración exacta que se ejecuta en el agente en tiempo real.

La configuración efectiva a veces puede diferir de la configuración que implementa, ya que tiene en cuenta cualquier anulación de fuentes como las variables de entorno locales. Esta vista es invaluable para cuestiones relacionadas con la resolución de problemas de configuración.

Esta pestaña contiene el historial completo de despliegue de la flota. Puede emplear la barra de búsqueda para encontrar una implementación específica por su nombre. La tabla enumera el nombre de cada implementación, su estado, la cantidad de agentes/configuraciones y la última fecha de actualización. La implementación exitosa más reciente está marcada con una etiqueta `Active` para indicar que es la que se está implementando actualmente en la flota.

Hacer clic en el nombre de un despliegue lo lleva a una vista de resumen para ese despliegue específico. Esta vista está organizada en dos pestañas:

#### Pestaña Detalles

La pestaña <DNT>details</DNT> proporciona un resumen del despliegue, que incluye:

* despliegue de metadatos (nombre, tipo, descripción, horas de inicio/finalización y un ID de despliegue para la resolución de problemas).
* Una tabla <DNT>**Agents**</DNT> que muestra el tipo de agente, la acción realizada (por ejemplo, instalar, actualizar) y la versión de la configuración.
* Una tabla <DNT>**Managed Entities**</DNT> que muestra exactamente qué entidades se vieron afectadas por el despliegue y si se marcaron como canarios.

#### Pestaña Eventos de despliegue

La pestaña <DNT>deployment events</DNT> proporciona visibilidad granular a nivel de evento de cada acción que ocurrió durante el despliegue. Esta pestaña es esencial para solucionar problemas de despliegue y entender las causas raíz. Puede:

* Visualice eventos atómicos con detalles que incluyen marca de tiempo, estado, nombre de la entidad administrada, nombre del agente, tipo de agente, tipo de evento, detalles y duración.
* Busque eventos específicos por palabra clave.
* Filtre eventos por entidad administrada para centrarse en un clúster o host específico (útil al solucionar fallas en grandes flotas).
* Filtre los eventos por estado (por ejemplo, Completado, Configuración aplicada, Aplicando).
* Acceda a la consulta NRQL subyacente para la resolución de problemas avanzada haciendo clic en el botón a la derecha del encabezado de la tabla de eventos.

Para obtener más información sobre el uso de la pestaña <DNT>deployment events</DNT> para la solución de problemas, consulta nuestra [documentación de solución de problemas](/docs/new-relic-control/fleet-control/troubleshooting).

## Entendiendo el despliegue

Un despliegue es un conjunto controlado de acciones que puede instalar, eliminar o actualizar agentes y sus configuraciones en las entidades administradas de su flota. Esta sección cubre aspectos clave del ciclo de vida del despliegue, incluyendo la comprensión de los diferentes estados de despliegue, el proceso de actualización automática para <DNT>Agent Control</DNT> y cómo utilizar funciones avanzadas como los despliegues en borrador y canary para gestionar los cambios de forma segura.

### Comprender los estados de implementación

El estado de cada implementación lo ayudará a comprender su resultado. Aquí están los posibles estados y lo que significan:

| Status                             | Descripción                                                                                                                                                                                                                                                                                  |
| :--------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 🔵 <DNT>**IN PROGRESS**</DNT>      | El despliegue se está implementando activamente en la entidad gestionada en la flota.                                                                                                                                                                                                        |
| 🟢 <DNT>**COMPLETED**</DNT>        | La implementación se completó exitosamente y se aplicaron todas las acciones previstas.                                                                                                                                                                                                      |
| 🔴 <DNT>**FAILED**</DNT>           | La implementación finalizó, pero al menos un agente de una entidad gestionada informó un problema. Esto podría significar que un agente no pudo iniciar, apagar correctamente o aplicar su nueva configuración. La implementación puede ser parcialmente exitosa en otra entidad gestionada. |
| 🟠 <DNT>**INTERNAL FAILURE**</DNT> | El despliegue no pudo completarse debido a un problema inesperado en el sistema <DNT>Fleet Control</DNT>. Esto no es un error con sus agentes o configuraciones. Vuelva a intentar el despliegue y, si el problema persiste, contacte al soporte de New Relic.                               |

### Comprensión del flujo de eventos de implementación

Cuando vea la pestaña <DNT>deployment events</DNT>, verá una serie de eventos que representan el ciclo de vida de su despliegue. Todos los eventos de despliegue se escriben en el tipo de evento personalizado `FleetDeployment`. Comprender este flujo de eventos le ayuda a interpretar lo que ve en la pestaña <DNT>deployment events</DNT> y a diagnosticar problemas con mayor eficacia.

<DNT>Fleet Control</DNT> despliega cambios de configuración en etapas llamadas &quot;anillos&quot; para reducir el riesgo. El proceso de despliegue funciona de la siguiente manera:

* **Si ha designado entidades canary:** La implementación utiliza primero un anillo canary (que contiene solo las entidades que marcó como canary), seguido de un anillo predeterminado para todas las entidades restantes. Cada anillo debe completarse con éxito antes de que comience el siguiente anillo. Si el anillo canary falla, el despliegue se detiene antes de proceder al anillo predeterminado, evitando problemas generalizados.
* **Si no ha designado ninguna entidad canario:** El despliegue utiliza solo el anillo predeterminado, que incluye todas las entidades de la flota. El despliegue se ejecutará hasta completarse en todas las entidades de ese único anillo.

#### Tipos de eventos de despliegue

Los siguientes eventos rastrean el progreso de un despliegue de principio a fin:

| evento                               | Descripción                                                                                  |
| :----------------------------------- | :------------------------------------------------------------------------------------------- |
| DespliegueDeFlotaIniciado            | Se inicia un despliegue para la flota. Los paquetes de configuración están preparados.       |
| Implementación en anillos iniciada   | Comienza un anillo de despliegue (por ejemplo, anillo canary o predeterminado).              |
| Despliegue del agente iniciado       | El despliegue de configuración comienza para un agente específico en una entidad gestionada. |
| Implementación del agente completada | Un agente termina de intentar aplicar la configuración.                                      |
| Implementación del anillo completada | Todos los agentes en un anillo han completado o agotado el tiempo de espera.                 |
| Despliegue de flota completado       | El despliegue general finaliza con un estado final.                                          |

#### Estados de despliegue del agente

Cuando un agente completa su intento de despliegue (AgentDeploymentCompleted), reporta uno de los siguientes estados:

| Status                     | Descripción                                                 |
| :------------------------- | :---------------------------------------------------------- |
| <DNT>CONFIG\_APPLIED</DNT> | El agente aplicó con éxito la nueva configuración.          |
| <DNT>FAILED</DNT>          | El agente encontró un error al aplicar la configuración.    |
| <DNT>TIMEOUT</DNT>         | El agente no respondió dentro del plazo esperado.           |
| <DNT>NOT\_REPORTING</DNT>  | El agente estaba desconectado cuando comenzó el despliegue. |

#### Cómo funciona el flujo de eventos

Cuando inicia un despliegue, se generan eventos en la siguiente secuencia:

1. **FleetDeploymentStarted** — El despliegue comienza para toda la flota.
2. **RingDeploymentStarted** — El primer anillo (típicamente canary) se inicia y se identifican todos los agentes en ese anillo.
3. **AgentDeploymentStarted** — Para cada agente en el anillo, se registra un evento de despliegue. Estos eventos ocurren en paralelo para todos los agentes en el anillo.
4. **AgentDeploymentCompleted** — Cada agente reporta su estado (<DNT>CONFIG\_APPLIED</DNT>, <DNT>FAILED</DNT>, <DNT>TIMEOUT</DNT> o <DNT>NOT\_REPORTING</DNT>).
5. **RingDeploymentCompleted** — Una vez que todos los agentes en el anillo finalizan o agotan el tiempo de espera, el anillo se marca como completo. Si el anillo tuvo éxito, comienza el siguiente anillo. Si falló, el despliegue se detiene.
6. **FleetDeploymentCompleted** — Después de que todos los anillos se completen (o un anillo falle), la implementación finaliza con un estado final de <DNT>COMPLETED</DNT>, <DNT>FAILED</DNT> o <DNT>INTERNAL\_FAILURE</DNT>.

#### Flujo visual de eventos

```
FleetDeploymentStarted
│
├─▶ RingDeploymentStarted (canary)
│   │
│   ├─▶ AgentDeploymentStarted (agent 1) ──▶ AgentDeploymentCompleted ─┐
│   ├─▶ AgentDeploymentStarted (agent 2) ──▶ AgentDeploymentCompleted ─┤ [parallel]
│   ├─▶ AgentDeploymentStarted (agent 3) ──▶ AgentDeploymentCompleted ─┘
│   │
│   └─▶ RingDeploymentCompleted (canary)
│       │
│       ├─ ✓ If SUCCESS → proceed to default ring
│       └─ ✗ If FAILED → stop deployment
│
├─▶ RingDeploymentStarted (default)
│   │
│   ├─▶ AgentDeploymentStarted (agent 4) ──▶ AgentDeploymentCompleted ─┐
│   ├─▶ AgentDeploymentStarted (agent 5) ──▶ AgentDeploymentCompleted ─┤ [parallel]
│   ├─▶ AgentDeploymentStarted (agent 6) ──▶ AgentDeploymentCompleted ─┘
│   │
│   └─▶ RingDeploymentCompleted (default)
│
└─▶ FleetDeploymentCompleted
```

**Opciones de estado de despliegue del agente:**

* ✅ **CONFIG\_APPLIED** — El agente aplicó la configuración correctamente
* ❌ **FAILED** — El agente encontró un error
* ⏱️ **TIMEOUT** — El agente no respondió a tiempo
* 📡 **NOT\_REPORTING** — El agente estaba desconectado cuando comenzó el despliegue

**Identificadores clave:** `fleetId`, `deploymentId`, `ringName`

Puede ver todos estos eventos en detalle utilizando la pestaña Eventos de despliegue. Consulte la [sección de la pestaña Despliegues](#the-deployments-tab) para obtener más información sobre cómo acceder y utilizar los eventos de despliegue para la resolución de problemas.

### Actualizaciones automáticas de control de agente

Para garantizar que sus entidades administradas se beneficien de los últimos parches de seguridad, mejoras de rendimiento y características, <DNT>Fleet Control</DNT> administra automáticamente el proceso de actualización de Agent Control. Cuando inicia un despliegue, <DNT>Fleet Control</DNT> verifica si hay una versión más reciente de Agent Control disponible para las entidades administradas en su flota.

Si se requiere una actualización, se incluye automáticamente como parte de su implementación. Verá una notificación en la UI durante el proceso de implementación informándole que se actualizará el agente Control. Esta actualización implícita ocurre junto con cualquier acción de implementación que esté realizando (como instalar un agente, eliminar un agente o cambiar la configuración del agente), lo que garantiza que su entorno se mantenga actualizado con un mínimo esfuerzo.

<Callout title="Importante: Error de actualización y recuperación" variant="caution">
  Actualmente, no existe una reversión automática en caso de una actualización fallida del Control del agente. Si falla una actualización, la implementación se marcará como **FAILED** y la entidad gestionada puede requerir intervención manual. Para obtener orientación, consulte nuestra documentación sobre [cómo actualizar manualmente el agente Control](/docs/new-relic-control/agent-control/upgrading).

  Estamos desarrollando activamente característica para proporcionar una reversión automática y simplificar la solución de fallas de actualización en una versión futura.
</Callout>

## Crear y gestionar borradores de implementación

Puede almacenar un despliegue como borrador para trabajar en él a lo largo del tiempo sin tener que desplegar los cambios inmediatamente.

### Crear y reanudar un borrador

1. Navegue a la pestaña <DNT>deployments</DNT> dentro de una flota específica.
2. Haga clic en <DNT>**Create deployment**</DNT>.
3. En el panel de implementación, proporcione un nombre obligatorio y una descripción opcional.
4. Realice los cambios que desee, como agregar un agente o actualizar su configuración.
5. En lugar de iniciar el despliegue, haga clic en <DNT>**Save draft**</DNT>.
6. Luego puede cerrar el panel de despliegue. Su nueva implementación aparecerá en la pestaña <DNT>deployments</DNT> con un estado de `Draft`.

Para continuar trabajando en el borrador, simplemente haga clic en su nombre en la tabla de historial de despliegue. Esto volverá a abrir el panel de implementación, lo que le permitirá finalizar los cambios y hacer clic en **Start deployment** cuando esté listo.

### Eliminar un borrador

Si ya no necesita un borrador de implementación:

1. Desde la pestaña <DNT>deployments</DNT>, haga clic en el nombre del borrador que desea eliminar.
2. En el panel de despliegue que se abre, haga clic en el botón de puntos suspensivos (`...`) junto a <DNT>Start deployment</DNT> en la parte superior de la pantalla.
3. Seleccione <DNT>**Delete draft**</DNT> del menú.
4. El borrador se eliminará permanentemente y regresará a la pestaña <DNT>deployments</DNT>.

1. Después de seleccionar el tipo de agente, aparece un menú desplegable <DNT>**Select a version**</DNT>.

2. La última versión está seleccionada de forma predeterminada.

3. Puede elegir entre otras versiones disponibles para la instalación.

4. Se pueden mostrar metadatos adicionales para cada versión, como:

   * Notas de la versión que destacan nuevas funciones y mejoras
   * Información de la actualización de seguridad
   * Otros detalles específicos de la versión

Esta capacidad de selección de versiones funciona en todos los tipos de infraestructura compatibles (clústeres de Kubernetes, hosts de Linux y hosts de Windows), lo que le brinda un control preciso sobre qué versión del agente se despliega en sus entidades administradas.

Una vez que haya seleccionado la versión de su agente, haga clic en <DNT>**Next**</DNT> para proceder a la selección de la configuración.

  * **Versión de la configuración:** La versión del archivo de configuración que ha creado para ese agente, administrada a través del menú de acciones superior.

  Cuando seleccionas &quot;Usar la versión más reciente&quot; o &quot;Cambiar versión de configuración&quot;, administras la versión de la configuración, no la versión del agente.
</Callout>

## Gestionar entidad en una flota

<Callout title="Importante: El principio de herencia de la flota" variant="important">
  Las flotas funcionan según un principio fundamental de **herencia**. Esto significa que cualquier entidad gestionada asignada a una flota se ajustará automáticamente al último despliegue activo de esa flota para garantizar la consistencia de la instrumentación. Comprender este comportamiento es fundamental al agregar o eliminar entidades, ya que puede resultar en la instalación o desinstalación automática del agente.
</Callout>

### Eliminar entidad de una flota

1. Navegue a la pestaña <DNT>summary</DNT> de la flota que actualmente contiene la entidad.
2. En la tabla <DNT>managed entities</DNT>, utilice la casilla de verificación para seleccionar una o más entidades que desee eliminar.
3. Haga clic en <DNT>Remove</DNT> que aparece.
4. Aparecerá un modal de verificación que enumerará la entidad que se eliminará. Para confirmar, escriba `remove` en el campo de texto.
5. Haga clic en <DNT>Remove entities</DNT>. La entidad ahora está desasignada de la flota.

<Callout title="Importante: Volver a la configuración local" variant="caution">
  Cuando una entidad gestionada se elimina de una flota, automáticamente vuelve a su **configuración local**original (el estado en el que se encontraba cuando se instaló por primera vez Agente Control). Esto puede provocar la desinstalación del agente. Por ejemplo, si la flota tenía instalados el agente de infraestructura y Fluent Bit, pero la configuración local de la entidad solo incluye el agente Control, ambos agentes se desinstalarán al eliminarlos para garantizar que coincidan con el estado local.
</Callout>

<img title="Remove entities confirmation" alt="The Remove entities confirmation panel with explanatory text, a text input field where the user must type 'REMOVE' to confirm, a table showing the entity to be removed (demo-cluster-eks-219-3), and the Remove entities button." src="/images/fleet-control-remove-me.webp" />

### Agregar una entidad gestionada no asignada a una flota

Para agregar una entidad gestionada no asignada a una flota, cree un nuevo despliegue. Este proceso asocia la entidad con la flota y su configuración de despliegue.

1. Navegue a la flota donde desea agregar la entidad y haga clic en <DNT>**Create a deployment**</DNT>.
2. Dale al despliegue un nombre y una descripción significativos.
3. En la tabla <DNT>managed entities</DNT> en la parte inferior de la pantalla de despliegue, haga clic en el botón <DNT>**Add managed entities**</DNT>.
4. Se abrirá un panel que mostrará una lista de todas las entidades gestionadas no asignadas.
5. Emplee los cuadros de selección para elegir una o más entidades que desee agregar a la flota.
6. Haga clic en <DNT>Add to deployment</DNT>. Las entidades seleccionadas aparecerán ahora en la tabla <DNT>managed entities</DNT> para esta implementación.
7. Haga clic en <DNT>Save draft</DNT> y luego en <DNT>Start deployment</DNT> para completar el proceso. Las entidades ahora se asignarán a la nueva flota.

<Callout title="Importante: Heredar el último despliegue activo" variant="important">
  Cuando agrega una entidad gestionada a una flota, se produce una implementación implícita en segundo plano. Este despliegue instala, actualiza o elimina automáticamente la instrumentación en la nueva entidad para garantizar que coincida perfectamente con el último **despliegue activo** de la flota. Por ejemplo, si el despliegue activo de la flota incluye los agentes de Infrastructure y Fluent Bit, pero la nueva entidad anteriormente solo tenía el agente OTel, el agente OTel se eliminará y se instalarán los agentes <DNT>Infrastructure</DNT> y <DNT>Fluent Bit</DNT>.
</Callout>

<img title="Add managed entities to deployment" alt="The Add managed entities selection pane showing a searchable table of available entities with checkboxes, allowing users to select which unassigned managed entities to add to the fleet via this deployment." src="/images/fleet-control-add-me-deployment.webp" />

## Usar despliegue canario

<DNT>Fleet Control</DNT> le permite designar una o más entidades administradas como &quot;canary&quot; durante el proceso de implementación. Esto proporciona un mecanismo de seguridad crucial para desplegar cambios.

Cuando crea un nuevo despliegue, puede emplear los puntos suspensivos (`...`) junto a cualquier entidad gestionada en la lista y designarla como canaria.

Cuando se inicia el despliegue:

1. Los cambios se implementan *únicamente* en las entidades gestionadas marcadas como canarias.
2. El proceso de implementación se detiene y espera hasta que la implementación canaria tenga éxito.
3. Si falla cualquier implementación canaria, toda la implementación se detiene inmediatamente y ninguna otra entidad gestionada se ve afectada.
4. Si todas las implementaciones canarias son exitosas, la implementación procede al resto de la entidad gestionada en la flota.

<Callout title="Importante: No hay reversión automática para canarios fallidos" variant="caution">
  Si falla una implementación canaria, la entidad gestionada en la que se implementó permanecerá en estado **Failed**. No existe una reversión automática a la configuración anterior. Necesitará crear una nueva implementación para revertir los cambios o solucionar el problema.
</Callout>

<img title="Canary deployment enablement" alt="The deployment pane's Managed entities table with the ellipsis menu expanded, showing the 'Use as a canary' option to designate a managed entity for staged canary deployment." src="/images/fleet-control-canary-action.webp" />

## La página de configuración

<Callout title="Importante: las configuraciones son globales" variant="important">
  Todas las configuraciones que cree en <DNT>Fleet Control</DNT> son **globales**. Esto significa que están disponibles para su uso en toda su organización de New Relic y se pueden implementar en cualquier flota que los requiera, independientemente de quién haya creado la configuración.
</Callout>

Al navegar a <DNT>New Relic Control → Configurations</DNT>, accederá a la página principal de Configuraciones, que es el centro principal para administrar todas las configuraciones de sus agentes.

### Descripción general de la página principal

Esta página proporciona una descripción general de alto nivel de todas las configuraciones disponibles en su cuenta. Desde aquí, puedes:

* **Ver recuentos resumidos:** vea su recuento total de configuración.
* **Ver distribución de la configuración:** vea un desglose de su configuración por tipo de agente (por ejemplo, New Relic Infrastructure, Fluent Bit, recolector NRDOT).
* **Buscar configuración:** emplee la barra de búsqueda para encontrar una configuración específica por su nombre o tipo de agente asociado.
* **Actualizar la lista:** Si acaba de crear una nueva configuración y no la ve en la tabla, haga clic en el botón de actualización ubicado a la izquierda de <DNT>**Create configuration**</DNT>.

La página muestra una tabla de su configuración con las siguientes columnas:

* **configuración:** El nombre de la configuración.
* **agente:** El tipo de agente al que se aplica la configuración.
* **Revisiones:** La cantidad de versiones que existen para la configuración.
* **Tipo:** El tipo de entidad administrada (por ejemplo, Clúster de Kubernetes, Host de Linux o Host de Windows).
* **Última modificación:** una timestamp de la última actualización.

<img title="Main Configurations page" alt="The Configurations page showing total configuration count, distribution chart by agent type, and a searchable table listing all configurations with their agent types, revision counts, entity types, and last modified dates." src="/images/fleet-control-configurations-page.webp" />

### Acciones de configuración

Desde la tabla de configuración principal, puede hacer clic en los puntos suspensivos (`...`) en el extremo derecho de cualquier configuración para **Edit**la. Esto abre el panel de edición principal donde puede modificar el archivo de configuración. Esta vista también proporciona botones para **Download** el archivo de configuración a su máquina local o **Copy** su contenido a su portapapeles.

### Entendiendo la configuración

La estructura de un archivo de configuración depende del **Tipo de entidad administrada** que seleccione (Clúster de Kubernetes, Host de Linux o Host de Windows).

* **Para los clústeres de Kubernetes,** la configuración que crea en la interfaz de usuario es un archivo `values.yaml`. Durante la implementación, <DNT>Fleet Control</DNT> empaqueta este archivo con el chart de Helm del agente para configurarlo en su clúster. Las plantillas para los agentes de Kubernetes están estructuradas como un archivo `values.yaml` de chart de Helm canónico. Cerca de la parte superior de la plantilla, encontrarás un enlace comentado al chart de Helm del agente en el [repositorio de GitHub de New Relic Helm Charts](https://github.com/newrelic/helm-charts). Esto le permite consultar fácilmente el esquema, los valores disponibles y otra documentación específica del agente.
* **Para los hosts,** el formato del archivo de configuración es específico del agente que se va a implementar.

<Callout title="Filtrado de configuración en despliegues" variant="tip">
  Al crear un despliegue para una flota, el sistema filtra automáticamente las configuraciones disponibles para mostrar solo aquellas que coinciden con el tipo de entidad gestionada de la flota. Por ejemplo:

  * Una **flota de clústeres de Kubernetes** solo mostrará configuraciones creadas para clústeres de Kubernetes.
  * Una **flota de hosts Linux** solo mostrará las configuraciones creadas para hosts Linux.
  * Una **flota de hosts de Windows** solo mostrará configuraciones creadas para hosts de Windows.

  Este filtrado automático garantiza que no pueda desplegar accidentalmente una configuración incompatible en sus entidades administradas, manteniendo la consistencia y evitando fallas en el despliegue.
</Callout>

<Callout title="Mejores prácticas de seguridad: Gestión de datos sensibles en Kubernetes" variant="important">
  Para despliegues de Kubernetes, nunca incluya datos sensibles como su `licenseKey` directamente en los archivos de configuración. En su lugar, utilice la gestión de secretos de Agent Control para inyectar valores sensibles de forma segura en tiempo de ejecución. Para obtener más información, consulta nuestra [documentación sobre gestión de secretos](/docs/new-relic-control/agent-control/configuration#secrets-management).
</Callout>

### Crear una configuración

Para crear una nueva configuración global desde cero:

1. Desde la página principal de Configuraciones, haga clic en <DNT>**Create configuration**</DNT>.

2. En el modal que aparece debes:

   * Seleccione el <DNT>**managed entity type**</DNT> (clúster de Kubernetes, host de Linux o host de Windows). Tenga en cuenta que el soporte para Linux Host y Windows Host está actualmente en vista previa pública.
   * Selecciona el <DNT>**Agent type**</DNT> (por ejemplo, New Relic Infrastructure, NRDOT collector, Fluent Bit) del menú desplegable. Los tipos de agentes disponibles se filtrarán según el tipo de entidad administrada que seleccionó.
   * Proporcione un <DNT>**Name**</DNT> significativo y único para la configuración.

<img title="Create configuration modal" alt="The Create configuration modal with dropdown fields for selecting Managed entity type, Agent type, and a text field for entering the configuration name." src="/images/fleet-control-create-configuration-modal.webp" />

3. Haga clic en <DNT>**Continue**</DNT>. Esto abre la vista de edición de configuración, que se divide en dos paneles.

   * **Panel izquierdo (configuración activa):** aquí es donde usted crea su configuración. Puede escribir directamente en este panel o pegar contenido de otra fuente.
   * **Panel derecho (Plantilla):** este panel contiene la última plantilla predeterminada para el tipo de agente que seleccionó. Estas plantillas proporcionan las configuraciones mínimas necesarias para ejecutar el agente, pero también incluyen muchas opciones de comentarios adicionales. Puede emplear estos ejemplos como referencia o descomentarlos para crear una configuración más personalizada.

4. Un flujo de trabajo común es hacer clic en <DNT>**Copy**</DNT> en el panel de plantillas y pegar el contenido en el panel de configuración activo a la izquierda para usarlo como punto de partida.

5. Una vez que esté satisfecho con su configuración en el panel izquierdo, haga clic en <DNT>**Save**</DNT>. La vista de edición se cerrará y regresará a la página principal de configuraciones, donde podrá encontrar su nueva configuración.

### Ver y versionar una configuración específica
