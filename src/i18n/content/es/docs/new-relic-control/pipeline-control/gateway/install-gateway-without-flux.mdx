Al instalar el gateway de Control de pipelines en modo de solo configuración, puede continuar administrando el ciclo de vida de la configuración de su pipeline (muestreo, filtrado, transformaciones) a través de la UI de New Relic mientras administra la infraestructura del gateway (escalado, versiones) mediante comandos de Kubernetes y Helm.

## Cuándo usar este modo de instalación

Elija este modo de instalación si:

* Sus políticas de seguridad restringen el acceso de administrador del clúster.
* Necesita control directo sobre el escalamiento y las actualizaciones de la infraestructura.
* Te sientes cómodo administrando recursos de Kubernetes usando comandos de kubectl y Helm.
* Está desplegando en entornos orientados al cumplimiento.

**Lo que obtiene:**

* Administrar la configuración de su pipeline (muestreo, filtrado, transformaciones) a través de la UI.
* Menores requisitos de permisos. New Relic solo necesita acceso `ConfigMap` (no de administrador del clúster) para desplegar la configuración de pipelines.
* Control directo sobre el escalamiento y las versiones del gateway.

**Lo que administra usted mismo:**

* Escalado de la infraestructura de gateway mediante Kubernetes Horizontal Pod Autoscaler (HPA)
* Actualizaciones de versión del Gateway mediante comandos de Helm

<Callout variant="tip">
  **¿Necesita automatización completa?** Si desea manejar toda la gestión de la infraestructura del gateway (escalado, versiones, actualizaciones) a través de la UI y su equipo de seguridad aprueba los permisos de administrador de clúster, consulte [Configurar y desplegar el gateway de Control de pipelines](/docs/new-relic-control/pipeline-control/set-up-and-deploy-pipeline-control) para la instalación de gestión completa.
</Callout>

## Consideraciones para el uso de la puerta de enlace

Al integrar y usar el componente de gateway, tenga en cuenta estas consideraciones:

### atributo y gestión de datos

* **Soporte de atributos:** El gateway utiliza las convenciones semánticas de OpenTelemetry para los atributos. Consulte la [documentación del gateway](/docs/new-relic-control/pipeline-control/gateway/overview) para obtener detalles sobre los atributos admitidos.
* **Manejo de datos/atributos:** la puerta de enlace reenvía sus requests para eliminar datos o atributos para datos de intervalos de tiempo métricos, trazas y trazas directamente a New Relic sin modificaciones.

### Análisis e integración de datos

* **Análisis de logs:** la puerta de enlace no incluye el análisis de Grok para logs. Es posible que necesite métodos alternativos para estructurar los datos de log para realizar consultas y visualizarlos.
* **Integración de Prometheus:** el gateway no admite la escritura remota de Prometheus para métricas.

### Licencias y cumplimiento

* **Gestión de claves de licencia:** el gateway no admite múltiples claves de licencia de fuentes nativas del protocolo OpenTelemetry (OTLP), como recolectores de agentes o SDK de APM.
* **Estándares de cumplimiento:** la puerta de enlace no cumple con los estándares de FedRAMP, lo que puede afectar su uso si necesita cumplir con los estándares de seguridad federales.

## Requisitos previos

* **Permisos de Kubernetes:** este modo de instalación solo requiere permisos CREATE/UPDATE de ConfigMap (no cluster-admin).

* **Herramientas necesarias:**

  * Helm v3.x o superior (verificar con helm version)
  * `kubectl` CLI con acceso a su clúster de Kubernetes

* **Agente de infraestructura de New Relic (opcional):** el gateway de Control de pipelines ya no incluye el agente de infraestructura de New Relic. Para ver las métricas de salud del clúster (uso de CPU, memoria y región) en el dashboard de salud del gateway, instale el agente de infraestructura por separado. Consulte [Instale la integración de Kubernetes](/install/kubernetes/) para obtener instrucciones de instalación.

**Qué sucede sin el agente de infraestructura:**

* La tabla de estado del Gateway mostrará &quot;N/A&quot; para: uso de CPU, uso de memoria y región.
* La configuración del Pipeline y el procesamiento de datos continúan funcionando normalmente.

<Callout variant="important">
  Si instalaste el gateway antes de este cambio, tu agente de infraestructura permanece en su lugar y continúa funcionando. Este requisito solo afecta a las nuevas instalaciones de gateway.
</Callout>

## Instalar el gateway de Control de pipelines

<img title="Install gateway without Flux" alt="Screenshot of Pipeline Control gateway installation wizard" src="/images/install-pipeline-control.webp" />

Para instalar el gateway de Control de pipelines sin Flux:

1. En su cuenta New Relic , en Integración y agente, haga clic en **Set up Pipeline Control** o busque **gateway** y seleccione **Pipeline Control gateway**.

2. Cerciorar de que la cuenta seleccionada sea la que desea emplear para el Control de tuberías. En caso contrario, haga clic en **Select an account**, seleccione la cuenta de su elección y luego haga clic en **Continue**.

3. Para ingresar sus credenciales, proporcione una clave de licencia existente o cree una nueva clave. Si crea una nueva clave, copie la clave. Luego, haga clic en **Continue**.

4. Seleccione o cree una flota de gateways:

   * Seleccione una flota existente si desea que su nuevo gateway comparta una configuración existente (reglas de muestreo, filtrado y transformación) con otros gateways en su organización.
   * Ingrese un nombre único para la flota de gateways si necesita una configuración nueva y aislada para un entorno o unidad de negocio específico.

5. Seleccione **Configuration only (without Flux)**.

6. Configure la puerta de enlace:

   * Ingrese el nombre de su clúster.
   * Ingrese un namespace para las instalaciones de gateway y Control de agente. El valor predeterminado es `newrelic`.

7. Instalar Agent Control:

   ```bash
   helm repo add newrelic https://helm-charts.newrelic.com && helm repo update && helm upgrade --install agent-control-deployment newrelic/agent-control-deployment -n newrelic --create-namespace --values agent-control-deployment-values.yaml
   ```

   <Callout variant="tip">
     El asistente de la UI proporciona el archivo de valores completo (`agent-control-deployment-values.yaml`) con su configuración específica. Descargue este archivo y úselo con el comando Helm.
   </Callout>

8. Verifique la instalación del Control del agente y luego haga clic en **Continue**.

9. Configure los ajustes del gateway (réplicas mínimas, réplicas máximas, umbral de CPU). Estos ajustes crean la configuración inicial del HPA de Kubernetes. Después de la instalación, los cambios en esta configuración a través de la UI no se aplican. Debe gestionar el escalado a través de HPA directamente.

10. Instalar la puerta de enlace:

    ```bash
    helm upgrade --install newrelic-pcg newrelic/pipeline-control-gateway -n newrelic --create-namespace --values pipeline-control-gateway-values.yaml
    ```

    <Callout variant="tip">
      El asistente de la UI proporciona el archivo de valores completo (`pipeline-control-gateway-values.yaml`) con su configuración específica. Descargue este archivo y úselo con el comando Helm.
    </Callout>

11. Haga clic en **Continue**.

12. Haga clic en **Test connection** y espere hasta que la conexión sea exitosa.

Para ver los detalles de la configuración de su puerta de enlace, vaya a New Relic Control y haga clic en **Gateway**.

## ¿Por qué usar múltiples flotas?

Imagine que una corporación (**The Corp**) tiene dos divisiones: **OrgA** (necesita el 100 % de los datos por motivos de auditoría) y **OrgB** (necesita un muestreo del 1 % para controlar los costos de alto volumen).

Con solo una flota, **The Corp** está atascada. Si **OrgB** aplica un muestreo del 1 % para ahorrar dinero, **OrgA** pierde el 99 % de sus datos de rendimiento. El control de costos de un equipo rompe el monitoreo de misión crítica del otro equipo.

Al usar múltiples flotas, **The Corp** puede separar sus &quot;cerebros&quot;:

* **Flota de misión crítica**: **OrgA** se une a esta flota para obtener el 100 % de retención de datos.
* **Flota de alto volumen**: **OrgB** se une a esta flota para un muestreo del 1 % y la redacción de datos personales.

Como resultado, los equipos independientes ahora pueden administrar reglas únicas de muestreo, seguridad y cumplimiento sin afectar al resto de la organización.

## Administrar el gateway después de la instalación

Después de la instalación, la página de configuración del gateway en la UI de New Relic muestra:

* **Versión del gateway** - Se muestra como solo lectura

* **Dos interruptores editables:**

  * **Recopilar registros de diagnóstico sobre el gateway** - Habilitar/deshabilitar el logging de diagnóstico
  * **Omitir las reglas del gateway si su CPU disponible es baja** - Alternar la omisión de reglas durante las limitaciones de recursos

Para administrar la infraestructura de gateway (escalamiento y versiones), utilice estos enfoques:

### Cambiar la configuración de escalamiento del gateway

Para modificar el mínimo/máximo de réplicas o los umbrales de CPU, actualice su HPA de Kubernetes:

**Opción 1: editar HPA directamente** Úselo para pruebas rápidas o ajustes temporales, respuesta inmediata a problemas de escalamiento o cuando no se necesite una actualización de Helm.

1. Ver el HPA actual:

   ```bash
   kubectl get hpa -n newrelic
   ```

   Este ejemplo asume que el namespace es `newrelic`. Si utiliza un namespace diferente, cámbielo en consecuencia aquí y en los siguientes comandos.

2. Editar HPA directamente:

   ```bash
   kubectl edit hpa pipeline-control-gateway-hpa -n newrelic                                                          
   ```

   Aquí, pipeline-control-gateway-hpa es el HPA que estás editando. Cámbielo por el nombre de su HPA.

   En el editor, modifique los campos que necesite, y luego guarde y salga. Tus cambios se aplican inmediatamente.

3. Verifique sus cambios:

   ```bash
   kubectl get hpa pipeline-control-gateway-hpa -n newrelic                                                          
   ```

   Confirme que se aplicaron los cambios verificando que la salida muestre los nuevos valores para las configuraciones que modificó.

**Opción 2: actualizar mediante Helm**

1. Edite su archivo `pipeline-control-gateway-values.yaml` .

2. Correr:

   ```bash
      helm upgrade newrelic-pcg newrelic/pipeline-control-gateway -n newrelic --values pipeline-control-gateway-values.yaml
   ```

3. Verifique sus cambios:

   ```bash
      kubectl get hpa -n newrelic
   ```

### Actualizar la versión del gateway

Para actualizar la versión del gateway de Control de pipelines:

1. Actualice la versión del chart en su comando de Helm:

   ```bash
   helm upgrade newrelic-pcg newrelic/pipeline-control-gateway --namespace newrelic --version <NEW_VERSION> --values pipeline-control-gateway-values.yaml
   ```

2. Verifique la actualización:

   ```bash
   helm list -n newrelic
   kubectl get pods -n newrelic
   ```

### Cambios de configuración del Pipeline

Estos se gestionan a través de la UI:

* Reglas de muestreo
* Filtros de eliminación
* Reglas de transformación
* Muestreo condicional

Los cambios en la configuración del pipeline se despliegan automáticamente mediante actualizaciones de ConfigMap, incluso en este modo.

## Agregar más clúster

Para agregar un nuevo clúster a su configuración de gateway:

1. Abra New Relic Control y luego seleccione **Gateway**.
2. En la tabla de clúster, haga clic en **Add a cluster**.
3. Siga las instrucciones de instalación anteriores.
4. Regrese a la página de resumen de la puerta de enlace. Deberías ver el clúster recién agregado.

Después de crear sus clústeres de gateway, configure el procesamiento del pipeline mediante la configuración YAML. Consulte [Configuración de Gateway](/docs/new-relic-control/pipeline-control/gateway/overview).
