Pipeline Control es el componente de gestión de datos de New Relic Control. Le ayuda a filtrar, transformar y gestionar datos de telemetría para reducir costos, mejorar la calidad de la señal y mantener el cumplimiento.

Pipeline Control ofrece dos capas de procesamiento que puede utilizar de forma independiente o conjunta:

## Gateway de control de pipelines

**Las reglas de Gateway** se ejecutan en su infraestructura:

* Procese los datos **antes de que salgan de su red** (reduce los costos de salida)
* Configuración a nivel de organización (las reglas se aplican a todas las cuentas de su organización)
* Capacidades completas de transformación, filtrado y muestreo
* Usted gestiona la infraestructura (requiere la implementación de un clúster de Kubernetes)
* Se cobra según el volumen de datos recibido en la puerta de enlace

El gateway es ideal cuando necesita:

* Para reducir los costos de salida de red al filtrar datos antes de que salgan de su infraestructura
* Capacidades de transformación de datos más completas para todos los tipos de MELT (p. ej. analizar logs, normalizar atributos, enriquecer con metadatos)
* Para descartar u ofuscar datos sensibles antes de que salgan de su red por requisitos de cumplimiento
* Para procesar mayores volúmenes de datos, donde el gateway es más económico que las reglas en la nube

Obtenga más información sobre el gateway en [Descripción general del gateway](/docs/new-relic-control/pipeline-control/gateway/overview).

## Reglas de nube de Pipeline Control

Las **reglas de la nube** se ejecutan en la infraestructura de New Relic después de que los datos llegan a los endpoints de New Relic:

* Procesa los datos **después de que lleguen a New Relic** (no se requiere gestión de infraestructura)
* Reglas específicas de la cuenta (cada regla se aplica a una sola cuenta de New Relic)
* Reglas de descarte basadas en NRQL para filtrar datos y atributos
* No se requiere configuración ni gestión de infraestructura
* Cobrado según los datos descartados (7.5 GB por aCCU) más los miles de millones de eventos escaneados (2.237 BSE por aCCU)

Las reglas de nube son mejores cuando:

* Desea reglas de descarte simples sin administrar la infraestructura de gateway
* Sus requisitos de procesamiento son modestos y no justifican un despliegue de gateway a gran escala
* Filtrado de menor volumen como un % de su ingesta total donde el precio de las reglas de la nube puede ser más rentable

Conozca más: [Crear reglas de nube](/docs/new-relic-control/pipeline-control/cloud-rules/create-pipeline-rules)

## Uso de ambos en conjunto

También puede usar reglas de gateway y de nube simultáneamente:

* Fuentes de telemetría (p. ej. agentes) pueden configurarse de manera diferente para enrutar a través de su gateway o no
* La estrategia de defensa en profundidad garantiza que los datos sensibles se descarten por completo
* Es conveniente gestionar algunas reglas a través de la API/UI en la nube mientras se utiliza el gateway para el procesamiento principal

## Tabla comparativa

| Característica                 | Reglas de la puerta de enlace                                                                | Reglas de la nube                                                        |
| ------------------------------ | -------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| **Ubicación de procesamiento** | Su infraestructura (Kubernetes)                                                              | Nube de New Relic                                                        |
| **Cuando procesa**             | Antes de que los datos salgan de su red                                                      | Después de que los datos llegan a New Relic                              |
| **Configuración requerida**    | Sí (clúster de Kubernetes, chart de Helm)                                                    | No                                                                       |
| **Método de configuración**    | YAML o UI                                                                                    | Interfaz de usuario o API de NerdGraph (basada en NRQL)                  |
| **Alcance de la regla**        | A nivel de organización                                                                      | Específico de la cuenta                                                  |
| **Capacidades**                | Transformar, descartar datos/atributos, muestrear por tasa                                   | Descartar datos/atributos                                                |
| **Tipos de datos admitidos**   | Métricas, Eventos, Logs, Trazas                                                              | Métricas, Eventos, Logs, Trazas                                          |
| **Ideal para**                 | Procesamiento de alto volumen, transformaciones, reducción de costos de egreso, cumplimiento | Reglas de descarte simples, bajo volumen, sin gestión de infraestructura |

Para obtener información detallada sobre precios y cálculos de costos, consulta [costos de Pipeline Control](/docs/new-relic-control/pipeline-control/costs).

## Requisitos previos

### Para reglas de la nube

* Una cuenta de New Relic en un rol con permisos RBAC para administrar las reglas de nube de Pipeline Control
* Comprensión de NRQL para crear reglas de filtrado

### Para la puerta de enlace

* Una cuenta de New Relic con permisos de administrador de productos de la organización o gerente de la organización

<Callout variant="important" title="importante">
  Se requieren los roles de usuario Administrador de productos de la organización o Gerente de la organización para:

  * Instalar puerta de enlace
  * Agregar clúster a una puerta de enlace
  * desplegar una configuración de puerta de enlace
  * Ver reglas de puerta de enlace
</Callout>

* Familiaridad con las operaciones y la gestión de Kubernetes
* Acceso de administrador al clúster de Kubernetes para implementar y gestionar recursos dentro de su clúster
* Licencias para crear y gestionar nombres de host DNS privados dentro de su red
* Licencias para crear y gestionar registros DNS
* Helm instalado localmente

<Callout variant="important">
  * Las reglas de control de pipelines solo se aplican a los datos que llegan desde el momento en que se crea la regla, no eliminan los datos que [ya se ingirieron](/docs/telemetry-data-platform/ingest-manage-data/manage-data/manage-data-retention#data-deletion).
  * Al crear reglas, usted es responsable de garantizar que las reglas identifiquen y descarten con precisión los datos que cumplan las condiciones que estableció. Usted también es responsable de monitorear la regla, así como los datos que divulga a New Relic.
</Callout>
