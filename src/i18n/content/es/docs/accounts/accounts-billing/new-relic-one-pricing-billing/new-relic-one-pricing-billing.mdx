Ya sea que esté pensando en usar New Relic o ya lo esté usando, la siguiente información es útil para comprender cómo New Relic fija el precio de los productos y factura su uso.

Para obtener más información, consulte las opciones de facturación que se enumeran a continuación y en nuestro [sitio web](https://newrelic.com/pricing).

/\* &lt;img title=&quot;Diagrama que muestra el modelo de precios, ediciones y programas de compra&quot; alt=&quot;Diagrama que muestra el modelo de precios, ediciones y programas de compra&quot; src=&quot;/images/accounts\_diagram\_pricing-diagram.webp&quot; /&gt; \*/

## Modelo de precios [#pricing-model]

Nuestro modelo de precios describe las capacidades existentes del producto e indica sus posibles opciones de facturación, como Data + usuario o Data + Core calcular para la observabilidad de todo el stack y Advanced calcular para la observabilidad inteligente. Comprender estos modelos de precios lo ayudará a elegir lo que mejor se adapte a las necesidades de su negocio.

<Callout title="Nota">
  El modelo [Core para calcular](/docs/licenses/license-information/product-definitions/new-relic-one-pricing-definitions/#compute-capacity-unit) precios basados en el consumo proporciona a toda su organización de ingeniería acceso ilimitado a la plataforma y observabilidad de todo el stack sin barreras de licencia por usuario. Dado que Core Calcular se encuentra actualmente en versión preliminar, si está interesado en probar el modelo de precios, debe comunicar con su ejecutivo de cuenta de New Relic.
</Callout>

<CollapserGroup>
  <Collapser id="data" title="Datos">
    Se refiere a la cantidad de datos ingeridos por New Relic, medida en [GB ingeridos](/docs/licenses/license-information/product-definitions/new-relic-one-pricing-definitions/#gb-ingested). Ofrecemos dos opciones para GB Ingested de la siguiente manera:

    * <b>Datos originales:</b> La opción de datos originales de New Relic proporciona un valor predeterminado de al menos 8 días de retención de datos y un límite máximo de consulta de 20 mil millones de puntos de datos por minuto.

    * <b>Data Plus:</b> la opción Data Plus de New Relic ofrece capacidades avanzadas como hasta 90 días de retención de datos extendida y un límite máximo de consulta aumentado de 60 mil millones de puntos de datos por minuto.

    Facturamos los datos por la cantidad de GB ingeridos, sin importar el tipo, lo que hace que su factura sea fácil de entender. Tienes acceso a 100 GB ingeridos por mes gratis. Para realizar un seguimiento de sus GB ingeridos, puede emplear nuestra página [Resumen de uso](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/usage-ui) o [ejecutar una consulta](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/usage-queries-alerts).

    ### ¿Qué se considera datos facturables? [#usage-calculation]

    En general, &quot;datos ingeridos&quot; se refiere a los datos almacenados en New Relic por su organización después de que aplicamos varias reglas de recorte y transformación de datos. En otras palabras, no es la cantidad de datos sin procesar enviados a New Relic, sino el tamaño de los datos que realmente se almacenan.

    En general, los datos empleados por nuestra función no cuentan para los GB ingeridos. Algunos ejemplos de datos de características y relacionados con ellas que no cuentan para GB ingeridos incluyen:

    * [Funcionalidad de alerta básica](/docs/alerts-applied-intelligence/new-relic-alerts/learn-alerts/alerts-concepts-workflow). Por ejemplo, `NrAiIncident`, un evento empleado para generar estas características, no cuenta para los GB ingeridos.
    * Datos creados a partir de datos brutos agregados (por ejemplo, [datos de infraestructura agregados](/docs/data-apis/manage-data/manage-data-retention/#infrastructure-data) o [datos métricos dimensionales](/docs/data-apis/understand-data/new-relic-data-types/#dimensional-metrics)).
    * [Métricas doradas](/docs/apis/nerdgraph/examples/golden-metrics-entities-nerdgraph-api-tutorial) (para más información sobre métricas doradas, consulte [las definiciones de entidad](https://github.com/newrelic/entity-definitions)).
    * Checks de monitoreo sintético. Para obtener detalles sobre la facturación de Sintéticos, [los límites y precios de Sintéticos](docs/synthetics/synthetic-monitoring/getting-started/monitor-limits/).
    * Datos de seguimiento de uso y facturación (por ejemplo, `NrConsumption`).
    * Datos relacionados con la administración de la cuenta (por ejemplo, `NrIntegrationError`, `NrAuditEvent`).

    Los GB ingeridos mensualmente se redondean al número entero más cercano. Por ejemplo, si su organización emplea 100,9 GB durante un mes, eso se cuenta como 100 GB.
  </Collapser>

  <Collapser id="user" title="Usuario">
    Un usuario en New Relic es un individuo que puede acceder y emplear la plataforma. El recuento de usuarios facturables es un factor de facturación.

    ### Usuario: Facturación y reglas [#billable-users]

    Son facturables en el momento en que se agregan como usuario facturable a una organización New Relic. Los siguientes son los tipos de usuarios de New Relic:

    * **Usuario básico:** como usuario básico, no se le cobrará y podrá configurar nuestras herramientas de observación, ejecutar consultas sobre datos y usar el dashboard personalizado (inicio rápido del dashboard por hasta 7 días). También tendrás acceso a algunas funciones de alerta básicas. Sin embargo, no puedes emplear nuestras experiencias seleccionadas, como la UI APM, la UI de monitoreo de browser o UI de monitoreo de aplicaciones móviles. Si bien su acceso no genera un cargo directo, puede impulsar la facturación por funciones basadas en el uso, incluido el complemento CCU.

    * **Usuario principal:** como usuario principal, obtiene todas las capacidades de usuario básico junto con acceso a poderosas funciones centradas en desarrolladores como New Relic CodeStream, Errors Inbox y nuestra UI de administración de logs.

    * **Usuario de plataforma completa:** Como usuario de plataforma completa, obtienes acceso a muchas de nuestras características de plataforma. Esto incluye nuestras experiencias UI observabilidad más seleccionadas, como APM, monitoreo de infraestructura, monitoreo de browser, monitoreo de celulares y monitoreo sintético.

    Para obtener más información, consulte [Tipos de usuario](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type/).

    <CollapserGroup>
      <Collapser id="user-count" title="Cómo se cuentan los usuarios facturables">
        La facturación a usuarios facturables se realiza por mes calendario. Para determinar el recuento de usuarios facturables de una organización en un mes calendario, contamos la cantidad de usuarios durante ese mes que tuvieron un <DNT>**billable user type**</DNT> de usuario de plataforma completa o de usuario principal. El <DNT>**billable user type**</DNT> de un usuario se define como el tipo de usuario más alto en el que se configuró un usuario durante un mes calendario. Empleamos la zona horaria UTC para definir el inicio y el final de un mes calendario.

        Por ejemplo, si un usuario se configura como usuario de plataforma completa en cualquier momento durante un mes calendario, su tipo de usuario facturable para ese mes será `full platform user` y no cambiará, incluso si cambia a una versión inferior más adelante en ese mes. Este es el caso incluso si ese usuario se cambia a un usuario de plataforma completa solo brevemente. El cambio se honrará el siguiente mes calendario y ese usuario no será contabilizado como usuario de la plataforma Full, a menos que cambie nuevamente posteriormente.

        Si planea agregar un usuario facturable o cambiar el tipo de usuario de su usuario, debe tener en cuenta estos consejos:

        * Si desea agregar un usuario facturable o actualizar un usuario, es mejor hacerlo a principios de mes.
        * Si deseas degradar a un usuario, probablemente querrás hacerlo a fin de mes.
        * Empleamos el tiempo UTC para determinar el inicio y el final de un mes. Esto significa que si, por ejemplo, estuvieras en la hora del Pacífico y quisieras cambiar a usuario a una categoría inferior el último día del mes, tendrías que hacer esos cambios antes de las 5:00 p. m., hora del Pacífico.

        Los usuarios de su organización con la misma dirección de email solo se cuentan como un usuario. Para obtener más información, consulte la sección [de seguimiento de usuarios](/docs/accounts/accounts-billing/account-setup/multiple-logins-found#user-records) de nuestra documentación.

        Si está en nuestro [modelo de usuario más nuevo](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models), los usuarios serán facturables cuando los administradores los agreguen como usuarios facturables. Incluso si un usuario facturable nunca inició sesión en New Relic y tiene una etiqueta `Pending invite` en la UI, aún es facturable (si está en nuestro [modelo de usuario original](/docs/accounts/original-accounts-billing/original-users-roles/users-roles-original-user-model), los usuarios `Pending invite` no son facturables).

        Para obtener más información sobre cómo agregar e invitar usuarios, consulte la sección [Agregar usuario](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-ui-and-tasks#add-users) de nuestra documentación.

        El costo de su usuario facturable depende de [la edición de precios](https://newrelic.com/pricing) de una organización (Estándar, Pro o Enterprise) o de cualquier precio personalizado que pueda tener con New Relic.

        Cuando una organización New Relic comienza a recibir facturas, prorratearemos su cantidad de usuarios facturables en función de su fecha de inicio dentro de ese mes. También prorratearemos tu último mes si cancelas tu subscripción.
      </Collapser>

      <Collapser id="user-downgrade-rules" title="Reglas de degradación de usuario">
        Las reglas sobre cuántas veces puedes degradar tu plan de usuario de la plataforma Full difieren según si tu [plan de uso](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing#usage-plans) es de pago por uso (PAYG) o un contrato comprometido.

        ### Reglas de degradación de pago por uso

        Para el plan de uso de pago por uso (PAYG), no existen reglas que limiten la degradación del usuario. Tenga en cuenta que el [cálculo de facturación](#user-count) puede afectar sus decisiones sobre cuándo actualizar o degradar el usuario.

        ### Reglas de degradación de contratos de compromiso

        Antes de actualizar o degradar a un usuario, debe comprender [cómo funciona la facturación de usuarios](#user-count).

        Para las organizaciones con contratos comprometidos, la regla que rige las actualizaciones y degradaciones de usuarios de la Plataforma Completa es la siguiente:

        <DNT>**If a Full Platform user is downgraded to a lower user type in a later month and then returned to a Full Platform user in a later month, and this downgrade/upgrade cycle happens twice in a contract year, that user will be billed as a Full Platform user for the remainder of the year, regardless of edits to their user type.**</DNT> No existen reglas que rijan los cambios de tipo de usuario que ocurren dentro de un mes debido a [cómo funciona la facturación de usuario](#user-count): el tipo de usuario facturable de un usuario para un mes calendario es el tipo de usuario más alto en el que está establecido durante ese mes.
      </Collapser>
    </CollapserGroup>

    ### Gestionar usuarios y costos [#manage]

    Para gestionar usuarios, consulte la sección [de administración de usuarios](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-ui-and-tasks) en nuestros documentos. Puede emplear la [UI de uso](/docs/accounts/accounts-billing/general-account-settings/introduction-account-settings/#pricing) para obtener una descripción general de su recuento de usuarios facturables. Si necesita más detalles que los que proporciona la UI, también puede ejecutar [una consulta NRQL relacionada con el uso](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/usage-queries-alerts).
  </Collapser>

  <Collapser id="core-compute-capacity-unit" title="Core Compute">
    Core Compute is precios basados en el consumo para la observabilidad de todas las capacidades de la stack con usuario ilimitado. Su factura se basa en [las unidades Core de capacidad de computación](/docs/licenses/license-information/product-definitions/new-relic-one-pricing-definitions/#compute-capacity-unit) (Core CCU) consumidas para ejecutar con éxito acciones como consulta, alertas, llamada API y carga de páginas. La tabla de modelos de precios indica para qué funcionalidades puede elegir Core calcular como modelo de precios alternativo al usuario.
  </Collapser>

  <Collapser id="advance-compute-capacity-unit" title="Advanced Compute">
    Calcular avanzado es un complemento a su plan Data + usuario o Data + Core Compute que proporciona precios basados en el consumo para la característica de observabilidad inteligente. Este complemento se mide en [unidades avanzadas de capacidad de computación](/docs/licenses/license-information/product-definitions/new-relic-one-pricing-definitions/#compute-capacity-unit) (aCCU). Se le facturará en función del uso que haga de las CCU avanzadas.

    Si las capacidades de observabilidad de todo el stack se combinan con cualquier característica de observabilidad inteligente, se le cobrará la unidad de capacidad de computación avanzada adecuada.

    <Callout variant="tip">
      Su uso de CodeStream bajo Calcular avanzado generará cargos de CCU avanzado, independientemente del tipo de usuario. Esto significa que el usuario básico puede acceder a la función Calcular característica avanzada e incurrir en cargos por CCU avanzadas.
    </Callout>
  </Collapser>
</CollapserGroup>

### Facturación de funciones adicionales [#add-on-billing]

Además de los factores de facturación principales de datos y usuarios facturables, también puedes usar complementos facturables opcionales para enriquecer tu experiencia con New Relic.

* **Advanced Compute**: este complemento para su plan Data + usuario o Data + Core Compute permite el acceso a la función de observabilidad inteligente.
* **Centro de datos de la UE para Datos Originales o Datos Plus**: este complemento se aplica a su opción de datos (Datos Originales o Datos Plus) cuando selecciona la Unión Europea como su región de datos, según disponibilidad.
* **Extended Retention for Original Data or Data Plus**: este complemento se aplica si excede el tiempo predeterminado durante el cual se conservan sus datos. Esto se aplica a todos sus datos, no solo a los logs, y es una buena opción si necesita realizar muchas consultas pequeñas o consultas sobre grandes volúmenes de datos.
* **Live Archives**: amplíe la duración del almacenamiento de logs hasta siete años. Live Archives también requiere Calcular avanzado.
* **New Relic Synthetic Checks**: este complemento se aplica si tus controles exceden la cantidad predeterminada de controles de monitoreo sintético.
