## Problema

No ves [datos AJAX](/docs/browser/new-relic-browser/browser-pro-features/ajax-page-identifying-time-consuming-calls) para la aplicación de tu navegador.

## Solución

Si su aplicación está instrumentada con [<InlinePopover type="browser" />](/docs/browser/new-relic-browser/installation-configuration/troubleshooting-browser-monitoring-installation)y está recopilando datos correctamente para otras [características Pro](/docs/browser/new-relic-browser/browser-pro-features), siga estos pasos:

<Steps>
  <Step>
    ## Verifique que la página esté instrumentada [#verify-page-intrum]

    Si está utilizando XMLHttpRequest, use la consola de desarrollo de su navegador para verificar que la página se haya instrumentado correctamente. Ingrese `XMLHttpRequest` y/o `fetch` en la consola. Si la instrumentación falló, verá algo como:

    ```js
    // XMLHttpRequest
    ƒ XMLHttpRequest() { [native code] }
    ```

    Si ve algo más, entonces la verificación ha tenido éxito.
  </Step>

  <Step>
    ## Verifique que la instrumentación de AJAX esté habilitada [#verify-instrum]

    La instrumentación AJAX no está incluida en el agente Lite.

    Para verificar la instrumentación de AJAX:

    1. Abra la herramienta para desarrolladores de la consola de su navegador y verifique el objeto `newrelic.initializedAgents`. Este objeto contiene una entrada para cada agente que se ejecuta en la página. Por lo general, solo hay un agente en una página.

    2. Inspeccione el objeto del agente inicializado en busca de un objeto `runtime`. Este objeto contiene información sobre el agente en ejecución, como `loaderType`. Si `loaderType` es `lite`, la instrumentación de AJAX no está incluida en el agente que se está utilizando.

    3. Si el `loaderType` no es `lite`, también puede revisar el agente inicializado en busca de los objetos `config` y `features`.

       * El objeto `config` contendrá la configuración de tiempo de ejecución del agente, incluido un objeto `ajax` con un booleano `enabled`.
       * El objeto `features` debe contener una entrada para cada característica inicializada por el agente, incluida una entrada `ajax`.

    4. Verifique que este objeto AJAX contenga un `enabled` booleano de `true` y una entrada `featAggregate`.

    Si alguno de estos elementos no es verdadero, es posible que la instrumentación AJAX no se haya inicializado debido a un cambio de configuración en New Relic en la entidad del navegador o posiblemente a un cambio manual en el objeto `NREUM.init` que se copió y pegó en el HTML.

    <Callout variant="important">
      `newrelic.initializedAgents` es la propiedad aceptada para comprobar la existencia del agente en una página, a menos que esté utilizando una versión antigua o no compatible del agente.
    </Callout>
  </Step>

  <Step>
    ## Verifique la configuración de la lista de denegación [#verify-deny-list]

    El agente se puede configurar para ignorar ciertos dominios y rutas al crear [el evento AjaxRequest](/attribute-dictionary/?event=AjaxRequest).

    Utilice la consola de desarrollo de su browser para comprobar las reglas actuales de la lista de denegación de AJAX ejecutando `newrelic.init`. Busque una propiedad `ajax` que contenga una matriz llamada `deny_list`.

    Si ve entradas en la lista de denegación que no le resultan familiares o que hacen que se filtren las llamadas AJAX, revise la documentación [del evento Filter AjaxRequest](/docs/browser/new-relic-browser/configuration/filter-ajax-request-events/) para obtener más orientación.
  </Step>

  <Step>
    ## Verifique el acceso a la red [#verify-network-access]

    Si el objeto está correctamente instrumentado, intente activar una llamada AJAX en su aplicación durante el monitoreo del tráfico de red en las herramientas para desarrolladores del navegador. Espere hasta un minuto y busque una llamada a `bam.nr-data.net/jserrors` con un parámetro `xhr`. Si la llamada falla, compruebe si hay problemas de red. Como alternativa, busque una solicitud a `bam.nr-data.net/events` con una carga que comience con `bel.7;2`.

    Si no ve esta llamada, si falla con un error no relacionado con el acceso a la red, o si tiene éxito pero aún no ve datos, continúe con la documentación de resolución de problemas.
  </Step>

  <Step>
    ## Revise cómo JSONP afecta a las requests [#jsonp]

    Si sus requests usan [JSONP](https://en.wikipedia.org/wiki/JSONP), estos requests no aparecerán en la [página de la UI de AJAX](/docs/browser/new-relic-browser/browser-pro-features/ajax-page-identifying-time-consuming-calls).

    Sin embargo, puede verlos como activos dentro del [rastreo de sesión](/docs/browser/new-relic-browser/browser-pro-features/session-traces-exploring-webpages-life-cycle).

    Si está utilizando el monitoreo de SPA, puede verlos en la pestaña <DNT>**AJAX**</DNT> de la página <DNT>**Page views**</DNT>.

    Requisitos para que JSONP sea reconocido:

    * Cada solicitud JSONP debe utilizar una función de devolución de llamada única. Las bibliotecas más populares (como jQuery) generan dinámicamente una función de devolución de llamada única para cada solicitud.
    * La cadena de consulta que contiene el nombre de la función de devolución de llamada debe denominarse `"callback"` o `"cb"` para que New Relic la reconozca. Este es el comportamiento predeterminado en la biblioteca más popular.
  </Step>

  <Step>
    ## Solucionar problemas de captura de carga [#payload-capture-issues]

    Si ve datos AJAX básicos pero la captura de carga no funciona como se esperaba, verifique estos problemas comunes:

    <CollapserGroup>
      <Collapser id="payload-not-appearing" title="La carga no aparece">
        Si ha habilitado la captura de carga pero no ve los nuevos atributos en sus eventos `AjaxRequest`:

        1. **Verifique el modo de captura**: asegúrese de que `capture_payloads` esté configurado en `failures` o `all` (no en `none`).
        2. **Verifique la versión del agente**: confirme que está ejecutando la versión 1.317.0 o superior del agente del browser. Puede verificar esto en la configuración del agente o mediante el atributo `agentVersion` en su telemetría.
        3. **Revisar la lista de denegación**: si el extremo está en la lista de denegación, no se capturarán datos (incluidas las cargas). Verifique la configuración de su lista de denegación en la configuración de la aplicación del navegador.
        4. **Comprobar la firma de error de GraphQL**: para el modo `failures` con códigos de estado 200, el agente busca indicadores de error en el cuerpo de la respuesta. Si su API de GraphQL usa un formato de error no estándar, es posible que no se capturen las cargas.
      </Collapser>

      <Collapser id="truncated" title="Las cargas están truncadas">
        Todos los atributos de carga tienen un límite estricto de 4096 bytes. Cuando los datos se truncan:

        * El agente reemplaza los últimos 4 bytes con `…` (puntos suspensivos con un espacio) para indicar el truncamiento
        * Aún puede consultar y buscar la porción capturada
        * Para la carga completa que supere los 4 KB, considere registrarla por separado o usar Session Replay para ver la actividad de red completa
      </Collapser>

      <Collapser id="sensitive-data" title="Los datos confidenciales son visibles en las cargas">
        Si aparece información confidencial en las cargas capturadas:

        1. **Agregue reglas de ofuscación**: cree patrones de expresiones regulares en la [configuración de ofuscación](/docs/browser/new-relic-browser/configuration/obfuscate-browser-agent-data) para enmascarar los datos antes de la ingesta.
        2. **Use patrones listos para usar**: el agente del browser incluye patrones predeterminados para tarjetas de crédito y SSN que se aplican a los eventos `AjaxRequest`.
        3. **Actualice la lista de denegación**: para los extremos que nunca deben capturarse, agréguelos a la lista de denegación para excluirlos por completo.
      </Collapser>

      <Collapser id="performance-impact" title="Impacto en el rendimiento">
        La captura de carga agrega datos a cada evento `AjaxRequest`, lo que puede aumentar:

        * **Ancho de banda de red**: las cargas se envían a New Relic como parte de la carga de recolección.
        * **Costos de ingesta de datos**: más datos de eventos dan como resultado un mayor consumo de datos.
        * **Sobrecarga del agente**: serializar y ofuscar las cargas agrega un tiempo de CPU mínimo.

        Para minimizar el impacto:

        * Comience con el modo `failures` para capturar solo casos de error.
        * Use reglas de ofuscación para reducir el tamaño de la carga. Evite patrones de ofuscación excesivos o complicados.
        * Monitoree su dashboard de ingesta de datos para rastrear los cambios de volumen.
      </Collapser>

      <Collapser id="content-type-filtering" title="Filtrado de tipo de contenido">
        Los cuerpos de la carga se capturan solo para los tipos de contenido legibles por humanos. Los tipos admitidos incluyen:

        * `text/*` (p. ej., `text/html`, `text/plain`)
        * `application/json`
        * `application/xml`
        * `application/xhtml+xml`
        * `application/ld+json`
        * `application/yaml`
        * `application/x-www-form-urlencoded`

        Los tipos de contenido binario (imágenes, videos, PDF) se excluyen automáticamente. Si no ve una carga para una solicitud específica, verifique el encabezado `Content-Type` para asegurarse de que esté en un formato compatible.
      </Collapser>
    </CollapserGroup>
  </Step>
</Steps>

<Callout variant="tip">
  Si alguno de estos pasos de resolución de problemas falla o aún tiene problemas con datos AJAX faltantes, obtenga soporte en [support.newrelic.com](https://support.newrelic.com).
</Callout>
