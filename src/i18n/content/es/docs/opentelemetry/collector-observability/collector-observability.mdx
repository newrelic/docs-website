Monitoree la salud y el rendimiento de su [colector de OpenTelemetry](https://opentelemetry.io/docs/collector/) con una experiencia de UI de APM dedicada. Cuando su colector falla o funciona incorrectamente, puede causar una interrupción de los datos de observabilidad, pérdida permanente de datos o información distorsionada. Por eso creamos Collector Observability: una experiencia de APM diseñada para el trabajo de streaming que realizan los recopiladores. Puede aprovechar la [telemetría interna](https://opentelemetry.io/docs/collector/internal-telemetry/) del recopilador para ver de un vistazo cómo se desempeña cada componente de su recopilador, de modo que pueda detectar problemas antes de que afecten su pipeline de observabilidad.

## Configurar el monitoreo del colector [#setup]

### Habilite la telemetría interna para su colector [#enable-telemetry]

De forma predeterminada, el recopilador no emite su [telemetría interna](https://opentelemetry.io/docs/collector/internal-telemetry/), por lo que deberá habilitarla primero.

#### Descargar el archivo de configuración

```bash
curl -L https://raw.githubusercontent.com/newrelic/nrdot-collector-releases/refs/tags/1.10.0/examples/internal-telemetry-config.yaml \
  --silent --output internal-telemetry-config.yaml
```

#### Establecer variables de entorno

* <DNT>`INTERNAL_TELEMETRY_NEW_RELIC_LICENSE_KEY`</DNT>: Clave de licencia de ingesta para la cuenta a la que se debe enviar la telemetría interna. Esta clave puede ser diferente a la clave que utiliza el colector para enviar datos regulares a New Relic, es decir, <DNT>`NEW_RELIC_LICENSE_KEY`</DNT> en el ejemplo a continuación.
* <DNT>`INTERNAL_TELEMETRY_SERVICE_NAME`</DNT>: El valor predeterminado es `otel-collector`, determina el nombre de la entidad en New Relic
* <DNT>`INTERNAL_TELEMETRY_OTLP_ENDPOINT`</DNT>: El valor predeterminado es EE. UU. `https://otlp.nr-data.net`; si está en la UE, establezca esto en `https://otlp.eu01.nr-data.net`

#### Ejecutar el colector con la configuración fusionada

Además de la configuración normal para componentes y pipelines (en el siguiente ejemplo `--config=/etc/nrdot-collector/config.yaml`), agregue un segundo argumento `--config` que combinará ambas configuraciones:

```bash
docker run \
  -e INTERNAL_TELEMETRY_NEW_RELIC_LICENSE_KEY='...' \
  -e NEW_RELIC_LICENSE_KEY='...' \
  -e INTERNAL_TELEMETRY_SERVICE_NAME='demo-collector' \
  -v './internal-telemetry-config.yaml:/etc/nrdot-collector/config-internal.yaml' \
  newrelic/nrdot-collector:1.10.0 --config=/etc/nrdot-collector/config.yaml \
  --config='/etc/nrdot-collector/config-internal.yaml'
```

<Callout variant="important">
  El orden de los argumentos de `--config` importa si tiene una configuración preexistente bajo el nodo `service::telemetry`. El colector utiliza una estrategia de &apos;el último gana&apos; a nivel de nodo al fusionar configuraciones y ciertas partes de la configuración (p. ej., listas, nodos hoja) no se pueden fusionar, por lo que se sobrescriben con el último argumento `--config`.
</Callout>

#### Alternativa (no recomendada para producción)

No recomendamos esto por razones de confiabilidad, pero para fines de prueba también puede hacer referencia a la configuración directamente y el colector la obtendrá al inicio:

```bash
docker run \
  -e INTERNAL_TELEMETRY_NEW_RELIC_LICENSE_KEY='...' \
  -e NEW_RELIC_LICENSE_KEY='...' \
  -e INTERNAL_TELEMETRY_SERVICE_NAME='demo-collector' \
  newrelic/nrdot-collector:1.10.0 --config=/etc/nrdot-collector/config.yaml \
  --config='https://raw.githubusercontent.com/newrelic/nrdot-collector-releases/refs/tags/1.10.0/examples/internal-telemetry-config.yaml'
```

### Agregar etiqueta de entidad [#add-tag]

La etiqueta <DNT>`newrelic.service.type: otel_collector`</DNT> actúa como una opción de participación en la experiencia a nivel de la interfaz de usuario. Seleccione una de las siguientes opciones:

* **Opción 1**: Use la configuración de ejemplo proporcionada arriba que contiene la configuración de la Opción 2.
* **Opción 2**: Agregue el argumento <DNT>`--config=yaml:service::telemetry::resource::newrelic.service.type: otel_collector`</DNT> al colector. Esto agrega el atributo como un atributo de recurso y New Relic realiza el etiquetado por usted en la ingesta. Si elimina esta opción, la etiqueta tardará un día en expirar.
* **Opción 3**: Agregar etiqueta mediante la UI de APM (parte superior de la página, junto al nombre de la entidad). También puede eliminar esto a través de la interfaz de usuario para volver a cambiar.

### Personalización de la configuración [#customize]

La configuración predeterminada expone variables de entorno adicionales de la forma `INTERNAL_TELEMETRY_...` para ajustar opciones comunes, como los niveles de detalle y el muestreo. Consulte la [configuración en sí](https://github.com/newrelic/nrdot-collector-releases/blob/main/examples/internal-telemetry-config.yaml) para obtener más detalles.

## Ver su colector en la interfaz de usuario [#view-ui]

Para ver la telemetría interna de su colector, navegue a <DNT>**APM &amp; Services &gt; Services - OpenTelemetry &gt; your\_collector\_name**</DNT> para explorar la entidad del colector.

## Facturación y sobrecarga de datos [#billing-overhead]

Collector Observability genera datos de telemetría adicionales que cuentan para su ingesta de datos. Entender el volumen de datos le ayuda a planificar los costos y optimizar su configuración.

### Datos generados [#data-generated]

La observabilidad del recopilador genera los siguientes tipos de telemetría:

* **Métricas**: Información a nivel de componente sobre receptores, procesadores y exportadores
* **Logs**: Logs muestreados con una sobrecarga mínima durante el funcionamiento normal
* **Trazas**: Soporte experimental opcional para el análisis profundo de pipelines

### Volumen de datos esperado [#expected-volume]

La cantidad de datos generados depende de la configuración de su colector:

* **Métricas de línea base**: Un colector con componentes estándar (receptor, procesador, exportador) genera aproximadamente **1-2 KB/minuto** de datos de métricas
* **Logs**: Con el muestreo predeterminado (1% de las declaraciones de logs), espere **0.5-1 KB/minuto** durante la operación normal. Durante errores o alta verbosidad, esto puede aumentar significativamente
* **Trazas** (opcional): Si se habilita, espere **5-10 KB/minuto** dependiendo de la complejidad del pipeline

Para un colector típico que procesa telemetría 24/7:

* **Ingesta mensual de métricas**: -2.5-5 MB
* **Ingesta mensual de logs**: -1.5-3 MB (funcionamiento normal)
* **Ingesta mensual de trazas**: -10-20 MB (si está habilitada)

### Controlar la sobrecarga de datos [#controlling-overhead]

Puede ajustar la generación de telemetría usando variables de entorno:

* <DNT>**`INTERNAL_TELEMETRY_METRICS_LEVEL`**</DNT>: Establecer en `none` para deshabilitar las métricas, `normal` (predeterminado), o `detailed`
* <DNT>**`INTERNAL_TELEMETRY_LOGS_LEVEL`**</DNT>: Establecer en `info` (predeterminado), `warn` o `error` para reducir el volumen de logs
* <DNT>**`INTERNAL_TELEMETRY_LOGS_SAMPLING`**</DNT>: Ajustar la tasa de muestreo (predeterminado: 1%)
* <DNT>**`INTERNAL_TELEMETRY_TRACES_ENABLED`**</DNT>: Establecer en `false` para deshabilitar las trazas (deshabilitado por defecto)

Para obtener detalles de configuración, consulte la sección [personalización de la configuración](#customize) más arriba.

## Limitaciones [#limitations]

* **Ciertas funciones comunes de APM** no se trasladan perfectamente al colector y su naturaleza de procesamiento de flujos, y se han ocultado de la página de Resumen y de la navegación lateral. Analizaremos cada uno de ellos para la Disponibilidad General a fin de determinar si podemos integrarlos y cómo hacerlo de la mejor manera. Los ejemplos son:

  * Puntuación Apdex
  * Métricas significativas de `apm.%`
  * Tasa de errores
  * Transacción

* **La telemetría del recopilador aún no es estable**:

  * La última versión admitida de telemetría emitida por los componentes del colector es la versión del componente en nuestras distribuciones NRDOT (consulte el [manifiesto](https://github.com/newrelic/nrdot-collector-releases/blob/main/distributions/nrdot-collector/manifest.yaml) como referencia).
  * Si la telemetría emitida cambia durante la Vista previa pública, nos reservamos el derecho de admitir solo la versión más reciente. Por ejemplo, admitimos <DNT>`http.client.request.duration`</DNT> (versión del recopilador &gt;=0.128.0, versión de NRDOT &gt;=1.2.0, nr-k8s-otel-collector &gt;=0.8.37) pero no <DNT>`http.request.duration`</DNT>.
  * Con la Disponibilidad general, comenzaremos a brindar retrocompatibilidad si los nombres de las métricas continúan cambiando.

* **Requisitos de formato de exportación**: La interfaz de usuario del recopilador espera telemetría en el formato exportado por el <DNT>`otlpexporter`</DNT>. No admite métricas exportadas a través de Prometheus. Por ejemplo, <DNT>`http.client.request.duration`</DNT> es compatible, pero <DNT>`http_client_request_duration`</DNT> no.

* **Métricas de componentes personalizados**: La telemetría interna que no aparece en la [documentación de telemetría interna](https://opentelemetry.io/docs/collector/internal-telemetry/) aún no se admite. Los componentes personalizados o [de contribución](https://github.com/open-telemetry/opentelemetry-collector-contrib) emiten métricas estándar, pero también pueden definir sus propias métricas. Seguimos trabajando en una forma de ayudarlo a obtener información de ellos sin crear dashboards personalizados.

* **Métricas doradas para contenedores OTel**: Aún no cuenta con soporte completo, lo que significa que algunas columnas en el panel de infraestructura podrían no completarse para los contenedores.
