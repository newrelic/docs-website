## Referencia completa de métricas

La integración recopila 30+ métricas de instancia en comandos, memoria, keyspace, conexiones y replicación — además de 9 métricas específicas del clúster para despliegues de Redis Cluster. Expanda las secciones a continuación para ver especificaciones detalladas de las métricas.

  Para obtener el catálogo completo de upstream, consulte la [documentación de redisreceiver](https://github.com/open-telemetry/opentelemetry-collector-contrib/blob/main/receiver/redisreceiver/documentation.md) de OpenTelemetry.
</Callout>

## Racimo métrico

Estas métricas solo están disponibles al monitorear Redis en modo Cluster. Requieren habilitar las métricas del clúster en el receptor de Redis (todas las métricas del clúster están deshabilitadas de forma predeterminada) o usar el enfoque del receptor de Prometheus con `redis_exporter --is-cluster`.

<table>
  <thead>
    <tr>
      <th style={{ width: "260px" }}>
        Métrica
      </th>

      <th>
        Descripción
      </th>

      <th style={{ width: "100px" }}>
        Tipo
      </th>
    </tr>
  </thead>

  <tbody>
    <tr>
      <td>
        `redis.cluster.state`
      </td>

      <td>
        Estado de salud general del clúster. 1 = OK (todas las ranuras servidas), 0 = FAIL (una o más ranuras no disponibles).
      </td>

      <td>
        Medidor
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.slots_assigned`
      </td>

      <td>
        Número de ranuras de hash asignadas a los nodos. Debe ser 16 384 para un clúster completamente operativo.
      </td>

      <td>
        Medidor
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.slots_ok`
      </td>

      <td>
        Número de ranuras que atienden requests activamente sin problemas.
      </td>

      <td>
        Medidor
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.slots_pfail`
      </td>

      <td>
        Número de ranuras en estado PFAIL (falla probable) — se sospecha que un nodo es inalcanzable pero aún no se ha confirmado.
      </td>

      <td>
        Medidor
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.slots_fail`
      </td>

      <td>
        Número de ranuras en estado FAIL confirmado — NO atienden tráfico.
      </td>

      <td>
        Medidor
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.known_nodes`
      </td>

      <td>
        Número total de nodos (maestros + réplicas) visibles en el clúster.
      </td>

      <td>
        Medidor
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.node.count`
      </td>

      <td>
        Número de nodos maestros que actualmente sirven ranuras de hash.
      </td>

      <td>
        Medidor
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.stats_messages_sent`
      </td>

      <td>
        Total de mensajes del bus del clúster enviados (ping, pong, meet, fail). El tráfico constante indica un protocolo gossip saludable.
      </td>

      <td>
        Acumulativo
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.stats_messages_received`
      </td>

      <td>
        Total de mensajes del bus del clúster recibidos de nodos pares. Una caída repentina indica una posible partición de la red.
      </td>

      <td>
        Acumulativo
      </td>
    </tr>
  </tbody>
</table>

<Callout variant="tip">
  Todos los nodos en un Cluster de Redis informan métricas de clúster idénticas (`CLUSTER INFO` devuelve los mismos datos desde cualquier nodo). Puede habilitar el monitoreo de clúster en todos los nodos para mayor resiliencia o en un solo nodo para minimizar los datos redundantes. Para el enfoque del receptor de Prometheus, use `redis_exporter --is-cluster`, que recopila automáticamente estas métricas.
</Callout>
