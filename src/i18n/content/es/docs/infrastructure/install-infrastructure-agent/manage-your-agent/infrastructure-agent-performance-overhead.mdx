El agente de infraestructura es una pieza de software liviana, diseñada para minimizar su impacto en el rendimiento de sus hosts. Sin embargo, la carga exacta varía dependiendo de la carga de trabajo de su host, particularmente de la cantidad de procesos que se ejecutan en el host. Esto se debe a que el agente recopila [datos](/docs/data-analysis/metrics/analyze-your-metrics/data-collection-metric-timeslice-event-data#event-data) detallados de cada proceso individual.

Como pauta general, New Relic ha recopilado puntos de referencia para algunos tipos comunes de hosts:

<CollapserGroup>
  <Collapser
    id="single-task"
    title="Host de tarea única de Linux"
  >
    El agente tiene una sobrecarga de rendimiento muy baja en un host clásico de una sola tarea. Por ejemplo, un servidor que ejecuta Apache, Unicorn o una única aplicación Java.

    Nuestro punto de referencia para este tipo de host se basa en un [Amazon EC2 t3.2xlarge](https://aws.amazon.com/ec2/instance-types/):

    * <DNT>**vCPUs**</DNT>: 8 vCPU

    * <DNT>**Memory**</DNT>: 32,0GB

    * <DNT>**Storage**</DNT>: 160,0GB

    * <DNT>**Operating system**</DNT>: CentOS 7

      Para este tipo de host clásico de tarea única, el uso típico es:

    * <DNT>**CPU**</DNT>: alrededor del 0,3%

    * <DNT>**Virtual memory**</DNT>: alrededor de 1GB

    * <DNT>**Resident memory**</DNT>: 25 a 35MB

    * <DNT>**Storage on disk**</DNT>: alrededor de 50 MB
  </Collapser>

  <Collapser
    id="docker-host"
    title="Host docker de Linux"
  >
    El agente tiene una sobrecarga de rendimiento muy baja en un host que ejecuta docker, y el uso exacto depende de la cantidad de contenedores docker que aloja su máquina y de si esos procesos son de larga o corta duración.

    Nuestro punto de referencia para este tipo de host se basa en un [Amazon EC2 t3.2xlarge](https://aws.amazon.com/ec2/instance-types/):

    * <DNT>**vCPUs**</DNT>: 8 vCPU

    * <DNT>**Memory**</DNT>: 32,0GB

    * <DNT>**Storage**</DNT>: 160,0GB

    * <DNT>**Number of containers**</DNT>: 25 contenedores, alrededor de 100 procesos de larga duración que se ejecutan en el contenedor

    * <DNT>**Operating system**</DNT>: CentOS 7

      Para este tipo de host Docker, el uso típico es:

    * <DNT>**CPU**</DNT>: alrededor del 0,8%

    * <DNT>**Virtual memory**</DNT>: alrededor de 1GB

    * <DNT>**Resident memory**</DNT>: 25 a 35MB

    * <DNT>**Storage on disk**</DNT>: alrededor de 50 MB
  </Collapser>

  <Collapser
    id="windows-host"
    title="Host de Windows"
  >
    El agente tiene una sobrecarga de rendimiento muy baja en un host típico de Windows que sirve aplicaciones web y ejecuta la stack de Windows/IIS.

    Nuestro punto de referencia para este tipo de host se basa en un [Amazon EC2 t2.small](https://aws.amazon.com/ec2/instance-types/):

    * <DNT>**vCPUs**</DNT>: 1

    * <DNT>**Memory**</DNT>: 2,0 GB

    * <DNT>**Storage**</DNT>: 30,0GB

    * <DNT>**Operating system**</DNT>: Servidor de Windows 2012 R2

      Para este tipo de host de Windows, el uso típico es:

    * <DNT>**CPU**</DNT>: 2 a 3%

    * <DNT>**Resident Memory**</DNT>: 30 MB

    * <DNT>**Storage on disk**</DNT>: alrededor de 50 MB
  </Collapser>

  <Collapser
    id="arm64"
    title="Anfitrión Linux ARM64"
  >
    El agente tiene una sobrecarga de rendimiento similar en un host ARM64 (Graviton 2) en EC2 en comparación con máquinas AMD64.

    El punto de referencia se basa en la instancia [Amazon EC2 t3.2xlarge vs. t4g.2xlarge](https://aws.amazon.com/ec2/instance-types/) .

    Amazon Linux 2 instancias EC2 con configuración predeterminada de agente de infraestructura:

    * <DNT>
        **CPU**
      </DNT>

      : alrededor del 0,1% en ARM frente al 0,13% en AMD

    * <DNT>
        **Virtual memory**
      </DNT>

      : aproximadamente 0,75 GB ARM frente a 1 GB AMD

    * <DNT>
        **Resident memory**
      </DNT>

      : ARM de 20 MB frente a AMD de 22 MB
  </Collapser>
</CollapserGroup>

Siempre estamos mejorando el rendimiento del agente de infraestructura. Si observa una sobrecarga inusualmente alta en el rendimiento del agente, obtenga asistencia en [support.newrelic.com](https://support.newrelic.com).

## Administrar datos [#manage-data]

Para saber cómo ajustar la cantidad de datos que nuestro monitoreo de infraestructura ingiere y reporta, consulte [Administrar datos de infraestructura](/docs/infrastructure/manage-your-data/data-instrumentation/manage-infrastructure-data-reporting).

## Utilización de recursos [#resource-utilization]

En los sistemas Linux, la infraestructura se instala con la configuración predeterminada para cada administrador de servicios admitido. Se aplica un límite de memoria de 1 Gigabyte. Considere revisar y ajustar la configuración predeterminada según su requisito del sistema.
