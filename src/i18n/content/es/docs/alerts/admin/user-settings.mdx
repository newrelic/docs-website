Para las alertas de New Relic, nuestra configuración de periodo de gracia controla cuánto tiempo debe pasar antes de que se le notifique sobre un problema. Nuestra configuración de problemas inactivos controla cuánto tiempo desea esperar antes de que un problema inactivo se cierre automáticamente.

Estas configuraciones se aplican a todos los datos que ingresan a su cuenta.

Para encontrar la configuración de tus alertas, ve a <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &gt; Alerts &gt; General**</DNT>.

## Política de periodo de gracia para cuentas que emplean decisiones [#grace-period]

Para las emisiones candidatas a nuestra lógica de correlación, puede establecer un período de gracia. El período de gracia es el tiempo entre el momento en que se crea un problema y el momento en que ese problema genera una notificación. Los períodos de gracia se utilizan para reducir la cantidad de notificaciones innecesarias.

Para cada nivel de prioridad, puede establecer el período de gracia de 0 a 20 minutos. Cuanto más largo sea el período de gracia, más tiempo tendrá nuestro sistema para analizar y correlacionar la nueva actividad con la actividad pasada.

Si hay alguna notificación para enviar, se envía cuando ha transcurrido el período de gracia.

Los valores predeterminados del período de calificación son:

* Baja: 15 minutos
* Medio: 10 minutos
* Alto: 5 minutos
* Crítico: 1 minuto

<DNT>
  **Note: Grace period is eligible only for users who use correlations**
</DNT>

## Ventana de problema de cierre automático [#auto-close-issue-window]

La configuración de cierre automático de problemas controla cuánto tiempo pasa antes de que un problema inactivo se cierre automáticamente. El rango es de 1 a 30 días y el valor predeterminado es 3 días.

<Callout variant="tip">
  Esta configuración está relacionada con la [configuración &quot;cerrar incidente abierto después de...&quot;](/docs/alerts-applied-intelligence/new-relic-alerts/alert-incidents/how-alert-condition-incidents-are-closed/#time-limit) para cada condición.

  Cuando los períodos de tiempo en estas dos configuraciones son diferentes, nuestro sistema utiliza el período de tiempo más corto, independientemente de la configuración. Por ejemplo, si el tiempo de cierre del incidente abierto es de 3 días y el tiempo de emisión inactiva es de 2 días, el problema (junto con todos los incidentes asociados) se cerrará en 2 días.
</Callout>

## Problema inactivo [#inactiv-issue]

Cuando un problema está inactivo, nuestro sistema no lo agregará ni correlacionará. Por ejemplo, si aparece un problema nuevo que es similar al problema inactivo, se trata como nuevo, en lugar de correlacionarlo con el problema inactivo.

Una emisión se considera inactiva cuando ha acumulado 5.000 incidentes.

Después de 30 días, un problema inactivo se cerrará automáticamente.

## Alertas entre cuentas [#cross-account-alerts]

La configuración de alertas entre cuentas le permite configurar una condición de alerta para enviar datos a una cuenta distinta de la que posee la condición.
