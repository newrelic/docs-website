Una vez que haya conectado su aplicación a New Relic y haya comenzado a explorar nuestros gráficos y dashboard, un buen siguiente paso es crear una alerta para mantener a su equipo actualizado sobre cualquier comportamiento inusual en sus datos. La capacidad <InlinePopover type="alerts" />eleva su experiencia New Relic desde una simple ingesta de datos hasta la adopción de medidas reflexivas y efectivas.

Aquí, lo guiaremos a través de los cinco pasos para crear su primera alerta para que pueda comenzar a aprender la característica de alerta de New Relic.

<Steps>
  <Step>
    ## Cree su condición de alerta a partir de un gráfico [#create-alert-condition]

    La forma más sencilla de comenzar con las alertas es crear una alerta a partir de un gráfico New Relic . Esta ruta es la misma que crear una condición de alerta NRQL desde cero, pero el gráfico ya tiene una consulta NRQL con la que puede trabajar.

    Una condición de alerta es esencialmente un contenedor que usted crea para definir qué condiciones deben cumplirse antes de que se le notifique cualquier comportamiento inusual. Para este ejemplo, creará una alerta que notifique a su equipo sobre cualquier problema de latencia con el tiempo de transacción web.

    Entonces, en este caso, si desea asegurarse de que las transacciones web nunca tarden más de 50 milisegundos, creará una condición de alerta para monitorear cuándo el tiempo de la transacción web supera los 50 milisegundos y crea un evento de alerta.

    <CollapserGroup>
      <Collapser id="select-your-query" title="Seleccione su consulta">
        Primero, vaya al gráfico etiquetado <DNT>**Web transactions time**</DNT> y haga clic en <DNT>**create an alert condition**</DNT>.

        <img title="Alert condition from a chart" alt="A screenshot depicting how to create an alert condition from a chart" src="/images/alerts_screenshot-crop_create-an-alert-condition-from-a-chart.webp" />

        <img title="Select your query" alt="A screenshot demonstrating how to choose your query" src="/images/alerts_screenshot-crop_select-your-query.webp" />
      </Collapser>

      <Collapser id="name-your-condition" title="Nombra tu condición">
        Es importante darle un nombre descriptivo a su condición de alerta. Supongamos que nombra esta condición <DNT>**response time**</DNT> y luego crea otra condición para una aplicación completamente diferente y etiqueta esa condición también como <DNT>**response time**</DNT> . Si esto ocurre, no podrá distinguir qué condición corresponde a cada aplicación. Por lo tanto, asegúrese de darle a su afección un nombre específico y único. En este caso, llamaríamos a esta condición <DNT>**Response time: Example-app**</DNT>.
      </Collapser>

      <Collapser id="define-your-signal" title="Define tu señal">
        Una vez que haya nombrado su condición, puede realizar cambios en la consulta NRQL si lo desea. Para su primera alerta, le recomendamos no ajustar la consulta de NRQL y dejar esta sección como está, pero si desea obtener más información sobre cómo usar NRQL para personalizar sus consultas, visite nuestros [documentos](/docs/query-your-data/nrql-new-relic-query-language/get-started/introduction-nrql-new-relics-query-language/).

        <img title="Define your signal" alt="A screenshot that shows you how to define your signal" src="/images/alerts_screenshot-crop_define-your-signal.webp" />
      </Collapser>
    </CollapserGroup>

  <Step>
    ## Establecer umbral para condición de alerta

    Si una condición de alerta es un contenedor, entonces los umbrales son las reglas que contiene cada condición de alerta. A medida que los datos fluyen hacia su sistema, la condición de alerta busca cualquier evento de alerta de estas reglas. Si la condición de alerta detecta datos provenientes de su sistema que cumplen con todas las condiciones que ha configurado, creará un evento de alerta. Un evento de alerta es una señal de que algo no está bien en su sistema y debe revisarlo.

    Su equipo está creando una condición de alerta para buscar problemas de latencia en el tiempo de transacción web. Ahora, creará las reglas que buscará esta condición.

    <CollapserGroup>
      <Collapser id="anomaly-threshold" title="Umbral de anomalía (recomendado)">
        Su equipo está creando esta condición de alerta para que se le notifique si el tiempo de transacción web demora más de lo habitual. Pero digamos que no le importa cuánto tiempo de transacción web sea más largo y solo quiere saber si el tiempo de transacción se está comportando de manera anormal. Para este caso de uso específico, recomendamos utilizar nuestro <DNT>**anomaly threshold**</DNT>. Nuestra detección de anomalías evalúa constantemente sus datos para comprender cómo se comporta normalmente su sistema. Al establecer un umbral de anomalía, puede utilizar nuestra detección de anomalías para alertar a su equipo si el tiempo de transacción web se desvía del rendimiento esperado. Dado que solo desea saber si el tiempo de transacción web se está comportando de manera inusual, seleccionará <DNT>**upper and lower**</DNT> porque desea recibir una notificación de *todas* las desviaciones. Pero, si solo desea alertar si el tiempo de transacción web demora *más* de lo habitual, seleccionará <DNT>**upper only.**</DNT>

        A continuación, debe establecer el nivel de prioridad. El nivel de prioridad determina qué creará un evento de alerta. Recomendamos configurar el nivel de prioridad como <DNT>**critical**</DNT> para su primera alerta. Puede obtener más información sobre los niveles de prioridad en nuestra [documentación de condiciones de alerta](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/advanced-techniques/set-thresholds-alert-condition#threshold-levels).

        A continuación, debe elegir qué define un <DNT>**critical anomaly threshold breach**</DNT>. Para esta primera alerta, recomendamos usar nuestra configuración predeterminada y ajustarla a sus necesidades según sea necesario. Por lo tanto, deje la configuración para abrir un evento de alerta &quot;cuando una consulta devuelve un valor que se desvía del valor predicho: <DNT>**for at least five minutes**</DNT> en <DNT>**3 standard deviations**</DNT>&quot;.

        Obtenga más información sobre la anomalía en nuestra [documentación](/docs/alerts-applied-intelligence/applied-intelligence/anomaly-detection/anomaly-detection-applied-intelligence/) sobre anomalías.

        <DNT>
          /* &lt;img title=&quot;Configure your anomaly thresholds&quot; alt=&quot;A screenshot demonstrating how to configure custom anomalies&quot; src=&quot;/images/alerts_screenshot-crop_configure-your-anomaly-thresholds.webp&quot; /&gt; */
        </DNT>
      </Collapser>

      <Collapser id="static-threshold" title="Umbral estático">
        A diferencia de los umbrales de anomalía, un umbral estático no analiza su conjunto de datos en su totalidad ni determina qué comportamiento es inusual basándose en el historial de su sistema. En su lugar, un umbral estático abrirá un evento de alerta siempre que su sistema se comporte de manera diferente a los criterios que estableció. Los umbrales de alerta estáticos son mucho más personalizables, y los recomendamos si tiene un conocimiento sólido de sus datos y de lo que está buscando.

        Obtenga más información sobre nuestra condición de alerta estática en nuestros [documentos NRQL](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/#threshold-types).
      </Collapser>

      <Collapser id="lost-signal" title="Umbral de señal perdida (opcional)">
        A veces, una señal entrante puede perderse, y es importante entender si se trata simplemente de un retraso o de un indicio de un problema mayor. Nuestro <DNT>**lost signal threshold**</DNT> nos permite saber cuántos segundos debe esperar el sistema desde el momento en que se detectó el último punto de datos antes de considerar que la señal se ha perdido. Si la señal no regresa antes del límite de tiempo que configuraste, tienes algunas opciones: abrir un nuevo evento de alerta, cerrar todos los eventos de alerta abiertos y/u omitir la apertura de un evento de alerta cuando se espera que la señal finalice. Haga clic [aquí](/docs/alerts/create-alert/create-alert-condition/create-nrql-alert-conditions/#signal-loss) para obtener más información sobre sus opciones de umbral de pérdida de señal.

        Establecer su umbral de señal perdida requiere conocimiento de su sistema y de lo que desea comprender. En el caso del tiempo de transacción web, digamos que New Relic recopila una señal cada minuto. Una señal perdida podría indicar un problema de latencia mucho mayor. Por lo tanto, le recomendamos configurar la hora según su nivel de comodidad y luego marcar la casilla <DNT>**open a new lost signal alert event**</DNT>.

        </DNT>

  <Step>
    ## Ajuste la configuración de señal avanzada [#advanced-signal-settings]

    New Relic observa constantemente los datos que fluyen desde su aplicación a nuestro sistema. Pero no todas las aplicaciones envían señales con la misma frecuencia o cadencia. Algunos eventos podrían enviar señales a nuestro sistema una vez por minuto, mientras que otros solo podrían reportar datos a New Relic una vez al día. Una condición de alerta es un contenedor <DNT>**specific**</DNT> diseñado para un caso de uso <DNT>**specific**</DNT> . Al crear una condición de alerta, esta sección es la más personalizable para los datos que está evaluando.

    Vamos a personalizar estas configuraciones de señal avanzadas para nuestra condición que busca problemas de latencia de transacciones web.

    <CollapserGroup>
      <Collapser id="window-duration" title="Duración de la ventana">
        <DNT>
          /* &lt;img title=&quot;Window duration&quot; alt=&quot;A screenshot depicting advanced custom settings for an alert condition.&quot; src=&quot;/images/alerts_screenshot-crop_window-duration.webp&quot; /&gt; */
        </DNT>

        Configurar la duración de la ventana para su condición de alerta le indica a New Relic cómo agrupar sus datos. Si está creando una condición de alerta para un conjunto de datos que envía una señal a New Relic una vez cada hora, querrá establecer la duración de la ventana en algo más cercano a sesenta minutos porque ayudará a detectar patrones y comportamientos inusuales. Pero, si está creando una condición de alerta para el tiempo de transacción web y New Relic recopila una señal para esos datos cada minuto, le recomendamos configurar la duración de la ventana en un minuto.

        Para su primera alerta, le recomendamos seguir con nuestra configuración predeterminada, pero cuanto más se familiarice con la creación de una condición de alerta, le recomendamos que personalice estos campos según su propia experiencia.
      </Collapser>

      <Collapser id="sliding-window" title="Usar agregación de ventanas deslizantes">
        A lo largo del día, los datos fluyen desde su aplicación a New Relic. En lugar de evaluar esos datos inmediatamente para eventos de alerta, las condiciones de alerta recopilan los datos durante un período de tiempo conocido como <DNT>**aggregation window**</DNT>. Un retraso adicional permite que los puntos de datos más lentos lleguen antes de que se agregue la ventana.

        Las ventanas deslizantes son útiles cuando necesitas suavizar gráficos &quot;puntiagudos&quot;. Un caso de uso común es utilizar ventanas deslizantes para suavizar gráficos de líneas que tienen mucha variación en períodos cortos de tiempo en los casos en que el agregado móvil es más importante que los agregados de ventanas de tiempo estrechas.

        Recomendamos utilizar nuestra agregación de ventana deslizante si no espera tener un flujo de datos constante y consistente, pero espera algunas caídas y picos en los datos.

        Puede obtener más información sobre la agregación de ventanas deslizantes en [este tutorial de NRQL](/docs/query-your-data/nrql-new-relic-query-language/nrql-query-tutorials/create-smoother-charts-sliding-windows/) o viendo este vídeo.

        <Video type="youtube" id="-5--8DZynFE" />
      </Collapser>

      <Collapser id="streaming-method" title="Método de transmisión">
        En general, recomendamos utilizar el método de transmisión <DNT>**event flow**</DNT> . Esto es mejor para los datos que ingresan a su sistema de manera frecuente y constante. Hay casos específicos en los que <DNT>**event timer**</DNT> podría ser un mejor método a elegir, pero para su primera alerta recomendamos nuestro método predeterminado, <DNT>**event flow**</DNT>. Para entender mejor qué método de streaming elegir, te recomendamos ver este breve vídeo.

        <Video type="wistia" id="n6nei987ln" />
      </Collapser>

      <Collapser id="delay" title="Demora">
        La función de retraso le protege contra la recolección de datos inconsistente. Le da a la condición de alerta un pequeño margen de maniobra antes de decidir crear un evento de alerta. Si en un minuto determinado sus datos llegan a New Relic con marcas de tiempo de un solo minuto, entonces una configuración de retraso bajo es suficiente. Por otro lado, si durante ese minuto New Relic recibe puntos de datos con marcas de tiempo de varios minutos atrás o varios minutos adelante, entonces su señal es más inconsistente y necesitará una configuración de retraso mayor.
      </Collapser>

      <Collapser id="gap-filling-strategy" title="Estrategia para llenar vacíos">
        Estamos creando una condición de alerta para notificar a nuestro equipo sobre cualquier problema de latencia con el tiempo de transacción web. En este caso, nuestra aplicación envía constantemente datos de New Relic. Hay un flujo constante de señales que se envían desde nuestra aplicación a New Relic y no se espera ninguna brecha en la señal, por lo que no necesitaremos seleccionar una estrategia para llenar las brechas. Para este caso de uso y para su primera alerta, recomendamos dejar <DNT>**gap-filling strategy**</DNT> configurado en <DNT>**none**</DNT>.

        Si tiene un conjunto de datos más inconsistente que envía señales a New Relic una vez cada veinticuatro horas, le recomendamos personalizar esta característica según las necesidades específicas de su equipo.

        Obtenga más información sobre las estrategias para llenar espacios en nuestros [documentos sobre señales perdidas](/docs/apis/nerdgraph/examples/nerdgraph-api-loss-signal-gap-filling/).
      </Collapser>
    </CollapserGroup>

  <Step>
    ## Conecte su condición a una póliza [#connect-condition-policy]

    Si hay algún problema de latencia con los tiempos de las transacciones web, nos gustaría recibir una notificación lo antes posible. La mejor acción rápida y eficiente es crear una condición de alerta que abrirá un evento de alerta si los tiempos de transacción web demoran demasiado.

    Esta condición de alerta es un contenedor que contiene todas las reglas: ¿estamos usando un umbral estático o de anomalía, estamos usando una agregación de ventana deslizante o simplemente dejamos el período de evaluación como de costumbre?

    En este punto del proceso, ya tenemos un contenedor completamente definido y hemos configurado todas las reglas para asegurarnos de que se abra un evento de alerta cuando lo deseemos. Según la configuración anterior, si nuestra condición de alerta reconoce este comportamiento en nuestro sistema que supera los umbrales que hemos establecido, creará un evento de alerta. Ahora, todo lo que tenemos que hacer es adjuntar este contenedor a una política.

    La política es el sistema de clasificación para el evento de alerta. Cuando creas una política, estás creando la herramienta que organiza todos tus eventos de alerta entrantes. Puede conectar políticas a <DNT>**[workflows](/docs/alerts/get-notified/alert-event-workflows/)**</DNT> que le indiquen a New Relic a dónde desea que vaya toda esta información entrante, con qué frecuencia desea que se envíe y a dónde.

    <CollapserGroup>
      <Collapser id="select-an-existing-policy" title="Seleccione una política existente">
        Si ya tiene una política que desea conectar a una condición de alerta, seleccione la política existente.

        Obtenga más información sobre cómo crear políticas [aquí](/docs/alerts/organize-alerts/specify-when-alerts-create-events).
      </Collapser>

      <Collapser id="create-a-new-policy" title="Crear una nueva política">
        Si queremos crear una nueva política para esta condición de alerta, esta es nuestra oportunidad. Recuerde que una política está conectada a flujos de trabajo, y los flujos de trabajo controlan la frecuencia con la que se nos notifica sobre cualquier evento de alerta. Es un delicado equilibrio entre garantizar que nos enteremos de cualquier problema con el tiempo de transacción web lo más rápido posible y asegurarnos de no recibir tantas alertas que nuestros desarrolladores experimenten fatiga y comiencen a perderse información importante debido a la sobrecarga de información.

        Las políticas pueden contener una o varias condiciones. Si está buscando monitor la latencia web de transacciones, tiene algunas opciones.

        Primero, puede crear una política que solo adjunte una única incidencia por política (la opción predeterminada). Un problema por política reduce el ruido, pero también requiere acción inmediata. Pero esto significa que si ha asociado múltiples condiciones a esta política, no solo a <DNT>**Response time: Example app**</DNT>, entonces, sin importar qué, todos los eventos de alerta en esta política se agruparán en una sola incidencia.

        O podríamos crear una incidencia por condición. Esto significa que cada vez que la condición <DNT>**Response time: Example app**</DNT> abra un evento de alerta, todos esos eventos de alerta se agruparán en un problema conectado a nuestra condición. Para este caso de uso específico, debe elegir esta opción porque cumple con el objetivo principal, que es monitorear los problemas de latencia con el tiempo de transacción web.

        O podríamos crear una incidencia para cada evento de alerta. Esta opción es la más ruidosa, pero puede funcionar bien si desea enviar información a un sistema externo.

        Obtenga más información sobre cómo crear políticas [aquí](/docs/alerts/organize-alerts/specify-when-alerts-create-events).
      </Collapser>
    </CollapserGroup>
  </Step>

  <Step>
    ## Ajustes adicionales [#additional-settings]

    <CollapserGroup>
      <Collapser id="close-open-alert events" title="Cerrar eventos de alerta abiertos">
        Un evento de alerta se cerrará automáticamente cuando la señal objetivo vuelva a un estado de no infracción durante el período de tiempo indicado en los umbrales de la condición. Este tiempo de espera se llama período de recuperación.

        Por ejemplo, si el comportamiento de infracción es &quot;el tiempo de transacción web es superior a .50 segundos al menos una vez en 5 minutos&quot;, entonces el evento de alerta se cerrará automáticamente cuando el tiempo de transacción web sea igual o inferior a .50 durante 5 minutos consecutivos.

        Cuando un evento de alerta se cierra automáticamente:

        1. La timestamp de cierre tiene una fecha retroactiva al inicio del período de recuperación.

        2. La evaluación se restablece y se reinicia desde el momento en que finalizó el evento de alerta anterior.

           Todas las condiciones tienen una configuración de límite de tiempo para eventos de alerta que forzará automáticamente el cierre de un evento de alerta de larga duración.

           Automáticamente utilizamos 3 días de forma predeterminada y le recomendamos que utilice nuestra configuración predeterminada para su primera alerta.
      </Collapser>

      <Collapser id="custom-alert-event-description" title="Enviar una descripción de evento de alerta personalizada">
        Dado que estamos creando una condición de alerta que nos permite saber si hay problemas de latencia con el tiempo de transacción web, queremos asegurarnos de que nuestros desarrolladores tengan toda la información que necesitan cuando se les notifique sobre este evento de alerta. Vamos a usar flujos de trabajo para notificar a un canal de Slack del equipo cuando se cree un evento de alerta.

        Obtén más información sobre las descripciones de eventos de alerta personalizadas en nuestra [documentación](/docs/alerts/create-alert/condition-details/custom-alert-event-descriptions).
      </Collapser>

      <Collapser id="runbook" title="Agregar URL del runbook">
        Si desea vincular a un runbook, puede colocar la URL en el campo URL del runbook.
      </Collapser>
    </CollapserGroup>
