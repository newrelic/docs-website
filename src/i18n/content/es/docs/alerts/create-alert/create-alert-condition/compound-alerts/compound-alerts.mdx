Cuando múltiples condiciones de alerta relacionadas se activan por separado, se crea ruido de alertas y se dificulta la identificación de problemas críticos. Si bien la característica de correlación existente de New Relic ayuda a reducir este ruido, es posible que necesite un control más preciso sobre cuándo se activan las alertas en función de combinaciones de condiciones específicas.

Las alertas compuestas le permiten combinar múltiples condiciones de alerta en una expresión de activación utilizando operadores lógicos <DNT>(AND, OR, NOT)</DNT>. Se activan solo cuando se cumplen combinaciones específicas de condiciones — o cuando una condición se activa pero otra no —, lo que le brinda menos notificaciones y más relevantes.

<img title="Overview of compound alerts" alt="A screenshot of how the compound alert builder works" src="/images/compound-alerts-overview.webp" />

Con <DNT>**Compound alerts**</DNT>, puedes:

* <DNT>**Reduce alert noise**</DNT>: reciba un único [evento de alerta compuesta](/docs/alerts/create-alert/create-alert-condition/compound-alerts/compound-alert-events) en lugar de alertas separadas de múltiples condiciones relacionadas, lo que reduce el volumen general de notificaciones.
* <DNT>**Combine up to five alert conditions**</DNT>: Seleccione hasta cinco condiciones de alerta existentes. Combínelas utilizando los operadores AND, OR y NOT con paréntesis para lógica compleja.
* <DNT>**Combine cross-account conditions**</DNT>: Incluya condiciones de alerta que apunten a datos de diferentes cuentas. Todas las condiciones de los componentes deben tener la misma cuenta propietaria, pero pueden consultar datos de otras cuentas.
* <DNT>**Control facet matching**</DNT>: Elija cómo manejar las facetas compartidas entre condiciones. La alerta compuesta puede activarse cuando el mismo valor de faceta cumple con el umbral en todas las condiciones, o activarse independientemente de los valores de faceta coincidentes.
* <DNT>**Silence component condition alert events**</DNT>: Configure los umbrales de las condiciones de alerta individuales para suprimir sus propios eventos de alerta cuando formen parte de una alerta compuesta. Esto reduce el ruido de las condiciones contribuyentes mientras mantiene activa la alerta compuesta.
* <DNT>**Add tags to compound conditions**</DNT>: Organice y filtre las condiciones de alerta compuesta utilizando etiquetas personalizadas, al igual que las condiciones de alerta estándar. Las etiquetas le ayudan a categorizar las alertas y agilizar la clasificación de incidentes.

## Casos de uso [#use-cases]

Aquí hay algunos casos de uso en los que podría usar alertas compuestas:

* <DNT>**Detect memory leaks**</DNT>: Alerta cuando la memoria &gt; 50 % (infraestructura) AND el rendimiento &lt; 1k (APM), lo que ayuda a identificar situaciones en las que la memoria aumenta sin el tráfico correspondiente.
* <DNT>**Database performance issues**</DNT>: Alerta cuando el tiempo de respuesta de la base de datos &gt; 2s AND (uso de CPU &gt; 80 % OR grupo de conexiones &gt; 90 %) AND NOT durante la ventana de mantenimiento, lo que ayuda a identificar la degradación del rendimiento y evita el ruido durante el mantenimiento planificado.

## Permisos y control de acceso [#permissions]

Las alertas compuestas utilizan el mismo modelo de permisos que las condiciones de alerta estándar:

* Los usuarios que pueden crear condiciones de alerta estándar también pueden [crear condiciones de alerta compuestas](/docs/alerts/create-alert/create-alert-condition/compound-alerts/create-compound-alert).
* Todos los usuarios con acceso a la cuenta pueden ver y administrar las condiciones de alerta compuestas en esa cuenta.

## Limitaciones [#limitations]

Cuando utilice alertas compuestas, tenga en cuenta estas limitaciones:

* <DNT>**Maximum conditions**</DNT>: Puede combinar hasta cinco condiciones en una sola alerta compuesta.
* <DNT>**No nesting**</DNT>: No puede usar una condición de alerta compuesta como condición de componente en otra alerta compuesta.
* <DNT>**Account scope**</DNT>: todas las condiciones de los componentes deben pertenecer a la misma cuenta propietaria (aunque pueden tener como objetivo datos de otras cuentas a través de condiciones entre cuentas).
* <DNT>**Facet matching**</DNT>: Cuando las condiciones están facetadas por múltiples atributos, se produce una coincidencia cuando al menos un atributo es el mismo (actualmente no se admite el control granular).
