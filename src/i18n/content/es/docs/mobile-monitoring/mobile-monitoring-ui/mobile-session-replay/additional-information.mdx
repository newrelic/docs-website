## Session Replay y el rendimiento de tu app [#app-performance]

Antes de habilitar Session Replay, considera el siguiente impacto en tu aplicación y usuarios:

* **Rendimiento de la aplicación**: Session Replay captura y transmite datos del viewport, lo que puede afectar el rendimiento de la aplicación en dispositivos más antiguos.
* **Consumo de datos**: la transmisión de datos de reproducción consumirá datos del usuario, especialmente cuando se capturan imágenes.
* **Requisitos de almacenamiento**: los datos capturados requieren una capacidad de almacenamiento adecuada en los dispositivos del usuario entre los ciclos de transmisión.
* **Optimización de red**: Todos los datos se comprimen del lado del cliente antes de la transmisión para minimizar el uso de la red.
* **Estrategia de muestreo**: Considere cuidadosamente sus tasas de muestreo para equilibrar las necesidades de recopilación de datos con el impacto en el rendimiento.

## Consumo de datos [#data-consumption]

Session Replay sigue el mismo precio de consumo que tus otros bytes móviles. La cantidad de bytes producidos depende del número, la longitud y los niveles de actividad del usuario de las sesiones, así como de la complejidad de tu aplicación.

La forma más precisa de proyectar su costo por repetición es habilitar la característica durante un corto período de prueba y medir su consumo real. Una fórmula sencilla para proyectar costos aproximados es:

```
sessions x sampling_rate x gb_per_replay x cost_per_gb
```

Por ejemplo:

1 millón (sesiones) × 5 % (tasa de muestreo) × 0.0053 (GB por repetición) × $0.40 (costo por GB) = **$106 por 50 000 repeticiones**

Para controlar su consumo, ajuste sus [tasas de muestreo](/docs/mobile-monitoring/mobile-monitoring-ui/mobile-session-replay/configuration/configure-sampling-rates).

## Preocupaciones de seguridad y privacidad del usuario [#data-security]

Priorizamos la privacidad de los datos y desarrollamos Session Replay con configuraciones predeterminadas sólidas para minimizar la posible exposición de información confidencial de los clientes. Puede personalizar aún más la [configuración de privacidad](/docs/mobile-monitoring/mobile-monitoring-ui/mobile-session-replay/configuration/configure-privacy-settings) para satisfacer sus necesidades específicas.

Session Replay no captura videos y, por defecto, las imágenes se bloquean y se reemplazan por marcadores de posición del mismo tamaño. En su lugar, registra y reconstruye las interacciones del usuario. Esto reduce significativamente la cantidad de datos almacenados.

### Cumplimiento de las leyes de privacidad de datos [#compliance]

New Relic actúa como procesador de datos según GDPR y proveedor de servicios según CCPA. Esto significa que procesamos datos de acuerdo con sus instrucciones. En última instancia, el cumplimiento de estas leyes depende de cómo implemente Session Replay dentro de su aplicación.

Por defecto, todo el texto y las entradas del usuario, incluidos los datos personales, se enmascaran dentro de las grabaciones. Puede desenmascarar selectivamente elementos específicos si es necesario, pero el valor predeterminado garantiza que ninguna información de identificación personal sea fácilmente visible.

## Marco de UI compatible [#supported-frameworks]

Actualmente, Session Replay es compatible con determinados frameworks de UI para aplicaciones iOS, Android y React Native. Estamos trabajando activamente para dar soporte a más UI y frameworks híbridos.

**Actualmente compatible:**

* iOS UIKit
* iOS SwiftUI
* Diseños XML de Android
* Android Jetpack Compose
* Vistas de React Native

**Apoyo previsto:**

* Vistas sitio web
* Flutter
* Cordova
* Capacitor
* MAUI

<Callout variant="tip">
  Es posible que algunos elementos no estén disponibles. No hay un elemento HTML nativo &quot;Switch&quot;. Los elementos de cámara, mapas y video no se capturan en las repeticiones — se muestran como espacio en blanco o con un SVG de marcador de posición. Comuníquese con soporte si falta algo o no se muestra correctamente.
</Callout>
