Los perfiles definen los OID SNMP que `ktranslate` recopila y cómo procesa los mensajes de captura SNMP recibidos. Siga las instrucciones aquí si desea modificar un perfil existente o crear uno nuevo.

## Requisitos [#requirements]

Ya sea que esté creando o modificando un perfil, asegúrese de tener lo siguiente:

* Una [cuenta de GitHub](https://docs.github.com/en/get-started/signing-up-for-github/signing-up-for-a-new-github-account) donde puedes bifurcar el repositorio [kentik/snmp-profiles](https://github.com/kentik/snmp-profiles) y crear \[la] solicitud de extracción para tus cambios.
* Un [recorrido SNMP](/docs/network-performance-monitoring/troubleshooting/snmp-walk/#capturing-snmpwalk) desde un dispositivo de red para sondear perfiles.
* Para los mensajes de captura SNMP, también debe tener acceso a la MIB del proveedor que define los mensajes.

<Callout variant="important">
  New Relic no proporciona soporte directo para el contenido de [kentik/snmp-profiles](https://github.com/kentik/snmp-profiles). Si un perfil SNMP existente devuelve datos incorrectos u otro comportamiento de interrupción, abra un problema en el repositorio [kentik/snmp-profiles](https://github.com/kentik/snmp-profiles) . No podemos proporcionar cronogramas para la modificación o creación de perfiles SNMP en terceros para `ktranslate`.

  Dependiendo del nivel de servicio asociado con su cuenta de New Relic, puede ser elegible para recibir ayuda del Soporte técnico global de New Relic. De acuerdo con nuestras [Ofertas de soporte técnico global](/docs/licenses/license-information/general-usage-licenses/global-technical-support-offerings/#tstools), haremos todo lo posible para determinar si un problema está relacionado con los productos de New Relic o si es causado por algo fuera de nuestro control y competencia. Para problemas con herramientas de terceros que están fuera de nuestro control, haremos todos los esfuerzos comercialmente razonables para contribuir con mejoras, parches u otros desarrollos funcionales al proyecto de código abierto ascendente.
</Callout>

    ## Fondo

    Cuando cree perfiles de sondeo SNMP, seguirá estos pasos generales:

    1. Recopile un [recorrido SNMP](/docs/network-performance-monitoring/troubleshooting/snmp-walk/#capturing-snmpwalk) desde un dispositivo de muestra.

    2. Identifique qué MIB son compatibles con el dispositivo, según los resultados de la caminata.

    3. En las MIB admitidas, identifique qué métrica le gustaría recopilar, como las siguientes:

       * OID escalares con contador, medidor o métrica enumerada
       * OID tabulares con contador, medidor o métrica enumerada
       * OID en formato escalar o tabular que devuelven valores de cadena que puede usar como etiqueta

    4. Cree o edite un perfil SNMP.

             El primer elemento a inspeccionar es el resultado de un recorrido SNMP. Esto se considera la "fuente de verdad" para su dispositivo y los resultados le muestran qué métrica específica realmente admite su dispositivo en particular.

             <Callout variant="tip">
               Es común encontrar OID en un archivo MIB que parecen prometedores, sólo para descubrir que su dispositivo no los admite. El proveedor del dispositivo controla explícitamente la compatibilidad con MIB y los OID individuales dentro de un archivo MIB. Es posible que haya opciones de configuración disponibles para agregar MIB adicionales a su dispositivo, pero debe consultar la documentación del proveedor para investigar esto.

           <TabsPageItem id="snmp-yml">
             Cada perfil SNMP utilizado por `ktranslate` es un archivo YAML. Puede seguir este tutorial utilizando el perfil [apc_ups.yml](https://github.com/kentik/snmp-profiles/blob/main/profiles/kentik_snmp/apc/apc_ups.yml) existente, que consta de estas secciones:

             <CollapserGroup>
               <Collapser
                 id="extends"
                 title="sección 'extiende'"
               >
                 Esta sección muestra la herencia de otros perfiles. Esto se utiliza para reducir la necesidad de agregar OID individuales desde MIB de alto nivel a cada perfil. Se pueden encontrar ejemplos comunes en el directorio [\_general](https://github.com/kentik/snmp-profiles/tree/main/profiles/kentik_snmp/_general) del repositorio.

                 La sintaxis aquí es una lista de nombres de archivos a los que desea hacer referencia.

                 ```yaml
                 extends:
                   - system-mib.yml
                   - ups-mib.yml
                 ```
               </Collapser>

               <Collapser
                 id="provider"
                 title="sección 'proveedor'"
               >
                 Esta sección identifica el valor de `provider` que será decorado como atributo para toda carga enviada a New Relic.

                 La sintaxis aquí es un valor de cadena que usa [kebab-case](https://www.freecodecamp.org/news/snake-case-vs-camel-case-vs-pascal-case-vs-kebab-case-whats-the-difference/#kebab-case).

                 ```yaml
                 provider: kentik-ups
                 ```

                 <Callout variant="tip">
                   El valor de `provider` es fundamental para [la síntesis de entidades](/docs/new-relic-solutions/new-relic-one/core-concepts/what-entity-new-relic/#entity-synthesis) en New Relic. Puedes encontrar más detalles sobre esto en la [documentación sobre configuración del dispositivo](/docs/network-performance-monitoring/advanced/advanced-config/#devices). Si está utilizando sus propios perfiles, deberá realizar una de las siguientes acciones:

                   * Utilice un valor existente que ya esté asignado a una entidad.
                   * Cree una solicitud de extracción \[la] al repositorio [newrelic/entidad-definitions](https://github.com/newrelic/entity-definitions) para asignar el valor de su proveedor a una definición existente o crear una nueva definición.
                   * Cree su propia experiencia personalizada en el panel de control para ver sus datos.
                 </Callout>
               </Collapser>

               <Collapser
                 id="sysobjectid"
                 title="sección 'sysobjectid'"
               >
                 Esta sección identifica el [sysObjectID](https://oidref.com/1.3.6.1.2.1.1.2) que `ktranslate` utiliza para asignar dispositivos a este perfil durante [el descubrimiento SNMP](/docs/network-performance-monitoring/troubleshooting/snmp-discovery-unexpected-profile/).

                 La sintaxis aquí es una lista de OID. Se proporciona compatibilidad con una coincidencia comodín utilizando el símbolo `*` para dar cuenta de grandes grupos de dispositivos sin anotarlos individualmente. Además, agregar un comentario YAML en línea aquí permitirá a `ktranslate` establecer el valor de `tags.kentik.model` que se decorará en la carga enviada a New Relic.

                 ```yaml
                 sysobjectid:
                   - 1.3.6.1.4.1.318.1.3.17    # APC SmartUPS 3 Phase
                   - 1.3.6.1.4.1.318.1.3.*     # APC UPS
                   - 1.3.6.1.4.1.318.*         # APC device
                 ```
               </Collapser>

               <Collapser
                 id="matches"
                 title="sección 'partidos'"
               >
                 Esta sección proporciona un segundo nivel opcional de filtrado que se utilizará con el mapeo `sysobjectid` durante el descubrimiento. En los casos en los que se utiliza un único SysObjectID para varios dispositivos, puede aislar aún más su dispositivo con una expresión regular que coincida con el contenido del OID [SysDescr](https://oidref.com/1.3.6.1.2.1.1.1) . La expresión coincidirá con una versión forzada en minúsculas de la cadena `SysDescr` . Puede ver un ejemplo en el perfil [net-snmp.yml](https://github.com/kentik/snmp-profiles/blob/main/profiles/kentik_snmp/_general/net-snmp.yml) .

                 La sintaxis aquí es un diccionario de expresiones regulares, entre comillas dobles como clave, con un valor asociado del nombre del archivo de perfil que debe asignarse cuando se realiza una coincidencia.

                 ```yaml
                 matches:
                   "^pfsense": pf_sense.yml
                 ```
               </Collapser>

               <Collapser
                 id="bulkwalk"
                 title="Sección 'no_use_bulkwalkall'"
               >
                 Esta sección proporciona una opción para deshabilitar la operación [snmpbulkwalk](http://www.net-snmp.org/docs/man/snmpbulkwalk.html) que `ktranslate` utiliza como estrategia de recopilación predeterminada. Establecer esta clave en `true` obligará a `ktranslate` a usar `GET`\|`GETNEXT`\|`GETTABLE` operaciones individualmente, lo cual es útil en algunos dispositivos más antiguos que pueden no admitir una operación masiva.

                 ```yaml
                 no_use_bulkwalkall: true
                 ```
               </Collapser>

                 id="metrics"
                 title="sección métrica"
               >
                 Esta sección describe los OID escalares y tabulares a los que `ktranslate` apuntará cuando utilice este perfil. La sintaxis de los OID escalares y tabulares varía ligeramente y se describirá en detalle en sus respectivas pestañas. En un nivel alto, la sintaxis aquí es una lista de OID con varias configuraciones que le permiten controlar completamente su colección.

                 ```yaml
                 metrics:

                   - MIB: $MIB_NAME
                     symbol:
                       OID: $SCALAR_OID

                   # Tabular OIDs
                   - MIB: $MIB_NAME

                       - OID: $COLUMN_OID
                         name: $TABULAR_OID_NAME_FROM_MIB
                 ```
               </Collapser>

               <Collapser
                 id="metric-tags"
                 title="Sección 'metric_tags'"
               >
                 Esta sección describe el atributo que se decorará según la métrica asociada. La dimensión métrica debe ser del tipo `count` \| `summary` \| `gauge`. Como tal, los valores de cadena no se admiten directamente como métricas y deben decorarse en una métrica existente utilizando el identificador `metric_tag` .

                 Este identificador se puede colocar en configuraciones de tablas MIB individuales, así como en la configuración del perfil global. Los casos de uso serían el [upsOutletGroupStatusName](https://oidref.com/1.3.6.1.4.1.318.1.1.1.12.1.2.1.2) para un grupo de salidas en la tabla `upsOutletGroupStatusTable` o el [upsAdvIdentSerialNumber](https://oidref.com/1.3.6.1.4.1.318.1.1.1.1.2.3) de su dispositivo UPS representado como un OID escalar.

                 ```
                 metrics:
                   - MIB: PowerNet-MIB_UPS
                     table:
                       OID: 1.3.6.1.4.1.318.1.1.1.12.1.2
                       name: upsOutletGroupStatusTable
                     symbols:
                       - ...
                     metric_tags:
                       - column:
                           OID: 1.3.6.1.4.1.318.1.1.1.12.1.2.1.2
                           name: upsOutletGroupStatusName
                 metric_tags:
                   - column:
                       OID: 1.3.6.1.4.1.318.1.1.1.1.2.3.0
                       name: upsAdvIdentSerialNumber
                 ```
               </Collapser>

                 id="conversions"
                 title="Conversiones métricas disponibles"
               >

                 <CollapserGroup>
                   <Collapser
                     id="tag"
                     title="etiqueta"
                   >
                     ```yaml
                     metrics:
                       ...
                         tag: $YOUR_TAG
                     ```

                     Se utiliza para anular el atributo `name` y presentar un nombre descriptivo que se enviará con la carga útil enviada a New Relic.
                   </Collapser>

                   <Collapser
                     id="enum"
                     title="enumeración"
                   >
                     ```yaml
                     metrics:
                       ...
                         enum:
                           up: 1
                           down: 2
                           testing: 3
                     ```

                     Se utiliza para manejar enumeraciones SNMP que convierten el valor entero de una métrica dimensional en el valor enumerado en un atributo decorado en la métrica dimensional (usando el mismo sufijo de nombre de métrica). Un ejemplo común es la conversión de [kentik.snmp.if_AdminStatus](https://oid-rep.orange-labs.fr/get/1.3.6.1.2.1.2.2.1.7) al valor enumerado de [if_AdminStatus](https://github.com/kentik/snmp-profiles/blob/main/profiles/kentik_snmp/_general/if-mib.yml#L59-L66) como `up`, `down` o `testing`.
                   </Collapser>

                   <Collapser
                     id="conversion-hextoint"
                     title="conversión: hextoint"
                   >
                     ```yaml
                     metrics:
                       ...
                         conversion: hextoint:$CURRENT:$DESIRED
                     ```

                     Se utiliza para convertir valores hexadecimales a formato entero. Opciones para <DNT>**current**</DNT>: `LittleEndian` \| `BigEndian`. Opciones para <DNT>**desired**</DNT>: `uint16` \| `uint32` \| `uint64`
                   </Collapser>

                   <Collapser
                     id="conversion-hextoip"
                     title="conversión: hextoip"
                   >
                     ```yaml
                     metrics:
                       ...
                         conversion: hextoip
                     ```

                     Se utiliza para convertir valores hexadecimales en cadenas IPv4 de 4 octetos.
                   </Collapser>

                   <Collapser
                     id="conversion-hwaddr"
                     title="conversión: hwaddr"
                   >
                     ```yaml
                     metrics:
                       ...
                         conversion: hwaddr
                     ```

                     Se utiliza para convertir valores hexadecimales en cadenas de direcciones MAC.
                   </Collapser>

                   <Collapser
                     id="conversion-powerset-status"
                     title="conversión: powerset_status"
                   >
                     ```yaml
                     metrics:
                       ...
                         conversion: powerset_status
                     ```

                     Se utiliza para la enumeración de la cadena ASCII [upsBasicStateOutputState](https://oid-rep.orange-labs.fr/get/1.3.6.1.4.1.318.1.1.1.11.1.1) en `POWERNET-MIB`.
                   </Collapser>

                   <Collapser
                     id="conversion-regexp"
                     title="conversión: expresión regular"
                   >
                     ```yaml
                     metrics:
                       ...
                         conversion: regexp:$YOUR_REGEX_PATTERN
                     ```

                     Coloca una coincidencia de expresiones regulares en la salida OID para capturar subcadenas; debe estar entre comillas y tener barras invertidas.
                   </Collapser>

                   <Collapser
                     id="conversion-to-one"
                     title="conversión: to_one"
                   >
                     ```yaml
                     metrics:
                       ...
                         conversion: to_one
                     ```

                     Se utiliza para crear un medidor métrico con el valor de `1` para sondear OID escalares no numéricos que no tienen opciones de enumeración. Un ejemplo es el OID [tlUpsTestResultsDetail](https://oid-rep.orange-labs.fr/get/1.3.6.1.4.1.850.100.1.7.2) que devuelve un valor del tipo [DisplayString](https://www.circitor.fr/Mibs/Html/S/SNMPv2-TC.php#DisplayString).
                   </Collapser>
                 </CollapserGroup>
               </Collapser>
             </CollapserGroup>

## Utilice perfiles personalizados de forma privada [#private]

El contenido del repositorio [kentik/snmp-profiles](https://github.com/kentik/snmp-profiles) se carga en `ktranslate` de forma predeterminada cuando crea el contenedor o servicio y, por lo tanto, se aplica globalmente a todas las instancias.

Para admitir opciones de personalización completas para que pueda mantener sus propios patrones y configuraciones de colección, haga lo siguiente:

1. Clona el repositorio.
2. Realice los cambios que considere oportunos.
3. Cargue su directorio de perfil personalizado en su contenedor docker o servicio de Linux al inicio.

La forma en que esto se hace es asignando sus archivos personalizados a la instancia de KTranslate dentro del directorio `etc/ktranslate/profiles/` .

Suponiendo que se encuentra en un directorio justo encima de `snmp-profiles`, el comando para agregar se ve así:

```shell
`pwd`/snmp-profiles/profiles:/etc/ktranslate/profiles
```

Para Docker, agregarías esto como un [montaje de volumen](https://docs.docker.com/storage/volumes/#start-a-container-with-a-volume), que sería similar a este:

```shell
docker run -d --name ktranslate-$CONTAINER_SERVICE --restart unless-stopped --pull=always -p 162:1620/udp \
  -v `pwd`/snmp-base.yaml:/snmp-base.yaml \
  -v `pwd`/snmp-profiles/profiles:/etc/ktranslate/profiles
  -e NEW_RELIC_API_KEY=$NR_LICENSE_KEY \
kentik/ktranslate:v2 \
  -snmp /snmp-base.yaml \
  -nr_account_id=$NR_ACCOUNT_ID \
  -metrics=jchf \
  -tee_logs=true \
  -service_name=$CONTAINER_SERVICE \
  -snmp_discovery_on_start=true \
  -snmp_discovery_min=180 \
  nr1.snmp
```

Para un servicio de Linux, editará el contenido de `/etc/ktranslate/profiles` en el host local.

<Callout variant="tip">
  Para eliminar el riesgo de comportamiento inconsistente en su entorno, asegúrese de pasar su versión personalizada de los perfiles cada vez que inicie un contenedor o servicio.
</Callout>

## Contribuir al repositorio público. [#public]

Como repositorio de código abierto, las contribuciones son bienvenidas de cualquier persona para crear nuevos perfiles o mejorar los perfiles existentes que mejorarán las capacidades de monitoreo para todos los usuarios.

New Relic no participa en el proceso de revisión de perfiles en el repositorio de terceros. Todas las solicitudes de cambios y actualizaciones deben dirigirse a los mantenedores de Kentik a través de problemas de Github.

Poco después de fusionar un perfil, la automatización CI/CD hace que los nuevos perfiles SNMP estén disponibles en una nueva versión de la imagen de la docker .

## ¿Qué tipos de OID debo incluir? [#oids]

En muchos casos, SNMP ofrece una gran cantidad de datos, pero poca información útil. El [recorrido SNMP de muestra](#snmp-polling) anterior tiene 2022 filas de resultados de OID y el [perfil resultante](https://github.com/kentik/snmp-profiles/blob/main/profiles/kentik_snmp/apc/apc_ups.yml) recopila alrededor de 50 OID en total. Siguiendo los patrones básicos de <DNT>**"Golden Signals"**</DNT> definidos por la [disciplina Site Reliability Engineering (ingeniería de confiabilidad del sitio (SRE))](https://sre.google/sre-book/monitoring-distributed-systems/); concéntrese en recopilar datos que le permitan saber si hay algo que impida que el dispositivo pueda realizar las funciones que espera que realice. A partir de eso, debe recopilar mediciones que le indiquen qué tan bien está desempeñando esas funciones.

**Ejemplo 1:**

Para un dispositivo que funcione como concentrador VPN, recogeríamos métricas de sistema de alto nivel como:

* Utilización de CPU y memoria
* Información del sensor de hardware para garantizar que el dispositivo no se apague solo debido a cosas como fallas en el ventilador.
* OID que nos informan sobre las conexiones agregadas y el rendimiento.

**Ejemplo 2:**

Una tabla OID que enumera todos los procesos en ejecución en un dispositivo de red es un ejemplo de datos que están disponibles pero no son útiles.

Recopilar todos los procesos en ejecución puede parecer útil, pero dado que se trata de un dispositivo, normalmente no tiene la capacidad ni la necesidad de hacer nada con los procesos que se ejecutan en su interior. Sondear y almacenar tablas con cientos de elementos con los que en realidad no se puede hacer nada no sería eficiente.
