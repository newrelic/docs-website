대부분의 앱은 캐싱, 세션 스토리지, 실시간 리더보드, 메시지 대기열과 같이 빨라야 하는 작업에 레디스를 사용합니다. 하지만 레디스가 느려지면 보통 느린 페이지나 시간 초과된 요청과 같이 다른 곳에서 먼저 알아차리게 됩니다. 가시성이 좋지 않으면 진짜 문제가 메모리 부족인지, 키가 너무 공격적으로 제거되는 것인지, 캐시 히트율이 떨어지는 것인지, 아니면 복제본이 뒤처지는 것인지 알기 어렵습니다.

뉴렐릭은 레디스를 위한 OpenTelemetry 기반 모니터링 경로를 제공합니다. OpenTelemetry Collector의 [레디스 수신기](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/receiver/redisreceiver) 는 레디스가 INFO 명령을 통해 노출하는 메트릭을 수집하고 별도의 에이전트 없이 뉴렐릭으로 전송합니다. 뉴렐릭은 각 레디스 인스턴스를 고유한 엔티티로 합성하고, 클러스터링된 배포를 레디스 Cluster 엔티티 아래에 그룹화하며, 사전 구축된 대시보드 및 알림을 제공합니다.

<Callout variant="tip">
  기존 뉴렐릭 레디스 온-호스트 통합(nri-redis)을 찾고 있다면 [레디스 모니터링 통합](/docs/infrastructure/host-integrations/host-integrations-list/redis/redis-integration/)을 참조하세요.
</Callout>

## 시작하다 [#get-started]

OpenTelemetry로 레디스를 모니터하려면 전제 조건 검토, 수집기 설정, 데이터 보기의 세 가지 단계를 완료하세요.

### 전제 조건 [#prereq]

환경이 배포를 위해 정의된 통합의 모든 요구 사항을 충족하는지 확인합니다:

* [자체 호스팅](/docs/opentelemetry/integrations/redis/host/#prereq)의 경우
* [도커](/docs/opentelemetry/integrations/redis/docker/#prereq)의 경우
* [Kubernetes](/docs/opentelemetry/integrations/redis/kubernetes/#prereq)의 경우

### 수집기 설정 [#collector-setup]

인프라에 따라 설치 경로를 선택하세요:

* [자체 호스팅](/docs/opentelemetry/integrations/redis/host)의 경우
* [Docker](/docs/opentelemetry/integrations/redis/docker)의 경우
* [Kubernetes](/docs/opentelemetry/integrations/redis/kubernetes)의 경우

### 데이터 보기 [#view-data]

수집기 설정을 완료하면 뉴렐릭에서 레디스 메트릭을 보고, NRQL로 쿼리하고, 대시보드를 빌드하고, 알림을 설정할 수 있습니다. 자세한 내용은 [데이터 찾기 및 쿼리](/docs/opentelemetry/integrations/redis/view-data) 문서를 참조하십시오.

뉴렐릭 레디스 통합은 다양한 메트릭을 추적합니다. 지원되는 데이터 유형 및 속성에 대해 자세히 알아보려면 [레디스 메트릭 참조](/docs/opentelemetry/integrations/redis/metrics) 가이드를 참조하세요.

## 다음 단계 [#next-steps]

통합을 설치한 후:

* **[데이터 보기](/docs/opentelemetry/integrations/redis/view-data)**: 대시보드를 탐색하고, NRQL 쿼리를 실행하며, 레디스 메트릭에 대한 알림을 설정합니다.
* **[메트릭 참조 검토](/docs/opentelemetry/integrations/redis/metrics)**: 사용 가능한 모든 메트릭과 해당 속성을 이해합니다.
* **[문제 해결](/docs/opentelemetry/integrations/redis/troubleshooting)**: 데이터 수집 및 엔티티 합성과 관련된 일반적인 문제를 해결합니다.
