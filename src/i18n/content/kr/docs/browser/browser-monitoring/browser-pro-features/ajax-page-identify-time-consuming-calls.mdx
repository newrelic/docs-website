URL, 상태 코드, 기간과 같은 메타데이터만 있는 경우 AJAX 실패를 진단하기 어렵습니다. GraphQL과 같은 최신 프레임워크는 이를 더욱 어렵게 만듭니다 ― 요청이 성공한 것처럼 보여도 응답 본문 안에 오류가 숨어 있을 수 있습니다.

당사의 <InlinePopover type="browser" /><DNT>AJAX</DNT> UI는 브라우저에서 외부 HTTP 또는 HTTPS 엔드포인트로의 AJAX requests에 대한 완전한 가시성을 제공합니다. 시간이 많이 걸리거나 실패하는 호출을 식별하고, 전체 요청 및 응답 페이로드를 캡처하고, 전체 스택에서 문제를 트레이스할 수 있습니다. 데스크톱 브라우저, 모바일 디바이스 및 태블릿을 포함한 디바이스 유형별로 문제를 검토할 수도 있습니다.

## AJAX requests 조사 [#ajax-dashboard]

앱에 대한 AJAX 요청 문제를 해결하려면:

1. <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &gt; Browser &gt; (select an app) &gt; AJAX**</DNT> 으)로 이동합니다.
2. 장치 유형을 선택하십시오: 데스크탑(기본값), 모바일 및 태블릿.
3. AJAX 요청을 클릭합니다.

AJAX 요청에 집중했다면 이제 조사할 차례입니다. 다음 전략 중 일부를 시도해 보십시오:

    <DNT>**Average response time**</DNT> 차트의 AJAX 엔드포인트는 4개 또는 2개의 기간으로 표시됩니다. 이는 [스티치 추적 헤더가](/docs/apm/transactions/cross-application-traces/cross-application-tracing#security) 있는지 여부에 따라 다릅니다.

    * 4개의 기간(<DNT>**Application time**</DNT>, <DNT>**Queue time**</DNT>, <DNT>**Network time**</DNT> 및 <DNT>**Callback time**</DNT>): 이러한 AJAX 호출은 스냅 추적 헤더를 제공하는 <InlinePopover type="apm" />에이전트의 앱 모니터에 대해 수행됩니다.
    * 두 기간(<DNT>**Response time**</DNT> 및 <DNT>**Callback time**</DNT>): 이러한 AJAX 호출은 APM 에이전트에 의한 모니터링이 아니거나 로그 추적 헤더를 제공하지 않는 에이전트에 의한 모니터링인 앱에 대해 수행됩니다.

    * `request.body`
    * `response.headers`
    * `response.body`

    ### 페이로드 캡처가 거부 목록과 상호 작용하는 방식 [#deny-list-interaction]

    페이로드 캡처는 기존 [AJAX request 거부 목록](/docs/browser/new-relic-browser/configuration/configure-ajax-request-events/)을 준수합니다. 모든 AJAX request의 경우:

    * URL이 거부 목록에 있는 경우 이벤트는 완전히 삭제되며 메타데이터나 페이로드는 캡처되지 않습니다.
    * URL이 거부 목록에 없는 경우, 페이로드 캡처 설정에 따라 에이전트는 request 및 응답 페이로드를 캡처하고 데이터를 뉴렐릭으로 보내기 전에 난독화 룰을 적용합니다.

    ### PII 보호 [#pii-protection]

    브라우저 에이전트는 수집 전에 캡처된 페이로드에서 신용 카드 번호, 사회 보장 번호, `password` 및 `token`과 같은 일반적인 키를 포함하여 일반적인 민감한 데이터 패턴을 자동으로 난독화합니다. 비즈니스별 민감한 데이터를 마스크하도록 사용자 지정 정규식 규칙을 구성할 수도 있습니다. 자세한 내용은 [브라우저 데이터 난독화](/docs/browser/new-relic-browser/configuration/obfuscate-browser-agent-data/#ingest-obfuscation)를 참조하십시오.
  </Collapser>

  <Collapser id="distributed tracing" title="처음부터 끝까지 요청 추적">

    * 귀하가 필터링한 모든 트레이스와 특정 요청 유형을 보려면 <DNT>**AJAX**</DNT> UI 에서 선택한 요청에 대한 독립 추적 링크를 클릭하세요.

  <Collapser id="time-percentage-example" title="시간 백분율 예">

    이 정보를 제공하기 위해 브라우저는 각 AJAX 범주에 대한 누적 요청 시간을 취하여 모든 AJAX 요청에 대한 누적 요청 시간으로 나눕니다. 높은 처리량 요청이 낮은 처리량 요청보다 선호되기 때문에 응답 시간을 최적화할 위치를 식별하는 데 도움이 됩니다.

* 핵심성과지표(KPI) [알림을](/docs/alerts-applied-intelligence/new-relic-alerts/get-started/introduction-alerts) 통해 문제 발생을 예방할 수 있습니다.
* [단일 페이지 앱(SPA) 모니터링을](/docs/browser/single-page-app-monitoring/get-started/introduction-single-page-app-monitoring) 사용합니다. 이는 AJAX 요청을 사용하여 콘텐츠를 동적으로 가져오고 유동적인 사용자 경험을 생성하는 모든 앱에 유용합니다.
* UI 또는 API로 [데이터를 쿼리합니다](/docs/query-your-data/explore-query-data/explore-data/introduction-querying-new-relic-data) . 예를 들어 [기본 브라우저 이벤트](/docs/insights/insights-data-sources/default-data/browser-default-events-insights) 로 쿼리하거나 지리 및 브라우저 데이터에 대해 [SPA `AjaxRequest`](/attribute-dictionary/?event=AjaxRequest) 를 사용하거나 고유한 [사용자 지정 데이터](/docs/telemetry-data-platform/get-data-new-relic/getting-started/introduction-new-relic-data-ingest-apis-sdks) 를 New Relic으로 가져올 수 있습니다.
* [차트](/docs/insights/use-insights-ui/manage-dashboards/chart-types) 와[대시보드](/docs/query-your-data/explore-query-data/dashboards/introduction-new-relic-one-dashboards) 로 데이터를 시각화하고 공유하세요.
* [AJAX 요청 거부 목록](/docs/browser/new-relic-browser/configuration/filter-ajax-request-events/)을 구성하여 페이로드 캡처 대상이 되는 URL을 제어합니다.
