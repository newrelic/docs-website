  랩의 각 절차는 마지막 절차를 기반으로 하므로 이 절차를 시작하기 전에 마지막 절차 [*인 애플리케이션의 오류 디버그 를*](/docs/browser/new-relic-browser/lab/debug-errors)완료했는지 확인하십시오.

애플리케이션에서 JavaScript 오류를 수정한 후 귀하와 귀하의 팀은 자신감을 느낍니다. 다운 타임을 준비하고 소셜 미디어로 이동하지만 Twitter를 확인하고 혼란스러워하는 고객을 봅니다.

<img title="unhappy customers" alt="unhappy customers" src="/images/browser-lab-screenshot-tweet.webp" />

어 오! 고객이 행복해 보이지 않습니다. 지연의 원인을 찾기 위해 New Relic 브라우저 모니터링을 사용할 때입니다.

<Callout variant="important">
  New Relic에서 데이터를 보려면 이 절차에 대해 브라우저 모니터링을 활성화해야 합니다.

  아직 하지 않았다면 [브라우저 에이전트로 앱을 계측하십시오](/docs/browser/new-relic-browser/lab/install-browser-agent).
</Callout>

## 애플리케이션의 속도 저하 디버그

<Steps>
  <Step>
    뉴렐릭 홈페이지에서 <DNT>**Browser**</DNT> 으로 이동하여 <DNT>**Relicstaurants**</DNT> 를 선택하세요.

    <img title="view relicstaurants" alt="view relicstaurants" src="/images/browser-lab-screenshot-view-relicstaurants.webp" />
  </Step>

  <Step>
    여기에서는 <DNT>**Page views with JavaScript errors**</DNT>, <DNT>**Core web vitals**</DNT>, <DNT>**User time on the site**</DNT>, <DNT>**Initial page load and route changes**</DNT> 등을 포함하여 브라우저 애플리케이션과 관련된 모든 데이터를 볼 수 있습니다.

    <img title="Relicstaurants summary" alt="Relicstaurants summary" src="/images/browser-lab-screenshot-relicstaurants-browser-app.webp" />

    <DNT>**largest contentful paint (LCP)**</DNT> 에 주목하세요.

    <img title="LCP" alt="LCP" src="/images/browser-lab-screenshot-LCP.webp" />

    콘텐츠가 포함된 최대 페인트(LCP)는 웹 페이지의 기본 콘텐츠가 로드되는 속도를 나타냅니다. 이상적으로는 콘텐츠를 로드하는 데 1-2초 이상 걸리지 않아야 합니다. 여기에서는 사이트가 5초 이상 로드되는 것을 볼 수 있습니다. 사용자가 불평하는 것도 당연합니다!

    하지만 이 지연의 원인은 무엇입니까? 백엔드?
  </Step>

  <Step>
    아래로 스크롤하여 <DNT>**Front end vs. back end**</DNT> 그래프를 확인하세요.

    <img title="frontend vs backend graph" alt="frontend vs backend graph" src="/images/browser-lab-screenshot-frontend-vs-backend.webp" />

    그래프를 필터링하고 백앤드를 로드하는 데 걸리는 시간을 확인하려면 <DNT>**Back end (time to first byte) (50%)**</DNT> 을 클릭하세요.

    <img title="backend time consumption" alt="backend time consumption" src="/images/browser-lab-screenshot-backend.webp" />

    그래프는 최악의 경우 백엔드가 요청을 처리하는 데 최대 140밀리초가 걸렸음을 나타냅니다. 이것은 프론트엔드가 지연을 일으킨다는 뜻인가요?

    <DNT>**Front end (Window load + AJAX) (50%)**</DNT> 을 클릭합니다.

    <img title="frontend time consumption" alt="frontend time consumption" src="/images/browser-lab-screenshot-frontend.webp" />

    문제가 있습니다! 그래프는 프런트 엔드에서 지연이 발생하고 있음을 나타냅니다.
  </Step>

  <Step>
    프론트엔드에서 지연이 발생하는 원인을 좁히려면 <DNT>**Initial page load and route change**</DNT> 그래프를 자세히 살펴보세요.

    <img title="page load graph" alt="page load graph" src="/images/browser-lab-screenshot-page-load-graph.webp" />
  </Step>

  <Step>
    <DNT>**Initial page load (Window load + AJAX)**</DNT> 을 클릭합니다.

    <img title="initial page load" alt="initial page load" src="/images/browser-lab-screenshot-initial-page-load.webp" />

    그래프는 <DNT>**Initial page load (Window load + AJAX)**</DNT> 5-6초가 걸리는 것을 나타냅니다. 이는 놀라운 일입니다.
  </Step>

  <Step>
    자세한 내용을 보려면 <DNT>**Initial page load and route change**</DNT> 클릭하세요.

    <img title="page load details" alt="page load details" src="/images/browser-lab-screenshot-page-load-detail.webp" />

    그러면 <DNT>**Page views**</DNT> 으로 이동합니다.

    <img title="page views" alt="page views" src="/images/browser-lab-screenshot-page-views.webp" />
  </Step>

  <Step>
    페이지를 <DNT>**Most time-consuming**</DNT> 기준으로 정렬합니다.

    <img title="sort page" alt="sort page" src="/images/browser-lab-screenshot-sort-pages.webp" />

    초기 페이지를 로드하는 데 거의 90%의 시간이 걸립니다.

    <img title="most time consuming pages" alt="most time consuming pages" src="/images/browser-lab-screenshot-most-time-consuming-page.webp" />

    그것을 클릭하면 세부 정보를 볼 수 있습니다.

    <img title="most time consuming page detail" alt="most time consuming page detail" src="/images/browser-lab-screenshot-most-time-consuming-page-detail.webp" />

    이 페이지에는 <DNT>**Page view breakdown**</DNT>, <DNT>**Median initial page load time**</DNT> 및 기타 중요한 세부정보가 표시됩니다. <DNT>**Page view breakdown**</DNT> 그래프는 페이지가 더 오래 걸리는 이유와 위치를 좁히는 데 도움이 되므로 여기서 특히 중요합니다. 이 그래프를 자세히 살펴보면 <DNT>**Page rendering**</DNT> 최대 5000밀리초까지 걸리는 것을 알 수 있습니다.

    <img title="page rendering" alt="page rendering" src="/images/browser-lab-screenshot-page-rendering.webp" />

    이제 초기 페이지를 렌더링하는 데 꽤 오랜 시간이 걸려 애플리케이션 속도가 느려진다는 사실을 알게 되었습니다. 다음으로, 렌더링 프로세스 속도를 늦추는 원인이 무엇인지 파악하기 위해 <DNT>**Session traces**</DNT> 를 관찰합니다.

    오른쪽 상단에 있는 <DNT>**X**</DNT> 를 클릭하여 이 보기를 종료합니다.
  </Step>

  <Step>
    왼쪽 탐색에서 <DNT>**Session traces**</DNT> 으로 이동하여 <DNT>**Page load**</DNT> 의 내림차순으로 정렬합니다.

    <img title="most time consuming pages" alt="most time consuming pages" src="/images/browser-lab-screenshot-session-traces.webp" />
  </Step>

  <Step>
    여기에서는 세션 트레이스가 <DNT>**Page load**</DNT> 시간 순서로 정렬된 것을 볼 수 있습니다.

    <img title="sorted session traces" alt="sorted session traces" src="/images/browser-lab-screenshot-sorted-session-traces.webp" />
  </Step>

  <Step>
    목록에서 첫 번째 항목을 클릭합니다.

    <img title="navigate to trace detail" alt="trace detail" src="/images/browser-lab-screenshot-navigate-to-trace-detail.webp" />

    ![정렬된 세션 추적을 보여주는 이미지](../../../images/nr-browser/navigate-to-trace-detail.webp)

    그러면 <DNT>**Session traces**</DNT> 세부정보 페이지로 이동됩니다.

    여기서는 특정 세션에 대한 전체 트레이스를 볼 수 있습니다. 이 페이지에는 <DNT>**Backend**</DNT>, <DNT>**Dom Processing**</DNT>, <DNT>**Page Load**</DNT> 및 기타 트레이스 관련 정보도 표시됩니다.

    <img title="full trace" alt="full trace" src="/images/browser-lab-screenshot-trace.webp" />
  </Step>

  <Step>
    <DNT>**Page load**</DNT> 이(가) 예상보다 오래 걸리고 있습니다. 로드에 대한 자세한 타임라인이 필요합니다. 타임라인을 조정하려면 포인터를 왼쪽과 오른쪽으로 스크롤하세요.

    <img title="adjust trace timeline" alt="adjust trace timeline" src="/images/browser-lab-screenshot-adjust-timeline.webp" />
  </Step>

  <Step>
    추적을 스크롤하여 시간 창을 이동하고 이 세션 동안 개별 이벤트의 세부 정보를 확인합니다.

    <img title="adjust time window" alt="adjust time window" src="/images/browser-lab-screenshot-trace-time-window-detail.webp" />
  </Step>

  <Step>
    특정 이벤트는 5초 이상 소요됩니다.

    <img title="image trace" alt="image trace" src="/images/browser-lab-screenshot-image-trace.webp" />
  </Step>

  <Step>
    이벤트를 클릭하면 세부 정보를 볼 수 있습니다.

    <img title="image trace detail" alt="image trace detail" src="/images/browser-lab-screenshot-image-trace-detail.webp" />

    이미지라는 점에 유의하십시오. 특히 로드하는 데 5 - 6초가 걸리고 지연을 유발하는 애플리케이션의 배경 이미지입니다.
  </Step>
</Steps>

이러한 결과를 바탕으로 여기에서 배경 이미지가 범인이라는 가설을 세웁니다. 최적화되지 않은 고해상도 이미지는 웹 사이트 속도 저하의 가장 일반적인 원인입니다. 좋은 소식! 이제 이유를 알았으므로 문제를 해결할 수 있습니다.

## 요약

요약하면 애플리케이션의 속도 저하를 관찰하고 New Relic 브라우저 모니터링을 사용하여 다음을 수행했습니다.

1. 사이트의 핵심 웹 바이탈 관찰
2. 속도 저하 원인 좁히기
