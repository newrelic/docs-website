페이지는 브라우저나 서버에 캐시될 수 있습니다. 캐시된 페이지에는 오래된(이전에 생성된) 응용 프로그램 및 대기열 시간이 포함됩니다.

## 캐시된 페이지 감지 [#browser-traces]

우리의 <InlinePopover type="browser"/>총 백앤드 시간을 앱 및 대기열 시간과 비교하여 캐시된 페이지를 감지합니다. 앱 시간과 대기열 시간의 합이 전체 백앤드 시간보다 큰 경우, 브라우저 모델링은 페이지가 캐시되었다고 가정하고 앱 시간과 대기열 시간을 0으로 하여 요청을 집계합니다.

다음은 캐시된 페이지에 대한 몇 가지 추가 팁입니다.

* <DNT>
    **Java**
  </DNT>

  : 앱 서버의 "작업" 캐시를 플러시합니다. 이렇게 하면 앱 서버가 강제로 다시 컴파일됩니다.

* <DNT>
    **.NET**
  </DNT>

  : `flush_dotnet_temp.cmd` 명령을 사용하여

  <DNT>
    **asp.net**
  </DNT>

  캐시 디렉토리가 깨끗한지 확인하세요. 이렇게 하면 앱 서버가 페이지 로드 타이밍 계측을 사용하여 ASP를 다시 컴파일하게 됩니다.

* <DNT>
    **Node.js**
  </DNT>

  : 기본적으로 Node.js는 페이지를 캐시하지 않습니다.

* <DNT>
    **PHP**
  </DNT>

  : 동적 페이지에 CDN 캐시가 있는 경우 캐시에 페이지 로드 타이밍 계측이 포함되어 있는지 확인하세요.

* <DNT>
    **Python**
  </DNT>

  : 앱을 다시 시작하세요.

* <DNT>
    **Ruby**
  </DNT>

  : 자동 계측과 수동 계측에 대한 헤더 정보를 확인하세요. 예를 들어 특정 웹페이지를 무시하려면 [사용자 인터페이스](/docs/browser/new-relic-browser/getting-started/browser-settings) 에서 자동 브라우저 모델링을 비활성화한 다음 태그로 지정하려는 웹페이지를 수동으로 추가해야 합니다.
