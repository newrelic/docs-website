클라우드 비용 인텔리전스를 설정한 후, 재무 한도를 초과하기 전에 사전 공지를 수신하고 예상치 못한 요금 청구를 방지할 수 있도록 임계값이 포함된 공지 정책을 생성하세요. 예산 내에서 예산 사용률을 기반으로 여러 공지 임계값을 설정할 수 있습니다.

## 새 경고 조건 만들기

공지 조건은 정의된 임계값에 대해 주어진 이벤트 세트를 측정하고 지정된 기간 동안 임계값이 충족될 때 인시던트를 여는 지속적으로 실행되는 쿼리입니다.

1. <DNT>[**one.newrelic.com**](https://one.newrelic.com)</DNT> &gt; <DNT>**Alerts**</DNT> &gt; <DNT>**Alert Policies**</DNT> 로 이동하세요.
2. 정책 목록 페이지에서 <DNT>**+ New alert condition**</DNT>를 클릭합니다.
3. 처음부터 알림을 구축하려면 <DNT>**Write your own query**</DNT>을(를) 사용하세요.

## 신호 동작 설정

NRQL 쿼리를 사용하여 공지 조건이 공지의 기반으로 사용할 신호를 정의할 수 있습니다. 이 예제에서는 이 쿼리를 사용합니다:

```sql
FROM CloudCost SELECT sum(line_item_unblended_cost) FACET product_region_code
```

공지 조건에 이 쿼리를 사용하면 제품 지역 코드별로 세분화된 비혼합 비용을 알고 싶다는 것을 뉴렐릭에 알려줍니다.

NRQL(뉴렐릭의 쿼리 언어) 사용에 대해 자세히 알아보려면 [NRQL 문서](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/)를 확인하십시오.

## 신호를 실행하고 미리 봅니다 [#advanced-signal-settings]

1. 신호를 정의한 후 <DNT>**Run**</DNT> 클릭합니다. 차트가 나타나고 귀하가 설정한 시위가 표시됩니다.

   <Callout variant="tip">
     교차 계정 알림을 설정하려면 드롭다운 목록에서 <DNT>**data account**</DNT> 을(를) 선택합니다. 계정 간 알림의 경우 한 번에 하나의 계정에서만 데이터를 쿼리할 수 있다는 점에 유의하십시오.
   </Callout>

   <img title="Create a new alert condition page with advanced signal settings highlighted" alt="Create a new alert condition page with advanced signal settings highlighted" src="/images/cci-create-new-alert.webp" />

   이 예의 경우 차트에는 회사의 cloud 리소스에 대한 예상 사용 비용이 제품 지역 코드별로 분류되어 표시됩니다. 이를 통해 여러 지역에 걸쳐 cloud 리소스의 비용을 모니터링할 수 있습니다.

2. <DNT>**Next**</DNT> 을(를) 클릭하고 공지 조건 구성을 시작합니다.

## 경고 조건에 대한 임계값 설정 [#thresholds]

이제 임계값이 위반될 경우 인시던트를 열기 위한 완전히 정의된 조건과 규칙이 있습니다. 조건 이름을 지정하고 정책에 연결하여 설정을 완료합니다.

## 경고 조건 세부 정보 추가 [#add-details]

공지 조건이 완전히 정의되었으며 임계값이 위반되면 인시던트를 생성합니다. 이 조건의 이름을 지정하고 정책에 연결하여 설정을 완료합니다.

정책은 인시던트를 위한 정렬 시스템입니다. 정책을 **[워크플로우](/docs/alerts-applied-intelligence/applied-intelligence/incident-workflows/incident-workflows/)** 에 연결하여 뉴렐릭이 이 정보를 보낼 위치와 빈도를 정의할 수 있습니다.

        1. **One issue per policy (default)** \[정책당 하나의 문제(기본값)]:

        2. **One issue per condition** \[조건당 하나의 문제]:

           * **장점**: 각 조건에 대해 별도의 문제를 생성하며, 특정 지연시간 문제를 분리하고 해결하는 데 이상적입니다.

        3. **모든 인시던트에 대한 이슈**:

           * **단점**: 가장 노이즈가 많은 옵션이며, 전반적인 추세를 추적하고 효과적으로 우선순위를 지정하기가 어렵습니다.
