Go 에이전트에 대한 구성 설정을 편집하여 New Relic이 앱을 모니터링하는 방법의 일부 측면을 제어할 수 있습니다. 예를 들어:

* 높은 보안 모드를 켭니다.
* UI에서 필터링 및 정렬을 위한 사용자 정의 태그를 추가합니다.
* 오류, 트랜잭션 이벤트, 트랜잭션 추적 및 사용자 지정 이벤트 수집을 끕니다.

## 구성 방법 및 우선 순위 [#options]

Go 에이전트를 구성하는 기본 방법은 표준 [설치 프로세스](/docs/agents/go-agent/installation/install-new-relic-go) 의 일부인 `newrelic.NewApplication()` 호출의 일부로 `newrelic.Config` 구조체를 수정하는 것입니다. [Go 에이전트 버전 2.7.0 이상](/docs/release-notes/agent-release-notes/go-release-notes) [에서는 UI에서 서버 측 구성](#server-side-configuration) 을 사용하여 제한된 수의 구성 옵션을 설정할 수도 있습니다.

Go 에이전트는 구성에 대해 이 우선 순위를 따릅니다. 사용하도록 설정하면 서버측 값이 비어 있는 경우에도 서버측 구성이 `newrelic.Config` 구조체의 **모든** 해당 값을 재정의합니다.

각 구성 방법에 대한 자세한 설명은 다음과 같습니다.

<CollapserGroup>
  <Collapser
    id="server-side-configuration"
    title="서버 측 구성(2.7.0 이상)"
  >
    [서버 측 구성](/docs/agents/manage-apm-agents/configuration/server-side-agent-configuration) 은 [Go 에이전트 버전 2.7.0 이상](/docs/release-notes/agent-release-notes/go-release-notes) 에서 사용할 수 있습니다. 이를 통해 UI에서 특정 설정을 구성할 수 있습니다. 이렇게 하면 변경 사항이 여러 호스트에서 실행되는 경우에도 모든 에이전트에 자동으로 적용됩니다. 사용 가능한 경우 이 문서에는 개별 구성 옵션 아래의 서버 측 구성에 대한 UI 레이블이 서버 **측 레이블** 로 포함됩니다.

    [진행 중인 구성에](#in-process-config) 설명된 단계에 따라 신청 프로세스에서 `newrelic.NewApplication()` 를 계속 호출해야 합니다. 서버 측 구성 옵션 세트는 로컬로 설정된 옵션을 덮어씁니다. 서버 측에서 모든 구성 옵션을 사용할 수 있는 것은 아니므로 `newrelic.Config` 구조체를 계속 업데이트할 수 있습니다.

    <Callout variant="caution">
      서버측 구성이 활성화된 경우 에이전트는 UI에서 설정할 **수** 있는 `newrelic.Config` 구조체의 모든 값을 무시합니다. UI 값이 비어 있어도 에이전트는 이를 빈 값으로 취급하여 `newrelic.Config` 값을 사용하지 않습니다.
    </Callout>
  </Collapser>

  <Collapser
    id="in-process-config"
    title={<>진행 <InlineCode>newrelic.Config</InlineCode> 구조체</>}
  >
    로컬 프로세스 `newrelic.Config` 구조에서 Go 에이전트를 구성합니다. 이 구조체는 `newrelic.NewApplication()` 을 호출할 때 액세스할 수 있습니다.

    1. `main` 함수 또는 `init` 블록에 다음을 추가합니다.

       ```go
       app, err := newrelic.NewApplication(
           newrelic.ConfigAppName("Your Application Name"),
           newrelic.ConfigLicense(os.Getenv("NEW_RELIC_LICENSE_KEY")),
       )
       ```

       `os.Getenv` 를 사용하여 `newrelic.ConfigLicense` 에 전달된 문자열 리터럴 값으로 하드 코딩하지 않고 환경에서 라이센스 키를 읽습니다. 라이선스 키 또는 기타 민감한 정보를 소스 코드에 넣지 않는 것이 좋습니다. 그럴 경우 SCM 리포지토리에 저장되고 승인되지 않은 당사자에게 공개될 수 있습니다.

    2. `newrelic.Config` 구조체의 값을 업데이트하여 `newrelic.ConfigOption` 을 사용하여 애플리케이션을 구성합니다. 이들은 `newrelic.Config` 구조체에 대한 포인터를 허용하는 함수입니다. 애플리케이션을 추가로 구성하려면 `newrelic.ConfigOption` 을 추가하세요. 예를 들어 사전 정의된 옵션 중 하나를 사용하여 공통 구성을 수행할 수 있습니다.

       ```go
       app, err := newrelic.NewApplication(
           newrelic.ConfigAppName("Your Application Name"),
           newrelic.ConfigLicense(os.Getenv("NEW_RELIC_LICENSE_KEY")),
           // add debug level logging to stdout
           newrelic.ConfigDebugLogger(os.Stdout),
       )
       ```

    3. 또는 더 복잡한 구성을 수행하기 위해 고유한 `newrelic.ConfigOption` 를 만들 수 있습니다.

       ```go
       app, err := newrelic.NewApplication(
           newrelic.ConfigAppName("Your Application Name"),
           newrelic.ConfigLicense(os.Getenv("NEW_RELIC_LICENSE_KEY")),
           newrelic.ConfigDebugLogger(os.Stdout),
           func(config *newrelic.Config) {
           	// add more specific configuration of the agent within a custom ConfigOption
           	config.HighSecurity = true
               config.CrossApplicationTracer.Enabled = false
           },
       )
       ```
  </Collapser>
</CollapserGroup>

## 구성 설정 변경 [#make-config-changes]

Go 에이전트 구성을 변경하려면 맞춤 `newrelic.ConfigOption` 내에서 `newrelic.Config` 구조체의 값을 설정합니다. 예를 들어 테스트 목적으로 New Relic 모니터링을 일시적으로 끄려면 `Enabled` 값을 `false` 으로 변경합니다.

```go
app, err := newrelic.NewApplication(
    newrelic.ConfigAppName("Your Application Name"),
    newrelic.ConfigLicense(os.Getenv("NEW_RELIC_LICENSE_KEY")),
    func(config *newrelic.Config) {
    	config.Enabled = false
    },
)
```

이 예와 다음 예에서 `config` 은 New Relic 구성 구조를 나타냅니다. 하지만 [Go 에이전트를 설치하고](/docs/agents/go-agent/get-started/get-new-relic-go) 앱에서 구성을 시작할 때 다른 변수 이름을 지정했을 수도 있습니다.

    `true` 일 때 에이전트는 앱에서 [New Relic 수집기](//docs/new-relic-solutions/get-started/glossary/#collector)로 데이터를 보냅니다.

    New Relic 모니터링을 끄려면 이것을 `false` 으로 설정하십시오.

    예를 들어:

    ```go
    app, err := newrelic.NewApplication(
        newrelic.ConfigAppName("Your Application Name"),
        newrelic.ConfigLicense(os.Getenv("NEW_RELIC_LICENSE_KEY")),
        func(config *newrelic.Config) {
        	config.Enabled = false
        },
    )
    ```

    이 작업을 더 쉽게 하기 위해 `ConfigEnabled` 옵션을 사용할 수 있습니다.

    ```go
    app, err := newrelic.NewApplication(
        newrelic.ConfigAppName("Your Application Name"),
        newrelic.ConfigLicense(os.Getenv("NEW_RELIC_LICENSE_KEY")),
    	      newrelic.ConfigEnabled(false),
    )
    ```

    이것은 개발 환경에 New Relic을 설치하거나 문제 해결을 위해 유용할 수 있습니다. `Enabled` 이 `false` 로 설정된 경우:

    * New Relic Go 에이전트는 New Relic 수집기와 통신하지 않습니다.
    * 에이전트는 고루틴을 생성하지 않습니다.
    * 라이센스 키는 설치 중에 필요하지 않습니다.

    * SSL 켜기

    * 오류 메시지 문자열 보고 끄기

    * 맞춤 이벤트 보고 끄기

      이 설정은 UI의 해당 계정 설정과 일치해야 합니다. 예를 들어:

      ```go
      app, err := newrelic.NewApplication(
          newrelic.ConfigAppName("Your Application Name"),
          newrelic.ConfigLicense(os.Getenv("NEW_RELIC_LICENSE_KEY")),
          func(config *newrelic.Config) {
              config.HighSecurity = true
          },
      )
      ```

      에이전트는 기본적으로 HTTPS를 통해 New Relic과 통신하며 New Relic은 APM 및 REST API에 대한 모든 트래픽에 대해 [HTTPS가 필요합니다](/docs/apis/rest-api-v2/troubleshooting/301-response-rest-api-calls) .
  </Collapser>

## 환경에서 구성 [#configuring-from-the-environment]

유연성을 높이기 위해 환경 변수를 애플리케이션의 소스 코드에 하드코딩하는 대신 설정하여 많은 구성 옵션을 설정할 수 있습니다. 이를 사용하려면 다른 구성 옵션 중에서 `ConfigFromEnvironment()` 에 대한 호출을 추가하세요.

```go
app, err := newrelic.NewApplication(
   newrelic.ConfigAppName("Your Application Name"),
   newrelic.ConfigLicense(os.Getenv("NEW_RELIC_LICENSE_KEY")),
   newrelic.ConfigFromEnvironment(),
)
```

`newrelic.ConfigFromEnvironment()` 이 나타나는 옵션 목록의 지점에서 환경 변수를 읽고 `newrelic.Config` 구조체의 해당 항목이 업데이트됩니다. `ConfigFromEnvironment` 뒤에 나열된 추가 구성 옵션이 있는 경우 `ConfigFromEnvironment` 에 의해 설정된 값을 재정의할 수 있습니다.

예를 들어 다음 환경 변수가 설정된 경우:

```ini
NEW_RELIC_LICENSE_KEY="your_license_key_here"
NEW_RELIC_APP_NAME="Your Application Name"
NEW_RELIC_CODE_LEVEL_METRICS_ENABLED="true"
NEW_RELIC_CODE_LEVEL_METRICS_PATH_PREFIX="myproject/src"
NEW_RELIC_LABELS="Env:Dev;Label2:label2;Label3:label3;Label4:label4"
```

다음 코드:

```go
app, err := newrelic.NewApplication(
    newrelic.ConfigFromEnvironment(),
)
```

하드 코딩된 것과 동일한 결과를 얻을 수 있습니다.

```go
app, err := newrelic.NewApplication(
    newrelic.ConfigAppName("Your Application Name"),
    newrelic.ConfigLicense("your_license_key_here"),
    newrelic.ConfigCodeLevelMetricsEnabled(true),
    newrelic.ConfigCodeLevelMetricsPathPrefix("myproject/src"),
    func(config *newrelic.Config) {
        config.Labels = map[string]string{
            "Env":    "Dev",
            "Label2": "label2",
            "Label3": "label3",
            "Label4": "label4",
        }
    },
)
```

가능한 모든 구성 옵션이 환경 변수를 통해 설정되는 것은 아닙니다. 아래 축소기 [의 환경 변수 및 기능 표](#env-var-table) 에는 사용 가능한 모든 구성 기능과 해당 환경 변수가 나열되어 있습니다. 이름이 지정된 구성 옵션은 `Config` 구조의 해당 필드에 값을 직접 할당하여 설정할 수 있지만 가능하면 구성 함수 및/또는 환경 변수를 사용하는 것이 좋습니다.

  <Collapser

    다음은 테이블 사용 방법에 대한 몇 가지 팁입니다.

    * 테이블에 환경 변수가 나열되어 있으면 명명된 환경 변수를 설정하여 해당 옵션을 설정할 수 있습니다. 에이전트가 모든 `NEW_RELIC_*` 환경 변수를 수락하도록 하는 `ConfigFromEnvironment()` 함수도 포함해야 합니다.

    * 구성 기능이 나열되어 있으면 `ConfigFromEnvironment()` 대신 해당 기능을 사용하여 해당 옵션을 설정할 수 있습니다. `ConfigFromEnvironment()` 를 포함하여 프로그램에 나열된 구성 기능은 코드에 나타나는 순서대로 해결된다는 점을 명심하십시오. 즉, 환경 변수를 만들고 `ConfigFromEnvironment()` 함수를 호출하면 이전에 특정 함수를 사용하여 설정했을 수 있는 해당 구성을 덮어씁니다. `ConfigFromEnvironment()` 이후의 후속 구성 옵션은 이전 구성 기능 및 환경 변수를 재정의합니다.

    * 각 기능을 사용하는 방법에 대한 자세한 내용은 여기 및 [Go 설명서 사이트](https://pkg.go.dev/github.com/newrelic/go-agent/v3@v3.20.0/newrelic#ConfigOption) 에서 설명서를 참조하세요.

      ### 표 참고 1: [#table-note-one]

      하위 기능을 활성화하기 위해 나열된 함수 중 하나를 호출하면 기본 기능도 활성화되고/하거나 다른 구성 값을 설정합니다.

      * `ConfigAppLogForwardingEnabled(true)` `ApplicationLogging.Forwarding.Enabled=true` 를 설정하지만 `ApplicationLogging.Enabled=true` 도 설정합니다.
      * `ConfigAppLogForwardingEnabled(false)` `ApplicationLogging.Forwarding.Enabled=false` 를 설정하지만 `ApplicationLogging.Forwarding.MaxSamplesStored=0` 도 설정합니다.
      * `ConfigAppLogDecoratingEnabled(true)` `ApplicationLogging.LocalDecorating.Enabled=true` 를 설정하지만 `ApplicationLogging.Enabled=true` 도 설정합니다.
      * `ConfigAppLogDecoratingEnabled(false)` `ApplicationLogging.LocalDecorating.Enabled=false` 을(를) 설정하지만 `ApplicationLogging.Enabled` 에는 영향을 미치지 않습니다.
      * `ConfigAppLogMetricsEnabled(true)` `ApplicationLogging.Metrics.Enabled=true` 를 설정하지만 `ApplicationLogging.Enabled=true` 도 설정합니다.
      * `ConfigAppLogMetricsEnabled(false)` `ApplicationLogging.Metrics.Enabled=false` 을(를) 설정하지만 `ApplicationLogging.Enabled` 에는 영향을 미치지 않습니다.

      ### 표 참고 2: [#table-note-two]

      `NEW_RELIC_LOG` 환경 변수를 통해 `Logger` 를 설정할 때 사용되는 로거 유형은 `NEW_RELIC_LOG_LEVEL` 값에 따라 다릅니다. 후자의 변수가 정의되고 값이 `debug` , `Debug` , `DEBUG` , `d` 또는 `D` 인 경우 표준 로거 대신 디버그 수준 로거가 사용됩니다. `NEW_RELIC_LOG` 은 `stdout` , `Stdout` , `STDOUT` , `stderr` , `Stderr` 또는 `STDERR` 값을 가질 수 있습니다.

    <Callout variant="tip">
      환경 변수는 `newrelic.ConfigFromEnvironment` 에서 읽을 수 있도록 비어 있지 않은 값을 가져야 합니다.
    </Callout>
  </Collapser>
</CollapserGroup>

## 버전 태그 설정 [#version-tag]

NEW_RELIC_METADATA_SERVICE_VERSION을 설정하면 이벤트 데이터에 `tag.service.version` 태그가 생성됩니다. 이 컨텍스트에서 서비스 버전은 배포되는 코드의 버전이며 대부분의 경우 1.2.3과 같은 시맨틱 버전이지만 항상 그런 것은 아닙니다. 이 정보를 보내면 배포된 소프트웨어 버전별로 원격 분석을 패싯할 수 있으므로 오류를 생성하는 소프트웨어 버전을 빠르게 식별할 수 있습니다.

## 애플리케이션 로깅 설정 [#application-logging]

에이전트에서 애플리케이션 로깅을 구성하기 위해 다음 설정을 사용할 수 있습니다. 컨텍스트에서 Go 에이전트 로그 사용에 대한 팁은 컨텍스트에서 [Go 로그를](/docs/logs/logs-context/configure-logs-context-go) 참조하십시오.

<Callout variant="important">
  Go 에이전트 버전 3.17.0 이상이 필요합니다.
</Callout>

    `ConfigAppLogForwardingMaxSamplesStored()` }을(를) 호출하여 `ApplicationLogging.Forwarding.MaxSamplesStored` 를 구성합니다.

    각 로그는 연결된 트랜잭션과 동일한 우선 순위를 받습니다. 트랜잭션 외부에서 발생하는 로그는 임의의 우선 순위를 받습니다. 일부 로그는 `MaxSamplesStored` 에 의해 제한되기 때문에 포함되지 않을 수 있습니다. 예를 들어, 로깅 `MaxSamplesStored` 이 10,000으로 설정되고 트랜잭션 1에 10,000개의 로그 항목이 있는 경우 트랜잭션 1에 대한 로그 항목만 기록됩니다. 트랜잭션 1의 로그가 10,000개 미만이면 트랜잭션 1에 대한 모든 로그를 받습니다. 여전히 공간이 있으면 트랜잭션 2에 대한 모든 로그를 받는 식입니다.

    샘플링된 트랜잭션에 대한 모든 로그가 기록된 후 `MaxSamplesStored` 의 한도에 도달하지 않은 경우 샘플링에 없는 트랜잭션에 대한 로그 메시지가 전송됩니다. 남아있는 경우 트랜잭션 외부의 로그 메시지가 기록됩니다.

## 모듈 종속성 메트릭 설정 [#mdm]

모듈 종속성 메트릭은 Go 에이전트에서 다양한 방식으로 구성할 수 있습니다. 모듈 종속성 메트릭은 코드 종속성 관리를 용이하게 하기 위해 Go 애플리케이션에서 사용하는 가져온 모듈 목록을 보고합니다. 여기에는 앱 모듈의 버전 정보도 포함됩니다.

<Callout variant="important">
  Go 에이전트 버전 3.20.0 필요 또는 더 높게
</Callout>

    `ConfigModuleDependencyMetricsEnabled` 를 호출하여 ModuleDependencyMetrics를 구성합니다.

    ```go
    app, err := newrelic.NewApplication(
        newrelic.ConfigModuleDependencyMetricsEnabled(true),
    )
    ```

    `NewApplication` 에 대한 호출에 `ConfigFromEnvironment()` 를 삽입하여 환경 변수를 통해 Go 에이전트 옵션 설정을 활성화할 수 있습니다. 이 작업을 수행한 경우 환경 변수를 설정하여 모듈 종속성 메트릭 수집을 활성화하거나 비활성화할 수 있습니다.

    ```ini
    NEW_RELIC_MODULE_DEPENDENCY_METRICS_ENABLED=true
    ```
