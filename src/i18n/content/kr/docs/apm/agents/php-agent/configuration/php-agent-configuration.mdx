우리의 PHP 에이전트에는 보고되는 데이터의 유형과 양을 미세 조정할 수 있는 다양한 설정이 있습니다. 대부분의 사용자에게 기본값은 오버헤드와 유틸리티의 최상의 조합을 생성합니다. 그러나 특정 요구 사항에 맞게 설정을 변경할 수 있습니다.

<Callout variant="important">
  INI 설정을 변경한 후에는 항상 웹 서버를 다시 시작하십시오. 그렇지 않으면 [즉시 적용](/docs/agents/php-agent/troubleshooting/ini-settings-not-taking-effect-immediately) 되지 않을 수 있습니다.
</Callout>

## 구성 우선 순위 [#config-options-precedence]

다음은 PHP 에이전트가 구성을 위해 따라야 하는 우선 순위에 대한 간단한 그림입니다. [서버 측 구성](/docs/agents/manage-apm-agents/configuration/server-side-agent-configuration) 은 적용할 수 없습니다. PHP 에이전트를 사용하는 앱의 UI에서 변경할 수 있는 유일한 값은 앱의 별칭과 Apdex-T 값입니다.

</figcaption>

## 구성 파일 변수 [#inivar-background]

설치하는 동안 [`newrelic-install`](/docs/agents/php-agent/installation/newrelic-install-script) 스크립트는 작성한 구성 파일에 대한 정보 또는 편집해야 하는 파일 목록을 제공합니다. 기본적으로 `newrelic.ini` 이라는 구성 파일을 생성하려고 시도합니다.

경우에 따라 `php.ini` 파일에 구성 옵션을 추가하라는 지시를 받을 수 있습니다. 필요한 경우에만 이 작업을 수행하십시오. 정확히 어떤 파일을 편집해야 하는지는 특정 버전의 PHP가 구성된 방식에 따라 다릅니다.

가장 일반적인 두 가지 구성은 다음과 같습니다.

* 단일 `newrelic.ini` 파일을 사용합니다. 이것은 특별한 옵션 없이 PHP를 직접 설치하거나 컴파일한 경우 일반적으로 기본값입니다.
* 모든 `.ini` 파일에 대해 지정된 디렉토리를 스캔합니다.

어떤 파일을 편집할지 확실하지 않은 경우:

* 명령줄에서 `php -i` 의 출력을 검토합니다.

* 브라우저에서 스크립트가 포함된 페이지의 출력을 검토합니다.

  ```php
  <?php phpinfo(); ?>
  ```

* `newrelic.ini` 파일이 나타나면 사용하십시오.

## 가변 범위 [#inivar-scope]

`newrelic.ini` 파일의 각 변수에는 정의된 범위가 있습니다. 범위는 설정을 지정하거나 수정할 수 있는 위치를 제어합니다.

New Relic 설정에 지원되는 두 가지 범위는 다음과 같습니다.

* <DNT>**SYSTEM**</DNT>: 전역 `newrelic.ini` 파일에 전역적으로 설정된 값입니다.
* <DNT>**PERDIR**</DNT>: [디렉토리별로](/docs/agents/php-agent/configuration/php-directory-ini-settings) 설정된 값입니다.

    이 구성 옵션은 APM에서 데이터가 보고되는 애플리케이션 이름을 설정합니다. 혼동과 의도하지 않은 데이터 집계를 방지하려면 기본 이름을 설명이 포함된 이름으로 바꾸는 것이 좋습니다. 동일한 이름을 가진 모든 애플리케이션의 데이터는 New Relic에 병합되므로 신중하게 설정하십시오.

    디렉토리별 설정을 포함한 모든 애플리케이션 이름 지정 옵션 [은 PHP 애플리케이션 이름 지정](/docs/agents/php-agent/configuration/name-your-php-application) 을 참조하십시오.

    설정 값은 최대 3개의 애플리케이션 이름이 세미콜론으로 구분된 목록입니다. 목록의 첫 번째 이름은 <DNT>**primary**</DNT> 애플리케이션 이름입니다. 각 계정마다 [고유해야 합니다](/docs/apm/new-relic-apm/installation-configuration/name-your-application#app-name) . 애플리케이션 이름은 캐시에 대한 키로 사용됩니다. [앱에 여러 이름을](/docs/agents/manage-apm-agents/app-naming/use-multiple-names-app) 사용하는 경우 뉴렐릭은 캐싱을 위해 첫 번째 이름만 사용합니다. 따라서 각 애플리케이션 이름은 첫 번째 요소 <DNT>**once**</DNT> 로만 표시될 수 있습니다.

    <CollapserGroup>
      <Collapser
        id="cache-example"
        title="캐싱 예제"
      >
        코드 세트 `newrelic.appname="App1;App3"` 에서 `newrelic.appname="App1;App2"` 이상을 설정하면 두 번째 코드가 작동하지 않는 것으로 나타납니다. 캐싱으로 인해 <DNT>**App1**</DNT> 및 <DNT>**App2**</DNT> 에 보고됩니다.
      </Collapser>
    </CollapserGroup>

    [중복되는 응용 프로그램 이름](/docs/agents/manage-apm-agents/app-naming/use-multiple-names-app) 이 필요한 경우 두 번째 또는 세 번째 위치에 공통 응용 프로그램 이름을 설정합니다. 예: `newrelic.appname="App2;App1"` 및 `newrelic.appname="App3;App1"` . 이는 예를 들어 응용 프로그램의 상위 집합을 추적하려는 경우에 유용합니다.
  </Collapser>

    New Relic PHP 에이전트는 지원하는 프레임워크를 자동으로 감지하지만, 실험적인 새 버전을 사용하거나 프레임워크를 사용자 정의한 경우 문제가 발생할 수 있습니다. 이 설정은 자동 프레임워크 감지를 비활성화하는 대신 에이전트에게 지정된 프레임워크에 따라 트랜잭션 이름을 지정하도록 지시합니다.

    `"no_framework"` 을 지정하면 프레임워크 관련 트랜잭션 이름이 완전히 비활성화됩니다.

    다음 값을 적절하게 사용합니다.

    * `"cakephp"`

    * `"codeigniter"`

    * `"drupal"` (Drupal 6 및 7용)

    * `"drupal8"` (Drupal 8 및 9용)

    * `"joomla"`

    * `"laravel"`

    * `"magento"`

    * `"magento2"`

    * `"mediawiki"`

    * `"slim"`

    * `"symfony2"` (Symfony 3의 경우 - Symfony 2.x는 지원되지 않음)

    * `"symfony4"` (심포니 4 및 5용)

    * `"wordpress"`

    * `"yii"`

    * `"zend"`

    * `"zend2"` (Zend 프레임워크 2 및 3용)

    * `"no_framework"` (감지된 경우에도 프레임워크를 강제 실행하지 않음)

      프레임워크 자동 감지가 실패하면 이 명령도 실패합니다.

      <Callout variant="tip">
        지원되는 프레임워크에 대한 최신 정보는 [PHP 호환성 페이지](/docs/apm/agents/php-agent/getting-started/php-agent-compatibility-requirements/) 를 확인하십시오.
      </Callout>

## 데몬 .ini 설정 [#inivar-daemon-settings]

이러한 설정 값은 데몬 시작을 제어합니다. 에이전트가 데몬을 시작해야 함을 감지하면 이러한 옵션을 데몬에 대한 적절한 명령줄 옵션으로 변환합니다.

이러한 모든 설정은 `newrelic.cfg` 파일의 설정을 미러링합니다. 모든 .ini 설정을 한 곳에 유지하기 위해 여기에서 반복됩니다. `newrelic.cfg` 의 각 설정에는 여기에 해당하는 항목이 있으며 `newrelic.daemon.` 가 접두사로 사용됩니다. 예를 들어, `newrelic.cfg` 의 `ssl` 설정은 .ini 파일의 `newrelic.daemon.ssl` 입니다.

<Callout variant="important">
  `/etc/newrelic/newrelic.cfg` 파일이 있는 경우 에이전트는 이러한 설정을 무시하고 에이전트는 데몬을 자동으로 시작 **하지 않습니다** .
</Callout>

데몬을 시작하는 방법과 외부 구성 파일을 사용하는 시기에 대한 자세한 내용은 [PHP 데몬 시작 모드](/docs/agents/php-agent/installation/starting-php-daemon) 를 참조하십시오.

## 컨텍스트의 로그인 [#logs-in-context]

PHP 에이전트 버전 10.1.0 [컨텍스트에서 APM 로그](/docs/apm/new-relic-apm/getting-started/get-started-logs-context) 와 함께 PHP 로그를 전달할 수 있습니다. 버전 10.3.0부터 로깅 지표 및 에이전트 로그 전달 기능이 기본적으로 활성화됩니다. `newrelic.application_logging.enabled` 값은 컨텍스트 기능의 로그가 활성인지 비활성인지 여부를 제어합니다.

로컬 에이전트 구성 파일(`newrelic.ini`)의 설정을 변경하려면 웹 서버를 다시 시작해야 적용됩니다. 그렇지 않으면 변경 사항이 [즉시 적용되지](/docs/agents/php-agent/troubleshooting/ini-settings-not-taking-effect-immediately) 않을 수 있습니다.

### 로그 전달 [#log-forwarding]

[지원되는 로깅 프레임워크를](/docs/logs/logs-context/configure-logs-context-php) 사용하는 경우 에이전트에 앱 로그를 New Relic으로 전달하도록 지시할 수 있습니다.

* `newrelic.application_logging.forwarding.log_level` 앱이 New Relic에 전달할 로그 종류를 선택할 수 있습니다.

    * 로그를 New Relic으로 전송하려면 이를 `true` 로 설정합니다. 이는 컨텍스트 로그인 기능이 작동할 수 있도록 하는 연결 메타데이터를 사용하여 PHP 에이전트에서 직접 로그를 전달합니다. PHP 에이전트는 애플리케이션 로그 출력에 메타데이터를 기록하지 않습니다.
    * 로그를 New Relic으로 전송하지 않으려면 이 값을 `false` 로 설정하세요. `enabled` `false` 로 설정되면 로그 데이터가 New Relic으로 전송되지 않으며 애플리케이션 로그 출력이 변경되지 않습니다.

      구성 파일의 예( `newrelic.ini` ):

      ```ini
      newrelic.application_logging.enabled = true
      ```

      기존 로그 포워딩 솔루션이 있고 컨텍스트에서 자동 로그인을 사용하도록 에이전트를 업데이트하는 경우 <DNT>**disable your old log forwarder**</DNT> 를 확인하세요. 그렇지 않으면 앱이 이중 로그 라인을 전송하게 됩니다. 귀하의 계정에 따라 이중 청구가 발생할 수 있습니다. 자세한 내용을 보려면 [특정 로그 전달자를](/docs/logs/forward-logs/enable-log-management-new-relic#log-forwarding) 비활성화하는 절차를 따르십시오.

      <Callout variant="important">

      </Callout>

#### 로그 컨텍스트 데이터 [#log-context-data]

PHP 에이전트는 Monolog 라이브러리에 대한 컨텍스트 데이터를 캡처하고 해당 컨텍스트를 전달된 로그에 속성으로 추가합니다. `forwarding` 섹션 내 `context_data` 섹션 아래 설정을 통해 에이전트가 추가하는 컨텍스트를 제어할 수 있습니다.

* `newrelic.application_logging.forwarding.context_data.exclude` 전달된 로그에서 제외되는 속성 키를 정의합니다.

    PHP 에이전트는 트랜잭션 추적 및 분산 추적을 세그먼트 모음으로 보고합니다. 각 세그먼트는 트랜잭션 추적의 메서드 또는 함수 호출을 나타냅니다. 이 구성의 기본값은 `0` 이며, 이는 PHP 에이전트가 웹 트랜잭션 중에 모든 세그먼트를 캡처해야 함을 나타냅니다. 트랜잭션이 끝나면 트랜잭션 추적에서 보고할 가장 높은 우선 순위 세그먼트를 조합합니다.

    수십만 또는 수백만 개의 함수 호출이 있는 장기 실행 PHP 프로세스의 경우 이 값을 `1` 보다 큰 값으로 설정하면 PHP 에이전트가 세그먼트를 기록할 때 시스템 메모리를 소진하는 것을 방지할 수 있습니다.

    세그먼트 크기는 해당 메서드 이름의 길이, 클래스 이름의 길이, 메서드에서 수행한 후속 호출 수에 따라 달라질 수 있습니다. 즉, 보수적인 추정치는 세그먼트당 400바이트입니다. 세그먼트 캡처를 위해 PHP 에이전트를 40MB로 제한하려면 이 값을 `100000` 으로 설정합니다. 이 값이 `2000` 보다 낮게 설정되면 트랜잭션 추적에 대해 보고되는 총 세그먼트 수를 추가로 제한합니다.

    이 구성 설정은 PHP 웹 프로세스에만 적용됩니다. PHP CLI 프로세스에는 영향을 미치지 않습니다. CLI 프로세스에 대한 제한을 설정하려면 `newrelic.transaction_tracer.max_segments_cli` 을 사용합니다.

## 받은편지함 구성 오류 [#errors-inbox-configuration]

다음 태그 중 하나를 설정하면 오류를 생성하는 소프트웨어 버전을 식별하는 데 도움이 됩니다.

* `NEW_RELIC_METADATA_SERVICE_VERSION` 배포된 코드 버전(대부분의 경우 1.2.3과 같은 의미 버전)이 포함된 이벤트 데이터에 대한 tagged.service.version을 생성하지만 항상 그런 것은 아닙니다.
* `NEW_RELIC_METADATA_RELEASE_TAG `릴리스 태그(예: v0.1.209)가 포함된 이벤트 데이터에 tagged.releaseTag를 생성합니다. 또는 릴리스-209).
* `NEW_RELIC_METADATA_COMMIT` 커밋 샤를 포함하는 이벤트 데이터에 대한 tagged.commit을 생성합니다. 전체 sha를 사용하거나 처음 7자만 사용할 수 있습니다(예: 734713b).

향후 릴리스된 오류 받은 편지함에서는 오류를 생성하는 소프트웨어 버전을 자동으로 추적합니다. 모든 버전 데이터는 [CodeStream](/docs/codestream/how-use-codestream/performance-monitoring/#buildsha) 에도 표시됩니다.

       에이전트를 전역적으로 비활성화해야 하는 경우 `newrelic.ini` 파일에서 `newrelic.enabled` 값을 `false` 로 설정합니다.

       에이전트가 전역적으로 비활성화되면 완전히 초기화되지 않습니다. 중요한 이유로 전역적으로 비활성화한다고 가정하고 에이전트는 PHP에 최대한 영향을 미치지 않도록 시도합니다.

       <Callout variant="important">
         에이전트를 전역적으로 비활성화한 다음 `.htaccess` 을 사용하여 디렉토리별로 선택적으로 활성화할 수 없습니다.
       </Callout>
  </Collapser>
