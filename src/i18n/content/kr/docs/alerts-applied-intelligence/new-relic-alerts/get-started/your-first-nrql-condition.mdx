애플리케이션을 New Relic에 연결하고 차트와 대시보드 탐색을 시작했다면 다음 단계는 팀이 데이터의 비정상적인 동작에 대해 최신 정보를 받을 수 있도록 알림을 만드는 것입니다. 알림은 단순히 데이터를 수집하는 것에서 사려 깊고 효과적인 조치를 취하는 것까지 New Relic 경험을 향상시킵니다.

여기서는 New Relic 경고 및 응용 인텔리전스 기능을 학습할 수 있도록 첫 번째 경고를 만드는 5단계를 안내합니다.

<Steps>
  <Step>
    ## 차트에서 알림 조건 만들기 [#create-alert-condition]

    알림을 시작하는 가장 쉬운 방법은 New Relic 차트에서 알림을 만드는 것입니다. 이 경로는 처음부터 NRQL 경고 조건을 생성하는 것과 동일하지만 차트에는 작업할 수 있는 NRQL 쿼리가 이미 있습니다.

    경고 조건은 기본적으로 비정상적인 동작에 대한 알림을 받기 전에 충족해야 하는 조건을 정의하기 위해 만드는 컨테이너입니다. 이 예에서는 웹 트랜잭션 시간과 관련된 대기 시간 문제를 팀에 알리는 경고를 생성합니다.

    따라서 이 경우 웹 트랜잭션이 50밀리초를 초과하지 않도록 하려면 웹 트랜잭션 시간이 50밀리초를 초과하고 인시던트를 생성할 때 모니터링하는 경고 조건을 구축합니다.

    ## 경고 조건에 대한 임계값 설정

    경고 조건이 컨테이너인 경우 임계값은 각 경고 조건에 포함된 규칙입니다. 데이터가 시스템으로 스트리밍될 때 경고 조건은 이러한 규칙의 위반을 검색합니다. 경고 조건이 설정한 모든 조건을 충족하는 시스템에서 들어오는 데이터를 확인하면 인시던트를 생성합니다. 인시던트는 시스템에서 문제가 발생하여 살펴봐야 한다는 신호입니다.

    팀에서 웹 트랜잭션 시간의 대기 시간 문제를 찾기 위해 경고 조건을 만들고 있습니다. 이제 이 조건이 찾을 규칙을 만들 것입니다.

    <CollapserGroup>
      <Collapser
        id="anomaly-threshold"
        title="이상 임계값(권장)"
      >
        웹 트랜잭션 시간이 평소보다 오래 걸리는 경우 알림을 받을 수 있도록 팀에서 이 경고 조건을 생성하고 있습니다. 그러나 웹 트랜잭션 시간이 얼마나 긴지는 신경 쓰지 않고 트랜잭션 시간이 비정상적으로 작동하는지 알고 싶다고 가정해 보겠습니다. 이 특정 사용 사례의 경우 **이상 임계값** 을 사용하는 것이 좋습니다. 당사의 이상 탐지는 데이터를 지속적으로 평가하여 시스템이 정상적으로 작동하는 방식을 이해합니다. 이상 임계값을 설정하면 웹 트랜잭션 시간이 예상 성능에서 벗어나는 경우 이상 감지를 사용하여 팀에 경고할 수 있습니다. 웹 트랜잭션 시간이 비정상적으로 작동하는지 여부만 알고 싶기 때문에 _모든_ 편차에 대한 알림을 원하므로 **상한과 하한** 을 선택합니다. 그러나 웹 트랜잭션 시간이 평소보다 _오래_ 걸리는 경우에만 알림을 받으려면 **상한만 선택합니다.**

        다음으로 우선 순위 수준을 설정해야 합니다. 우선 순위 수준은 인시던트를 생성할 항목을 결정합니다. 첫 번째 알림에 대해 우선순위 수준을 **중요** 로 설정하는 것이 좋습니다. [경보 조건 문서](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/advanced-techniques/set-thresholds-alert-condition#threshold-levels) 에서 우선 순위 수준에 대해 자세히 알아볼 수 있습니다.

        다음으로 **중요한 이상 임계값 위반** 을 정의하는 항목을 선택해야 합니다. 이 첫 번째 알림의 경우 기본 설정을 사용하고 필요에 따라 필요에 맞게 조정하는 것이 좋습니다. 따라서 "쿼리가 예측 값에서 벗어난 값을 반환할 때: **최소 5분 동안** **3 표준 편차** " 인시던트를 열도록 설정을 그대로 둡니다.

        이상 [문서](/docs/alerts-applied-intelligence/applied-intelligence/anomaly-detection/anomaly-detection-applied-intelligence/) 에서 이상에 대해 자세히 알아보십시오.

      <Collapser
        id="static-threshold"
        title="정적 임계값"
      >
        이상 임계값과 달리 정적 임계값은 데이터 세트를 전체적으로 보지 않고 시스템 기록을 기반으로 비정상적인 동작을 결정합니다. 대신 정적 임계값은 시스템이 **설정** 한 기준과 다르게 작동할 때마다 인시던트를 엽니다. 정적 경고 임계값은 훨씬 더 사용자 정의할 수 있으며 데이터와 원하는 것을 잘 알고 있는 경우 권장합니다.

        [NRQL 문서](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/#threshold-types) 에서 정적 경보 조건에 대해 자세히 알아보십시오.
      </Collapser>

      <Collapser
        id="lost-signal"
        title="손실 신호 임계값(선택 사항)"
      >
        때로는 들어오는 신호가 손실될 수 있으며 단순히 지연인지 또는 더 넓은 문제의 표시인지 이해하는 것이 중요합니다. **손실 신호 임계값** 은 신호 손실을 고려하기 전에 마지막 데이터 포인트가 감지된 시간부터 시스템이 대기해야 하는 시간(초)을 알려줍니다. 설정한 시간 제한 전에 신호가 반환되지 않으면 새 인시던트 열기 및/또는 열려 있는 모든 인시던트 닫기의 두 가지 작업 중 하나 또는 둘 모두를 선택할 수 있습니다. 손실된 신호가 이 엔터티의 다른 인시던트를 대체하거나 신호 손실이 예상되는 경우 관련 열려 있는 인시던트를 닫을 수 있습니다.

        손실된 신호 임계값을 설정하려면 시스템에 대한 지식과 이해하려는 내용이 필요합니다. 웹 트랜잭션 시간의 경우 New Relic이 1분마다 신호를 수집한다고 가정해 보겠습니다. 손실된 신호는 훨씬 더 큰 대기 시간 문제를 나타낼 수 있습니다. 따라서 편안한 수준으로 시간을 설정한 다음 확인란을 **선택하여 새로운 신호 끊김 사건을 여는** 것이 좋습니다.

  <Step>
    ## 고급 신호 설정 미세 조정 [#advanced-signal-settings]

    New Relic은 애플리케이션에서 시스템으로 스트리밍되는 데이터를 지속적으로 관찰합니다. 그러나 모든 애플리케이션이 동일한 주파수 또는 케이던스로 신호를 보내는 것은 아닙니다. 일부 이벤트는 1분에 한 번 시스템에 신호를 보낼 수 있는 반면 다른 이벤트는 매일 한 번만 New Relic에 데이터를 보고할 수 있습니다. 경고 조건은 **특정** 사용 사례를 위해 설계된 **특정** 컨테이너입니다. 경고 조건을 생성할 때 이 섹션은 평가 중인 데이터에 대해 가장 사용자 정의할 수 있습니다.

    웹 트랜잭션 대기 시간 문제를 찾는 조건에 대해 이러한 고급 신호 설정을 사용자 정의할 것입니다.

        첫 번째 알림의 경우 기본 설정을 그대로 사용하는 것이 좋지만 알림 조건 생성에 익숙해질수록 자신의 경험에 따라 이러한 필드를 사용자 지정하는 것이 좋습니다.
      </Collapser>

      <Collapser
        id="sliding-window"
        title="슬라이딩 윈도우 집계 사용"
      >
        하루 종일 데이터는 애플리케이션에서 New Relic으로 스트리밍됩니다. 사고에 대해 해당 데이터를 즉시 평가하는 대신 경보 조건은 **집계 창** 이라는 일정 기간 동안 데이터를 수집합니다. 추가 지연을 통해 기간이 집계되기 전에 더 느린 데이터 포인트가 도착할 수 있습니다.

        슬라이딩 윈도우는 "뾰족한" 차트를 매끄럽게 만들어야 할 때 유용합니다. 일반적인 사용 사례 중 하나는 슬라이딩 윈도우를 사용하여 좁은 기간의 집계보다 롤링 집계가 더 중요한 경우 짧은 시간 동안 많은 변동이 있는 선 그래프를 매끄럽게 만드는 것입니다.

        꾸준하고 일관된 데이터 스트림을 기대하지 않지만 데이터의 급감과 급증이 예상되는 경우 슬라이딩 윈도우 집계를 사용하는 것이 좋습니다.

        지연 기능은 일관성 없는 데이터 수집으로부터 사용자를 보호합니다. 인시던트 생성을 결정하기 전에 경고 조건에 약간의 흔들림 공간을 제공합니다. 주어진 시간에 데이터가 단 1분의 타임스탬프와 함께 New Relic에 도착하면 낮은 지연 설정으로 충분합니다. 반면에 그 분 동안 New Relic이 몇 분 전 또는 몇 분 후의 타임스탬프가 있는 데이터 포인트를 수신하면 신호가 더 일관성이 없으며 더 높은 지연 설정이 필요합니다.

      <Collapser
        id="gap-filling-strategy"
        title="갭 메우기 전략"
      >
        웹 트랜잭션 시간과 관련된 대기 시간 문제를 팀에 알리기 위해 경고 조건을 만들고 있습니다. 이 경우 애플리케이션은 지속적으로 New Relic 데이터를 전송합니다. 애플리케이션에서 New Relic으로 전송되는 신호의 지속적인 스트림이 있으며 신호에 예상되는 간격이 없으므로 간격 채우기 전략을 선택할 필요가 없습니다. 이 사용 사례와 첫 번째 알림의 경우 **간격 채우기 전략** 을 **none** 으로 설정하는 것이 좋습니다.

        24시간마다 한 번씩 New Relic에 신호를 보내는 일관되지 않은 데이터 세트가 있는 경우 팀의 특정 요구 사항에 따라 이 기능을 사용자 지정하는 것이 좋습니다.

        [손실된 신호 문서](/docs/apis/nerdgraph/examples/nerdgraph-api-loss-signal-gap-filling/) 에서 갭 채우기 전략에 대해 자세히 알아보세요.
      </Collapser>
    </CollapserGroup>
  </Step>

  <Step>
    ## 조건을 정책에 연결 [#connect-condition-policy]

    웹 트랜잭션 시간에 지연 문제가 있는 경우 가능한 한 빨리 알림을 받고 싶습니다. 가장 빠르고 효율적인 조치는 웹 트랜잭션 시간이 너무 오래 걸리는 경우 인시던트를 여는 경고 조건을 만드는 것입니다.

    이 경고 조건은 모든 규칙을 보관하는 컨테이너입니다. 정적 또는 이상 임계값을 사용하고 있습니까, 슬라이딩 윈도우 집계를 사용하고 있습니까, 아니면 평가 기간을 정상적으로 유지하고 있습니까?

    프로세스의 이 시점에서 우리는 이제 완전히 정의된 컨테이너를 갖게 되었으며 우리가 원할 때 인시던트가 열리도록 모든 규칙을 설정했습니다. 위의 설정에 따라 경고 조건이 시스템에서 설정한 임계값을 "위반"하는 이 동작을 인식하는 경우 인시던트가 생성됩니다. 이제 이 컨테이너를 정책에 연결하기만 하면 됩니다.

    정책은 인시던트를 분류하는 시스템입니다. 정책을 생성하면 들어오는 모든 인시던트를 구성하는 도구가 생성됩니다. 이 모든 수신 정보를 원하는 위치, 전송 빈도 및 위치를 New Relic에 알려주는 **[워크플로우](/docs/alerts-applied-intelligence/applied-intelligence/incident-workflows/incident-workflows/)** 에 정책을 연결할 수 있습니다.

    <CollapserGroup>
      <Collapser
        id="select-an-existing-policy"
        title="기존 정책 선택"
      >
        경고 조건에 연결할 정책이 이미 있는 경우 기존 정책을 선택합니다.

        [여기에서](/docs/alerts-applied-intelligence/new-relic-alerts/alert-policies/specify-when-alerts-create-incidents/) 정책을 만드는 방법에 대해 자세히 알아보세요.
      </Collapser>

      <Collapser
        id="create-a-new-policy"
        title="새 정책 만들기"
      >
        이 경고 조건에 대한 새 정책을 만들려면 여기 기회가 있습니다. 정책은 워크플로에 연결되어 있으며 워크플로는 사건에 대한 알림을 받는 빈도를 제어합니다. 웹 트랜잭션 시간과 관련된 모든 문제에 대해 가능한 한 빨리 파악하는 것과 개발자가 피로를 느끼고 정보 과부하로 인해 중요한 정보를 놓치기 시작하는 경고를 너무 많이 받지 않도록 하는 것 사이에 균형이 잘 잡혀 있습니다.

        정책은 하나 이상의 조건을 보유할 수 있습니다. 웹 트랜잭션 대기 시간을 모니터링하려는 경우 몇 가지 옵션이 있습니다.

        첫째, 정책당 단일 문제만 첨부하는 정책을 만들 수 있습니다(기본 옵션). 정책당 하나의 문제는 잡음을 줄이지만 즉각적인 조치가 필요합니다. 그러나 이는 **Response time: Example app** 뿐만 아니라 이 정책에 여러 조건을 첨부한 경우 어떤 경우에도 이 정책의 모든 인시던트가 하나의 단일 문제로 그룹화됨을 의미합니다.

        또는 조건당 하나의 문제를 만들 수 있습니다. 즉, **응답 시간: 예제 앱** 조건이 인시던트를 열 때마다 해당 모든 인시던트가 조건과 연결된 하나의 문제로 통합됩니다. 이 특정 사용 사례의 경우 웹 트랜잭션 시간의 대기 시간 문제를 모니터링하는 기본 목표를 충족하므로 이 옵션을 선택해야 합니다.

        또는 모든 사건에 대해 문제를 만들 수도 있습니다. 이 옵션은 가장 시끄럽지만 외부 시스템에 정보를 보내려는 경우 잘 작동할 수 있습니다.

        [여기에서](/docs/alerts-applied-intelligence/new-relic-alerts/alert-policies/specify-when-alerts-create-incidents/) 정책을 만드는 방법에 대해 자세히 알아보세요.
      </Collapser>
    </CollapserGroup>
  </Step>

    ## 추가 세팅 [#additional-settings]

    <CollapserGroup>
      <Collapser
        id="close-open-incidents"
        title="열린 인시던트 닫기"
      >
        대상 신호가 조건의 임계값에 표시된 기간 동안 비위반 상태로 돌아오면 인시던트가 자동으로 닫힙니다. 이 대기 시간을 복구 기간이라고 합니다.

        예를 들어 위반 행위가 "웹 거래 시간이 .50보다 깁니다. 5분에 한 번 이상 초"라고 입력하면 웹 트랜잭션 시간이 .50 이하일 때 인시던트가 자동으로 닫힙니다. 5분 연속.

        인시던트가 자동으로 종료되는 경우:

        1. 마감 타임스탬프는 복구 기간 시작으로 소급됩니다.

        2. 평가가 재설정되고 이전 인시던트가 종료된 시점부터 다시 시작됩니다.

           모든 조건에는 오래 지속되는 인시던트를 자동으로 강제 종료하는 인시던트 시간 제한 설정이 있습니다.

           자동으로 기본값은 3일이며 첫 번째 알림에는 기본 설정을 사용하는 것이 좋습니다.
