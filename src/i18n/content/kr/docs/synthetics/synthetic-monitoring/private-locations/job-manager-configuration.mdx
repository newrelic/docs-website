구성에서 [환경 변수](#environment-variables) 를 사용하여 [합성 작업 관리자](/docs/synthetics/synthetic-monitoring/private-locations/install-job-manager) 를 구성하는 방법을 알아봅니다.

<Callout variant="important">
  [사용자 정의 모듈](/docs/synthetics/synthetic-monitors/private-locations/containerized-private-minion-cpm-configuration/#custom-modules) , [영구 데이터 저장소](/docs/synthetics/synthetic-monitors/private-locations/containerized-private-minion-cpm-configuration/#preserve-data-volume) 및 [사용자 정의 환경 변수](/docs/synthetics/synthetic-monitors/private-locations/containerized-private-minion-cpm-configuration/#vars-scripted-monitors) 는 현재 합성 작업 관리자에 대해 지원되지 않습니다.
</Callout>

참고로 New Relic은 합성 작업 관리자 파일에 대한 수정 사항에 대해 책임을 지지 않습니다.

## 환경 변수 [#environment-variables]

환경 변수를 사용하면 특정 환경 및 기능 요구 사항을 충족하도록 합성 작업 관리자 구성을 미세 조정할 수 있습니다.

## Kubernetes 및 Docker에 대한 크기 조정 고려 사항 [#kubernetes-sizing]

<Callout variant="tip">
  Docker 특정 크기 고려 사항은 곧 제공될 예정입니다.
</Callout>

대규모 환경에서 작업하는 경우 합성 모니터를 효율적으로 실행하기 위한 최소 요구 사항을 충족하도록 작업 관리자 구성을 사용자 지정해야 할 수 있습니다. 다음을 포함하여 많은 요소가 합성 작업 관리자 배포의 크기 요구 사항에 영향을 줄 수 있습니다.

* 예상 사용량에 따라 모든 런타임이 필요한 경우
* 모니터 유형(ping, 단순 또는 스크립트 브라우저, 스크립트 API)별 분당 작업 수
* 작업 시간(약 3분에 시간 초과되는 작업 포함)
* 작업 실패 횟수입니다. 작업 실패의 경우 모니터가 내장된 3/3 재시도 논리를 제공하는 데 실패하기 시작할 때 자동 재시도가 예약됩니다. 이러한 추가 작업은 합성 작업 관리자의 처리량 요구 사항에 추가됩니다.

아래 나열된 크기 조정 구성 설정 외에도 동일한 개인 위치 키로 추가 합성 작업 관리자를 배포하여 여러 환경에서 작업 부하를 분산할 수 있습니다.

    K8s 합성 작업 관리자가 사용하는 각 런타임은 [helm 차트](https://github.com/newrelic/helm-charts/tree/master/charts/synthetics-job-manager) 에서 값을 설정하여 독립적으로 크기를 조정할 수 있습니다.

    기본값 `1` 에서 `ping-runtime.replicaCount` 설정을 늘려 핑 모니터 로드를 실행하는 데 도움이 되도록 추가 핑 런타임을 시작할 수 있습니다.

    Node.js API 및 Node.js 브라우저 런타임은 `parallelism` 및 `completions` 설정의 조합을 사용하여 독립적으로 크기가 조정됩니다. 이러한 설정에 대한 이상적인 구성은 고객 요구 사항에 따라 달라집니다.

    `parallelism` 설정은 K8 클러스터에서 동시에 실행할 수 있는 해당 유형의 작업 수(Node.js API 또는 Node.js 브라우저)를 제어합니다. `parallelism` 설정은 컨테이너화된 비공개 미니언(CPM)의 `synthetics.heavyWorkers` 구성과 동일합니다. K8s 클러스터에 [리소스 요청 및 제한 값](/docs/synthetics/synthetic-monitoring/private-locations/install-job-manager/#kubernetes-requirements) 을 기반으로 이 수의 동시 런타임을 실행하는 데 사용할 수 있는 리소스가 충분한지 확인하십시오.

    `completions` 설정은 이 런타임 유형의 추가 작업을 시작하기 위해 1분마다 `CronJob` 일정을 기다리기 전에 완료해야 하는 이 유형의 작업 수를 제어합니다. `completions` 설정은 `parallelism` 설정을 고려하여 1분에 완료할 수 있는 작업의 추정치로 설정해야 합니다. `completions` 가 1보다 크면 완료된 작업은 `kubectl get pods` 에 표시되지만 해당 작업이 완료 또는 실패로 표시되는 즉시 리소스가 해제됩니다.

    각 런타임에 대한 `completions` 및 `parallelism` 의 기준을 계산하는 데 도움이 되는 공식:

    ```m
    completions = 60 / avg job duration
    parallelism = jobs per minute / completions
    ```

    런타임마다 작업 기간과 속도가 다를 수 있습니다. 다음 쿼리를 사용하여 개인 위치에 대한 평균 기간 및 요금을 얻을 수 있습니다.

    ```sql
    # non-ping average job duration by runtime type
    FROM SyntheticCheck SELECT average(duration) AS 'avg job duration' WHERE type != 'SIMPLE' AND location = 'YOUR_PRIVATE_LOCATION' FACET type SINCE 1 hour ago

    # non-ping jobs per minute by runtime type
    FROM SyntheticCheck SELECT rate(uniqueCount(id), 1 minute) AS 'jobs per minute' WHERE type != 'SIMPLE' AND location = 'YOUR_PRIVATE_LOCATION' FACET type SINCE 1 hour ago
    ```

    <Callout variant="tip">
      위 쿼리는 현재 결과를 기반으로 합니다. 개인 위치에 결과가 없거나 작업 관리자가 최상의 성능을 발휘하지 못하는 경우 쿼리 결과가 정확하지 않을 수 있습니다. 이 경우 작업이 완료(충분한 완료)로 1분 이상 채워지고 대기열이 증가하지 않을 때까지(충분한 병렬 처리) `completions` 및 `parallelism` 에 대해 몇 가지 다른 값을 시도하십시오.
    </Callout>

    작업을 완료하는 데 시간이 더 오래 걸리면 1분을 채우는 데 필요한 완료 횟수가 줄어들지만 병렬 포드가 더 많이 필요합니다. 마찬가지로 분당 더 많은 작업을 처리해야 하는 경우 더 많은 병렬 포드가 필요합니다.

    `completions` 값을 사용하여 &#x7B; `parallelism` } 값의 좋은 시작점을 얻을 수 있습니다. `parallelism` 설정은 분당 실행할 수 있는 작업 수에 직접적인 영향을 줍니다. 값이 너무 작으면 대기열이 커질 수 있습니다. 값이 너무 크면 노드가 리소스 제약을 받을 수 있습니다.

    대기열을 0으로 유지하기 위해 `parallelism` 설정이 제대로 작동하는 경우 `60 / avg job duration` 에서 처음에 계산된 것보다 `completions` 에 더 높은 값을 설정하는 것은 나쁜 생각이 아닙니다. 이는 작업이 예상보다 빨리 완료되는 경우를 수용하고 1분이 경과하기 전에 `completions` 값에 도달하는 시나리오를 방지하는 데 도움이 될 수 있습니다. 가장 효율적인 `completions` 설정은 전체 1분을 작업 실행 및 완료로 채웁니다. 완료가 충분하지 않은 경우 해당 런타임에 대한 작업 처리는 다음 `CronJob` 가 다른 `completions` 세트를 가동할 때까지 몇 초 동안 유휴 상태로 유지됩니다.
