애플리케이션 아키텍처가 무엇이든 당사는 [로그를 New Relic에 보고하는 솔루션을](/docs/logs/get-started/get-started-log-management#integrate-logs) 보유하고 있습니다.

<InlinePopover type="apm"/>에이전트, 인프라 에이전트 또는 OpenTelemetry 통합을 사용하여 로그 데이터를 보고하면 다른 플랫폼 UI 경험의 맥락에서 로그 데이터를 볼 수 있는 기능인 <DNT>**logs in context**</DNT> 을 얻게 됩니다. 예를 들어, 몇 가지 예를 UI 들면 APM, 관측, 분리 추적, 내어스 인박스(errors inbox)와 같은 경험에서 데이터를 거부하는 것을 볼 수 있습니다. 이러한 상황별 지식을 사용하면 로그 데이터를 수동으로 검색할 필요 없이 앱이나 호스트의 성능을 더 깊이 파고들 수 있습니다.

컨텍스트 내 로그가 앱과 호스트 문제의 근본 원인을 찾는 데 어떻게 도움이 되는지 알아보려면 이 짧은 동영상(약 20분)을 시청하세요. 4분):

## 플랫폼 전반에 걸친 문제의 근본 원인 확인 [#compatibility-requirements]

모든 애플리케이션 및 인프라 데이터를 단일 솔루션으로 통합하면 문제의 근본 원인을 더 빨리 파악할 수 있습니다. 컨텍스트에 로그인하면 의미 있는 패턴과 추세를 빠르게 확인할 수 있습니다.

플랫폼의 다른 부분에서 모든 로그의 범위를 좁히려고 추가 시간을 소비하지 마십시오. 대신 컨텍스트에서 로그를 활성화하여 문제를 식별하고 해결하는 데 필요한 정확한 로그 라인을 확인하십시오.

## 컨텍스트의 APM 로그인 [#apm-automatic]

우리의 <InlinePopover type="apm"/>에이전트에는 로그 포워딩을 포함한 로그가 기본적으로 활성화되어 있습니다. 컨텍스트에서 <InlinePopover type="apm"/>로그를 가져오고 사용하는 방법에 대한 자세한 내용은 [APM 로그를](/docs/apm/new-relic-apm/getting-started/get-started-logs-context) 참조하세요.

## 다음은 뭐지? [#what-next]

APM 또는 인프라 모니터링을 위해 컨텍스트에서 로그를 설정한 후 New Relic UI에서 로깅 데이터를 최대한 활용하십시오.

* [로그 UI를](/docs/logs/log-management/ui-data/use-logs-ui)사용하여 플랫폼 전체에서 로깅 데이터를 탐색합니다.
* [APM UI](/docs/logs/log-management/ui-data/use-logs-ui/#links) 의 앱 성능 컨텍스트에서 로그를 확인합니다. [분산 추적](/docs/apm/distributed-tracing/ui-data/understand-use-distributed-tracing-data) , 스택 추적, 애플리케이션 로그 등으로 [오류](/docs/apm/apm-ui-pages/error-analytics/errors-page-find-fix-verify-problems/) 를 해결합니다.
* [경보](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-alert-conditions/)를 설정하십시오.
* [데이터를 조회](/docs/query-your-data/explore-query-data/get-started/introduction-querying-new-relic-data/) 하고 [대시보드를 작성](/docs/query-your-data/explore-query-data/dashboards/introduction-dashboards/)하십시오.
