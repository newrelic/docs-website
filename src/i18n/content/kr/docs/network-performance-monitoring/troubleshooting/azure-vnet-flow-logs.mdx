[Azure VNet Flow 로그 모니터링을 설정](/docs/network-performance-monitoring/setup-performance-monitoring/cloud-flow-logs/azure-vnet-flow-log-monitoring) 하고 문제가 발생하는 경우, 다음 섹션을 사용하여 VNet Flow 로그 전달자의 일반적인 문제를 진단하세요.

## 뉴렐릭에 플로우 로그 데이터 없음 [#no-data]

flow 로그 데이터가 뉴렐릭에 표시되지 않으면 다음을 확인하세요:

* `NR_LICENSE_KEY` 이(가) 올바르게 설정되었는지 확인하세요.
* VNet Flow 로그가 활성화되어 있고 스토리지 계정에 PT1H.json Blob을 활발히 쓰고 있는지 확인하세요.
* Event Grid 구독이 활성화되어 있고 `BlobCreated` 이벤트를 Event Hub로 라우팅하는지 확인하세요.
* `EVENTHUB_CONSUMER_CONNECTION` 및 `EVENTHUB_NAME` 설정이 이벤트를 수신하는 Event Hub와 일치하는지 확인하세요.
* Azure에서 Function App 로그에 오류가 있는지 확인하세요. 자세한 로그를 보려면 `AzureFunctionsJobHost__logging__logLevel__Function` 앱 설정을 `Debug`(으)로 설정하세요.

## 중복 플로우 로그 레코드 [#duplicate-data]

커서 메커니즘은 정상적인 조건에서 중복 처리를 방지합니다. 중복이 표시되는 경우 다음을 확인하세요:

* `CURSOR_STORAGE_CONNECTION` 이(가) 올바르게 설정되어 있고 테이블에 액세스할 수 있는지 확인하세요.
* 각 Event Hub 파티션을 처리하는 소비자의 활성 인스턴스가 하나만 있는지 확인하세요.

## 건너뛰는 이벤트(포이즌 이벤트 보호) [#poison-events]

동일한 Blob을 처리하는 데 `MAX_CONSECUTIVE_FAILURES` 번 연속 실패(기본값 5)하면 포워더는 이를 포이즌 이벤트로 표시하고 무한 재시도를 방지하기 위해 건너뜁니다. Function App 로그에서 초기 실패의 근본 원인을 확인하고, 문제를 해결한 다음, `nrvnetflowlogscursors` 테이블에서 해당 행을 삭제하여 커서를 재설정하세요. 포워더는 뉴렐릭 전송 오류(429, 5xx 및 재시도 가능한 네트워크 오류)를 먼저 재시도합니다. 실패 카운터는 이러한 재시도가 소진되거나 다른 하드 실패가 발생한 후에만 증가합니다.
