프로필은 `ktranslate` 수집하는 SNMP OID와 수신된 SNMP 트랩 메시지를 처리하는 방법을 모두 정의합니다. 기존 프로필을 수정하거나 새 프로필을 생성하려면 여기 지침을 따르세요.

## 요구 사항 [#requirements]

프로필을 생성하거나 수정하는 경우 다음 사항이 있는지 확인하세요.

* [kentik/snmp-profiles](https://github.com/kentik/snmp-profiles) 저장소를 포크하고 변경 사항에 대한 풀 [요청을 생성할 수 있는 GitHub 계정 .](https://docs.github.com/en/get-started/signing-up-for-github/signing-up-for-a-new-github-account)
* 프로필 폴링을 위한 네트워크 장치에서의 [SNMP 워크입니다](/docs/network-performance-monitoring/troubleshooting/snmp-walk/#capturing-snmpwalk) .
* SNMP 트랩 메시지의 경우 메시지를 정의하는 공급업체의 MIB에도 액세스할 수 있어야 합니다.

<Callout variant="important">
  New Relic은 [kentik/snmp-profiles](https://github.com/kentik/snmp-profiles) 내용에 대한 직접적인 지원을 제공하지 않습니다. 기존 SNMP 프로필이 잘못된 데이터 또는 다른 중단 동작을 반환하는 경우 [kentik/snmp-profiles](https://github.com/kentik/snmp-profiles) 저장소에서 문제를 엽니다. `ktranslate` 에 대한 타사의 SNMP 프로필 수정 또는 생성에 대한 일정을 제공할 수 없습니다.

  New Relic 계정과 관련된 서비스 수준에 따라 New Relic 글로벌 기술 지원의 도움을 받을 수 있습니다. 당사의 [글로벌 기술 지원 서비스](/docs/licenses/license-information/general-usage-licenses/global-technical-support-offerings/#tstools) 에 따라 당사는 문제가 New Relic 제품에 발생한 것인지 아니면 당사의 통제 및 권한 범위를 벗어난 원인으로 인해 발생한 것인지 판단하기 위해 최선을 다할 것입니다. 당사의 통제 범위를 벗어나는 제3자 도구 관련 문제의 경우, 당사는 업스트림 오픈 소스 프로젝트에 개선 사항, 패치 또는 기타 기능 개발을 제공하기 위해 상업적으로 합당한 노력을 기울일 것입니다.
</Callout>

    ## 배경

    SNMP 폴링 프로필을 생성할 때 다음과 같은 일반적인 단계를 따르게 됩니다.

    1. 샘플 장치에서 [SNMP Walk를](/docs/network-performance-monitoring/troubleshooting/snmp-walk/#capturing-snmpwalk) 수집합니다.

    2. 걷기 결과에 따라 장치에서 지원되는 MIB를 식별합니다.

    3. 지원되는 MIB에서 다음과 같이 수집하려는 측정항목을 식별합니다.

       * 카운터, 게이지 또는 열거된 측정항목이 있는 스칼라 OID
       * 카운터, 게이지 또는 열거된 측정항목이 포함된 테이블 형식 OID
       * 태그로 사용할 수 있는 문자열 값을 반환하는 스칼라 또는 테이블 형식의 OID

    4. SNMP 프로필을 생성하거나 편집합니다.

                   * 엔터티에 이미 매핑된 기존 값을 사용합니다.
                   * [newrelic/entity-definitions](https://github.com/newrelic/entity-definitions) 저장소에 대한 풀 요청을 생성하여 공급자 값을 기존 정의에 매핑하거나 새 정의를 구축합니다.
                   * 자신만의 맞춤형 대시보드 환경을 만들어 데이터를 확인하세요.
                 </Callout>

                 id="sysobjectid"
                 title="'sysobjectid' 섹션"
               >
                 이 [](https://oidref.com/1.3.6.1.2.1.1.2)섹션에서는 `ktranslate` [SNMP 검색](/docs/network-performance-monitoring/troubleshooting/snmp-discovery-unexpected-profile/) 중에 장치를 이 프로필에 매핑하는 데 사용하는 sysObjectID를 식별합니다.

                 여기의 구문은 OID 목록입니다. 개별적으로 주석을 달지 않고도 대규모 장치 그룹을 설명하기 위해 `*` 기호를 사용하여 와일드카드 일치에 대한 지원이 제공됩니다. 또한 여기에 YAML 주석을 추가하면 `ktranslate` New Relic으로 전송되는 페이로드에 장식될 `tags.kentik.model` 값을 설정할 수 있습니다.

                 ```yaml
                 sysobjectid:
                   - 1.3.6.1.4.1.318.1.3.17    # APC SmartUPS 3 Phase
                   - 1.3.6.1.4.1.318.1.3.*     # APC UPS
                   - 1.3.6.1.4.1.318.*         # APC device
                 ```
               </Collapser>

[net-snmp.yml](https://github.com/kentik/snmp-profiles/blob/main/profiles/kentik_snmp/_general/net-snmp.yml) 프로필에서 예를 볼 수 있습니다.

                 ```yaml
                 matches:
                   "^pfsense": pf_sense.yml
                 ```

               <Collapser
                 id="metrics"
                 title="'측정항목' 섹션"
               >
                 이 섹션에서는 이 프로필을 사용할 때 `ktranslate` 대상으로 삼는 스칼라 및 테이블 형식 OID에 대해 간략하게 설명합니다. 스칼라 및 테이블 형식 OID의 구문은 약간씩 다르며 해당 탭에서 자세히 설명합니다. 높은 수준에서 여기에 있는 구문은 컬렉션을 완전히 제어할 수 있는 다양한 설정이 포함된 OID 목록입니다.

                 ```yaml
                 metrics:
                   # Scalar OID
                   - MIB: $MIB_NAME
                     symbol:
                       OID: $SCALAR_OID
                       name: $OID_NAME_FROM_MIB
                   # Tabular OIDs
                   - MIB: $MIB_NAME
                     table:
                       OID: $TABLE_OID
                       name: $TABLE_NAME
                     symbols:
                       - OID: $COLUMN_OID
                         name: $TABULAR_OID_NAME_FROM_MIB
                 ```
               </Collapser>

               <Collapser
                 id="metric-tags"
                 title="'metric_tags' 섹션"
               >
                 이 섹션에서는 관련 측정항목으로 장식될 속성에 대해 간략하게 설명합니다. 차원 측정항목은 `count` 유형이어야 합니다. | `summary` \| `gauge`. 따라서 문자열 값은 측정항목으로 직접 지원되지 않으며 `metric_tag` 식별자를 사용하여 기존 측정항목에 장식되어야 합니다.

                 이 식별자는 개별 MIB 테이블 구성은 물론 전역 프로필 설정에도 배치될 수 있습니다. 사용 사례는 `upsOutletGroupStatusTable` 테이블의 콘센트 그룹에 대한 [upsOutletGroupStatusName](https://oidref.com/1.3.6.1.4.1.318.1.1.1.12.1.2.1.2) 이거나 스칼라 OID로 표시되는 UPS 장치의 [upsAdvIdentSerialNumber](https://oidref.com/1.3.6.1.4.1.318.1.1.1.1.2.3) 입니다.

                 ```
                 metrics:
                   - MIB: PowerNet-MIB_UPS
                     table:
                       OID: 1.3.6.1.4.1.318.1.1.1.12.1.2
                       name: upsOutletGroupStatusTable
                     symbols:
                       - ...
                     metric_tags:
                       - column:
                           OID: 1.3.6.1.4.1.318.1.1.1.12.1.2.1.2
                           name: upsOutletGroupStatusName
                 metric_tags:
                   - column:
                       OID: 1.3.6.1.4.1.318.1.1.1.1.2.3.0
                       name: upsAdvIdentSerialNumber
                 ```
               </Collapser>

    ## 배경

    SNMP 트랩 메시지 프로필(경고, 이벤트, 알림 또는 알림이라고도 함) 생성은 매우 간단합니다. 일반적으로 다양한 트랩 메시지 옵션을 이해하려면 공급업체의 MIB 파일만 조사하면 됩니다.

          이 예에서는 VMware vCenter 어플라이언스에서 사용할 수 있는 다양한 트랩 메시지를 정의하는 [VMWARE-VC-EVENT-MIB.mib를](https://www.circitor.fr/Mibs/Html/V/VMWARE-VC-EVENT-MIB.php) 사용합니다.

          <Callout variant="important">
            장치에 의한 SNMP 트랩 메시지 생성은 전적으로 해당 장치의 구성에 따라 달라집니다. 이러한 메시지는 원치 않는 메시지이며 `ktranslate` 은 이 시나리오에서 메시지를 처리하고 이를 New Relic에 전달하는 수신 엔드포인트일 뿐입니다. 또한 메시지는 MIB에서 정의한 개별 이벤트에서만 생성됩니다.

            New Relic에 표시되지 않는 데이터가 표시될 것으로 예상되는 경우 장치 구성을 확인하고 메시지를 생성해야 하는 이벤트가 전달되었는지 확인해야 합니다.
          </Callout>

          이 MIB에서 우리가 관심을 갖는 부분을 분리하기 위해 사용할 관련 조각은 다음과 같습니다.

          ```
          -- **********************************************************
          -- Notifications by an agent that implements this mib module
          -- **********************************************************

          vpxdAlarm NOTIFICATION-TYPE
              OBJECTS { vmwVpxdTrapType, vmwVpxdHostName, vmwVpxdVMName,
          	      vmwVpxdOldStatus, vmwVpxdNewStatus, vmwVpxdObjValue }
              STATUS      obsolete
              DESCRIPTION
                  "This notification is sent on entity alarm state change, by the vCenter Server SNMP agent.
                   This information is also available through the vSphere client, through the Alarms screen,
                   or through the Managed Object Browser(MOB) interface for alarms at
                   https://<vCenter Server machine address>/mob/?moid=AlarmManager.
                   Listing individual objects of a specific type or ID can be done through the PropertyCollector SDK API.
                   See http://www.vmware.com/support/developer/vc-sdk/visdk2xpubs/ReferenceGuide/vmodl.query.PropertyCollector.html
                   for details."
              ::= { vmwVCNotifications 201 }

          vpxdDiagnostic NOTIFICATION-TYPE
              STATUS      current
              DESCRIPTION
                  "This notification is sent on starting or restarting vCenter Server,
                   on requesting a test notification explicitly, and can also be
                   configured to be sent periodically at a specified time interval via
                   vCenter Server configuration by the vCenter Server SNMP agent."
              ::= { vmwVCNotifications 202 }

          vpxdAlarmInfo NOTIFICATION-TYPE
              OBJECTS { vmwVpxdTargetObjType, vmwVpxdOldStatus, vmwVpxdNewStatus, vmwVpxdObjValue, vmwVpxdTargetObj }
              STATUS      current
              DESCRIPTION
                  "This notification is sent on entity alarm state change, by the vCenter Server SNMP agent.
                   This information is also available through the vSphere client, through the Alarms screen,
                   or through the Managed Object Browser(MOB) interface for alarms at
                   https://<vCenter Server machine address>/mob/?moid=AlarmManager.
                   Listing individual objects of a specific type or ID can be done through the PropertyCollector SDK API.
                   See http://www.vmware.com/support/developer/vc-sdk/visdk2xpubs/ReferenceGuide/vmodl.query.PropertyCollector.html
                   for details."
              ::= { vmwVCNotifications 203 }

          -- **********************************************************
          -- Notification parameters
          -- **********************************************************

          vmwVpxdTrapType OBJECT-TYPE
              SYNTAX     SnmpAdminString
              MAX-ACCESS accessible-for-notify
              STATUS     obsolete
              DESCRIPTION
          	"This is the alarm notification type."
              ::= { vmwVC 301 }

          vmwVpxdHostName OBJECT-TYPE
              SYNTAX     SnmpAdminString
              MAX-ACCESS accessible-for-notify
              STATUS     obsolete
              DESCRIPTION
          	"This is the name of the vSphere host in the notification."
              ::= { vmwVC 302 }

          vmwVpxdVMName OBJECT-TYPE
              SYNTAX     SnmpAdminString
              MAX-ACCESS accessible-for-notify
              STATUS     obsolete
              DESCRIPTION
                  "This is the name of the VM in the notification."
              ::= { vmwVC 303 }

          vmwVpxdOldStatus OBJECT-TYPE
              SYNTAX     SnmpAdminString
              MAX-ACCESS accessible-for-notify
              STATUS     current
              DESCRIPTION
          	"This is the old status in the notification."
              ::= { vmwVC 304 }

          vmwVpxdNewStatus OBJECT-TYPE
              SYNTAX     SnmpAdminString
              MAX-ACCESS accessible-for-notify
              STATUS     current
              DESCRIPTION
          	"This is the new status in the notification."
              ::= { vmwVC 305 }

          vmwVpxdObjValue OBJECT-TYPE
              SYNTAX     VmwLongSnmpAdminString
              MAX-ACCESS accessible-for-notify
              STATUS     current
              DESCRIPTION
          	"This is the current object value in the notification."
              ::= { vmwVC 306 }

          vmwVpxdTargetObj OBJECT-TYPE
              SYNTAX     SnmpAdminString
              MAX-ACCESS accessible-for-notify
              STATUS     current
              DESCRIPTION
          	"This is the current object in the notification. This may be
          	one of esx host name, vm name, or other. This value must not be empty."
              ::= { vmwVC 307 }

          vmwVpxdTargetObjType OBJECT-TYPE
              SYNTAX   INTEGER { unknown(1), host(2), vm(3), other(4)}
              MAX-ACCESS accessible-for-notify
              STATUS     current
              DESCRIPTION
          	"This is the alarm target object type."
              ::= { vmwVC 308 }
          ```

          이 스니펫에는 2개의 주요 섹션이 포함되어 있습니다.

          * <DNT>
              **Notifications by an agent that implements this MIB module**
            </DNT>

            : 고유한 OID로 식별되는 실제 트랩 메시지의 정의입니다.

            * [vmwVCNotifications](https://www.circitor.fr/Mibs/Html/V/VMWARE-VC-EVENT-MIB.php#vmwVCNotifications): 이는 사용 가능한 세 가지 정의된 메시지 유형( [vpxdAlarm](https://www.circitor.fr/Mibs/Html/V/VMWARE-VC-EVENT-MIB.php#vpxdAlarm), [vpxdDiagnostic](https://www.circitor.fr/Mibs/Html/V/VMWARE-VC-EVENT-MIB.php#vpxdDiagnostic) 및 [vpxdAlarmInfo )](https://www.circitor.fr/Mibs/Html/V/VMWARE-VC-EVENT-MIB.php#vpxdAlarmInfo) 을 다루는 루트 알림 OID입니다.

          * <DNT>
              **Notification parameters**
            </DNT>

            : 일반적으로 `variable bindings` 또는 `varbinds` 이라고 하며 이는 SNMP 에이전트가 트랩 메시지에 장식으로 보낼 수 있는 고유한 `key:value` 공지 데이터 쌍입니다.

            * 이 MIB에는 8개의 가능한 varbind가 있으며 모두 `1.3.6.1.4.1.6876.4.3.301` - `1.3.6.1.4.1.6876.4.3.308` 범위의 OID로 참조됩니다. 이는 모든 공급업체와 MIB에 고유합니다.

          </Callout>

## 사용자 정의 프로필을 비공개로 사용 [#private]

[kentik/snmp-profiles](https://github.com/kentik/snmp-profiles) 저장소의 콘텐츠는 컨테이너 또는 서비스를 빌드할 때 기본적으로 `ktranslate` 에 로드되므로 모든 인스턴스에 전역적으로 적용됩니다.

전체 사용자 정의 옵션을 지원하여 자신만의 컬렉션 설정과 패턴을 유지하려면 다음을 수행하세요.

1. 저장소를 복제합니다.
2. 적절하다고 생각되는 대로 변경하세요.
3. 시작 시 사용자 정의 프로필 디렉터리를 Docker 컨테이너 또는 Linux 서비스에 로드합니다.

이를 수행하는 방법은 사용자 정의된 파일을 `etc/ktranslate/profiles/` 디렉토리 내의 KTranslate 인스턴스에 매핑하는 것입니다.

`snmp-profiles` 바로 위의 디렉터리에 있다고 가정하면 추가하는 명령은 다음과 같습니다.

```shell
`pwd`/snmp-profiles/profiles:/etc/ktranslate/profiles
```

Docker의 경우 이를 [볼륨 마운트](https://docs.docker.com/storage/volumes/#start-a-container-with-a-volume) 로 추가하면 다음과 유사합니다.

```shell
docker run -d --name ktranslate-$CONTAINER_SERVICE --restart unless-stopped --pull=always -p 162:1620/udp \
  -v `pwd`/snmp-base.yaml:/snmp-base.yaml \
  -v `pwd`/snmp-profiles/profiles:/etc/ktranslate/profiles
  -e NEW_RELIC_API_KEY=$NR_LICENSE_KEY \
kentik/ktranslate:v2 \
  -snmp /snmp-base.yaml \
  -nr_account_id=$NR_ACCOUNT_ID \
  -metrics=jchf \
  -tee_logs=true \
  -service_name=$CONTAINER_SERVICE \
  -snmp_discovery_on_start=true \
  -snmp_discovery_min=180 \
  nr1.snmp
```

Linux 서비스의 경우 로컬 호스트에서 `/etc/ktranslate/profiles` 의 콘텐츠를 편집합니다.

<Callout variant="tip">
  귀하의 환경에서 일관되지 않은 동작의 위험을 제거하려면 컨테이너 또는 서비스를 시작할 때마다 사용자 정의 버전의 프로필을 전달해야 합니다.
</Callout>

## 공개 저장소에 기여 [#public]

오픈 소스 저장소로서 새 프로필을 만들거나 기존 프로필을 개선하여 모든 사용자의 모니터링 기능을 향상하려는 모든 사람의 기여를 환영합니다.

New Relic은 타사 저장소의 프로필을 검토하는 과정에 관여하지 않습니다. 변경 및 업데이트에 대한 모든 요청은 Github 문제를 통해 Kentik의 관리자에게 전달되어야 합니다.

프로필이 병합된 직후 CI/CD 자동화를 통해 새 버전의 Docker 이미지에서 새 SNMP 프로필을 사용할 수 있습니다.

## 어떤 종류의 OID를 포함해야 합니까? [#oids]

많은 경우 SNMP는 많은 데이터를 제공하지만 실행 가능한 정보는 거의 제공하지 않습니다. 위의 [샘플 SNMP 워크](#snmp-polling) 에는 OID의 결과 2,022개 행이 있으며 [결과 프로필은](https://github.com/kentik/snmp-profiles/blob/main/profiles/kentik_snmp/apc/apc_ups.yml) 총 50개 정도의 OID를 수집합니다. [SRE(사이트 신뢰성 엔지니어링) 분야](https://sre.google/sre-book/monitoring-distributed-systems/) 에서 정의한 <DNT>**"Golden Signals"**</DNT> 의 기본 패턴을 따릅니다. 장치가 수행할 것으로 예상되는 모든 기능을 수행하는 데 방해가 되는 요소가 있는지 알려주는 데이터 수집에 집중하세요. 이를 토대로 해당 기능을 얼마나 잘 수행하는지 알려주는 측정값을 수집해야 합니다.

**예시 1:**

VPN 집중 장치로 작동하는 장치의 경우 다음과 같은 높은 수준의 시스템 측정항목을 수집합니다.

* CPU 및 메모리 사용률
* 팬 고장 등으로 인해 장치가 자동으로 종료되지 않도록 하는 하드웨어 센서 정보
* 집계된 연결 및 처리량에 대해 알려주는 OID

**예 2:**

네트워크 어플라이언스에서 실행 중인 모든 프로세스를 나열하는 OID 테이블은 사용 가능하지만 도움이 되지 않는 데이터의 예입니다.

실행 중인 모든 프로세스를 수집하는 것이 유용할 수 있지만 이는 어플라이언스이므로 일반적으로 내부에서 실행되는 프로세스에 대해 어떤 작업도 수행할 수 있는 기능이나 필요성이 없습니다. 실제로 아무것도 할 수 없는 수백 개의 항목이 포함된 테이블을 폴링하고 저장하는 것은 효율적이지 않습니다.
