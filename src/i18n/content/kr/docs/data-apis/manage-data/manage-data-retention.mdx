New Relic의 **Data Retention** UI 페이지에서는 다양한 유형의 데이터가 New Relic 데이터베이스(NRDB)에 저장되는 기간을 검토하고 관리할 수 있습니다. 민감한 정보를 보호하고 쿼리를 더 빨리 반환하려면 데이터를 더 짧은 기간 동안 보관하고, 장기간에 걸쳐 데이터와 추세를 보고 분석할 수 있도록 더 장기간 보관합니다.

## 요구 사항 [#requirements]

이 문서는 [사용량 기반 가격 책정 모델을 사용](/docs/accounts/original-accounts-billing/original-product-based-pricing/overview-pricing-models) 하는 조직에 적용됩니다. 원래 가격 모델에 대한 데이터 보존은 [원래 데이터 보존](/docs/accounts/original-accounts-billing/product-based-pricing/overview-data-retention-components) 을 참조하십시오.

데이터 보존을 편집하려면 Pro 또는 Enterprise 에디션이 필요합니다.

권한 관련 요구 사항:

* [최신 사용자 모델](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models) : [**결제** 관리 설정](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts#admin-settings) 이 필요합니다.
* [원래 사용자 모델](/docs/accounts/original-accounts-billing/original-users-roles/users-roles-original-user-model) : **관리자** , **소유자** 및 **데이터 보존 추가 기능 관리자** 가 가지고 있는 **Insights 이벤트 보존** 기능이 필요합니다.

## 데이터 보존 UI 찾기 [#find-ui]

**데이터 보존** UI 페이지는 [데이터 관리 허브](/docs/data-apis/manage-data/manage-your-data) 에 있습니다.

## 다양한 데이터 유형에 대한 보존 세부정보 [#data-details]

이 섹션에서는 변경할 수 없는 보존 설정이 있는 일부 데이터 유형을 포함하여 몇 가지 다른 유형의 데이터에 대한 세부 정보를 제공합니다.

<CollapserGroup>
  <Collapser
    id="dimensional-metrics"
    title="차원 측정항목(Metric API)"
  >
    원시 [차원 측정항목](/docs/data-apis/understand-data/new-relic-data-types/#dimensional-metrics) 데이터( `Metric` 데이터 유형)는 몇 가지 다른 시간 범위에 대해 집계(롤업)되며 이러한 집계는 다른 기간 동안 유지됩니다. 원시 차원 메트릭 데이터 요소는 30일 동안 저장됩니다. 집계된 모든 데이터 포인트(1분에 집계 시작)는 13개월 동안 유지됩니다.

    원시 데이터 수집만 사용량 계산에 포함되며 집계된 데이터는 계산에 포함되지 않습니다.

    측정항목 보존 기간은 편집할 수 없습니다.
  </Collapser>

  <Collapser
    id="metric-timeslice"
    title="메트릭 타임슬라이스 데이터(APM, 브라우저, 모바일)"
  >
    [메트릭 타임슬라이스 데이터](/docs/using-new-relic/data/understand-data/new-relic-data-types#timeslice-data) 는 APM(사용자 지정 메트릭 및 기본 메트릭 모두), 모바일 및 브라우저 에이전트에 의해 생성됩니다. 메트릭 타임슬라이스는 메트릭 이름, 메트릭이 나타내는 시간 세그먼트("타임슬라이스") 및 숫자 값(측정)의 세 부분으로 구성됩니다. APM, 브라우저 및 모바일 에이전트는 오류율, 대역폭 사용량 및 가비지 수집 시간과 같은 다양한 성능 메트릭에 대해 분당 수천 개의 메트릭 타임슬라이스를 수집할 수 있습니다.

    메트릭 타임슬라이스 데이터는 다음과 같이 집계됩니다.

    * 8일 동안 저장된 처음 - 1분 롤업

      초기 원시 데이터 수집만 사용량 계산에 포함되며 집계된 데이터는 계산에 포함되지 않습니다.

## 보존 값 수정 [#adjust-retention]

[요구 사항을 충족하는](#requirements) 경우 일부 데이터 유형의 보존 기간을 늘릴 수 있습니다. 예를 들어 APM 데이터의 보존 값을 줄여 해당 데이터 쿼리 성능을 최적화할 수 있습니다. 반대로, 더 긴 시간 범위에 걸쳐 보다 세분화된 분석을 수행하거나 데이터 규정 준수 요구 사항을 충족할 수 있도록 일부 데이터 유형에 대한 보존을 늘릴 수 있습니다.

네임스페이스의 [기본 보존 기간](#retention-periods) 에 30일의 보존 기간이 추가될 때마다 비용은 매월 수집된 GB당 $0.05입니다.

### 보존 가치 감소 [#reduce-retention]

특정 데이터 원본이 유지되는 일 수를 줄이면 해당 원본 유형의 기존 데이터가 즉시 사라지지 않습니다. 해당 데이터는 최대 몇 주 동안 조회되고 쿼리에 반환될 수 있습니다. 실수로 데이터를 삭제하지 않도록 보호 장치가 마련되어 있지만, 이는 또한 보존 값을 줄이는 것이 [데이터를 빠르게 제거](#data-deletion) 하는 좋은 전략이 아님을 의미합니다.

즉, 변경 사항은 12시간 이내에 시스템에 등록되고 조정된 소스 유형의 새 데이터는 새 기간 동안 유지됩니다.

계약된 보존 값 아래로 데이터 보존을 줄이면 New Relic 데이터 수집 비용이 줄어들지 않습니다. 반대로, 대상 원격 측정 데이터 유형에 대한 보존을 늘리면 데이터 수집 요금 조정에 대해 New Relic과의 대화가 촉발될 수 있습니다.

## 데이터 삭제 [#deleting-data]

[원격 측정 데이터](/docs/using-new-relic/data/understand-data/new-relic-data-types) (이벤트, 메트릭, 로그, 추적)가 New Relic에 보고되고 쿼리에 사용할 수 있게 되면 해당 데이터를 편집하거나 삭제할 수 없습니다. 이것은 New Relic의 속도와 성능을 최적화하는 의도적인 설계 결정입니다. [데이터 보유 기간](#adjust-retention) 이 종료되면 데이터가 만료됩니다. 이미 보고된 데이터를 삭제해야 하는 경우 New Relic 담당자에게 문의하세요.

## API를 통한 보존 관리 [#api-examples]

UI에서 보존 설정을 변경하는 것 외에도 [NerdGraph API](/docs/apis/nerdgraph/get-started/introduction-new-relic-nerdgraph) 를 사용할 수 있습니다. 몇 가지 예제 NerdGraph 요청이 아래에 포함되어 있습니다. 데이터 보존 변경 사항은 업데이트 후 24시간 이내에 적용됩니다.

<CollapserGroup>
  <Collapser
    id="api-list-namespaces"
    title="사용자 지정 가능한 보존 이벤트 네임스페이스 나열"
  >
    계정에 대한 사용자 정의 가능한 보존 이벤트 이름 공간을 나열하려면:

    ```
    query {
     actor {
       account(id: <var>YOUR_ACCOUNT_ID</var>) {
         dataManagement {
           customizableRetention {
             eventNamespaces {
               namespace
             }
           }
         }
       }
     }
    }
    ```
  </Collapser>

  <Collapser
    id="api-list-rules"
    title="계정의 활성 규칙 나열"
  >
    계정의 활성 규칙을 나열하려면 다음을 수행합니다.

    ```
    query {
     actor {
       account(id: <var>YOUR_ACCOUNT_ID</var>) {
         dataManagement {
           eventRetentionRules {
             id
             deletedAt
             deletedById
             createdAt
             createdById
             retentionInDays
             namespace
           }
         }
       }
     }
    }
    ```
  </Collapser>

  <Collapser
    id="api-show-active-rule"
    title="계정/네임스페이스 쌍에 대한 활성 규칙 표시"
  >
    특정 계정/네임스페이스 쌍에 대한 활성 규칙을 표시하려면:

    ```
    query {
     actor {
       account(id: <var>YOUR_ACCOUNT_ID</var>) {
         dataManagement {
           eventRetentionRule(namespace: "APM") {
             id
             deletedById
             deletedAt
             createdById
             createdAt
             retentionInDays
             namespace
           }
         }
       }
     }
    }
    ```
  </Collapser>

  <Collapser
    id="api-create-new-rule"
    title="새 규칙 만들기"
  >
    새 규칙을 생성하려면:

    ```
    mutation {
     dataManagementCreateEventRetentionRule(accountId: <var>YOUR_ACCOUNT_ID</var>, namespace: "APM", retentionInDays: 8) {
       id
       deletedById
       deletedAt
       createdById
       createdAt
       retentionInDays
       namespace
     }
    }
    ```
  </Collapser>

  <Collapser
    id="api-delete-rule"
    title="기존 규칙 삭제"
  >
    하위 계정에 적용된 규칙을 삭제하면 해당 하위 계정은 지정된 네임스페이스에 대한 상위 계정의 현재 보존 값을 적용합니다. 상위 계정에 적용된 규칙을 삭제하면 지정된 네임스페이스에 기본 보존 값이 다시 적용됩니다. 기존 규칙을 삭제하려면:

    ```
    mutation {
     dataManagementDeleteEventRetentionRule(accountId: <var>YOUR_ACCOUNT_ID</var>, namespace: "APM") {
       id
       deletedById
       deletedAt
       createdById
       createdAt
       retentionInDays
       namespace
     }
    }
    ```
  </Collapser>

  <Collapser
    id="api-delete-rule"
    title="기존 하위 계정 규칙을 다른 사용자에게 복사"
  >
    하위 계정을 템플릿 또는 원본 계정으로 지정하고 다른 대상 하위 계정이 원본 하위 계정과 동일한 네임스페이스 보존 값을 채택하도록 할 수 있습니다. 참고: 모든 하위 계정은 동일한 상위 계정에 있어야 합니다. 기존 보존 세트를 복사하려면 다음을 수행합니다.

    ```
    mutation {
      dataManagementCopyRetentions(
        sourceAccountId: <var>SOURCE_ACCOUNT_ID</var>,
        destinationAccountIds: [<var>DESTINATION_ACCOUNT_ID_1</var>, <var>DESTINATION_ACCOUNT_ID_2</var>, <var>DESTINATION_ACCOUNT_ID_3</var>])
      {
        success
        failure
      }
    }
    ```

    성공 및 실패는 원본 계정에서 규칙을 복사하는 데 성공한(또는 실패한) 대상 계정 ID 목록을 반환합니다.
  </Collapser>
</CollapserGroup>

## 계정 상속 [#account-structure]

상위 계정의 데이터 보존 설정은[하위 계정](/docs/accounts/original-accounts-billing/original-users-roles/mastersub-account-structure) 에 상속됩니다. 이 컨텍스트에서 조직의 첫 번째 계정은 상위 계정으로 처리되고 조직에 추가된 다른 계정은 해당 계정의 설정을 상속합니다. 이러한 설정을 무시하고 계정별 데이터 보존 설정을 지정할 수도 있습니다.
