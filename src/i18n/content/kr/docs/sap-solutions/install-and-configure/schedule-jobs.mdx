## RFC 검사 프로그램 일정 [#rfc-check]

`/NEWR/REGISTRY_RFC_CHECK` 1～3분마다 예약하여 <DNT>**data collection job**</DNT>보다 더 자주 실행되도록 합니다. 작업 이름 `NR_RFC_CHECK`을 사용하세요.

## ABAP 데이터 수집 관리자 일정 [#abap-collection-manager]

다음 데이터 공급자 그룹에 초점을 맞춘 변형을 사용하여 세 개의 데이터 수집 작업을 예약하십시오. `SP03` 이전에 컬렉션 관리자 작업을 예약하는 방법에 대한 지침은 [SP03 이전의 변형 및 작업](/docs/sap-solutions/reference/variants-and-jobs-before-sp03)을 참조하십시오. 스케줄링을 위해 데이터 공급자를 세 개의 그룹으로 나눕니다. 모든 데이터 제공업체의 전체 목록은 [데이터 제공업체 참조](/docs/sap-solutions/monitor/data-providers-reference)를 참조하십시오.

* <DNT>**TRACE group**</DNT> 성능 데이터베이스(STAD) 관련 데이터 제공자가 포함됩니다:

  * [`NR.SAP.TRANSACTION`](/docs/sap-solutions/monitor/data-providers-reference/#sap-appl-metrics): 트레이스 데이터 제공자
  * [`NR.SAP.TRANSACTION.COMPLETE`](/docs/sap-solutions/monitor/data-providers-reference/#sap-appl-metrics): 병렬 백그라운드 RFC 프로세스를 통한 전체 성능 데이터베이스 수집
  * `NR.SAP.SYSTEM.PROCESS` 및 `NR_SAP_PROGRAM`: SP02 이후 일시 중단됨
  * `NR_SAP_WORKLOAD_COMPLETE` 그리고 `NR.SAP.WORKLOAD.COMPLETE`

* <DNT>**IDOC group**</DNT> IDOC 관련 공급자 포함(프로덕션에서는 상당한 시간이 걸릴 수 있음):

  * [`NR_SAP_IDOC`](/docs/sap-solutions/monitor/data-providers-reference/#idoc-all): 메인 IDOC 상태 이벤트 데이터 제공자
  * [`NR_SAP_IDOC_COMPLETE`](/docs/sap-solutions/monitor/data-providers-reference/#idoc-all): 백그라운드 RFC 수집을 통한 전체 IDOC 검색
  * `NR_SAP_IDOC_Configuration`: 성능 향상을 위해 IDOC 마스터 데이터 다운로드

* <DNT>**SYSTEM group**</DNT> 나머지 모든 데이터 공급자가 포함됩니다. [모니터링 그룹별 전체 목록](/docs/sap-solutions/monitor/data-providers-reference)을 참조하십시오.

#### 각 작업 일정 예약

1. 프로세서 `SM36` 을(를) 시작하세요.
2. 각 작업에 대해 <DNT>**Job Name**</DNT> 을 입력하고 <DNT>**Job Class**</DNT> 을 <DNT>**C**</DNT>로 설정합니다.
3. <DNT>**Step**</DNT>를 클릭하고 <DNT>**Program Name**</DNT> `/NEWR/ONE_COLLECTION`를 입력한 다음 <DNT>**Variant**</DNT>를 선택합니다.
4. <DNT>**Start Condition**</DNT>를 클릭하고, <DNT>**Start Date**</DNT> 및 <DNT>**Time**</DNT>를 설정한 다음, <DNT>**Periodic Job**</DNT> 체크박스를 선택합니다.
5. <DNT>**Period Values**</DNT>으로 이동하여 <DNT>**Other Period**</DNT>을(를) 클릭하고 분에 <DNT>**5**</DNT> 을(를) 입력합니다.

## 일정 재제출 프로그램 [#resubmit-program]

1. 전송 오류 발생 시 <DNT>**New Relic account**</DNT> 의 실패한 전송을 재제출하도록 프로그램 `/NEWR/RESEND_FAILED_API_TRANS` 에 대해 <DNT>**DEFAULT**</DNT> (이)라는 이름의 <DNT>**variant**</DNT> 을(를) 생성합니다.

2. 다음 값을 지정하세요.

   * [<DNT>**New Relic Account ID**</DNT>](/docs/accounts/accounts-billing/account-setup/account-id)
   * <DNT>**HTTP Response Codes**</DNT>: `000` 및 `413`, 그리고 `200` ～ `299` 범위를 제외합니다.

   <img title="Resend failed API transactions" alt="A screenshot displaying the resend failed API transactions" src="/images/resend-failed-api-transaction.webp" width="700" />

3. 이 작업을 15분마다 실행되도록 예약하여 <DNT>**HTTP response code**</DNT> `500`이(가) 있는 전송과 같은 실패한 전송을 다시 제출하고 주기적으로 <DNT>**transmission log entries**</DNT>을(를) 삭제합니다.
