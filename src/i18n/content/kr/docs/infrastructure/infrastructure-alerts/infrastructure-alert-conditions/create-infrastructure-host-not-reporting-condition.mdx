<Callout variant="tip">
  NRQL 조건 안내 모드는 [인프라 &quot;호스트 비보고&quot;(HNR) NRQL 조건을 생성하기](/docs/alerts/create-alert/create-alert-condition/create-nrql-host-not-reporting-conditions/) 위한 큐레이션된 경험을 제공합니다. 이는 인프라 &amp;quot;호스트 보고 없음&amp;quot; 조건을 생성하는 것보다 선호되는 대안입니다.
</Callout>

인프라 에이전트로부터 데이터 수신이 중단되면 알림을 받으려면 인프라 모니터링의 <DNT>**Host not reporting**</DNT> [조건을](/docs/infrastructure/new-relic-infrastructure/configuration/infrastructure-alerts-add-edit-or-view-host-alert-information) 사용하세요. 이를 통해 호스트 그룹에 대해 동적으로 공지하고, 기능 기능을 5-60분으로 구성하고, <InlinePopover type="alerts" />공지를 최대한 활용할 수 있습니다.

## 특징 [#features]

가장 중요한 호스트 집합을 기반으로 조건을 정의하고, 필터링된 각 호스트 집합에 적합한 청년 값을 구성할 수 있습니다. 인프라 에이전트의 데이터가 지정한 시간 내에 [수집기에](/docs/accounts-partnerships/education/getting-started-new-relic/glossary#collector) 도달하지 않으면 <DNT>**Host not reporting**</DNT> 이벤트가 트리거됩니다.

<Callout variant="caution">
  태그 또는 레이블을 사용하여 `Host Not Reporting` 조건을 필터링한 다음 대상 호스트에서 중요한 태그 또는 레이블을 제거한 경우 시스템은 해당 호스트가 연결이 끊어진 것으로 특성화하므로 호스트 보고 안 함 사건을 엽니다.
</Callout>

이 기능의 유연성을 통해 모니터링 대상과 선택한 개인 또는 팀에 통지할 시기를 쉽게 사용자 지정할 수 있습니다. 또한 이메일 알림에는 상황을 신속하게 해결하는 데 도움이 되는 링크가 포함되어 있습니다.

## &quot;호스트가 보고하지 않음&quot; 조건 생성 [#create-condition]

<DNT>**Host not reporting**</DNT> 조건 기준을 정의하려면 다음을 수행하십시오.

1. 표준 절차에 따라 [기반 시설 조건을 만드](/docs/infrastructure/new-relic-infrastructure/configuration/infrastructure-alerts-add-edit-or-view-host-alert-information) 십시오.

2. <DNT>
     **Alert type**
   </DNT>

3. 공지를 트리거하기 위한 <DNT>**Critical**</DNT> 레버 값을 정의합니다: 최소 5분, 최대 60분.

   <DNT>
     **Host not reporting**
   </DNT>

   <Callout variant="tip">
     또한, 위에서 언급한 옵션을 확인하는 것과 함께 호스트에 `hostStatus: shutdown` [태그를](/docs/new-relic-one/use-new-relic-one/core-concepts/use-tags-help-organize-find-your-data/) 추가할 수 있습니다. 이렇게 하면 태그가 있는 한 에이전트 버전이나 OS에 관계 없이 해당 호스트에 대한 모든 <DNT>**Host not reporting**</DNT> 인시던트가 열리지 않습니다. 태그를 제거하면 시스템이 해당 호스트에 대한 <DNT>**Host not reporting**</DNT> 인시던트를 다시 열 수 있습니다.
   </Callout>

   <Callout variant="tip">
     `hostStatus: shutdown` [태그](/docs/new-relic-one/use-new-relic-one/core-concepts/use-tags-help-organize-find-your-data/) 대신 호스트 엔터티에 `termination: expected` 태그를 추가할 수 있습니다. 이는 New Relic에게 호스트가 종료될 것으로 예상되었으며 **Don&apos;t trigger alerts for hosts that perform a clean shutdown** \[완전한 종료를 수행하는 호스트에 대해 경고를 트리거하지 않음]을 수행하는 한 모든 **Host not reporting** \[호스트가 보고하지 않음] 사고가 열리는 것을 중지함을 알려줍니다. Strong&amp;gt; 옵션이 조건에서 선택되었습니다.
   </Callout>

   <DNT>
     **Critical**
   </DNT>

   레버 값을 정의합니다: 최소 5분, 최대 60분.

4. 배열을 통해 호스트가 종료되도록 설정된 경우 잘못된 알림을 방지하려면

   <DNT>
     **Don't trigger alerts for hosts that perform a clean shutdown**
   </DNT>

   옵션을 활성화하세요. 현재 이 기능은 systemd를 사용하는 모든 Windows 시스템 및 Linux 시스템에서 지원됩니다.

   <Callout variant="tip">
     또한 위에 언급된 옵션을 확인하면서 호스트에 `hostStatus: shutdown` [태그를](/docs/new-relic-one/use-new-relic-one/core-concepts/use-tags-help-organize-find-your-data/) 추가할 수 있습니다. 이렇게 하면 에이전트 버전이나 OS에 관계없이 해당 태그가 있는 한 해당 호스트에 대해 모든 <DNT>**Host not reporting**</DNT> 인시던트가 열리지 않습니다. 태그를 제거하면 시스템이 해당 호스트에 대해 <DNT>**Host not reporting**</DNT> 인시던트를 다시 열 수 있습니다.
   </Callout>

정책의 [인시던트 기본 설정](/docs/alerts/new-relic-alerts/configuring-alert-policies/specify-when-new-relic-creates-incidents) 에 따라 조건에 대해 정의된 <DNT>**Critical**</DNT> 주니어 값, 참조가 통과될 때 사용할 공지 채널을 정의합니다. '오탐'을 방지하려면 호스트는 인시던트가 개설되기 전 전체 기간 동안 신고를 중단해야 합니다.

<DNT>**Example:**</DNT> 필터링된 호스트 집합 중 하나라도 <DNT>**seven**</DNT> 분 동안 데이터 보고를 중지하면 인시던트를 개시하는 조건을 생성합니다.

* 호스트가 5분 동안 보고를 중지한 후 보고를 재개하면 조건

  <DNT>
    **does not**
  </DNT>

  이 인시던트를 엽니다.

* 호스트가 7분 동안 보고를 중지하면 다른 호스트는 괜찮더라도 조건

  <DNT>
    **does**
  </DNT>

  이 인시던트를 시작합니다.

## 문제 조사 [#view-incidents]

호스트가 데이터를 보고하지 않는 이유를 자세히 조사하려면 다음을 수행하십시오.

1. 이메일 알림의 세부정보를 검토하세요.

2. 전자 메일 공지의 링크를 사용하여 어셈블리 의

   <DNT>
     [**Events**](/docs/infrastructure/new-relic-infrastructure/infrastructure-ui-pages/infrastructure-events-page-live-feed-every-config-change)
   </DNT>

   [페이지 에서 환경의 진행 중인 변경](/docs/infrastructure/new-relic-infrastructure/infrastructure-ui-pages/infrastructure-events-page-live-feed-every-config-change) UI 사항을 모니터링할 수 있습니다. 예를 들어,

   <DNT>
     **Events**
   </DNT>

   페이지를 사용하면 루트 사용자가 호스트에 대한 설정을 변경한 직후 호스트 연결이 끊어졌는지 확인하는 데 도움이 됩니다.

3. 선택 사항: 이메일 공지의

   <DNT>
     [**Acknowledge**](/docs/alerts/new-relic-alerts/reviewing-alert-incidents/acknowledge-alert-incidents)
   </DNT>

   [링크를](/docs/alerts/new-relic-alerts/reviewing-alert-incidents/acknowledge-alert-incidents) 사용하여 알림 인시던트를 인지하고 있으며 이에 대한 소유권을 갖고 있는지 확인하세요.

4. 이메일 링크를 사용하여

   <DNT>
     [**Incident details**](/docs/alerts-applied-intelligence/new-relic-alerts/alert-incidents/view-event-details-incidents)
   </DNT>

   [페이지](/docs/alerts-applied-intelligence/new-relic-alerts/alert-incidents/view-event-details-incidents) 의 추가 세부정보를 검토하세요.

## 의도적 중단 [#intentional]

<DNT>**Don&apos;t trigger alerts for hosts that perform a clean shutdown**</DNT> 옵션을 사용하면 예상치 못한 상황과 계획된 상황을 구분할 수 있습니다. 다음과 같은 상황에서는 이 옵션을 사용하십시오.

* 호스트가 의도적으로 오프라인 상태가 되었습니다.
* 호스트가 유지 관리를 위해 계획된 가동 중지 시간이 있습니다.
* 호스트가 종료되었거나 폐기되었습니다.
* 호스트를 자동 확장하거나 클라우드 콘솔에서 인스턴스를 종료합니다.

우리는 완전한 종료를 표시하기 위해 Linux 및 Windows 종료 신호에 의존합니다.

에이전트가 다음 시나리오를 감지했음을 확인했습니다.

* systemd를 사용하는 EC2 인스턴스의 AWS Auto-scaling 이벤트(Amazon Linux, CentOs/RedHat 7 이상, Ubuntu 16 이상, Suse 12 이상, Debian 9 이상)
* Windows 시스템의 사용자 시작 종료
* systemd를 사용하는 Linux 시스템의 사용자 시작 종료(Amazon Linux, CentOs/RedHat 7 이상, Ubuntu 16 이상, Suse 12 이상, Debian 9 이상)

에이전트가 다음 시나리오를 감지 **하지 못한다** 는 것을 알고 있습니다.

* systemd를 사용하지 않는 Linux 시스템의 사용자 시작 종료(CentOs/RedHat 6 이하, Ubuntu 14, Debian 8). 여기에는 여전히 Upstart 또는 SysV init 시스템을 사용하는 다른 최신 Linux 시스템이 포함됩니다.
* systemd를 사용하지 않는 EC2 인스턴스의 AWS Auto-scaling 이벤트(CentOs/RedHat 6 이하, Ubuntu 14, Debian 8). 여기에는 여전히 Upstart 또는 SysV init 시스템을 사용하는 다른 최신 Linux 시스템이 포함됩니다.
