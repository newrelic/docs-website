인프라 에이전트는 호스트 성능에 미치는 영향을 최소화하도록 설계된 경량 소프트웨어입니다. 그러나 정확한 로드는 호스트의 워크로드, 특히 호스트에서 실행 중인 프로세스 수에 따라 다릅니다. 에이전트가 각 개별 프로세스에서 자세한 [데이터](/docs/data-analysis/metrics/analyze-your-metrics/data-collection-metric-timeslice-event-data#event-data) 를 수집하기 때문입니다.

일반적인 지침으로 New Relic은 몇 가지 일반적인 호스트 유형에 대한 벤치마크를 수집했습니다.

<CollapserGroup>
  <Collapser
    id="single-task"
    title="Linux 단일 작업 호스트"
  >
    에이전트는 클래식 단일 작업 호스트에서 성능 오버헤드가 매우 낮습니다. 예를 들어 Apache, Unicorn 또는 단일 Java 응용 프로그램을 실행하는 서버입니다.

    이 유형의 호스트에 대한 벤치마크는 [Amazon EC2 t3.2xlarge](https://aws.amazon.com/ec2/instance-types/) 를 기반으로 합니다.

    * <DNT>**vCPUs**</DNT>: vCPU 8개

    * <DNT>**Memory**</DNT>: 32.0GB

    * <DNT>**Storage**</DNT>: 160.0GB

    * <DNT>**Operating system**</DNT>: 센트OS 7

      이러한 유형의 클래식 단일 작업 호스트의 경우 일반적인 사용법은 다음과 같습니다.

    * <DNT>**CPU**</DNT>: 약 0.3%

    * <DNT>**Virtual memory**</DNT>: 약 1GB

    * <DNT>**Resident memory**</DNT>: 25-35MB

    * <DNT>**Storage on disk**</DNT>: 약 50MB
  </Collapser>

  <Collapser
    id="docker-host"
    title="리눅스 도커 호스트"
  >
    에이전트는 Docker를 실행하는 호스트에서 성능 오버헤드가 매우 낮으며, 머신이 호스트하는 Docker 컨테이너의 수와 해당 프로세스의 수명이 긴지 또는 짧은지에 따라 정확한 사용량이 결정됩니다.

    이 유형의 호스트에 대한 벤치마크는 [Amazon EC2 t3.2xlarge](https://aws.amazon.com/ec2/instance-types/) 를 기반으로 합니다.

    * <DNT>**vCPUs**</DNT>: vCPU 8개

    * <DNT>**Memory**</DNT>: 32.0GB

    * <DNT>**Storage**</DNT>: 160.0GB

    * <DNT>**Number of containers**</DNT>: 25개의 컨테이너, 컨테이너에서 실행되는 약 100개의 장기 프로세스

    * <DNT>**Operating system**</DNT>: 센트OS 7

      이 유형의 Docker 호스트의 경우 일반적인 사용법은 다음과 같습니다.

    * <DNT>**CPU**</DNT>: 약 0.8%

    * <DNT>**Virtual memory**</DNT>: 약 1GB

    * <DNT>**Resident memory**</DNT>: 25-35MB

    * <DNT>**Storage on disk**</DNT>: 약 50MB
  </Collapser>

  <Collapser
    id="windows-host"
    title="Windows 호스트"
  >
    에이전트는 웹 앱을 제공하고 Windows/IIS 스택을 실행하는 일반적인 Windows 호스트에서 성능 오버헤드가 매우 낮습니다.

    이 유형의 호스트에 대한 벤치마크는 [Amazon EC2 t2.small](https://aws.amazon.com/ec2/instance-types/) 을 기반으로 합니다.

    * <DNT>**vCPUs**</DNT>: 1

    * <DNT>**Memory**</DNT>: 2.0GB

    * <DNT>**Storage**</DNT>: 30.0GB

    * <DNT>**Operating system**</DNT>: 윈도우 서버 2012 R2

      이 유형의 Windows 호스트의 경우 일반적인 사용법은 다음과 같습니다.

    * <DNT>**CPU**</DNT>: 2-3%

    * <DNT>**Resident Memory**</DNT>: 30MB

    * <DNT>**Storage on disk**</DNT>: 약 50MB
  </Collapser>

  <Collapser
    id="arm64"
    title="리눅스 ARM64 호스트"
  >
    에이전트는 AMD64 시스템과 비교할 때 EC2의 ARM64(Graviton 2) 호스트에서 유사한 성능 오버헤드를 갖습니다.

    벤치마크는 [Amazon EC2 t3.2xlarge 대 t4g.2xlarge](https://aws.amazon.com/ec2/instance-types/) 인스턴스를 기반으로 합니다.

    인프라 에이전트 기본 설정이 있는 Amazon Linux 2 EC2 인스턴스:

    * <DNT>
        **CPU**
      </DNT>

      : ARM에서 약 0.1% vs AMD에서 약 0.13%

    * <DNT>
        **Virtual memory**
      </DNT>

      : 약 0.75GB ARM 대 1GB AMD

    * <DNT>
        **Resident memory**
      </DNT>

      : 20MB ARM 대 22MB AMD
  </Collapser>
</CollapserGroup>

우리는 항상 인프라 에이전트의 성능을 개선하고 있습니다. 에이전트 성능 오버헤드가 비정상적으로 높은 경우 [support.newrelic.com](https://support.newrelic.com) 에서 지원을 받으십시오.

## 데이터 관리 [#manage-data]

인프라 모니터링이 수집하고 보고하는 데이터의 양을 조정하는 방법을 알아보려면 [인프라 데이터 관리](/docs/infrastructure/manage-your-data/data-instrumentation/manage-infrastructure-data-reporting) 를 참조하십시오.

## 자원 활용 [#resource-utilization]

Linux 시스템에서 인프라는 지원되는 각 서비스 관리자에 대한 기본 설정으로 설치됩니다. 1GB의 메모리 제한이 적용됩니다. 시스템 요구 사항에 따라 기본 구성을 검토하고 조정하는 것을 고려하십시오.
