당사의 인프라 에이전트는 귀하의 환경에서 실행되는 애플리케이션과 인프라 및 로그 소스를 발견하고 어떤 것을 계측해야 하는지 권장합니다. 설치는 계측하도록 선택한 각 시스템의 구성 및 배포를 자동화합니다.

## 중요한 이유 [#why-it-matters]

가이드 설치를 통해 애플리케이션과 인프라를 계측하고 몇 분 만에 New Relic에서 데이터를 볼 수 있습니다.

가이드 설치는 명령줄 인터페이스(CLI), 호스트 환경을 위한 [인프라 에이전트](/docs/infrastructure/install-infrastructure-agent/get-started/install-infrastructure-agent) , 설치 레시피 라이브러리를 사용하여 애플리케이션과 인프라를 계측합니다. 그것은 당신을 위해 더 적은 수고를 의미합니다.

당사의 계측 레시피는 오픈 소스이므로 필요에 따라 기존 레시피를 수정하거나 새 레시피를 구축할 수 있습니다.

## 일부 기술 세부 사항 [#technical-detail]

New Relic 가이드 설치는 오픈 소스 설치 레시피를 사용하여 호스트 내 통합을 계측합니다. 이러한 레시피에는 설치 및 설정 명령, 로그에 대한 정보, 설치 중인 항목과 관련된 메타데이터가 포함됩니다. 각 유형의 시스템에 대해 YAML 파일에 수집되며 특정 통합을 위해 인프라 에이전트를 설치하는 데 필요한 모든 설치 세부 정보가 있습니다.

<Callout variant="important">
  Windows에서 가이드 설치는 Microsoft SQL Server, 로그 및 인프라 에이전트만 지원합니다.

  다른 모든 통합은 Linux에서만 지원됩니다.
</Callout>

## 호스트 내 통합(OHI) 레시피 [#ohi-recipes]

안내식 설치는 OHI의 검색, 구성 및 설치를 자동화합니다. 그러나 CLI 설치 명령을 사용하여 하나씩 계측하려는 경우가 있을 수 있습니다.

개별 호스트 내 통합을 설치하려면 설치할 통합 유형을 지정하는 다음과 유사한 명령을 사용합니다. 다음은 Linux용 구문입니다.

```
curl -Ls https://download.newrelic.com/install/newrelic-cli/scripts/install.sh | bash && sudo NEW_RELIC_API_KEY=<var>INSERT_YOUR_API_KEY</var> NEW_RELIC_ACCOUNT_ID=<var>INSERT_YOUR_RELIC_ACCOUNT_ID /usr/local/bin/newrelic install -n <var>INSERT_THE_INTEGRATION_NAME</var>
```

예를 들어 Apache 통합을 설치하려면 다음을 수행하십시오.

```
curl -Ls https://download.newrelic.com/install/newrelic-cli/scripts/install.sh | bash && sudo NEW_RELIC_API_KEY=<API_KEY> NEW_RELIC_ACCOUNT_ID=<ACCOUNT_ID> /usr/local/bin/newrelic install -n apache-open-source-integration
```

이 표에는 안내식 설치 CLI 명령에서 지원하는 통합이 나열되어 있습니다. 참조용으로 특정 온-호스트 통합 명령이 제공됩니다.

당사의 오픈 소스 통합은 성능 메트릭 및 인벤토리 데이터를 서버 및 애플리케이션에서 New Relic 플랫폼으로 보냅니다. 메트릭 데이터의 미리 작성된 대시보드를 보고, 경고 정책을 만들고, 사용자 지정 쿼리 및 차트를 만들 수 있습니다.

## 일반적인 문제 해결 [#troubleshoot]

안내식 설치가 실패하는 영역을 식별하면 여기에 문서화하고 몇 가지 문제 해결 지침을 제공합니다.

<CollapserGroup>
  <Collapser
    className="freq-link"
    id="mysql"
    title="MySQL: 잘못된 사용자 권한"
  >
    MySQL 상태 데이터를 모니터링하려면 특정 권한이 있는 유효한 사용자 이름과 암호가 필요합니다.

    다음 명령은 사용자를 생성하고 필요한 권한을 부여합니다.

    특정 비밀번호로 사용자 `newrelic@localhost` 을(를) 만듭니다.

    ```
    sudo mysql -e "CREATE USER 'newrelic'@'localhost' IDENTIFIED BY '<var>YOUR_SELECTED_PASSWORD</var>';"
    ```

    최대 5개의 연결로 `newrelic@localhost` 에 복제 권한을 부여합니다.

    ```
    sudo mysql -e "GRANT REPLICATION CLIENT ON *.* TO 'newrelic'@'localhost' WITH MAX_USER_CONNECTIONS 5;"
    ```

    최대 5개의 연결로 `newrelic@localhost` 에 선택 권한을 부여합니다.

    ```
    sudo mysql -e "GRANT SELECT ON *.* TO 'newrelic'@'localhost' WITH MAX_USER_CONNECTIONS 5;"
    ```

    완료되면 다음 안내 설치 시도가 작동합니다.
  </Collapser>

  <Collapser
    className="freq-link"
    id="nginx"
    title="NGINX: 상태 URL 없음"
  >
    NGINX 서버를 모니터링하려면 유효한 상태 URL을 구성해야 합니다.

    `status_url`: 상태 모듈을 사용하여 메트릭을 제공하도록 설정된 URL입니다. 기본값 `127.0.0.1` 이 잘못된 경우 시스템의 주소/FQDN/URL을 대체하십시오. 예시:

    `status_url: http://127.0.0.1/status`

    다음 NGINX 문서에서 `status_url` 에 대해 자세히 알아볼 수 있습니다.

    * NGINX 오픈 소스의 경우: [HTTP 스텁 상태 모듈](https://nginx.org/en/docs/http/ngx_http_stub_status_module.html)

    * NGINX Plus: [HTTP 상태 모듈](https://nginx.org/en/docs/http/ngx_http_status_module.html) 및 [HTTP API 모듈](http://nginx.org/en/docs/http/ngx_http_api_module.html)

      NGINX가 설치된 방법에 따라 `status_url` 을 설정하는 여러 가지 방법이 있습니다.

    * Kubernetes를 통해 활성화된 경우: [Kubernetes에서 실행 중인 서비스 모니터링을](/docs/monitor-service-running-kubernetes) 참조하세요.

    * Amazon ECS를 통해 활성화된 경우: [ECS에서 실행 중인 서비스 모니터링을](/docs/integrations/host-integrations/host-integrations-list/monitor-services-running-amazon-ecs) 참조하십시오.

    * 호스트에 설치된 경우: 통합의 YAML 구성 파일 `nginx-config.yml` 에서 구성을 편집합니다.
  </Collapser>
</CollapserGroup>
