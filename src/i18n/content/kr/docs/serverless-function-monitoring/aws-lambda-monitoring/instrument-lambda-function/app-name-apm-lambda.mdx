`NEW_RELIC_APM_LAMBDA_MODE` 이(가) `true`일 때, `NEW_RELIC_APP_NAME` 환경 변수는 뉴렐릭에서 Lambda 함수가 APM 엔티티로 표시되는 방식을 제어합니다. 이 페이지에서는 이 변수를 설정하기 전에 이해해야 할 주요 동작을 설명합니다.

## 엔티티 동작

동일한 `NEW_RELIC_APP_NAME` 값과 뉴렐릭 계정을 공유하는 함수는 뉴렐릭 UI에서 단일 APM 엔티티로 그룹화됩니다.

설정한 이름이 기존 APM 엔티티(예: Lambda가 아닌 서비스에서 이미 생성한 엔티티)와 일치하는 경우, Lambda 함수의 텔레메트리가 해당 엔티티에 자동으로 병합됩니다. 별도의 엔티티가 생성되지 않습니다.

<Callout variant="important">
  `NEW_RELIC_APP_NAME` 을(를) 변경하면 새로운 APM 엔티티가 생성됩니다. 이전 이름과 관련된 이력 데이터는 새 엔티티로 마이그레이션되지 않습니다.
</Callout>

## 다중 리전 보고서

서로 다른 AWS 리전에 배포된 여러 Lambda 함수는 동일한 `NEW_RELIC_APP_NAME`을(를) 공유할 수 있습니다. 이 모든 함수는 배포된 리전에 관계없이 단일 APM 엔티티에 보고합니다.

그룹화된 엔티티 내에서 개별 함수 또는 리전별 배포를 구분하려면 NRQL에서 `tags.aws.arn` 속성으로 필터링하거나 엔티티 뷰에서 태그 필터 표시줄을 사용하세요.

## 교차 런타임 보고

동일한 `NEW_RELIC_APP_NAME` 을(를) 공유하는 서로 다른 런타임을 사용하는 Lambda 함수(예: 파이썬 함수와 Node.js 함수)는 **별도의** APM 엔티티를 생성합니다 ― 런타임당 하나씩. 그룹화는 런타임 경계를 넘지 않습니다.

## Lambda 및 비-Lambda 서비스 간 동일한 앱 이름

Lambda 함수와 Lambda가 아닌 APM 서비스(예: 기존 웹 애플리케이션 또는 마이크로서비스)가 모두 동일한 `NEW_RELIC_APP_NAME`(으)로 구성된 경우, **동일한** APM 엔티티에 보고합니다.

이는 의도적일 수 있습니다. Lambda와 서버 기반 인프라 모두에서 실행되는 서비스의 통합된 뷰를 원할 경우, 이름을 공유하면 모든 텔레메트리 ― 트랜잭션, 오류, 처리량 ― 가 한 곳에 표시되도록 할 수 있습니다.

하지만 의도치 않게 수행된 경우 혼란을 초래할 수 있습니다. Lambda 함수와 비 Lambda 서비스의 메트릭, 오류율 및 처리량이 결합되어, 문제를 진단하거나 성능을 독립적으로 이해하기가 더 어려워질 수 있습니다.
