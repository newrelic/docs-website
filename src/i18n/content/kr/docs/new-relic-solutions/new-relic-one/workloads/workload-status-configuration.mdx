  이 페이지는 [표준 워크로드](/docs/new-relic-solutions/new-relic-one/workloads/create-workloads) 와 [지능형 워크로드](/docs/new-relic-solutions/new-relic-one/workloads/workloads-overview#intelligent-workloads) 모두에 적용됩니다. 상태 설정은 두 유형 모두 동일한 방식으로 작동합니다.
</Callout>

다음 세 가지 방법을 사용하여 트래픽 상태를 구성할 수 있습니다.

* <DNT>**Static**</DNT>유지 관리 기간 또는 특정 워크로드 상태를 전달하려는 기타 시나리오에 대해 고정된 상태를 수동으로 설정합니다.
* <DNT>**Automatic**</DNT>상태는 각 엔티티 그룹의 상태와 전체 워크로드를 결정하기 위해 구성한 규칙에 따라 계산됩니다.
* <DNT>**Alert policy and conditions**</DNT>: 공지 상태를 기반으로 워크로드 상태를 판단하기 위한 맞춤형 정책 및 조건을 설정합니다.

워크로드 상태 설정을 관리하려면:

1. 워크로드로 이동하세요
2. 왼쪽 사이드바의 <DNT>**SETTINGS**</DNT> 아래에서 클릭하세요. <DNT>**Configure health status**</DNT>
3. <DNT>**Workload status setup**</DNT> 페이지가 열리고 세 가지 설정 방법 옵션이 표시됩니다.

## 자동 워크로드 상태 구성 [#configure-automatic]

자동 상태 계산은 워크로드에 있는 각 엔티티의 상태를 사용하여 사용자가 정의한 규칙에 따라 단일의 전체 워크로드 상태 값을 결정합니다.

작업 부하의 모든 엔터티가 성능, 오류 또는 가용성 관점에서 똑같이 중요하지는 않습니다.

* 필수 종합 모니터나 서비스에 중요한 인시던트가 진행 중인 경우 워크로드가 중단된 것으로 표시되기를 원할 수 있습니다.
* 그러나 호스트 인프라에 일정 수준의 중복성과 장애 복원력이 있는 경우 단일 호스트에서 공지 인시던트가 발생했다는 이유만으로 쿼리 상태를 <DNT>**Operational**</DNT> 에서 변경할 필요가 없을 수도 있습니다.

### 자동 상태 작동 방식

자동 상태 설정은 다음과 같은 규칙 기반 시스템을 사용합니다.

* 각 규칙은 분리 유형, 태그, GUID 또는 조합을 기준으로 그룹을 평가합니다.
* 각 규칙은 해당 그룹 내 엔티티 상태를 기반으로 상태를 생성합니다.
* 해당 상태의 결과는 모든 규칙 결과 중 최악의 상태와 같습니다.

기본적으로 워크로드를 생성할 때 워크로드 상태를 계산하기 위한 다음 규칙이 추가됩니다.

1. 디지털 경험에 가까운 엔터티 유형(즉, 합성 모니터, 브라우저 응용 프로그램, 모바일 응용 프로그램 및 서비스)의 경우 사용 가능한 최악의 상태가 전파됩니다.
2. 기본적으로 인프라 엔터티인 다른 엔터티 유형의 경우 사용 가능한 최상의 상태가 전파됩니다.

자동 상태를 설정하려면:

1. <DNT>**Workload status setup**</DNT> 페이지에서 <DNT>**Automatic**</DNT> 옵션을 선택하세요.

2. <DNT>**Automatic Status Configuration**</DNT> 섹션이 확장되어 다음과 같이 표시됩니다.

   * 자동 상태 계산을 활성화 또는 비활성화하는 토글 버튼입니다.
   * 그룹 상태 및 전체 상태 결정 규칙
   * 설정에 따라 현재 워크로드 상태를 보여주는 미리보기

3. 토글 스위치가 켜져 있지 않으면 켜주세요.

4. 기존 규칙을 검토하거나 필요에 따라 새로운 규칙을 추가하세요.

### 사용자 지정 규칙 만들기 [#create-custom-tules]

자동 워크로드 상태를 사용자 지정하기 위해 고유한 규칙을 정의할 수 있습니다. 규칙은 엔터티 그룹과 롤업 모드로 구성됩니다.

<img title="new-relic-workloads-status-rule.png" alt="new-relic-workloads-status-rule.png" src="/images/new-relic-workloads-status-rule.webp" />

1. 엔터티 유형, 태그 값, GUID 또는 이들 모두의 조합을 기반으로 엔터티 그룹을 정의합니다.
2. 이러한 엔터티의 상태를 그룹 상태로 전파하는 방법을 결정합니다.

* <DNT>**Roll up the best status**</DNT>: 그룹의 지위는 소속된 모든 구성원의 덜 중요한 지위와 일치합니다. 그룹 내 하나 이상의 엔티티가 여전히 작동 중인 경우 그룹 상태를 &apos;작동 중&apos;으로 유지하려면 이 옵션을 사용하십시오.
* <DNT>**Roll up the worst status**</DNT>: 그룹의 지위는 소속된 모든 구성원 중 가장 중요한 지위와 일치합니다. 그룹 내 요소 중 하나라도 작동하지 않으면 그룹 상태가 즉시 서비스 저하 또는 중단을 나타내도록 하려면 이 옵션을 사용하십시오. 또한 특정 개수의 엔티티가 작동 중지된 후에만 최악의 상태를 집계하도록 설정할 수도 있습니다.

3. 규칙을 저장하고 필요한 경우 계속해서 다른 규칙을 만드십시오.

최종 작업 부하 상태는 모든 개별 그룹 상태 중 최악의 상태와 같습니다.

<Callout variant="important">
  이 최종 계산에서는 워크로드 관리자가 [정적 상태](#configure-static) 를 설정한 경우 자동으로 계산된 상태가 무시됩니다.
</Callout>

### 나머지 엔티티에 대한 규칙 [#remaining-entities]

보다 쉽고 동적인 상태 구성을 위해 이전에 정의한 다른 규칙에서 평가되지 않은 모든 엔터티에 대해 롤업 유형을 사용할 수 있습니다. 특히 다른 규칙을 전혀 추가하지 않으면 나머지 엔터티에 대한 규칙이 워크로드의 모든 엔터티를 고려합니다.

항목 유형별 그룹화 옵션과 결합하면 이 특수 규칙을 사용하면 각 항목 유형에 대한 규칙을 구성하지 않고도 워크로드의 각 항목 유형이 어떻게 작동하는지에 대한 일반적인 이해를 얻을 수 있습니다. 따라서 항상 유형별로 그룹화된 나머지 모든 엔터티에 대한 규칙을 설정하고 워크로드 스택 계층의 모든 엔터티가 작동하지 않을 때 신속하게 감지할 수 있도록 최악의 상태를 롤업하는 것이 좋습니다.

<Callout variant="important">
  6개월 동안 활동이 없으면 상태 계산이 자동으로 중지됩니다. 워크로드에 액세스하고 이를 활성으로 설정하여 다시 활성화할 수 있습니다.
</Callout>

## 공지사항 및 조건 구성 [#alert-policy-conditions]

워크로드 상태를 판단하는 세 번째 방법은 커스텀 공지 및 조건을 설정하는 것이다. 이 방법을 사용하면 공지 상태에 따라 상태 변경을 유발하는 특정 조건을 정의할 수 있습니다.

공지사항 및 조건을 구성하려면:

1. <DNT>**Workload status setup**</DNT> 페이지에서 <DNT>**Alert policy and conditions**</DNT> 옵션을 선택하세요.

2. <DNT>**Configure**</DNT> 클릭하여 <DNT>**Workload health status conditions**</DNT> 페이지를 여세요

3. 이용약관 페이지에서 다음을 수행할 수 있습니다.

   * <DNT>**Search**</DNT> 기존 조건을 이름, 조건 ID 또는 정책 이름으로 검색할 수 있습니다.
   * <DNT>**Filter**</DNT> 조건 이름 및 정책 이름에 대해 드롭다운 메뉴 필터를 사용하는 조건
   * <DNT>**Create new alert policy**</DNT><DNT>**+ New alert policy**</DNT> 클릭하여 새 정책을 정의하세요.
   * <DNT>**Create new alert condition**</DNT>특정 조건을 추가하려면 <DNT>**+ New alert condition**</DNT> 클릭하세요.

이러한 조건들의 상태가 워크로드의 건전성을 결정합니다. 이러한 조건으로 인해 발생하는 모든 문제는 해당 보고서를 건강하지 않은 것으로 만들 것입니다.

<Callout variant="tip">
  워크로드를 건강에 해롭게 만드는 요인을 세밀하게 제어하고 싶을 때 공지 사항 및 조건을 사용하세요. 이 방법은 특정 조건, 임계값 또는 조건 조합이 있을 때 해당 조건의 상태 변화를 유발해야 하는 경우에 특히 유용합니다.
</Callout>

## 정적 워크로드 상태 설정 [#configure-static]

규칙을 기반으로 하는 [자동 계산](#configure-automatic) 에 관계없이 워크로드의 상태를 전달하려는 경우 [사용 가능한 상태 값](/docs/workload-status#obtain-status) 중 하나에서 워크로드에 대한 정적 상태 값을 설정할 수 있습니다.

이는 유지 관리 작업 중에 워크로드 상태가 중단되었음을 다른 팀에 알리거나 추가 정보를 제공하거나 워크로드가 다시 작동할 것으로 예상하는 시간을 제공하는 데 유용합니다.

<Callout variant="tip">
  배포 또는 운영 프로세스로 인해 일시적인 상태를 정기적으로 전달해야 하는 경우 [API](/docs/apis/nerdgraph/tutorials/nerdgraph-workloads-api-tutorials) 를 워크플로에 통합하여 정적 상태 설정을 자동화할 수 있습니다.
</Callout>

정적 작업 부하 상태를 설정하려면 다음을 수행합니다.

1. <DNT>**Workload status setup**</DNT> 페이지에서 <DNT>**Static**</DNT> 옵션을 선택하세요.
2. <DNT>**Configure**</DNT> 클릭하여 <DNT>**Set static status**</DNT> 패널을 엽니다.
3. <DNT>**Select status**</DNT> 드롭다운 메뉴에서 상태 값(예: 작동 중, 성능 저하, 심각, 중단)을 선택하세요.
4. 선택적으로, <DNT>**Summary**</DNT> 필드에 간단한 요약을 입력하여 상태를 한눈에 파악할 수 있습니다.
5. 선택적으로, <DNT>**Description**</DNT> 필드에 워크로드에 어떤 변화가 발생하는지 자세히 설명하는 내용을 입력할 수 있습니다.
6. 정적 상태를 적용하려면 <DNT>**Save**</DNT> 클릭하십시오.
7. 메인 상태 설정 페이지로 돌아가서 <DNT>**Save**</DNT> 클릭하여 구성을 확인하세요.

설정된 정적 상태 값은 [자동으로 계산된](#configure-automatic) 다른 모든 상태 값을 항상 재정의합니다. 정적 상태는 자동 규칙을 무시하고 유지 관리 기간이나 기타 계획된 이벤트 동안 특정 상태를 전달하려는 경우에 특히 유용합니다.

## 상태 값 이해 [#understand-value]

<DNT>**Workload status setup**</DNT> 페이지에서는 상태 계산 구성 방식과 전체 워크로드 상태 값을 결정하는 모든 규칙 및/또는 정적 상태의 결과를 확인할 수 있습니다.

### 미리보기 패널

페이지 오른쪽의 <DNT>**Preview**</DNT> 패널에는 다음 내용이 표시됩니다.

* <DNT>**Unsaved changes indicator**</DNT>설정 변경 사항이 있을 경우 배너가 표시됩니다.
* <DNT>**Entity status**</DNT>워크로드에 있는 엔티티의 현재 상태
* <DNT>**Workload status**</DNT>계산된 전체 상태(예: &quot;운영 중&quot;)를 색상으로 표시함
* <DNT>**Status explanation**</DNT>: 현재 상태를 결정하는 규칙이나 조건에 대한 세부 정보

상태 설정에 대한 변경 사항은 변경 내용을 저장해야만 적용됩니다. 설정을 변경하면 미리 보기가 실시간으로 업데이트되어 해당 시점에 설정을 저장했을 때 어떤 상태가 될지 보여줍니다.

설정을 저장하려면:

1. 상태 계산 결과가 예상과 일치하는지 미리보기를 통해 확인하십시오.
2. 페이지 오른쪽 하단에 있는 <DNT>**Save**</DNT> 클릭하세요.
3. 새로운 설정은 즉시 적용됩니다.

### 상태 계산 순서

여러 설정 방법이 활성화된 경우:

1. <DNT>**Static status**</DNT> 항상 우선권을 가지며 자동 계산이나 공지 기반 계산을 무시합니다.
2. 고정 상태가 설정되지 않은 경우, <DNT>**Automatic rules**</DNT> 또는 <DNT>**Alert policy and conditions**</DNT> 구성에 따라 상태를 결정합니다.
3. 해당 상태는 평가된 모든 규칙 또는 조건 중에서 가장 나쁜 상태와 같습니다.
