## 문제

<InlinePopover type="apm" />, <InlinePopover type="browser" />및 <InlinePopover type="mobile" />기능의 경우 계정 또는 로그를 그룹화하여 더 잘 관리할 수 있는 많은 개별 [그래프 타임슬라이스 데이터 포인트를](/docs/data-analysis/metrics/analyze-your-metrics/data-collection-metric-timeslice-event-data#timeslice-data) 보내는 경우가 있을 수 있습니다. 이 상황을 설명하기 위해 <DNT>**metric grouping issue**</DNT> 또는 <DNT>**MGI**</DNT> 용어를 사용합니다. 이런 일이 발생하면 에이전트는 불필요하게 많은 양의 데이터를 뉴렐릭으로 전송하게 되어 뉴렐릭 차트, 표, 보고서의 효과가 감소합니다.

메트릭 그룹화 문제는 특히 이름이 URL을 기반으로 하는 경우 웹 트랜잭션에서 가장 일반적으로 발생합니다. 애플리케이션에서 보고한 다른 메트릭에서도 발생할 수 있습니다. 예를 들어:

* 애플리케이션이 인터넷을 크롤링하고 각 외부 호출이 다른 도메인으로 이동하는 경우
* 요청을 받을 때마다 소프트웨어가 임시 데이터베이스 테이블을 동적으로 생성하는 경우
* UUID, 기사 이름 또는 유사한 고유 구성 요소를 포함하는 [사용자 정의 계측](/docs/agents/manage-apm-agents/agent-metrics/custom-instrumentation) 을 사용하는 경우

메트릭을 효과적으로 그룹화하는 대신(컨트롤러, 영구 데이터베이스 테이블 또는 특정 외부 서비스와 같이) 잠재적으로 무한한 메트릭 목록을 생성할 수 있는 모든 상황은 메트릭 그룹화 문제가 될 수 있습니다.

## 해결책

메트릭 그룹화가 무엇이며 어떻게 문제가 발생할 수 있는지 이해함으로써 New Relic이 애플리케이션과 어떻게 작동하여 메트릭을 효과적으로 그룹화하고 메트릭 그룹화 문제가 발생하지 않도록 방지하는지 더 잘 이해할 수 있습니다.

앱에서 메트릭 그룹화 문제가 발생하지 않도록 하려면:

1. [New Relic 릴리스 정보를](/docs/release-notes) 확인하여 최신 버전의 New Relic 에이전트를 실행하고 있는지 확인하세요.
2. 필요한 경우 [APM/모바일/브라우저 에이전트를 최신 버전으로 업데이트하세요](/docs/agents/manage-apm-agents/installation/update-new-relic-agent) .
3. 몇 분 기다렸다가 New Relic UI에서 새 데이터를 확인하세요.
4. 이름이 `MetricCardinalityNearLimit` 인 이벤트에 대해 [`NrIntegrationError`](/docs/data-apis/manage-data/nrintegrationerror/) 를) 확인/쿼리합니다. 이러한 이벤트는 앱이 카디널리티 제한에 가까워지면 발생합니다.

문제가 지속되면 에이전트에 대한 절차를 따르십시오.

## 원인 [#cause]

메트릭 그룹화 문제는 메트릭 이름(대부분 웹 트랜잭션 이름)의 세분성이 너무 미세하여 소수의 코드 경로에 대해 수백 또는 수천 개의 서로 다른 웹 트랜잭션 이름이 생성될 때 발생합니다. 몇 가지 주요 코드 경로는 고유한 문서, 기사 또는 페이지 등에 대한 다양한 전체 URL 경로를 생성할 수 있으며 URL 경로의 고유한 요소가 트랜잭션 이름에 포함된 경우 이러한 공통 경로 각각에는 고유한 고유한 경로가 있습니다. 이름.

<CollapserGroup>
  <Collapser id="example" title="MGI 예">
    이 예에는 사용자가 모든 주제에 대해 기사를 작성하고 다른 사용자가 볼 수 있도록 게시할 수 있는 애플리케이션이 있습니다. 애플리케이션에는 기사 추가, 기사 검색, 기사 표시의 세 가지 주요 기능이 있습니다.

    검색 엔진 최적화(SEO)를 개선하기 위해 &quot;기사 보기&quot; 코드는 각 기사를 참조하는 고유한 URL을 생성합니다. 예를 들어 다음 URL은 각각 예제 웹 사이트의 다른 문서를 참조합니다.

    ```
    http://example.com/article/view/How_to_Install_New_Relic
    http://example.com/article/view/How_New_Relic_Saved_the_Day
    http://example.com/article/view/Where_do_I_get_New_Relic
    ```

    세 기사 모두 다릅니다. 다른 콘텐츠를 포함하고 URL이 다릅니다. 그러나 각 기사를 생성하는 코드 경로는 동일합니다. 모두 &quot;기사 보기&quot; 기능을 사용합니다.

    많은 웹 프레임워크가 이 기술을 사용합니다. URL의 일부로 컨트롤러 또는 경로(이 경우 `article/view` )가 있습니다. New Relic은 이러한 패턴을 자동으로 식별하고 유사한 경로를 함께 그룹화하여 메트릭 그룹화 문제가 발생하지 않도록 합니다.

    컨트롤러를 감지하는 메커니즘이 없다면, 예시 애플리케이션은 사이트 방문자가 요청한 각 개별 URL에 대한 메트릭을 전송하게 됩니다. 게시물이 백만 개 있고 사이트가 인기 있다면, 매분 수천 개의 고유 URL이 방문될 수 있습니다. 이로 인해 각 [하베스트 주기](/docs/accounts-partnerships/education/getting-started-new-relic/glossary#harvest-cycle) 에 대해 뉴렐릭으로 전송해야 할 추가 데이터가 상당히 많아지고 [APM <DNT>**Transactions**</DNT> 페이지는](/docs/apm/applications-menu/monitoring/transactions-page) 수천 개의 고유 URL을 나열하려고 시도하여 지표 그룹화 문제가 발생합니다.
  </Collapser>
</CollapserGroup>

애플리케이션 성능을 모니터링하고 개선하려면 각 개별 기사가 얼마나 빨리 표시되는지 아는 것보다 기능(예: 사이트에서 기사 보기)에 대한 평균 성능을 아는 것이 훨씬 더 유용합니다. 지표 그룹화 문제를 방지하기 위해 뉴렐릭은 일반적으로 [APM <DNT>**Transactions**</DNT> 페이지](/docs/apm/applications-menu/monitoring/transactions-page) 에 해당 기능에 대한 단일 항목(예: `/article/view/*`)을 표시합니다.

이 그룹화를 통해 기사를 보는 데 얼마나 많은 시간이 소요되었는지 훨씬 더 잘 알 수 있으며 기사 보기와 관련된 성능 문제를 쉽게 찾을 수 있습니다. 이러한 통계가 수백 또는 수천 개의 트랜잭션에 분산되어 있으면 추세, 회귀 또는 성능 개선을 감지하는 것이 매우 어려울 것입니다.

각 APM 에이전트는 컨트롤러와 프레임워크를 감지하는 방법이 다릅니다. 대부분은 자동으로 적용되지만, 일부는 [구성 파일](/docs/agents/manage-apm-agents/configuration/configure-agent) 에서 옵션을 활성화하거나 비활성화해야 합니다. 또한, 메트릭 그룹화 문제가 발생하지 않도록 방지하기 위해 당사의 권장 사항을 따르세요.
