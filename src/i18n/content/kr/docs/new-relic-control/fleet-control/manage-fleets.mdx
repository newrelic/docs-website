<Callout title="기능 가용성" variant="important">
  <DNT>Fleet Control</DNT> 쿠버네티스 클러스터는 정식 출시(GA)되었습니다. Linux 및 Windows 호스트에서 에이전트를 관리하는 기능은 현재 공개 미리보기 단계에 있습니다.

  지원되는 에이전트와 해당 환경의 전체 목록은 [에이전트 유형 호환성 문서를](/docs/new-relic-control/agent-control/agent-types) 참조하세요.

  공개 미리보기 기능은 [출시 전 정책](/docs/licenses/license-information/referenced-policies/new-relic-pre-release-policy) 에 따라 제공됩니다.
</Callout>

[설정 가이드를](/docs/new-relic-control/fleet-control/setup) 사용하여 첫 번째 함대를 생성한 후에는 대부분의 일상적인 작업을 <DNT>Fleet Control</DNT> UI에서 수행하게 됩니다. 이러한 작업은 API 통해 프로그래밍 방식으로 수행할 수도 있지만, 이 페이지에서는 플릿, 구현, 배포 및 설정을 관리하기 위한 UI 워크플로우에 중점을 둡니다.

## 함대 유형 이해하기

<DNT>Fleet Control</DNT> 세 가지 유형의 플릿을 지원하며, 각 플릿은 특정 관리 대상 엔티티 유형에 맞게 설계되었습니다.

* **쿠버네티스 클러스터 플릿:** 에이전트가 구현되고 배포되는 쿠버네티스 클러스터를 Helm 차트로 관리합니다.
* **Linux 호스트 플릿:** 에이전트가 호스트 기반 프로세스로 실행되는 Linux 서버를 관리합니다. (공개 미리보기)
* **Windows 호스트 플릿:** 에이전트가 Windows 서비스 또는 호스트 기반 프로세스로 실행되는 Windows 서버를 관리합니다. (공개 미리보기)

함대를 생성할 때 관리 대상 엔티티 유형을 선택합니다. 이 선택은 해당 함대와 협력하는 방식에 중요한 영향을 미칩니다.

* **함대 할당:** 함대 유형과 일치하는 관리형 함대만 추가할 수 있습니다. 예를 들어, Windows 호스트 플릿은 Windows 호스트만 포함할 수 있고 쿠버네티스 클러스터 플릿은 쿠버네티스 클러스터만 포함할 수 있습니다.
* **설정 필터링:** 구현, 배포 중에 함대 유형과 일치하는 설정만 표시됩니다. Windows 호스트 환경을 사용하는 경우 설정 선택기에는 Windows 호스트 설정만 표시됩니다. 이 자동 필터링 기능은 잘못된 설정을 방지하고 구현 및 배포 프로세스를 간소화합니다.
* **에이전트 가용성:** 구현, 배포에 사용 가능한 에이전트 목록은 플릿 유형과 해당 환경에 지원되는 항목을 기준으로 필터링됩니다. 예를 들어 특정 에이전트는 Kubernetes 환경이나 특정 호스트 운영 시스템에서만 사용할 수 있습니다.

이 설계는 플릿 내 모든 관리 대상에서 일관성을 보장하고, 실수로 Kubernetes Helm 차트 설정을 Windows 호스트에 구현하거나 배포하려고 시도하는 등의 잘못된 구성을 방지합니다.

## 테이블에서 데이터 정렬

<DNT>Fleet Control</DNT> UI 전체에 있는 많은 데이터 테이블은 정렬 가능합니다. 특정 열을 기준으로 표를 정렬하려면 해당 열 머리글 위로 마우스를 가져가세요. 희미한 회색 위 또는 아래 화살표가 나타나면 헤더를 클릭하여 해당 열의 데이터를 기준으로 테이블 행의 순서를 변경할 수 있습니다.

## 함대 페이지

<DNT>New Relic Control → Fleets</DNT> 으로 이동하면 전체 <DNT>Fleet Control</DNT> 환경에 대한 개요를 제공하는 메인 <DNT>Fleets</DNT> 페이지로 이동합니다. 여기에서 다음을 수행할 수 있습니다.

* **함대 검색:** 검색 창을 사용하여 이름으로 특정 함대를 찾으세요.
* **요약 개수 보기:** 총 차량 수와 할당되지 않은 관리 차량 수를 확인하세요.
* **함대 분포 보기:** 유형(쿠버네티스 클러스터 또는 호스트)별로 함대 분석을 봅니다.

<img title="Main Fleets page" alt="The main Fleets page showing fleet count, unassigned entities count, distribution chart, and a table listing all fleets with their types, entity counts, and status indicators." src="/images/fleet-control-main-fleets-page.webp" />

### 함대 활동

주요 함대 페이지에서 함대의 가장 오른쪽에 있는 줄임표(`...`)를 클릭하면 작업 메뉴가 표시됩니다.

* **설정 편집:** 함대의 이름과 설명을 변경할 수 있습니다.

* **권한 편집:** 연관된 그룹이나 역할을 변경하여 플릿에 대한 세분화된 액세스 제어(FGA)를 관리할 수 있습니다.

* **함대 삭제:** 함대를 영구적으로 제거합니다. 함대를 삭제하려면 다음 조건을 충족해야 합니다.

  * 모든 관리 대상 선박은 함대에서 제거되어야 합니다.
  * 함대에는 현재 진행 중인 구현, 배포가 있을 수 없습니다.

<img title="Fleet actions menu" alt="Fleet actions menu revealed by clicking the ellipsis button in a fleet's row, showing Edit settings, Edit permissions, and Delete a fleet options." src="/images/fleet-control-fleet-actions-menu.webp" />

### 함대 권한(세분화된 액세스)

<DNT>Fleet Control</DNT> 세분화된 액세스 제어(FGA)를 사용하여 관리자가 특정 사용자 또는 그룹에게 특정 함대 관리 권한을 위임할 수 있도록 합니다. 이 기능은 팀에 조직 전체 수준의 광범위한 권한을 부여하지 않고도 자체적으로 차량 관리를 할 수 있도록 지원하려는 경우에 유용합니다.

액세스 권한을 위임하는 주요 방법은 특정 플릿에 대해 미리 정의된 <DNT>**Fleet Manager**</DNT> 역할을 사용자 그룹에 할당하는 것입니다. 이를 통해 해당 그룹의 사용자는 구현 생성, 배포, 설정 관리, 관리되는 구성 추가 또는 제거 기능을 포함하여 해당 플릿에 대한 모든 권한을 갖게 됩니다.

#### 관리자를 위한 전제 조건

함대 권한을 설정하려면 <DNT>Organization Manager</DNT> 및 <DNT>AuthDomain Manager</DNT> 역할이 있어야 합니다.

#### 설정 흐름

이 프로세스에는 두 가지 주요 단계가 포함됩니다. 첫째, 관리자가 사용자 그룹을 만들고 둘째, 해당 그룹에 특정 함대에 대한 <DNT>`Fleet Manager`</DNT> 역할을 할당합니다.

**1단계: 사용자 그룹 만들기**

1. 왼쪽 탐색 메뉴 하단에서 사용자 이름을 클릭한 다음 <DNT>**Administration**</DNT> 선택합니다.
2. <DNT>Administration</DNT> 페이지에서 <DNT>**Access Management**</DNT> 선택하세요.
3. <DNT>groups</DNT> 탭으로 이동하여 **Create a new group** \[새 그룹 만들기]를 클릭합니다.
4. 그룹에 의미 있는 이름을 지정합니다(예: `k8s-platform-team-fleet-managers`).
5. <DNT>members</DNT> 섹션 아래에 차량 관리 권한을 위임할 사용자를 추가하세요.
6. <DNT>**Add user to the group**</DNT> 클릭하세요. 여기서는 계정 또는 조직 수준의 역할을 할당할 필요가 없습니다. 권한은 함대 단위로 부여됩니다.

**2단계: 함대에 그룹 및 역할 지정**

새로운 함대를 만들거나 기존 함대를 편집하여 권한을 할당할 수 있습니다.

**기존 함대에 권한을 할당하려면:**

1. 메인 <DNT>fleets</DNT> 페이지에서 관리하려는 함대를 찾으세요.

2. 행 끝에 있는 줄임표(`...`)를 클릭하고 **Edit permissions** \[권한 편집을] 선택합니다.

3. <DNT>fleet permissions</DNT> 모달에서:

   * <DNT>**Select a group**</DNT>이전 단계에서 생성한 사용자 그룹을 선택하세요.
   * **역할을 선택하세요:** <DNT>`Fleet Manager`</DNT> 역할을 선택하세요.

4. <DNT>**Save**</DNT> 클릭하세요. 선택된 그룹의 사용자들은 이제 해당 차량 관리 시스템에 대한 모든 관리 권한을 갖게 되었습니다.

**함대 생성 중에 권한을 할당하려면:**

1. [설정 가이드에서 함대 생성](/docs/new-relic-control/fleet-control/setup) 단계를 따르세요.
2. **Create Fleet** \[플릿 생성] 을 클릭하기 전에 **Access management** \[액세스 관리] 섹션을 확장하세요.
3. 다운 메뉴를 사용하여 원하는 **Group** \[그]룹 과 **Fleet Manager** \[차량 관리자] 역할을 선택하세요.
4. **Create Fleet** \[차량 만들기]를 클릭합니다.

<img title="Fleet permissions configuration" alt="The Create a fleet modal with the Access management section expanded, showing dropdowns to select an existing group (FleetROGroup) and role (Fleet Manager) for fine-grained access control, with an Add Access Grant button." src="/images/fleet-control-fga-expanded.webp" />

## 특정 함대 관리

함대 이름을 클릭하면 자세한 관리 페이지로 이동합니다.

<Callout title="중요: 함대 설정 규칙" variant="important">
  플릿 관리 시스템은 관리 대상 모든 엔티티에서 일관성을 보장하도록 설계되었습니다. 이를 달성하기 위해서는 세 가지 중요한 규칙이 적용됩니다.

  * **관리형 부분 유형 일관성:** 각 집합에는 집합이 생성될 때 설정되는 특정 관리형 분리 유형(쿠버네티스 분리, Linux 호스트 또는 Windows 호스트)이 있습니다. 이 유형과 일치하는 관리 대상 엔티티만 플릿에 할당할 수 있습니다. 이는 플릿의 모든 부분이 동일한 설정 및 구현, 배포 전략과 호환되도록 보장합니다. 자세한 내용은 [함대 유형 이해를](#understanding-fleet-types) 참조하세요.

  * **에이전트 유형당 하나의 설정:** 단일 함대 내에서 각 에이전트 유형에 대해 하나의 설정만 가질 수 있습니다. 예를 들어, 플릿에 쿠버네티스 클러스터 3개가 있는 경우, 모두 뉴렐릭 인프라 에이전트에 대해 동일한 단일 설정을 사용해야 합니다. 함대 내에서 에이전트 유형에 대한 설정 카디널리티는 항상 1:1입니다.

  * **독점적인 부하 측정, 리소스:** 시스템은 뉴렐릭 인프라 에이전트와 뉴렐릭 NRDOT(Distribution of OpenTelemetry Collector)를 모두 동일한 플릿에 구현하고 배포하는 것을 방지합니다. 이 유효성 검사는 감시를 위해 관리되는 부분을 실수로 이중 계측하는 것을 방지하는 데 도움이 됩니다.
</Callout>

### 함대 헤더: 태그 및 메타데이터

페이지 상단에는 함대 이름과 함께 클릭 가능한 **Tags** \[태그] 및 **Metadata** \[메타데이터] 버튼이 표시됩니다. 이를 통해 함대에 대한 중요한 식별 정보에 신속하게 접근할 수 있습니다.

* **태그:** 이를 클릭하면 함대와 관련된 핵심 가치 쌍 데이터(예: `accountId`, 함대의 `description`, 생성 또는 업데이트 타임스탬프)의 포괄적인 세트가 표시됩니다.
* **여유:** 이를 클릭하면 `Entity guid`, `Account ID`, `Managed entities type` 와 같은 함대의 핵심 식별 세부 정보가 뉴렐릭 참고로서 표시됩니다.

세부 관리 페이지는 크게 **Summary**, **Agents** \[에이전트], **Deployments** \[배포] 3가지 탭으로 구성됩니다.

<img title="Fleet metadata panel" alt="The fleet summary page with the metadata panel opened on the right side, displaying Entity guid, Account ID, Managed entities type, Operating system, and Created by information." src="/images/fleet-control-fleet-metadata-panel.webp" />

### 요약 탭

이 탭은 함대의 현재 상태에 대한 스냅샷을 제공하며, 이는 세 개의 표로 구성되어 있습니다.

* <DNT>**Active deployment table:**</DNT> 가장 최근 구현의 이름, 배포 버전, 포함된 구성/설정 수, 마지막 업데이트 날짜를 표시합니다.
* <DNT>**Agents table:**</DNT> 에이전트 유형, 관련 설정 및 버전, 설정이 생성된 시기를 나열합니다.
* <DNT>**Managed entities table:**</DNT> 플릿에 할당된 모든 영역, 해당 계정, 유형, 현재 측정, 로그 상태를 나열합니다. 또한 뉴렐릭 플랫폼(예: 쿠버네티스 클러스터 탐색기)의 다른 곳에서 분리 메인 페이지로 이동하는 **View entity** \[보기] 링크도 포함되어 있습니다.

<img title="Fleet Summary tab" alt="The Summary tab showing the fleet's active deployment status, Agents section with agent type and configuration details, and Managed entities table listing all clusters with health status indicators and entity selection checkboxes." src="/images/fleet-control-fleet-summary-tab.webp" />

### 에이전트 탭

이 탭은 관리되는 플릿 내에서 실행 중인 모든 에이전트에 대한 자세한 보기를 제공합니다. 이 탭 상단의 검색창을 사용하면 이름이나 유형으로 특정 에이전트를 찾을 수 있습니다.

중요한 세부 사항은 <DNT>New Relic Agent Control</DNT> 감독자도 여기에 에이전트 유형으로 나열되어 있다는 것입니다. 각 에이전트에 대해 연관된 관리되는 부분, 측정, 로그 상태, 원격 설정 상태 및 하트비트가 마지막으로 수신된 시기를 볼 수 있습니다.

#### 효과적인 설정 보기

<DNT>agents</DNT> 탭의 주요 기능은 에이전트의 *효과적인 설정을* 볼 수 있는 기능입니다. 인물의 이름을 클릭하면 상세 정보를 볼 수 있습니다. 여기서는 Helm 차트 버전처럼 데이터 데이터를 볼 수 있지만, 가장 중요한 것은 에이전트에서 실행 중인 정확한 설정을 볼 수 있다는 것입니다.

효과적인 설정은 로컬 환경 변수와 같은 소스의 재정의를 설명하므로 구현하거나 배포하는 설정과 다를 수 있습니다. 이 보기는 문제 해결, 설정 관련 문제 해결에 매우 중요합니다.

<img title="Agent detail view" alt="Detailed agent information showing agent name, managed entity, agent type, version, Helm chart version, instrumentation status, remote configuration status, last seen timestamp, configuration hash, configuration version, and effective configuration link." src="/images/fleet-control-agent-detail.webp" />

### 구현, 배포 탭

이 탭에는 함대의 전체 구현, 배포 내역이 포함되어 있습니다. 검색창을 사용하여 이름으로 특정 구현, 배포판을 찾을 수 있습니다. 표에는 각 구현, 배포의 이름, 상태, 에이전트/구성 수, 마지막 업데이트 날짜가 나열되어 있습니다. 가장 최근에 성공적으로 구현된 배포는 현재 함대에서 시행되고 있음을 나타내기 위해 `Active` 레이블로 표시됩니다.

<img title="Fleet Deployments tab" alt="The Deployments tab showing the active deployment card with deployment details and a deployment history table listing past deployments with dates, names, status badges, durations, and agent counts. Includes Status and Deployed by filter dropdowns and a Create deployment button." src="/images/fleet-control-fleet-deployments-tab.webp" />

구현의 이름을 클릭하면 해당 특정 구현, 배포에 대한 요약 보기로 이동됩니다. 이 화면은 두 개의 탭으로 구성되어 있습니다.

#### 세부 정보 탭

<DNT>details</DNT> 탭은 다음을 포함하여 구현, 배포의 요약을 제공합니다.

* 구현, 배포데이터(이름, 유형, 설명, 시작/종료 시간 및 구현, 문제 해결을 위한 배포 ID, 처리).
* <DNT>**Agents**</DNT> 테이블은 도구 유형, 수행된 작업(예: 설치, 업데이트) 및 설정 버전을 보여줍니다.
* 구현, 배포 및 카나리아로 표시되었는지 여부에 따라 어떤 개체가 영향을 받았는지 정확하게 보여주는 <DNT>**Managed Entities**</DNT> 표입니다.

#### 구현, 배포 이벤트 탭

<DNT>deployment events</DNT> 탭은 구현, 배포 중에 발생한 모든 작업에 대한 세부적인 이벤트 수준 가시성을 제공합니다. 이 탭은 문제 해결, 해결 구현, 배포 문제 및 근본 원인을 이해하는 데 필수적입니다. 다음과 같은 일을 할 수 있습니다:

* 타임스탬프, 상태, 관리되는 부분 이름, 에이전트 이름, 에이전트 유형, 이벤트 유형, 세부 정보, 기간 등의 세부 정보가 포함된 원자 이벤트를 확인하세요.
* 키워드로 특정 이벤트를 검색하세요.
* 특정 클러스터나 호스트에 집중하기 위해 관리되는 부분별로 이벤트를 필터링합니다(문제 해결, 대규모 플릿의 오류 해결 시 유용함).
* 상태별로 이벤트를 필터링합니다(예: 완료됨, 구성 적용됨, 적용 중).
* 이벤트 테이블 헤더 오른쪽에 있는 버튼을 클릭하여 고급 문제 해결을 위한 기본 NRQL 쿼리에 액세스하세요.

문제 해결, 해결을 위한 <DNT>deployment events</DNT> 탭 사용에 대한 자세한 내용은 [문제 해결, 해결 문서를](/docs/new-relic-control/fleet-control/troubleshooting) 참조하세요.

## 구현, 배포 이해

구현, 배포는 플릿의 관리되는 에이전트에 에이전트와 해당 설정을 설치, 삭제 또는 업데이트할 수 있는 제어된 작업 세트입니다. 이 섹션에서는 구현, 배포 수명주기의 주요 측면을 다룹니다. 여기에는 다양한 구현, 배포 상태 이해, <DNT>Agent Control</DNT> 에 대한 자동 업그레이드 프로세스, 초안 및 카나리 구현, 배포와 같은 고급 기능을 사용하여 변경 사항을 안전하게 관리하는 방법이 포함됩니다.

### 구현, 배포 상태 이해

각 구현, 배포의 상태는 결과를 이해하는 데 도움이 됩니다. 가능한 상태와 그 의미는 다음과 같습니다.

| 상태                                 | 설명                                                                                                                                                       |
| :--------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 🔵 <DNT>**IN PROGRESS**</DNT>      | 구현, 배포는 함대의 관리되는 부분에 적극적으로 도입되고 있습니다.                                                                                                                    |
| 🟢 <DNT>**COMPLETED**</DNT>        | 구현, 배포가 성공적으로 완료되었으며, 의도된 모든 작업이 적용되었습니다.                                                                                                                |
| 🔴 <DNT>**FAILED**</DNT>           | 구현, 배포가 완료되었지만 관리되는 부분에서 최소한 하나의 에이전트가 문제를 보고했습니다. 이는 에이전트가 제대로 시작하지 못했거나, 종료하지 못했거나, 새로운 설정을 적용하지 못했음을 의미할 수 있습니다. 구현, 배포는 다른 관리 대상에서 부분적으로 성공할 수 있습니다. |
| 🟠 <DNT>**INTERNAL FAILURE**</DNT> | <DNT>Fleet Control</DNT> 시스템 내의 예기치 않은 문제로 인해 구현 및 배포를 완료할 수 없었습니다. 이는 에이전트나 설정의 오류가 아닙니다. 구현 및 배포를 다시 시도해 보시고, 문제가 지속되면 뉴렐릭 고객 지원팀에 문의하십시오.             |

### 구현, 배포 이벤트 흐름 이해

<DNT>deployment events</DNT> 탭을 보면 구현체의 수명 주기, 배포 등을 나타내는 일련의 이벤트가 표시됩니다. 모든 구현, 배포 이벤트는 `FleetDeployment` 사용자 정의 대시보드 유형에 기록됩니다. 이 이벤트 흐름을 이해하면 <DNT>deployment events</DNT> 탭에서 보이는 내용을 해석하고 문제를 보다 효과적으로 진단할 수 있습니다.

<DNT>Fleet Control</DNT> 구현하다, 배포하다 설정은 위험을 줄이기 위해 &quot;링&quot;이라는 단계에서 변경됩니다. 구현, 배포 프로세스는 다음과 같이 작동합니다.

* **카나리 정보를 지정한 경우** 구현 및 배포는 먼저 카나리 링(사용자가 카나리로 표시한 정보만 포함)을 사용한 다음 나머지 모든 정보에 대한 기본 링을 사용합니다. 각 단계는 다음 단계가 시작되기 전에 성공적으로 완료되어야 합니다. 카나리 링에 문제가 발생하면 기본 링으로 진행하기 전에 구현 및 배포가 중단되어 광범위한 문제를 방지합니다.
* **카나리 대상을 지정하지 않은 경우** 구현 및 배포는 함대의 모든 대상을 포함하는 기본 링만 사용합니다. 구현, 배포는 해당 단일 링의 모든 부분에 걸쳐 완료될 때까지 실행됩니다.

#### 구현, 배포 이벤트 유형

다음 이벤트는 구현, 배포의 진행 상황을 처음부터 끝까지 추적합니다.

| 이벤트         | 설명                                     |
| :---------- | :------------------------------------- |
| 함대 배치 시작됨   | 구현, 배포가 함대에 대해 시작됩니다. 설정 패키지가 준비되었습니다. |
| 링 배포 시작됨    | 구현, 배포 링 시작(예: 카나리 링 또는 기본 링).         |
| 에이전트 배포 시작됨 | 설정 구현, 배포는 관리되는 특정 에이전트에 대해 시작됩니다.     |
| 에이전트 배포 완료  | 대리인이 설정 적용 시도를 마쳤습니다.                  |
| 링 배치 완료     | 링 안의 모든 참가자가 완료되었거나 시간 초과되었습니다.        |
| 함대 배치 완료    | 전반적인 구현은 배포가 최종 상태로 완료됩니다.             |

#### 에이전트 구현, 배포 상태

에이전트가 구현, 배포 시도(AgentDeploymentCompleted)를 완료하면 다음 상태 중 하나를 보고합니다.

| 상태                         | 설명                                 |
| :------------------------- | :--------------------------------- |
| <DNT>CONFIG\_APPLIED</DNT> | 에이전트가 새로운 설정을 성공적으로 적용했습니다.        |
| <DNT>FAILED</DNT>          | 에이전트에서 설정을 적용하는 중에 오류가 발생했습니다.     |
| <DNT>TIMEOUT</DNT>         | 그쪽에서 예상했던 시간 내에 응답하지 않았습니다.        |
| <DNT>NOT\_REPORTING</DNT>  | 구현, 배포가 시작되었을 때 에이전트는 오프라인 상태였습니다. |

#### 이벤트 흐름 작동 방식

구현을 시작하면 배포, 이벤트가 다음 순서로 생성됩니다.

1. **FleetDeploymentStarted** — 전체 플릿에 대한 구현, 배포가 시작됩니다.
2. **RingDeploymentStarted** — 첫 번째 링(일반적으로 카나리 링)이 시작되고 해당 링의 모든 에이전트가 식별됩니다.
3. **AgentDeploymentStarted** — 링의 각 에이전트에 대해 구현, 배포 이벤트가 기록됩니다. 이러한 이벤트는 링의 모든 에이전트에 대해 동시에 발생합니다.
4. **AgentDeploymentCompleted** — 각 에이전트는 상태(<DNT>CONFIG\_APPLIED</DNT>, <DNT>FAILED</DNT>, <DNT>TIMEOUT</DNT>, 또는 <DNT>NOT\_REPORTING</DNT>)를 보고합니다.
5. **RingDeploymentCompleted** — 링에 속한 모든 에이전트가 완료되거나 시간 초과되면 링이 완료로 표시됩니다. 벨소리가 성공하면 다음 벨소리가 시작됩니다. 실패하면 구현, 배포가 중지됩니다.
6. **FleetDeploymentCompleted** — 모든 링이 완료되거나 링이 실패하면 구현, 배포가 <DNT>COMPLETED</DNT>, <DNT>FAILED</DNT> 또는 <DNT>INTERNAL\_FAILURE</DNT> 의 최종 상태로 완료됩니다.

#### 시각적 이벤트 흐름

```
FleetDeploymentStarted
│
├─▶ RingDeploymentStarted (canary)
│   │
│   ├─▶ AgentDeploymentStarted (agent 1) ──▶ AgentDeploymentCompleted ─┐
│   ├─▶ AgentDeploymentStarted (agent 2) ──▶ AgentDeploymentCompleted ─┤ [parallel]
│   ├─▶ AgentDeploymentStarted (agent 3) ──▶ AgentDeploymentCompleted ─┘
│   │
│   └─▶ RingDeploymentCompleted (canary)
│       │
│       ├─ ✓ If SUCCESS → proceed to default ring
│       └─ ✗ If FAILED → stop deployment
│
├─▶ RingDeploymentStarted (default)
│   │
│   ├─▶ AgentDeploymentStarted (agent 4) ──▶ AgentDeploymentCompleted ─┐
│   ├─▶ AgentDeploymentStarted (agent 5) ──▶ AgentDeploymentCompleted ─┤ [parallel]
│   ├─▶ AgentDeploymentStarted (agent 6) ──▶ AgentDeploymentCompleted ─┘
│   │
│   └─▶ RingDeploymentCompleted (default)
│
└─▶ FleetDeploymentCompleted
```

**에이전트 구현, 배포 상태 옵션:**

* ✅ **CONFIG\_APPLIED** — 에이전트가 설정을 성공적으로 적용했습니다.
* ❌ **FAILED** — 에이전트에 오류가 발생했습니다.
* ⏱️ **TIMEOUT** — 에이전트가 제 시간에 응답하지 않았습니다.
* 📡 **NOT\_REPORTING** — 에이전트 구현, 배포 시작 시 오프라인 상태였습니다.

**주요 식별자:** `fleetId`, `deploymentId`, `ringName`

구현, 배포 이벤트 탭을 사용하면 이러한 모든 이벤트를 자세히 볼 수 있습니다. 문제 해결을 위한 구현, 배포 이벤트 액세스 및 사용에 대한 자세한 내용은 [구현, 배포 탭 섹션을](#the-deployments-tab) 참조하세요.

### 자동 에이전트 제어 업그레이드

관리되는 대상이 최신 보안 패치, 성능 향상 및 기능을 활용할 수 있도록 <DNT>Fleet Control</DNT> 대상 제어에 대한 업그레이드 프로세스를 자동으로 관리합니다. 구현을 시작하면 <DNT>Fleet Control</DNT> 플릿의 관리형 부분에 대해 최신 버전의 에이전트 제어를 사용할 수 있는지 확인합니다.

업그레이드가 필요한 경우, 이는 자동으로 귀하의 구현, 배포의 일부로 포함됩니다. 구현, 배포 프로세스 중에 UI 에 에이전트 컨트롤이 업그레이드될 것임을 알리는 공지가 표시됩니다. 이 암시적 업그레이드는 수행 중인 구현, 배포 작업(예: 에이전트 설치, 에이전트 제거 또는 에이전트 설정 변경)과 함께 발생하여 최소한의 노력으로 환경을 최신 상태로 유지합니다.

<Callout title="중요: 업그레이드 실패 및 복구" variant="caution">
  현재로선 에이전트 제어 업그레이드에 실패한 경우 자동 롤백 기능이 제공되지 않습니다. 업그레이드가 실패하면 구현, 배포가 **FAILED** \[실패] 로 표시되고 관리되는 항목에 수동 개입이 필요할 수 있습니다. 자세한 내용은 [에이전트 Control 수동 업그레이드에](/docs/new-relic-control/agent-control/upgrading) 대한 설명서를 참조하세요.

  향후 릴리스에서는 업그레이드 실패 시 자동 롤백 기능을 제공하고 수정 과정을 간소화하는 기능을 적극적으로 개발하고 있습니다.
</Callout>

## 초안 구현 생성 및 관리, 배포

구현, 배포를 초안으로 저장하여 변경 사항을 즉시 구현, 배포하지 않고도 시간이 지남에 따라 작업할 수 있습니다.

### 초안을 만들고 다시 시작합니다.

1. 특정 함대 내의 <DNT>deployments</DNT> 탭으로 이동합니다.
2. <DNT>**Create deployment**</DNT> 을(를) 클릭합니다.
3. 구현 및 배포 창에서 필수 이름으로, 선택 사항으로 설명을 입력합니다.
4. 에이전트를 추가하거나 설정을 업데이트하는 등 원하는 대로 변경하세요.
5. 구현, 배포를 시작하는 대신 <DNT>**Save draft**</DNT> 클릭합니다.
6. 그런 다음 구현, 배포 창을 닫을 수 있습니다. 새로운 구현인 배포는 <DNT>deployments</DNT> 탭에 `Draft` 상태로 표시됩니다.

초안 작업을 계속하려면 구현, 배포 내역 표에서 해당 이름을 클릭하기만 하면 됩니다. 이렇게 하면 구현, 배포 창이 다시 열리고 변경 작업을 완료할 수 있으며 준비가 되면 **Start deployment** \[배포 시작을] 클릭할 수 있습니다.

### 초안 삭제

초안이 더 이상 필요하지 않은 경우 배포합니다.

1. <DNT>deployments</DNT> 탭에서 삭제하려는 초안의 이름을 클릭하세요.
2. 열리는 구현, 배포 창에서 화면 상단의 <DNT>Start deployment</DNT> 옆에 있는 점 세 개(`...`) 버튼을 클릭합니다.
3. 메뉴에서 <DNT>**Delete draft**</DNT> 선택하세요.
4. 초안은 영구적으로 삭제되며, <DNT>deployments</DNT> 탭으로 돌아갑니다.

### 구현, 배포 중 에이전트 버전 선택

구현, 배포에 에이전트를 추가하면 설정을 선택하기 전에 에이전트 버전을 선택하게 됩니다.

1. 에이전트 유형을 선택하면 <DNT>**Select a version**</DNT> 드롭다운 메뉴가 나타납니다.

2. 최신 버전이 기본적으로 선택됩니다.

3. 설치 가능한 다른 버전을 선택할 수 있습니다.

4. 각 버전에 대해 다음과 같은 추가 메타데이터가 표시될 수 있습니다.

   * 이번 릴리스 노트에서는 새로운 기능과 개선 사항을 소개합니다.
   * 보안 업데이트 정보
   * 기타 버전별 세부 정보

이 버전 선택 기능은 지원되는 모든 베어링 유형(쿠버네티스 클러스터, Linux 호스트 및 Windows 호스트)에서 작동하므로 관리되는 에이전트 버전이 구현되거나 배포되는지를 정확하게 제어할 수 있습니다.

에이전트 버전을 선택한 후 <DNT>**Next**</DNT> 클릭하여 설정 선택을 진행하세요.

<img title="Select agent version" alt="The Select an agent modal showing agent type cards and a version dropdown displaying available versions with release notes and security update information." src="/images/fleet-control-select-agent-version.webp" />

### 구현의 에이전트 작업, 배포

구현, 배포를 생성하거나 편집할 때 구현, 배포 창의 **Agents** \[에이전트] 테이블은 각 에이전트에 대한 여러 작업을 제공합니다. 에이전트 행의 오른쪽에 있는 줄임표(`...`)를 클릭하면 다음 옵션에 액세스할 수 있습니다.

* **설정 편집:** 설정 편집기를 열어 구현, 배포의 일부로 이 에이전트에 대한 설정을 수정할 수 있습니다.
* **최신 버전 사용:** 설정을 여러 버전으로 생성한 경우, 이 옵션을 선택하면 구현 및 배포가 자동으로 최신 버전의 설정 설정을 사용하도록 업데이트됩니다.
* **설정 버전 변경:** 구현에 사용할 설정의 특정 이전 버전을 수동으로 선택할 수 있습니다.
* **구현, 배포에서 제거:** 이 구현, 배포에서 에이전트 및 해당 설정을 완전히 제거합니다.

<Callout title="에이전트 버전과 설정 버전" variant="tip">
  <DNT>Fleet Control</DNT> 에는 두 가지 유형의 버전이 있다는 점을 구분하는 것이 중요합니다.

  * **에이전트 버전:** 에이전트를 배치에 추가할 때 초기 에이전트 선택 단계에서 선택된 에이전트 소프트웨어 자체의 버전(예: 에이전트 에이전트 v1.54.0)입니다.
  * **설정 버전:** 위의 작업 메뉴를 통해 관리할 수 있는 해당 개체에 대해 생성한 설정 파일의 버전입니다.

  &quot;최신 버전 사용&quot; 또는 &quot;설정 버전 변경&quot;을 선택하면 설정 버전을 관리하는 것이지, 기기 버전을 관리하는 것이 아닙니다.
</Callout>

<img title="Agent actions in deployment" alt="The deployment pane's Agents table with the ellipsis menu expanded, showing options to Edit configuration, Change configuration version, and Remove from deployment for the New Relic Infrastructure agent." src="/images/fleet-control-deployment-agent-actions.webp" />

## 관리하다 함대에서

<Callout title="중요: 함대 상속의 원칙" variant="important">
  함대는 **상속** 의 기본 원칙에 따라 운영됩니다. 이는 플릿에 할당된 모든 관리 부분이 해당 플릿의 마지막 활성 구현을 자동으로 준수하고 측정, 내용 일관성을 보장하기 위해 배포됨을 의미합니다. 에이전트를 추가하거나 제거할 때 이러한 동작을 이해하는 것이 중요합니다. 에이전트가 자동으로 설치되거나 제거될 수 있기 때문입니다.
</Callout>

### 함대에서 이를 제거하세요

1. 현재 해당 엔티티가 포함된 함대의 <DNT>summary</DNT> 탭으로 이동합니다.
2. <DNT>managed entities</DNT> 표에서 체크박스를 사용하여 제거하려는 항목을 하나 이상 선택하세요.
3. 나타나는 <DNT>Remove</DNT> 클릭하세요.
4. 제거할 항목을 나열하는 확인 모달이 나타납니다. 확인하려면 텍스트 필드에 `remove` 입력하세요.
5. <DNT>Remove entities</DNT> 클릭하세요. 해당 함정은 현재 함대에서 제외되었습니다.

<Callout title="중요: 로컬 설정으로 되돌리기" variant="caution">
  관리되는 블로그가 함대에서 제거되면 에이전트 Control이 처음 설치되었을 때의 상태인 원래 **로컬 설정**으로 자동 복귀됩니다. 이로 인해 에이전트가 제거될 수 있습니다. 예를 들어, 플릿에 베어링 및 Fluent Bit 에이전트가 설치되어 있지만 이동의 로컬 설정에 에이전트 제어만 포함되어 있는 경우 제거 시 로컬 상태와 일치하는지 확인하기 위해 두 에이전트가 모두 제거됩니다.
</Callout>

<img title="Remove entities confirmation" alt="The Remove entities confirmation panel with explanatory text, a text input field where the user must type 'REMOVE' to confirm, a table showing the entity to be removed (demo-cluster-eks-219-3), and the Remove entities button." src="/images/fleet-control-remove-me.webp" />

### 할당되지 않은 관리 항목을 함대에 추가

새로운 구현, 배포를 생성하여 할당되지 않은 관리 부분을 플릿에 추가합니다. 이 프로세스는 Fleet 및 해당 구성, 배포 설정과 연결됩니다.

1. 엔티티를 추가하려는 함대로 이동하여 <DNT>**Create a deployment**</DNT> 클릭합니다.
2. 구현, 배포에 의미 있는 이름과 설명을 지정하세요.
3. 구현, 배포 화면 하단의 <DNT>managed entities</DNT> 표에서 <DNT>**Add managed entities**</DNT> 버튼을 클릭합니다.
4. 할당되지 않은 모든 관리 항목 목록이 표시된 창이 열립니다.
5. 선택 상자를 사용하여 함대에 추가하려는 하나 이상의 항목을 선택하세요.
6. <DNT>Add to deployment</DNT> 클릭하세요. 선택한 부분은 이제 이 구현에 대한 <DNT>managed entities</DNT> 테이블에 표시됩니다.
7. <DNT>Save draft</DNT> 클릭한 다음 <DNT>Start deployment</DNT> 클릭하여 프로세스를 완료하세요. 이제 해당 인원은 새로운 함대에 배정될 것입니다.

<Callout title="중요: 마지막 활성 구현 상속, 배포" variant="important">
  플릿에 관리형 인스턴스를 추가하면 암묵적인 구현이 이루어지고 백그라운드에서 배포가 발생합니다. 이 구현은 배포가 새 부분에 대한 측정, 리소스를 자동으로 설치, 업데이트 또는 제거하여 플릿의 마지막 **활성 구현인 배포와** 완벽하게 일치하는지 확인합니다. 예를 들어, 플릿의 활성 구현, 배포에 하드웨어 및 Fluent Bit 에이전트가 포함되어 있지만 이전에 새 부품에 OTel 에이전트만 있었던 경우 OTel 에이전트는 제거되고 <DNT>Infrastructure</DNT> 및 <DNT>Fluent Bit</DNT> 에이전트가 설치됩니다.
</Callout>

<img title="Add managed entities to deployment" alt="The Add managed entities selection pane showing a searchable table of available entities with checkboxes, allowing users to select which unassigned managed entities to add to the fleet via this deployment." src="/images/fleet-control-add-me-deployment.webp" />

## 카나리아 구현, 배포 사용

<DNT>Fleet Control</DNT> 구현 및 배포 프로세스 중에 하나 이상의 관리 대상을 &quot;카나리&quot;로 지정할 수 있습니다. 이는 변경 사항을 배포하는 데 있어 매우 중요한 안전 장치를 제공합니다.

새로운 구현, 배포를 만들 때 목록에서 관리되는 모든 항목 옆에 줄임표(`...`)를 사용하여 카나리아로 지정할 수 있습니다.

구현이 시작되면 배포가 시작됩니다.

1. 변경 사항은 카나리아로 표시된 관리되는 부분 *에만* 구현, 배포됩니다.
2. 구현, 배포 프로세스는 일시 중지되고 카나리아 구현, 배포가 성공할 때까지 기다립니다.
3. Canary 구현, 배포가 실패하면 전체 구현, 배포가 즉시 중지되고 다른 관리되는 구성은 영향을 받지 않습니다.
4. 모든 Canary 구현, 배포가 성공하면 구현, 배포가 Fleet의 나머지 관리되는 부분으로 진행됩니다.

<Callout title="중요: 실패한 카나리아에 대한 자동 롤백은 없습니다." variant="caution">
  Canary 구현, 배포가 실패하는 경우 관리되는 부분은 구현, 배포는 **Failed** 상태로 유지됩니다. 이전 설정으로 자동 롤백되는 기능은 없습니다. 변경 사항을 되돌리거나 문제를 해결하려면 새로운 구현, 배포를 만들어야 합니다.
</Callout>

<img title="Canary deployment enablement" alt="The deployment pane's Managed entities table with the ellipsis menu expanded, showing the 'Use as a canary' option to designate a managed entity for staged canary deployment." src="/images/fleet-control-canary-action.webp" />

## 설정 페이지

<Callout title="중요: 설정은 전역적입니다." variant="important">
  <DNT>Fleet Control</DNT> 에서 생성하는 모든 설정은 **전역 설정** 입니다. 즉, 해당 설정은 전체 뉴럴릭 조직에서 사용할 수 있으며, 설정을 생성한 사람이 누구인지에 관계없이 필요한 모든 함대에 구현, 배포할 수 있습니다.
</Callout>

<DNT>New Relic Control → Configurations</DNT> 으로 이동하면 모든 설정 항목을 관리하는 중앙 허브인 기본 설정 페이지로 이동합니다.

### 메인 페이지 개요

이 페이지는 귀하의 계정에서 사용 가능한 모든 설정에 대한 간략한 개요를 제공합니다. 여기에서 다음을 수행할 수 있습니다.

* **요약 카운트 보기:** 전체 설정 카운트를 확인하세요.
* **설정 분포 보기:** 에이전트 유형별 설정 분석을 봅니다(예: 뉴렐릭 인프라 , Fluent Bit, NRDOT 수집기).
* **설정 검색:** 검색 표시줄을 사용하여 이름이나 관련 에이전트 유형으로 특정 설정을 찾습니다.
* **목록을 새로 고치세요:** 새 설정을 방금 만들었는데 표에 표시되지 않으면 <DNT>**Create configuration**</DNT> 왼쪽에 있는 새로 고침 버튼을 클릭하세요.

이 페이지에는 다음 열이 포함된 설정 표가 표시됩니다.

* **설정:** 설정의 이름입니다.
* **에이전트:** 설정이 적용되는 에이전트 유형입니다.
* **개정판:** 설정에 존재하는 버전 수.
* **유형:** 관리되는 부분 유형(예: 쿠버네티스 클러스터, Linux Host 또는 Windows Host)입니다.
* **최종 수정:** 마지막 업데이트의 타임스탬프입니다.

<img title="Main Configurations page" alt="The Configurations page showing total configuration count, distribution chart by agent type, and a searchable table listing all configurations with their agent types, revision counts, entity types, and last modified dates." src="/images/fleet-control-configurations-page.webp" />

### 설정 작업

기본 설정 테이블에서 모든 설정의 맨 오른쪽에 있는 줄임표(`...`)를 클릭하면 해당 설정을 **Edit** \[편집] 할 수 있습니다. 이렇게 하면 설정 파일을 수정할 수 있는 기본 편집 창이 열립니다. 이 보기에서는 설정 파일을 로컬 컴퓨터에 **Download** \[다운로드하거나] 파일의 내용을 클립보드에 **Copy** \[복사하는] 버튼도 제공됩니다.

### 설정 이해

설정 파일의 구조는 선택한 **관리형 분리 유형** (쿠버네티스 클러스터, Linux 호스트 또는 Windows 호스트)에 따라 다릅니다.

* **쿠버네티스 클러스터의 경우** UI 에서 생성하는 설정은 `values.yaml` 파일입니다. 구현하는 동안 <DNT>Fleet Control</DNT> 클러스터에서 구성하기 위해 에이전트의 Helm 차트와 함께 이 파일을 패키징합니다. Kubernetes 에이전트용 템플릿은 표준 Helm 차트 `values.yaml` 파일로 구성됩니다. 템플릿 상단 근처에서 [뉴렐릭 Helm Charts GitHub](https://github.com/newrelic/helm-charts) 에 있는 에이전트의 Helm 차트에 대한 패턴화 링크를 찾을 수 있습니다. 이를 통해 스키마, 사용 가능한 값 및 기타 에이전트별 문서를 쉽게 참조할 수 있습니다.
* **호스트의 경우** 설정 파일 형식은 에이전트가 구현하다, 배포하다에 따라 다릅니다.

<Callout title="구현에서 필터링 설정, 배포" variant="tip">
  플릿에 대한 구현 및 배포를 생성할 때 시스템은 사용 가능한 설정을 자동으로 필터링하여 플릿의 관리 대상 유형과 일치하는 설정만 표시합니다. 예를 들어:

  * **쿠버네티스 클러스터 함대는** 쿠버네티스 클러스터에 대해 생성된 설정만 표시합니다.
  * **Linux 호스트 플릿에는** Linux 호스트용으로 생성된 설정만 표시됩니다.
  * **Windows 호스트 플릿에는** Windows 호스트용으로 생성된 설정만 표시됩니다.

  이 자동 필터링은 실수로 관리되는 구성에 대해 호환되지 않는 설정을 구현하고 배포하는 일을 방지하고 일관성을 유지하며 구현, 배포 실패를 방지합니다.
</Callout>

<Callout title="보안 모범 사례: Kubernetes에서 민감한 데이터 관리" variant="important">
  Kubernetes 구현, 배포의 경우 설정 파일에 직접 `licenseKey` 와 같은 민감한 데이터를 포함하지 마세요. 대신, Control의 비밀 관리 기능을 사용하여 런타임에 민감한 값을 안전하게 주입하세요. 자세한 내용은 [비밀 관리 문서를](/docs/new-relic-control/agent-control/configuration#secrets-management) 참조하십시오.
</Callout>

### 구성 만들기

새로운 글로벌 세트를 처음부터 만들려면:

1. 기본 설정 페이지에서 <DNT>**Create configuration**</DNT> 클릭합니다.

2. 나타나는 모달에서 다음을 수행해야 합니다.

   * <DNT>**managed entity type**</DNT> (쿠버네티스 클러스터, Linux 호스트 또는 Windows 호스트)을 선택합니다. 참고로 Linux 호스트 및 Windows 호스트 지원은 현재 공개 미리보기 단계입니다.
   * 드롭다운 메뉴에서 <DNT>**Agent type**</DNT> (예: 뉴렐릭 인프라, NRDOT 수집기, Fluent Bit)을 선택합니다. 사용 가능한 에이전트 유형은 선택한 관리 대상 엔티티 유형에 따라 필터링됩니다.
   * 설정에 의미 있고 고유한 <DNT>**Name**</DNT> 제공하세요.

<img title="Create configuration modal" alt="The Create configuration modal with dropdown fields for selecting Managed entity type, Agent type, and a text field for entering the configuration name." src="/images/fleet-control-create-configuration-modal.webp" />

3. <DNT>**Continue**</DNT> 클릭하세요. 이렇게 하면 두 개의 창으로 나뉜 설정 편집 화면이 열립니다.

   * **왼쪽 창(활성 설정):** 여기에서 설정을 작성합니다. 이 창에 직접 입력하거나 다른 소스에서 콘텐츠를 붙여넣을 수 있습니다.
   * **오른쪽 창(템플릿):** 이 창에는 선택한 에이전트 유형에 대한 최신 기본 템플릿이 포함되어 있습니다. 이러한 템플릿은 에이전트를 실행하는 데 필요한 최소한의 설정을 제공하지만, 추가적으로 많은 주석화 옵션도 포함하고 있습니다. 이러한 예제를 참조로 사용하거나 주석 처리를 제거하여 더욱 사용자 정의된 설정을 구축할 수 있습니다.

4. 일반적인 방법은 템플릿 창에서 <DNT>**Copy**</DNT> 클릭하고 내용을 왼쪽의 활성 설정 창에 붙여넣어 시작점으로 사용하는 것입니다.

5. 왼쪽 창에서 설정을 완료했으면 <DNT>**Save**</DNT> 클릭하세요. 편집 화면이 닫히고 기본 설정 페이지로 돌아가면 새 설정을 확인할 수 있습니다.

### 특정 설정 보기 및 버전

설정에 대한 자세한 내용을 보고 버전을 관리하려면 기본 목록에서 해당 이름을 클릭하세요. 이렇게 하면 해당 설정에 대한 자세한 창이 열립니다.

이 보기에서는 다음 내용을 볼 수 있습니다.

* 설정의 이름, 에이전트 유형 및 사용 가능한 버전 목록입니다.
* **구현 상태는** 설정이 모든 플릿에 구현되었는지, 배포되었는지, 그렇다면 어떤 플릿인지를 나타냅니다.

이 창에서 **Download** \[다운로드], **Copy** \[복사], **Clone** \[복제] 등 여러 작업을 수행할 수도 있습니다. 여기서 가장 중요한 작업은 설정의 새로운 버전을 만드는 것입니다.

새 버전을 만들려면:

1. <DNT>**Create config version**</DNT> 을(를) 클릭합니다.
2. 이렇게 하면 설정을 수정할 수 있는 편집 창이 열립니다.
3. 편집을 마치셨으면 <DNT>**Save**</DNT> 클릭하세요.
4. 메인 설정 페이지로 돌아가게 되며, 해당 설정의 <DNT>**revisions**</DNT> 카운트가 1 증가한 것을 확인할 수 있습니다.

설정으로 돌아가면 목록에서 새 버전을 선택하여 볼 수 있습니다.

<img title="Configuration details pane" alt="The configuration details pane showing tabs, YAML content, version information, deployment status indicating which fleets use this configuration, and action buttons for Download, Copy, and Clone." src="/images/fleet-control-config-detail.webp" />

## 설정 및 해당 버전 삭제

설정을 삭제하려면 먼저 해당 설정의 모든 개별 버전을 삭제해야 합니다. 마지막 버전이 제거되면 상위 설정은 자동으로 삭제됩니다.

<Callout title="중요: 삭제 제한" variant="caution">
  현재 활성 구현, 배포의 일부인 설정 버전을 삭제할 수 없습니다. 설정 세부 정보 창의 구현, 배포 상태 섹션에는 해당 버전이 구현되고, 배포되는 플릿이 표시됩니다. 삭제하기 전에 먼저 다른 설정 버전을 사용하는 해당 함대에 대한 새 구현, 배포를 만들어야 합니다.
</Callout>

설정 버전을 삭제하려면:

1. 설정 페이지로 이동하세요.
2. 수정하려는 설정의 이름을 클릭하여 세부 정보 창을 엽니다.
3. 왼쪽에는 모든 버전 목록이 표시됩니다. 제거하려는 버전 위에 마우스를 올려놓고 나타나는 줄임표(...)를 클릭합니다.
4. 메뉴에서 <DNT>**Delete**</DNT> 선택하세요.
5. 나머지 모든 버전에 대해 이 과정을 반복하십시오. 마지막 버전이 삭제되면 메인 설정 페이지의 목록에서 상위 설정이 제거됩니다.

<img title="Configuration version view" alt="Configuration details showing version selectors demonstrating the ability to view, select, and delete different configuration versions." src="/images/fleet-control-config-version-view.webp" />

### 설정 복제

복제는 기존 설정을 기반으로 새로운 설정을 만드는 빠른 방법입니다.

1. 메인 설정 페이지에서 메인 <DNT>**Create configuration**</DNT> 옆에 있는 점 세 개(`...`) 버튼을 클릭합니다.
2. 다운 메뉴에서 복제의 시작점으로 사용하려는 기존 설정을 선택하세요.
3. 이렇게 하면 선택한 설정의 내용이 미리 채워진 두 창으로 구성된 편집 보기가 열립니다.
4. 이제 새로운 설정을 작성하고 저장할 수 있습니다.

## 관리되는 주요 논문 페이지

<DNT>**New Relic Control → Managed Entities**</DNT> 으로 이동하면 쿠버네티스 클러스터 또는 Agent Control이 설치된 호스트 등 관리되는 모든 부분에 대한 중앙 집중식 보기를 얻을 수 있습니다. 이 페이지는 <DNT>Fleet Control</DNT> 에서 관리할 수 있는 모든 엔티티에 대한 완전한 자산 목록 역할을 하며 <DNT>**Fleet managed**</DNT> 및 <DNT>**Unassigned**</DNT> 의 두 탭으로 구성됩니다.

### 차량 관리 탭

이 탭에는 현재 함대에 할당된 모든 관리 항목이 나열됩니다. 표에는 다음 정보가 나와 있습니다.

* <DNT>**Entity name:**</DNT> 이름을 클릭하면 쿠버네티스 클러스터 탐색기와 같은 뉴렐릭 플랫폼에서 해당 모임의 탐색기 환경으로 바로 이동합니다.
* <DNT>**Entity type:**</DNT> 관리되는 분리의 유형(예: 쿠버네티스 클러스터 또는 호스트)입니다.
* <DNT>**Instrumentation status:**</DNT> 측정의 현재 상태, 부분에 대한 로그입니다.
* <DNT>**Fleet:**</DNT> 해당 함대가 현재 어느 함대에 소속되어 있습니까? 차량 이름을 클릭하면 해당 차량 관리 페이지로 바로 이동할 수 있습니다.
* <DNT>**Account name:**</DNT> 관리 대상과 연결된 뉴럴릭 계정입니다.

<img title="Managed entities page" alt="The Fleet managed tab showing all managed entities with columns for entity name, type, instrumentation status with health indicators, fleet assignment, and account name." src="/images/fleet-control-managed-entities-page.webp" />

### 할당되지 않은 탭

이 탭에는 <DNT>Agent Control</DNT> 설치되어 있지만 현재 어떤 플릿에도 할당되지 않은 모든 관리 대상 엔티티 목록이 표시됩니다. 이 표는 다음과 같은 정보를 제공합니다.

* <DNT>**Entity name:**</DNT> 이름을 클릭하면 뉴렐릭 플랫폼에 있는 해당 부품의 탐색기 환경으로 바로 이동됩니다.
* <DNT>**Entity type:**</DNT> 관리 대상 엔티티의 유형입니다.
* <DNT>**Instrumentation status:**</DNT> 측정값의 현재 상태입니다.
* <DNT>**Account name:**</DNT> 관리 대상과 연결된 뉴럴릭 계정입니다.

이 보기는 <DNT>Fleet Control</DNT> 에서 아직 관리되지 않는 자산을 식별하는 데 특히 유용합니다. 모든 관리되는 파트를 하나의 차량에 할당하여 일관되게 구성되고 모니터링되도록 하는 모범 사례입니다.

<img title="Unassigned managed entities tab" alt="The Unassigned tab on the Managed Entities page showing a table of managed entities that have Agent Control installed but are not currently assigned to any fleet, with columns for entity name, type, instrumentation status, and account name." src="/images/fleet-control-me-unassigned-tab.webp" />

## 주요 에이전트 페이지

<DNT>**New Relic Control → Agents**</DNT> 으로 이동하면 조직 전체에서 <DNT>Fleet Control</DNT> 관리하는 모든 에이전트에 대한 전체 개요를 볼 수 있습니다.

페이지 상단에는 다음을 포함하여 에이전트의 상태를 빠르게 살펴볼 수 있는 요약 카드가 있습니다.

* 연결된 모든 에이전트의 수입니다.
* 연결이 끊긴 에이전트의 수로, 문제 해결, 해결이 필요할 수 있는 에이전트를 신속하게 식별하는 데 도움이 됩니다.
* 상위 500개 에이전트 유형에 대한 분포입니다.

요약 카드 아래에는 모든 관리 에이전트가 나열된 포괄적인 표가 있습니다. 이 표에서 에이전트 이름을 검색하고 에이전트 유형이나 보고 상태별로 필터링할 수 있습니다. 표에는 다음과 같은 열이 포함되어 있습니다.

* <DNT>**Type:**</DNT> 에이전트의 유형입니다(예: 뉴렐릭 인프라).
* <DNT>**Agent Name:**</DNT> 에이전트의 구체적인 이름입니다.
* <DNT>**Managed Entity:**</DNT> 에이전트가 연결된 관리 대상 엔티티입니다.
* <DNT>**Connection Status:**</DNT> 에이전트의 현재 연결 상태입니다.
* <DNT>**Fleet:**</DNT> 에이전트가 속한 함대입니다.
* <DNT>**Agent Version:**</DNT> 에이전트의 버전 번호입니다.
