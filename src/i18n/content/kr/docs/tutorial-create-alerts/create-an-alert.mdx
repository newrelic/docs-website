New Relic에서 경고 조건을 생성하는 방법에는 차트, NRQL 쿼리 또는 골든 메트릭을 사용하는 세 가지 방법이 있습니다. 선택한 옵션에 따라 생성 프로세스의 시작 부분이 다르지만 이후 단계는 모두 동일합니다. 즉, 한 가지 방법으로 조건을 생성하면 이를 통해 얻은 경험이 다른 옵션 중 하나를 사용하여 생성한 다른 조건으로 이전됩니다.

이 튜토리얼에는 앞서 언급한 세 가지 옵션 모두에 대한 단계가 포함되어 있으므로 원하는 방식으로 첫 번째 조건을 시작할 수 있습니다. 첫 번째 단계에서 따라가고 싶은 경로를 선택하기만 하면 절차가 끝날 때쯤 New Relic을 사용하여 첫 번째 경고 조건을 성공적으로 만들게 됩니다!

<Callout variant="tip">
  아래 단계에서 다루는 옵션이나 용어에 대한 추가 정보가 필요한 경우 [알림 문서를](/docs/alerts-applied-intelligence/overview/#concepts-terms) 확인하세요.
</Callout>

          차트를 사용하는 것은 미리 만들어진 쿼리에서 경고 조건을 만드는 좋은 방법이지만 경우에 따라 처음부터 만들어야 합니다. NRQL 쿼리에서 새 경고 조건을 생성하는 프로세스는 차트에서 생성하는 것과 거의 동일합니다. 주요 차이점은 미리 생성된 쿼리를 사용하는 대신 수동으로 NRQL 쿼리를 작성한다는 것입니다.

          <Callout variant="tip">
            NRQL은 New Relic의 쿼리 언어(New Relic&apos;s Query Language)의 약자로 모든 차트를 지원하고 데이터를 분석하고 쿼리하는 데 사용할 수 있는 표준화된 쿼리 방법입니다. NRQL을 처음 사용하는 경우 [NRQL 문서를](/docs/query-your-data/nrql-new-relic-query-language/get-started/introduction-nrql-new-relics-query-language/) 확인하여 [작동 방식](/docs/query-your-data/nrql-new-relic-query-language/get-started/introduction-nrql-how-nrql-works/) 및 [소개 자습서를](/docs/query-your-data/nrql-new-relic-query-language/get-started/introduction-nrql-tutorial/)포함하여 NRQL에 대해 자세히 알아볼 수 있습니다.
          </Callout>

          <img title="NRQL alert" alt="A screenshot displaying an alert condition created with a simple NRQL query" src="/images/alerts_screenshot-crop_NRQL-query.webp" />

          1. <DNT>**one.newrelic.com**</DNT> 에서 [<DNT>**Alerts**</DNT>](https://one.newrelic.com/alerts-ai) 선택한 다음 [<DNT>**Alert conditions (policies)**</DNT>](https://one.newrelic.com/alerts-ai/alerts-classic/policies) 선택합니다.

          2. <DNT>**New alert condition**</DNT> 선택합니다.

          3. NRQL 쿼리를 작성하여 조건을 만듭니다. 예를 들어 느린 트랜잭션을 모니터링하려는 경우 다음을 입력할 수 있습니다.

             ```sql
             SELECT average(duration) from Transaction facet name
             ```

    ## 신호 동작 조정

    <img title="Tuning your signal" alt="A screenshot displaying how to set options for a signal in an alert condition" src="/images/alerts_screenshot-crop_signal-tune.webp" />

    1. <DNT>**Set the data aggregation options**</DNT>:

       * <DNT>**Window duration**</DNT>: <DNT>**Window duration**</DNT> 설정을 사용하여 5분마다 또는 1시간마다 그룹화하는 등 데이터를 집계하는 빈도를 결정합니다. 이 기능의 설정은 모니터링하는 데이터의 종류에 따라 다릅니다. 무엇을 설정해야 할지 확실하지 않은 경우 기본 설정으로 그대로 둘 수 있습니다. 자세한 내용은 [첫 번째 공지 문서 만들기 를](/docs/alerts-applied-intelligence/new-relic-alerts/get-started/your-first-nrql-condition/#window-duration) 참조하세요.
       * <DNT>**Sliding window aggregation**</DNT>: [슬라이딩 윈도우 집계를](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/#sliding-window-aggregation) 사용할지 여부를 선택합니다. 이 기능은 겹치는 집계 창을 만들어 불규칙한 데이터를 처리할 때 더 부드러운 차트를 만듭니다. 이 기능은 기본적으로 비활성화되어 있으며 첫 번째 기능에서는 비활성화된 상태로 유지하는 것이 좋습니다.
       * <DNT>**Streaming method**</DNT>: 가장 일반적으로 사용되는 옵션은 <DNT>**Event flow**</DNT> 과 <DNT>**Event timer**</DNT> 입니다. <DNT>**Event flow**</DNT> 는 빈번하고 일관되게 데이터를 보고하는 데 가장 적합한 반면, <DNT>**Event timer**</DNT> 일관되지 않게, 드물게 보고하거나 일괄적으로 수집하는 데이터에 가장 적합합니다. 각 시나리오에서 어떤 방법을 사용해야 하는지 알아보기 위해 개별 설정에 대해 자세히 알아보려면 [문서를](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/understand-technical-concepts/streaming-alerts-key-terms-concepts/#aggregation-methods)참조하세요.

    2. <DNT>**Set the timing option**</DNT>: 이는 각 윈도우에서 이벤트를 평가하기 전에 기다리는 시간을 결정합니다. 지연 시간이 길수록 허위 알림이 줄어드는 것을 의미하지만, 공지 이벤트가 열리기 전 잠재적인 다운타임이 길어질 수도 있습니다. 이 기능은 선택한 옵션에 따라 다르게 표시됩니다. <DNT>**Event flow**</DNT> 및 <DNT>**Cadence**</DNT> 옵션에서는 <DNT>**Delay**</DNT> 으로, <DNT>**Event timer**</DNT> 옵션에서는 <DNT>**Timer**</DNT> 표시됩니다. 첫 번째 알림에는 기본 시간 설정을 유지하는 것을 권장합니다.

    3. <DNT>**Select the gap filling strategy**</DNT>: Gap-filling은 사용자가 정의한 신세틱스 값으로 데이터 보고의 빈칸을 채워주는 반응형 기능입니다. 데이터 보고가 산발적일 것이라고 생각하는 경우 빈 집계 창에 이 데이터를 채워 잘못된 알림을 방지할 수 있습니다. 첫 번째 공지의 경우 <DNT>**Gap-filling strategy**</DNT> 아래에서 <DNT>**None**</DNT> 선택하세요.

    4. <DNT>**Select the evaluation delay**</DNT>: 이 옵션을 사용하면 뉴렐릭이 설정된 기준에 따라 새 신호를 평가하기 전에 일정 시간 동안 대기할 수 있습니다. 데이터 스트림이 보고되기 시작할 때 오탐 공지 이벤트를 방지하는 데 유용합니다. 첫 번째 알림의 경우, 해당 옵션이 데이터에 적용되는지 확실히 알고 있지 않다면 비활성화 토글을 그대로 두셔도 됩니다.

    5. 완료한 후 <DNT>**Next**</DNT> 선택하여 진행 상황을 저장하고 다음 단계로 이동하세요.
  </Step>

    ## 정적 임계값 설정

    /* &lt;img title=&quot;정적 청년값 설정, 예측&quot; alt=&quot;정적 청년값 설정 방법을 표시하는 스크린샷, 공지 조건에 대한 내용&quot; src=&quot;/images/alerts_screenshot-crop_set-임계값, 이에.webp&quot; /&gt; */

    1. <DNT>**Security level**</DNT> 섹션에서 레버 값이 <DNT>**Critical**</DNT> 또는 <DNT>**Warning**</DNT> 상태인지 선택하여 시작하세요.

    2. 충족 시 공지 이벤트가 열리는 한도를 설정하세요. 이 값들은 조건이 어떤 역할을 하도록 설정하느냐에 따라 달라집니다. 예를 들어, 오류 메시지에 대한 조건을 생성하려는 경우, 레버 값, 참조를 <DNT>**equal to 1 at least once in 5 minutes**</DNT> 로 설정할 수 있지만, 지연 시간 문제의 장기간에 대한 조건을 생성하려는 경우, 옛 값, 참조를 <DNT>**above or equal to 50 for at least 15 minutes**</DNT> 로 설정할 수 있습니다.

    3. 다른 레버 값을 추가하려면 필요에 따라 <DNT>**Add threshold**</DNT> 또는 <DNT>**Add lost signal threshold**</DNT> 선택하세요. 손실된 신호의 분리 값은 보고를 중지할 수 있는 감시 작업이나 [null 값을 반환](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/#query-order) 할 수 있는 쿼리에 중요합니다.

    4. 완료한 후 <DNT>**Next**</DNT> 선택하여 진행 상황을 저장하고 다음 단계로 이동하세요.
  </Step>

  <Step>
    ## 경고 조건 세부 정보 추가

    /* &lt;img title=&quot;공지 조건 세부정보&quot; alt=&quot;공지 세부정보 생성 방법을 보여주는 스크린샷&quot; src=&quot;/images/alerts_screenshot-crop_condition-details.webp&quot; /&gt; */

    1. <DNT>**Name**</DNT> 필드에 공지 이름을 추가하세요. 이는 이 조건을 식별하는 데 도움이 될 뿐만 아니라 나중에 조건을 편집하거나 제거해야 하는 경우를 찾는 데도 도움이 됩니다.

    2. 연결할 이 조건에 대한 정책을 선택하세요. <DNT>**Existing policy**</DNT> 이 있는 경우 드롭다운에서 선택합니다. 그렇지 않은 경우 <DNT>**New policy**</DNT> 선택하고 여기에서 생성하여 [정책을 생성](/docs/alerts-applied-intelligence/new-relic-alerts/alert-policies/create-edit-or-find-alert-policy/#alert-policy-name) 해야 합니다.

    3. 알림에 대한 추가 설정을 지정하세요. 여기에는 공개 공지 이벤트가 자동으로 종료될 때까지의 기간을 설정하고 공지 이벤트에 대한 [사용자 정의 설명을](/docs/alerts/create-alert/condition-details/custom-alert-event-descriptions) 입력하는 것이 포함됩니다.

    4. 완료하려면 <DNT>**Save condition**</DNT> 선택하세요.
  </Step>
</Steps>

2. 기존 공지 조건을 생성하려는 정책을 클릭하거나 새 정책을 생성하세요.

3. <DNT>**New alert condition**</DNT>를 선택한 다음 <DNT>**Build a classic alert**</DNT>를 클릭합니다.

4. 나열된 카테고리 중 하나에서 조건을 구축하십시오. 이는 공지 조건이 귀하에게 통지하는 내용을 결정합니다.

5. 다음 단계에 대한 옵션 범위를 좁히려면 조건 유형을 선택하세요. 범위를 선택해야 할 수도 있습니다.

6. 모니터링하려는 부분 또는 부분을 선택하세요. 검색창을 사용하여 모임 목록의 범위를 좁히세요. APM 차트 유형 조건을 선택한 경우 특정 태그를 공유하는 목표 애플리케이션을 동적으로 선택할 수 있습니다.

7. 알림 설정, 알림 값 및 기타 조건 설정을 설정하세요. 조건의 이름을 지정한 후 저장하세요.

이제 공지조건이 생성되었습니다! 알림에 대한 다양한 설정 옵션을 제공하므로 모니터링 요구 사항에 맞게 알림을 미세 조정할 수 있습니다. NRQL 쿼리에서 조건을 생성할 수도 있습니다. 이에 대해서는 이 튜토리얼 시리즈에서 다음에 다루겠습니다.

## 알림에서 더 많은 것을 얻으십시오 [#more-alerting]

팀에서 알림을 보다 효율적이고 관리하기 쉽게 만드는 데 도움이 되는 고급 알림 기능이 있습니다.

* 알림 수신 위치와 시간을 더욱 세밀하게 제어하고 문제 발생 시 적절한 담당자에게 알림을 보내려면, [알림우](/docs/alerts/get-notified/alert-event-workflows/) 사용 방법을 참조하세요.
* 거의 즉각적인 알림 알림을 활성화하여 평균 해결 시간(MTTR)(MTTR)을 줄이는 데 도움을 주고 싶다면 [이상 작용 감지](/docs/alerts-applied-intelligence/applied-intelligence/anomaly-detection/anomaly-detection-applied-intelligence/) 사용 방법을 참조하세요.
* API를 사용하여 New Relic 알림을 관리하려면 [REST API 또는 GraphQL을](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/alerts-api/intro-alerts-api/)사용하면 됩니다.
