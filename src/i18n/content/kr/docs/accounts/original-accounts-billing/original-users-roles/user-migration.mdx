거의 모든 New Relic 사용자는 2020년 7월에 도입된 새로운 사용자 모델을 사용하고 있습니다. 이 튜토리얼은 원래 사용자 모델에 여전히 사용자가 있는 New Relic 고객을 대상으로 새 사용자 모델로 마이그레이션합니다.

## 배경 [#background]

2020년 7월에 새롭고 개선된 사용자 모델을 출시했습니다. 이 새로운 사용자 모델은 사용자와 역할 및 계정에 대한 액세스를 관리하는 더 간단하고 효율적인 방법을 제공합니다.

처음에 이 새 모델은 주로 신규 고객에게 제공되었지만 기존 New Relic 조직의 사용자는 원래 사용자 모델을 유지했습니다.그러나 이제 일부 [요구 사항](#requirements) 을 충족하는 일부 원래 사용자 모델 조직은 마이그레이션 마법사를 사용하여 사용자를 새 모델로 마이그레이션할 수 있습니다.마이그레이션 프로세스가 완료되면 사용자는 최신 사용자 모델을 [사용하게 되며 사용자 및 계정 액세스를 관리하기 위한 새로운 절차를](/docs/accounts/accounts-billing/new-relic-one-user-management/introduction-managing-users/) 갖게 됩니다.

## 혜택 [#benefits]

사용자를 이 모델로 마이그레이션하면 다음과 같은 이점이 있습니다.

* 한 곳에서 여러 계정의 모든 사용자를 보고 관리합니다.
* 사용자를 추가하고 관리하는 단계가 더 적습니다.
* 유연한 인증 옵션.
* 사용자 관리를 위한 보다 세분화된 역할.
* Pro 및 Enterprise 고객의 경우: ID 공급자를 통한 자동화된 사용자 관리에 대한 액세스.

[새로운 사용자 모델의 이점에 대해 자세히 알아보십시오.](https://blog.newrelic.com/product-news/user-management-and-access-security/)

## 요구 사항 [#requirements]

요구 사항은 다음과 같습니다.

* 귀하와 귀하의 사용자는 [원래 사용자 모델](/docs/accounts/original-accounts-billing/original-product-based-pricing/overview-user-models) 에 있어야 합니다. 어느 쪽인지 확실하지 않은 경우 [사용자 모델 결정](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models/#determine-user-model) 을 참조하세요.
* 사용자 마이그레이션 마법사를 사용하려면 [소유자 역할](/docs/accounts/original-accounts-billing/original-users-roles/users-roles-original-user-model/#roles) 이 있고 전체 플랫폼 사용자여야 합니다.
* 사용자에게 특정 역할과 특정 계정을 부여하기 위해 권한을 설정해야 하는 경우 사용자에 대한 권한을 설정해야 하며 시작하기 전에 새로운 사용자 관리 개념을 이해해야 합니다(자세한 내용은 아래 참조).

## 권장: 청구 목적으로 사용자의 사용자 유형 검토 [#user-type]

[사용량 기반 가격 책정 모델을 사용하는 조직의 경우](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing) 사용자의 [사용자 유형](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type) 이 청구 요소입니다.

고려해야 할 사항:

* 현재 사용량 기반 가격 책정 모델을 사용 중이든 곧[해당 모델로 전환할](/docs/accounts/original-accounts-billing/original-product-based-pricing/switch-new-models) 계획이든 사용자를 마이그레이션하기 전에 사용자 유형을 검토하고 수정하는 것이 좋습니다.그 이유 중 하나는 원래 **사용자 및 역할** UI를 통해 사용자가 New Relic을 마지막으로 사용한 시간을 알 수 있지만 새 UI에는 아직 이 기능이 없으며 원래 UI는 어떤 사용자 유형을 사용할지 결정하는 데 유용할 수 있습니다. 선택하다.이에 대한 팁은 [사용자 유형 편집](/docs/accounts/original-accounts-billing/original-users-roles/users-roles-original-user-model/#full-user-count) 을 참조하십시오.
* 사용량 기반 가격 책정 모델의 경우 청구 가능한 사용자는 이전에 New Relic에 로그인했는지 여부에 관계없이 추가되는 즉시 청구할 수 있습니다.즉, UI에 `Pending invite` 태그가 있는 사용자는 청구 가능한 사용자 유형이 있는 경우에도 여전히 청구할 수 있습니다.이에 대한 자세한 내용은 [청구 가능한 사용자](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/user-count-billing) 를 참조하십시오.

## 사용자 관리 개념 이해 [#user-mgmt-concepts]

특정 계정 및/또는 특정 역할에 대한 사용자의 액세스를 제어해야 하는 경우(모든 사용자에게 모든 액세스 권한을 부여하는 것과 대조적으로) 완전히 다른 새로운 사용자 관리 개념에 대한 몇 가지 기본 사항을 배워야 합니다. 오래된 개념에서. **사용자의 기존 역할 및 권한은 새 사용자 레코드** 로 이전되지 않으므로 사용자 마이그레이션 프로세스의 일부에서는 역할 및 계정에 대한 필수 액세스 권한이 있는 사용자 그룹을 설정해야 합니다.

다음은 새 사용자 모델이 작동하는 방식에 대한 가장 중요한 몇 가지 개념입니다.

* 사용자는 "인증 도메인"이라는 컨테이너에 있습니다.이 도메인은 사용자가 New Relic에 추가되는 방식을 수동(UI에서) 또는 자동(SCIM을 통해)으로 제어합니다.또한 사용자가 수동으로(이메일/비밀번호를 사용하여) 로그인하거나 SAML SSO를 사용하여 로그인하는 방법을 제어합니다.대부분의 조직에는 하나 또는 두 개의 인증 도메인이 있습니다. 하나는 기본 수동 설정을 위한 것이고 다른 하나는 보다 자동적인 방법을 위한 것입니다.
* 사용자는 하나 이상의 그룹(예: 기본 **Admin** 그룹 또는 **Contractors** 같은 사용자 지정 그룹)에 할당될 수 있습니다. 대규모 조직의 경우 사용자가 여러 그룹에 할당되는 경우가 많습니다.
* 사용자 그룹에 특정 역할 및 특정 계정에 대한 액세스 권한을 부여하려면 이를 신규 또는 기존 그룹에 추가해야 합니다. 예를 들어 **계약자** 그룹에 하나 이상의 계정에 대한 기본 **모든 제품 관리자** 역할에 대한 액세스 권한을 부여하거나 해당 그룹에 사용자 지정 역할을 부여할 수 있습니다.

자세히 알아보려면:

* [사용자 관리 개념을](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts#understand-concepts) 검토합니다.
* 이 [예제 계획 스프레드시트](https://docs.google.com/spreadsheets/d/1FnguDXRUX9FGY14oV4Gx6O08v4vNC2Pv0GGCsU7Pxuw/edit?usp=sharing) 를 사용하여 사용자 그룹 액세스를 계획하십시오.

## 0단계: 마이그레이션 마법사 찾기 및 시작 [#start-migration]

시작하기 전에 위 [의 요구 사항](#requirements) 과 기타 권장 사항을 읽었는지 확인하십시오. 마법사 사용을 시작하려면:

1. **[one.newrelic.com > All capabilities](https://one.newrelic.com/all-capabilities) > Apps** 으로 이동합니다.
2. 앱 표에서 **사용자 마이그레이션 연습** 앱을 클릭합니다.
3. 추가 도움말과 컨텍스트가 필요하면 아래 섹션에서 특정 마이그레이션 마법사 페이지에 대한 팁을 참조하세요. 이 문서에서 혼란스러운 부분을 발견하면 이 페이지의 **문제 만들기** 버튼을 사용하여 피드백을 보내주십시오.

이 페이지에서 마이그레이션할 관리자를 선택합니다. 이 단계에서는 선택한 관리자에 대한 새 모델에 대한 사용자 레코드를 생성하고 이를 **관리자** 그룹에 할당합니다. 완료되면 해당 관리자는 New Relic에 로그인할 때 새 사용자 레코드를 사용할 수 있으며(아래 이미지 참조) 마이그레이션 마법사 프로세스가 완료될 때까지 이전 사용자 레코드와 새 사용자 레코드에 모두 액세스할 수 있습니다. 이 단계에서 마이그레이션하는 모든 관리자는 마이그레이션 마법사를 완료하는 데 도움을 줄 수 있습니다. 이는 관리자를 먼저 마이그레이션하는 것의 이점 중 하나입니다. 이 단계가 완료된 후 마이그레이션 마법사에 액세스하려면 모든 사용자가 새 사용자 기록으로 로그인해야 합니다. 이를 위해서는 New Relic에서 로그아웃하고 로그인 화면에 다시 액세스하여 모든 로그인 정보를 볼 수 있어야 합니다. 옵션에서.

마이그레이션 프로세스를 완료한 후 언제든지 관리자를 더 추가하거나 기존 관리자를 제거 및 편집할 수 있습니다. 이것이 지금 많은 또는 모든 관리자를 데려오고 나중에 설정을 조정해야 하는 이유일 수 있습니다.

다음은 **관리자 추가** 페이지 사용에 대한 몇 가지 팁입니다.

* 마이그레이션이 수행되는 계정을 이해하고 있는지 확인해야 합니다. 사용자 마이그레이션은 선택한 상위 계정( **포함된 계정** 드롭다운에서 볼 수 있음) 및 하위 계정( **연결된 모든 계정 보기** 버튼을 사용하여 볼 수 있음)에만 적용됩니다. 예상한 모든 계정이 표시되지 않는 경우 조직에 여러 상위/하위 계정 구조가 있기 때문일 수 있으며 이는 마이그레이션 프로세스를 두 번 이상 수행해야 함을 의미합니다.

* 일부 사용자만 새 사용자 모델로 마이그레이션하여 시작할 계획이라면 원래 사용자 모델 사용자를 관리할 관리자가 있도록 일부 원래 사용자 모델 관리자를 남겨두는 것이 좋습니다.

## 2단계: 조직 설정 [#page2]

<img
  title="user-migration-page-2.png"
  alt="User migration page 2"
  src={accountsUserMigrationPage2}
/>

a) 더 많은 구성 옵션을 허용하는 안내 설정 또는 b) 더 적은 단계의 자동 설정을 선택할 수 있습니다. 이것을 선택하는 몇 가지 팁:

* 마이그레이션하려는 사용자에 대해 SAML SSO 또는 SCIM을 사용하지 않고 마이그레이션된 모든 사용자에게 모든 계정에 대한 액세스 권한을 부여해도 된다면 자동 설정 옵션을 사용하는 것이 좋습니다. (나중에 언제든지 계정에 대한 사용자 액세스를 더 많이 분할할 수 있습니다.)
* SCIM 프로비저닝 관련: 가까운 시일 내에 SCIM 프로비저닝을 통해 New Relic 사용자를 관리할 계획이라면 SCIM이 활성화된 상태로 마이그레이션할 수 있도록 마이그레이션을 기다리는 것을 고려해야 합니다. 이는 사용자가 마이그레이션되면 특정 인증 도메인에 상주하고 해당 도메인에 사용자가 추가되면 해당 도메인에서 SCIM과 비SCIM(수동) 간에 전환할 수 없기 때문입니다.

## 4단계: 인증 도메인 설정 [#page4]

<img
  title="user-migration-page-4.png"
  alt="User migration page 4"
  src={accountsUserMigrationPage4}
/>

이 섹션은 사용자 관리 방법(New Relic에 추가 및 업데이트 방법)과 인증 방법(로그인)을 제어합니다.이 페이지에 대한 중요 참고 사항:

* SCIM 옵션("ID 공급자별 관리") 또는 SAML SSO 옵션을 선택하면 마이그레이션 마법사를 종료하고 New Relic UI 및 ID 공급자의 다른 곳에서 구성한 다음 마법사에서 작업을 완료해야 합니다.
* SCIM을 사용하는 경우 사용자 자산을 마이그레이션하는 이 사용자 마이그레이션 프로세스의 마지막 단계 전에 ID 공급자로부터 그룹 및 사용자를 푸시하고 액세스를 구성해야 합니다. 이렇게 하면 사용자의 자산이 SCIM 프로비저닝된 사용자 레코드로 마이그레이션되고 해당 사용자가 로그인할 때 액세스할 수 있습니다.
* 인증 도메인이 관리되는 방식(ID 제공자에 의해 또는 수동으로)을 설정한 후에는 변경할 수 없으며 해당 설정을 조정하려면 새 인증 도메인을 만들고 새 사용자 레코드를 추가해야 합니다.가능한 경우 사용자 관리 방식을 곧 변경할 수 있으므로 사용자 기록 이전을 기다려야 하는 이유가 될 수 있습니다.

두 개의 인증 도메인 섹션에 대한 자세한 내용은 다음과 같습니다.

### 사용자 관리(수동 및 ID 제공자) [#managing-users]

사용자 추가 및 관리 방법에 대해 **수동** 또는 **ID 공급자** (SCIM)를 선택할 수 있습니다. ID 공급자를 사용하여 SCIM을 통해 사용자를 프로비저닝하는 옵션은 조직에 [Pro 또는 Enterprise 버전이](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing/#editions)있는 경우에만 사용할 수 있습니다.

**ID 공급자** 를 선택하는 경우 [자동화된 사용자 관리 단계를](/docs/accounts/accounts/automated-user-management/automated-user-provisioning-single-sign#how-to) 따라야 하지만 이 프로세스의 뒷부분에서 수행할 그룹 생성과 관련된 단계는 건너뜁니다. 자동화된 사용자 관리 단계를 완료하면 마이그레이션 마법사와 이 문서로 돌아갑니다.

이 단계를 완료한 후에는 최대한 빨리 사용자 마이그레이션 프로세스를 완료하는 것이 좋습니다. 이 단계를 완료하고 마법사 절차를 완료할 때까지 사용자는 동일한 로그인( [1단계의 로그인 스크린샷 참조)과 연결된 두 개의 사용자 레코드를 갖게 되거나(1단계](#page1) 의 로그인 스크린샷 참조) 볼 것으로 예상하는 자산(예: 대시보드)이 누락될 수 있습니다.

ID 공급자를 New Relic과 동기화하고 그룹 액세스를 설정하기 위한 몇 가지 팁:

* Okta, Azure 또는 OneLogin에 대해 New Relic 앱을 이미 사용하고 있다면 오래된 버전을 사용하고 있을 가능성이 큽니다. 오래된 앱의 제목은 "New Relic by account"이고 최신 필수 앱의 제목은 "New Relic by organization"입니다.
* 이러한 단계를 완료하면 새 사용자 모델에서 새 사용자 레코드가 생성되고 ID 제공자 구성을 기반으로 New Relic에서 동기화됩니다. 사용자 프로비저닝을 완료한 후 새 [**사용자 관리** UI](#manage-users) 에 해당 사용자 레코드가 표시되는지 확인합니다.
* 새로운 New Relic 사용자 관리 UI에 액세스하려면 새 사용자 기록을 통해 로그인해야 합니다. 이를 위해서는 로그아웃하고 다시 로그인한 다음 이메일을 확인해야 이메일과 관련된 모든 로그인이 표시될 수 있습니다.

### 로그인 방법(수동 및 SSO) [#login-methods]

로그인 방법을 통해 해당 사용자가 로그인하는 방법을 선택할 수 있습니다. a) 이메일/비밀번호 로그인 또는 b) 싱글 사인온(SSO)을 선택할 수 있습니다. SSO는 Pro 또는 Enterprise 버전이 있는 조직에서만 사용할 수 있습니다.

SSO를 사용 중이지만 SCIM을 사용하지 않는 경우 SSO [를 설정하려면 추가 단계를](/docs/accounts/accounts-billing/new-relic-one-user-management/authentication-domains-saml-sso-scim-more/#saml) 완료해야 합니다. (이미 이전 단계의 SCIM 절차를 따랐다면 SAML SSO를 이미 설정했을 것입니다.)

SAML SSO 설정을 위한 몇 가지 팁:

* Okta, Azure 또는 OneLogin에 대해 New Relic 앱을 이미 사용하고 있다면 오래된 버전을 사용하고 있을 가능성이 큽니다. 오래된 앱의 제목은 "New Relic by account"이고 최신 필수 앱의 제목은 "New Relic by organization"입니다.
* 새로운 New Relic 사용자 관리 UI에 액세스하려면 새 사용자 기록을 통해 로그인했는지 확인해야 합니다. 해당 이메일과 관련된 모든 로그인을 보려면 로그아웃하고 다시 로그인한 다음 이메일을 확인해야 할 수 있습니다.
* SSO 설정 절차를 완료한 다음 마이그레이션 마법사로 돌아와 마이그레이션 프로세스를 계속할 수 있습니다.
* 인증 방법을 두 개 이상 선택하는 경우 새 [인증 도메인](/docs/accounts/accounts-billing/new-relic-one-user-management/authentication-domains-saml-sso-scim-more/#auth-domain-definition) 을 추가해야 합니다.

    사용자 목록을 다시 업로드하고 이 단계를 완료하면 최신 사용자 모델에 대한 사용자 레코드가 생성됩니다.이후 단계에서 이러한 사용자의 자산을 전환할 수 있습니다.

    * 새 사용자의 이메일 주소가 대소문자 일치를 포함하여 원래 사용자 기록 이메일 주소와 일치하는지 확인합니다. 이메일 주소를 키 값으로 사용하여 사용자를 일치시키고 이후 단계에서 사용자 관련 자산을 전환합니다.

## 6단계: 설정에 액세스 [#access-settings]

<img
  title="user-migration-page-6.png"
  alt="User migration page 6"
  src={accountsUserMigrationPage6}
/>

이 단계는 사용자 그룹이 액세스할 수 있는 역할과 계정을 설정하는 것입니다. **사용자의 기존 역할 및 권한은 유지되지 않습니다.** 즉, 사용자가 특정 계정이나 특정 역할에 액세스할 수 있도록 설정해야 하는 경우 이 단계에서 그룹 액세스를 구성하는 것이 좋습니다.

<Callout variant="important">
  SCIM 프로비저닝 사용자의 경우: SCIM을 사용하여 ID 제공자로부터 사용자 및 그룹을 가져오는 경우 New Relic에서 사용자 및 그룹을 편집할 수 없습니다. 모든 사용자 및 그룹 변경 사항은 아이덴티티 공급자에서 처리됩니다.
</Callout>

사용자 그룹에 액세스 권한을 부여하려는 각 계정 및 역할에 대해 새 그룹이나 기존 그룹에 추가해야 합니다. 그룹 액세스를 이해하는 데 도움이 되는 리소스:

* 사용자 액세스 작동 방식에 대한 기본 사항은 [사용자 관리 개념](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts/#understand-concepts) 을 참조하십시오.
* 그룹 액세스를 추가하고 편집하는 방법에 대한 자습서는 [사용자 관리 자습서](/docs/accounts/accounts-billing/new-relic-one-user-management/tutorial-add-new-user-groups-roles-new-relic-one-user-model/#group-access) 를 참조하십시오.
* 그룹 액세스 계획에 대한 도움말은 [사용자 그룹 계획 스프레드시트](https://docs.google.com/spreadsheets/d/1FnguDXRUX9FGY14oV4Gx6O08v4vNC2Pv0GGCsU7Pxuw/edit?usp=sharing) 를 참조하십시오.
* 유용한 비디오는 [사용자 관리 비디오](/docs/accounts/accounts-billing/new-relic-one-user-management/user-mgmt-videos) 를 참조하십시오.

## 7단계: 사용자 자산 마이그레이션 [#migrate-assets]

<img
  title="user-migration-page-7.png"
  alt="User migration page 7"
  src={accountsUserMigrationPage7}
/>

이 단계가 완료되면 사용자의 개인 자산이 새 사용자 레코드로 마이그레이션되고 원래 사용자 레코드가 삭제됩니다. 현재 New Relic에 로그인한 사용자의 경우 이 단계를 완료하면 로그아웃하거나 현재 브라우저 세션이 만료될 때까지 현재 New Relic 세션이 중단되지 않습니다.

마이그레이션되는 사용자 자산은 다음과 같습니다.

* 대시보드
* 즐겨찾기
* 주간 이메일 설정
* 이메일 수신 동의/선택 해제 기본 설정
* 사용자별 [사용자 키](/docs/apis/intro-apis/new-relic-api-keys/#user-api-key)
* 새로운 유물 앱 [NerdStorage 데이터](https://developer.newrelic.com/explore-docs/nerdstorage)

사용자가 New Relic을 사용하는 여러 조직에 액세스할 수 있는 경우(예: 해당 사용자가 계약자인 경우) 해당 조직이 모두 사용자를 마이그레이션할 때까지 원래 사용자 모델 레코드가 완전히 삭제되지 않습니다. 이러한 사용자는 원래 사용자 레코드와 하나 이상의 새 사용자 레코드를 모두 갖게 되며 이 경우 로그인할 때 표시됩니다( [1페이지 섹션의 로그인 스크린샷 참조](#page1) ).

## 문제점 해결 [#troubleshooting]

마이그레이션 후 몇 가지 일반적인 문제:

* 관리자 수준의 역할이 할당되었지만 New Relic 플랫폼 기능에 액세스하려고 할 때 오류 메시지가 표시되는 경우, 역할이 아닌 [관리 설정](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts#admin-settings) ( **조직 설정** 및/또는 **인증 도메인 설정** )이 할당되었기 때문일 수 있습니다. 특정 계정의 New Relic 기능에 액세스하려면 하나 이상의 역할(예: **모든 제품 관리자** 또는 사용자 지정 역할)이 필요합니다.

* 마이그레이션을 완료했거나 마이그레이션 중인데도 원래 사용자 관리 UI( **계정 설정** 탭을 통해 액세스한 UI)가 계속 표시되는 경우 원래 사용자 모델 레코드에 여전히 로그인되어 있기 때문일 수 있습니다. 이에 대한 몇 가지 해결책:

  * New Relic에서 로그아웃했다가 다시 로그인하여 **이메일 확인** 옵션을 선택합니다. 이메일을 확인했으면 "Original New Relic account"가 아닌 "Organization"이라는 로그인 옵션을 선택하십시오.
  * 그래도 문제가 지속되면 브라우저 캐시를 지우고 다시 로그인해 보세요.

* 로그인할 때 선택할 계정이 여러 개인 경우 다음과 같은 이유로 인해 발생할 수 있습니다.

  * 새 모델에서 사용자가 생성되었지만 마이그레이션 프로세스가 완료되지 않은 경우 원래 사용자와 새 사용자 모두에 액세스할 수 있습니다.
  * 여러 조직에 속한 경우 마이그레이션되지 않은 조직의 이전 로그인과 새 사용자 로그인의 조합을 볼 수 있습니다.
  * 여러 조직 또는 동일한 조직 내의 여러 인증 도메인에 속해 있기 때문에 이메일 주소에 여러 사용자 레코드가 있을 수 있습니다.

## 완료한 후 [#manage-users]

사용자가 새 사용자 모델로 마이그레이션되면 [사용자 메뉴](/docs/accounts/accounts-billing/general-account-settings/intro-account-settings) , **관리** 를 클릭하고 다음 UI 페이지를 사용하여 사용자를 찾고 관리할 수 있습니다.

* **사용자 관리** : 사용자를 보고 추가하고 유형(기본 또는 전체) 변경, 그룹 변경, 사용자 업그레이드 요청 승인에 사용합니다.
* **액세스 관리** : 그룹 액세스를 생성 및 편집하고 인증 도메인(SAML SSO 설정 및 SCIM 설정 등)을 구성하는 데 사용합니다.

그룹 액세스 계획에 대한 몇 가지 [팁은 사용자 관리 개념에](#user-mgmt-concepts) 대한 팁을 참조하십시오.

이러한 도구 및 개념에 대한 자세한 내용은[사용자 관리 문서](/docs/accounts/accounts-billing/new-relic-one-user-management/introduction-managing-users) 를 참조하십시오.
