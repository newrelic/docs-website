2021년 4월 12일부터 원래 사용자 모델의 사용자가 있는 일부 고객이 자체 서비스를 제공하고 해당 사용자를 New Relic One 사용자 모델로 마이그레이션할 수 있습니다.

## 배경 [#background]

2020년 7월 30일, 우리는 New Relic One 사용자 모델이라는 새롭고 향상된 사용자 모델을 출시했습니다. 이 새로운 모델은 사용자와 역할 및 계정에 대한 액세스를 관리하는 더 간단하고 효율적인 방법을 제공합니다.

처음에 이 새 모델은 주로 신규 고객에게 제공되었지만 기존 New Relic 조직의 사용자는 원래 사용자 모델을 유지했습니다. 그러나 이제 일부 [요구 사항](#requirements) 을 충족하는 일부 원래 사용자 모델 조직은 마이그레이션 마법사를 사용하여 사용자를 새 모델로 마이그레이션할 수 있습니다. 마이그레이션 프로세스가 완료되면 사용자는 New Relic One 사용자 모델을 [사용하게 되며 사용자 및 계정 액세스를 관리하기 위한 새로운 절차를](/docs/accounts/accounts-billing/new-relic-one-user-management/introduction-managing-users/) 갖게 됩니다.

## 혜택 [#benefits]

사용자를 이 모델로 마이그레이션하면 다음과 같은 이점이 있습니다.

* 한 곳에서 여러 계정의 모든 사용자를 보고 관리합니다.
* 사용자를 추가하고 관리하는 단계가 더 적습니다.
* 유연한 인증 옵션.
* 사용자 관리를 위한 보다 세분화된 역할.
* Pro 및 Enterprise 고객의 경우: ID 공급자를 통한 자동화된 사용자 관리에 대한 액세스.

[새로운 사용자 모델의 이점에 대해 자세히 알아보십시오.](https://blog.newrelic.com/product-news/user-management-and-access-security/)

## 요구 사항 [#requirements]

요구 사항은 다음과 같습니다.

* 귀하와 귀하의 사용자는 [원래 사용자 모델](/docs/accounts/original-accounts-billing/original-product-based-pricing/overview-user-models) 에 있어야 합니다. 어느 쪽인지 확실하지 않은 경우 [사용자 모델 결정](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models/#determine-user-model) 을 참조하세요.
* 사용자 마이그레이션 마법사를 사용하려면 [소유자 역할](/docs/accounts/original-accounts-billing/original-users-roles/users-roles-original-user-model/#roles) 이 있고 전체 플랫폼 사용자여야 합니다.

## 사용자를 마이그레이션해야 하는지 결정 [#recommendations]

자체적으로 사용자 마이그레이션을 수행하기로 선택한 조직에 적용되는 몇 가지 권장 사항이 아래에 나와 있습니다. **New Relic 담당자의 도움을 받는 경우 이러한 고려 사항이 적용되지 않습니다** . 이 경우 계정 담당자가 안내를 드릴 것입니다.

다음과 같은 상황에서는 사용자 마이그레이션 마법사를 사용 **하지 않는** 것이 좋습니다.

* [새로운 사용자 모델이 미치는 영향과 제한 사항](/docs/accounts/original-accounts-billing/original-product-based-pricing/overview-user-models/#differences) 이 부정적인 영향을 미칠 수 있다고 생각하는 경우.
* 사용자 관리 기능을 위해 계정 수준 역할이 필요한 경우. 사용자 관리와 관련된 역할(사용자 추가 및 업데이트, 사용자 유형 변경, 액세스 권한 생성)은 현재 전체 조직에 적용되며 특정 계정에 할당할 수 없습니다.

사용자 마이그레이션 마법사를 사용해야 하는지 여부에 대한 질문이 있는 경우 New Relic 계정 담당자에게 문의하세요.

## 선택 사항: 사용자의 사용자 유형 검토 [#user-type]

[사용량 기반 가격 책정 모델을 사용하는 조직의 경우](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing) 사용자의 [사용자 유형](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type) 이 청구 요소입니다.

현재 사용량 기반 가격 책정 모델을 사용 중이든 곧[해당 가격 책정 모델로 전환할](/docs/accounts/original-accounts-billing/original-product-based-pricing/switch-new-models) 계획이든 사용자 마이그레이션을 수행하기 전에 사용자의 사용자 유형을 수정하는 것이 좋습니다. 그 이유 중 하나는 원래 **사용자 및 역할** UI를 통해 사용자가 New Relic을 마지막으로 사용한 시간(새 UI에는 아직 없음)을 볼 수 있고, 이는 어떤 사용자 유형을 만들지 결정하는 데 유용할 수 있기 때문입니다. 그들을. 이를 수행하는 방법에 대한 팁은 [사용자 유형 편집](/docs/accounts/original-accounts-billing/original-users-roles/users-roles-original-user-model/#full-user-count) 을 참조하십시오.

## 선택 사항: 사용자 관리 개념 이해 [#before-beginning]

특정 계정이나 특정 역할에 대한 사용자의 액세스를 제어해야 하는 경우가 많을수록 새로운 사용자 관리 개념에 대한 몇 가지 기본 사항을 이해하는 데 도움이 됩니다. 다음은 간략한 요약입니다.

* 사용자는 "인증 도메인"이라는 컨테이너에 있습니다. 도메인은 사용자가 New Relic에 추가되는 방식을 제어합니다(UI를 사용하여 수동으로 또는 SCIM을 통해 자동으로). 또한 사용자가 로그인하는 방법(수동으로 이메일/비밀번호 사용 또는 SAML SSO 사용)을 제어합니다. 대부분의 조직에는 하나 또는 두 개의 인증 도메인이 있습니다. 하나는 기본 수동 설정을 위한 것이고 다른 하나는 보다 자동적인 방법을 위한 것입니다.
* 사용자는 하나 이상의 그룹(예: 기본 **Admin** 그룹 또는 **Contractors** 같은 사용자 지정 그룹)에 할당될 수 있습니다. 대규모 조직의 경우 사용자가 여러 그룹에 할당되는 경우가 많습니다.
* 사용자 그룹에 특정 역할 및 특정 계정에 대한 액세스 권한을 부여하려면 **액세스 권한 부여** 를 생성해야 합니다. 예를 들어 **계약자** 그룹에 하나 이상의 계정에 대한 기본 **모든 제품 관리자** 역할에 대한 액세스 권한을 부여하거나 해당 그룹에 사용자 지정 역할을 부여할 수 있습니다.

자세히 알아보려면:

* [사용자 관리 개념을](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts#understand-concepts) 검토합니다.
* 이 [예제 계획 스프레드시트](https://docs.google.com/spreadsheets/d/1FnguDXRUX9FGY14oV4Gx6O08v4vNC2Pv0GGCsU7Pxuw/edit?usp=sharing) 를 사용하여 그룹 및 액세스 권한을 계획하세요.

## 0단계: 마이그레이션 마법사 찾기 및 시작 [#start-migration]

시작하기 전에 위 [의 요구 사항](#requirements) 과 기타 권장 사항을 읽었는지 확인하십시오. 마법사 사용을 시작하려면:

1. [one.newrelic.com](https://one.newrelic.com) 의 상단 탐색에서 **앱** 을 클릭합니다.
2. 앱 표에서 **사용자 마이그레이션 연습** 앱을 클릭합니다.
3. 추가 도움말과 컨텍스트가 필요하면 아래 섹션에서 특정 마이그레이션 마법사 페이지에 대한 팁을 참조하세요. 이 문서에서 혼란스러운 부분을 발견하면 이 페이지의 **문제 만들기** 버튼을 사용하여 피드백을 보내주십시오.

이 페이지에서 마이그레이션할 관리자를 선택합니다. 이 단계에서는 선택한 관리자에 대한 새 모델에 대한 사용자 레코드를 생성하고 이를 **관리자** 그룹에 할당합니다. 완료되면 선택한 관리자는 New Relic에 로그인할 때 사용할 수 있는 새 사용자 레코드를 갖게 되며(아래 이미지 참조) 마이그레이션 마법사 프로세스가 완료될 때까지 이전 사용자 레코드와 새 사용자 레코드에 모두 액세스할 수 있습니다. 여기에서 선택한 관리자는 새 사용자 레코드에서 마이그레이션 마법사를 사용할 수 있습니다. 이는 다른 팀 구성원을 선택하여 마이그레이션을 완료하려는 경우에 유용할 수 있습니다.

마이그레이션 프로세스를 완료한 후 언제든지 관리자를 더 추가하거나 기존 관리자를 제거 및 편집할 수 있습니다. 이것이 지금 많은 또는 모든 관리자를 데려오고 나중에 설정을 조정해야 하는 이유일 수 있습니다.

다음은 **관리자 추가** 페이지 사용에 대한 몇 가지 팁입니다.

* 마이그레이션이 수행되는 계정을 이해하고 있는지 확인해야 합니다. 사용자 마이그레이션은 선택한 상위 계정( **포함된 계정** 드롭다운에서 볼 수 있음) 및 하위 계정( **연결된 모든 계정 보기** 버튼을 사용하여 볼 수 있음)에만 적용됩니다. 예상한 모든 계정이 표시되지 않는 경우 조직에 여러 상위/하위 계정 구조가 있기 때문일 수 있으며 이는 마이그레이션 프로세스를 두 번 이상 수행해야 함을 의미합니다.

* 일부 사용자만 새 사용자 모델로 마이그레이션하여 시작할 계획이라면 원래 사용자 모델 사용자를 관리할 관리자가 있도록 일부 원래 사용자 모델 관리자를 남겨두는 것이 좋습니다.

## 2단계: 조직 설정 [#page2]

<img
  title="user-migration-page-2.png"
  alt="User migration page 2"
  src={userMigrationPage2}
/>

a) 더 많은 구성 옵션을 허용하는 안내 설정 또는 b) 더 적은 단계의 자동 설정을 선택할 수 있습니다. 이것을 선택하는 몇 가지 팁:

* 마이그레이션하려는 사용자에 대해 SAML SSO 또는 SCIM을 사용하지 않고 마이그레이션된 모든 사용자에게 모든 계정에 대한 액세스 권한을 부여해도 된다면 자동 설정 옵션을 사용하는 것이 좋습니다. (나중에 언제든지 계정에 대한 사용자 액세스를 더 많이 분할할 수 있습니다.)
* SCIM 프로비저닝 관련: 가까운 시일 내에 SCIM 프로비저닝을 통해 New Relic 사용자를 관리할 계획이라면 SCIM이 활성화된 상태로 마이그레이션할 수 있도록 마이그레이션을 기다리는 것을 고려해야 합니다. 이는 사용자가 마이그레이션되면 특정 인증 도메인에 상주하고 해당 도메인에 사용자가 추가되면 해당 도메인에서 SCIM과 비SCIM(수동) 간에 전환할 수 없기 때문입니다.

## 4단계: 인증 도메인 설정 [#page4]

<img
  title="user-migration-page-4.png"
  alt="User migration page 4"
  src={userMigrationPage4}
/>

이 섹션은 사용자가 프로비저닝되는 방식(New Relic에 추가됨)과 인증 방식(로그인)을 제어합니다. SAML SSO 또는 SCIM 설정을 선택하면 마이그레이션 마법사를 종료하고 New Relic UI 및 ID 공급자의 다른 위치에서 구성한 다음 마법사로 돌아가야 합니다.

두 개의 인증 도메인 섹션에 대한 자세한 내용은 다음과 같습니다.

### 사용자 관리(수동 및 ID 제공자) [#managing-users]

사용자 추가 및 관리 방법은 **수동** 또는 **ID 공급자** (SCIM)를 선택할 수 있습니다. ID 공급자를 사용하여 SCIM을 통해 사용자를 프로비저닝하는 옵션은 조직에 [Pro 또는 Enterprise 버전](https://newrelic.com/pricing) 이 있는 경우에만 사용할 수 있습니다.

**ID 공급자** 를 선택하는 경우 [자동화된 사용자 관리 단계를](/docs/accounts/accounts/automated-user-management/automated-user-provisioning-single-sign#how-to) 따라야 하지만 이 프로세스의 뒷부분에서 수행할 액세스 권한 설정과 관련된 단계는 건너뜁니다. 자동화된 사용자 관리 단계를 완료하면 마이그레이션 마법사와 이 문서로 돌아갑니다.

이 단계를 완료한 후에는 최대한 빨리 사용자 마이그레이션 프로세스를 완료하는 것이 좋습니다. 이 단계를 완료하고 마법사 절차를 완료할 때까지 사용자는 동일한 로그인( [1단계의 로그인 스크린샷 참조)과 연결된 두 개의 사용자 레코드를 갖게 되거나(1단계](#page1) 의 로그인 스크린샷 참조) 볼 것으로 예상하는 자산(예: 대시보드)이 누락될 수 있습니다.

ID 공급자를 New Relic과 동기화하고 액세스 권한을 설정하기 위한 몇 가지 팁:

* Okta, Azure 또는 OneLogin에 대해 New Relic 앱을 이미 사용하고 있다면 오래된 버전을 사용하고 있을 가능성이 큽니다. 오래된 앱의 제목은 "New Relic by account"이고 최신 필수 앱의 제목은 "New Relic by organization"입니다.
* 이러한 단계를 완료하면 새 사용자 모델에서 새 사용자 레코드가 생성되고 ID 제공자 구성을 기반으로 New Relic에서 동기화됩니다. 사용자 프로비저닝을 완료한 후 새 [**사용자 관리** UI](#manage-users) 에 해당 사용자 레코드가 표시되는지 확인합니다.
* 새로운 New Relic 사용자 관리 UI에 액세스하려면 새 사용자 기록을 통해 로그인해야 합니다. 이를 위해서는 로그아웃하고 다시 로그인한 다음 이메일을 확인해야 이메일과 관련된 모든 로그인이 표시될 수 있습니다.

### 로그인 방법(수동 및 SSO) [#login-methods]

로그인 방법을 통해 해당 사용자가 로그인하는 방법을 선택할 수 있습니다. a) 이메일/비밀번호 로그인 또는 b) 싱글 사인온(SSO)을 선택할 수 있습니다. SSO는 Pro 또는 Enterprise 버전이 있는 조직에서만 사용할 수 있습니다.

SSO를 사용 중이지만 SCIM을 사용하지 않는 경우 SSO [를 설정하려면 추가 단계를](/docs/accounts/accounts-billing/new-relic-one-user-management/authentication-domains-saml-sso-scim-more/#saml) 완료해야 합니다. (이미 이전 단계의 SCIM 절차를 따랐다면 SAML SSO를 이미 설정했을 것입니다.)

SAML SSO 설정을 위한 몇 가지 팁:

* Okta, Azure 또는 OneLogin에 대해 New Relic 앱을 이미 사용하고 있다면 오래된 버전을 사용하고 있을 가능성이 큽니다. 오래된 앱의 제목은 "New Relic by account"이고 최신 필수 앱의 제목은 "New Relic by organization"입니다.
* 새로운 New Relic 사용자 관리 UI에 액세스하려면 새 사용자 기록을 통해 로그인했는지 확인해야 합니다. 해당 이메일과 관련된 모든 로그인을 보려면 로그아웃하고 다시 로그인한 다음 이메일을 확인해야 할 수 있습니다.
* SSO 설정 절차를 완료한 다음 마이그레이션 마법사로 돌아와 마이그레이션 프로세스를 계속할 수 있습니다.
* 인증 방법을 두 개 이상 선택하는 경우 새 [인증 도메인](/docs/accounts/accounts-billing/new-relic-one-user-management/authentication-domains-saml-sso-scim-more/#auth-domain-definition) 을 추가해야 합니다.

New Relic 사용자를 추가하고 관리하는 방법에는 두 가지가 있습니다. 지침 및 팁에 사용할 방법을 선택하십시오.

<CollapserGroup>
  <Collapser

    **권장 사항** : 사용자 가져오기를 선택하기 전에 기존 원래 사용자 모델 사용자의 전체 목록을 다운로드하십시오. 이것은 유용한 리소스가 될 것이며 필요한 경우 백업 역할을 합니다. 또한 이 목록을 사용하여 어떤 사용자가 어떤 계정에 액세스할 수 있는지 파악하고 이후 단계에서 그룹을 만들고 액세스 권한을 부여할 수 있습니다.

    이 마이그레이션 중에 사용자를 수동으로 추가하는 작업을 훨씬 쉽게 하기 위해 사용자를 대량으로 업로드하는 기능을 제공합니다. 원래 사용자 모델 사용자를 다운로드한 후 모든 사용자 또는 일부만 업로드할 수 있습니다. 이 사용자 목록을 검토하고 제공된 목록과 [원래 사용자 모델에 대한 New Relic UI에서 이전하지 않으려는 사용자(예: 조직에서 더 이상 근무하지 않는 사용자)를 제거하는 것이 좋습니다.](/docs/accounts/original-accounts-billing/original-users-roles/users-roles-original-user-model#deleting)

    사용자 목록을 다시 업로드하고 이 단계를 완료하면 New Relic One 사용자 모델에 대한 사용자 레코드가 생성됩니다. 이후 단계에서 이러한 사용자의 자산을 전환할 수 있습니다.

    이 단계를 완료하고 새 사용자 레코드를 만든 후에는 나머지 마이그레이션 프로세스를 최대한 빨리 완료하는 것이 좋습니다. 자산을 마이그레이션하고 원래 사용자 레코드를 삭제하는 단계를 완료하지 않으면 사용자에게 동일한 로그인과 연결된 두 개의 사용자 레코드가 있거나( [1단계의 로그인 스크린샷](#page1) 참조) 볼 것으로 예상되는 자산(예: 대시보드).

    몇 가지 팁:

    * 비밀번호를 재설정할 필요가 없습니다. 사용자 로그인에 사용자 이름/비밀번호를 사용하는 경우 생성된 새 사용자 레코드에 대해 동일한 로그인 자격 증명을 갖게 됩니다. 사용자에게 이메일 확인 보류 상태(확인 보류 중)가 있는 경우 해당 상태도 전환됩니다.
    * 새 사용자의 이메일 주소가 대소문자 일치를 포함하여 원래 사용자 기록 이메일 주소와 일치하는지 확인합니다. 이메일 주소를 키 값으로 사용하여 사용자를 일치시키고 이후 단계에서 사용자 관련 자산을 전환합니다.

## 6단계: 설정에 액세스 [#access-settings]

<img
  title="user-migration-page-6.png"
  alt="User migration page 6"
  src={userMigrationPage6}
/>

이 단계는 사용자 그룹이 액세스할 수 있는 역할과 계정을 설정하는 것입니다. 특정 계정 및 역할에 대한 사용자 액세스를 매핑해야 하는 경우 이 단계에서 그룹 및 액세스 권한을 설정하는 것이 좋습니다.

SCIM 프로비저닝을 사용하여 ID 제공자로부터 사용자 및 그룹을 가져오는 경우 New Relic에서 사용자 또는 그룹을 편집할 수 없습니다. 모든 사용자 및 그룹 변경 사항은 ID 제공자 측에서 처리됩니다.

그룹에 액세스 권한을 부여하려는 각 계정 및 역할에 대한 액세스 권한을 생성해야 합니다. 액세스 권한을 이해하는 데 도움이 되는 리소스:

* 액세스 권한이 작동하는 방식에 대한 기본 사항은 [액세스](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts/#understand-concepts) 권한을 참조하세요.
* 액세스 권한을 생성하는 방법에 대한 자습서는 [사용자 관리 자습서](/docs/accounts/accounts-billing/new-relic-one-user-management/tutorial-add-new-user-groups-roles-new-relic-one-user-model/#group-access) 를 참조하십시오.
* 액세스 권한 부여 계획에 대한 도움말은 [그룹 및 액세스 권한 부여 계획 스프레드시트](https://docs.google.com/spreadsheets/d/1FnguDXRUX9FGY14oV4Gx6O08v4vNC2Pv0GGCsU7Pxuw/edit?usp=sharing) 를 참조하십시오.

## 7단계: 사용자 자산 마이그레이션 [#migrate-assets]

<img
  title="user-migration-page-7.png"
  alt="User migration page 7"
  src={userMigrationPage7}
/>

이 단계가 완료되면 사용자의 개인 자산이 새 사용자 레코드로 마이그레이션되고 원래 사용자 레코드가 삭제됩니다. 현재 New Relic에 로그인한 사용자의 경우 이 단계를 완료하면 로그아웃하거나 현재 브라우저 세션이 만료될 때까지 현재 New Relic 세션이 중단되지 않습니다.

마이그레이션되는 사용자 자산은 다음과 같습니다.

* 대시보드
* 즐겨찾기
* 주간 이메일 설정
* 이메일 수신 동의/선택 해제 기본 설정
* 사용자별 [사용자 키](/docs/apis/intro-apis/new-relic-api-keys/#user-api-key)
* 새로운 Relic One 앱 [NerdStorage 데이터](https://developer.newrelic.com/explore-docs/nerdstorage)

사용자가 New Relic을 사용하는 여러 조직에 액세스할 수 있는 경우(예: 해당 사용자가 계약자인 경우) 해당 조직이 모두 사용자를 마이그레이션할 때까지 원래 사용자 모델 레코드가 완전히 삭제되지 않습니다. 이러한 사용자는 원래 사용자 레코드와 하나 이상의 새 사용자 레코드를 모두 갖게 되며 이 경우 로그인할 때 표시됩니다( [1페이지 섹션의 로그인 스크린샷 참조](#page1) ).

## 문제점 해결 [#troubleshooting]

마이그레이션 후 몇 가지 일반적인 문제:

* 관리자 수준 역할이 할당되었지만 New Relic 플랫폼 기능에 액세스하려고 할 때 오류 메시지가 표시되는 경우 조직 범위 역할( **조직 관리자** 및/또는 **인증 도메인 관리자** )이 할당되었지만 계정이 없기 때문일 수 있습니다. 범위가 지정된 역할. 특정 계정의 New Relic 기능에 액세스하려면 하나 이상의 계정 범위 역할(예: **모든 제품 관리자** 또는 사용자 지정 역할)이 필요합니다.

* 마이그레이션을 완료했거나 마이그레이션 중인데도 원래 사용자 관리 UI( **계정 설정** 탭을 통해 액세스한 UI)가 계속 표시되는 경우 원래 사용자 모델 레코드에 여전히 로그인되어 있기 때문일 수 있습니다. 이에 대한 몇 가지 해결책:

  * New Relic에서 로그아웃했다가 다시 로그인하여 **이메일 확인** 옵션을 선택합니다. 이메일을 확인했으면 "Original New Relic account"가 아닌 "Organization"이라는 로그인 옵션을 선택하십시오.
  * 그래도 문제가 지속되면 브라우저 캐시를 지우고 다시 로그인해 보세요.

## 완료한 후 [#manage-users]

사용자가 새 사용자 모델로 마이그레이션되면 [계정 드롭다운](/docs/using-new-relic/welcome-new-relic/get-started/glossary/#account-dropdown) , **관리** 를 차례로 클릭하고 다음 UI 페이지를 사용하여 사용자를 찾고 관리할 수 있습니다.

* **사용자 관리** : 사용자를 보고 추가하고 유형(기본 또는 전체) 변경, 그룹 변경, 사용자 업그레이드 요청 승인에 사용합니다.
* **액세스 관리** : 액세스 권한 부여(그룹에 역할 및 계정에 대한 액세스 권한 부여)를 생성하고 인증 도메인(SAML SSO 설정 및 SCIM 설정 등)을 구성하는 데 사용합니다.

액세스 권한 부여 계획에 대한 몇 가지 팁은 액세스 권한 부여에 대한 [팁 을](#before-beginning) 참조하십시오.

이러한 도구 및 개념에 대한 자세한 내용은[사용자 관리 문서](/docs/accounts/accounts-billing/new-relic-one-user-management/introduction-managing-users) 를 참조하십시오.
