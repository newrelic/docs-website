[원래 사용자 모델](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models) 의 사용자: 이 모델에서 사용자 역할 및 권한이 작동하는 방식, 사용자를 추가하고 관리하는 방법.

## 요구 사항 [#who-is-doc-for]

문서의 이 섹션에서는 [원래 사용자 모델](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models) 에서 사용자를 관리하는 방법을 보여줍니다. [**사용자 및 역할** UI](#view-access) 에서 사용자를 볼 수 있는 경우 해당 사용자는 원래 사용자 모델에 있는 것입니다.

New Relic 사용자는 자신의 로그인 정보를 다른 사람과 공유할 수 없습니다. New Relic 사용자는 최대 3개의 동시 활성 세션 또는 주어진 시간에 3개의 고유한 IP 주소를 사용할 수 있습니다.

## 새로운 사용자 모델에 대한 업데이트 [#migrate]

2020년 7월에 [New Relic One 사용자](/docs/accounts/original-accounts-billing/original-product-based-pricing/overview-user-models) 모델이라는 새로운 사용자 모델을 출시했습니다. 이 모델은 [조직과 사용자를 관리하는 방법 측면에서 많은 이점](https://newrelic.com/blog/how-to-relic/user-management-and-access-security) 을 제공합니다. 처음에는 신규 가입만 가능했지만 시간이 지남에 따라 이전 고객을 새 모델로 마이그레이션했습니다. 일부 오래된 고객은 [사용자 마이그레이션을 자체적으로 처리할 수](/docs/accounts/original-accounts-billing/original-users-roles/user-migration/) 있습니다. 원래 모델이 완전히 사용되지 않을 때까지 사용자를 새 모델로 마이그레이션하는 작업을 계속할 것입니다.

새로운 사용자 모델의 영향 중 하나는 이제 사용자가 동일한 이메일과 연결된 여러 로그인을 가질 수 있다는 것입니다. 예를 들어, 여러 조직에 대한 액세스 권한이 있는 사용자(예: 계약자)는 사용자 레코드를 한 조직의 새 사용자 모델로 업데이트하여 원래 로그인 방법 및 레코드와 New Relic One 사용자 모델 레코드를 갖게 될 수 있습니다. 이로 인해 사용자가 New Relic에 로그인되어 원하는 계정을 찾지 못할 수 있습니다. 이에 대한 자세한 내용은 [액세스에 영향을 미치는 요소](/docs/accounts/accounts-billing/general-account-settings/factors-affecting-access-features-data/#account-access) 를 참조하십시오.

![여러 계정](./images/login-multiple-accounts-found.png "로그인-다중-계정-found.png")<figcaption>사용자의 이메일이 둘 이상의 로그인과 연결된 경우 로그인할 때 "여러 계정을 찾았습니다"라는 메모가 표시됩니다.</figcaption>

## UI에서 사용자 보기 및 관리 [#view-access]

**사용자 및 역할** UI는 [원래 사용자 모델](#who-is-doc-for) 에서 사용자를 관리하는 데 사용됩니다. 이 UI를 찾으려면[계정 드롭다운](/docs/accounts-partnerships/education/getting-started-new-relic/glossary#account-dropdown) , **계정 설정** , **사용자 및 역할** 을 차례로 클릭합니다.

New Relic One 가격 책정 모델을 사용 중인 경우 몇 가지 [중요한 고려 사항](#full-user-count) 을 참조하십시오.

[New Relic REST API](/docs/apis/rest-api-v2/account-examples-v2/listing-users-your-account) 를 사용하여 계정의 사용자 목록을 가져올 수도 있습니다.

다음은 몇 가지 일반적인 사용자 관리 작업 및 시나리오에 대한 지침 및 고려 사항입니다.

    6. **사용자 추가를** 선택합니다.

       새 사용자는 New Relic에서 자동으로 이메일 알림을 받게 됩니다.

       <Callout variant="important">
         New Relic은 사용자당 최대 1,000개의 계정을 권장합니다. 추가 계정으로 인해 일부 New Relic 기능에 대한 액세스가 제한될 수 있습니다.

  <Collapser
    id="full-user-count"
    title="청구 가능한 사용자 관리"
  >
    [New Relic One 가격 책정 모델](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing/) 을 사용 중인 경우 사용자의 사용자 유형은 청구서의 한 요소입니다.

    조직에 여러 계정이 있는 경우 원래 사용자 모델에서 청구 가능한 사용자를 관리할 때 다음과 같은 몇 가지 중요한 고려 사항이 있습니다.

    * 단일 UI에서 조직의 모든 사용자를 볼 수는 없습니다. 모든 사용자를 검토하고 편집하는 것이 목표인 경우(예: 청구 목적으로 [사용자 유형](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type) 편집) 각 개별 계정의 **사용자 및 역할** UI로 이동해야 합니다. (사용자를 보다 쉽게 관리할 수 있는 기능은 [최신 사용자 모델](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models) 의 이점 중 하나입니다.)
    * 위의 점 때문에 **사용량 보기** UI에 표시되는 청구 가능한 사용자 수가 특정 계정의 **사용자 및 역할** UI로 이동할 때 표시되는 사용자 수와 일치하지 않을 수 있습니다.
    * **계정 설정** UI에서 해당 계정이 "활성" 또는 "비활성"(보고 데이터 아님)으로 표시되는지 여부에 관계없이 조직의 모든 계정에서 청구 가능한 사용자에 대해 비용을 청구할 수 있습니다.

    사용자 유형 편집에 대한 팁은 사용자 유형 [관리](#update-user-type) 를 참조하십시오.
  </Collapser>

    title="사용자 유형 및 사용자 업그레이드 관리"

    사용자 유형 관리를 위한 몇 가지 중요한 팁:

    * [New Relic One 가격 책정 모델](/docs/accounts/original-accounts-billing/original-product-based-pricing/overview-pricing-models) 을 사용하는 경우 사용자의 [사용자 유형](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type) 이 청구 요소입니다.
    * 사용자를 상위 사용자 유형으로 업그레이드하면(UI 또는 업그레이드 요청 이메일을 통해) 조직의 모든 계정에서 해당 사용자 유형이 변경됩니다.

    사용자의 사용자 유형을 업데이트하려면:

    1. 이동: **[계정 드롭다운](/docs/accounts-partnerships/education/getting-started-new-relic/glossary#account-dropdown) > 계정 설정 > 사용자 및 역할 > 사용자** .
    2. 사용자를 선택하고 유형을 수정하거나 [여러 사용자의 유형을 일괄 업데이트합니다](#bulk) . 조직에 여러 계정이 있는 경우 단일 UI 위치에서 모든 사용자를 볼 수는 없지만 각 계정의 **사용자 및 역할** UI로 이동해야 합니다.

    업그레이드를 요청한 사용자를 보고 승인하려면:

    1. 이동: **[계정 드롭다운](/docs/accounts-partnerships/education/getting-started-new-relic/glossary#account-dropdown) > 계정 설정 > 사용자 및 역할** .
    2. **업그레이드 요청** 을 클릭합니다. 조직에 여러 계정이 있는 경우 단일 UI 위치에서 모든 사용자를 볼 수는 없지만 각 계정의 **사용자 및 역할** UI로 이동해야 합니다.

    사용자 업그레이드 방법을 제어하려면 **사용자 및 역할** UI에서 **액세스 요청** 을 선택할 수 있습니다. 두 가지 옵션이 있습니다.

    * **자동 승인** : 이 옵션을 사용하면 사용자 유형을 자동으로 즉시 업그레이드할 수 있습니다. 이 옵션을 사용하면 사용자가 문제를 보다 쉽게 해결할 수 있습니다.
    * **검토 필요** : 이 옵션을 사용하면 사용자가 업그레이드를 요청하고 관리자가 먼저 업그레이드해야 할 때 관리자에게 알림 이메일이 전송됩니다. 알림 이메일이나 **사용자 및 역할** UI의 사용자 기록에서 승인할 수 있습니다.

    사용자 유형에 대한 자세한 내용은 [사용자](#user-type) 유형을 참조하십시오.

    3. 현재 계정에 대한 관리자 역할이 있는 사람을 선택하십시오.

    4. 변경 사항을 적용하려면 페이지를 새로고침하세요.

    6. **기본 역할** 열을 보고 계정 소유자를 찾습니다.

       그만큼 <Icon name="fe-repeat"/>**소유자 변경** 버튼은 현재 계정 소유자에게만 표시됩니다. 현재 소유자가 역할을 변경할 수 없는 경우(예: 해당 사람이 더 이상 조직에 없는 경우) New Relic의 계정 담당자에게 문의하거나 [support.newrelic.com](https://support.newrelic.com) 에서 지원을 받으십시오.

## 사용자 유형 [#user-type]

사용자 유형은 [New Relic One 가격 책정 모델](/docs/accounts/original-accounts-billing/original-product-based-pricing/overview-pricing-models) 을 사용하는 조직에만 적용되는 청구 요소입니다.

사용자의 사용자 유형은 최대 허용된 New Relic 기능 집합을 제어합니다. 실제로 사용자에게는 다양한 방식으로 기능을 제한하는 [역할](#roles) 이 할당되는 경우가 많지만 사용자 유형은 이론적인 기능의 최대 집합을 나타냅니다. 자세한 내용은 [사용자 유형](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type) 을 참조하십시오.

조직의 사용자가 다른 계정에서 다른 사용자 유형으로 설정된 경우 해당 사용자는 가장 높은 사용자 유형이 무엇이든 상관없이 간주됩니다.

사용자 유형을 편집하는 방법은 [사용자 유형 관리](#update-user-type) 를 참조하십시오.

## 추가 역할 [#add-on]

![원래 사용자 모델 애드온 역할 UI](./images/original-user-model-roles-ui.png "원래 사용자 모델 애드온 역할 UI")<figcaption>경보 관리자의 기능 UI 보기</figcaption>

추가 기능을 사용하면 사용자에게 보다 구체적이고 세분화된 액세스 권한을 부여할 수 있습니다. 사용자 또는 제한된 사용자에게 추가 기능 관리자 액세스 권한을 부여하면 해당 기능 범주에 대한 **관리** 기능과 동등한 권한이 부여됩니다. 그들은 계속해서 다른 모든 New Relic 제품에 대해 사용자 또는 제한된 사용자 기능을 갖습니다. 예를 들어 회사의 소프트웨어 엔지니어를 사용자로 만들고 **APM 관리자** 역할을 할당하여 APM 기능을 제공할 수 있습니다.

상위 계정의 개인은 자동으로 상위 계정의 모든 하위 계정에 대해 동일한 수준의 액세스 권한을 갖습니다.

다음과 같은 추가 기능 관리자 역할이 있습니다.

* 알림 관리자
* APM 매니저
* 응용 인텔리전스 관리자
* 브라우저 관리자
* 데이터 보존 관리자
* 사건 정보 관리자
* 인시던트 워크플로 관리자
* 인프라 관리자
* 인사이트 관리자
* 로그 관리자
* 모바일 매니저
* 너드팩 매니저
* 합성 관리자
* 워크로드 관리자

역할의 기능을 이해하려면 **[계정 드롭다운](/docs/glossary/glossary/#account-dropdown) > 계정 설정 > 사용자 및 역할 > 역할** 로 이동하여 특정 역할을 선택합니다. 특정 기능 위로 마우스를 가져가면 자세히 알아볼 수 있습니다.

사용자 지정 기능이 있는 사용자 지정 역할을 만들려면 **[계정 드롭다운](/docs/glossary/glossary/#account-dropdown) > 계정 설정 > 사용자 및 역할 > 역할** 로 이동하고 **새 사용자 지정 추가 기능** 역할을 클릭합니다.

## 대량 사용자 관리 [#bulk]

**대량 사용자 작업** 기능을 사용하면 한 번에 여러 사용자를 추가, 업데이트 또는 삭제할 수 있습니다. 다음과 같은 경우에 도움이 될 수 있습니다.

* 여러 명의 새 직원이 시작될 때 역할 추가
* 여러 직원이 떠날 때 역할 삭제
* 여러 직원에게 관리자 역할 부여

### 사용자 일괄 업데이트 [#update]

대량 사용자 작업을 수행하기 위한 몇 가지 중요한 규칙 및 권장 사항:

* 자신의 역할이나 소유자 역할은 업데이트할 수 없습니다.
* 기존 사용자의 이메일 주소나 이름은 수정할 수 없습니다.
* 기존 사용자를 삭제했다가 다시 추가하는 방식으로 편집하는 것은 피해야 합니다. 이는 의도하지 않은 결과를 초래할 수 있기 때문입니다(예: 원래 사용자와 연결된 API 키가 손실됨).

새 사용자 역할을 추가하려면 기존 사용자 역할을 업데이트하거나 [원래 사용자 모델](/docs/accounts/original-accounts-billing/original-product-based-pricing/overview-user-models) 의 사용자에 대한 사용자 역할을 삭제하십시오.

1. **[계정 드롭다운](/docs/accounts-partnerships/education/getting-started-new-relic/glossary#account-dropdown) > 계정 설정 > 사용자 및 역할** 로 이동하고 URL 끝에 `/bulk_actions` 를 추가합니다.

   예시 URL:

   ```
   https://account.newrelic.com/accounts/<var>123456789</var>/users/bulk_actions
   ```

2. **백업 CSV 파일** 을 다운로드합니다. 백업 파일을 다운로드하면 변경 사항이 적용되기 전에 계정의 사용자 기록이 유지되고 실수로 제거될 수 있는 사용자를 쉽게 다시 추가할 수 있습니다.

3. **사용자** 의 CSV 또는 **CSV 템플릿** 을 다운로드합니다. 각 일괄 작업(추가, 업데이트 또는 삭제)에는 자체 CSV 파일이 필요합니다. New Relic은 계정 번호, 날짜 및 수행 중인 일괄 작업과 함께 파일을 저장할 것을 권장합니다. 예: `account_123456789_delete_users_2018-06-29`

4. 선택한 일괄 작업을 적용할 역할의 사용자로만 시트를 채웁니다. 역할을 변경하지 않으려는 사용자를 스프레드시트에서 제거합니다.

    title="백업 CSV 파일이 있는 경우"
  >
    백업 CSV 파일이 저장되어 있는 경우:

    1. 백업 CSV 파일을 엽니다.
    2. 역할이 수정될 사용자로 백업 CSV 파일을 채웁니다.
    3. 새 CSV 파일에 대한 CSV 작업 선택: **추가** , **업데이트** 또는 **삭제**
    4. 새 CSV를 업로드하고 **변경 사항 저장** 을 선택합니다.
  </Collapser>
