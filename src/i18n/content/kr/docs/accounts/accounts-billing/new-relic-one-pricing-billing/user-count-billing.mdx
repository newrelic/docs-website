[가격 책정 모델](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing) 에서 청구 가능한 사용자 수는 청구 요소입니다.

## 비용에 대해 알아보기 [#cost]

아직 New Relic 고객이 아니고 비용에 대해 궁금한 경우 먼저 [기본 가격 페이지를](https://newrelic.com/pricing)참조하십시오.

기존 New Relic 고객이고 청구를 이해하려면 [청구 UI를](/docs/accounts/accounts-billing/general-account-settings/introduction-account-settings)참조하십시오. 조직에 여러 계정이 있는 경우 조직의 모든 사용자를 보려면 기본 보고 계정(일반적으로 조직에서 생성된 첫 번째 계정)에 있어야 합니다.

## 청구 가능한 사용자는 누구입니까? [#billable-users]

청구 가능 사용자는 [사용자 유형](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type) 이 코어 사용자 또는 전체 플랫폼 사용자인 사용자입니다. 기본 사용자는 무료입니다.

사용자가 청구할 수 있는 이유에 대한 기타 세부정보:

* [최신 사용자 모델](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models)사용자의 경우: 사용자는 New Relic 조직에 청구 가능 사용자로 추가되는 순간 청구 가능합니다. 청구 가능 사용자가 New Relic에 로그인한 적이 없고 UI에 `Pending invite` 태그가 있더라도 여전히 청구 가능합니다.
* [기본 사용자 청구 버전](#pricing-versions)의 경우: 청구 가능한 사용자에 대한 월별 요금이 있으며 비례 배분되지 않습니다. 즉, 사용자가 한 달 내내 청구 가능한 사용자 유형이든 해당 월의 한 시간 동안만 비용이 청구됩니다. [사용자 계산에 대해 자세히 알아보세요.](#user-count)

## 사용자 관리 [#manage]

사용자 관리 방법은 [사용자 관리를](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-ui-and-tasks)참조하십시오.

## 사용자 청구 버전 [#pricing-versions]

사용자 요금 청구 방법에는 두 가지 버전이 있습니다.

* **기본 사용자 청구 버전입니다.** 이것은 최신 버전입니다. 한 달에 청구 가능한 총 사용자 수를 기준으로 요금이 청구됩니다. 이것은 새로 생성된 모든 New Relic 조직이 가지고 있는 버전이며 귀하의 조직이 가지고 있을 가능성이 있는 버전입니다.
* **원래 사용자 청구 버전입니다.** 이것은 일부 조직에서 여전히 가지고 있는 이전 버전입니다.

조직의 버전에 따라 사용자 청구 방식과 일부 사용자 다운그레이드 규칙이 결정됩니다. 이러한 규칙은 아래에서 자세히 설명합니다.

사용 중인 버전이 확실하지 않습니까? 다음은 몇 가지 팁입니다.

* [billing UI](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing#usage-ui) \[결제 UI] 의 **Plan summary** \[요금제 요약] 페이지에 `monthly provisioned users`표시되면 원래 버전을 사용 중인 것입니다.
* 인보이스에 `New Relic One - Pro Users` 또는 `New Relic One - Enterprise Users`표시되면 원래 버전을 사용 중인 것입니다. 여기에 사용자 유형(예: `New Relic - Pro Full Platform Users`)이 포함되어 있으면 최신 버전을 사용하고 있는 것입니다.

## 기본 사용자 결제 버전 [#primary-version]

[사용량 UI](/docs/accounts/accounts-billing/general-account-settings/introduction-account-settings/#pricing) 를 사용하여 청구 가능한 사용자 수에 대한 개요를 확인할 수 있습니다. UI가 제공하는 것보다 더 자세한 정보가 필요한 경우 [사용 관련 NRQL 쿼리](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/usage-queries-alerts) 를 실행할 수도 있습니다.

  >
    <Callout variant="important">
      이러한 규칙은 [기본 사용자 결제 버전](#pricing-versions) 의 조직에 적용됩니다.
    </Callout>

    청구 가능한 사용자에 대한 청구는 역월별로 이루어집니다. 한 달 동안 조직의 청구 가능 사용자 수를 결정하기 위해 **billable user type** \[청구 가능 사용자 유형이] 전체 플랫폼 사용자 또는 핵심 사용자인 해당 월의 사용자 수를 계산합니다. 사용자의 **청구 가능 사용자 유형은** 한 달 동안 사용자가 설정된 가장 높은 사용자 유형으로 정의됩니다. UTC 시간대를 사용하여 한 달의 시작과 끝을 정의합니다.

    이것이 실제로 어떻게 작동하는지에 대한 예: 사용자가 한 달 중 어느 시점에든 전체 플랫폼 사용자로 설정되면 해당 월에 청구 가능한 사용자 유형은 `full platform user`이며 다운그레이드하더라도 변경되지 않습니다. 그 달 말에. 이는 해당 사용자가 잠시 동안만 전체 플랫폼 사용자로 변경된 경우에도 마찬가지입니다.

    청구 가능한 사용자를 추가하거나 사용자 유형을 변경할 계획이라면 이러한 규칙을 염두에 두어야 합니다. 몇 가지 팁:

    * 청구 가능한 사용자를 추가하거나 사용자를 업그레이드하려는 경우 월초에 수행하는 것이 좋습니다.
    * 사용자를 다운그레이드하려면 월말에 다운그레이드하는 것이 좋습니다.
    * UTC 시간을 사용하여 한 달의 시작과 끝을 결정합니다. 즉, 예를 들어 태평양 표준시를 기준으로 월말에 사용자를 다운그레이드하려면 태평양 표준시 오후 5시까지 변경해야 합니다.

    동일한 이메일 주소를 사용하는 조직 내 사용자는 한 명의 사용자로만 계산됩니다. 자세한 내용은 [사용자 추적을](/docs/accounts/accounts-billing/account-setup/multiple-logins-found#user-records) 참조하세요.

    [최신 사용자 모델](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models) 의 사용자: 관리자가 청구 가능한 사용자로 추가한 사용자는 청구할 수 있습니다. 청구 가능한 사용자가 New Relic에 로그인한 적이 없고 UI에 `Pending invite` 태그가 있더라도 여전히 청구 가능합니다. ( [원래 사용자 모델](/docs/accounts/original-accounts-billing/original-users-roles/users-roles-original-user-model) 의 사용자는 `Pending invite` 명의 사용자에게 청구할 수 없습니다.)

    사용자 추가 및 초대에 대한 자세한 내용은 사용자 [추가](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-ui-and-tasks#add-users) 를 참조하십시오.

    청구 가능한 사용자 비용은 조직의 [가격 책정 에디션](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing/#editions) (Standard, Pro 또는 Enterprise) 또는 New Relic과의 사용자 정의 거래에 따라 다릅니다.

    New Relic 조직에 처음 요금이 청구되기 시작하면 청구 가능한 사용자 수는 시작한 달의 시기를 기준으로 일할 계산됩니다. 조직에서 구독을 취소하면 지난 달에 대해서도 일할 계산이 적용됩니다.
  </Collapser>

  <Collapser
    id="user-downgrade-rules"
    title="사용자 다운그레이드 규칙"
  >
    <Callout variant="important">
      이러한 규칙은 [기본 사용자 결제 버전](#pricing-versions) 의 조직에 적용됩니다.
    </Callout>

    전체 플랫폼 사용자를 다운그레이드할 수 있는 횟수와 관련된 규칙은 [사용량 계획](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing#usage-plans) 이 종량제인지 약정 계약인지에 따라 다릅니다.

    ### 종량제 다운그레이드 규칙

    [종량제 사용 계획](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing#usage-plans)의 경우 사용자 다운그레이드를 제한하는 규칙이 없습니다. 그러나 [청구 계산은](#user-count) 사용자를 업그레이드하거나 다운그레이드할 시기를 결정하는 데 영향을 미칠 수 있습니다.

    ### 약정 계약 다운그레이드 규칙

    사용자를 업그레이드하거나 다운그레이드하기 전에 [사용자 결제 방식을](#user-count)이해해야 합니다.

    [약정 계약이](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing/#usage-plans)있는 조직의 경우 전체 플랫폼 사용자 다운그레이드 및 업그레이드에 적용되는 규칙은 다음과 같습니다.

    **정식 플랫폼 사용자가 다음 달에 더 낮은 사용자 유형으로 다운그레이드된 후 다음 달에 정식 플랫폼 사용자로 돌아가고 이 다운그레이드/업그레이드 주기가 약정 연도에 두 번 발생하는 경우 해당 사용자에게 전체 요금이 청구됩니다. 사용자 유형에 대한 편집과 관계없이 해당 연도의 나머지 기간 동안 플랫폼 사용자.** [사용자 청구가 작동하는 방식](#user-count)으로 인해 한 달 내에 발생하는 사용자 유형 변경을 관리하는 규칙은 없습니다. 한 달 동안 사용자의 청구 가능한 사용자 유형은 해당 월에 설정된 가장 높은 사용자 유형입니다.

    다음은 이 규칙의 맥락에서 사용되는 용어에 대한 세부 정보입니다.

    * **다운그레이드는** 한 달 동안 전체 플랫폼 사용자로 청구되는 것에서 이후 달에 더 낮은 사용자 유형(코어 또는 기본)으로 청구되거나 삭제되는 것을 의미합니다.
    * **업그레이드는** 한 달 동안 핵심 사용자, 기본 사용자(또는 이전에 삭제된 사용자)로 청구되는 것에서 이후 달에 전체 플랫폼 사용자로 청구되는 것으로 바뀌는 것을 의미합니다.
    * **계약 연도는** 계약 시작 시점 또는 해당 시점의 기념일에 시작됩니다. 조직에서 다른 요금제로 시작하여 약정 계약으로 전환한 경우 선택한 날짜부터 계약 갱신 날짜 또는 약정 기간의 연간 기념일 중 더 빠른 날짜까지 사용자 유형 다운그레이드 규칙이 적용됩니다.

    <Callout variant="tip">
      **전체 플랫폼 사용자 다운그레이드/업그레이드에 적용되는 규칙이 있는 이유는 무엇입니까?**

      전체 플랫폼 사용자 유형은 장기적인 분류를 목적으로 합니다. 우리는 약정 계약을 생성할 때 청구 가능한 사용자의 추정치를 사용하며 이러한 규칙을 통해 이러한 계약 제한을 일관되게 구현하고 있습니다.
    </Callout>

    #### 다운그레이드 한도의 예

    계약 연도가 3월에 시작한다고 가정해 보겠습니다. 그리고 사용자가 다음 계정 유형을 매월 순환한다고 가정해 보겠습니다.

    1. 사용자는 3월에 전체 플랫폼 사용자로 청구됩니다.
    2. 사용자는 4월에 기본 사용자로 청구됩니다(**다운그레이드**).
    3. 사용자는 6월에 전체 플랫폼 사용자로 청구됩니다(**업그레이드**).
    4. 사용자는 7월에 기본 사용자로 청구됩니다(**다운그레이드**).

    해당 사용자가 다음 달에 전체 플랫폼 사용자로 돌아오면 이후 기본 사용자 또는 핵심 사용자로 다시 다운그레이드되더라도 해당 계약 연도의 남은 기간 동안 전체 플랫폼 사용자로 요금이 청구됩니다.
  </Collapser>
</CollapserGroup>

## 원래 사용자 결제 버전 [#original-version]

[원래 사용자 청구 버전](#pricing-versions) 에 대한 세부 정보는 다음과 같습니다.

<CollapserGroup>
  <Collapser
    id="original-user-count"
    title="청구 가능한 사용자 계산 방법"
  >
    다음 규칙은 원래 [사용자 결제 버전](#pricing-versions) 의 조직에만 적용됩니다.

    * **청구 가능한 사용자를 결정하는 방법입니다.** 역월의 경우 해당 월의 청구 가능한 사용자 수 계산을 기준으로 조직에 요금이 청구됩니다.
    * **첫 번째 달과 마지막 달에 비례 배분합니다.** 청구 가능한 사용자 수는 New Relic 조직이 구독을 시작하는 시점 또는 사용자가 월간 프로비저닝 사용자가 되는 시점(월간 프로비저닝 사용자로 추가되거나 전환됨)을 기준으로 비례 배분됩니다.
    * **사용자는 프로비저닝될 때 청구 가능합니다.** 사용자는 New Relic에서 청구 가능한 사용자 유형으로 설정되는 순간 청구 가능한 것으로 간주됩니다. 이는 해당 사용자가 New Relic에 로그인했거나 사용한 적이 있는지 여부에 관계없이 적용됩니다.
    * **이메일 주소를 기준으로 한 사용자 수입니다.** 이에 대한 자세한 내용은 [사용자 추적을](/docs/accounts/accounts-billing/account-setup/multiple-logins-found#user-records) 참조하세요.
    * **원래 사용자 모델에 대한 경고입니다.** 조직에 [원래 사용자 모델](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models) 의 사용자가 있는 경우: 사용자가 조직의 한 계정에서는 기본 사용자로 설정되고 다른 계정에서는 청구 가능 사용자로 설정된 경우 청구 가능 사용자 상태가 우선합니다.
  </Collapser>
