조직 내에서 특정 데이터에 누가 액세스하는지 제어하는 것은 보안, 규정 준수 및 데이터 거버넌스에 필수적입니다. 뉴렐릭의 [역할 기반 액세스 제어는](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts) 사용자가 액세스할 수 있는 기능을 관리하는 반면, <DNT>**data access control**</DNT> (는) 사용자가 볼 수 있는 특정 데이터를 관리합니다.

현재 로그 데이터에 대한 데이터 액세스 제어를 사용할 수 있어 특정 [로그 파티션](/docs/logs/ui-data/data-partitions/) 에 대한 사용자 액세스를 제한할 수 있습니다.

## 작동 원리 [#how-it-works]

대규모 조직에서는 다양한 팀, 서비스, 환경에 걸쳐 엄청난 양의 로그를 생성합니다. 보안팀은 개인 식별 정보(PII)를 포함할 수 있는 중요한 보안 로그를 모니터링해야 하는 반면, 개발팀은 해당 마이크로서비스와 관련된 로그만 확인하면 됩니다. <DNT>**Data access control**</DNT> 사용하면 각 팀이 액세스할 수 있는 로그 파티션을 정확하게 정의하는 정책을 만들 수 있습니다.

특정 로그 파티션에 대한 액세스를 제한하는 데이터 액세스 정책을 만들고 액세스 권한을 부여하여 사용자 그룹에 할당합니다. 할당된 정책이 있는 그룹의 사용자는 해당 정책에서 허용한 로그 파티션에만 액세스할 수 있습니다.

모든 접근을 거부하는 방식으로 시작한 다음, 필요한 파티션에만 접근 권한을 부여하는 허용 정책을 추가하는 것이 좋습니다. 이는 최소 권한 원칙을 따르며 사용자가 필요한 데이터에만 액세스할 수 있도록 보장합니다.

주요 개념:

* **데이터 액세스 정책**: 허용 목록이나 거부 목록을 사용하여 사용자가 액세스할 수 있는 로그 파티션을 정의합니다.
* **정책 할당**: 정책은 <DNT>*Access Management*\*</DNT> UI를 통해 그룹에 할당됩니다.
* **와일드카드 매칭**: `%` 와일드카드로 사용하여 여러 파티션을 매칭합니다(예: `log_prod%` 은 &quot;log\_prod&quot;로 시작하는 모든 파티션과 매칭됩니다).
* **여러 정책**: 사용자가 서로 다른 정책을 가진 여러 그룹에 속해 있는 경우 가장 허용적인 액세스 권한이 적용됩니다.

<Callout variant="important">
  액세스가 제한된 사용자는 [기록 내보내기](/docs/apis/nerdgraph/examples/nerdgraph-historical-data-export/), [공지 공지](/docs/alerts/get-notified/incident-workflows/) 또는 [Insights API](/docs/apis/insights-apis/query-insights-event-data-api/) 통해 제한된 로그 파티션의 정보를 계속 볼 수 있습니다. 자세한 내용과 완화책을 알아보려면 [잠재적인 데이터 노출 위험을](#data-exposure-risks) 참조하세요.
</Callout>

## 요구 사항 [#requirements]

데이터 액세스 정책을 만들고 관리하려면 다음이 필요합니다.

* [Pro 또는 에디션](https://newrelic.com/pricing) 계정
* [인증 도메인 관리자](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts#admin-settings) 역할
* [핵심 사용자 또는 전체 플랫폼 사용자](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type) 유형

## 데이터 액세스 정책 생성 및 할당 [#create-policies]

UI 또는 [NerdGraph API를](/docs/apis/nerdgraph/examples/nerdgraph-data-access-control) 통해 데이터 액세스 정책을 만들고 할당할 수 있습니다. UI를 통해 정책을 만들려면 다음 단계를 참조하세요.

<Steps>
  <Step>
    ### 정책 만들기 [#create]

    1. <DNT>**[one.newrelic.com &gt; Administration](https://one.newrelic.com/admin-portal) &gt; Access Management &gt; Data access policies**</DNT> 으)로 이동합니다.
    2. <DNT>**Create a policy**</DNT> 클릭합니다(기존 정책이 있는 경우 <DNT>**Add a policy**</DNT> 클릭합니다).

    <img title="Add data access policy" alt="Screenshot showing how to add a data access policy" src="/images/add-data-policy.webp" />

    3. 정책 이름을 입력하세요.

    4. 조건을 선택하세요:

       * <DNT>**Deny selected**</DNT>: 선택한 로그 파티션에 대한 액세스를 차단합니다.
       * <DNT>**Allow selected**</DNT>: 선택된 로그 파티션에만 액세스를 허용합니다.

    5. <DNT>**Filter partitions by account**</DNT> 드롭다운 메뉴에서 계정을 선택하여 로그인 파티션 목록을 선택하세요.

    6. 정책에 포함할 로그 파티션을 선택하세요.

       * 드롭다운 메뉴에서 파티션을 선택하거나
       * 파티션 이름이나 와일드카드(예: 여러 파티션을 일치시키려면 `log_go%` )를 쉼표로 구분하여 입력합니다.

    7. <DNT>**Review policy**</DNT> 클릭하여 설정을 확인하세요.

    8. <DNT>**Create policy**</DNT> 을(를) 클릭합니다.

    정책은 <DNT>**Data access policies**</DNT> 탭에 나타납니다. 이를 보거나 편집하거나 삭제하려면 다음을 클릭하세요. <Icon name="fe-more-horizontal" /> 메뉴.
  </Step>

  <Step>
    ### 그룹에 정책 할당 [#assign]

    1. <DNT>**Administration &gt; Access Management &gt; Grants**</DNT> 으)로 이동합니다.
    2. 정책을 할당할 권한을 찾은 다음 클릭하세요. <Icon name="fe-more-horizontal" /> 옆에 메뉴가 있습니다.
    3. <DNT>**Add data access policy**</DNT> 선택합니다.
    4. <DNT>**Policies**</DNT> 드롭다운 메뉴에서 정책을 선택하세요.
    5. <DNT>**Save**</DNT> 을(를) 클릭합니다.

    <DNT>**Policy name**</DNT> 열에는 할당된 정책이 표시됩니다. 제거하거나 변경하려면 다음을 사용하세요. <Icon name="fe-more-horizontal" /> 메뉴.
  </Step>
</Steps>

## 잠재적인 데이터 노출 위험 [#data-exposure-risks]

데이터 액세스 제어는 대부분의 블루렐릭 기능에서 로그 데이터 액세스를 제한하지만 사용자에게 제한된 데이터가 계속 표시될 수 있는 시나리오가 있습니다.

<CollapserGroup>
  <Collapser id="api-keys" title="구형 Insights API 키">
    **위험**: 구형 Insights API 키를 사용하는 사용자는 제한된 로그인 파티션을 쿼리할 수 있습니다. 이러한 키는 데이터 액세스 정책을 따르지 않기 때문입니다.

    **완화책**:

    1. [API 키 페이지](https://one.newrelic.com/admin-portal/api-keys/home)로 이동하세요
    2. 구형 Insights API 키 식별
    3. 이 키를 회전하거나 삭제하세요
    4. 데이터 액세스 정책을 준수하는 새 [사용자 키를](/docs/apis/intro-apis/new-relic-api-keys/#user-key) 만듭니다.
  </Collapser>

  <Collapser id="alerts" title="알림 및 공지">
    **위험**:

    민감한 데이터가 포함된 공지 조건 쿼리를 생성하면 해당 계정에 대한 액세스 권한이 있는 사용자는 다음을 수행할 수 있습니다.

    * 쿼리가 파티션을 제한하는 조건을 포함하여 모든 공지 조건에서 공지를 수신합니다.
    * 제한된 데이터를 포함할 수 있는 다음 이벤트를 쿼리합니다: `NrAiSignal`, `NrAiIncident`, `NrAiIssue`, `NrAiNotification` 또는 `NrIntegrationError`
    * 조건이 제한된 파티션을 쿼리하는 경우에도 조건 세부 정보 페이지와 문제 페이지를 봅니다.

    **완화책**:

    1. 알림 및 공지플로우에 접근하지 않고 [맞춤 역할](/docs/accounts/accounts-billing/new-relic-one-user-management/account-user-mgmt-tutorial/#roles) 생성
    2. 알림을 관리하지 않아야 하는 사용자에게 할당합니다.
    3. 공지 조건에서 민감한 필드에 대한 패싯을 피하세요
  </Collapser>

  <Collapser id="data-partition-modify" title="데이터 파티션 규칙 기능에 대한 액세스 수정">
    **위험**:

    <DNT>**Data partitions rules**</DNT> 기능에 대한 <DNT>**Modify**</DNT> 액세스 권한이 있는 사용자는 데이터 파티션 설정을 수정할 수 있습니다. 이를 통해 사용자는 파티션 규칙이나 설정을 변경하여 제한된 로그 데이터에 액세스할 수 있습니다. 이 기능은 데이터 접근 제어 제한을 우회합니다.

    **완화책**:

    * 파티션 설정을 관리해야 하는 신뢰할 수 있는 관리자에게만 이 액세스 권한을 부여하세요.
    * 로그만 보고 싶은 사용자를 위해 이 기능 없이 [사용자 정의 역할을](/docs/accounts/accounts-billing/new-relic-one-user-management/account-user-mgmt-tutorial/#roles) 만듭니다.
  </Collapser>

  <Collapser id="query-keys-modify" title="쿼리 키 기능에 대한 액세스 수정">
    **위험**: <DNT>**Query keys**</DNT> 기능에 대한 <DNT>**Modify**</DNT> 액세스 권한이 있는 사용자는 계정 전체 쿼리 기능을 제공하는 쿼리 키를 생성할 수 있습니다. 이러한 키는 사용자 수준의 데이터 액세스 제한을 우회합니다. 이를 통해 제한된 파티션을 포함하여 전체 계정의 모든 로그 데이터를 쿼리할 수 있습니다.

    **완화책**:

    * 쿼리 키를 생성해야 하는 신뢰할 수 있는 관리자에게만 이 액세스 권한을 부여하세요.
    * 일반 사용자에게는 이 기능이 없는 [사용자 지정 역할을](/docs/accounts/accounts-billing/new-relic-one-user-management/account-user-mgmt-tutorial/#roles) 만듭니다.
  </Collapser>
</CollapserGroup>
