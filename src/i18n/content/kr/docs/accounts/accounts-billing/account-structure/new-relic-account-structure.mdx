New Relic 조직 및 계정의 구조를 이해하면 New Relic에서 데이터를 찾고 사용자 액세스를 설정하는 데 도움이 됩니다.

## 조직 및 계정 [#organization-accounts]

뉴렐릭에서 <DNT>**organization**</DNT> 은 뉴렐릭 고객을 나타냅니다. 귀하의 뉴렐릭 조직에는 조직의 계정, 사용자, 데이터 등 귀하와 귀하의 팀 구성원과 관련된 모든 것이 포함되어 있습니다.

New Relic에 가입하면 단일 계정을 포함하는 조직이 생성됩니다. [무료 및 표준 버전은](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing#editions) 단일 계정만 가질 수 있습니다. Pro 및 Enterprise 버전 조직은 계정을 추가할 수 있습니다.

계정에는 여러 데이터 보고 소스가 있을 수 있습니다. 예를 들어 단일 계정에는 인프라 에이전트, <InlinePopover type="apm"/>에이전트 및 기타 통합의 데이터 보고가 있을 수 있습니다. 뉴렐릭에 보고된 모든 데이터에는 해당 데이터가 어느 계정에 속해 있는지 알려주는 [계정 ID가](/docs/accounts/accounts-billing/account-structure/account-id) 필요합니다. 해당 ID는 API 호출과 같은 일부 계정별 작업에도 사용됩니다.

[사용자 액세스를 설정할](#account-access)때 사용자에게 특정 계정에 대한 액세스 권한을 부여합니다.

## 교차 계정 액세스 [#cross-account-access]

뉴렐릭은 계정 간에 데이터를 공유할 수 있는 크로스 플랫폼 기능을 제공합니다. 예를 들어 사용자를 여러 계정에 할당하면 해당 사용자가 계정 소유자가 아니더라도 해당 계정의 데이터를 사용자와 공유할 수 있습니다. 해당 계정에 대한 액세스 권한이 할당되어 있는 한, 제외 추적, custom <InlinePopover type="dashboards"/>및 워크로드와 같은 다른 기능은 엔터티 모니터의 데이터를 다른 계정에 공유합니다.

그러나 일부 교차 계정 경계가 있습니다. 예를 들어, [쿼리 빌더를 사용](/docs/query-your-data/explore-query-data/query-builder/introduction-query-builder) 하면 현재 속한 계정에서만 쿼리할 수 있습니다(계정 간 쿼리를 수행하려면 [NerdGraph API를](/docs/apis/nerdgraph/examples/nerdgraph-nrql-tutorial) 사용해야 함). 이것이 귀하가 보유한 계정 수를 최소화하기로 선택한 이유 중 하나입니다. 명심해야 할 또 다른 제한 사항은 교차 계정 알림을 생성할 수 없다는 것입니다. 여러 계정에 걸쳐 데이터를 집계해야 하는 경우 이로 인해 문제가 발생할 수 있습니다.

## 계정을 추가하는 이유 [#add-accounts]

더 많은 계정을 추가하는 이유는 목표와 구조에 따라 다릅니다. 계정을 추가하는 몇 가지 이유는 다음과 같습니다.

* 프로덕션 환경과 테스트 환경을 분리합니다. 예를 들어, 한 계정의 이름은 `Invoice processing dev` 이고 다른 계정은 `Invoice processing prod` 입니다.
* 다른 부서에 대한 청구를 쉽게 구분할 수 있도록 합니다.
* 상당히 다른 데이터 세트가 있는 모든 종류의 프로젝트를 분리합니다.
* 보안을 위해 다른 프로젝트 또는 데이터 세트에 대한 사용자 액세스를 제어합니다. 예를 들어 특정 엔터티의 데이터만 포함하는 외부 계약자용 계정을 설정하여 해당 그룹이 액세스할 수 있는 항목을 제한할 수 있습니다.

계정을 추가하고 이름을 바꾸는 방법은 계정 [추가](/docs/accounts/accounts-billing/account-structure/add-accounts) 를 참조하십시오.

## 사용자가 계정에 액세스하는 방법 [#account-access]

조직에서 New Relic 사용자는 의무 및 책임과 관련된 특정 계정에 대한 액세스 권한이 부여됩니다. 계정에 대한 사용자 액세스를 관리하려면 [특정 계정의 특정 역할에 대한 액세스 권한을 그룹에 부여합니다](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts#understand-concepts) . 당사의 사용자 관리 시스템을 사용하면 몇 개의 계정에서 몇 가지 역할로 비교적 간단한 설정이든 여러 계정에서 많은 역할을 갖는 복잡한 설정이든 상관없이 필요한 사용자 액세스를 생성할 수 있습니다. [사용자 관리에 대해 자세히 알아보세요.](/docs/accounts/accounts-billing/new-relic-one-user-management)

<InlinePopover type="dashboards"/>및 워크로드와 같은 일부 기능은 조직 내 여러 계정의 데이터를 표시할 수 있습니다. 즉, 사용자에게 모든 관련 계정에 대한 액세스 권한이 부여되지 않으면 데이터가 누락될 수 있습니다.

액세스 문제에 대해 자세히 알아보려면 액세스[에 영향을 미치는 요소](/docs/accounts/accounts-billing/account-structure/factors-affecting-access-features-data/) 를 참조하십시오.

## 원래 사용자 모델 [#original-model]

사용자가 여전히 원래 사용자 모델을 사용하는 경우 계정 구조와 액세스가 다르게 작동합니다. [원래 사용자 모델 계정 액세스](/docs/accounts/original-accounts-billing/original-users-roles/original-account-structure) 를 참조하십시오.
