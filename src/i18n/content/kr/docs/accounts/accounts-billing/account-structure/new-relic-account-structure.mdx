## 조직 및 계정 [#organization-accounts]

New Relic에서 **조직** 은 New Relic 고객을 나타냅니다. New Relic 조직에는 조직의 계정, 사용자 및 데이터와 같이 귀하 및 팀 구성원과 관련된 모든 것이 포함됩니다.

New Relic에 가입하면 단일 계정을 포함하는 조직이 생성됩니다. Standard Edition 조직은 하나의 계정만 가질 수 있습니다. Pro 및 Enterprise 에디션 조직은 더 많은 계정을 추가할 수 있습니다.

계정에는 보고하는 데이터 소스가 많이 있을 수 있습니다. 예를 들어, 단일 계정에는 인프라 에이전트, APM 에이전트 및 기타 통합의 데이터 보고가 있을 수 있습니다. New Relic에 보고되는 모든 데이터에는 [계정 ID](/docs/accounts/accounts-billing/account-structure/account-id) 가 필요합니다. 이 ID는 New Relic에 해당 데이터를 넣을 계정을 알려줍니다. 이 ID는 API 호출과 같은 일부 계정별 작업에도 사용됩니다.

[사용자 액세스를 설정할](#account-access) 때 사용자에게 특정 계정에 대한 액세스 권한을 부여합니다.

## 교차 계정 액세스 [#cross-account-access]

New Relic은 사용자가 여러 계정의 데이터에 액세스할 수 있는 다양한 플랫폼 간 기능을 제공합니다. 예를 들어 사용자에게 여러 계정에 대한 액세스 권한을 할당할 수 있습니다. 또한 분산 추적, 사용자 지정 대시보드 및 워크로드와 같은 기능을 사용하면 해당 계정에 대한 액세스 권한이 할당된 경우 다른 계정에서 모니터링되는 엔터티의 데이터를 볼 수 있습니다.

그러나 교차 계정 장애물이 있을 수 있습니다. 예를 들어 [쿼리 빌더](/docs/query-your-data/explore-query-data/query-builder/introduction-query-builder) 를 사용하면 현재 있는 계정 내에서만 쿼리할 수 있습니다(교차 계정 쿼리를 작성하려면 [NerdGraph API](/docs/apis/nerdgraph/examples/nerdgraph-nrql-tutorial) 를 사용해야 함). 이것이 귀하의 계정 수를 최소화하려는 한 가지 이유입니다.

## 계정을 추가하는 이유 [#add-accounts]

계정을 추가하는 이유는 목표와 구조에 따라 다릅니다. 아주 큰 조직도 단일 계정 또는 두세 개의 계정만 보유하는 것을 방해하는 것은 없습니다.

계정을 계획할 때 가능한 한 조직의 계정 수를 최소화해야 합니다. 이는 두 가지 이유 때문입니다.

* [사용자가 액세스할 수 있는 계정](#account-access) 을 보다 쉽게 계획하고 관리할 수 있습니다.
* 이를 통해 사용자는 일부 New Relic 기능을 더 쉽게 사용할 수 있습니다(예: [쿼리 작성기에서 교차 계정 제한](#cross-account-access) 이 더 적음).

계정을 추가해야 하는 몇 가지 이유:

* 프로덕션 환경과 테스트 환경을 분리합니다. 예를 들어, 한 계정의 이름은 `Invoice processing dev` 이고 다른 계정은 `Invoice processing prod` 입니다.
* 상당히 별개의 데이터 세트를 포함하는 모든 종류의 프로젝트를 분리합니다.
* 다른 프로젝트 또는 데이터 세트에 대한 사용자 액세스를 제어합니다. 예를 들어 특정 엔터티의 데이터만 포함하는 외부 계약자용 계정을 설정할 수 있으며 별도의 계정을 만드는 것이 한 가지 방법입니다.

계정을 추가하고 이름을 바꾸는 방법은 계정 [추가](/docs/accounts/accounts-billing/account-structure/add-accounts) 를 참조하십시오.

## 사용자가 계정에 액세스하는 방법 [#account-access]

조직에서 New Relic 사용자는 의무 및 책임과 관련된 특정 계정에 대한 액세스 권한이 부여됩니다. 계정에 대한 사용자의 액세스를 관리하려면 특정 계정의 특정 역할에 사용자 그룹을 할당하는 [액세스 권한 부여](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts#understand-concepts) 를 생성합니다. 예를 들어, [**결제 관리자** 역할](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts#standard-roles) 을 사용하여 일부 계정에서 결제를 관리하는 기능을 그룹에 할당하고, 일부 계정에서 일부 사용자를 비관리자 전체 플랫폼 사용자로 할당하고, 일부 사용자를 일부 계정에서 기본 사용자로 할당할 수 있습니다. 당사의 사용자 관리 시스템을 사용하면 몇 개의 계정에 걸쳐 몇 가지 역할이 있는 비교적 간단한 설정이든 여러 계정에 걸쳐 많은 역할을 갖는 복잡한 설정이든 상관없이 필요한 사용자 액세스 권한을 생성할 수 있습니다. [사용자 관리에 대해 자세히 알아보세요.](/docs/accounts/accounts-billing/new-relic-one-user-management/)

대시보드 및 워크로드와 같은 일부 기능은 조직 내 여러 계정의 데이터를 표시할 수 있습니다. 즉, 사용자에게 모든 관련 계정에 대한 액세스 권한이 부여되지 않은 경우 데이터가 누락될 수 있습니다.

액세스 문제에 대해 자세히 알아보려면 액세스[에 영향을 미치는 요소](/docs/accounts/accounts-billing/account-structure/factors-affecting-access-features-data/) 를 참조하십시오.

## 원래 사용자 모델 [#original-model]

사용자가 여전히 원래 사용자 모델을 사용하는 경우 계정 구조와 액세스가 다르게 작동합니다. [원래 사용자 모델 계정 액세스](/docs/accounts/original-accounts-billing/original-users-roles/original-account-structure) 를 참조하십시오.
