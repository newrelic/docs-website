Impact \[#impact] 최소한의 리소스 오버헤드로 더 심층적인 옵저버티를 달성하고 문제를 더 빠르게 해결할 수 있으므로 모델링이 부품 성능에 영향을 주지 않습니다. 에이전트는 시스템 역학에 대한 포괄적인 이해를 위해 고유한 커널 수준 및 네트워크 세부 정보를 제공합니다. \*/ /\* 현대적인 설계는 쿠버네티스 클러스터부터 가상머신 및 cloud 외에 기존 Linux 호스트에 이르기까지 복잡하고 혼합된 버전에서 실행되는 경우가 많기 때문에 어렵습니다. 이러한 환경이 확장될수록 상당한 수동 설정 없이는 Performance에 대한 가시성을 확보하기가 더 어려워집니다. 엔지니어는 클러스터에서 어떤 애플리케이션이 실행되고 있는지, 어떤 애플리케이션이 성능 저하를 일으키는지 알아야 합니다. 때로는 애플리케이션 소유자가 애플리케이션 수준 도구만으로는 해결할 수 없는 기본 인프라 문제로 인해 애플리케이션 성능이 저하되는 경우가 있습니다. 귀하의 팀이 다양한 자산을 관리한다고 상상해 보십시오. \* 신속하게 문제를 해결해야 하지만, 포함된 통합 플랫폼과 상관관계를 지정하는 것은 쉽지 않습니다. \* 새로운 범위는 구현하다, 배포하다이지만, 계측된 것은 코드를 소유하지 않거나, 개발팀이 외부에 있거나, 단순히 서비스가 존재한다는 사실을 모르기 때문에 어려운 일입니다. 두 시나리오 모두 모든 애플리케이션의 언어나 아키텍처에 대한 사전 지식이 필요 없이 전체 시스템에 대한 광범위한 가시성을 제공하는 솔루션이 필요합니다. 뉴렐릭 eBPF 에이전트는 이러한 과제를 해결합니다. 이는 이러한 과제를 해결하도록 설계된 강력하고 비침투적인 도구입니다. 이는 Linux 커널에서 직접 인사이트를 얻는 안전하고 효율적인 방법을 제공하는 업계 최고의 기술인 확장된 Berkeley Packet Filter(eBPF)를 활용합니다. 이 eBPF 에이전트를 사용하면 커널에서 직접 풍부한 서비스 포괄적인 설명서를 수집하여 전체 Linux 자산에 대한 통합되고 자동화된 보기를 얻을 수 있습니다. 이 접근 방식을 사용하면 코드를 변경하지 않고도 모든 애플리케이션에 대한 심층적인 가시성을 얻을 수 있습니다. \*/ /\* ## 주요 기능 \[#key-features] 뉴렐릭 eBPF 에이전트는 Linux 호스트 및 쿠버네티스 클러스터에 포괄적인 옵저버빌리티를 제공하도록 설계되었습니다. 전체 시스템 상태를 효과적으로 모니터링하는 데 도움이 되는 다양한 기능을 제공합니다. \* 제로 코드 측정, 코드를 수정하지 않고도 기능 성능에 대한 심층적인 가시성을 확보하거나 배포하다, 언어별 에이전트. 이러한 접근 방식은 운영 오버헤드를 줄이고 전체 인프라에서 가치 실현 시간을 단축합니다. \* 서비스를 자동으로 검색하여 실행 중인 모든 서비스와 해당 의존성/종속성을 지속적으로 식별하고 매핑합니다. 이를 통해 애플리케이션 아키텍처를 실시간으로 볼 수 있습니다. \* Kubernetes에서 에이전트는 클러스터의 서비스 수준 정보를 사용하여 애플리케이션에 대한 기본 컨텍스트를 제공합니다. \* Linux 호스트에서는 환경 변수나 기본 프로세스 정보에서 서비스 이름을 파생시켜 애플리케이션 동작을 명확하게 볼 수 있습니다. \* Kubernetes 에서 애플리케이션을 실행하든 기존 Linux 호스트에서 실행하든, 전체 환경에서 일관된 가시성을 제공하기 위해 통합된 텔레메트리를 제공합니다. \* 인사이트의 서비스 결합, 요청률, 지연 시간, 오류 등을 파악하기 위한 서비스 기반 성능 분석. 이를 통해 실제 상황에서 애플리케이션이 어떻게 수행되는지 이해하는 데 도움이 됩니다. \* 현대적인 마이크로서비스이든 인테리어이든 모든 언어로 작성된 모니터에 대한 언어에 구애받지 않는 가시성. 이를 통해 모든 모니터링 요구 사항에 단일 도구를 사용할 수 있습니다. \* 상세한 델 데이터를 수집하는 동시에 시스템 성능에 미치는 영향을 최소화하기 위한 낮은 리소스 오버헤드입니다. 이렇게 하면 모니터링이 애플리케이션 성능을 방해하지 않습니다. \* 인사이트 기능이 기본 시스템과 상호 작용하는 방식을 제공하기 위한 커널 및 사용자 공간 증폭에 대한 심층적인 가시성. 이는 고대에서 트레이스 이벤트를 캡처하는 데 도움이 됩니다. \* 리치 데이터 데이터는 문제 해결, 해결 및 성능 분석에 도움이 되도록 시스템 운영에 대한 세부적인 세부 정보를 수집합니다. \*/ /\* \`IMAGE Needed\` \*/ /\* ## eBPF 에이전트 작동 방식 \[#how-it-works] 뉴렐릭 eBPF 에이전트는 Kubernetes 에서 helm 차트를 사용하고 Linux 호스트에서 Debian/RPM 패키지를 사용하여 배포합니다. Kubernetes에서는 \`daemonset\`으로 실행되고 Linux 호스트에서는 시스템 서비스로 실행됩니다. 에이전트를 설치하면 해당 환경에서 실행 중인 모든 서비스와 애플리케이션을 자동으로 감지합니다. 이러한 부분은 eBPF 부분 아래 뉴렐릭의 APM &amp; Service UI 에 통합되어 있습니다. 뉴렐릭 eBPF 에이전트에는 두 가지 모드가 있습니다. \* \*\*검색 모드:\*\* 활성 모니터링 없이 환경에서 자동으로 검색합니다. 이 모드는 초기 설정이나 덜 방해가 되는 모니터링에 적합합니다. \* eAPM 모드:\*\* 시스템 역학에 대한 포괄적인 이해를 위해 적극적으로 모델링하고 하위 레벨 지표를 수집하여 더 깊은 수준의 옵저버빌리티를 잠금 해제합니다. ## eBPF가 APM 보완하는 방식 \[#ebpf-APM] \*/

현대식 그리드 그리드는 쿠버네티스 클러스터부터 기존 Linux 호스트까지 혼합된 가구 환경에서 실행되는 이기종의 상호 연결된 서비스로 구성됩니다. 이러한 복잡성은 다음과 같은 문제 해결 속도를 늦추는 근본적인 문제를 야기합니다.

* **옵저버빌리티 성능과 전망 사이의 격차:** 성능 모니터링은 코드 성능에 대한 심층적인 인사이트를 제공하는 동시에 전망 모니터링은 시스템 상태를 추적합니다. 하지만 이러한 구성 요소 간의 네트워크 계층은 종종 눈에 띄지 않습니다. 인지던트 중에 근본 원인이 애플리케이션 로직에 있는지, 아니면 네트워크 관련 문제에 있는지 확인하려면 추가 조사가 필요합니다.

* **이기종 에이전트의 운영 오버헤드:** 마이크로서비스 환경은 종종 구현하다, 배포하다 여러 가지 다른 분석 에이전트를 필요로 합니다. 이로 인해 운영상의 복잡성, 잠재적인 에이전트 충돌이 발생하고 애플리케이션 수정 및 재시작이 필요하게 되는데, 이는 DevOps 팀에게 큰 어려움입니다.

* **Siled 텔레메트리 데이터:** 데이터 수집 후, 데이터 수집을 네트워크 범위 및 포함 지표와 연관시키려면 광범위한 수동 작업이 필요하므로 불완전한 옵저버빌리티 및 인시던트 해결 지연이 발생합니다.

* **제한된 플랫폼 팀 제어:** 플랫폼 엔지니어는 전체 자산에 걸쳐 쉽게 추가할 수 있는 능력이나 권한이 부족한 경우가 많지만 여전히 전반적인 사이트 안정성과 성능에 대한 책임은 남아 있습니다.

뉴렐릭 eBPF 옵저버빌리티 솔루션은 통합과 단순화를 통해 이러한 과제를 해결합니다. 우리의 단일하고 가볍고 언어에 구애받지 않는 에이전트는 eBPF 기술을 활용하여 Linux 커널에서 직접 심층적인 &quot;외부에서 내부로&quot; 인사이트를 얻습니다.

## 주요 특징들 [#key-features]

* **침입 없는 완벽한 가시성:** 프로그래밍 언어에 관계없이 쿠버네티스 클러스터 및 Linux 호스트/컨테이너 전반에 걸쳐 새로운 구형, 타사 워크로드에 대한 완벽한 옵저버빌리티를 확보하세요.

* **통합 데이터 수집:** 커널 수준에서 작동하는 당사의 단일 에이전트는 포함 범위, 쿼리, 네트워크 트래픽 및 포함 지표를 포함한 옵저버빌리티 데이터를 동시에 수집하여 데이터 사일로 문제를 해결합니다.

* **단순화된 구현, 배포:** 옵저버빌리티를 커널로 이동함으로써 수정이 필요하지 않으며 다시 시작하거나 코드를 변경할 필요 없이 마이크로서비스가 많은 환경에서 롤아웃을 단순화합니다.

이 솔루션은 플랫폼 엔지니어가 직면한 중요한 과제를 해결합니다.

* **더 빠른 문제 해결, 해결:** 외관 및 네트워크 인사이트와 통합 텔레메트리의 상관관계를 확인하세요. 이를 통해 성능 문제, 특히 기본 시스템에서 발생하는 문제를 빠르게 진단할 수 있습니다.

## eBPF 기반 APM(eAPM) [#ebpf-apm]

eBPF 기반 APM(eAPM)은 코드 없이, 언어에 구애받지 않는 애플리케이션 모니터링을 위한 표준입니다. 호스트 또는 Kubernetes 노드에서 실행 중인 모든 서비스를 자동으로 검색하고 성능 데이터로 새로운 APM UI 채웁니다. 이 기능은 알 수 없는 서비스나 타사 애플리케이션 등 수동으로 측정할 수 없는 서비스를 포함하여 모든 서비스의 전체 인벤토리를 얻는 데 이상적입니다.

### 주요 특징들

* **지속적인 서비스 검색:** 실행 중인 모든 애플리케이션 서비스를 자동으로 검색하고 매핑하여 아키텍처에 대한 실시간 보기를 제공합니다.
* **제로 코드 측정, 기능:** 코드 또는 구현을 수정하지 않고도 기능 성능(MELT 데이터 포함)에 대한 심층적인 가시성을 제공합니다.
* **광범위한 환경 지원:** **x86 및 ARM64** 모두에서 Linux 호스트(다양한 배포판) 및 쿠버네티스 클러스터에 대한 포괄적인 지원을 제공합니다.
* **고급 프로세서 추적:** 성능 병목 현상, 병목지점을 정확히 찾아내기 위해 트랜잭션을 개별 구성 요소(애플리케이션 코드, 데이터베이스 호출, 외부 서비스 호출)로 분해하는 상세한 세그먼트 연결을 지원합니다. 이러한 수준의 가시성은 이전에는 기존 APM 에이전트를 통해서만 제공되었습니다.

### 사용 사례

* **플랫폼 엔지니어의 경우:** 귀하가 소유하지 않았거나 귀하가 모르는 사이에 구현, 배포되었던 서비스를 포함하여 귀하의 플랫폼에서 실행 중인 모든 서비스의 즉각적이고 완전한 인벤토리를 확보하세요.
* **불투명한 워크로드를 사용하는 팀의 경우:** 코드를 수정할 수 없는(또는 수정하고 싶지 않은) 언어로 작성된 서비스, 즉시 모니터 실외기, 타사 바이너리 또는 서비스를 사용하세요.
* **전체 환경 범위:** 기존 APM 측정, 계측을 보완하여 eAPM 사용하여 수동으로 계측되지 않는 모든 서비스를 포괄하고 100% 가시성을 보장합니다.

## eBPF 네트워크 메트릭 [#ebpf-network-metrics]

<Callout title="시사">
  이 기능은 아직 개발 중이지만 꼭 사용해 보시기 바랍니다!

  이 기능은 현재 [사전 출시 정책](/docs/licenses/license-information/referenced-policies/new-relic-pre-release-policy/) 에 따라 미리보기의 일부로 제공됩니다. HIPAA 또는 FedRAMP 규정을 준수하는 고객은 이용할 수 없습니다.
</Callout>

eBPF 네트워크 메트릭은 애플리케이션의 네트워크 통신에 대한 심층적인 가시성을 제공합니다. 커널에서 직접 세부적인 TCP, IP 및 DNS 텔레메트리를 캡처하여 뛰어난 성능 문제의 주요 소스를 해결합니다. 이 기능은 APM(eAPM과 기존 APM 모두)을 보완하며 독립적으로 작동하여 애플리케이션의 통신 상태에 대한 전체적인 그림을 제공하여 &quot;애플리케이션인가, 네트워크인가?&quot;라는 질문에 확실하게 답하는 데 도움이 됩니다.

### 구현, 배포 유연성

* **기존 뉴렐릭 APM 클라이언트의 경우:** 기존 APM 에이전트와 함께 eBPF 네트워크 지표를 활성화하여 현재 측정, 리소스를 중단하지 않고 네트워크 계층 가시성을 얻을 수 있습니다. eBPF 에이전트는 네트워크 전용 모드에서 작동하여 APM 데이터를 커널 수준 네트워크 인사이트로 보완합니다.

* **포괄적인 관찰을 위해:** 구현, 전투 및 네트워크 계층 전반에 걸쳐 완전한 옵저버빌리티를 위한 eAPM 및 네트워크 지표 기능을 모두 포함하는 전체 eBPF 에이전트를 배포합니다.

네트워크 지표를 보려면 [네트워크 지표](/docs/ebpf/network-metrics/) 문서를 참조하세요.

### 주요 특징들

* **오류-컨텍스트 네트워크 데이터:** 모든 네트워크 지표는 이를 시작한 특정 구성, 서비스 및 프로세스와 자동으로 상관되어 문제 해결을 쉽게 만듭니다.

* **세부적인 DNS 모니터링:** DNS 확인 실패를 추적하고 연관된 정규화된 도메인 이름(FQDN)을 보고합니다.

* **자세한 TCP 연결 분석:** 다음을 포함하여 연결 상태 및 지연 시간을 측정합니다.

  * **핸드셰이크:** 3-웨이 핸드셰이크 지연시간 및 성공/실패 카운트.
  * **연결 상태:** 연결 지속 시간, 데이터 전송 지속 시간, SYN, ACK, FIN, RST 패킷 수.
  * **오류:** 비정상적인 연결 종료를 추적합니다.

* **패킷 수준 IP 인사이트:** 예측을 통해 패킷 손실 및 네트워크 최적화 기회를 식별합니다.

  * 전송 및 수신된 바이트
  * 패킷 재전송/삭제
  * 패킷 전달 지연시간

### 사용 사례

* **문제 해결, 느린 외부 API 호출 해결:** 외부 서비스에 저하된 응답 시간을 표시하는 경우 네트워크 메트릭은 문제가 특수 로직이 아닌 TCP 핸드셰이크 지연, 패킷 손실 또는 재전송 문제에서 비롯되는지 여부를 식별하는 데 도움이 됩니다.
* **성능 저하에 대한 근본 원인 분석:** TCP 연결 상태, 패킷 전달 지연 시간 및 연결 상태 변경을 검사하여 뛰어난 성능 문제와 네트워크 계층 문제의 상관 관계를 분석합니다.
* **서비스 간 통신 모니터링:** 클러스터 전체에 걸쳐 마이크로서비스-마이크로서비스 통신 상태를 모니터링하여 네트워크 정체 또는 서비스 간 라우팅 문제를 식별합니다.
* **외부 의존성/종속성 문제 해결, 해결:** 아웃바운드 네트워크 패턴 및 연결 품질을 분석하여 성능 문제가 외부 API 엔드포인트 또는 타사 서비스에서 발생하는 경우를 신속하게 식별합니다.

## eBPF 에이전트를 설정하는 방법 [#setup]

Linux 호스트 또는 쿠버네티스 클러스터에서 뉴렐릭 eBPF 에이전트를 설정할 수 있습니다.

<Steps>
  <Step>
    귀하의 환경이 당사의 [호환성과 귀하의 환경 요구 사항을](/docs/ebpf/requirements) 충족하는지 확인하세요.

    * [리눅스 요구 사항](/docs/ebpf/requirements#linux)
    * [Kubernetes 요구 사항](/docs/ebpf/requirements#k8s)
  </Step>

  <Step>
    사용자 환경에 따라 적절한 설치 가이드를 따르세요.

    * [리눅스 설치 가이드](/docs/ebpf/linux-installation/)
    * [Kubernetes 가이드 설치](/docs/ebpf/k8s-installation/)
  </Step>

  <Step>
    ### 뉴렐릭에서 데이터 찾기 [#data]

    eBPF 에이전트가 설치되면 Linux 호스트에서 자동으로 데이터를 수집하기 시작합니다. 이 데이터는 뉴렐릭의 OpenTelemetry UI 에서 액세스할 수 있습니다. 뉴렐릭 OpenTelemetry UI 에 대한 자세한 내용은 [OpenTelemetry APM UI](/docs/opentelemetry/get-started/apm-monitoring/opentelemetry-apm-ui) 참조하세요.
  </Step>
</Steps>
