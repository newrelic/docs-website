  <TabsPages>
    <TabsPageItem id="k8s">
      * Kubernetesノードに必要[なLinuxディストリビューションの要件が](/docs/ebpf/requirements/#kernel-requirements)すべて満たされていることを確認してください。

      ### Kubernetes環境 [#kubernetes-environments]

      以下のKubernetes環境がサポートされています。

      * Amazon Elastic Kubernetes Service（EKS）
      * Google Kubernetes Engine（GKE）
      * Azure Kubernetes Service（AKS）
      * Red Hat OpenShift
      * Microk8s

      ### コンテナランタイムのサポート [#container-runtime]

      以下のコンテナランタイムがサポートされています。

      * Dockerのデフォルトランタイム（runc）
      * コンテナ
      * CRI-O
      * ポッドマン
      * ようき
      * クラン
    </TabsPageItem>

    <TabsPageItem id="linux">
      ### カーネルの要件 [#kernel-requirements]

      * Linuxカーネルのバージョン：

        * `x86-64` (amd64): `v5.8`以上
        * `arm64` (aarch64): `v5.8`以上

      <Callout variant="important">
        Linuxディストリビューションの中には、新しいOSリリース上で古いカーネルバージョンを実行できるものがある。ディストリビューションが[以下にリストされて](#supported-linux-distributions)いる場合でも、カーネル バージョンが`5.8`より低い場合、eBPF エージェントは機能しません。
      </Callout>

      ### サポートされているLinuxディストリビューション [#supported-linux-distributions]

      以下のLinuxディストリビューションがサポートされています。

      * CentOS `9`以降
      * Debian `11`以降
      * Red Hat Enterprise Linux `8.1`以降
      * Ubuntu `20.04`以降
      * Amazon Linux `2` および `2023`
      * Bottleロケット`1.51.0`以上
      * AlmaLinux `9.7`

      ### システム要求 [#system-requirements]

      * root権限（インストールおよび操作に必要）
      * 最低`2 GB`のRAM
    </TabsPageItem>
  </TabsPages>
</Tabs>

### 追加のプロトコルサポート [#protocol-support]

eBPF エージェントは、次のプロトコルの監視もサポートしています。

* HTTP

* Apache・スリフト

* gRPC

* カサンドラ

* カフカ

* AMQP

  <Callout variant="tip">
    Cassandra、Kafka、および AMQP の場合、eBPF エージェントはセグメント リンクをサポートしません。
  </Callout>

### ネットワーク要件 [#network-requirements]

<table>
  <thead>
    <tr>

      </td>

      <td>
        * `otlp.nr-data.net:443`
        * `otlp.eu01.nr-data.net:443` （EU 地域のアカウントのみ必要）

    </tr>

    <tr>
      <td>

      </td>

      <td>

      </td>

<Callout variant="tip">
  eBPF 監視は言語に依存せず、あらゆるアプリケーションにレイテンシとスループット メトリクスを提供します。 ただし、[トランザクションセグメントのリンクは](/docs/apm/transactions/intro-transactions/transactions-new-relic-apm/#segments)言語に依存しており、現在PHPとNode.jsではサポートされていません。
</Callout>

{/* ## Querying eBPF data [#querying-data]

Once the eBPF agent is deployed, telemetry data flows to New Relic and can be queried using NRQL. Here are common query patterns:

### Query spans

Including full-body requests and trace data:

```sql
FROM Span
SELECT * WHERE
deployment.name = '<deployment-name>' LIMIT MAX
```

### Query metrics

Including TCP stats and performance metrics:

```sql
FROM Metric
SELECT * WHERE
deployment.name = '<deployment-name>' LIMIT MAX
```

<Callout variant="tip">
  Replace `<deployment-name>` with the actual deployment name you configured during installation.
</Callout>

### Entity naming

The eBPF agent automatically generates entity names differently depending on the environment:

* **In hosts or Docker**: Names are a combination of the process name, its directory or container ID, and the listening port. For example, `ruby:/home/ubuntu/app:[5678]` or `java:f4aead533895:[8080]`.

* **In Kubernetes**: Names are derived from the service name, for example, `mysql-database-service`.

## Data access in New Relic [#data-access]

eBPF data is accessible through multiple New Relic interfaces:

1. **APM & Services UI**: Go to **[one.newrelic.com](https://one.newrelic.com) > APM & Services** and filter by `instrumentation.name = nr_ebpf`

2. **NRQL queries**: Use the query examples above to explore your data

3. **OpenTelemetry UI**: eBPF data integrates with New Relic's OpenTelemetry experience

For more information on accessing your data, see [OpenTelemetry APM UI](/docs/opentelemetry/get-started/apm-monitoring/opentelemetry-apm-ui). */}
