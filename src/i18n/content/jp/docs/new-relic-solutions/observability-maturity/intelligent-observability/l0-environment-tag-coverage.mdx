このルールは、システム資産内の環境タグの存在に基づいてスコアを提供します。このドキュメントでは、スコアの解釈について説明し、タグ付け戦略を強化するために実行できるアクションに関するガイダンスを提供します。

## 説明

スコアは最新のエンティティスキャンに基づいており、ターゲットエンティティに「環境」タグの値がない場合、失敗します。

## スコアの解釈方法

環境タグ カバレッジ スコアは、適切な環境タグを持つシステム内のエンティティの割合を反映します。 結果を理解する方法は次のとおりです。

* **高スコア (80-100%)** : ほとんどのエンティティが適切にタグ付けされており、タグ付けの実践が成熟していることを示しています。
* **中程度のスコア（50～79%）** : 一貫性を改善する機会がある中程度のカバレッジ
* **低スコア (0-49%)** : システム組織やトラブルシューティングを妨げる可能性のある限られた環境タグ

スコアが低い場合、多くの場合、組織がインフラストラクチャ全体にメタデータ標準またはセマンティック規則を完全に実装していないことを示します。これにより、次のことが困難になる可能性があります。

* 問題がどの環境に影響を与えるかを素早く特定する
* 環境固有の監視およびアラートルールを適用する
* 観察データを効果的に整理およびフィルタリングする

<Callout variant="important">
  環境タグは、運用の明確化にとって重要です。これらがないと、本番環境のインシデントがステージングの問題と混同され、応答時間の遅れや潜在的なビジネスへの影響につながる可能性があります。
</Callout>

## 推奨されるアクション

環境タグのカバレッジ スコアを向上させるには、次の戦略を使用します。

### 1. タグ付けのアプローチを確認し、文書化する

まず、明確なタグ付け標準を確立することから始めます。

* 環境のカテゴリを定義します（例：本番環境、ステージング環境、開発環境、テスト環境）
* 地理的セグメントを考慮する（例：米国東部、EU西部、アジア太平洋）
* 機能部門（顧客対応、社内、実験など）を考慮する
* タグ付けの規則を文書化し、チームと共有する

### 2. 組織に合わせてルールをカスタマイズする

特定の要件に合わせてスコアカード ルールを調整します。

* 組織固有のタグ値を含めるようにルールを変更します
* 必要に応じて複数の環境関連タグの検証を追加する
* コンテキストを提供する隣接タグ（例：リージョン、クラスター、チーム）を含める
* 特定のタグ形式や命名規則を期待するルールを設定する

### 3. 一貫したエージェント デプロイメントの実践を実施する

インフラストラクチャ全体にタグが体系的に適用されていることを確認します。

* エージェント展開中に環境変数を使用してタグを設定する
* エージェント拡張機能を設定して環境タグを自動的に適用する
* タグ付けを含むコードとしてのインフラストラクチャの実践を実装する
* 一貫したタグ付けを強制するデプロイメントテンプレートを作成する

### 4. 外部データソースを活用する

既存のシステムからの自動タグ付けを検討してください。

* cloudプロバイダータグと統合 ( AWS 、 Azure 、 GCP )
* ServiceNowなどの設定管理データベースからタグを同期
* コンテナ化された環境にKubernetesラベルとアノテーションを使用する
* CI/CDパイプラインと接続してデプロイサービスを自動タグ付け

### 5. タグ付け戦略を整理し検証する

長期にわたってタグの品質を維持するためのプロセスを作成します。

* 環境タグでフィルタリングするワークロードを使用して、類似のシステムをグループ化します。
* さまざまな環境セグメントにわたってタグの完全性を定期的に監査する
* ダッシュボードを作成してタグのコンプライアンスを監視する
* さまざまな環境でタグを維持するためのチームの責任を確立する

### 6. 包括的なタグ付け戦略を策定する

タグ付けに関する組織の調整を構築します。

* 命名規則と必要なタグを説明するタグ ガイドラインを作成する
* 一貫したタグ付けの重要性についてチームをトレーニングする
* タグガバナンスと品質管理のプロセスを確立する
* タグがインシデント対応とシステム管理をどのようにサポートするかを文書化する

## 重要な考慮事項

環境タグ カバレッジ スコアを操作するときは、次の要素に留意してください。

**組織に合わせてカスタマイズ**: スコアカード ルールは一般的なガイダンスを提供しますが、特定の環境の構造は異なる場合があります。運用ニーズ、組織構造、展開パターンに基づいて結果を評価します。

**継続的な改善に重点を置く**: タグ付け戦略は、インフラストラクチャと組織の変更に合わせて進化する必要があります。システムが成長し、変化してもタグが関連性と有用性を維持するように、定期的にアプローチを見直してください。

**標準化と柔軟性のバランスをとる**:一貫性は重要ですが、環境固有のニーズを考慮します。一部のサービスでは、他のサービスでは必要のない追加のタグが必要な場合がありますが、コア環境タグが存在している限りは問題ありません。

**自動化を検討してください**。手動でのタグ付けはエラーが発生しやすく、拡張性がありません。デプロイメント パイプラインおよびインフラストラクチャ管理ツールと統合される自動タグ付けソリューションに投資します。

## 次のステップ

環境タグの範囲を改善するには:

1. **小さく始める**: 最も重要な本番サービスから始め、適切な環境タグが付いていることを確認します。
2. **可能な場合は自動化する**: 一貫性を確保するためにタグをデプロイメント プロセスに統合します。
3. **進捗状況の監視**: スコアカードを使用して、時間の経過とともに改善を追跡します
4. **段階的に拡張**: コアサービスにタグを付けたら、開発環境とステージング環境に拡張します。

一貫した環境タグは、インフラストラクチャ全体にわたる効果的な監視、より高速なインシデント対応、より優れたシステム組織のための基盤を作成します。
