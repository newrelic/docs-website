将来のデータ取り込みを予測するためには、データソースの一部または全部に影響を与える可能性のある成長要因について理解する必要があります。以下のセクションでは、一般的な成長ドライバーと呼ばれるものについて説明します。最後に、インジェスト（ [Budgeting](/docs/new-relic-solutions/observability-maturity/operational-efficiency/dg-budgeting) ）プロセスで使用できる成長ドライバーのワークシートの概念を紹介します。

## 成長ドライバーを理解する [#drivers]

### 季節的および景気循環の成長 [#cycle]

1年を通して、また何年にもわたって発生するテレメトリの成長の原因を理解することは非常に重要です。これらの中には、一般的に予想されるものもあれば、予想外のものや、完全に異常なものもあります。これらの概念は、基本予算と成長目標を設定する際に重要であり、また、予期しない遠隔測定による成長をその場しのぎで解決する際にも役に立ちます。このようなユーザーの増加は歓迎すべきことですが、データガバナンスの計画がない場合、圧倒されるような印象を与えることもあります。ビジネスが成長し、新しいユーザーが増え、それぞれのユーザーの活動によって、さらにブラウザ、APM、ログデータが放出されるようになりました。K8sクラスタ、ロードバランサー、Kafkaのようなサポートプラットフォームの拡張の必要性も、送信される遠隔測定の増加を引き起こします。もう1つのタイプの成長は、ユーザー数の明らかな増加を伴わないビジネストランザクションの増加によって引き起こされるものです。たとえば、1種類の商品（靴）を販売していたウェブサイトが、帽子や手袋などの商品も販売するようになった場合です。この結果、ユーザーあたりのビジネス・トランザクションが増加し、ユーザー数の増加と同様のカスケード現象がテレメトリーで発生します。

### コードリファクタリング [#code-refactor]

アプリケーションコードの変更により、ユーザーやビジネストランザクションを追加することなく、遠隔測定量が突然増加するシナリオがあります。いくつかの例を挙げます。

* 開発者は、ユーザーがページにアクセスするたびに、バックエンドと対話する追加のjava javascriptコードを追加します。
* ある開発者が、非常に頻繁に呼び出されるいくつかのアプリケーションのメソッドに、新しいロギングコードを追加しました。
* 新しいデータベーススキーマでは、以前は1回で済んだデータベース呼び出しが複数回必要になります。
* モノリシックなアプリケーションを5つのマイクロサービスに分割し、それぞれについてAPMとdistriubutedトレースデータを出力しています。

### 計装の設定ミス [#misconfiguration]

現場の例ある組織では、以前は約2000台のホストのコアオペレーティングシステムのメトリクスに30秒のサンプリングレートを使用していました。設定ミスまたは不正な変更により、10秒に変更すると、これらのホストから取得されるOSメトリクスの遠隔測定が3倍になりました。

後ほど、テレメトリー標準について説明します。テレメトリー規格で規定されることが多いのは、様々なモニタリング活動のサンプリングレートです。

### 可観測性の幅を広げる [#breadth]

現場の例。観測可能性を拡大するのは、継続的な改善プロセスの一部です。2年近くKafkaブローカーの健全性を監視していたある組織は、Kafaトピックオフセットの監視を開始することにしました。トピック監視データの冗長性を認識していなかった彼らは、Kafka監視からの遠隔測定フットプリントが5倍になったときに驚きました。

<Callout variant="caution">
  合併や買収は、テレメトリの増加"が組織に忍び寄る一般的な方法です" 。我々は、統合プロセスの一環として、観測可能性の統合を正式なアクションアイテムとして組み込むことを提案します。これは、 _クラウドコンピューティングの統合_ を全体的な統合計画に追加するのと同じことです。また、このような事象は予期しないこともあるため、New Relic アカウントチームに頼るべき領域でもあります。
</Callout>

### 予期しないサードパーティの変更 [#third-party]

フィールドの例JMX統合は、サードパーティアプリケーションが公開する任意のメトリックを、接頭辞"Transaction_" で取得するように設計されています。このアプリケーションのバージョン1.0では、サンプルごとに10イベントが得られます。サードパーティアプリケーションを保守しているチームは、"Transaction_" という接頭辞を持つ10個のメトリクスを追加しています。私たちのチームが新しいコードをインストールすると、どの JMX イベントが 2 倍になったかを知って少し驚きました。

## 成長ドライバーワークシート [#driver-worksheet]

テレメトリの成長ドライバーを理解することは、バリュードライバーを理解することと同じくらい重要です。成長ドライバーについては、相関性の高い方法がありますが、一般的には、主要かつ最も直接的なドライバーを選択すべきです。以下の成長ドライバーは、テレメトリバックログの情報を補強し、四半期および年間の成長特性を理解するために使用することができます。

* アクティブユーザーの増加
* 製品数の増加
* インフラストラクチャスケーリングイニシアチブ
* イノベーションイニシアチブ
* 効率化の取り組み（テレメトリ摂取の削減）
* コードの大幅なリファクタリング

もし、あなたのビジネスが成長しているのであれば、成長を考慮することは理にかなっています。ユーザーが20％増えたり、製品が1年で2倍になったりするにもかかわらず、システムを導入してクラウドの計算コストが横ばいになるとしたら、それは意味がないでしょう。このような成長要因への配慮も同じです。

データガバナンスの一環として、各サブアカウントにこのようなシートを作成することをお勧めします。これらのドライバーシートは、プランニングフレームワークで、テレメトリの成長予測を作成するために使用されます。

| チーム             | 成長ドライバー                                                                                                                                                                                                                            |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ストリーミングビデオ      | このチームは、K8sのインフラをリファクタリングしています。現在、彼らのデータセンターでは、オンプレミスのK8sクラスタが管理されています。AWS VPCで新しいクラスタを立ち上げる予定で、この四半期の大半は冗長インフラを持つ可能性がある。K8s Telemetry SMEは、この四半期に5％弱の成長率を達成しました。この後の四半期は、オンプレミスのクラスターを縮小するため、横ばいかマイナスになる可能性がある。                    |
| クラウドプラットフォームチーム | このチームは、いくつかのクラウド・サービスから過剰におしゃべりな、価値の低いログを取り除くことで、ログ量を大幅に削減する計画を持っています。 `bytecountestimate()` を使って深く掘り下げた分析を行い、四半期でインジェストを5%削減する計画を思いつきました。つまり、90日間の成長率はマイナスになるはずです。                                                                |
| 国際サービス          | このチームは、さらに2カ国のサポートを追加する予定です。APM K8s と Mobile SMEs と協力して、7.5%の成長率を見積もることができましたが、これはモバイルイベントの増加によるものに違いありません。彼らは、どの程度のユーザーアクティビティを見るべきかの良い予測を持っているので、現在サポートしている5カ国での現在のインジェストに基づき、比較的良いモデルを構築することが出来ました。                             |
| 出荷& 受領          | このチームは、今期中にアプリケーションログを追加する予定です。現在ディスクに記録されているログの数から推定し、追加されるログインコンテキストタグを考慮し、いくつかの要素を使用しています。このチームは今期、12.6%の成長を見込んでいます。ロギング SME は、New Relic のドロップルールの使い方と Fluentbit でのデータの効率化について素晴らしいガイダンスを与えてくれたので、彼らはこの見積もりに舵を切ることができると確信しています。 |
| マーケティングテクノロジー   | このチームは、Javaのモノリスを3つか4つの別々のマイクロサービスにリファクタリングしています。他のリファクタリングのコード解析と、モノリスの Telemetry 動作の慎重な監査に基づいて、このチームは成長率を 26.7% と予測している。これは比較的大きな数字です。しかし、このリファクタリングは、あと3年から5年はコードベースを比較的安定させることができるはずです。                                        |

この分析の一環として、ベースラインレポートと成長ドライバーシートの分析に基づいて、アカウント全体の成長因子を生成することもお勧めします。この成長因子は、最終的な遠隔測定予算シートで使用される1つの数値です。
