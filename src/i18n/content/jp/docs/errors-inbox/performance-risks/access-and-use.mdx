<DNT>Performance Risks inbox</DNT>にアクセスして使用することで、エンドユーザーに影響が及ぶ前に、アプリケーションパフォーマンスの問題をプロアクティブに特定して解決できます。このガイドでは、<DNT>Performance Risks inbox</DNT>にアクセスし、検出閾値を更新し、New Relic MCPサーバー内でツールを効果的に活用する方法について説明します。<DNT>Performance Risks inbox</DNT>とは何か、およびその主な利点の詳細については、[<DNT>Performance Risks inbox</DNT>の概要](/docs/errors-inbox/performance-risks/introduction)を参照してください。

## パフォーマンス リスク インボックスにアクセスする [#access]

アカウントおよびエンティティレベルで<DNT>Performance Risks inbox</DNT>にアクセスできます：

* [エンティティレベルでのアクセス](#access-entity)
* [アカウントレベルでのアクセス](#access-account)

### エンティティレベルでのアクセス [#access-entity]

特定のエンティティの<DNT>Performance Risks inbox</DNT>にアクセスして、そのエンティティおよび関連するテレメトリーデータに関するパフォーマンスの問題を表示できます。これにより、特定のアプリケーションやサービスのパフォーマンスに焦点を当て、それがエンドユーザーにどのような影響を与えている可能性があるかを理解することができます。

**エンティティの<DNT>Performance Risks inbox</DNT>にアクセスするには：**

1. <DNT>[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &gt; APM &amp; Services</DNT>に移動します。
2. 必要なエンティティを選択します。
3. エンティティナビゲーションペインで、<DNT>**Triage**</DNT>セクションの\[<DNT>**Performance Risks**</DNT>]をクリックします。

### アカウントレベルでのアクセス [#access-account]

アカウント内のすべてのエンティティにわたるパフォーマンスの問題の全体像を把握するために、アカウントレベルで<DNT>Performance Risks inbox</DNT>にアクセスすることもできます。これにより、複数のアプリケーションやサービスに影響を及ぼしている可能性のある共通のパフォーマンス問題を特定し、それに応じて最適化作業の優先順位を付けることができます。

**アカウントの<DNT>Performance Risks inbox</DNT>にアクセスするには：**

1. <DNT>[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &gt; Errors Inbox</DNT>に移動します。
2. エンティティナビゲーションペインで、\[<DNT>**Performance Risks**</DNT>]をクリックします。

## パフォーマンスリスクの表示と分析 [#view-analyze]

<DNT>Performance Risks inbox</DNT>にアクセスすると、検出された問題を2つの方法で表示できます：<DNT>**Triage**</DNT>ビューと<DNT>**Group performance risks**</DNT>ビューです。どちらのビューも同じ基になるデータを表示しますが、異なる分析方法を提供します。

### トリアージ [#triage]

<DNT>**Triage**</DNT>ビューは、類似したパフォーマンスリスクを自動的に処理してグループ化します。類似のクエリは計算、保存され、単一のパフォーマンスリスクグループにマージされます。たとえば、`IN`句のバリエーションなど、同じパターンを共有する複数のクエリは、個別の項目として表示されるのではなく、1つのエントリにグループ化されます。これにより、個々の発生を個別に確認することなく、最も影響の大きい問題を特定して優先順位を付けることが容易になります。

アプリケーション全体のパターンをすばやく特定するために、パフォーマンスの問題を事前にグループ化したサマリーが必要な場合は、<DNT>**Triage**</DNT>ビューを使用してください。

<img title="Screenshot of the Triage view in Performance Risks inbox showing pre-grouped performance issues." alt="Screenshot of the Triage view in Performance Risks inbox showing pre-grouped performance issues." src="/images/performancerisks_triage.webp" />

### グループのパフォーマンスリスク [#group]

<DNT>**Group performance risks**</DNT>ビューでは、事前計算されたグループ化を行わずに、未加工のパフォーマンスリスクデータを表示します。正規化されたクエリ表現（カンマや疑問符のプレースホルダーなど）が表示され、次のようなさまざまな属性に基づいてデータをフィルタリングおよびグループ化できます：

* パフォーマンスリスクの影響を受けるユーザー
* コールを生成するサービス
* クエリをトリガーするAPI

調査に関連する特定の属性に基づいてパフォーマンスリスクを分析する必要がある場合は、<DNT>**Group performance risks**</DNT>ビューを使用してください。

<img title="Screenshot of the Group performance risks view in Performance Risks inbox showing raw performance risk data." alt="Screenshot of the Group performance risks view in Performance Risks inbox showing raw performance risk data." src="/images/performancerisks_grpperrisk.webp" />

## 閾値の設定 [#set-thresholds]

<DNT>Performance Risks inbox</DNT> 設定可能な閾値を使用して、パフォーマンスの異常を自動的に検出します。検出基準が特定のパフォーマンス監視のニーズと優先事項に確実に合致するように、アカウントレベルとエンティティレベルの両方でこれらの閾値を更新できます。これらの閾値は、何がパフォーマンスリスクに該当するかを決定し、アプリケーション内の最も関連性が高く対処可能なパフォーマンスの問題について、確実にアラートが通知されるようにするのに役立ちます。

アカウントまたはエンティティレベルでパフォーマンスリスクの閾値を設定できます：

* [アカウントレベルの閾値](#account-thresholds)
* [エンティティレベルの閾値](#entity-thresholds)

### アカウントレベルの閾値を設定します。 [#account-thresholds]

アカウントレベルで閾値を設定すると、アカウント内のすべてのエンティティに適用されるデフォルトのパフォーマンスリスク検出基準を確立できます。これは、一貫した監視基準を維持し、すべてのアプリケーションが同じパフォーマンスベンチマークに基づいて評価されるようにするために役立ちます。

**アカウントレベルで閾値を設定するには：**

<CollapserGroup>
  <Collapser id="new-account-thresholds" title="新しいアカウントレベルの閾値">
    1. <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &gt; Errors Inbox &gt; Performance Thresholds**.</DNT>に移動します。
    2. **Accounts** \[アカウント] タブで、<DNT>**+ New**</DNT>をクリックします。
    3. <DNT>**Account**</DNT>ドロップダウンから、新しい閾値を作成するアカウントを選択します。
    4. 各アナライザーに必要な制限を設定します。
    5. <DNT>**Submit**</DNT>をクリックして保存します。
  </Collapser>

  <Collapser id="edit-account-thresholds" title="既存のアカウントレベルの閾値を編集">
    1. <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &gt; Errors Inbox &gt; Performance Thresholds**.</DNT>に移動します。
    2. **Accounts** \[アカウント]タブで、更新するアカウントの横にある<DNT>**Edit**</DNT>ボタンをクリックします。
    3. 各アナライザーに必要な制限を設定します。
    4. <DNT>**Submit**</DNT>をクリックして保存します。

    <Callout variant="tip">
      閾値を変更すると、誰が変更を行ったかを追跡するためにユーザー名が表示されます。
    </Callout>
  </Collapser>
</CollapserGroup>

### エンティティレベルの閾値を設定します [#entity-thresholds]

エンティティレベルで閾値を設定すると、独自のパフォーマンス特性や要件を持つ可能性のある特定のアプリケーションやサービスに対して、パフォーマンスリスク検出基準をカスタマイズできます。これは、さまざまな種類のアプリケーション、環境、またはビジネス上の優先事項に合わせて監視を調整する場合に特に役立ちます。

**エンティティレベルで閾値を設定するには：**

特定のエンティティの閾値は、次のいずれかの方法で設定できます：

<CollapserGroup>
  <Collapser id="entity-triage" title="パフォーマンスリスクページから">
    1. <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &gt; APM &amp; Services**.</DNT>に移動します。
    2. 必要なエンティティを選択します。
    3. エンティティナビゲーションペインで、<DNT>**Triage**</DNT>セクションの\[<DNT>**Performance Risks**</DNT>]をクリックします。
    4. 右上隅の<DNT>**Adjust performance thresholds**</DNT>をクリックします。
    5. 各アナライザーに必要な制限を設定します。
    6. <DNT>**Submit**</DNT>をクリックして保存します。

    <Callout variant="tip">
      閾値を変更すると、誰が変更を行ったかを追跡するためにユーザー名が表示されます。

    4. <DNT>**Entity**</DNT>ドロップダウンから、更新するエンティティを選択します。
    5. 各アナライザーに必要な制限を設定します。

## New Relic MCPサーバーのパフォーマンスリスクインボックスツールを使用してください。 [#using]

[New Relic MCPサーバー](/docs/agentic-ai/mcp/overview/)で<DNT>Performance Risks inbox</DNT>ツール（`list_performance_risks()`）にアクセスすることもできます。このツールは、特定のエンティティに対して検出されたすべてのパフォーマンスリスクを、リスクタイプ、重大度、発生回数などの詳細とともに一覧表示します。このツールを使用して、パフォーマンスリスクデータをプログラムで取得できます。
