[IBM](https://www.ibm.com/products/mq) MQ[統合](/docs/integrations/host-integrations/getting-started/introduction-host-integrations)は、IBM MQからディメンションメトリックを収集して送信します。

この[メトリックデータ](#metrics)は、事前に作成されたダッシュボードで表示したり、アラートポリシーを作成したり、カスタムクエリやグラフを作成したりできます。

<img
  title="IBM MQ quickstart dashboard"
  alt="Image of the dashboard available through the IBM MQ quickstart"
  src="/images/infrastructure_screenshot-crop_ibmmq-dashboard.webp"
/>

<figcaption>
  [New Relic IBM MQ Monitorクイックスタート](https://one.newrelic.com/nr1-core?state=c95ed954-7277-e44d-7fd7-351a77b3ff37)を介してインストールされたダッシュボード。
</figcaption>

IBM MQ インフラストラクチャーからデータを取得するために、IBM MQ 統合では、構成で指定されたポートで Prometheus メトリックを公開するサード・パーティーの [Prometheus エクスポーター](https://github.com/ibm-messaging/mq-metric-samples/tree/master/cmd/mq_prometheus)を使用します。この統合は、これらのメトリクスを収集し、エンティティに変換し、フィルタリングして、New Relic に送信します。

<Callout variant="important">
  IBM MQ統合は、ARM64アーキテクチャーをサポートしていません。
</Callout>

IBM MQモニター統合をインストールするには、以下のステップを実行してください。

1. [インテグレーションのインストールとアクティブ化](#install)。
2. [インテグレーションの設定](#config)。
3. [データの検索と使用](#find-and-use)。

## 統合を構成する [#config]

統合を構成するには、統合のYAML構成ファイル`ibmmq-config.yml`の構成を編集します。統合のYAML形式の構成では、必要なログインクレデンシャルを配置し、データの収集方法を構成できます。どのオプションを変更するかは、設定と設定によって異なります。構成ファイルには、すべての統合に適用できる共通の設定があります。

これらの一般的な設定についてすべて読むには、 [構成フォーマット](/docs/create-integrations/infrastructure-integrations-sdk/specifications/host-integrations-newer-configuration-format/#configuration-basics)のドキュメントを参照してください。

### 権限 [#permissions]

IBM MQ インフラストラクチャを監視するには、インテグレーション構成で指定された`username`に特定の権限を付与する必要があります。 これらの権限は、モニターのキュー、チャネル、トピックによって異なります。

一般に、 `username`には次の権限が必要です。

* `DISPLAY`と`INQUIRY`が有効になっており、キューマネージャーに接続できる
* インテグレーション設定で指定されたチャネルにアクセスします（デフォルトでは`SYSTEM.DEF.SVRCONN` ）
* `DISPLAY`権限を持つキュー`SYSTEM.DEFAULT.MODEL.QUEUE`および`SYSTEM.ADMIN.COMMAND.QUEUE`に対して特権コマンドを操作します。
* トピック`SYSTEM.ADMIN.TOPIC`の`DISPLAY`と`SUBSCRIBE`でアクセスし、 `SYSTEM.BROKER.ADMIN.STREAM`

特定の設定に応じて追加の権限が必要になる場合があることに注意してください。

## ibmmq-config.ymlサンプルファイル [#examples]

<CollapserGroup>
  <Collapser
    id="basic-config"
    title="基本構成"
  >
    これは、メトリックを収集するために使用される基本構成です。

    ```yml
    integrations:
      - name: nri-ibmmq
        config:
          hostname: localhost
          port: 1414
          queue_manager: QM1
          channel: SYSTEM.DEF.SVRCONN
          username: your_mq_user
          password: your_mq_password
          monitored_queues: '!SYSTEM.*,*'
          monitored_channels: '*'
          monitored_topics: '*'
          monitored_subscriptions: '*'
          exporter_port: 9157
    ```
  </Collapser>

  <Collapser
    id="PasswordLess-config"
    title="パスワードなしの構成"
  >
    パスワードが指定されていない場合、ユーザー/パスワード認証は無効になり、構成でユーザー名を指定する必要はありません。

    エクスポーターは、統合を実行しているユーザーを clientID として送信して接続を試みます。エージェントのインストール モードに応じて、「root」または「nri-agent」になります。

    このようなユーザーは、パスワードを入力せずにキュー マネージャーにアクセスできる必要があります。

    ```yml
    integrations:
      - name: nri-ibmmq
        config:
          hostname: localhost
          port: 1414
          queue_manager: QM1
          channel: SYSTEM.DEF.SVRCONN
          monitored_queues: '!SYSTEM.*,*'
          monitored_channels: '*'
          monitored_topics: '*'
          monitored_subscriptions: '*'
          exporter_port: 9157
    ```
  </Collapser>

  <Collapser
    id="filters-config"
    title="フィルタを使用した構成"
  >
    これは、フィルターを使用して監視対象キューを制限する基本構成です。<br/>この例では、 `DEV.QUEUE`という名前のすべてのキューを監視し、特にキュー`DEV.QUEUE.1`を無視します。他のパターンは、コンマで区切って追加できます。

    ```yml
    integrations:
      - name: nri-ibmmq
        config:
          hostname: localhost
          port: 1414
          queue_manager: QM1
          channel: SYSTEM.DEF.SVRCONN
          username: your_mq_user
          password: your_mq_password
          monitored_queues: 'DEV.QUEUE*,!DEV.QUEUE.1'
          monitored_channels: '*'
          monitored_topics: '*'
          monitored_subscriptions: '*'
          exporter_port: 9157
    ```
  </Collapser>

  <Collapser
    id="mtls-config"
    title="mTLS と基本認証を使用した構成"
  >
    これは、mTLS のセットアップ方法を示す高度な構成です。`ccdt`ファイルは、 `transmissionSecurity`セクションの`certificateLabel`フィールドを通じてサーバーに提示する証明書を示す必要があります。証明書は、そのようなラベルが付いたキーストアにあり、サーバーによって信頼されている必要があります。さらに、キーストアには、サーバーによって提示された証明書を検証できる証明書が含まれている必要があります。

    ```yml
    integrations:
      - name: nri-ibmmq
        config:
          queue_manager: QM1
          channel: SYSTEM.DEF.SVRCONN
          username: your_mq_user
          password: your_mq_password
          monitored_queues: 'DEV.QUEUE*,!DEV.QUEUE.1'
          monitored_channels: '*'
          monitored_topics: '*'
          monitored_subscriptions: '*'
          exporter_port: 9157

          # Configuring mTLS
          mqsslkeyr: "/key"
          ccdt_url: "file:///ccdt.json"
          home: /tmp
    ```
  </Collapser>

## データを見つけて使用する [#find-and-use]

* [New Relic IBM MQモニターのクイックスタート](https://one.newrelic.com/nr1-core?state=c95ed954-7277-e44d-7fd7-351a77b3ff37)は、 [Instant Observability（I / O）](https://one.newrelic.com/nr1-core?state=7f373f09-8149-ffbf-c409-9e988bbb808e)から利用できます。データを簡単に探索し、コンテキストを理解し、問題をより迅速に解決できるダッシュボードを提供します。
* データを検索して使用する方法の詳細については、[統合データ](/docs/infrastructure/integrations/find-use-infrastructure-integration-data)についてを参照してください。
* メトリックは、エンティティ`IBMMQ_MANAGER`および`IBMMQ_QUEUE`のメトリックサンプルおよび[イベントタイプ](/docs/using-new-relic/data/understand-data/new-relic-data-types#events-new-relic)に添付されます。
* トラブルシューティングの目的で、または[カスタムチャートやダッシュボード](/docs/query-your-data/explore-query-data/dashboards/introduction-dashboards/)を作成するために、 [NewRelicのデータをクエリ](/docs/using-new-relic/data/understand-data/query-new-relic-data)できます。
