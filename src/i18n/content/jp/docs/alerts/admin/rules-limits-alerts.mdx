## NRDB アラート クエリに一致する 1 分あたりのデータ ポイント [#query-limit]

アラート条件 `Matched data points per minute` 制限は、New Relic [アカウント](/docs/accounts/accounts-billing/account-structure/new-relic-account-structure) 内のアラート クエリの一致 [データ ポイント](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/understand-technical-concepts/streaming-alerts-key-terms-concepts)の合計レートに適用されます。

この制限を超えると、レートが制限を下回るまで、影響を受けるアカウントの条件を作成または更新できなくなります。 既存のアラート条件は影響を受けませ**ん**。

[制限 UI](/docs/data-apis/manage-data/view-system-limits)で、一致したデータ ポイントと制限インシデントを確認できます。

どのような条件が最大のスループットにつながっているかを理解するには、次のようなクエリを実行できます。

```sql
FROM NrAiSignal
SELECT sum(aggregatedDataPointsCount) AS 'alert matched data points'
FACET conditionId
```

一致したデータ ポイントを最適化するためのヒント:

* [スライディング ウィンドウ](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/#sliding-window-aggregation)を使用している場合は、データ ポイントの数が大幅に増加する可能性があることに注意してください。データ ポイントの数を減らすには、より長い集計期間を使用できます。
* `WHERE` 句を使用して、アラート対象のデータ量を絞り込みます。 `FACET` の代わりに `WHERE` を使用すると、場合によってはより効率的なアラートを生成できます。
* 同様のアラートを結合します。類似したアラート条件が複数ある場合は、組み合わせフィルターを使用してそれらをグループ化することを検討してください。

制限の引き上げをリクエストするには、New Relic アカウント担当者にお問い合わせください。

[スライディング ウィンドウ](/docs/query-your-data/nrql-new-relic-query-language/nrql-query-tutorials/create-smoother-charts-sliding-windows) を使用すると、データ ポイントの数が大幅に増加する可能性があることに注意してください。生成されるデータ ポイントの数を減らすために、スライディング ウィンドウ集計の期間を長くすることを検討してください。

## まとめて 1 分あたりのスキャン操作数 [#query-scan-limit]

アラート条件`Alert query scan operations per minute`の制限は、取り込まれたイベントに対するクエリ スキャン操作の合計レートに適用されます。 書き込みスキャン操作は、取り込まれたイベントとNew Relic アカウント に登録されている一括書き込みを照合するためにNew Relic[ ](/docs/accounts/accounts-billing/account-structure/new-relic-account-structure)パイプラインによって実行される作業です。

この制限を超えると、レートが制限を下回るまで、影響を受けるアカウントの条件を作成または更新できなくなります。 既存のアラート条件は影響を受けませ**ん**。

クエリスキャン操作と制限インシデントは、[制限 UI](/docs/data-apis/manage-data/view-system-limits)で確認できます。

イベントをまとめてと一致させる場合、その書き込みが参照する[データ型](/docs/nrql/get-started/introduction-nrql-new-relics-query-language/#what-you-can-query)からのすべてのイベントを検査する必要があります。 特定のデータ型でのイベントを減らす一般的な方法をいくつか示します (これにより、一括書き込みスキャン操作が減少します)。

* ログ データに関するアラートを発行する場合は、[ログ パーティションを](/docs/tutorial-manage-large-log-volume/organize-large-logs/)使用して、一括書き込みのためにスキャンされるログを制限します。

* カスタムイベントでアラートを送信する場合は、より大きなカスタムイベント タイプを分割します。

* イベントのアラートの代わりにカスタムイベントを使用してください。

* データを集約するために[メトリクスを作成します](/docs/data-apis/convert-to-metrics/analyze-monitor-data-trends-metrics/)。

* 可能な場合は、瞬間イベントでアラートを送信する代わりに[、メトリック タイムスライスを](/docs/data-apis/understand-data/metric-data/query-apm-metric-timeslice-data-nrql/)使用してください。

上記のヒントに加えて、未使用または不要な一括書き込み (アラート条件) をクリーンアップすると、書き込みスキャン操作の回数が減ります。

制限の引き上げをリクエストするには、New Relic アカウント担当者にお問い合わせください。
