プロファイルは、 `ktranslate`が収集する SNMP OID と、受信した SNMP トラップ メッセージの処理方法の両方を定義します。既存のプロファイルを変更する場合、または新しいプロファイルを作成する場合は、ここの手順に従ってください。

## 要件 [#requirements]

プロファイルを作成する場合でも、プロファイルを変更する場合でも、次のものが揃っていることを確認してください。

* [kentik/snmp-profiles](https://github.com/kentik/snmp-profiles) リポジトリをフォークし、変更のプル リクエストを作成できる GitHub[ アカウント 。](https://docs.github.com/en/get-started/signing-up-for-github/signing-up-for-a-new-github-account)
* プロファイルをポーリングするためのネットワーク デバイスからの[SNMP ウォーク](/docs/network-performance-monitoring/troubleshooting/snmp-walk/#capturing-snmpwalk)。
* SNMP トラップ メッセージの場合は、メッセージを定義するベンダーの MIB にもアクセスできる必要があります。

<Callout variant="important">
  New Relic は[、 kentik/snmp-profiles](https://github.com/kentik/snmp-profiles)の内容に対する直接のサポートを提供しません。既存の SNMP プロファイルが不正なデータを返したり、別の破壊的な動作を返したりする場合は、 [kentik/snmp-profiles](https://github.com/kentik/snmp-profiles)リポジトリで問題をオープンしてください。`ktranslate`のサードパーティでの SNMP プロファイルの変更または作成のタイムラインを提供することはできません。

  New Relic アカウントに関連付けられたサービス レベルに応じて、New Relic グローバル テクニカル サポートからのサポートを受けることができる場合があります。当社の[グローバル テクニカル サポート サービス](/docs/licenses/license-information/general-usage-licenses/global-technical-support-offerings/#tstools)に従って、問題が New Relic 製品にあるのか、それとも当社の制御と範囲外にあるものによって引き起こされたのかを判断するために最善を尽くします。当社の制御が及ばないサードパーティ ツールの問題については、商業的に合理的な努力を払い、機能強化、パッチ、またはその他の機能開発を上流のオープンソース プロジェクトに提供します。
</Callout>

    ## 背景

    SNMP ポーリング プロファイルを作成するときは、次の一般的な手順に従います。

    1. サンプル デバイスから[SNMP ウォーク](/docs/network-performance-monitoring/troubleshooting/snmp-walk/#capturing-snmpwalk)を収集します。

    2. ウォークの結果に基づいて、デバイスでどの MIB がサポートされているかを特定します。

    3. サポートされている MIB で、次のような収集するメトリックを特定します。

       * カウンター、ゲージ、または列挙されたメトリックを含むスカラー OID
       * カウンタ、ゲージ、または列挙されたメトリクスを含む表形式の OID
       * タグとして使用できる文字列値を返すスカラーまたは表形式の OID

    4. SNMP プロファイルを作成または編集します。

               <Collapser
                 id="provider"
                 title="「プロバイダー」セクション"
               >
                 このセクションでは、New Relic に送信されるすべてのペイロードの属性として装飾される`provider`の値を特定します。

                 ここでの構文は、 [kebab-case](https://www.freecodecamp.org/news/snake-case-vs-camel-case-vs-pascal-case-vs-kebab-case-whats-the-difference/#kebab-case)を使用した文字列値です。

                 ```yaml
                 provider: kentik-ups
                 ```

                 <Callout variant="tip">
                   `provider`の値は、New Relic での[エンティティ合成](/docs/new-relic-solutions/new-relic-one/core-concepts/what-entity-new-relic/#entity-synthesis)にとって重要です。詳細については、[デバイス構成に関するドキュメント](/docs/network-performance-monitoring/advanced/advanced-config/#devices)を参照してください。独自のプロファイルを使用している場合は、次のいずれかを実行する必要があります。

                   * すでにエンティティにマップされている既存の値を使用します。
                   * [newrelic/entity-settings](https://github.com/newrelic/entity-definitions)リポジトリへのプル リクエストを作成して、プロバイダーの値を既存の定義にマップするか、新しい定義を構築します。
                   * 独自のカスタム ダッシュボード エクスペリエンスを作成してデータを表示します。
                 </Callout>
               </Collapser>

               <Collapser
                 id="sysobjectid"
                 title="「sysobjectid」セクション"
               >
                 このセクションでは[ 、SNMP 検出 中に](/docs/network-performance-monitoring/troubleshooting/snmp-discovery-unexpected-profile/) がデバイスをこのプロファイルにマッピングするために使用する[ sysObjectID](https://oidref.com/1.3.6.1.2.1.1.2) `ktranslate`を特定します。

                 ここの構文は OID のリストです。`*`記号を使用したワイルドカード一致のサポートが提供され、デバイスに個別に注釈を付けることなく、大規模なデバイス グループを考慮できます。さらに、ここに YAML コメントを追加すると、 `ktranslate`が New Relic に送信されるペイロードに装飾される`tags.kentik.model`の値を設定できるようになります。

                 ```yaml
                 sysobjectid:
                   - 1.3.6.1.4.1.318.1.3.17    # APC SmartUPS 3 Phase
                   - 1.3.6.1.4.1.318.1.3.*     # APC UPS
                   - 1.3.6.1.4.1.318.*         # APC device
                 ```
               </Collapser>

               <Collapser
                 id="matches"
                 title="「一致」セクション"
               >
                 このセクションでは、検出中に`sysobjectid`マッピングで使用されるオプションの第 2 レベルのフィルタリングを提供します。単一の SysObjectID がさまざまなデバイスに使用されている場合は、 [SysDescr](https://oidref.com/1.3.6.1.2.1.1.1) OID の内容と一致する正規表現を使用してデバイスをさらに分離できます。この式は、 `SysDescr`文字列の強制小文字バージョンと一致します。[net-snmp.yml](https://github.com/kentik/snmp-profiles/blob/main/profiles/kentik_snmp/_general/net-snmp.yml)プロファイルで例を確認できます。

                 ここでの構文は、キーとして二重引用符で囲まれた正規表現の辞書であり、一致したときにマップされるプロファイル ファイル名の関連値が含まれています。

                 ```yaml
                 matches:
                   "^pfsense": pf_sense.yml
                 ```
               </Collapser>

               <Collapser
                 id="bulkwalk"
                 title="「no_use_bulkwalkall」セクション"
               >
                 このセクションでは、 `ktranslate`がデフォルトの収集戦略として使用する[snmpbulkwalk](http://www.net-snmp.org/docs/man/snmpbulkwalk.html)操作を無効にするオプションを提供します。このキーを`true`に設定すると、 `ktranslate`強制的に`GET`を使用します。 `GETNEXT` \| `GETTABLE`は個別に操作します。これは、一括操作をサポートしていない一部の古いデバイスで便利です。

                 ```yaml
                 no_use_bulkwalkall: true
                 ```
               </Collapser>

               <Collapser
                 id="metrics"
                 title="「メトリクス」セクション"
               >
                 このセクションでは、このプロファイルを使用するときに`ktranslate`がターゲットとするスカラー OID と表形式の OID について概要を説明します。スカラー OID と表形式 OID の構文は両方とも若干異なりますが、それぞれのタブで詳しく説明します。大まかに言うと、ここでの構文は、OID のコレクションを完全に制御できるさまざまな設定を備えた OID のリストです。

                 ```yaml
                 metrics:
                   # Scalar OID
                   - MIB: $MIB_NAME
                     symbol:
                       OID: $SCALAR_OID
                       name: $OID_NAME_FROM_MIB
                   # Tabular OIDs
                   - MIB: $MIB_NAME
                     table:
                       OID: $TABLE_OID
                       name: $TABLE_NAME
                     symbols:
                       - OID: $COLUMN_OID
                         name: $TABULAR_OID_NAME_FROM_MIB
                 ```
               </Collapser>

               <Collapser
                 id="metric-tags"
                 title="「metric_tags」セクション"
               >
                 このセクションでは、関連するメトリックに装飾される属性の概要を説明します。ディメンション メトリックはタイプ`count`である必要があります | `summary` \| `gauge` 。そのため、文字列値はメトリクスとして直接サポートされていないため、 `metric_tag`識別子を使用して既存のメトリクスに修飾する必要があります。

                 この識別子は、グローバル プロファイル設定だけでなく、個々の MIB テーブル設定にも配置できます。使用例としては、 `upsOutletGroupStatusTable`テーブル内のコンセント グループの[upsOutletGroupStatusName](https://oidref.com/1.3.6.1.4.1.318.1.1.1.12.1.2.1.2) 、またはスカラー OID として表される UPS デバイスの[upsAdvIdentSerialNumber](https://oidref.com/1.3.6.1.4.1.318.1.1.1.1.2.3)などがあります。

                 ```
                 metrics:
                   - MIB: PowerNet-MIB_UPS
                     table:
                       OID: 1.3.6.1.4.1.318.1.1.1.12.1.2
                       name: upsOutletGroupStatusTable
                     symbols:
                       - ...
                     metric_tags:
                       - column:
                           OID: 1.3.6.1.4.1.318.1.1.1.12.1.2.1.2
                           name: upsOutletGroupStatusName
                 metric_tags:
                   - column:
                       OID: 1.3.6.1.4.1.318.1.1.1.1.2.3.0
                       name: upsAdvIdentSerialNumber
                 ```
               </Collapser>

## カスタムプロファイルを非公開で使用する [#private]

[kentik/snmp-profiles](https://github.com/kentik/snmp-profiles)リポジトリの内容は、コンテナまたはサービスを構築するときにデフォルトで`ktranslate`にロードされるため、すべてのインスタンスにグローバルに適用されます。

独自のコレクション設定とパターンを維持できるように完全なカスタマイズ オプションをサポートするには、次の手順を実行します。

1. リポジトリのクローンを作成します。
2. 必要に応じて変更を加えます。
3. 起動時にカスタム プロファイル ディレクトリを Docker コンテナまたは Linux サービスにロードします。

これを行うには、カスタマイズしたファイルを`etc/ktranslate/profiles/`ディレクトリ内の KTranslate インスタンスにマッピングします。

`snmp-profiles`のすぐ上のディレクトリにいると仮定すると、追加するコマンドは次のようになります。

```shell
`pwd`/snmp-profiles/profiles:/etc/ktranslate/profiles
```

Docker の場合、これを[ボリューム マウント](https://docs.docker.com/storage/volumes/#start-a-container-with-a-volume)として追加します。これは次のようになります。

```shell
docker run -d --name ktranslate-$CONTAINER_SERVICE --restart unless-stopped --pull=always -p 162:1620/udp \
  -v `pwd`/snmp-base.yaml:/snmp-base.yaml \
  -v `pwd`/snmp-profiles/profiles:/etc/ktranslate/profiles
  -e NEW_RELIC_API_KEY=$NR_LICENSE_KEY \
kentik/ktranslate:v2 \
  -snmp /snmp-base.yaml \
  -nr_account_id=$NR_ACCOUNT_ID \
  -metrics=jchf \
  -tee_logs=true \
  -service_name=$CONTAINER_SERVICE \
  -snmp_discovery_on_start=true \
  -snmp_discovery_min=180 \
  nr1.snmp
```

Linux サービスの場合は、ローカル ホスト上の`/etc/ktranslate/profiles`の内容を編集します。

<Callout variant="tip">
  環境内で一貫性のない動作が発生するリスクを回避するには、コンテナーまたはサービスを起動するたびに、必ずカスタム バージョンのプロファイルを渡してください。
</Callout>

## パブリックリポジトリに貢献する [#public]

オープンソース リポジトリとして、新しいプロファイルを作成したり、すべてのユーザーの監視機能を強化する既存のプロファイルを改善したりするための貢献を誰からでも歓迎します。

New Relic は、サードパーティ リポジトリのプロファイルをレビューするプロセスには関与しません。変更と更新のすべてのリクエストは、Github の問題を通じて Kentik のメンテナーに対処する必要があります。

プロファイルがマージされた直後、CI/CD 自動化により、新しい SNMP プロファイルが新しいバージョンの Docker イメージで使用できるようになります。

## どのような種類の OID を含める必要がありますか? [#oids]

多くの場合、SNMP は大量のデータを提供しますが、実用的な情報はほとんど提供されません。 上記の[サンプル SNMP ウォーク](#snmp-polling)には OID からの結果が 2,022 行あり、[結果のプロファイルには](https://github.com/kentik/snmp-profiles/blob/main/profiles/kentik_snmp/apc/apc_ups.yml)合計約 50 個の OID が収集されます。 [サイト信頼性エンジニアリング (SRE) 分野](https://sre.google/sre-book/monitoring-distributed-systems/)で定義された<DNT>**"Golden Signals"**</DNT>の基本パターンに従い、デバイスが期待どおりに機能を実行できなくなる原因となるものがあるかどうかを確認できるデータの収集に重点を置きます。 それを基にして、それらの機能がどの程度うまく実行されているかを示す測定値を収集する必要があります。

**例 1:**

VPN コンセントレーターとして動作するデバイスの場合、次のような高レベルのシステム メトリクスを収集します。

* CPUとメモリの使用率
* ファンの故障などが原因でデバイスがシャットダウンしないようにするためのハードウェアセンサー情報
* 集約された接続とスループットについて教えてくれるOID

**例 2:**

ネットワーク アプライアンス上で実行中のすべてのプロセスをリストする OID テーブルは、利用可能ではあるが役に立たないデータの例です。

実行中のすべてのプロセスを収集することは便利に聞こえるかもしれませんが、これはアプライアンスであるため、通常は内部で実行されているプロセスに対して何もする機能がなく、またその必要もありません。実際には何もできない数百の項目を含むテーブルをポーリングして保存するのは効率的ではありません。
