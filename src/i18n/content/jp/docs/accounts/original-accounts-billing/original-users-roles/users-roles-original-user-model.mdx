[元のユーザーモデルのユーザーの場合](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models)：このモデルでユーザーの役割と権限がどのように機能するか、およびユーザーを追加および管理する方法。

## 要件 [#who-is-doc-for]

このドキュメントのセクションで[は、オリジナルのユーザー モデル](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models)でユーザーを管理する方法を説明します。 [<DNT>**Users and roles**</DNT> UI](#view-access)にユーザーが表示される場合、それらのユーザーは元のユーザー モデルに存在します。

New Relic ユーザーが持つことができるセッションと IP アドレスの数には制限があります。詳細については、 [アクセスに影響を与える要因を](/docs/accounts/accounts-billing/account-structure/factors-affecting-access-features-data#session-limits)参照してください。

## 新しいユーザーモデルに関する最新情報 [#migrate]

2020年7月に[、組織とユーザーの管理方法に関して多くのメリット](https://newrelic.com/blog/how-to-relic/user-management-and-access-security)を提供する[新しいユーザーモデル](/docs/accounts/original-accounts-billing/original-product-based-pricing/overview-user-models)をリリースしました。当初、これは新規のサインアップでのみ利用可能でしたが、時間の経過とともに、古い顧客を新しいモデルに移行してきました。一部の古い顧客は[、ユーザーの移行をセルフサービス](/docs/accounts/original-accounts-billing/original-users-roles/user-migration)できます。元のモデルが完全に非推奨になるまで、ユーザーを新しいモデルに移行する作業を続けます。

新しいユーザーモデルの影響の1つは、ユーザーが同じ電子メールに複数のログインを関連付けることができるようになったことです。たとえば、複数の組織（請負業者など）にアクセスできるユーザーは、1つの組織でユーザーレコードを新しいユーザーモデルに更新し、その結果、元のログイン方法とレコード、および新しいモデルのユーザーレコードの両方を持つことができます。これにより、ユーザーがNew Relicにログインし、探しているアカウントを見つけることができなくなる可能性があります。詳細については、 [アクセスに影響を与える要因を](/docs/accounts/accounts-billing/general-account-settings/factors-affecting-access-features-data/#account-access)参照してください。

## UIでのユーザーの表示と管理 [#view-access]

<DNT>**Users and roles**</DNT> UI [、オリジナルのユーザー モデル](#who-is-doc-for)でユーザーを管理するために使用されます。 この UI を見つけるには、[ユーザー メニューを](/docs/accounts/accounts-billing/general-account-settings/intro-account-settings)クリックし、 <DNT>**Account settings**</DNT>をクリックして、 <DNT>**Users and roles**</DNT>をクリックします。

新しい使用量ベースの料金モデルを使用している場合は、いくつかの[重要な考慮事項](#full-user-count)を参照してください。

また、 [New Relic REST API](/docs/apis/rest-api-v2/account-examples-v2/listing-users-your-account) を使用して、アカウントのユーザーのリストを取得することもできます。

ここでは、一般的なユーザー管理のタスクやシナリオについて、手順や注意点を説明します。

    3. 適切な名前とEメールアドレスを入力してください。

    5. ベースロールをAdmin、User、Restrictedのいずれかに選択します。

    6. <DNT>**Add user**</DNT>を選択します。

       </Callout>

    [使用量ベースの料金モデルを使用して](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing/)いる場合、ユーザーのユーザータイプが請求の要素になります。

    組織内に複数のアカウントがある場合、オリジナルユーザーモデルで請求書発行可能なユーザーを管理する際の重要な注意点をご紹介します。

    * 組織のすべてのユーザーを単一の UI で確認することはできません。 すべてのユーザーを確認して編集することが目的の場合 (たとえば、請求目的で[ユーザー タイプ](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type)を編集するなど)、個々のアカウントごとに <DNT>**Users and roles**</DNT> UIに移動する必要があります。 (ユーザーをより簡単に管理できることは[、新しいユーザー モデル](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models)の利点の 1 つです。)

    * 上記の理由により、 <DNT>**View your usage**</DNT> UI に表示される課金対象ユーザーの数は、特定のアカウントの<DNT>**Users and roles**</DNT> UI に移動したときに表示されるユーザー数と一致しない可能性があります。

    * <DNT>**Account settings**</DNT> UI でアカウントが「アクティブ」または「非アクティブ」（データをレポートしない）としてマークされているかどうかに関係なく、組織内のすべてのアカウントの課金対象ユーザーに対して課金される可能性があります。

      ユーザータイプの編集に関するヒントは、 [Manage user type](#update-user-type) を参照してください。
  </Collapser>

  <Collapser

    title="ユーザータイプとユーザーアップグレードの管理"
  >
    ユーザータイプを管理するための重要なヒントがあります。

    * [使用量ベースの料金モデルを使用して](/docs/accounts/original-accounts-billing/original-product-based-pricing/overview-pricing-models)いる場合、ユーザーの[ユーザータイプ](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type)が課金要素になります。

    * ユーザーをより高いユーザータイプにアップグレードすると（UIまたはアップグレードリクエストメールを介して）、組織内のすべてのアカウントでそのユーザーのユーザータイプが変更されます。

      ユーザーのユーザータイプを更新するには

    1. <DNT>**[user menu](/docs/accounts/accounts-billing/general-account-settings/intro-account-settings) > Account settings > Users and roles > Users**</DNT>に移動します。

    2. ユーザーを選択してそのタイプを編集するか、[複数のユーザーのタイプを一括更新します](#bulk)。 組織に複数のアカウントがある場合、すべてのユーザーを 1 つの UI で確認することはできないことに注意してください。異なるアカウントのユーザーを管理するには、各アカウントの<DNT>**Users and roles**</DNT> UI に移動する必要があります。

       アップグレードを要求したユーザーを表示し、承認する。

    3. <DNT>**[user menu](/docs/accounts/accounts-billing/general-account-settings/intro-account-settings) > Account settings > Users and roles**</DNT>に移動します。

    4. <DNT>**Upgrade requests**</DNT>クリックします。 組織に複数のアカウントがある場合、すべてのユーザーを 1 つの UI で確認することはできないことに注意してください。異なるアカウントのユーザーを管理するには、各アカウントの<DNT>**Users and roles**</DNT> UI に移動する必要があります。

       ユーザーのアップグレード方法を制御するには、 <DNT>**Users and roles**</DNT> UI から<DNT>**Access requests**</DNT>を選択します。 次の 2 つのオプションがあります。

    * <DNT>**Automatic approval**</DNT>: このオプションを使用すると、ユーザーはユーザー タイプを自動的かつ即座にアップグレードできます。 このオプションを使用すると、ユーザーは問題をより簡単にトラブルシューティングできます。

    * <DNT>**Require review**</DNT>: このオプションを選択すると、ユーザーがアップグレードをリクエストしたときに管理者に通知メールが届き、管理者が最初にユーザーをアップグレードする必要があります。 通知メールまたは<DNT>**Users and roles**</DNT> UI のユーザー レコードから承認できます。

      ユーザータイプについては、 [ユーザータイプ](#user-type) を参照してください。
  </Collapser>

    New Relic アカウントでは、Owner [ロール](/docs/accounts-partnerships/accounts/account-billing-usage/users-roles#roles) を一度に 1 つだけ持つことができます。現在アカウントの管理者ロールを持っている人にロールを変更するには、現在のアカウントオーナーである必要があります。 [現在のオーナーが不在の場合](#no-owner) 、New Relic のアカウント担当者にお問い合わせいただくか、 [support.newrelic.com](https://support.newrelic.com) でサポートを受けてください。

    割り当てられたOwnerロールを削除または解除することはできません。ただし、アカウントに1つ以上のAdminロールがある場合は、OwnerをAdminに変更することができます。

    1. <DNT>**[user menu](/docs/accounts/accounts-billing/general-account-settings/intro-account-settings) > Account settings > Account > Users and roles**</DNT>に移動します。

    2. アクティブユーザーリストの上で、 <Icon name="fe-repeat"/><DNT>**Change owner**</DNT> 。 アカウントに管理者がいない場合、このボタンは使用できません。

    3. 現在、アカウントのAdminロールを持っている人を選択します。

    4. 変更を有効にするには、ページを更新してください。

       以前のオーナーの役割は、自動的に管理者の役割に変更されます。

       現在のアサインされたオーナーを知ることができます。

    5. <DNT>**[user menu](/docs/accounts/accounts-billing/general-account-settings/intro-account-settings) > Account settings > Account > Users and roles**</DNT>に移動します。

    6. アカウント所有者を見つけるには、 <DNT>**Base role**</DNT>列を表示します。

       の<Icon name="fe-repeat"/><DNT>**Change owner**</DNT>ボタンは現在のアカウント所有者にのみ表示されます。 現在の所有者が役割を変更できない場合 (たとえば、その人が組織に所属していない場合) は、New Relic のアカウント担当者に連絡するか、 [support.newrelic.com](https://support.newrelic.com)でサポートを受けてください。

## ユーザータイプ [#user-type]

[使用量ベースの価格設定モデルを使用](/docs/accounts/original-accounts-billing/original-product-based-pricing/overview-pricing-models)している組織の場合、ユーザータイプが課金要素になります。

ユーザーのユーザータイプは、New Relic 機能への最大許可アクセスを制御します。実際には、ユーザーにはさまざまな方法で権限を制限する [役割が](#roles) 割り当てられていることがよくありますが、ユーザー タイプは理論上の最大アクセス権を表します。詳細については、 [「ユーザー タイプ」](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type)を参照してください。

組織内のユーザーが異なるアカウントで異なるユーザータイプに設定されている場合、そのユーザーは最も高いユーザータイプとみなされます。

ユーザーのタイプを編集する方法については、 [Manage user type](#update-user-type) を参照してください。

## 追加の役割 [#add-on]

アドオン ロールを使用すると、ユーザーにさらに具体的かつ詳細なアクセス権を付与できます。 ユーザーまたは制限付きユーザーにアドオン マネージャー アクセス権を付与すると、その機能カテゴリに対する<DNT>**Admin**</DNT>権限と同等の権限が付与されます。 他のすべての New Relic 製品に対するユーザー権限または制限付きユーザー権限は引き続き保持されます。 たとえば、社内のソフトウェア エンジニアをユーザーとして設定し、 <DNT>**APM manager**</DNT>ロールを割り当てて<InlinePopover type="apm"/>権限を付与することができます。

## カスタムおよびアドオンの役割を管理する

以下に、管理されたアドオンロールとカスタムアドオンロールの両方を管理するためのオプションを示します。

<CollapserGroup>
  <Collapser
    id="view-roles"
    title="役割を見る"
  >
    [アカウントに割り当てられている個人とその現在の役割](/docs/accounts-partnerships/accounts/account-setup/adding-updating-users)のリストを表示するには、 <DNT>**[user menu](/docs/accounts/accounts-billing/general-account-settings/intro-account-settings) > Account settings > Users and roles**</DNT>に移動します。
  </Collapser>

  <Collapser
    id="assign-role"
    title="マネージドロールの割り当て"
  >
    <Callout variant="tip">
      <DNT>
        **Owner and Admins**
      </DNT>
    </Callout>

    マネージド アドオン ロールは、New Relic の各製品にデフォルトで用意されています。ユーザーにマネージドロールを追加すると、割り当てられた製品の [管理者レベルの権限が付与されます](/docs/accounts-partnerships/accounts/account-billing-usage/users-roles) 。また、編集や削除はできません。自分のアカウントの User または Restricted User にマネージド アドオン ロールを割り当てるには、以下の手順に従います。

    1. <DNT>**[user menu](/docs/accounts/accounts-billing/general-account-settings/intro-account-settings) > Account settings > Users and roles**</DNT>に移動します。

    2. アカウントに関連付けられているユーザーのリストから、ユーザー名を選択します。

    3. <DNT>**Add-on roles**</DNT>の下で、ユーザーの管理者ロールのタイプを選択します。

    4. 追加できる権限を理解するには、 <DNT>**Capabilities preview**</DNT>チャートを使用します。

       <DNT>**Capabilities preview**</DNT>チャートの機能は、サブスクリプションレベルで利用可能な機能と正確に一致しない可能性があります。

       <Callout variant="tip">
         また、 [CSVファイル](#bulk) を使って、ユーザーの追加、更新、削除を一括して行うことができます。
       </Callout>
  </Collapser>

  <Collapser
    id="create"
    title="カスタムロールの作成"
  >
    自分のアカウントにカスタムアドオンロールを作成するには

    1. <DNT>
         **[user menu](/docs/accounts/accounts-billing/general-account-settings/intro-account-settings) > Account settings > Users and roles > Roles**
       </DNT>

       に移動します。

    2. <DNT>
         **<Icon name="fe-plus-circle"/> New custom add-on role**
       </DNT>

       を選択します。

    3. 新しいカスタム ロールに必要な権限を選択し、

       <DNT>
         **Create role**
       </DNT>

       選択します。
  </Collapser>

  <Collapser
    id="assign-custom"
    title="カスタムロールの割り当て"
  >
    <Callout variant="tip">
      <DNT>
        **Owners and Admins**
      </DNT>
    </Callout>

    ユーザーに割り当てる前に、カスタムロールを作成する必要があります。アカウントのUserまたはRestricted Userにカスタムアドオンロールを割り当てるには、以下の手順に従います。

    1. <DNT>
         **[user menu](/docs/accounts/accounts-billing/general-account-settings/intro-account-settings) > Account settings > Users and roles > Users**
       </DNT>

       に移動します。

    2. アカウントに関連付けられているユーザーのリストから、そのユーザーの名前を選択します ]。

    3. <DNT>
         **Add-on roles**
       </DNT>

       の下で、ユーザーのカスタム ロールを選択します。

    4. <DNT>
         **Update user**
       </DNT>

       をクリックします。
  </Collapser>

  <Collapser
    id="edit"
    title="カスタムロールの編集・削除"
  >
    <Callout variant="tip">
      <DNT>
        **Owners and Admins**
      </DNT>
    </Callout>

    New Relic のデフォルトのロールを編集または削除することはできません。しかし、自分のアカウントのカスタムアドオンロールを編集または削除することは可能です。

    1. <DNT>
         **[user menu](/docs/accounts/accounts-billing/general-account-settings/intro-account-settings) > Account settings > Users and roles > Roles**
       </DNT>

       に移動します。

    2. <DNT>
         **Add-on roles**
       </DNT>

       リストからカスタム アドオン ロールを選択し、必要に応じて

       <DNT>
         **<Icon name="fe-edit-2"/> Edit role**
       </DNT>

       または

       <DNT>
         **<Icon name="fe-trash-2"/> Delete role**
       </DNT>

       を選択します。
  </Collapser>
</CollapserGroup>

## ユーザー一括管理 [#bulk]

<DNT>**Bulk user actions**</DNT>機能を使用すると、複数のユーザーを一度に追加、更新、または削除できます。 これは次の場合に役立ちます:

* 複数の新入社員が入社した際のロールの追加
* 複数の社員が退職する際のロールの削除
* 複数の社員にAdminの役割を与える

### ユーザーを一括で更新 [#update]

一括してユーザーアクションを行う際の重要なルールと推奨事項をご紹介します。

* 自分のロールやOwnerのロールを更新することはできません。
* 既存のユーザーのEメールアドレスや名前を編集することはできません。
* 既存のユーザーを削除して再度追加するような編集は、意図しない結果を招く可能性があるため、避けるべきです（例えば、元のユーザーに関連付けられていたAPIキーが失われます）。

[オリジナル・ユーザー・モデル](/docs/accounts/original-accounts-billing/original-product-based-pricing/overview-user-models) のユーザーに対して、新しいユーザー・ロールの追加、既存のユーザー・ロールの更新、またはユーザー・ロールの削除を行うには、次のようにします。

1. <DNT>**[user menu](/docs/accounts/accounts-billing/general-account-settings/intro-account-settings) > Account settings > Users and roles**</DNT>に移動し、URL の末尾に`/bulk_actions`を追加します。

   例のURLです。

   ```
   https://account.newrelic.com/accounts/123456789/users/bulk_actions
   ```

2. <DNT>**Backup CSV file**</DNT>をダウンロードします。 バックアップ ファイルをダウンロードすると、変更が行われる前のアカウント内のユーザーの記録が保持され、誤って削除された可能性のあるユーザーを簡単に再追加できるようになります。

3. <DNT>**CSV of users**</DNT>または<DNT>**CSV template**</DNT>をダウンロードします。 各一括アクション (追加、更新、削除) には、独自の CSV ファイルが必要です。 New Relic では、アカウント番号、日付、実行された一括アクションをファイルに保存することをお勧めします。 例えば： `account_123456789_delete_users_2018-06-29`

4. 選択した一括アクションを適用するロールを持つユーザーのみを、このシートに入力します。ロールを変更したくないユーザーをスプレッドシートから削除します。

### バルクユーザー管理のトラブルシューティング [#re-add]

<Callout variant="important">
  ユーザーを削除して再度追加すると、関連するパーミッションが失われることがありますのでご注意ください。例えば、関連するAPIキーは再度追加する必要があります。
</Callout>
