モニターOpenTelemetry Collectorインストールして構成することで、Linux ホスト上で実行されるRabbitMQ 。 この包括的なガイドでは、Debian、Ubuntu、RHEL、および CentOS システムのインストール、設定、検証について説明します。

## インストレーション手順 [#install]

セルフホスト型 RabbitMQ インスタンスの監視を設定するには、次の手順に従います。

<Steps>
  <Step>
    ### あなたが始める前に [#requirements]

    環境が次の要件を満たしていることを確認してください。

    <CollapserGroup>
      <Collapser id="rabbitmq-reqs" title="RabbitMQの要件">
        * **バージョン**: RabbitMQ 3.8 以上
        * **管理プラグイン**: APIを公開するには有効にする必要があります
        * **ユーザーアカウント**: 管理APIへのアクセス権限を持つ Adminユーザー
      </Collapser>

      <Collapser id="system-reqs" title="システム要求">
        * **OS** : Debian、Ubuntu、RHEL、または CentOS
        * **シェルアクセス**: sudo権限を持つユーザー
        * **ネットワーク**: New Relic の[OTLP エンドポイント](https://docs.newrelic.com/docs/opentelemetry/best-practices/opentelemetry-otlp/#configure-endpoint-port-protocol)へのアウトバウンド HTTPS 接続
        * **リソース**: コレクター用に最低 512MB の RAM と 1 つの CPU コアが使用可能
      </Collapser>

      <Collapser id="newrelic-reqs" title="New Relicの要件">
        * **アカウント**: アクティブな New Relic アカウント
        * **ライセンスキー**: アカウントの[New Relicキー](/docs/apis/intro-apis/new-relic-api-keys/#license-key)
      </Collapser>

      <Collapser id="software-reqs" title="ソフトウェア要件">
        * **OpenTelemetry Collector** : Contrib バージョン 0.140.0 以上
        * **システム パッケージ**: `wget` 、 `systemd` (通常はプリインストール済み)
      </Collapser>
    </CollapserGroup>
  </Step>

  <Step>
    ### RabbitMQ管理プラグインを有効にする [#enable-plugin]

    管理プラグインは、OpenTelemetry Collector がスクレイピングする API を公開します。

    ```bash
    sudo rabbitmq-plugins enable rabbitmq_management
    ```

    動作していることを確認します:

    ```bash
    curl -I -u admin:password http://localhost:15672/api/overview
    ```

    応答に`HTTP/1.1 200 OK`が表示されます。
  </Step>

  <Step>
    ### OpenTelemetry Collectorをインストールする [#install-collector]

    すべての Linux ディストリビューションの場合:

    [OpenTelemetry Collectorリリース](https://github.com/open-telemetry/opentelemetry-collector-releases/releases/latest)から、ホスト OS 用のOpenTelemetry Collector Contrib バイナリをダウンロードしてインストールします。
  </Step>

  <Step>
    ### コレクターを構成する [#configure]

    `/etc/otelcol-contrib/config.yaml`を作成または置換:

    ```yaml
    receivers:
      rabbitmq:
        endpoint: http://localhost:15672
        username: admin
        password: password
        collection_interval: 30s
        metrics:
          # Queue Metrics (essential for message flow and backlog)
          rabbitmq.consumer.count:
            enabled: true
          rabbitmq.message.delivered:
            enabled: true
          rabbitmq.message.published:
            enabled: true
          rabbitmq.message.acknowledged:
            enabled: true
          rabbitmq.message.dropped:
            enabled: true
          rabbitmq.message.current:
            enabled: true # Crucial for monitoring queue backlog, includes 'ready' and 'unacknowledged' states

          # Node Health Metrics (critical for server resource monitoring)
          rabbitmq.node.disk_free:
            enabled: true
          rabbitmq.node.disk_free_limit:
            enabled: true
          rabbitmq.node.disk_free_alarm:
            enabled: true
          rabbitmq.node.mem_used:
            enabled: true
          rabbitmq.node.mem_limit:
            enabled: true
          rabbitmq.node.mem_alarm:
            enabled: true
          rabbitmq.node.mem_used_details.rate:
            enabled: true
          rabbitmq.node.fd_used:
            enabled: true
          rabbitmq.node.fd_total:
            enabled: true
          rabbitmq.node.sockets_used:
            enabled: true
          rabbitmq.node.sockets_total:
            enabled: true
          rabbitmq.node.proc_used:
            enabled: true
          rabbitmq.node.proc_total:
            enabled: true
          rabbitmq.node.uptime:
            enabled: true
          rabbitmq.node.run_queue:
            enabled: true
          rabbitmq.node.processors:
            enabled: true
          rabbitmq.node.context_switches_details.rate:
            enabled: true
          rabbitmq.node.gc_num_details.rate:
            enabled: true
          rabbitmq.node.gc_bytes_reclaimed_details.rate:
            enabled: true

          # I/O Metrics (important for understanding disk and network activity)
          rabbitmq.node.io_read_count_details.rate:
            enabled: true
          rabbitmq.node.io_read_bytes_details.rate:
            enabled: true
          rabbitmq.node.io_read_avg_time_details.rate:
            enabled: true
          rabbitmq.node.io_write_count_details.rate:
            enabled: true
          rabbitmq.node.io_write_bytes_details.rate:
            enabled: true
          rabbitmq.node.io_write_avg_time_details.rate:
            enabled: true
          rabbitmq.node.io_sync_count_details.rate:
            enabled: true
          rabbitmq.node.io_sync_avg_time_details.rate:
            enabled: true
          rabbitmq.node.io_seek_count_details.rate:
            enabled: true
          rabbitmq.node.io_seek_avg_time_details.rate:
            enabled: true
          rabbitmq.node.io_reopen_count_details.rate:
            enabled: true

          # Mnesia and Store Metrics (for internal database and message storage)
          rabbitmq.node.mnesia_ram_tx_count_details.rate:
            enabled: true
          rabbitmq.node.mnesia_disk_tx_count_details.rate:
            enabled: true
          rabbitmq.node.msg_store_read_count_details.rate:
            enabled: true
          rabbitmq.node.msg_store_write_count_details.rate:
            enabled: true
          rabbitmq.node.queue_index_write_count_details.rate:
            enabled: true
          rabbitmq.node.queue_index_read_count_details.rate:
            enabled: true

          # Connection/Channel/Queue Lifecycle Metrics
          rabbitmq.node.connection_created_details.rate:
            enabled: true
          rabbitmq.node.connection_closed_details.rate:
            enabled: true
          rabbitmq.node.channel_created_details.rate:
            enabled: true
          rabbitmq.node.channel_closed_details.rate:
            enabled: true
          rabbitmq.node.queue_declared_details.rate:
            enabled: true
          rabbitmq.node.queue_created_details.rate:
            enabled: true
          rabbitmq.node.queue_deleted_details.rate:
            enabled: true

    # processors: Process data before exporting.
    processors:
      resourcedetection:
        detectors: [system]
        system:
          resource_attributes:
            host.name:
              enabled: true
            host.id:
              enabled: true
      resource:
        attributes:
          - key: instrumentation.provider
            value: opentelemetry
            action: upsert
          - key: rabbitmq.deployment.name
            value: my-rabbitmq-server  # Replace with your server name
            action: upsert
      batch:
        send_batch_size: 1024
        timeout: 30s

    exporters:
      otlphttp/newrelic:
        endpoint: ${env:NEWRELIC_OTLP_ENDPOINT}
        headers:
          api-key: ${env:NEWRELIC_LICENSE_KEY}
        compression: gzip

    service:
      pipelines:
        metrics:
          receivers: [rabbitmq]
          processors: [resourcedetection, resource, batch]
          exporters: [otlphttp/newrelic]
    ```

    設定で次の値を更新します。

    * `endpoint`: RabbitMQ 管理 API URL (デフォルト: `http://localhost:15672`)
    * `username` `password` : RabbitMQ の認証情報
    * `rabbitmq.deployment.name`: このRabbitMQの一意の名前
    * `collection_interval`: メトリクスをスクレイピングする頻度（デフォルト: 30秒）
  </Step>

  <Step>
    ### 環境変数の設定 [#env-vars]

    環境変数の systemd オーバーライド ファイルを作成します。

    ```bash
    sudo mkdir -p /etc/systemd/system/otelcol-contrib.service.d
    ```

    New Relic のリージョンを選択してください:

    <Tabs>
      <TabsBar>
        <TabsBarItem id="us">米国地域</TabsBarItem><TabsBarItem id="eu">EU地域</TabsBarItem>
      </TabsBar>

      <TabsPages>
        <TabsPageItem id="us">
          ```bash
          cat <<EOF | sudo tee /etc/systemd/system/otelcol-contrib.service.d/environment.conf
          [Service]
          Environment="NEWRELIC_OTLP_ENDPOINT=https://otlp.nr-data.net:4318"
          Environment="NEWRELIC_LICENSE_KEY=YOUR_LICENSE_KEY"
          EOF
          ```

          `YOUR_LICENSE_KEY` [New Relicライセンスキー](/docs/apis/intro-apis/new-relic-api-keys/#license-key)に置き換えます。
        </TabsPageItem>

        <TabsPageItem id="eu">
          ```bash
          cat <<EOF | sudo tee /etc/systemd/system/otelcol-contrib.service.d/environment.conf
          [Service]
          Environment="NEWRELIC_OTLP_ENDPOINT=https://otlp.eu01.nr-data.net:4318"
          Environment="NEWRELIC_LICENSE_KEY=YOUR_LICENSE_KEY"
          EOF
          ```

          `YOUR_LICENSE_KEY` [New Relicライセンスキー](/docs/apis/intro-apis/new-relic-api-keys/#license-key)に置き換えます。
        </TabsPageItem>
      </TabsPages>
    </Tabs>

    systemd をリロードし、コレクターを再起動します。

    ```bash
    sudo systemctl daemon-reload
    sudo systemctl restart otelcol-contrib
    ```
  </Step>

  <Step>
    ### New Relicでデータを検証する [#verify]

    サービスステータスを確認します。

    ```bash
    sudo systemctl status otelcol-contrib
    ```

    出力には`active (running)`が表示されます。

    最近のログを表示:

    ```bash
    sudo journalctl -u otelcol-contrib -n 50 --no-pager
    ```

    メトリクス収集が成功したことを示すメッセージを探します。

    ```
    INFO    RabbitmqReceiver        Successfully scraped rabbitmq metrics
    ```

    New Relicでメトリクスを検証する:

    データが表示されるまで 2 ～ 3 分待ってから、[書き込みビルダー](/docs/query-your-data/explore-query-data/query-builder/introduction-query-builder)で次のコマンドを実行します。

    ```sql
    SELECT count(*)
    FROM Metric
    WHERE metricName LIKE 'rabbitmq.%'
      AND instrumentation.provider = 'opentelemetry'
      AND rabbitmq.deployment.name = 'my-rabbitmq-server'
    FACET metricName
    SINCE 10 minutes ago
    ```

    次のようなメトリクス名が表示されるはずです。

    * `rabbitmq.queue.count` - キューの数
    * `rabbitmq.queue.message.count` - キュー全体のメッセージの合計
    * `rabbitmq.connection.count` - アクティブな接続
    * `rabbitmq.consumer.count` - アクティブな消費者

    <Callout variant="tip">
      5 分経ってもデータが表示されない場合は、以下の[トラブルシューティングのセクション](#troubleshoot)を確認してください。
    </Callout>
  </Step>

  <Step>
    ### (オプション) RabbitMQをNew Relicに転送する [#logs]

    Metrix に加えて、 RabbitMQログ アプリケーションをNew Relicに転送して、包括的な観察を行うことができます。 これは、メトリクスとログ全体で問題を相互に関連付けるのに役立ちます。

    <Callout variant="important">
      ログ転送には追加の設定権限とシステム権限が必要です。 続行する前に必要なアクセス権があることを確認してください。
    </Callout>

    ファイルログ レシーバーを追加するには、 **`/etc/otelcol-contrib/config.yaml`を更新します**。

    ```yaml
    receivers:
      rabbitmq:
        # ... existing rabbitmq config ...

      # Add filelog receiver for RabbitMQ logs
      filelog/rabbitmq:
        include:
          - /var/log/rabbitmq/*.log
          - /var/log/rabbitmq/**/*.log
        include_file_path: true
        include_file_name: false
        operators:
          - type: regex_parser
            regex: '^(?P<timestamp>\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2}\.\d{3}) \[(?P<level>\w+)\] <(?P<pid>[^>]+)> (?P<message>.*)$'
            timestamp:
              parse_from: attributes.timestamp
              layout: '%Y-%m-%d %H:%M:%S.%L'
            severity:
              parse_from: attributes.severity

    processors:
      # ... existing processors ...

      transform/rabbitmq_logs:
        log_statements:
          - context: resource
            statements:
              - set(attributes["rabbitmq.display.name"], Concat(["server", attributes["rabbitmq.deployment.name"]], ":"))

    service:
      pipelines:
        metrics:
          receivers: [rabbitmq]
          processors: [resourcedetection, resource, batch]
          exporters: [otlphttp/newrelic]

        logs/rabbitmq:
          receivers: [filelog]
          processors: [batch, resource, transform/rabbitmq_logs]
          exporters: [otlphttp/newrelic]
    ```

    コレクターにログの読み取り権限を付与します。1 つの方法を選択してください:

    rabbitmq グループに追加:

    ```bash
    sudo usermod -a -G rabbitmq otelcol-contrib
    ```

    または、ファイルの権限を設定します。

    ```bash
    sudo chmod 644 /var/log/rabbitmq/*.log
    ```

    コレクターを再起動します。

    ```bash
    sudo systemctl restart otelcol-contrib
    ```

    New Relic でログが収集されていることを確認します。

    ```sql
    SELECT count(*)
    FROM Log
    WHERE service.name = 'rabbitmq'
      AND rabbitmq.deployment.name = 'my-rabbitmq-server'
    SINCE 10 minutes ago
    ```
  </Step>
</Steps>

## データを検索する [#find-data]

数分後、 RabbitMQメトリクスがNew Relicに表示されるはずです。 New Relic UIのさまざまなビューでRabbitMQメトリクスを探索する詳細な手順については、 [「データの検索と書き込み」を](/docs/opentelemetry/integrations/rabbitmq/find-and-query-data)参照してください。

NRQL を使用してデータをクエリすることもできます。

```sql
FROM Metric SELECT * WHERE rabbitmq.deployment.name = 'my-rabbitmq-server'
```

## トラブルシューティング [#troubleshoot]

<CollapserGroup>
  <Collapser id="no-data" title="New Relicにデータが表示されない">
    インストール後にNew Relicにメトリクスが表示されない場合:

    **1. 管理プラグインが有効になっていることを確認します。**

    ```bash
    sudo rabbitmq-plugins list | grep management
    ```

    有効になっていて実行中であることを示す`[E*] rabbitmq_management`が表示されます。

    **2. 管理 API をテストします。**

    ```bash
    curl -u admin:password http://localhost:15672/api/overview
    ```

    これにより、RabbitMQ インスタンスに関する JSON データが返されます。そうでない場合、管理プラグインは正しく動作していません。

    **3. コレクターログでエラーを確認します。**

    ```bash
    sudo journalctl -u otelcol-contrib -f
    ```

    探す：

    * 認証エラー（ユーザー名/パスワードが正しくありません）
    * 接続タイムアウト（ネットワークの問題）
    * 解析エラー（設定構文の問題）

    **4. コレクター サービスが実行されていることを確認します。**

    ```bash
    sudo systemctl status otelcol-contrib
    ```

    `active (running)`と表示されるはずです。そうでない場合は、起動に失敗した理由をログで確認してください。

    **5. ライセンスキーを確認します。**

    ```bash
    sudo cat /etc/systemd/system/otelcol-contrib.service.d/environment.conf
    ```

    `NEWRELIC_LICENSE_KEY`が正しく設定されており、余分なスペースや引用符が含まれていないことを確認してください。
  </Collapser>

  <Collapser id="connection-refused" title="接続拒否エラー">
    コレクター ログに`connection refused`が表示される場合:

    **RabbitMQ がポート 15672 でリッスンしていることを確認します。**

    ```bash
    sudo netstat -tlnp | grep 15672
    ```

    RabbitMQ がリッスンしているのが見えるはずです。そうでない場合、管理プラグインが有効になっていないか、予期されたポートにバインドされていません。

    **ファイアウォール ルールを確認します。**

    ```bash
    sudo iptables -L -n | grep 15672
    ```

    ファイアウォール ルールによってポート 15672 がブロックされていないことを確認します。

    **エンドポイント構成を確認します。**\
    `/etc/otelcol-contrib/config.yaml`の`endpoint`が RabbitMQ の設定 (ホスト名、ポート) と一致していることを確認してください。
  </Collapser>

  <Collapser id="auth-failures" title="認証失敗（401 認証失敗）">
    `401 Unauthorized`エラーが表示された場合:

    **資格情報を確認します:**\
    `/etc/otelcol-contrib/config.yaml`ユーザー名とパスワードが RabbitMQ ユーザーと一致していることを確認してください。

    **アドミニストレータ権限を確認します。**

    ```bash
    sudo rabbitmqctl list_users
    ```

    ユーザーには`[administrator]`タグが必要です。そうでない場合は、管理者権限を付与します。

    ```bash
    sudo rabbitmqctl set_user_tags your-username administrator
    ```
  </Collapser>

  <Collapser id="high-memory" title="メモリまたはCPUの使用率が高い">
    コレクターが過剰なリソースを消費する場合:

    **バッチサイズを縮小する:**\
    `batch`プロセッサで、 `send_batch_size` 512 から 256 以下に減らします。

    **収集間隔を増やす:**\
    スクレイピングの頻度を減らすには、 `collection_interval` 30 秒から 60 秒以上に変更します。

    **不要なメトリクスを無効にします。**\
    コメントアウト`rabbitmq`受信機の設定には不要なメトリクスがあります。

    例：

    ```yaml
    receivers:
      rabbitmq:
        collection_interval: 60s  # Increased
        metrics:
          rabbitmq.queue.message.count:
            enabled: true
          # Disable less critical metrics
          # rabbitmq.queue.count:
          #   enabled: false
    ```
  </Collapser>

  <Collapser id="log-permissions" title="ログのアクセスが拒否されました">
    ログを収集するときに権限エラーが表示される場合:

    **ファイルアクセスをテストします:**

    ```bash
    sudo -u otelcol-contrib cat /var/log/rabbitmq/rabbit@$(hostname).log
    ```

    これが失敗すると、コレクターユーザーはログファイルを読み取ることができなくなります。

    **ファイルの権限を確認します:**

    ```bash
    ls -la /var/log/rabbitmq/
    ```

    **権限を修正:**\
    コレクター ユーザーをrabbitmq グループに追加します。

    ```bash
    sudo usermod -a -G rabbitmq otelcol-contrib
    sudo systemctl restart otelcol-contrib
    ```
  </Collapser>
</CollapserGroup>

RabbitMQ監視の設定が完了したので、監視機能を強化できます。

**データを探索する:**

**監視の強化:**

* [アラートの作成](/docs/alerts-applied-intelligence/new-relic-alerts/get-started/introduction-alerts)- キューの深さとメッセージのバックログに関するアラートを設定します
* [ダッシュボードの構築](/docs/query-your-data/explore-query-data/dashboards/introduction-dashboards)- RabbitMQ メトリクスを視覚化するカスタムダッシュボードを作成します
