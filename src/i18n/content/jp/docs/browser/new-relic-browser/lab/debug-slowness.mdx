  ラボの各手順は最後の手順に基づいているため、この手順を開始する前に、最後の手順 [*であるアプリケーションのエラーをデバッグするを*](/docs/browser/new-relic-browser/lab/debug-errors)完了していることを確認してください。

アプリケーションの JavaScript エラーを修正した後、あなたとあなたのチームは自信を持っています。ダウンタイムの準備ができて、ソーシャル メディアに向かいますが、Twitter をチェックすると、混乱している顧客が何人かいます。

<img title="unhappy customers" alt="unhappy customers" src="/images/browser-lab-screenshot-tweet.webp" />

ええとああ！あなたの顧客は幸せそうに見えません。New Relic ブラウザー監視を使用して、遅延の原因を発見する時が来ました。

## アプリケーションのデバッグの遅さ

<Steps>
  <Step>
    New Relic ホームページから<DNT>**Browser**</DNT>に移動し、 <DNT>**Relicstaurants**</DNT>アプリケーションを選択します。

    <img title="view relicstaurants" alt="view relicstaurants" src="/images/browser-lab-screenshot-view-relicstaurants.webp" />
  </Step>

  <Step>
    ここには、<DNT>**Page views with JavaScript errors**</DNT>、<DNT>**Core web vitals**</DNT>、<DNT>**User time on the site**</DNT>、<DNT>**Initial page load and route changes**</DNT> などを含む、 browserアプリケーションに関連するすべてのデータが表示されます。

    <img title="Relicstaurants summary" alt="Relicstaurants summary" src="/images/browser-lab-screenshot-relicstaurants-browser-app.webp" />

    <DNT>**largest contentful paint (LCP)**</DNT>に注目してください。

    <img title="LCP" alt="LCP" src="/images/browser-lab-screenshot-LCP.webp" />

    最大コンテンツフル ペイント (LCP) は、Web ページのメイン コンテンツが読み込まれる速度を表します。 理想的には、コンテンツの読み込みに 1 ～ 2 秒以上かかることはありません。 ここでは、サイトの読み込みに 5 秒以上かかっていることがわかります。 ユーザーが不満を言うのも不思議ではありません。

    しかし、この遅延の原因は何ですか?バックエンド？
  </Step>

  <Step>
    下にスクロールして、 <DNT>**Front end vs. back end**</DNT>グラフに注目してください。

    <img title="frontend vs backend graph" alt="frontend vs backend graph" src="/images/browser-lab-screenshot-frontend-vs-backend.webp" />

    <DNT>**Back end (time to first byte) (50%)**</DNT>をクリックしてグラフをフィルタリングし、バックエンドの読み込みにかかる時間を確認します。

    <img title="backend time consumption" alt="backend time consumption" src="/images/browser-lab-screenshot-backend.webp" />

    グラフは、最悪の場合、バックエンドがリクエストを処理するのに最大 140 ミリ秒かかったことを示しています。これはフロントエンドが遅延を引き起こしていることを意味しますか?

    <DNT>**Front end (Window load + AJAX) (50%)**</DNT>をクリックします。

    <img title="frontend time consumption" alt="frontend time consumption" src="/images/browser-lab-screenshot-frontend.webp" />

    そこで問題です！グラフは、遅延がフロントエンドで発生していることを示しています。
  </Step>

  <Step>
    フロントエンドでの遅延の原因を絞り込むには、 <DNT>**Initial page load and route change**</DNT>グラフを詳しく見てください。

    <img title="page load graph" alt="page load graph" src="/images/browser-lab-screenshot-page-load-graph.webp" />
  </Step>

  <Step>
    <DNT>**Initial page load (Window load + AJAX)**</DNT>をクリックします。

    <img title="initial page load" alt="initial page load" src="/images/browser-lab-screenshot-initial-page-load.webp" />

    グラフは、 <DNT>**Initial page load (Window load + AJAX)**</DNT>に 5 ～ 6 秒かかっていることを示しており、これは憂慮すべきことです。
  </Step>

  <Step>
    詳細を表示するには、 <DNT>**Initial page load and route change**</DNT>をクリックしてください。

    <img title="page load details" alt="page load details" src="/images/browser-lab-screenshot-page-load-detail.webp" />

    これにより、 <DNT>**Page views**</DNT>に移動します。

    <img title="page views" alt="page views" src="/images/browser-lab-screenshot-page-views.webp" />
  </Step>

  <Step>
    ページを<DNT>**Most time-consuming**</DNT>で並べ替えます。

    <img title="sort page" alt="sort page" src="/images/browser-lab-screenshot-sort-pages.webp" />

    最初のページの読み込みに約 90% の時間がかかっていることに注意してください。

    <img title="most time consuming pages" alt="most time consuming pages" src="/images/browser-lab-screenshot-most-time-consuming-page.webp" />

    それをクリックして詳細を表示します。

    <img title="most time consuming page detail" alt="most time consuming page detail" src="/images/browser-lab-screenshot-most-time-consuming-page-detail.webp" />

    このページには、 <DNT>**Page view breakdown**</DNT> 、 <DNT>**Median initial page load time**</DNT> 、およびその他の重要な詳細が表示されます。 <DNT>**Page view breakdown**</DNT>グラフは、ページに時間がかかる理由と場所を絞り込むのに役立つため、ここでは特に重要です。 このグラフを詳しく見ると、 <DNT>**Page rendering**</DNT> 5000 ミリ秒もの時間がかかっていることがわかります。

    <img title="page rendering" alt="page rendering" src="/images/browser-lab-screenshot-page-rendering.webp" />

    最初のページのレンダリングにかなりの時間がかかり、アプリケーションが遅くなっていることがわかりました。 次に、 <DNT>**Session traces**</DNT>を観察して、レンダリング プロセスを遅らせている原因を特定します。

    右上隅にある<DNT>**X**</DNT>をクリックして、このビューを終了します。
  </Step>

  <Step>
    左側のナビゲーションから<DNT>**Session traces**</DNT>に移動し、 <DNT>**Page load**</DNT>の降順に並べ替えます。

    <img title="most time consuming pages" alt="most time consuming pages" src="/images/browser-lab-screenshot-session-traces.webp" />
  </Step>

  <Step>
    ここでは、セッショントレースが<DNT>**Page load**</DNT>時間の順に並べ替えられていることがわかります。

    <img title="sorted session traces" alt="sorted session traces" src="/images/browser-lab-screenshot-sorted-session-traces.webp" />
  </Step>

  <Step>
    リストから、最初のものをクリックします。

    <img title="navigate to trace detail" alt="trace detail" src="/images/browser-lab-screenshot-navigate-to-trace-detail.webp" />

    ![並べ替えられたセッション トレースを示す画像](../../../images/nr-browser/navigate-to-trace-detail.webp)

    これにより、 <DNT>**Session traces**</DNT>の詳細ページが表示されます。

    ここでは、その特定のセッションの完全なトレースが表示されます。 このページには、 <DNT>**Backend**</DNT> 、 <DNT>**Dom Processing**</DNT> 、 <DNT>**Page Load**</DNT> 、およびその他のトレース関連情報も表示されます。

    <img title="full trace" alt="full trace" src="/images/browser-lab-screenshot-trace.webp" />
  </Step>

  <Step>
    <DNT>**Page load**</DNT>には予想より時間がかかっていることに注意してください。 負荷の詳細なタイムラインが必要です。 ポインタを左右にスクロールしてタイムラインを調整します。

    <img title="adjust trace timeline" alt="adjust trace timeline" src="/images/browser-lab-screenshot-adjust-timeline.webp" />
  </Step>

  <Step>
    トレースをスクロールしてタイム ウィンドウを移動し、このセッション中の個々のイベントの詳細を確認します。

    <img title="adjust time window" alt="adjust time window" src="/images/browser-lab-screenshot-trace-time-window-detail.webp" />
  </Step>

  <Step>
    特定のイベントに 5 秒以上かかっていることに注意してください。

    <img title="image trace" alt="image trace" src="/images/browser-lab-screenshot-image-trace.webp" />
  </Step>

  <Step>
    イベントをクリックすると詳細が表示されます。

    <img title="image trace detail" alt="image trace detail" src="/images/browser-lab-screenshot-image-trace-detail.webp" />

    イメージですのでご了承ください。特に、読み込みに 5 ～ 6 秒かかり、遅延の原因となっているのはアプリケーションの背景画像です。
  </Step>
</Steps>

これらの調査結果に基づいて、背景画像がここでの犯人であるという仮説を立てます。高解像度で最適化されていない画像は、Web サイトの速度低下の最も一般的な原因です。朗報です！理由がわかったので、問題を解決できます。

## 概要

要約すると、アプリケーションの遅さを観察し、New Relic ブラウザ監視を使用して次のことを行いました。

1. サイトの主要なウェブ バイタルを観察する
2. 速度低下の原因を絞り込む
