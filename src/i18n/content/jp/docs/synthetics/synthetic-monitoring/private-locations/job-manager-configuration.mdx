構成で[環境変数](#environment-variables)を使用して、[合成ジョブ マネージャー](/docs/synthetics/synthetic-monitoring/private-locations/install-job-manager)を構成する方法について説明します。

<Callout variant="important">
  [カスタム モジュール](/docs/synthetics/synthetic-monitors/private-locations/containerized-private-minion-cpm-configuration/#custom-modules)、 [永続的なデータ ストレージ](/docs/synthetics/synthetic-monitors/private-locations/containerized-private-minion-cpm-configuration/#preserve-data-volume)、および[ユーザー定義の環境変数](/docs/synthetics/synthetic-monitors/private-locations/containerized-private-minion-cpm-configuration/#vars-scripted-monitors)は、現時点では合成ジョブ マネージャーではサポートされていません。
</Callout>

注意として、New Relic は、Synthetics ジョブ マネージャー ファイルに加えた変更について責任を負いません。

## Kubernetes と Docker のサイジングに関する考慮事項 [#kubernetes-sizing]

<Callout variant="tip">
  Docker 固有のサイジングに関する考慮事項は、まもなく利用可能になります。
</Callout>

大規模な環境で作業している場合は、合成モニターを効率的に実行するための最小要件を満たすように、ジョブ マネージャーの構成をカスタマイズする必要がある場合があります。次のような多くの要因が、合成ジョブ マネージャーの展開のサイジング要件に影響を与える可能性があります。

* 予想される使用量に基づいてすべてのランタイムが必要な場合
* モニターの種類 (ping、シンプルまたはスクリプト化されたブラウザー、およびスクリプト化された API) ごとの 1 分あたりのジョブ数
* 約 3 分でタイムアウトするジョブを含む、ジョブの所要時間
* ジョブの失敗数。ジョブが失敗した場合、組み込みの 3/3 再試行ロジックを提供するためにモニターが失敗し始めると、自動再試行がスケジュールされます。これらの追加のジョブは、合成ジョブ マネージャーのスループット要件に追加されます。

以下にリストされているサイジング構成設定に加えて、同じプライベート ロケーション キーを使用して追加の合成ジョブ マネージャーを展開し、複数の環境間でジョブの負荷を分散することができます。

<CollapserGroup>
  <Collapser
    id="kubernetes"
    title="Kubernetes"
  >
    K8s 合成ジョブ マネージャーで使用される各ランタイムは、[ヘルム チャート](https://github.com/newrelic/helm-charts/tree/master/charts/synthetics-job-manager)で値を設定することにより、個別にサイズを設定できます。

    `1`のデフォルト値から`ping-runtime.replicaCount`設定を増やすことで、追加の ping ランタイムを開始して、ping モニター負荷の実行を支援できます。

    Node.js API と Node.js ブラウザ ランタイムは、 `parallelism`と`completions`の設定の組み合わせを使用して個別にサイズ設定されます。これらの設定の理想的な構成は、お客様の要件によって異なります。

    `parallelism`設定は、K8s クラスタで同時に実行できるそのタイプ（Node.js API または Node.js ブラウザ）のジョブの数を制御します。`parallelism`設定は、コンテナー化されたプライベート ミニオン (CPM) の`synthetics.heavyWorkers`構成と同等です。[リソースのリクエストと制限の値](/docs/synthetics/synthetic-monitoring/private-locations/install-job-manager/#kubernetes-requirements)に基づいて、この数の同時ランタイムを実行するのに十分なリソースが K8s クラスターにあることを確認してください。

    `completions`設定は、このランタイム タイプの追加のジョブを開始するために 1 分ごとの`CronJob`スケジュールで待機する前に、このタイプのジョブをいくつ完了する必要があるかを制御します。`completions`設定は、 `parallelism`設定を考慮して、1 分間に完了できるジョブ数の見積もりに設定する必要があります。`completions`が 1 より大きい場合、完了したジョブは`kubectl get pods`に表示されますが、それらのジョブが完了または失敗とマークされるとすぐにリソースが解放されます。

    各ランタイムの`completions`と`parallelism`のベースラインの計算に役立つ数式:

    ```m
    completions = 60 / avg job duration
    parallelism = jobs per minute / completions
    ```

    ランタイムが異なれば、ジョブの所要時間とレートも異なる可能性があります。次のクエリを使用して、プライベートな場所の平均期間とレートを取得できます。

    ```sql
    # non-ping average job duration by runtime type
    FROM SyntheticCheck SELECT average(duration) AS 'avg job duration' WHERE type != 'SIMPLE' AND location = 'YOUR_PRIVATE_LOCATION' FACET type SINCE 1 hour ago

    # non-ping jobs per minute by runtime type
    FROM SyntheticCheck SELECT rate(uniqueCount(id), 1 minute) AS 'jobs per minute' WHERE type != 'SIMPLE' AND location = 'YOUR_PRIVATE_LOCATION' FACET type SINCE 1 hour ago
    ```

    <Callout variant="tip">
      上記のクエリは、現在の結果に基づいています。個人の場所に結果がない場合、またはジョブ マネージャーのパフォーマンスが最適でない場合、クエリの結果は正確ではない可能性があります。その場合、 `completions`と`parallelism`にいくつかの異なる値を試して、ジョブが少なくとも 1 分間完了 (十分な完了) で満たされ、キューが大きくならないようになる (十分な並列処理) まで試してください。
    </Callout>

    ジョブの完了に時間がかかる場合、1 分間を満たすために必要な完了数は少なくなりますが、より多くの並列ポッドが必要になります。同様に、1 分あたりにより多くのジョブを処理する必要がある場合は、より多くの並列ポッドが必要になります。

    `completions`の値を使用して、&#x7B; `parallelism` } の値の適切な開始点を取得できます。`parallelism`設定は、1 分間に実行できるジョブの数に直接影響します。値が小さすぎると、キューが大きくなる可能性があります。値が大きすぎると、ノードのリソースが制限される可能性があります。

    `parallelism`設定がうまく機能してキューをゼロに保つ場合は、 `60 / avg job duration`から最初に計算された値よりも高い値を`completions`に設定することは悪い考えではありません。これは、ジョブが予想よりも速く完了する場合に対応し、1 分が経過する前に`completions`値に達するシナリオを回避するのに役立ちます。最も効率的な`completions`設定では、実行中のジョブと完了したジョブで 1 分間が満たされます。十分な完了がない場合、そのランタイムのジョブ処理は、次の`CronJob`が別の`completions`のセットを起動できるようになるまで、数秒間アイドル状態になります。
