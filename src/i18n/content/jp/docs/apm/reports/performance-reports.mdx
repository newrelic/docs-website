当社のパフォーマンス分析レポートは、アプリケーションの正常性とパフォーマンスのシンプルで事前定義されたビューを提供します。詳細については、以下のレポートを選択してください。

<CollapserGroup>
  <Collapser className="freq-link" id="background" title="バックグラウンドジョブ分析">
    <DNT>**Background jobs analysis**</DNT> レポートは、`DelayedJob` や `Resque` など、New Relic によってインストゥルメントされたフレームワークを使用する、本番環境で実行されているジョブやその他の[非ウェブ トランスポート](/docs/apm/transactions/intro-transactions/monitor-background-processes-other-non-web-transactions)New Relicパフォーマンスを分析するのに役立ちます。 レポートには、スループットに費やされた時間 (1 分あたりのリクエスト数または<DNT>**rpm**</DNT> )、ジョブに費やされた合計時間、ジョブの実行にかかった平均時間など、過去 24 時間から過去 7 日間までの比較情報が表示されます。

    ## UIでのレポート表示 [#procedures]

    アプリの<DNT>**Background jobs analysis**</DNT>レポートを表示するには、 <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &gt; APM &amp; services &gt; (select an app) &gt; Reports &gt; Performance &gt; Non-web transactions**</DNT>タブに移動します。 デフォルトのレポートでは、過去 24 時間とその前の 24 時間、7 日前を比較し、過去 7 日間の平均を算出します。

  <Collapser className="freq-link" id="capacity" title="容量分析">
    APMのキャパシティ分析レポートでは、アプリのインスタンスが何台稼働していて、どのくらいビジー状態なのかが表示されます。この情報は、アプリがリクエスト負荷をサポートするのに十分なインスタンスを配置しているかどうかを分析するのに役立ちます。また、この情報を利用して、最適なパフォーマンスを得るために構成を微調整することができます。

    容量分析レポートは、 [Python](#agent-capacity-analysis) と Ruby 言語のエージェントでのみ利用できます。

    ## 容量分析レポートを見る [#procedures]

    1. <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &gt; APM &amp; services &gt; (select an app) &gt; Reports &gt; Capacity**</DNT>に移動します。
    2. New Relic の標準的な [ページ機能](/docs/accounts-partnerships/education/getting-started-new-relic/standard-dashboard-features) を使って、詳細な情報にドリルダウンすることができます。

    ## Pythonのシングルまたはマルチスレッドアプリ [#agent-capacity-analysis]

    Python エージェントの場合、キャパシティ分析のレポートは、従来のシングルスレッドまたはマルチスレッドのアプリケーションに対してのみ機能します。これは、New Relic がシステム負荷を計算する方法が、コルーチンベースのシステムには適用されないためです。

    従来のシングルまたはマルチスレッドのアプリケーションを使用している場合、Pythonエージェントによって生成されたメトリック情報は、オプションのC拡張コンポーネントがコンパイルおよびインストールできる場合にのみ取得され、レポートされます。geventやeventletのようなコルーチンベースのシステムをWSGIサーバーと組み合わせて使用する場合、容量分析レポートにはエージェントの再起動以外のデータは表示されません。
  </Collapser>

  <Collapser className="freq-link" id="scalability" title="スケーラビリティ分析">
    APM のスケーラビリティ分析レポートには、アプリケーション負荷に関連した応答時間、データベース時間、および CPU 時間が表示されます。これは、パフォーマンスを評価し、アプリの成長に伴う潜在的なハードウェア要件を特定するのに役立ちます。

    ## スケーラビリティ分析レポートを見る [#procedures]

    自分のアプリのレポートを表示するには

    <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &gt; APM &amp; services &gt; (select an app) &gt; Reports &gt; Scalability**</DNT>に移動します。

    グラフには 2 つの線があります。データ ポイントを結ぶ主線と、「平滑化」とラベル付けされ、主線の平滑化バージョンを示す副線です。セカンダリ ラインは、プライマリ ラインで観察される予期せぬ変動を気にすることなく、アプリケーションのパフォーマンスを視覚化するのに役立ちます。

    これらのチャートは他の New Relic チャートとは少し異なる方法で構築されており、一般的に使用できるデフォルトのチャート アクションはありません。

    ## 時間帯の選択 [#time]

    スケーラビリティ レポートには、さまざまな期間にわたるデータを表示できる [時間ピッカーが](/docs/query-your-data/explore-query-data/dashboards/manage-your-dashboard/#dash-time-picker) 含まれています。アプリケーションのスケーラビリティを分析する場合、これより短い時間範囲では全体像が得られないため、推奨される最小期間は 30 分です。時間ピッカーは、過去 30 分のデフォルト値を使用します。

    ## 個々のインスタンスのデータを表示する [#instances]

    各タブには、特定のインスタンスを選択できるインスタンス ピッカーが含まれており、グラフとビルボードがそのインスタンスのデータにフィルターされます。

    ### アプリケーションサーバーメトリクス [#nrql-app-server]

    このグループの各クエリは、`SINCE '<monday>' UNTIL '<next_monday>'`を使用して週の範囲を指定し、`appName = '<app_name>'`を使用してアプリを選択することで、1つのアプリケーションの単一の週次値をレポートします。

    **Requests（100万）:** その週に処理されたHTTP requestsの総数（100万単位）。

    ```sql
    SELECT sum(newrelic.timeslice.value[count]) / 1e6 AS 'Requests millions'
    FROM Metric
    WHERE metricTimesliceName = 'HttpDispatcher' AND appName = '<app_name>'
    SINCE '<monday>' UNTIL '<next_monday>'
    ```

    **レスポンスタイム（ms）:** すべてのHTTPトランザクション全体の平均応答時間（ミリ秒単位）。

    ```sql
    SELECT sum(newrelic.timeslice.value) / sum(newrelic.timeslice.value[count]) * 1000 AS 'Resp time ms'
    FROM Metric
    WHERE metricTimesliceName = 'HttpDispatcher' AND appName = '<app_name>'
    SINCE '<monday>' UNTIL '<next_monday>'
    ```

    **スループット（RPM）:** 1週間の1分あたりの平均requests。

    ```sql
    SELECT sum(newrelic.timeslice.value[count]) / 10080 AS 'Throughput RPM'
    FROM Metric
    WHERE metricTimesliceName = 'HttpDispatcher' AND appName = '<app_name>'
    SINCE '<monday>' UNTIL '<next_monday>'
    ```

    **エラー率（％）:** すべてのウェブトランザクションと非ウェブトランザクションに占めるエラーの割合。

    ```sql
    SELECT filter(sum(newrelic.timeslice.value[count]), WHERE metricTimesliceName = 'Errors/all') /
      (filter(sum(newrelic.timeslice.value[count]), WHERE metricTimesliceName = 'HttpDispatcher') +
       filter(sum(newrelic.timeslice.value[count]), WHERE metricTimesliceName = 'OtherTransaction/all')) * 100 AS 'Error rate'
    FROM Metric
    WHERE metricTimesliceName IN ('Errors/all', 'HttpDispatcher', 'OtherTransaction/all')
      AND appName = '<app_name>'
    SINCE '<monday>' UNTIL '<next_monday>'
    ```

    **Apdex:** アプリケーションの週次[Apdex](/docs/apm/new-relic-apm/apdex/apdex-measuring-user-satisfaction)スコア。

    ```sql
    SELECT (sum(newrelic.timeslice.value[count]) + sum(newrelic.timeslice.value[total]) / 2.0) /
      (sum(newrelic.timeslice.value[count]) + sum(newrelic.timeslice.value[total]) + sum(newrelic.timeslice.value[exclusiveTotal])) AS 'Apdex'
    FROM Metric
    WHERE metricTimesliceName = 'Apdex' AND appName = '<app_name>'
    SINCE '<monday>' UNTIL '<next_monday>'
    ```

    **満足（％）/容認（％）/不満（％）:** パーセンテージで表した3つのApdexコンポーネント。これは、基礎となるリクエストカテゴリに分割することで、上記の単一のApdexスコアを補完します。

    ```sql
    SELECT sum(newrelic.timeslice.value[count]) /
      (sum(newrelic.timeslice.value[count]) + sum(newrelic.timeslice.value[total]) + sum(newrelic.timeslice.value[exclusiveTotal])) * 100 AS '% Satisfied',
      sum(newrelic.timeslice.value[total]) /
      (sum(newrelic.timeslice.value[count]) + sum(newrelic.timeslice.value[total]) + sum(newrelic.timeslice.value[exclusiveTotal])) * 100 AS '% Tolerating',
      sum(newrelic.timeslice.value[exclusiveTotal]) /
      (sum(newrelic.timeslice.value[count]) + sum(newrelic.timeslice.value[total]) + sum(newrelic.timeslice.value[exclusiveTotal])) * 100 AS '% Frustrated'
    FROM Metric
    WHERE metricTimesliceName = 'Apdex' AND appName = '<app_name>'
    SINCE '<monday>' UNTIL '<next_monday>'
    ```

    **CPU（％）:** パーセンテージで表した、1週間の平均CPUユーザー時間。

    ```sql
    SELECT sum(newrelic.timeslice.value) / 604800 * 100 AS 'CPU %'
    FROM Metric
    WHERE metricTimesliceName = 'CPU/User Time' AND appName = '<app_name>'
    SINCE '<monday>' UNTIL '<next_monday>'
    ```

    **メモリ（MB）:** メガバイト単位の平均物理メモリ使用量。

    ```sql
    SELECT average(newrelic.timeslice.value) AS 'Memory MB'
    FROM Metric
    WHERE metricTimesliceName = 'Memory/Physical' AND appName = '<app_name>'
    SINCE '<monday>' UNTIL '<next_monday>'
    ```

    **DB（％）:** 合計ウェブトランザクションタイムに対する、ウェブデータストア呼び出しに費やした時間の割合。

    ```sql
    SELECT filter(sum(newrelic.timeslice.value), WHERE metricTimesliceName = 'Datastore/allWeb') /
      filter(sum(newrelic.timeslice.value), WHERE metricTimesliceName = 'HttpDispatcher') * 100 AS 'DB %'
    FROM Metric
    WHERE metricTimesliceName IN ('Datastore/allWeb', 'HttpDispatcher') AND appName = '<app_name>'
    SINCE '<monday>' UNTIL '<next_monday>'
    ```

    <Callout title="注記">
      データベースの割合は、ウェブトランザクションの合計レスポンスタイムに対するデータベースオペレーション時間の比率を表しています。計算式：（データベース時間/合計レスポンスタイム）×100。100％を超える割合は、データベース呼び出しの並列実行または非同期実行を示します。
    </Callout>

    ### Traffic パターン [#nrql-traffic]

    これらのクエリは、レポートの上部に表示される2か月間の日次スパークラインを生成します。`SINCE 2 months ago`よりも`TIMESERIES 1 day`を使用します。アプリがどのようにインストゥルメントされているかに応じて、2つのバリアントがあります：アプリサーバーアプリは`HttpDispatcher`タイムスライス（rpm）をキーとし、ブラウザでインストゥルメントされたアプリは`EndUser`タイムスライス（ppm）を使用します。

    **スループット — アプリケーションサーバーアプリ（rpm）：**`HttpDispatcher`のタイムスライスを報告するアプリの1日あたりの1分あたりのリクエスト率。

    ```sql
    SELECT rate(sum(newrelic.timeslice.value), 1 minute) AS 'Throughput rpm'
    FROM Metric
    WHERE metricTimesliceName = 'HttpDispatcher' AND entity.guid = '<APP_GUID>'
    SINCE 2 months ago TIMESERIES 1 day
    ```

    **スループット — ブラウザでインストゥルメントされたアプリ（ppm）:** `EndUser`タイムスライスを報告するアプリの1日あたりの1分間のページロード率。

    ```sql
    SELECT rate(sum(newrelic.timeslice.value), 1 minute) AS 'Throughput ppm'
    FROM Metric
    WHERE metricTimesliceName = 'EndUser' AND entity.guid = '<APP_GUID>'
    SINCE 2 months ago TIMESERIES 1 day
    ```

    ### エンドユーザーのブラウザメトリクス [#nrql-enduser]

    これらのクエリは、[ブラウザ監視](/docs/browser/browser-monitoring/getting-started/introduction-browser-monitoring)が関連付けられたアプリケーションのレポートのブラウザセクションにデータを提供します。`TIMESERIES 1 week`を使用して、直近`{N}`週間の週ごとに1つのポイントを返します。

    **エンドユーザーのレスポンスタイム(s):** Browserエージェントからの平均ページロードタイム。

    ```sql
    SELECT average(newrelic.timeslice.value) AS 'End User Resp Time (s)'
    FROM Metric
    WHERE metricTimesliceName = 'EndUser' AND entity.guid = '<APP_GUID>'
    SINCE {N} weeks ago TIMESERIES 1 week
    ```

    **エンドユーザーApdex（ページロード）:** エンドユーザーのページロードのApdexスコア。

    ```sql
    SELECT apdex(newrelic.timeslice.value) AS 'End User Apdex'
    FROM Metric
    WHERE metricTimesliceName = 'EndUser/Apdex' AND entity.guid = '<APP_GUID>'
    SINCE {N} weeks ago TIMESERIES 1 week
    ```

    ## 選択されたアプリケーション [#featured\_apps][#featured_apps]

    New Relicでは、これらの条件を満たす通話回数の多いアプリを上位20位まで選出しています。

    * アプリは最低でも25リクエスト/分（rpm）のスループットが必要です。
    * アプリにはウェブトランザクションが含まれている必要があります。

    ブラウザの監視機能が連動しているアプリケーションは、コール数が少なくても、連動していないアプリケーションよりも優先されます。
