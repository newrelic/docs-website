Para alertas da New Relic, nossa configuração de período de carência controla quanto tempo deve passar antes que você seja notificado sobre um problema. Nossa configuração de problemas inativos controla quanto tempo você deseja esperar antes que um problema inativo seja fechado automaticamente.

Essas configurações se aplicam a todos os dados que chegam à sua conta.

Para encontrar suas configurações de alertas, acesse <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &gt; Alerts &gt; General**</DNT>.

## Política de período de carência para contas que usam decisões [#grace-period]

Para problemas candidatos à nossa lógica de correlação, você pode definir um período de carência. O período de carência é o tempo entre o momento em que um problema é criado e o momento em que esse problema gera a notificação. Os períodos de carência são utilizados para reduzir o número de notificações desnecessárias.

Para cada nível de prioridade, você pode definir o período de carência de 0 a 20 minutos. Quanto maior o período de carência, mais tempo nosso sistema terá para analisar e correlacionar novas atividades com atividades passadas.

Caso haja alguma notificação a ser enviada, ela será enviada decorrido o prazo de carência.

Os valores padrão do período de notas são:

* Baixo: 15 minutos
* Médio: 10 minutos
* Alto: 5 minutos
* Crítico: 1 minuto

<DNT>
  **Note: Grace period is eligible only for users who use correlations**
</DNT>

## Janela de problema de fechamento automático [#auto-close-issue-window]

A configuração de fechamento automático de problemas controla quanto tempo decorre antes que um problema inativo seja fechado automaticamente. O intervalo é de 1 a 30 dias e o valor padrão é 3 dias.

<Callout variant="tip">
  Esta configuração está relacionada à [configuração &quot;fechar incidente aberto após...&quot;](/docs/alerts-applied-intelligence/new-relic-alerts/alert-incidents/how-alert-condition-incidents-are-closed/#time-limit) para cada condição.

  Quando os períodos de tempo nessas duas configurações são diferentes, nosso sistema utiliza o período de tempo mais curto, independentemente da configuração. Por exemplo, se a configuração de tempo de fechamento do incidente aberto for 3 dias e a configuração de tempo de problema inativo for 2 dias, o problema (juntamente com todos os incidentes associados) será encerrado em 2 dias.
</Callout>

## Problema inativo [#inactiv-issue]

Quando um problema está inativo, ele não será agregado ou correlacionado pelo nosso sistema. Por exemplo, se surgir um novo problema semelhante ao problema inativo, ele será tratado como novo, em vez de correlacionado ao problema inativo.

Um problema é considerado inativo quando coleta 5.000 incidentes.

Após 30 dias, um problema inativo será automaticamente encerrado.

## Alertas entre contas [#cross-account-alerts]

As configurações de alertas entre contas permitem que você configure uma condição do alerta para dados de destino reportados a uma conta diferente daquela que possui a condição.
