Embora a configuração de amostragem padrão funcione para todos os serviços, ela pode não ser a ideal. Os agentes de APM fornecem opções de configuração de amostragem para que você possa personalizar a amostragem de acordo com as necessidades do seu serviço. Esta página apresenta alguns exemplos reais de diferentes tipos de serviços e como você pode modificar a configuração de amostragem para atender a essas necessidades.

## Exemplo 1: Má distribuição de endpoints capturados devido à amostragem upstream

A configuração padrão utiliza o amostrador adaptativo. Isso significa que qualquer trace de entrada que foi amostrado por um serviço upstream também será amostrado pelo serviço atual. Às vezes, isso pode fazer com que um endpoint específico seja muito amostrado, enquanto outros endpoints são negligenciados e quase não são amostrados.

### Solução 1: Use o amostrador de proporção para amostrar traces amostrados por um serviço upstream

A configuração a seguir resolverá este problema ao amostrar qualquer trace de entrada que tenha sido amostrado por um serviço upstream apenas 10% das vezes. Isso abre espaço para que outros endpoints sejam amostrados, não preenchendo o reservatório de spans com traces que foram amostrados por um serviço upstream.

```json
{
    distributed_tracing: {
        sampler: {
            remote_parent_sampled: {
                trace_ratio_id_based: {
                    ratio: 0.1
                }
            }
        }
    }
}
```

### Solução 2: Use o amostrador de proporção para capturar uma parte dos traces amostrados por um serviço upstream com granularidade total e capturar o restante com granularidade parcial

Se você ainda precisar capturar informações mínimas sobre um trace quando ele for amostrado por um serviço upstream, você pode usar a seguinte configuração. Esta configuração captura 5% de todos os traces amostrados por um serviço upstream com granulosidade total e o restante com granulosidade parcial. Ao capturar a maioria dos traces amostrados por um serviço upstream com granularidade parcial, o reservatório de spans não enche tão rapidamente e deixa espaço para que outros endpoints sejam amostrados.

```json
{
    distributed_tracing:{
        sampler: {
            remote_parent_sampled: {
                trace_ratio_id_based: {
                    ratio: 0.5
                }
            }
            partial_granularity: {
                enabled: true,
                type: "compact",  # If capturing AI data set this to "essential"
                remote_parent_sampled: always_on
            }
        }
    }
}
```

## Exemplo 2: Má distribuição de endpoints capturados devido a alguns endpoints serem raramente chamados

A configuração padrão utiliza o amostrador adaptativo. Isso significa que ele captura cerca de 10 traces por minuto. Devido à probabilidade aleatória, o amostrador adaptativo terá maior probabilidade de capturar chamadas de endpoint que ocorrem com mais frequência. Isso significa que endpoints raramente chamados têm maior probabilidade de serem perdidos e não amostrados.

### Solução 1: Use o amostrador de proporção para amostrar traces amostrados por um serviço upstream

A configuração a seguir resolverá este problema ao amostrar qualquer trace de entrada que tenha sido amostrado por um serviço upstream apenas 10% das vezes. Isso abre espaço para que outros endpoints sejam amostrados, não preenchendo o reservatório de spans com traces que foram amostrados por um serviço upstream.

```json
{
    distributed_tracing: {
        sampler: {
            remote_parent_sampled: {
                trace_ratio_id_based: {
                    ratio: 0.1
                }
            }
        }
    }
}
```
