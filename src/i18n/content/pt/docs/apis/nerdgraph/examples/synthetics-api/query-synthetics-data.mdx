Após configurar sua infraestrutura de monitoramento, você poderá utilizar a consulta para recuperar informações sobre sua entidade Sintética. consulta faz requests para buscar dados sobre monitores, localização privada, credenciais e período de inatividade. Este tutorial fornece exemplos de como usar a API do NerdGraph para consultar dados sintéticos de monitoramento.

<Callout variant="tip">
  **Quick Navigation**: use os links abaixo para ir para tipos específicos de consulta:

  * [Consultas gerais de monitores](#query-monitors): consulte todos os monitores ou tipos específicos
  * [Consultas específicas do monitor](#query-monitor-script): recuperar o script de um monitor com script, ou as etapas ordenadas de um monitor de etapas
  * [Suporte a consultas de entidade](#query-private-locations): consultar localizações privadas, credenciais seguras e monitorar períodos de inatividade
  * [Consultas avançadas](#query-guid-mapping): mapear IDs de monitores legados e verificar o status de atualização do runtime
</Callout>

Para conhecer os recursos adicionais de consulta disponíveis para sua entidade Sintético, confira [o tutorial API NerdGraph entidade](/docs/apis/nerdgraph/examples/nerdgraph-entities-api-tutorial/).

    ```graphql

                name
                accountId
                monitorType
                tags {
                  key
                  values

        }
      }
    }
    ```

## Consultas específicas do monitor [#monitor-specific-queries]

  <Collapser id="query-monitor-script" title="Scriptdo monitor de consulta">
    Esta consulta recupera o conteúdo do script usado em uma API com script ou monitor de browser com script. O script contém o código JavaScript que define o comportamento do monitor, como chamadas de API, interações do navegador ou lógica de validação personalizada. Esta consulta funciona apenas com tipos de monitor com script: outros tipos de monitor (simples, ping, etapa, verificação de certificado, links quebrados) retornarão um erro, pois não usam scripts personalizados.

  <Collapser id="query-monitor-steps" title="Etapas do monitor de consulta">
    Esta consulta recupera as etapas configuradas para um monitor de etapas. Os monitores de etapas fornecem monitoramento baseado em browser de várias etapas sem código por meio de uma sequência de ações predefinidas, como navegação, cliques, entradas de formulário e asserções. Cada etapa possui uma posição ordinal, um tipo que define a ação e valores que contêm os dados de configuração da etapa. Esta consulta só funciona com monitores de etapa: outros tipos de monitor retornarão um erro, já que não usam configurações baseadas em etapas.

## Consultas avançadas [#advanced-queries]

<CollapserGroup>
  <Collapser id="query-guid-mapping" title="Consulta para mapear o ID do monitor para o GUID da entidade">
    Esta consulta recupera o GUID da entidade para um monitor Sintético utilizando o ID do monitor. Isso é útil quando você tem o ID numérico legado do monitor e precisa convertê-lo para o formato GUID da entidade, exigido para a maioria das operações do NerdGraph. O GUID da entidade é o identificador moderno usado para atualizações, exclusões e outras tarefas de gerenciamento de monitores, enquanto o ID do monitor é o identificador numérico mais antigo que pode aparecer em URLs ou integrações legadas.

  <Collapser id="query-runtime-upgrade-all" title="status da atualização do runtime do consulta (todos os monitores)">
    Esta consulta recupera o status de todos os testes de atualização de tempo de execução para monitores de tempo de execução legados em sua conta. Esses testes validam se os monitores que usam ambientes de execução mais antigos (como Chrome 72 ou API legada Node.js ) podem ser executados com sucesso em ambientes de execução mais recentes (Chrome 100+ ou Node.js 16.10). Os resultados são exibidos na [interface de atualização de tempo de execução](/docs/synthetics/synthetic-monitoring/using-monitors/runtime-upgrade-ui/) e ajudam você a identificar quais monitores estão prontos para atualização. O resultado do teste é armazenado na tag `validationStatus` e, se o teste de atualização falhar, informações detalhadas sobre o erro estarão disponíveis na tag `validationError`.
