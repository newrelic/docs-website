Se você está pensando em usar o New Relic ou já o utiliza, as informações a seguir serão úteis para entender como o New Relic precifica seus produtos e cobra pelo uso.

Para saber mais, confira as opções de faturamento listadas abaixo e em nosso [site](https://newrelic.com/pricing).

## Modelo de preços [#pricing-model]

Nosso modelo de preços descreve as funcionalidades existentes do produto e indica as opções de faturamento disponíveis, como Dados + usuário ou Dados + Computador principal para observabilidadestack e Computador avançado para observabilidade inteligente. Compreender esses modelos de precificação ajuda você a escolher o que é melhor para as necessidades do seu negócio.

<Callout title="Observação">
  O modelo [Core computar](/docs/licenses/license-information/product-definitions/new-relic-one-pricing-definitions/#compute-capacity-unit) preços baseados no consumo fornece a toda a sua organização de engenharia acesso ilimitado à plataforma e capacidade de observação full-stack sem barreiras de licenciamento por usuário. Como o Core Computar está atualmente em versão prévia, se você estiver interessado em experimentar o modelo de preços, entre em contato com seu executivo de contas New Relic.
</Callout>

<CollapserGroup>
  <Collapser id="data" title="Dados">
    Refere-se à quantidade de dados ingeridos pelo New Relic, medida em [GB ingeridos](/docs/licenses/license-information/product-definitions/new-relic-one-pricing-definitions/#gb-ingested). Oferecemos duas opções para GB Ingested, conforme descrito a seguir:

    * <b>Dados Originais:</b> A opção de dados originais da New Relic fornece um padrão de pelo menos 8 dias de retenção de dados e um limite máximo de consulta de 20 bilhões de pontos de dados por minuto.

    * <b>Data Plus:</b> A opção Data Plus da New Relic oferece recursos avançados, como retenção de dados estendida por até 90 dias e um limite máximo de consulta aumentado para 60 bilhões de pontos de dados por minuto.

    Cobramos pelos dados com base na quantidade de GB consumidos, independentemente do tipo, o que facilita a compreensão da sua fatura. Você tem acesso a 100 GB de dados ingeridos por mês gratuitamente. Para acompanhar o seu consumo de GB, você pode usar nossa página [de Resumo de Uso](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/usage-ui) ou [executar o comando consulta](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/usage-queries-alerts).

    ### O que é considerado dado faturável? [#usage-calculation]

    De forma geral, &quot;dados ingeridos&quot; se refere aos dados armazenados no New Relic pela sua organização após a aplicação de diversas regras de corte e transformação de dados. Em outras palavras, não é a quantidade de dados brutos enviados para a New Relic, mas sim o tamanho dos dados que acabam sendo armazenados.

    Em geral, os dados utilizados pelo nosso recurso não são contabilizados no total de GB ingeridos. Exemplos de recursos e dados relacionados a recursos que não são contabilizados no GB ingerido incluem:

    * [Funcionalidade básica de alerta](/docs/alerts-applied-intelligence/new-relic-alerts/learn-alerts/alerts-concepts-workflow). Por exemplo, `NrAiIncident`, um evento usado para gerar esses recursos, não conta para GB ingeridos.
    * Dados criados a partir de dados brutos agregados (por exemplo, [dados de infraestrutura agregados](/docs/data-apis/manage-data/manage-data-retention/#infrastructure-data) ou [dados dimensionais métricos](/docs/data-apis/understand-data/new-relic-data-types/#dimensional-metrics)).
    * [Levantamentos clássicos](/docs/apis/nerdgraph/examples/golden-metrics-entities-nerdgraph-api-tutorial) (para mais informações sobre levantamentos clássicos, consulte [definições de entidade](https://github.com/newrelic/entity-definitions)).
    * Verificações do monitoramento sintético. Para obter detalhes sobre a faturação, [os limites e os preços da Sintéticos](docs/synthetics/synthetic-monitoring/getting-started/monitor-limits/).
    * Dados de rastreamento de uso e faturamento (por exemplo, `NrConsumption`).
    * Dados relacionados à administração da conta (por exemplo, `NrIntegrationError`, `NrAuditEvent`).

    O GB ingerido mensalmente é arredondado para o número inteiro mais próximo. Por exemplo, se sua organização usa 100,9 GB durante um mês, isso é contado como 100 GB.
  </Collapser>

  <Collapser id="user" title="Usuário">
    Um usuário no New Relic é um indivíduo que pode acessar e usar a plataforma. O número de usuários faturáveis é um fator de faturamento.

    ### Usuários: Faturamento e regras [#billable-users]

    Usuário são faturáveis no momento em que são adicionados como usuário faturável a uma organização New Relic. A seguir, apresentamos os tipos de usuários do New Relic:

    * **Usuário básico:** Como usuário básico, você não será cobrado e poderá configurar nossas ferramentas de observabilidade, executar consultas de dados e usar o dashboard personalizado (início rápido do dashboard por até 7 dias). Você também terá acesso a alguns recursos básicos de alerta. No entanto, você não poderá usar nossas experiências selecionadas, como a interface APM, a interface de monitoramento de browser ou a interface de mobile. Embora seu acesso não implique uma cobrança direta, você pode gerenciar a cobrança de recursos com base no uso, incluindo o complemento CCU.

    * **Usuário principal:** Como usuário principal, você obtém todos os recursos básicos do usuário, além de acesso a poderosos recursos centrados em desenvolvedores, como New Relic CodeStream, Caixa de entrada de erros e nossa interface de gerenciamento de logs.

    * **Usuário de plataforma completa:** Como usuário de plataforma completa, você tem acesso a muitos de nossos recursos de plataforma. Isso inclui nossas experiências de interface de observabilidade mais selecionadas, como APM, monitoramento de infraestrutura, monitoramento de browser, monitoramento de mobile e monitoramento sintético.

    Para saber mais, consulte os [tipos de usuário](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type/).

    <CollapserGroup>
      <Collapser id="user-count" title="Como os usuários faturáveis são contados">
        A cobrança para usuários faturáveis é feita mensalmente, conforme o calendário. Para determinar a contagem de usuários faturáveis de uma organização em um mês, contamos o número de usuários durante esse mês que tiveram um <DNT>**billable user type**</DNT> de usuário de plataforma completa ou de usuário principal. O <DNT>**billable user type**</DNT> de um usuário é definido como o tipo de usuário mais alto que um usuário foi definido durante um mês civil. Usamos o fuso horário UTC para definir o início e o fim de um mês do calendário.

        Por exemplo, se um usuário for definido como usuário da plataforma completa em algum momento durante um mês civil, seu tipo de usuário faturável para esse mês será `full platform user` e não mudará, mesmo que ele faça um downgrade mais tarde nesse mês. Isso ocorre mesmo que o usuário seja alterado para um usuário da plataforma completa apenas brevemente. A alteração será aplicada no mês seguinte e esse usuário não será contabilizado como usuário da plataforma completa, a menos que seja alterado novamente posteriormente.

        Se você pretende adicionar usuários faturáveis ou alterar o tipo de usuário de um usuário, considere estas dicas:

        * Se você deseja adicionar um usuário faturável ou atualizar o plano de um usuário, o ideal é fazer isso no início do mês.
        * Se você quiser rebaixar o plano de um usuário, provavelmente é melhor fazer isso no final do mês.
        * Usamos o horário UTC para determinar o início e o fim de um mês. Isso significa que, por exemplo, se você estivesse no fuso horário do Pacífico e quisesse fazer um downgrade do seu plano de usuário no último dia do mês, teria que fazer essas alterações até as 17h, horário do Pacífico.

        Um usuário em sua organização com o mesmo endereço de e-mail conta apenas como um usuário. Para obter mais informações, consulte a seção [de rastreamento de usuários](/docs/accounts/accounts-billing/account-setup/multiple-logins-found#user-records) em nossa documentação.

        Se você utiliza nosso [modelo de usuário mais recente](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models), os usuários são cobrados quando adicionados como usuários faturáveis pelos administradores. Mesmo que um usuário faturável nunca tenha feito login no New Relic e tenha uma tag `Pending invite` na interface, ele ainda será faturável (se você estiver em nosso [modelo de usuário original](/docs/accounts/original-accounts-billing/original-users-roles/users-roles-original-user-model), os usuários `Pending invite` não são faturáveis).

        Para mais informações sobre como adicionar e convidar usuários, consulte a seção [Adicionar usuário](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-ui-and-tasks#add-users) em nossa documentação.

        O custo do seu usuário faturável depende da [edição de preços](https://newrelic.com/pricing) da sua organização (Standard, Pro ou Enterprise) ou de qualquer plano de preços personalizado que você tenha com New Relic.

        Quando uma organização New Relic começa a ser faturada, calcularemos proporcionalmente o número de usuários faturáveis com base na data de início dentro daquele mês. Também cobraremos o valor proporcional do seu último mês caso você cancele a assinatura.
      </Collapser>

      <Collapser id="user-downgrade-rules" title="Regras de downgrade do usuário">
        As regras relativas ao número de vezes que você pode fazer downgrade da sua conta Full Plataforma Usuário variam dependendo se o seu [plano de uso](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing#usage-plans) é pré-pago (PAYG) ou possui um contrato com fidelização.

        ### Regras de downgrade com pagamento conforme o uso

        Para o plano de uso pré-pago (PAYG), não há regras que limitem o downgrade do usuário. Observe que o [cálculo da fatura](#user-count) pode influenciar suas decisões sobre quando atualizar ou fazer downgrade do plano de usuário.

        ### Regras de downgrade de contratos de compromisso

        Antes de fazer upgrade ou downgrade de um usuário, você deve entender [como funciona o faturamento do usuário](#user-count).

        Para organizações com contratos comprometidos, a regra que rege downgrades e upgrades da plataforma Full usuário é a seguinte:

        <DNT>**If a Full Platform user is downgraded to a lower user type in a later month and then returned to a Full Platform user in a later month, and this downgrade/upgrade cycle happens twice in a contract year, that user will be billed as a Full Platform user for the remainder of the year, regardless of edits to their user type.**</DNT> Não há regras que regem as alterações de tipo de usuário que ocorrem dentro de um mês devido ao [modo como o faturamento do usuário funciona](#user-count): o tipo de usuário faturável de um usuário para um mês civil é o tipo de usuário mais alto definido durante esse mês.
      </Collapser>
    </CollapserGroup>

    ### Gerenciar usuários e custos [#manage]

    Para gerenciar usuários, consulte a seção [de gerenciamento de usuários](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-ui-and-tasks) em nossa documentação. Você pode usar a [interface de usuário](/docs/accounts/accounts-billing/general-account-settings/introduction-account-settings/#pricing) para obter uma visão geral da sua contagem de usuários faturáveis. Se você precisar de mais detalhes do que a interface fornece, também pode executar [consultas NRQL relacionadas ao uso](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/usage-queries-alerts).
  </Collapser>

  <Collapser id="core-compute-capacity-unit" title="Core Compute">
    Core Compute é baseado em preços sem consumo para observação de recursos full-stack com usuário ilimitado. Sua fatura é baseada na [unidade principal de capacidade de computação](/docs/licenses/license-information/product-definitions/new-relic-one-pricing-definitions/#compute-capacity-unit) (Core CCUs) consumida para executar com êxito ações como consultas, alertas, chamadas de API e carregamentos de páginas. A tabela de modelos de preços indica para quais recursos você pode escolher o Core Compute como um modelo de preços alternativo ao do usuário.
  </Collapser>

  <Collapser id="advance-compute-capacity-unit" title="Advanced Compute">
    Advanced computar é um complemento ao seu plano de computação Data + usuário ou Data + Core Compute que fornece preços baseados no consumo para recurso de observabilidade inteligente. Esse complemento é medido em [Unidade Avançada de Capacidade de Computação](/docs/licenses/license-information/product-definitions/new-relic-one-pricing-definitions/#compute-capacity-unit) (aCCUs). A cobrança é feita com base no seu uso de CCUs avançados.

    Se os recursos full-stack de observabilidade forem combinados com qualquer recurso de observabilidade inteligente, será cobrada a unidade avançada de capacidade de computação apropriada.

    <Callout variant="tip">
      O uso do CodeStream em computadores avançados acarretará cobranças de CCU avançado, independentemente do tipo de usuário. Isso significa que o usuário básico pode acessar o recurso de computador avançado e incorrer em cobranças pelas CCUs avançadas.
    </Callout>
  </Collapser>
</CollapserGroup>

### Cobrança de recurso adicional [#add-on-billing]

Além dos principais fatores de faturamento, como dados e usuários faturáveis, você também pode usar complementos faturáveis opcionais para enriquecer sua experiência com New Relic.

* **Advanced Compute**: Este complemento ao seu plano de computação Data + usuário ou Data + Core Compute permite o acesso ao recurso de observabilidade inteligente.
* **Centro de dados da UE para Dados Originais ou Dados Plus**: Este complemento aplica-se à sua opção de dados (Dados Originais ou Dados Plus) quando seleciona a União Europeia como a sua região de dados, conforme disponibilidade.
* **Extended Retention for Original Data or Data Plus**: este complemento será aplicado se você exceder o período padrão de retenção dos seus dados. Isso se aplica a todos os seus dados — não apenas aos logs — e é uma boa opção se você precisar fazer muitas consultas pequenas ou consultar grandes volumes de dados.
* **Live Archives**: estenda a duração do armazenamento de logs em até sete anos. Live Archives também requer um computador avançado.
* **New Relic Synthetic Checks**: este complemento é aplicado caso suas verificações excedam o número padrão de verificações do monitoramento sintético.
