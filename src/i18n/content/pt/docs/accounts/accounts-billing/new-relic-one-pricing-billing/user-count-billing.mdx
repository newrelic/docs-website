Em [nosso modelo de precificação](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing), a contagem de usuários faturáveis é um fator de faturamento.

## Saiba mais sobre custos [#cost]

Se você ainda não é cliente New Relic e está curioso sobre custos, consulte primeiro [nossa página principal de preços](https://newrelic.com/pricing).

Se você já é cliente da New Relic e deseja entender seu faturamento, consulte a [interface de faturamento](/docs/accounts/accounts-billing/general-account-settings/introduction-account-settings). Se a sua organização contiver diversas contas, você deverá estar na conta de relatório principal (normalmente a primeira conta criada na sua organização) para ver todos os usuários da sua organização.

## Quais usuários são faturáveis? [#billable-users]

Usuários faturáveis são usuários com um [tipo de usuário](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type) núcleo ou usuário de plataforma completa. O usuário básico é gratuito.

usuário são faturáveis no momento em que são adicionados como usuário faturável a uma organização New Relic . Mesmo que um usuário faturável nunca tenha feito login no New Relic e tenha uma tag `Pending invite` na interface, ele ainda será faturável.

## Gerenciar custos do usuário [#primary-version]

Você pode usar a [interface de uso](/docs/accounts/accounts-billing/general-account-settings/introduction-account-settings/#pricing) para obter uma visão geral da contagem de usuários faturáveis. Se precisar de mais detalhes do que a interface fornece, você também pode executar [a consulta NRQL relacionada ao uso](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/usage-queries-alerts).

<CollapserGroup>
  <Collapser id="user-count" title="Como os usuários faturáveis são contados">
    A cobrança do usuário faturável é feita por mês. Para determinar a contagem de usuários faturáveis de uma organização em um mês, contamos os usuários durante esse mês que tinham um <DNT>**billable user type**</DNT> de usuário completo da plataforma ou de usuário principal. O <DNT>**billable user type**</DNT> de um usuário é definido como o tipo de usuário mais alto no qual um usuário foi definido durante um mês civil. Usamos o fuso horário UTC para definir o início e o fim de um mês.

    Para ver um exemplo de como isso funciona na prática: se um usuário for definido como usuário completo da plataforma em qualquer momento durante um mês, o tipo de usuário faturável para esse mês será `full platform user` e não será alterado, mesmo se ele fizer downgrade. mais tarde naquele mês. Este é o caso mesmo que esse usuário seja alterado para um usuário completo da plataforma apenas brevemente.

    Se você planeja adicionar um usuário faturável ou alterar o tipo de usuário do seu usuário, lembre-se dessas regras. Algumas dicas:

    * Se quiser adicionar um usuário faturável ou atualizar um usuário, provavelmente você desejará fazer isso no início do mês.
    * Se você quiser fazer o downgrade de um usuário, provavelmente fará isso no final do mês.
    * Usamos o horário UTC para determinar o início e o fim de um mês. Isso significa que se, por exemplo, você estivesse no horário do Pacífico e quisesse fazer downgrade do usuário no último dia do mês, você teria que fazer essas alterações até as 17h, horário do Pacífico.

    Os usuários da sua organização com o mesmo endereço de e-mail são contados apenas como um usuário. Para obter mais informações, consulte [rastreamento de usuário](/docs/accounts/accounts-billing/account-setup/multiple-logins-found#user-records).

    Para usuário em nosso [modelo de usuário mais recente](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models): usuário é faturável quando adicionado pelos administradores como usuário faturável. Mesmo que um usuário faturável nunca tenha feito login no New Relic e tenha uma tag `Pending invite` na interface, ele ainda será faturável. (Para usuário em nosso [modelo de usuário original](/docs/accounts/original-accounts-billing/original-users-roles/users-roles-original-user-model), `Pending invite` usuário não são faturáveis.)

    Para obter mais informações sobre como adicionar e convidar usuários, consulte [Adicionar usuário](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-ui-and-tasks#add-users).

    O custo do seu usuário faturável depende da [edição de preços](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing/#editions) da organização (Standard, Pro ou Enterprise) ou de qualquer acordo personalizado que você possa ter com a New Relic.

    Quando uma organização New Relic começa a ser cobrada pela primeira vez, sua contagem de usuários faturáveis é calculada proporcionalmente com base em quando, durante o mês, eles começaram. Se uma organização cancelar sua assinatura, o rateio também será aplicado ao último mês.
  </Collapser>

  <Collapser id="user-downgrade-rules" title="Regras de downgrade do usuário">
    As regras referentes a quantas vezes você pode fazer downgrade de usuário completo da plataforma diferem dependendo se o seu [plano de uso](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing#usage-plans) é pré-pago ou um contrato de compromisso.

    ### Regras de downgrade com pagamento conforme o uso

    Para o [plano de utilização pré-pago](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing#usage-plans), não há regras que limitem o downgrade de usuário. Mas observe que o [cálculo do faturamento](#user-count) pode impactar suas decisões sobre quando fazer upgrade ou downgrade do usuário.

    ### Regras de downgrade de contratos de compromisso

    Antes de fazer upgrade ou downgrade de um usuário, você deve entender [como funciona o faturamento do usuário](#user-count).

    Para organizações com [contratos de compromisso](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing/#usage-plans), a regra que rege downgrades e upgrades completos de usuários da plataforma é esta:

    <DNT>**If a full platform user is downgraded to a lower user type in a later month and then returned to a full platform user in a later month, and this downgrade/upgrade cycle happens twice in a contract year, that user will be billed as a full platform user for the remainder of the year, regardless of edits to their user type.**</DNT> Não há regras que regem as alterações de tipo de usuário que ocorrem dentro de um mês devido ao [modo como o faturamento do usuário funciona](#user-count): o tipo de usuário faturável de um usuário para um mês civil é o tipo de usuário mais alto definido durante esse mês.

    Aqui estão detalhes sobre os termos usados no contexto desta regra:

    * Um <DNT>**downgrade**</DNT> significa deixar de ser cobrado como um usuário completo da plataforma em um mês para, em um mês posterior, ser cobrado como um tipo de usuário inferior (principal ou básico) ou excluído.
    * Um <DNT>**upgrade**</DNT> significa deixar de ser cobrado como usuário principal, usuário básico (ou usuário excluído anteriormente) em um mês para, em um mês posterior, ser cobrado como usuário completo da plataforma.
    * Um <DNT>**contract year**</DNT> começa no ponto inicial do seu contrato ou no aniversário desse ponto. Se sua organização iniciou com um plano de preços diferente e mudou para um contrato de compromisso, a regra de downgrade de tipo de usuário será aplicada a partir da data em que você optou por participar até a data de renovação do contrato ou a data de aniversário anual do seu termo de compromisso, o que ocorrer primeiro.

    <Callout variant="tip">
      <DNT>
        **Why do we have rules governing full platform user downgrades/upgrades?**
      </DNT>

      O tipo de usuário da plataforma completa pretende ser uma classificação de longo prazo. Contamos com estimativas de usuários faturáveis quando criamos um contrato de compromisso, e essas regras garantem que estamos implementando esses limites contratuais de forma consistente.
    </Callout>

    #### Exemplo de limites de downgrade

    Digamos que um ano de contrato comece em março. E digamos que o usuário percorra esses tipos de conta mês a mês:

    1. O usuário é cobrado como usuário completo da plataforma em março
    2. O usuário será cobrado como usuário básico em abril (<DNT>**downgrade**</DNT>)
    3. O usuário será cobrado como usuário completo da plataforma em junho (<DNT>**upgrade**</DNT>)
    4. O usuário será cobrado como usuário básico em julho (<DNT>**downgrade**</DNT>)

    Se esse usuário voltar a ser um usuário completo da plataforma em qualquer mês subsequente, ele será cobrado pelo restante do ano do contrato como um usuário completo da plataforma, mesmo que seja rebaixado novamente para ser um usuário básico ou usuário principal depois disso.
  </Collapser>
</CollapserGroup>
