Neste documento você aprenderá como definimos <DNT>**user type**</DNT>, a qual recurso cada tipo de usuário tem acesso e como decidir sobre um tipo de usuário.

Quer saber como os usuários são calculados para fins de faturamento? Consulte [preços da New Relic](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing). Se ainda não o fez, certifique-se de [se inscrever para uma conta New Relic](https://newrelic.com/signup). É grátis, para sempre.

## Qual é o tipo de usuário? [#user-type-defined]

O <DNT>**user type**</DNT> de um usuário do New Relic determina o conjunto máximo de permissões do New Relic que ele pode acessar. O tipo de usuário deve ser uma configuração de longo prazo com base nas responsabilidades esperadas do usuário durante os próximos meses ou mais.

A escolha do tipo de usuário de um usuário é principalmente uma decisão relacionada ao faturamento. O usuário principal e o usuário completo da plataforma são faturáveis, enquanto o usuário básico não. É uma questão de quanto valor uma organização espera obter com o uso da New Relic por um membro da equipe. (Para obter detalhes sobre cobrança, consulte [Faturamento de usuário](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/user-count-billing).)

O tipo de usuário **não** deve ser usado como forma de controlar as permissões de um usuário. Isso ocorre porque o New Relic ocasionalmente ajusta as permissões disponíveis para cada tipo de usuário. Para restringir o acesso às permissões, você deve usar [funções](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts).

Existem três tipos de usuário:

* <DNT>**Basic user**</DNT>: acesso a vários recursos básicos, mas poderosos, da plataforma New Relic.
* <DNT>**Core user**</DNT>: acesso a vários recursos a mais que um usuário básico.
* <DNT>**Full platform user**</DNT>: acesso a todos os recursos.

Se você tiver a tarefa de [adicionar usuários do New Relic](/docs/accounts/accounts-billing/new-relic-one-user-management/tutorial-add-new-user-groups-roles-new-relic-one-user-model/#add-users), uma das principais decisões a tomar é qual tipo de usuário definir. Se não tiver certeza a princípio, você pode adicioná-los como usuário básico e depois decidir qual usuário deseja atualizar. Para saber como ajustar o tipo de usuário, consulte [Gerenciar tipo de usuário](#manage-user-type).

## Como escolher um tipo de usuário [#choose-user-type]

Antes de decidir sobre os tipos de usuário, convém entender os [impactos no faturamento e as regras de downgrade](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/user-count-billing). Para uma comparação detalhada de acesso, consulte a [tabela de comparação de acesso de tipo de usuário](#user-type-comparison-table).

Aqui estão algumas dicas para decidir sobre um tipo de usuário:

<DNT>
  **Reasons to make someone a full platform user:**
</DNT>

* Eles precisam de acesso total à plataforma New Relic (nosso painel com curadoria e outras experiências), e não apenas da capacidade de consultar e criar gráficos personalizados.
* Eles desempenham um papel fundamental nas fases de desenvolvimento, teste, implantação e manutenção do ciclo de vida de desenvolvimento do aplicativo.
* Eles quebram/consertam código regularmente; eles são responsáveis pela triagem do fluxo de trabalho, resolução de problemas ou gerenciamento de usuários e funções de sua equipe.
* Eles possuem práticas de DevOps (por exemplo, sistemas de controle de versão e implementação de CI/CD).

<DNT>
  **Reasons to make someone a core user:**
</DNT>

* São desenvolvedores que não têm como principal responsabilidade a confiabilidade e o ritmo de operação.

* Eles não exigem acesso completo à plataforma, mas se beneficiariam de algumas funcionalidades específicas oferecidas ao usuário principal, como:

  * Visualizando erros em um só lugar da sua stack com [a Caixa de entrada de erros](/docs/errors-inbox/errors-inbox).
  * Usando nossa [UI de log](/docs/logs/ui-data/use-logs-ui) para detectar problemas e padrões no log.
  * A capacidade de usar aplicativos do nosso [catálogo de aplicativos New Relic](https://opensource.newrelic.com/nerdpacks).

* Eles precisam ser capazes de gerenciar o usuário e/ou o faturamento (tanto o usuário principal quanto o completo da plataforma podem fazer isso).

<DNT>
  **Reasons to make someone a basic user:**
</DNT>

* Eles não precisam de acesso total à plataforma para usar nossas experiências selecionadas e <InlinePopover type="dashboards"/>, mas se beneficiariam com a criação de consultas personalizadas e gráficos de dados.
* Eles desempenham um papel fundamental na fase de planejamento do ciclo de vida de desenvolvimento do aplicativo.
* Eles usam e configuram ferramentas da New Relic para inserir dados na New Relic e acessam, configuram e usam <InlinePopover type="alerts"/>nesses dados, mas não são necessariamente responsáveis pela triagem do fluxo de trabalho, resolução de problemas ou pelo gerenciamento de usuários e funções para seus equipe.
* Eles desejam ver análises e métricas de negócios de alto nível para planejamento futuro (como costuma acontecer com executivos de alto nível).
* Eles não gerenciam usuários ou faturamento.

Abaixo está uma comparação detalhada do recurso ao qual cada tipo de usuário tem acesso. Pontos importantes sobre esta tabela:

* A tabela vem da [nossa página de preços](https://newrelic.com/pricing). Para encontrar a tabela, visite o cabeçalho <DNT>**User costs**</DNT> e clique em <DNT>**View permissions**</DNT>.
* Muitos dos recursos requerem acesso às nossas experiências de interface, não aos dados subjacentes. Todos os usuários podem consultar todos os dados das contas que acessam e podem criar e visualizar gráficos personalizados. Por exemplo, o usuário básico pode acessar dados <InlinePopover type="apm"/>, dados de monitoramento do browser e muito mais, mas não pode acessar experiências de interface selecionadas.
* O tipo de usuário deve ser uma configuração de longo prazo. Tanto o tipo de usuário quanto as funções controlam o acesso ao recurso New Relic. [Saiba mais sobre fatores de acesso do usuário](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts).

## Tipo de usuário e funções [#user-type-and-roles]

Aqui está uma explicação de como o tipo de usuário e [o acesso baseado em função](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts) diferem:

* <DNT>**[user type](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type)**</DNT> de um usuário: qual tipo de usuário transformar em usuário é uma questão de o que uma organização espera que os membros de sua equipe façam com a New Relic e quanto valor eles esperam obter com esse trabalho. É principalmente uma decisão relacionada ao faturamento. Ele define o máximo de permissões permitidas que um usuário pode acessar. O tipo de usuário **não** deve ser usado para controlar o acesso e as permissões de um usuário: para isso, você deve usar funções.
* <DNT>**roles**</DNT> de um usuário: as funções controlam o acesso de um usuário. As funções são compostas por <DNT>**permissions**</DNT>, que concedem acesso para fazer coisas específicas no New Relic (por exemplo, a capacidade de modificar configurações de APM). As funções são atribuídas aplicando-as a um [grupo de usuários](#groups) e podem estar presentes em uma ou mais contas em uma organização.

Um usuário do New Relic recebe permissão para usar um recurso do New Relic pela combinação de a) tipo de usuário e b) permissões de função. Para que um usuário do New Relic acesse algo, seu tipo de usuário e as funções atribuídas a ele devem permitir esse acesso.

Por exemplo, digamos que um usuário básico tenha uma função com amplo acesso ao New Relic, como [<DNT>**All product admin**</DNT>](#standard-roles) (que os grupos padrão <DNT>**User**</DNT> e <DNT>**Admin**</DNT> têm). Seu tipo de usuário (usuário básico) os impediria de usar muitos dos recursos que um usuário principal ou usuário completo da plataforma com essa função pode acessar. Para obter mais acesso, o usuário básico teria que se tornar um usuário principal ou completo da plataforma.

Outro exemplo: digamos que um usuário completo da plataforma tenha uma função restritiva atribuída a uma conta específica (como [<DNT>**Read only**</DNT>](#standard-roles)). Um usuário completo da plataforma pode, teoricamente, acessar toda a New Relic, mas, neste caso, a função atribuída a essa conta restringe bastante seu acesso. Para obter mais acesso, as funções atribuídas precisariam ser alteradas (por exemplo, atribuindo-os a um grupo diferente ou ajustando as funções atribuídas ao grupo).

Para obter mais informações sobre funções e grupos, consulte [Conceitos de gerenciamento de usuários](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts).
