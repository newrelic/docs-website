Estamos apresentando nosso mais recente runtime de monitor sintético, com a versão **Latest** do Chrome e o Node.js 22. O runtime **Latest** usa automaticamente a versão mais recente do Chrome, garantindo que seus monitores com script sejam executados em um ambiente que atenda aos padrões de navegadores modernos, proporcionando maior segurança, estabilidade e desempenho.

## Ação necessária: impacto da atualização do Node.js 22

Como parte da atualização do Chrome 134 para o Chrome **Latest**, a versão do Node no backend está mudando do Node.js 16 para o Node.js 22. Este runtime mais recente é mais rigoroso em relação aos ciclos de vida dos processos. Se os scripts do seu monitor contiverem **handles abertos não tratados** (como promises não resolvidas, temporizadores pendentes ou conexões de rede não fechadas), seus monitores falharão.

**Para resolver isso:** Valide seus scripts de monitor no runtime **Latest** para garantir que todos os handles sejam fechados corretamente antes que o script termine de ser executado. As validações com falha aparecem na UI de Runtime Upgrades. Consulte [Solucionar erros de atualização de runtime](/docs/synthetics/synthetic-monitoring/troubleshooting/runtime-upgrade-troubleshooting).

## Quem é afetado

A New Relic gerencia automaticamente o processo de atualização para:

* Monitores de navegador com script em locais públicos
* Monitores de API em locais públicos
* Monitores em locais públicos e privados (híbrido)

A New Relic aplica validação e atualização automatizadas a todas as configurações de local.

<Callout variant="important">
  Monitores executados exclusivamente em locais privados requerem atualização manual.
</Callout>

## O que é o Mais recente?

A opção de runtime **Latest** no menu suspenso de criação/atualização do monitor usa automaticamente a versão mais recente do Chrome suportada pela New Relic. Em vez de fixar em uma versão específica do Chrome (como Chrome 134 ou Chrome 145), **Latest** garante que seus monitores sempre sejam executados na versão mais recente disponível do Chrome com os últimos recursos e atualizações de segurança.

## Atualizar monitores via NerdGraph

Para atualizar os runtimes do seu monitor programaticamente, use as mutações do NerdGraph abaixo. Você precisa do GUID da entidade do seu monitor (disponível nas configurações do monitor).

### Monitorar o browser com script

```graphql
mutation {
  syntheticsUpdateScriptBrowserMonitor(
    guid: "YOUR_MONITOR_GUID",
    monitor: {
      runtime: {
        runtimeType: "CHROME_BROWSER",
        runtimeTypeVersion: "LATEST",
      }
    }
  ) {
    errors {
      description
      type
    }
  }
}
```

### Monitor de API com script

```graphql
mutation {
  syntheticsUpdateScriptApiMonitor(
    guid: "YOUR_MONITOR_GUID",
    monitor: {
      runtime: {
        runtimeType: "NODE_API",
        runtimeTypeVersion: "LATEST",
      }
    }
  ) {
    errors {
      description
      type
    }
  }
}
```

## O processo de atualização automatizado

Para garantir uma transição tranquila para o runtime mais recente do Chrome, a New Relic testa proativamente seus monitores existentes. O processo inclui:

<Steps>
  <Step>
    ## Validação de backend

    A New Relic testa automaticamente seus monitores existentes no runtime mais recente do Chrome / Node.js 22 no backend.

    <Callout variant="tip">
      Essa validação não consome suas verificações sintéticas nem afeta seus resultados de produção.
    </Callout>
  </Step>

  <Step>
    ## Atualização automática

    **Se a validação for bem-sucedida:** a New Relic atualiza o monitor para o runtime mais recente em seu nome.
  </Step>

  <Step>
    ## Revisão manual de falhas

    **Se a validação falhar:** New Relic não força a atualização. Isso geralmente acontece devido ao tratamento rigoroso de handles abertos pelo Node.js 22. Em vez disso, a New Relic sinaliza os monitores que falharam na validação e os exibe no recurso **Runtime Upgrades** na UI do seu Synthetics Nerdlet.

    Você pode revisar os detalhes da falha lá, solucionar problemas na sintaxe do seu script e validar e atualizar manualmente o monitor após corrigir o script.
  </Step>
</Steps>

## Solução de problemas de atualização

O problema mais comum ao atualizar para o Node.js 22 são os manipuladores abertos não tratados, incluindo promises não resolvidas, temporizadores pendentes, conexões não fechadas e conexões HTTP/HTTPS. Para soluções detalhadas e exemplos de código para cada tipo de problema, consulte [Solucionar erros de atualização de runtime](/docs/synthetics/synthetic-monitoring/troubleshooting/runtime-upgrade-troubleshooting).

## Perguntas frequentes

**Quais tipos de monitores são afetados?** Monitores de navegador com script e monitores de API com script. Monitores de navegador simples e monitores de ping não são afetados.

**Posso testar meus scripts localmente antes de atualizar?** Sim. Instale o Node.js 22 localmente e teste seus scripts de monitoramento para identificar problemas antes da atualização.

**Posso fazer rollback se meu monitor falhar após a atualização?** Sim. Você pode alterar a versão do runtime de volta nas configurações do monitor ou via NerdGraph.

**Quando meus monitores serão atualizados automaticamente?** A New Relic realiza a validação primeiro. Se a validação for bem-sucedida, a New Relic atualiza automaticamente os monitores. Se a validação falhar, você verá a falha na UI de Runtime Upgrades e deverá corrigir e atualizar manualmente.

## Precisa de ajuda?

Se você encontrar problemas durante o processo de atualização:

1. Verifique a **UI de Runtime Upgrades** no seu Nerdlet do Synthetics para obter mensagens de erro específicas.
2. Revise seus scripts de monitoramento em busca de promises não tratadas e conexões abertas.
3. Teste seus scripts localmente com o Node.js 22 antes de atualizar
4. Entre em contato com o Suporte da New Relic para obter assistência na solução de problemas de falhas de validação.
