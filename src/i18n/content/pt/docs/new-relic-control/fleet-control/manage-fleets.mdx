Depois de criar sua primeira frota usando o [guia de configuração](/docs/new-relic-control/fleet-control/setup), você realizará a maior parte do seu trabalho diário na interface do usuário <DNT>Fleet Control</DNT>. ## Entendendo os tipos de frota

<DNT>Fleet Control</DNT> suporta três tipos de frotas, cada uma projetada para um tipo específico de entidade gerenciada:

* **Frotas de clusters Kubernetes:** Gerencie clusters Kubernetes onde os agentes são implantados como Helm charts.
* **Frotas de hosts Linux:** Gerencie servidores Linux onde os agentes são executados como processos baseados em host. (Versão prévia pública)
* **Frotas de hosts Windows:** Gerencie servidores Windows onde os agentes são executados como serviços do Windows ou processos baseados em host. (Versão prévia pública)

Ao criar uma frota, você seleciona o tipo de entidade gerenciada. Essa escolha tem implicações importantes na forma como você trabalha com essa frota:

* **Atribuição de entidade:** Você só pode adicionar entidades gerenciadas que correspondam ao tipo da frota. Por exemplo, uma frota de hosts Windows só pode conter hosts Windows, e uma frota de clusters Kubernetes só pode conter clusters Kubernetes.
* **Filtragem de configuração:** Durante a implantação, serão exibidas apenas as configurações que correspondem ao tipo da frota. Se você estiver trabalhando com uma frota de hosts Windows, apenas as configurações de host Windows aparecerão no seletor de configurações. Essa filtragem automática evita configurações incorretas e agiliza o processo de implantação.
* **Disponibilidade do agente:** A lista de agentes disponíveis para implantação será filtrada com base no tipo de frota e no que é suportado para esse ambiente. Por exemplo, determinados agentes podem estar disponíveis apenas para ambientes Kubernetes ou sistemas operacionais de host específicos.

Este design garante a consistência em todas as entidades gerenciadas em uma frota e evita configurações incorretas, como tentar implantar acidentalmente uma configuração de chart Helm do Kubernetes em um host Windows.

### Permissões de frota (acesso detalhado)

<DNT>Fleet Control</DNT> usa acesso de granularidade fina (FGA) para permitir que administradores deleguem o gerenciamento de frotas específicas a certos usuários ou grupos. Isso é útil quando você deseja capacitar uma equipe para gerenciar suas próprias frotas sem conceder permissões amplas em nível de organização.

O método principal para delegar acesso é atribuir a função predefinida <DNT>**Fleet Manager**</DNT> a um grupo de usuários para uma frota específica. Isso concede aos usuários desse grupo controle total sobre essa frota, incluindo a capacidade de criar implantações, gerenciar configurações e adicionar ou remover entidades gerenciadas.

#### Pré-requisitos para administrador

Para configurar permissões do Fleet, você deve ter as funções <DNT>Organization Manager</DNT> e <DNT>AuthDomain Manager</DNT>.

#### Configurar fluxo de trabalho

O processo envolve duas etapas principais: primeiro, um administrador cria um grupo de usuários e, segundo, ele atribui a esse grupo a função <DNT>`Fleet Manager`</DNT> para uma frota específica.

**Etapa 1: crie um grupo de usuários**

1. Na parte inferior do menu de navegação esquerdo, clique no seu nome de usuário e, em seguida, selecione <DNT>**Administration**</DNT>.
2. Na página <DNT>Administration</DNT>, selecione <DNT>**Access Management**</DNT>.
3. Vá para a aba <DNT>groups</DNT> e clique em **Create a new group**.
4. Dê ao grupo um nome significativo (por exemplo, `k8s-platform-team-fleet-managers`).
5. Na seção <DNT>members</DNT>, adicione os usuários aos quais você deseja delegar o gerenciamento de frota.
6. Clique em <DNT>**Add user to the group**</DNT>. Você não precisa atribuir nenhuma função de nível de conta ou organização aqui. As permissões serão concedidas no nível da frota.

**Etapa 2: atribuir um grupo e uma função a uma frota**

Você pode atribuir permissões ao criar uma nova frota ou editando uma existente.

**Para atribuir permissões a uma frota existente:**

1. Na página principal do <DNT>fleets</DNT>, localize a frota que você deseja gerenciar.

2. Clique nas reticências (`...`) no final da linha e selecione **Edit permissions**.

3. No modal <DNT>fleet permissions</DNT> :

   * <DNT>**Select a group**</DNT>: Escolha o grupo de usuários que você criou na etapa anterior.
   * **Selecione uma função:** Escolha a função <DNT>`Fleet Manager`</DNT>.

4. Clique em <DNT>**Save**</DNT>. Os usuários no grupo selecionado agora têm permissões totais de gerenciamento para esta frota.

**Para atribuir permissões durante a criação da frota:**

1. Siga as etapas para [criar uma frota em nosso guia de configuração](/docs/new-relic-control/fleet-control/setup).
2. Antes de clicar em **Create Fleet**, expanda a seção **Access management**.
3. Use o menu suspenso para selecionar o **Group** desejado e a função de **Fleet Manager**.
4. Clique em **Create Fleet**.

## Gerenciando uma frota específica

Clicar no nome de uma frota leva você para sua página de gerenciamento detalhado.

<Callout title="Importante: Regras de configuração da frota" variant="important">
  As frotas são projetadas para garantir a consistência em todas as suas entidades gerenciadas. Para alcançar isso, três regras importantes se aplicam:

  * **Consistência do tipo de entidade gerenciada:** Cada frota tem um tipo específico de entidade gerenciada (Cluster Kubernetes, Host Linux ou Host Windows) que é definido quando a frota é criada. Você só pode atribuir entidades gerenciadas que correspondam a este tipo à frota. Isso garante que todas as entidades da frota sejam compatíveis com as mesmas configurações e estratégias de implantação. Consulte [Entendendo os tipos de frota](#understanding-fleet-types) para mais detalhes.

  * **Uma configuração por tipo de agente:** dentro de uma única frota, você só pode ter uma configuração para cada tipo de agente. Por exemplo, se você tiver três clusters do Kubernetes em uma frota, todos eles devem usar a mesma configuração única para o agente New Relic Infrastructure. A cardinalidade de um tipo de agente para sua configuração dentro de uma frota é sempre 1:1.

  * **Instrumentação de infraestrutura exclusiva:** O sistema impede que você implante tanto o agente de infraestrutura do New Relic quanto a Distribuição New Relic do Coletor OpenTelemetry (NRDOT) na mesma frota. Essa validação ajuda a evitar a instrumentação dupla acidental das suas entidades gerenciadas para o monitoramento de infraestrutura.
</Callout>

## Compreendendo a implantação

Uma implantação é um conjunto controlado de ações que pode instalar, excluir ou atualizar agentes e suas configurações nas entidades gerenciadas em sua frota. Esta seção aborda os principais aspectos do ciclo de vida da implantação, incluindo a compreensão dos diferentes status de implantação, o processo de atualização automática para <DNT>Agent Control</DNT> e como usar recursos avançados, como implantações de rascunho e canary, para gerenciar alterações com segurança.

### Compreendendo os status de implantação

O status de cada implantação ajuda você a entender seu resultado. Aqui estão os possíveis status e o que eles significam:

| Status                             | Descrição                                                                                                                                                                                                                                                                                                  |
| :--------------------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 🔵 <DNT>**IN PROGRESS**</DNT>      | A implantação está sendo ativamente implementada na entidade gerenciada da frota.                                                                                                                                                                                                                          |
| 🟢 <DNT>**COMPLETED**</DNT>        | A implantação foi concluída com sucesso e todas as ações pretendidas foram aplicadas.                                                                                                                                                                                                                      |
| 🔴 <DNT>**FAILED**</DNT>           | A implantação foi concluída, mas pelo menos um agente em uma entidade gerenciada relatou um problema. Isso pode significar que um agente não conseguiu iniciar, desligar corretamente ou aplicar sua nova configuração. A implantação poderá ser parcialmente bem sucedida em outra entidade administrada. |
| 🟠 <DNT>**INTERNAL FAILURE**</DNT> | A implantação não pôde ser concluída devido a um problema inesperado no sistema <DNT>Fleet Control</DNT>. Isso não é um erro com seus agentes ou configurações. Tente a implantação novamente e, se o problema persistir, entre em contato com o suporte da New Relic.                                     |

### Compreendendo o fluxo de eventos de implantação

Ao visualizar a aba <DNT>deployment events</DNT>, você verá uma série de eventos que representam o ciclo de vida da sua implantação. Todos os eventos de implantação são gravados no tipo de evento personalizado `FleetDeployment`. Entender esse fluxo de eventos ajuda a interpretar o que você vê na aba <DNT>deployment events</DNT> e a diagnosticar problemas com mais eficácia.

<DNT>Fleet Control</DNT> implanta alterações de configuração em estágios chamados &quot;anéis&quot; para reduzir o risco. O processo de implantação funciona da seguinte maneira:

* **Se você designou entidades canário:** A implantação usa um anel canário primeiro (contendo apenas as entidades que você marcou como canários), seguido por um anel padrão para todas as entidades restantes. Cada anel deve ser concluído com sucesso antes que o próximo anel comece. Se o anel canário falhar, a implantação para antes de prosseguir para o anel padrão, evitando problemas generalizados.
* **Se você não designou nenhuma entidade canário:** A implantação usa apenas o anel padrão, que inclui todas as entidades da frota. A implantação será executada até a conclusão em todas as entidades nesse único anel.

#### Tipos de eventos de implantação

Os eventos a seguir acompanham o progresso de uma implantação do início ao fim:

| Evento                          | Descrição                                                                                  |
| :------------------------------ | :----------------------------------------------------------------------------------------- |
| Implantação da frota iniciada   | Uma implantação é iniciada para a frota. Os pacotes de configuração estão preparados.      |
| Implantação em anel iniciada    | Um anel de implantação é iniciado (por exemplo, anel canário ou padrão).                   |
| Implantação do agente iniciada  | A implantação da configuração começa para um agente específico em uma entidade gerenciada. |
| Implantação do agente concluída | Um agente conclui a tentativa de aplicar a configuração.                                   |
| Implantação do anel concluída   | Todos os agentes em um anel concluíram ou atingiram o tempo limite.                        |
| Implantação do Fleet concluída  | A implantação geral termina com um status final.                                           |

#### Status de implantação do agente

Quando um agente conclui sua tentativa de implantação (AgentDeploymentCompleted), ele reporta um dos seguintes status:

| Status                     | Descrição                                             |
| :------------------------- | :---------------------------------------------------- |
| <DNT>CONFIG\_APPLIED</DNT> | O agente aplicou a nova configuração com sucesso.     |
| <DNT>FAILED</DNT>          | O agente encontrou um erro ao aplicar a configuração. |
| <DNT>TIMEOUT</DNT>         | O agente não respondeu dentro do prazo esperado.      |
| <DNT>NOT\_REPORTING</DNT>  | O agente estava offline quando a implantação começou. |

#### Como funciona o fluxo de eventos

Quando você inicia uma implantação, eventos são gerados na seguinte sequência:

1. **FleetDeploymentStarted** — A implantação começa para toda a frota.
2. **RingDeploymentStarted** — O primeiro anel (tipicamente canário) é iniciado e todos os agentes nesse anel são identificados.
3. **AgentDeploymentStarted** — Para cada agente no anel, um evento de implantação é registrado. Esses eventos ocorrem em paralelo para todos os agentes no anel.
4. **AgentDeploymentCompleted** — Cada agente reporta seu status (<DNT>CONFIG\_APPLIED</DNT>, <DNT>FAILED</DNT>, <DNT>TIMEOUT</DNT> ou <DNT>NOT\_REPORTING</DNT>).
5. **RingDeploymentCompleted** — Assim que todos os agentes no anel concluírem ou excederem o tempo limite, o anel será marcado como concluído. Se o anel for bem-sucedido, o próximo anel começa. Se falhar, a implantação para.
6. **FleetDeploymentCompleted** — Depois que todos os anéis forem concluídos (ou um anel falhar), a implantação termina com um status final de <DNT>COMPLETED</DNT>, <DNT>FAILED</DNT> ou <DNT>INTERNAL\_FAILURE</DNT>.

#### Fluxo visual de eventos

```
FleetDeploymentStarted
│
├─▶ RingDeploymentStarted (canary)
│   │
│   ├─▶ AgentDeploymentStarted (agent 1) ──▶ AgentDeploymentCompleted ─┐
│   ├─▶ AgentDeploymentStarted (agent 2) ──▶ AgentDeploymentCompleted ─┤ [parallel]
│   ├─▶ AgentDeploymentStarted (agent 3) ──▶ AgentDeploymentCompleted ─┘
│   │
│   └─▶ RingDeploymentCompleted (canary)
│       │
│       ├─ ✓ If SUCCESS → proceed to default ring
│       └─ ✗ If FAILED → stop deployment
│
├─▶ RingDeploymentStarted (default)
│   │
│   ├─▶ AgentDeploymentStarted (agent 4) ──▶ AgentDeploymentCompleted ─┐
│   ├─▶ AgentDeploymentStarted (agent 5) ──▶ AgentDeploymentCompleted ─┤ [parallel]
│   ├─▶ AgentDeploymentStarted (agent 6) ──▶ AgentDeploymentCompleted ─┘
│   │
│   └─▶ RingDeploymentCompleted (default)
│
└─▶ FleetDeploymentCompleted
```

**Opções de status de implantação do agente:**

* ✅ **CONFIG\_APPLIED** — O Agente aplicou a configuração com sucesso
* ❌ **FALHOU** — O Agente encontrou um erro
* ⏱️ **TIMEOUT** — O agente não respondeu a tempo
* 📡 **NOT\_REPORTING** — O agente estava offline quando a implantação começou

**Principais identificadores:** `fleetId`, `deploymentId`, `ringName`

Você pode visualizar todos esses eventos em detalhes usando a aba Eventos de Implantação. Consulte a [seção da aba Deployments](#the-deployments-tab) para mais informações sobre como acessar e usar eventos de implantação para solução de problemas.

### Atualizações do Controle Automático de Agente

Para garantir que suas entidades gerenciadas se beneficiem dos patches de segurança, melhorias de desempenho e recursos mais recentes, o <DNT>Fleet Control</DNT> gerencia automaticamente o processo de atualização do Agent Control. Ao iniciar uma implantação, o <DNT>Fleet Control</DNT> verifica se uma versão mais recente do Agent Control está disponível para as entidades gerenciadas em sua frota.

Se uma atualização for necessária, ela será incluída automaticamente como parte da sua implantação. Você verá uma notificação na interface durante o processo de implantação informando que o Agente Control será atualizado. Essa atualização implícita acontece junto com quaisquer ações de implantação que você esteja executando (como instalar o agente, remover o agente ou alterar a configuração do agente), garantindo que seu ambiente permaneça atualizado com o mínimo de esforço.

<Callout title="Importante: Falha de atualização e recuperação" variant="caution">
  Atualmente, não há reversão automática para uma atualização de controle de agente com falha. Se uma atualização falhar, a implantação será marcada como **FAILED** e a entidade gerenciada poderá exigir intervenção manual. Para obter orientação, consulte nossa documentação sobre [atualização manual do agente Control](/docs/new-relic-control/agent-control/upgrading).

  Estamos desenvolvendo ativamente o recurso para fornecer reversão automática e simplificar a correção de falhas de atualização em uma versão futura.
</Callout>

## Criar e gerenciar projeto de implantação

Você pode salvar uma implantação como rascunho para trabalhar nela ao longo do tempo, sem implantar as alterações imediatamente.

### Criar e retomar um rascunho

1. Navegue até a aba <DNT>deployments</DNT> em uma frota específica.
2. Clique em <DNT>**Create deployment**</DNT>.
3. No painel de implantação, forneça um nome obrigatório e uma descrição opcional.
4. Faça as alterações desejadas, como adicionar um agente ou atualizar sua configuração.
5. Em vez de iniciar a implantação, clique em <DNT>**Save draft**</DNT>.
6. Em seguida, você pode fechar o painel de implantação. Sua nova implantação será listada na aba <DNT>deployments</DNT> com o status de `Draft`.

Para continuar trabalhando no rascunho, basta clicar em seu nome na tabela de histórico de implantação. Isso reabrirá o painel de implantação, permitindo que você conclua suas alterações e clique em **Start deployment** quando estiver pronto.

### Excluir um rascunho

Se você não precisar mais de uma implantação de rascunho:

1. Na guia <DNT>deployments</DNT>, clique no nome do rascunho que deseja excluir.
2. No painel de implantação que abrir, clique no botão de reticências (`...`) ao lado de <DNT>Start deployment</DNT> na parte superior da tela.
3. Selecione <DNT>**Delete draft**</DNT> no menu.
4. O rascunho será removido permanentemente e você retornará para a aba <DNT>deployments</DNT>.

### Seleção de versões do agente durante a implantação

Ao adicionar um agente a uma implantação, você será solicitado a selecionar uma versão do agente antes de escolher uma configuração:

1. Após selecionar o tipo de agente, um menu suspenso <DNT>**Select a version**</DNT> é exibido.

2. A versão mais recente é selecionada por padrão.

3. Você pode escolher outras versões disponíveis para instalação.

4. Metadados adicionais podem ser exibidos para cada versão, como:

   * Notas de versão destacando novos recursos e melhorias
   * Informações de atualização de segurança
   * Outros detalhes específicos da versão

Essa capacidade de seleção de versão funciona em todos os tipos de infraestrutura suportados (clusters Kubernetes, hosts Linux e hosts Windows), oferecendo controle preciso sobre qual versão do agente é implantada em suas entidades gerenciadas.

Após selecionar a versão do agente, clique em <DNT>**Next**</DNT> para prosseguir para a seleção de configuração.

### Ações do agente em uma implantação

Ao criar ou editar uma implantação, a tabela de **Agents** no painel de implantação fornece diversas ações para cada agente. Clique nas reticências (`...`) no lado direito da linha de um agente para acessar as seguintes opções:

* **Editar configuração:** abre o editor de configuração, permitindo que você modifique a configuração deste agente como parte da implantação.
* **Usar a versão mais recente:** Se você criou várias versões de uma configuração, esta opção atualiza automaticamente a implantação para usar a versão de configuração salva mais recente.
* **Alterar versão da configuração:** Permite selecionar manualmente uma versão anterior específica da configuração para usar na implantação.
* **Remover da implantação:** remove completamente o agente e sua configuração desta implantação.

<Callout title="Versão do agente vs. Versão da configuração" variant="tip">
  É importante distinguir entre dois tipos de versões no <DNT>Fleet Control</DNT>:

  * **Versão do agente:** A versão do próprio software do agente (por exemplo, agente de infraestrutura v1.54.0), selecionada durante a etapa inicial de seleção do agente ao adicionar um agente a uma implantação.
  * **Versão da configuração:** A versão do arquivo de configuração que você criou para esse agente, gerenciada pelo menu de ações acima.

  Ao selecionar &quot;Usar versão mais recente&quot; ou &quot;Alterar versão de configuração&quot;, você está gerenciando a versão da configuração, não a versão do agente.
</Callout>

## Gerenciar entidade em uma frota

<Callout title="Importante: O princípio da herança da frota" variant="important">
  As frotas operam com base num princípio fundamental de **herança**. Isso significa que qualquer entidade gerenciada atribuída a uma frota se conformará automaticamente à última implantação ativa dessa frota para garantir a consistência da instrumentação. Entender esse comportamento é fundamental ao adicionar ou remover uma entidade, pois isso pode resultar na instalação ou desinstalação automática do agente.
</Callout>

### Remover entidade de uma frota

1. Navegue até a guia <DNT>summary</DNT> da frota que contém atualmente a entidade.
2. Na tabela <DNT>managed entities</DNT>, use a caixa de seleção para selecionar uma ou mais entidades que você deseja remover.
3. Clique em <DNT>Remove</DNT> que aparecer.
4. Um modal de verificação aparecerá, listando a entidade a ser removida. Para confirmar, digite `remove` no campo de texto.
5. Clique em <DNT>Remove entities</DNT>. A entidade agora está desatribuída da frota.

<Callout title="Importante: Revertendo para a configuração local" variant="caution">
  Quando uma entidade gerenciada é removida de uma frota, ela retorna automaticamente à sua **configuração local**original — o estado em que estava quando o Agente Control foi instalado pela primeira vez. Isso pode resultar na desinstalação do agente. Por exemplo, se a frota tiver o agente de infraestrutura e Fluent Bit instalados, mas a configuração local da entidade incluir apenas o agente Control, ambos os agentes serão desinstalados na remoção para garantir que correspondam ao estado local.
</Callout>

<img title="Remove entities confirmation" alt="The Remove entities confirmation panel with explanatory text, a text input field where the user must type 'REMOVE' to confirm, a table showing the entity to be removed (demo-cluster-eks-219-3), and the Remove entities button." src="/images/fleet-control-remove-me.webp" />

### Adicionar entidade gerenciada não atribuída a uma frota

Adicione uma entidade gerenciada não atribuída a uma frota criando uma nova implantação. Este processo associa a entidade à frota e à sua configuração implantar.

1. Navegue até a frota onde você deseja adicionar a entidade e clique em <DNT>**Create a deployment**</DNT>.
2. Dê à implantação um nome e uma descrição significativos.
3. Na tabela <DNT>managed entities</DNT>, na parte inferior da tela de implantação, clique no botão <DNT>**Add managed entities**</DNT>.
4. Um painel será aberto, exibindo uma lista de todas as suas entidades gerenciadas não atribuídas.
5. Use as caixas de seleção para escolher uma ou mais entidades que você deseja adicionar à frota.
6. Clique em <DNT>Add to deployment</DNT>. As entidades selecionadas agora aparecerão na tabela <DNT>managed entities</DNT> para esta implantação.
7. Clique em <DNT>Save draft</DNT> e, em seguida, em <DNT>Start deployment</DNT> para concluir o processo. As entidades agora serão atribuídas à nova frota.

<Callout title="Importante: Herdar a última implantação ativa" variant="important">
  Quando você adiciona uma entidade gerenciada a uma frota, uma implantação implícita ocorre em segundo plano. Esta implantação instala, atualiza ou remove automaticamente a instrumentação na nova entidade para garantir que ela corresponda perfeitamente à última **implantação ativa** da frota. Por exemplo, se a implantação ativa da frota incluir os agentes Infrastructure e Fluent Bit, mas a nova entidade tinha anteriormente apenas o agente OTel, o agente OTel será removido e os agentes <DNT>Infrastructure</DNT> e <DNT>Fluent Bit</DNT> serão instalados.
</Callout>

<img title="Add managed entities to deployment" alt="The Add managed entities selection pane showing a searchable table of available entities with checkboxes, allowing users to select which unassigned managed entities to add to the fleet via this deployment." src="/images/fleet-control-add-me-deployment.webp" />

## Use implantação canário

<DNT>Fleet Control</DNT> permite que você designe uma ou mais entidades gerenciadas como um &quot;canary&quot; durante o processo de implantação. Isso oferece um mecanismo de segurança crucial para a implementação de alterações.

Ao criar uma nova implantação, você pode usar as reticências (`...`) ao lado de qualquer entidade gerenciada na lista e designá-la como um canário.

Quando a implantação começa:

1. As alterações são implantadas *somente* na entidade gerenciada marcada como canários.
2. O processo de implantação é pausado e aguarda o sucesso da implantação canária.
3. Se qualquer implantação canário falhar, toda a implantação será interrompida imediatamente e nenhuma outra entidade gerenciada será afetada.
4. Se todas as implantações canárias forem bem-sucedidas, a implantação prossegue para o restante da entidade administrada na frota.

<Callout title="Importante: Não há reversão automática para canários com falha" variant="caution">
  Se uma implantação canária falhar, a entidade gerenciada na qual ela foi implantada permanecerá em um estado de **Failed**. Não há reversão automática para a configuração anterior. Você precisará criar uma nova implantação para reverter as alterações ou corrigir o problema.
</Callout>

<img title="Canary deployment enablement" alt="The deployment pane's Managed entities table with the ellipsis menu expanded, showing the 'Use as a canary' option to designate a managed entity for staged canary deployment." src="/images/fleet-control-canary-action.webp" />

## A página de configuração

<Callout title="Importante: as configurações são globais" variant="important">
  Todas as configurações criadas em <DNT>Fleet Control</DNT> são **globais**. Isso significa que eles estão disponíveis para uso em toda a sua organização New Relic e podem ser implantados em qualquer frota que precise deles, independentemente de quem criou a configuração.
</Callout>

Ao navegar para <DNT>New Relic Control → Configurations</DNT>, você chega à página principal de Configurações, que é a central para gerenciar todas as configurações dos seus agentes.

### Visão geral da página principal

Esta página fornece uma visão geral de alto nível de todas as configurações disponíveis em sua conta. A partir daqui, você pode:

* **Pesquisar configuração:** Utilize a barra de pesquisa para localizar uma configuração específica por seu nome ou tipo de agente associado.

A página exibe uma tabela da sua configuração com as seguintes colunas:

* **Configuração:** O nome da configuração.
* **agente:** O tipo de agente ao qual a configuração se aplica.
* **Revisões:** O número de versões existentes para a configuração.
* **Tipo:** O tipo de entidade gerenciada (por exemplo, Cluster Kubernetes, Host Linux ou Host Windows).
* **Última modificação:** um timestamp da última atualização.

### Ações de configuração

Na tabela de configuração principal, você pode clicar nas reticências (`...`) na extrema direita de qualquer configuração para **Edit-** la. Isso abre o painel de edição principal, onde você pode modificar o arquivo de configuração. Esta visualização também fornece botões para **Download** o arquivo de configuração para sua máquina local ou **Copy** seu conteúdo para sua área de transferência.

### Compreendendo a configuração

A estrutura de um arquivo de configuração depende do **Tipo de Entidade Gerenciada** que você selecionar (Cluster Kubernetes, Host Linux ou Host Windows).

* **Para clusters Kubernetes,** a configuração que você cria na interface é um arquivo `values.yaml`. Durante a implantação, <DNT>Fleet Control</DNT> empacota este arquivo com o chart Helm do agente para configurá-lo em seu cluster. Os modelos para agentes do Kubernetes são estruturados como um arquivo de Helm chart `values.yaml` canônico. Perto do topo do template, você encontrará um link comentado para o chart Helm do agente no [repositório do GitHub de Helm Charts da New Relic](https://github.com/newrelic/helm-charts). Isso permite que você consulte facilmente o esquema, os valores disponíveis e outras documentações específicas do agente.
* **Para hosts,** o formato do arquivo de configuração é específico do agente que está sendo implantado.

<Callout title="Filtragem de configuração em implantações" variant="tip">
  Ao criar uma implantação para uma frota, o sistema filtra automaticamente as configurações disponíveis para exibir apenas aquelas que correspondem ao tipo de entidade gerenciada da frota. Por exemplo:

  * Uma **frota de clusters Kubernetes** exibirá apenas configurações criadas para clusters Kubernetes.
  * Uma **frota de hosts Linux** exibirá apenas configurações criadas para hosts Linux.
  * Uma **frota de hosts Windows** exibirá apenas configurações criadas para hosts Windows.

  Essa filtragem automática garante que você não implante acidentalmente uma configuração incompatível em suas entidades gerenciadas, mantendo a consistência e evitando falhas de implantação.
</Callout>

<Callout title="Melhores práticas de segurança: Gerenciamento de dados sensíveis no Kubernetes" variant="important">
  Para implantações do Kubernetes, nunca inclua dados confidenciais como seu `licenseKey` diretamente em arquivos de configuração. Em vez disso, use o gerenciamento de segredos do Agent Control para injetar valores sensíveis de forma segura em tempo de execução. Para mais informações, consulte nossa [documentação de gerenciamento de segredos](/docs/new-relic-control/agent-control/configuration#secrets-management).
</Callout>

### Crie uma configuração

Para criar uma nova configuração global do zero:

1. Na página principal de Configurações, clique em <DNT>**Create configuration**</DNT>.

2. No modal que aparece, você deve:

   * Selecione o <DNT>**managed entity type**</DNT> (Cluster Kubernetes, Host Linux ou Host Windows). Observe que o suporte ao Linux Host e ao Windows Host está atualmente em pré-visualização pública.
   * Selecione o <DNT>**Agent type**</DNT> (por exemplo, New Relic Infrastructure, NRDOT collector, Fluent Bit) no menu suspenso. Os tipos de agente disponíveis serão filtrados com base no tipo de entidade gerenciada que você selecionou.
   * Forneça um <DNT>**Name**</DNT> significativo e exclusivo para a configuração.

<img title="Create configuration modal" alt="The Create configuration modal with dropdown fields for selecting Managed entity type, Agent type, and a text field for entering the configuration name." src="/images/fleet-control-create-configuration-modal.webp" />

3. Clique em <DNT>**Continue**</DNT>. Isso abre a visualização de edição de configuração, que é dividida em dois painéis.

   * **Painel Esquerdo (Configuração Ativa):** É aqui que você cria sua configuração. Você pode digitar diretamente neste painel ou colar conteúdo de outra fonte.
   * **Painel direito (modelo):** este painel contém o modelo padrão mais recente para o tipo de agente selecionado. Esses modelos fornecem as configurações mínimas necessárias para executar o agente, mas também incluem muitas opções adicionais de comentários. Você pode usar esses exemplos como referência ou descomentá-los para criar uma configuração mais personalizada.

4. Um fluxo de trabalho comum é clicar em <DNT>**Copy**</DNT> no painel de modelos e colar o conteúdo no painel de configuração ativo à esquerda para usar como ponto de partida.

5. Quando estiver satisfeito com sua configuração no painel esquerdo, clique em <DNT>**Save**</DNT>. A visualização de edição será fechada, retornando à página principal de configurações, onde você poderá encontrar sua nova configuração.

## Excluir uma configuração e suas versões

Para excluir uma configuração, você deve primeiro excluir todas as suas versões individuais. A configuração pai é excluída automaticamente quando sua última versão é removida.

4. Selecione <DNT>**Delete**</DNT> no menu.
5. Repita este processo para todas as versões restantes. Assim que a última versão for excluída, a configuração pai será removida da lista na página principal de configurações.
