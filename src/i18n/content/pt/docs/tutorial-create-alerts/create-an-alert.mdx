Existem três maneiras de criar uma condição de alerta no New Relic: Com um gráfico, uma consulta NRQL ou com métricas clássicas. Embora o início do processo de criação seja diferente dependendo da opção escolhida, todas as etapas subsequentes são iguais. Isso significa que, depois de criar uma condição de uma maneira, a experiência que você ganha ao fazer isso será transferida para outras condições que você criar usando qualquer uma das outras opções.

Para este tutorial, incluímos etapas para todas as três opções mencionadas acima, para que você possa começar com sua primeira condição da maneira que desejar. Basta selecionar o caminho que deseja seguir no primeiro passo e ao final do procedimento você terá feito com sucesso sua primeira condição de alerta usando New Relic!

<Callout variant="tip">
  Se precisar de mais informações sobre qualquer uma das opções ou termos abordados nas etapas abaixo, confira nossos [documentos de alerta](/docs/alerts-applied-intelligence/overview/#concepts-terms).
</Callout>

          Embora usar gráficos seja uma ótima maneira de criar condições de alerta a partir de consultas pré-fabricadas, às vezes é necessário criá-los do zero. O processo para criar uma nova condição do alerta a partir de uma consulta NRQL é quase exatamente o mesmo que criá-la a partir de um gráfico: a principal diferença é que você escreve a consulta NRQL manualmente em vez de usar uma consulta pré-gerada.

          <Callout variant="tip">
            NRQL significa New Relic&apos;s Query Language — um método de consulta padronizado que alimenta todos os nossos gráficos e que você pode usar para analisar e consultar seus dados. Se esta é sua primeira vez trabalhando com NRQL, você pode conferir nossos [documentos sobre NRQL](/docs/query-your-data/nrql-new-relic-query-language/get-started/introduction-nrql-new-relics-query-language/) para saber mais sobre NRQL, incluindo [como ele funciona](/docs/query-your-data/nrql-new-relic-query-language/get-started/introduction-nrql-how-nrql-works/) e um [tutorial introdutório](/docs/query-your-data/nrql-new-relic-query-language/get-started/introduction-nrql-tutorial/).
          </Callout>

          2. Selecione <DNT>**New alert condition**</DNT>.

          3. Escreva a consulta NRQL para criar a condição. Por exemplo, se você quiser monitor transações lentas, você pode inserir:

             ```sql
             SELECT average(duration) from Transaction facet name
             ```

  <Step>
    ## Ajuste o comportamento do sinal

    <img title="Tuning your signal" alt="A screenshot displaying how to set options for a signal in an alert condition" src="/images/alerts_screenshot-crop_signal-tune.webp" />

    1. <DNT>**Set the data aggregation options**</DNT>:

       * <DNT>**Window duration**</DNT>: use a configuração <DNT>**Window duration**</DNT> para determinar a frequência com que agregamos seus dados, como agrupamentos a cada 5 minutos ou a cada hora. A configuração deste recurso depende do tipo de dados que você está monitorando. Se não tiver certeza de como deveria ser, você pode deixá-lo na configuração padrão. Você pode saber mais em nosso [documento Crie seu primeiro alerta](/docs/alerts-applied-intelligence/new-relic-alerts/get-started/your-first-nrql-condition/#window-duration).
       * <DNT>**Sliding window aggregation**</DNT>: escolha se deseja ou não usar [agregação de janela deslizante](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/#sliding-window-aggregation). Este recurso cria gráficos mais suaves ao lidar com dados erráticos, criando janelas de agregação sobrepostas. Este recurso está desativado por padrão e recomendamos mantê-lo desativado no seu primeiro alerta
       * <DNT>**Streaming method**</DNT>: As opções mais comumente usadas são <DNT>**Event flow**</DNT> e <DNT>**Event timer**</DNT>. <DNT>**Event flow**</DNT> é mais adequado para relatórios de dados frequentes e consistentes, enquanto <DNT>**Event timer**</DNT> é mais adequado para dados que são relatados de forma inconsistente, infrequente ou ingeridos em lotes. Se você quiser saber mais sobre as configurações individuais para ver qual método deve ser usado em cada cenário, consulte [nossa documentação](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/understand-technical-concepts/streaming-alerts-key-terms-concepts/#aggregation-methods)

    2. <DNT>**Set the timing option**</DNT>: Isso determina quanto tempo esperamos por eventos em cada janela antes de avaliá-los. Um atraso maior significará menos alertas falsos, mas também significará um tempo de inatividade potencial mais longo antes que um evento de alerta seja aberto. Esse recurso também é exibido de forma diferente dependendo da opção selecionada: é chamado de <DNT>**Delay**</DNT> nas opções <DNT>**Event flow**</DNT> e <DNT>**Cadence**</DNT> e <DNT>**Timer**</DNT> em <DNT>**Event timer**</DNT>. Para o seu primeiro alerta, recomendamos manter a configuração de tempo padrão.

    3. <DNT>**Select the gap filling strategy**</DNT>: Preenchimento de lacunas é um recurso reativo que preenche períodos em branco no relatório de dados com valor Sintético definido pelo usuário. Se você acredita que o relatório de dados será esporádico, você pode evitar falsos alertas preenchendo janelas de agregação vazias com esses dados. Para seu primeiro alerta, selecione <DNT>**None**</DNT> em <DNT>**Gap-filling strategy**</DNT>.

    4. <DNT>**Select the evaluation delay**</DNT>: Se usada, esta opção habilita um período de tempo para o New Relic aguardar antes de avaliar um novo sinal em relação ao limite definido. É útil para evitar eventos de alerta falsos positivos quando um fluxo de dados começa a reportar. Para o seu primeiro alerta, você pode deixar o botão de alternância desativado, a menos que saiba que essa opção se aplica aos seus dados.

    5. Selecione <DNT>**Next**</DNT> quando terminar para salvar seu progresso e passar para a próxima etapa.
  </Step>

  <Step>
    ## Defina o limite estático

    /* &lt;img title=&quot;Definir limites estáticos&quot; alt=&quot;Uma captura de tela exibindo como definir limites estáticos para uma condição de alerta&quot; src=&quot;/images/alerts_screenshot-crop_set-thresholds.webp&quot; /&gt; */

    1. Comece selecionando se o limite terá um status <DNT>**Critical**</DNT> ou <DNT>**Warning**</DNT> na seção <DNT>**Security level**</DNT> .

    2. Defina o limite que, quando atingido, abrirá um evento de alerta. Esses valores mudarão dependendo do que você deseja que a condição faça. Por exemplo, se você quisesse criar uma condição para uma mensagem de erro, poderia definir o limite como <DNT>**equal to 1 at least once in 5 minutes**</DNT>, mas se quisesse criar uma condição para períodos mais longos de problemas de latência, poderia definir o limite como <DNT>**above or equal to 50 for at least 15 minutes**</DNT>.

    3. Se desejar adicionar outro limite, selecione <DNT>**Add threshold**</DNT> ou <DNT>**Add lost signal threshold**</DNT>, dependendo de suas necessidades. Limites de sinais perdidos são importantes para monitoramento de entidade que podem deixar de reportar ou consultar que podem [retornar valores nulos](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/#query-order).

    4. Selecione <DNT>**Next**</DNT> quando terminar para salvar seu progresso e passar para a próxima etapa.
  </Step>

  <Step>
    ## Adicionar detalhes da condição do alerta

    /* &lt;img title=&quot;Detalhes da condição de alerta&quot; alt=&quot;Uma captura de tela exibindo como criar os detalhes para um alerta&quot; src=&quot;/images/alerts_screenshot-crop_condition-details.webp&quot; /&gt; */

    1. Adicione um nome para seu alerta no campo <DNT>**Name**</DNT> . Isso o ajudará a identificar essa condição, bem como a encontrá-la no futuro, caso precise editá-la ou removê-la.

    2. Escolha a política à qual esta condição se conectará. Se você tiver um <DNT>**Existing policy**</DNT>, selecione-o no menu suspenso. Caso contrário, você precisará [criar uma política](/docs/alerts-applied-intelligence/new-relic-alerts/alert-policies/create-edit-or-find-alert-policy/#alert-policy-name) selecionando <DNT>**New policy**</DNT> e criando-a aqui.

    3. Defina quaisquer configurações adicionais para o alerta. Isso inclui definir quanto tempo até que eventos de alerta abertos sejam fechados automaticamente e inserir quaisquer [descrições personalizadas](/docs/alerts/create-alert/condition-details/custom-alert-event-descriptions) para os eventos de alerta.

    4. Selecione <DNT>**Save condition**</DNT> para finalizar.
  </Step>
</Steps>

## Crie um alerta usando o construtor de alerta clássico

<Callout variant="tip">
  Recomendamos usar os métodos acima para criar um alerta NRQL. Se desejar criar um alerta clássico, você pode usar o formulário original do construtor de alerta.
</Callout>

/\* &lt;img title=&quot;Criar um alerta com o construtor&quot; alt=&quot;Uma captura de tela mostrando o construtor de alertas&quot; src=&quot;/images/alerts\_screenshot-crop\_alert-builder.webp&quot; /&gt; \*/

1. Em <DNT>**one.newrelic.com**</DNT>, selecione [<DNT>**Alerts**</DNT>](https://one.newrelic.com/alerts-ai) e depois [<DNT>**Alert policies**</DNT>](https://one.newrelic.com/alerts-ai/condition-builder/policy-list).

2. Clique na política para a qual deseja criar a condição clássica do alerta ou crie uma nova política.

3. Selecione <DNT>**New alert condition**</DNT> e depois selecione <DNT>**Build a classic alert**</DNT>.

4. Construa sua condição a partir de qualquer uma das categorias listadas. Isso determina sobre o que a condição do alerta notifica você.

5. Selecione um tipo de condição para restringir as opções para a próxima etapa. Também pode ser necessário selecionar um escopo.

6. Selecione a entidade ou entidades que pretende monitor. Use a barra de pesquisa para restringir a lista de entidades. Se você selecionou uma condição de tipo de métrica de aplicativo APM, poderá optar por destino dinâmico de aplicativos que compartilham determinada tag.

7. Defina seu limite de alerta e quaisquer outras configurações de condição. Nomeie sua condição e salve.

Você acabou de criar uma condição do alerta! Oferecemos diferentes opções de configuração para alerta, para que você possa ajustar seu alerta de acordo com suas necessidades de monitoramento. Você também pode criar condições a partir da consulta NRQL, que abordaremos a seguir nesta série de tutoriais.

## Aproveite mais os alertas [#more-alerting]

Temos alguns recursos de alertas avançados que podem ajudar você a tornar os alertas mais eficientes e gerenciáveis para sua equipe:

* Se você deseja maior controle sobre onde e quando recebe alertas e garantir que as pessoas certas sejam notificadas sobre problemas, veja como usar [Workflows](/docs/alerts/get-notified/alert-event-workflows/).
* Se você deseja ativar a notificação de alerta crítico quase instantânea, ajudando a reduzir seu tempo médio de resolução (MTTR), veja como usar nossa [detecção de anomalia](/docs/alerts-applied-intelligence/applied-intelligence/anomaly-detection/anomaly-detection-applied-intelligence/).
* Se quiser usar APIs para gerenciar alertas da New Relic, você pode fazer isso com [REST API ou GraphQL](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/alerts-api/intro-alerts-api/).
