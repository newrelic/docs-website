Quando seus gerenciadores de filas do IBM MQ estão sendo executados em infraestrutura tradicional, como VMs, servidores bare metal ou instâncias de EC2, você precisa de uma maneira confiável de monitorar seu desempenho sem a complexidade da orquestração de contêineres. Este guia mostra como configurar o monitoramento abrangente do IBM MQ usando o [coletor da Distribution of OpenTelemetry (NRDOT) do New Relic](/docs/opentelemetry/nrdot/nrdot-collector) ou o [OpenTelemetry Collector Contrib](https://github.com/open-telemetry/opentelemetry-collector-contrib).

Você configurará o coletor para coletar automaticamente as profundidades de fila, métricas de taxas de transferência e o status do canal dos seus gerenciadores de filas e, em seguida, enviará esses dados para o New Relic, onde eles aparecem como entidades organizadas com dashboards e alertas prontos para uso. Essa abordagem funciona melhor quando você tem um conjunto estável de gerenciadores de filas que não mudam com frequência.

<Callout variant="tip">
  Se, em vez disso, seus gerenciadores de filas forem executados no Kubernetes, consulte [Monitorar o IBM MQ no Kubernetes](/docs/opentelemetry/integrations/ibm-mq/kubernetes).
</Callout>

## Antes de você começar [#prereq]

Você precisará desses componentes antes de configurar o coletor:

* Conta do New Relic com uma [chave de licença](/docs/apis/intro-apis/new-relic-api-keys/#ingest-license-key)válida
* [Endpoint OTLP](/docs/opentelemetry/best-practices/opentelemetry-otlp) da New Relic para sua região
* Gerenciadores de filas do IBM MQ em execução com canais de conexão de servidor que têm permissões de consulta PCF
* Exportador do IBM MQ [mq-metric-samples](https://github.com/ibm-messaging/mq-metric-samples) instalado e expondo métrica em cada gerenciador de filas. Um exportador por gerenciador de filas
* Host Linux para instalar o coletor NRDOT ou o OpenTelemetry Collector Contrib

<Callout variant="tip">
  Recomendamos ativar as estatísticas MQI em seus gerenciadores de filas usando `ALTER QMGR STATMQI(ON) STATQ(ON)` para obter melhores métricas de taxas de transferência.
</Callout>

        <Step>
          ### Instalar o coletor NRDOT [#install-nrdot]

          Baixe e instale o pacote NRDOT para a sua distribuição Linux. Substitua `<NRDOT_VERSION>` pela tag de versão mais recente da página [nrdot-collector-releases](https://github.com/newrelic/nrdot-collector-releases/releases) (por exemplo, `v0.12.0`).

          * Para Debian/Ubuntu:

            ```bash
            curl -LO https://github.com/newrelic/nrdot-collector-releases/releases/download/<NRDOT_VERSION>/nrdot-collector_<NRDOT_VERSION>_linux_amd64.deb
            sudo dpkg -i nrdot-collector_<NRDOT_VERSION>_linux_amd64.deb
            ```

          * Para RHEL/Rocky/Amazon Linux:

            ```bash
            curl -LO https://github.com/newrelic/nrdot-collector-releases/releases/download/<NRDOT_VERSION>/nrdot-collector_<NRDOT_VERSION>_linux_amd64.rpm
            sudo dnf install ./nrdot-collector_<NRDOT_VERSION>_linux_amd64.rpm
            ```
        </Step>

        <Step>
          ### Configurar o NRDOT Collector [#config-nrdot]

          Configure o NRDOT Collector para coletar métricas do IBM MQ e enviá-las para o New Relic. Esta configuração lida com três tarefas principais:

          * Conecte-se aos seus gerenciadores de filas
          * Limpe os dados
          * Enviar os dados para o New Relic

          1. Para criar o diretório e o arquivo de configuração do coletor com as permissões necessárias, execute:

          ```bash
          # Config directory — group-readable by nrdot only (it will hold secrets)
          sudo install -d -o root -g nrdot -m 0750 /etc/nrdot-collector

          # Create the config file with correct permissions
          sudo install -o root -g nrdot -m 0640 /dev/null /etc/nrdot-collector/ibmmq-config.yaml
          sudo -u nrdot ${EDITOR:-vi} /etc/nrdot-collector/ibmmq-config.yaml
          ```

          2. Cole a seguinte configuração no arquivo de configuração criado, substituindo os espaços reservados pelos seus valores reais:

          ```yaml
          extensions:
            health_check:
              endpoint: 127.0.0.1:13133

          processors:

            filter/ibmmq-overhead:
              metrics:
                exclude:
                  match_type: regexp
                  metric_names:
                    - "^go_.*"
                    - "^process_.*"
                    - "^promhttp_.*"
                    - "^scrape_.*"

            filter/ibmmq-queues:
              metrics:
                datapoint:
                  - 'attributes["queue"] != nil and IsMatch(attributes["queue"], "^SYSTEM\\.(ADMIN\\.|MQSC\\.|DEFAULT\\.|AUTH\\.|CHANNEL\\.|CHLAUTH\\.|CICS\\.|SYNCPOINT\\.|INTERNAL\\.|PENDING\\.|PROTECTION\\.|BROKER\\.|AMQP\\.|DOTNET\\.|REST\\.|RETAINED\\.|SELECTION\\.|DURABLE\\.|HIERARCHY\\.|DDELAY\\.)")'
                  - 'attributes["queue"] != nil and IsMatch(attributes["queue"], "^SYSTEM\\.CLUSTER\\.(COMMAND|HISTORY)\\.QUEUE$")'
                  - 'attributes["queue"] != nil and IsMatch(attributes["queue"], "^AMQ\\.")'

            resourcedetection:
              detectors: [env, ec2, system]
              system:
                resource_attributes:
                  host.name:
                    enabled: true
                  host.id:
                    enabled: true

            transform/ibmmq-cleanup:
              metric_statements:
                - context: resource
                  statements:
                    - delete_key(attributes, "server.address")
                    - delete_key(attributes, "server.port")
                    - delete_key(attributes, "url.scheme")
                - context: datapoint
                  statements:
                    # Rename the injected identity labels to their OTel dotted form — the
                    # names New Relic's IBM MQ entity synthesis keys on. qmgr / queue stay raw.
                    - set(attributes["target.name"], attributes["targetName"]) where attributes["targetName"] != nil
                    - delete_key(attributes, "targetName")
                    - set(attributes["cluster.name"], attributes["clusterName"]) where attributes["clusterName"] != nil
                    - delete_key(attributes, "clusterName")
                    - delete_key(attributes, "instance")
                    - delete_key(attributes, "job")

            memory_limiter/ibmmq:
              check_interval: 1s
              limit_mib: 512
              spike_limit_mib: 256

            cumulativetodelta/ibmmq: {}

            batch/ibmmq:
              send_batch_size: 1000
              timeout: 200ms

          exporters:

            otlphttp/ibmmq:
              endpoint: ${env:NEW_RELIC_OTLP_ENDPOINT}
              headers:
                api-key: ${env:NEW_RELIC_LICENSE_KEY}

          receivers:

            prometheus/ibmmq-qm1:
              config:
                scrape_configs:
                  - job_name: 'ibmmq-qm1'
                    scrape_interval: ${env:IBMMQ_SCRAPE_INTERVAL:-60s}
                    scrape_timeout: 15s
                    static_configs:
                      - targets:
                          - "${env:IBMMQ_QM1_ENDPOINT:-localhost:9157}"
                        labels:
                          targetName: "${env:TARGET_NAME}"
                          clusterName: "${env:IBMMQ_CLUSTER_NAME:-}"

          service:
            pipelines:
              metrics/ibmmq-qm1:
                receivers: [prometheus/ibmmq-qm1]
                processors:
                  - filter/ibmmq-overhead
                  - filter/ibmmq-queues
                  - resourcedetection
                  - transform/ibmmq-cleanup
                  - memory_limiter/ibmmq
                  - cumulativetodelta/ibmmq
                  - batch/ibmmq
                exporters: [otlphttp/ibmmq]

            extensions: [health_check]
          ```

          <Callout variant="important">
            Não reordene nem remova processadores. A ordem da cadeia de processadores é fundamental para a síntese de entidade. O processador `transform/ibmmq-cleanup` deve ser executado após o processador `resourcedetection`. Inverter essa ordem faz com que as métricas do IBM MQ se associem à própria entidade do coletor em vez de entidades `IBMMQ_MANAGER`, o que impede que os dados sejam exibidos nos dashboards.
          </Callout>

          #### O que esta configuração faz [#config-explanation]

          Esta configuração cria um pipeline que coleta métricas de gerenciadores de filas do IBM MQ e roteia dados estruturados para o New Relic. Cada componente executa uma função específica:

          | Componente                | Descrição                                                                                                                                                                                                                                                                                                                                                       |
          | ------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
          | `health_check`            | Fornece um endpoint de integridade em `127.0.0.1:13133` que retorna `{"status":"Server available"}` para verificar se o coletor está em execução e para solucionar problemas de inicialização.                                                                                                                                                                  |
          | `prometheus/ibmmq-qm1`    | Conecta-se ao exportador de cada gerenciador de filas a cada 60 segundos para coletar métricas. Cada gerenciador de filas usa um receptor independente; se um gerenciador de filas falhar, os receptores restantes continuarão operando normalmente. Para adicionar mais gerenciadores de filas, consulte a seção [Adicionar outro gerenciador de filas](#add). |
          | `filter/ibmmq-overhead`   | Remove as métricas próprias internas do exportador (como `go_*` e `process_*`) que não contêm um sinal do IBM MQ. Isso otimiza os custos de ingestão de dados ao filtrar a telemetria desnecessária.                                                                                                                                                            |
          | `filter/ibmmq-queues`     | Exclui as filas internas do sistema IBM MQ (`SYSTEM.*` e `AMQ.*`) para que apenas as filas do aplicativo se tornem entidades `IBMMQ_QUEUE`. Isso reduz a contagem total de entidades e concentra o monitoramento em ativos críticos.                                                                                                                            |
          | `transform/ibmmq-cleanup` | Este é um componente crítico que mapeia métricas para as entidades apropriadas do IBM MQ no New Relic. Sem esse componente, os dados aparecem como métricas genéricas do coletor em vez de sintetizar as entidades `IBMMQ_MANAGER` e `IBMMQ_QUEUE` com dashboards e alertas associados.                                                                         |
          | `memory_limiter/ibmmq`    | Limita o uso de memória a 512MB (com tolerância a picos de 256MB) para evitar que o coletor exceda os limites de recursos do sistema.                                                                                                                                                                                                                           |
          | `batch/ibmmq`             | Agrupa as métricas antes da transmissão para reduzir a sobrecarga da rede, agrupando até 1.000 pontos de dados por solicitação.                                                                                                                                                                                                                                 |
          | `otlphttp/ibmmq`          | Exporta as métricas processadas para o New Relic usando a chave de licença configurada para autenticação e o endpoint regional designado.                                                                                                                                                                                                                       |

          Cada gerenciador de filas usa um pipeline independente para que um problema com uma instância não afete as outras. A ordem dos processadores é crítica, não reordene ou remova processadores, ou as métricas falharão ao mapear corretamente para as entidades do IBM MQ.
        </Step>

        <Step>
          ### Definir variáveis de ambiente do NRDOT [#set-nrdot]

          A configuração do seu coletor usa variáveis de ambiente para valores específicos da implantação, como a chave de licença do New Relic e os endpoints do gerenciador de filas. Essa abordagem mantém os segredos separados dos arquivos de configuração e facilita o ajuste das configurações sem editar o YAML.

          1. Crie um arquivo de ambiente para o coletor NRDOT:

          ```bash
          # Owner nrdot, mode 0600 — this file holds the license key
          sudo install -o nrdot -g nrdot -m 0600 /dev/null /etc/nrdot-collector/ibmmq-env
          sudo -u nrdot ${EDITOR:-vi} /etc/nrdot-collector/ibmmq-env
          ```

          2. Adicione seu valor como `KEY=VALUE` linhas ao arquivo, substituindo os espaços reservados por seus valores reais:

          ```ini
          # /etc/nrdot-collector/ibmmq-env

          # --- Required ---

          # New Relic ingest license key (40 chars, suffix NRAL)
          NEW_RELIC_LICENSE_KEY=<YOUR-LICENSE-KEY>

          # Stable host identity — first segment of every IBMMQ entity GUID
          TARGET_NAME=<YOUR-HOSTNAME>

          # mq-metric-samples exporter endpoints, one per queue manager
          IBMMQ_QM1_ENDPOINT=localhost:9157

          # --- Optional (shown with defaults) ---

          # OTLP endpoint: US prod default shown; For more information, refer to [New Relic OTLP endpoint](https://docs.newrelic.com/docs/opentelemetry/best-practices/opentelemetry-otlp).
          NEW_RELIC_OTLP_ENDPOINT=https://otlp.nr-data.net:4318

          # Optional tag grouping for the IBMMQ_MANAGER entity (does not affect the entity GUID).
          # Leave unset to omit cluster.name (nullable); set it to group QMs under one cluster tag.
          # IBMMQ_CLUSTER_NAME=prod-mq-cluster

          # Scrape interval — must be >= the mq-metric-samples pollInterval (default 60s)
          IBMMQ_SCRAPE_INTERVAL=60s
          ```

          <Callout variant="caution">
            Não altere o valor de `TARGET_NAME` após a implantação inicial. Esse valor forma o primeiro segmento de cada GUID de entidade `IBMMQ_MANAGER` e `IBMMQ_QUEUE` (`target.name:qmgr`). Alterar esse valor após a implantação cria novas entidades e deixa as existentes órfãs, o que quebra os dashboards e alertas associados. Selecione uma string estável durante a configuração inicial, como o QMID do gerenciador de filas. Para obter mais informações, consulte [O Modelo de Entidade do IBM MQ](/docs/opentelemetry/integrations/ibm-mq/overview#entity-model).
          </Callout>

          Defina as seguintes variáveis de ambiente no seu shell para verificar a configuração antes de iniciar o coletor:

          | Variável                  | Obrigatório | Descrição                                                                                                                                                                   |
          | ------------------------- | ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
          | `NEW_RELIC_LICENSE_KEY`   | Sim         | Sua [chave de licença de ingestão](/docs/apis/intro-apis/new-relic-api-keys/#ingest-license-key) da New Relic.                                                              |
          | `TARGET_NAME`             | Sim         | Um identificador para o seu host, como nome do host ou ID do gerenciador de filas. Não altere isso após a implantação inicial, pois se torna parte dos IDs da sua entidade. |
          | `IBMMQ_QM1_ENDPOINT`      | Sim         | Localização do exportador do seu primeiro gerenciador de filas. Por exemplo, `localhost:9157` se estiver em execução no mesmo host.                                         |
          | `NEW_RELIC_OTLP_ENDPOINT` | Não         | Endpoint OTLP da New Relic para sua região. Para mais informações, consulte o [endpoint OTLP da New Relic](/docs/opentelemetry/best-practices/opentelemetry-otlp).          |
          | `IBMMQ_CLUSTER_NAME`      | Não         | Nome do Cluster para agrupar seus gerenciadores de filas no New Relic.                                                                                                      |
          | `IBMMQ_SCRAPE_INTERVAL`   | Não         | Intervalo de coleta para exportar métricas para o New Relic. O intervalo de coleta padrão é de 60 segundos.                                                                 |
        </Step>

        <Step>
          ### Configure o serviço systemd do NRDOT [#nrdot-systemd]

          1. Crie um arquivo drop-in `systemd` para carregar os arquivos de configuração e ambiente.

          ```bash
          sudo mkdir -p /etc/systemd/system/nrdot-collector.service.d
          sudo tee /etc/systemd/system/nrdot-collector.service.d/ibmmq.conf > /dev/null << 'EOF'
          [Service]
          EnvironmentFile=/etc/nrdot-collector/ibmmq-env
          ExecStart=
          ExecStart=/usr/bin/nrdot-collector --config /etc/nrdot-collector/ibmmq-config.yaml
          MemoryMax=1G
          LimitNOFILE=65536
          NoNewPrivileges=true
          ProtectSystem=strict
          ProtectHome=true
          EOF
          ```

          A linha `ExecStart=` vazia limpa o comando padrão, e a segunda linha define o comando personalizado. Isso evita que o coletor inicie duas vezes. As outras configurações carregam o arquivo de ambiente, limitam o uso de memória e melhoram a segurança.

          2. Após criar o drop-in, verifique se o systemd pode analisá-lo sem erros:

          ```bash
          sudo systemd-analyze verify /etc/systemd/system/nrdot-collector.service
          # No output means no syntax errors
          ```
        </Step>

        <Step>
          ### Iniciar o coletor NRDOT e verificar [#start-nrdot]

          1. Inicie o coletor e ative-o para ser executado automaticamente na inicialização:

          ```bash
          sudo systemctl daemon-reload
          sudo systemctl enable --now nrdot-collector.service
          sudo systemctl status nrdot-collector.service --no-pager
          ```

          O comando `status` deve mostrar `Active: active (running)`. Se você vir `Active: failed`, verifique os logs com `journalctl -u nrdot-collector.service -n 200 --no-pager`.

          2. Verifique se o coletor está saudável:

          ```bash
          curl -sS http://127.0.0.1:13133
          # Expected: {"status":"Server available", ...}
          ```

          3. Confirme se sua métrica do IBM MQ está fluindo para o New Relic. Aguarde 60 segundos após o início do coletor e, em seguida, verifique seus dados usando a consulta de verificação em [Encontre e consulte seus dados](/docs/opentelemetry/integrations/ibm-mq/view-data).
        </Step>

        <Step>
          ### Visualize seus dados no New Relic [#view-nrdot]

          Assim que seu coletor estiver em execução e as métricas estiverem fluindo, você verá seus gerenciadores de filas como entidades `IBMMQ_MANAGER` no New Relic, com suas filas como entidades `IBMMQ_QUEUE` filhas. Para obter detalhes sobre como encontrar seus dados, executar consultas e configurar dashboards e alertas, consulte [Encontrar e consultar seus dados](/docs/opentelemetry/integrations/ibm-mq/view-data).
        </Step>

          | `filter/ibmmq-overhead`   | Remove as métricas próprias internas do exportador (como `go_*` e `process_*`) que não contêm um sinal do IBM MQ. Isso otimiza os custos de ingestão de dados ao filtrar a telemetria desnecessária.                                                                                                                                                            |
          | `filter/ibmmq-queues`     | Exclui as filas internas do sistema IBM MQ (`SYSTEM.*` e `AMQ.*`) para que apenas as filas do aplicativo se tornem entidades `IBMMQ_QUEUE`. Isso reduz a contagem total de entidades e concentra o monitoramento em ativos críticos.                                                                                                                            |
          | `transform/ibmmq-cleanup` | Este é um componente crítico que mapeia métricas para as entidades apropriadas do IBM MQ no New Relic. Sem esse componente, os dados aparecem como métricas genéricas do coletor em vez de sintetizar as entidades `IBMMQ_MANAGER` e `IBMMQ_QUEUE` com dashboards e alertas associados.                                                                         |
          | `memory_limiter/ibmmq`    | Limita o uso de memória a 512MB (com tolerância a picos de 256MB) para evitar que o coletor exceda os limites de recursos do sistema.                                                                                                                                                                                                                           |
          | `batch/ibmmq`             | Agrupa as métricas antes da transmissão para reduzir a sobrecarga da rede, agrupando até 1.000 pontos de dados por solicitação.                                                                                                                                                                                                                                 |
          | `otlphttp/ibmmq`          | Exporta as métricas processadas para o New Relic usando a chave de licença configurada para autenticação e o endpoint regional designado.                                                                                                                                                                                                                       |

          Cada gerenciador de filas usa um pipeline independente para que um problema com uma instância não afete as outras. A ordem dos processadores é crítica, não reordene ou remova processadores, ou as métricas falharão ao mapear corretamente para as entidades do IBM MQ.
        </Step>

        <Step>
          ### Definir variáveis de ambiente do OpenTelemetry [#set-otelcol]

          A configuração do seu coletor usa variáveis de ambiente para valores específicos da implantação, como a chave de licença do New Relic e os endpoints do gerenciador de filas. Essa abordagem mantém os segredos separados dos arquivos de configuração e facilita o ajuste das configurações sem editar o YAML.

          1. Crie um arquivo de ambiente para o OpenTelemetry Collector:

          ```bash
          # Owner root, mode 0600 — this file holds the license key
          sudo touch /etc/otelcol-contrib/ibmmq-env
          sudo chmod 600 /etc/otelcol-contrib/ibmmq-env
          sudo ${EDITOR:-vi} /etc/otelcol-contrib/ibmmq-env
          ```

          2. Adicione seu valor como `KEY=VALUE` linhas ao arquivo, substituindo os espaços reservados por seus valores reais:

          ```ini
          # /etc/otelcol-contrib/ibmmq-env

          # --- Required ---

          # New Relic ingest license key (40 chars, suffix NRAL)
          NEW_RELIC_LICENSE_KEY=<YOUR-LICENSE-KEY>

          # Stable host identity — first segment of every IBMMQ entity GUID
          TARGET_NAME=<YOUR-HOSTNAME>

          # mq-metric-samples exporter endpoints, one per queue manager
          IBMMQ_QM1_ENDPOINT=localhost:9157

          # --- Optional (shown with defaults) ---

          # OTLP endpoint: US prod default shown; For more information, refer to [New Relic OTLP endpoint](https://docs.newrelic.com/docs/opentelemetry/best-practices/opentelemetry-otlp).
          NEW_RELIC_OTLP_ENDPOINT=https://otlp.nr-data.net:4318

          # Optional tag grouping for the IBMMQ_MANAGER entity (does not affect the entity GUID).
          # Leave unset to omit cluster.name (nullable); set it to group QMs under one cluster tag.
          # IBMMQ_CLUSTER_NAME=prod-mq-cluster

          # Scrape interval — must be >= the mq-metric-samples pollInterval (default 60s)
          IBMMQ_SCRAPE_INTERVAL=60s
          ```

          <Callout variant="caution">
            Não altere o valor de `TARGET_NAME` após a implantação inicial. Esse valor forma o primeiro segmento de cada GUID de entidade `IBMMQ_MANAGER` e `IBMMQ_QUEUE` (`target.name:qmgr`). Alterar esse valor após a implantação cria novas entidades e deixa as existentes órfãs, o que quebra os dashboards e alertas associados. Selecione uma string estável durante a configuração inicial, como o QMID do gerenciador de filas. Para obter mais informações, consulte [O Modelo de Entidade do IBM MQ](/docs/opentelemetry/integrations/ibm-mq/overview#entity-model).
          </Callout>

          Defina as seguintes variáveis de ambiente no seu shell para verificar a configuração antes de iniciar o coletor:

          | Variável                  | Obrigatório | Descrição                                                                                                                                                                   |
          | ------------------------- | ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
          | `NEW_RELIC_LICENSE_KEY`   | Sim         | Sua [chave de licença de ingestão](/docs/apis/intro-apis/new-relic-api-keys/#ingest-license-key) da New Relic.                                                              |
          | `TARGET_NAME`             | Sim         | Um identificador para o seu host, como nome do host ou ID do gerenciador de filas. Não altere isso após a implantação inicial, pois se torna parte dos IDs da sua entidade. |
          | `IBMMQ_QM1_ENDPOINT`      | Sim         | Localização do exportador do seu primeiro gerenciador de filas. Por exemplo, `localhost:9157` se estiver em execução no mesmo host.                                         |
          | `NEW_RELIC_OTLP_ENDPOINT` | Não         | Endpoint OTLP da New Relic para sua região. Para mais informações, consulte o [endpoint OTLP da New Relic](/docs/opentelemetry/best-practices/opentelemetry-otlp).          |
          | `IBMMQ_CLUSTER_NAME`      | Não         | Nome do Cluster para agrupar seus gerenciadores de filas no New Relic.                                                                                                      |
          | `IBMMQ_SCRAPE_INTERVAL`   | Não         | Intervalo de coleta para exportar métricas para o New Relic. O intervalo de coleta padrão é de 60 segundos.                                                                 |
        </Step>

        <Step>
          ### Configure o serviço systemd do OpenTelemetry [#otelcol-systemd]

          1. Crie um arquivo de serviço do systemd para o OpenTelemetry Collector:

          ```poweshell
            sudo tee /etc/systemd/system/otelcol-contrib.service > /dev/null << 'EOF'
            [Unit]
            Description=OpenTelemetry Collector Contrib
            After=network.target

            [Service]
            Type=simple
            User=nobody
            Group=nogroup
            ExecStart=/usr/bin/otelcol-contrib --config=/etc/otelcol-contrib/ibmmq-config.yaml
            EnvironmentFile=/etc/otelcol-contrib/ibmmq-env
            Restart=on-failure
            RestartSec=5
            StandardOutput=journal
            StandardError=journal
            SyslogIdentifier=otelcol-contrib
            KillMode=mixed
            KillSignal=SIGTERM
            MemoryMax=1G
            LimitNOFILE=65536
            NoNewPrivileges=true
            ProtectSystem=strict
            ProtectHome=true
            ReadWritePaths=/tmp

            [Install]
            WantedBy=multi-user.target
            EOF
          ```

          O serviço é executado como `nobody:nogroup` por segurança, carrega o arquivo de ambiente e inclui limites de memória e fortalecimento de segurança.

          2. Após criar o serviço, verifique se o systemd consegue analisá-lo sem erros:

          ```bash
          sudo systemctl daemon-reload
          sudo systemctl status otelcol-contrib.service
          # No output means no syntax errors
          ```
        </Step>

        <Step>
          ### Inicie o coletor do OpenTelemetry e verifique [#start-otelcol]

          1. Inicie o coletor e ative-o para ser executado automaticamente na inicialização:

          ```bash
          sudo systemctl enable --now otelcol-contrib.service
          sudo systemctl status otelcol-contrib.service --no-pager
          ```

          O comando `status` deve mostrar `Active: active (running)`. Se você vir `Active: failed`, verifique os logs com `journalctl -u otelcol-contrib.service -n 200 --no-pager`.

          2. Verifique se o coletor está saudável:

          ```bash
          curl -sS http://127.0.0.1:13133
          # Expected: {"status":"Server available", ...}
          ```

          3. Confirme se sua métrica do IBM MQ está fluindo para o New Relic. Aguarde 60 segundos após o início do coletor e, em seguida, verifique seus dados usando a consulta de verificação em [Encontre e consulte seus dados](/docs/opentelemetry/integrations/ibm-mq/view-data).
        </Step>

        <Step>
          ### Visualize seus dados no New Relic [#view-otelcol]

          Assim que seu coletor estiver em execução e as métricas estiverem fluindo, você verá seus gerenciadores de filas como entidades `IBMMQ_MANAGER` no New Relic, com suas filas como entidades `IBMMQ_QUEUE` filhas. Para obter detalhes sobre como encontrar seus dados, executar consultas e configurar dashboards e alertas, consulte [Encontrar e consultar seus dados](/docs/opentelemetry/integrations/ibm-mq/view-data).
        </Step>
      </Steps>

## Configuração avançada [#advanced]

Use essas configurações para personalizar sua configuração para diferentes cenários.

### Escalonamento e gerenciamento do gerenciador de filas

<CollapserGroup>
  <Collapser id="add" title="Adicione outro gerenciador de filas">
    Para adicionar outro gerenciador de filas (como `QM2` na porta `9160`), você precisará atualizar os arquivos de configuração e de ambiente:

    1. Confirme se o exportador mq-metric-samples para `QM2` está em execução e acessível:

       ```bash
       curl -sS http://localhost:9160/metrics | grep -E '^ibmmq_qmgr_status' | head -1
       ```

    2. Adicione `IBMMQ_QM2_ENDPOINT` ao arquivo env em `/etc/nrdot-collector/ibmmq-env`:

       ```ini
       IBMMQ_QM2_ENDPOINT=localhost:9160
       ```

    3. Abra `/etc/nrdot-collector/ibmmq-config.yaml` e adicione um quarto bloco de receptor dentro de `receivers:`:

       ```yaml
       prometheus/ibmmq-qm2:
         config:
           scrape_configs:
             - job_name: 'ibmmq-qm2'
               scrape_interval: ${env:IBMMQ_SCRAPE_INTERVAL:-60s}
               scrape_timeout: 15s
               static_configs:
                 - targets:
                     - "${env:IBMMQ_QM2_ENDPOINT:-localhost:9160}"
                   labels:
                     targetName: "${env:TARGET_NAME}"
                     clusterName: "${env:IBMMQ_CLUSTER_NAME:-}"
       ```

    4. Adicione o pipeline correspondente dentro de `service.pipelines:`:

       ```yaml
       metrics/ibmmq-qm2:
         receivers: [prometheus/ibmmq-qm2]
         processors:
           - filter/ibmmq-overhead
           - filter/ibmmq-queues
           - resourcedetection
           - transform/ibmmq-cleanup
           - memory_limiter/ibmmq
           - cumulativetodelta/ibmmq
           - batch/ibmmq
         exporters: [otlphttp/ibmmq]
       ```

    5. Reinicie o coletor:

       ```bash
       sudo systemctl restart nrdot-collector.service
       ```

    Para **remover** um gerenciador de filas, desative seu serviço de exportador, exclua seus blocos de receptor e pipeline de `ibmmq-config.yaml`, remova seu `IBMMQ_QMN_ENDPOINT` do arquivo env e reinicie o coletor. A entidade expira de acordo com a janela de relatórios do New Relic.
  </Collapser>
</CollapserGroup>

### Segurança e redes

<CollapserGroup>
  <Collapser id="tls-basic-auth" title="TLS e autenticação básica para a coleta do exportador">
    Se o seu exportador mq-metric-samples estiver configurado para exigir HTTPS ou autenticação básica, adicione os campos relevantes ao bloco `scrape_configs` de cada receptor. O exemplo a seguir mostra ambas as opções para `prometheus/ibmmq-qm1` — aplique o mesmo padrão a `qm2` e `qm3`:

    ```yaml
    prometheus/ibmmq-qm1:
      config:
        scrape_configs:
          - job_name: 'ibmmq-qm1'
            scrape_interval: ${env:IBMMQ_SCRAPE_INTERVAL:-60s}
            scrape_timeout: 15s
            scheme: https
            tls_config:
              ca_file: /etc/nrdot-collector/ca.pem
              insecure_skip_verify: false
            basic_auth:
              username: ${env:IBMMQ_EXPORTER_USER}
              password: ${env:IBMMQ_EXPORTER_PASSWORD}
            static_configs:
              - targets:
                  - "${env:IBMMQ_QM1_ENDPOINT:-localhost:9157}"
                labels:
                  targetName: "${env:TARGET_NAME}"
                  clusterName: "${env:IBMMQ_CLUSTER_NAME:-}"
    ```

    Adicione as variáveis correspondentes a `/etc/nrdot-collector/ibmmq-env`:

    ```ini
    IBMMQ_EXPORTER_USER=mqmonitor
    IBMMQ_EXPORTER_PASSWORD=<YOUR_EXPORTER_PASSWORD>
    ```
  </Collapser>

  <Collapser id="remote-exporter" title="Exportador remoto (coletor e gerenciador de filas em hosts separados)">
    Quando o exportador mq-metric-samples for executado em um host diferente do coletor, altere os valores de `IBMMQ_QMN_ENDPOINT` no arquivo env para apontar para o host remoto:

    ```ini
    IBMMQ_QM1_ENDPOINT=mq-host-1.example.com:9157
    IBMMQ_QM2_ENDPOINT=mq-host-1.example.com:9158
    IBMMQ_QM3_ENDPOINT=mq-host-2.example.com:9159
    ```

    Configure o firewall nas portas do exportador para que apenas o IP do coletor possa alcançá-las. No Ubuntu/UFW:

    ```bash
    sudo ufw allow from <collector-ip> to any port 9157 proto tcp
    sudo ufw allow from <collector-ip> to any port 9158 proto tcp
    sudo ufw allow from <collector-ip> to any port 9159 proto tcp
    ```

    No RHEL/firewalld, crie uma zona dedicada com escopo para o IP de origem do coletor e as três portas.

    Se o coletor estiver atrás de um proxy de saída, adicione `HTTPS_PROXY` ao arquivo env e inclua `NO_PROXY=localhost,127.0.0.1` para evitar que o proxy intercepte os requests de scrape locais:

    ```ini
    HTTPS_PROXY=http://proxy.example.com:3128
    NO_PROXY=localhost,127.0.0.1
    ```
  </Collapser>
</CollapserGroup>
