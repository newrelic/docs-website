Esta página fornece uma referência completa para todas as métricas do HAProxy coletadas via OpenTelemetry, incluindo os caminhos do haproxyreceiver e do receptor Prometheus.

## Visão geral [#overview]

A integração do HAProxy OpenTelemetry coleta métricas do endpoint de estatísticas CSV do HAProxy por meio do [haproxyreceiver](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/receiver/haproxyreceiver). As métricas são organizadas em dois níveis:

* **Métricas padrão (17)**: coletadas automaticamente com a configuração padrão — suficientes para a maioria dos casos de uso de monitoramento
* **Métricas opcionais (16)**: desativadas por padrão, ativadas individualmente na seção `metrics` do receiver — úteis para depuração detalhada e planejamento de capacidade

Ao usar a instalação guiada do NRDOT, você pode escolher entre duas configurações:

* **Otimizado** (menor ingestão): 17 métricas padrão, intervalo de coleta de 60s, processadores de otimização aplicados
* **Abrangente** (detalhes completos): 27 métricas (17 padrão + 10 opcionais), intervalo de coleta de 15s

<Callout variant="tip">
  Para habilitar métricas opcionais, adicione-as à seção `metrics` da configuração do seu receiver haproxy com `enabled: true`. Consulte [Monitore o HAProxy auto-hospedado com OpenTelemetry](/docs/opentelemetry/integrations/haproxy/self-hosted/) para detalhes de configuração.
</Callout>

<Callout variant="important">
  **Limitações conhecidas do caminho do Prometheus:**

  * **Sem FACET por código de status**: o haproxyreceiver analisa `hrsp_*` campos CSV em um atributo `status_code` em `haproxy.requests.total`. O caminho do Prometheus não suporta isso — a divisão do código de resposta HTTP está disponível apenas como métricas separadas (`haproxy_frontend_http_responses_total{code="2xx"}`) não incluídas na configuração de transformação.
  * **As métricas de taxa são máximos vitalícios**: `haproxy.connections.rate` mapeia para `haproxy_frontend_connections_rate_max` e `haproxy.requests.rate` mapeia para `haproxy_frontend_http_requests_rate_max`. Esses são valores máximos vitalícios, não taxas instantâneas. Para taxas atuais reais, use a função NRQL `rate()` em métricas de contador cumulativo.
  * **-136 métricas do Prometheus não têm equivalente no OTel**: o endpoint do Prometheus do HAProxy expõe -204 métricas. Apenas 36 são mapeadas para as 17 métricas padrão do OTel (nos níveis de frontend/backend/servidor). O restante (detalhes internos do processo, detalhes de verificação de integridade, estatísticas do resolvedor DNS) não tem equivalente no haproxyreceiver.
</Callout>

## Consultas NRQL de exemplo [#sample-queries]

Use estas consultas para explorar suas métricas do HAProxy na New Relic:

```sql
-- Current sessions across all frontends
FROM Metric SELECT latest(haproxy.sessions.count)
FACET haproxy.proxy_name
WHERE haproxy.service_name = 'FRONTEND'

-- Request rate per frontend over time
FROM Metric SELECT average(haproxy.requests.rate)
FACET haproxy.proxy_name
WHERE haproxy.service_name = 'FRONTEND'
TIMESERIES

-- Backend error rate (responses.errors as rate)
FROM Metric SELECT rate(sum(haproxy.responses.errors), 1 second)
FACET haproxy.proxy_name
WHERE haproxy.service_name = 'BACKEND'
TIMESERIES

-- Bytes throughput per backend
FROM Metric SELECT rate(sum(haproxy.bytes.input), 1 minute) AS 'Bytes In/min',
  rate(sum(haproxy.bytes.output), 1 minute) AS 'Bytes Out/min'
FACET haproxy.proxy_name
WHERE haproxy.service_name = 'BACKEND'

-- Average session time per backend (latency indicator)
FROM Metric SELECT average(haproxy.sessions.average)
FACET haproxy.proxy_name
WHERE haproxy.service_name = 'BACKEND'

-- Per-server session count (individual backend servers)
FROM Metric SELECT latest(haproxy.sessions.count)
FACET haproxy.proxy_name, haproxy.service_name
WHERE haproxy.service_name NOT IN ('FRONTEND', 'BACKEND')

-- Queued requests (indicates backend saturation)
FROM Metric SELECT max(haproxy.requests.queued)
FACET haproxy.proxy_name
WHERE haproxy.service_name = 'BACKEND'
TIMESERIES
```
