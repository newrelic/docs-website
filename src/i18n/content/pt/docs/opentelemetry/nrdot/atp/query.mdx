O ATP coleta métricas de processo por meio do OpenTelemetry `hostmetricsreceiver` e as enriquece com metadados de filtragem inteligente. Você pode consultar estes dados para:

* Monitore os estados dos processos e identifique `stuck` ou processos problemáticos
* Acompanhe como os limites dinâmicos do ATP se adaptam aos seus padrões de carga de trabalho
* Meça a eficiência da filtragem para entender a eficácia da redução de dados
* Detectar anomalias e comportamento incomum de processos
* Configurar alertas para o monitoramento de processos críticos

Todas as métricas de processo estão disponíveis no namespace `process.*` no New Relic, com metadados específicos do ATP armazenados no campo `process.atp`.

## Monitoramento do estado do processo [#process-state]

<CollapserGroup>
  <Collapser id="process-count-by-name" title="Distribuição de processos por estado">
    Esta consulta mostra um detalhamento de todos os processos em execução no seu host, categorizados por seu estado de execução. É essencial para identificar processos `stuck` ou esgotamento de recursos.

    ```sql
    FROM Metric
    SELECT uniqueCount(process.pid) as 'Process Count'
    FACET process.state, CASES(
      WHERE process.state = 'R' AS 'Running',
      WHERE process.state = 'S' AS 'Sleeping (Interruptible)',
      WHERE process.state = 'D' AS 'Uninterruptible Sleep (Disk I/O)',
      WHERE process.state = 'Z' AS 'Zombie',
      WHERE process.state = 'T' AS 'Stopped',
      WHERE process.state = 'I' AS 'Idle Kernel Thread',
      WHERE process.state = 'X' AS 'Dead',
      WHERE process.state = 'W' AS 'Paging'
    )
    ```
  </Collapser>
</CollapserGroup>

## Monitoramento de inteligência de ATP [#atp-intelligence]

<CollapserGroup>
  <Collapser id="dynamic-thresholds" title="Rastreamento de limiares dinâmicos">
    Esta consulta mostra os valores atuais de limite dinâmico comparados às métricas reais do processo, ajudando você a entender o quão próximos os processos estão de acionar ações de ATP.

    ```sql
    FROM Metric
    SELECT
      latest(numeric(jparse(`process.atp`)[threshold_details][`process.cpu.utilization`][threshold])) * 100 AS 'Process CPU Utilization',
      latest(numeric(jparse(`process.atp`)[threshold_details][`process.memory.usage`][threshold])) AS 'Memory Usage'
    WHERE host.name = 'YOUR_HOST_NAME'
    TIMESERIES MAX
    ```

    **Como usar esta consulta:**

    * Substitua `YOUR_HOST_NAME` pelo seu nome do host real
    * Você pode modificar os nomes das métricas na função `jparse` para rastrear limites de outras métricas
    * Observe padrões onde os valores reais se aproximam consistentemente dos valores de limite
  </Collapser>

  <Collapser id="filtering-efficiency" title="Eficiência de filtragem ATP">
    Esta consulta mede a eficácia com que o ATP está reduzindo o volume de telemetria enquanto retém dados críticos.

    ```sql
    FROM Metric
    SELECT latest(numeric(jparse(`process.atp`)['filtering_summary']['efficiency_ratio'])) * 100 as 'Filtering Efficiency'
    WHERE metricName like 'process.%'
    AND `process.atp` IS NOT NULL
    ```
  </Collapser>
</CollapserGroup>

## Consultas de detecção de anomalias [#anomaly-detection]

<CollapserGroup>
  <Collapser id="anomaly-overview" title="Contagem de anomalias ao longo do tempo">
    Esta consulta visualiza a frequência com que o ATP detecta anomalias em diferentes executáveis, ajudando você a identificar quais processos são mais propensos a comportamentos incomuns.

    ```sql
    FROM Metric
    SELECT latest(numeric(jparse(`process.atp`, 'filtering_summary.stage_hits.anomaly_detection'))) AS 'Anomaly Count'
    WHERE metricName like 'process.%'
    FACET process.executable.name
    TIMESERIES MAX
    ```
  </Collapser>

  <Collapser id="memory-usuage" title="Anomalias de uso de memória">
    Esta consulta destaca processos específicos que apresentaram anomalias de uso de memória detectadas pelo ATP.

    ```sql
    FROM Metric
    SELECT max(process.memory.usage) as 'Memory Usage'
    WHERE metricName = 'process.memory.usage'
    AND numeric(jparse(`process.atp`)['filtering_summary']['stage_hits']['anomaly_detection']) > 0
    AND process.atp IS NOT NULL
    FACET process.pid, process.executable.name, host.name
    TIMESERIES MAX
    ```
  </Collapser>

  <Collapser id="cpu-usage" title="Anomalias de uso de CPU">
    Esta consulta mostra processos que apresentaram anomalias de utilização de CPU.

    ```sql
    FROM Metric
    SELECT max(process.cpu.utilization) * 100 as 'CPU Utilization'
    WHERE metricName like 'process.cpu.utilization'
    AND numeric(jparse(`process.atp`)['filtering_summary']['stage_hits']['anomaly_detection']) > 0
    AND process.atp IS NOT NULL
    FACET process.pid, process.executable.name, host.name
    TIMESERIES MAX
    ```

    **Nota**: Você pode modificar essas consultas para monitorar anomalias em qualquer outra métrica de processo alterando o filtro `metricName`.
  </Collapser>
</CollapserGroup>

## Monitoramento de desempenho [#performance-monitoring]

<CollapserGroup>
  <Collapser id="context-switches" title="Trocas de contexto por minuto">
    Esta consulta rastreia a taxa de troca de contexto por processo. Altas taxas de troca de contexto frequentemente indicam contenção de CPU ou multi-threading ineficiente.

    ```sql
    SELECT rate(sum(process.context_switches), 1 minute) AS 'Context Switches/min'
    FROM Metric
    WHERE metricName = 'process.context_switches'
    FACET process.executable.name, process.pid
    TIMESERIES MAX
    ```
  </Collapser>
</CollapserGroup>

## Configurando alertas

<CollapserGroup>
  <Collapser id="data-alert" title="Alerta de lacuna de dados">
    Use esta consulta para criar um alerta que é acionado quando a telemetria do processo para de fluir completamente, indicando uma possível falha do agente ou problemas de rede. Para mais informações sobre a configuração de alertas, consulte [Criar condições de alerta](/docs/alerts/create-alert/create-alert-condition/alert-conditions/).

    ```sql
    FROM Metric
    SELECT count(*)
    WHERE metricName LIKE 'process.%'
    ```
  </Collapser>
</CollapserGroup>
