Il existe trois façons de créer une condition d&apos;alerte dans New Relic : avec un graphique, une requête NRQL ou avec des métriques dorées. Bien que le début du processus de création soit différent selon l’option choisie, toutes les étapes suivantes sont les mêmes. Cela signifie qu&apos;une fois que vous avez créé une condition dans un sens, l&apos;expérience que vous gagnez en le faisant sera transférée à d&apos;autres que vous créez en utilisant l&apos;une ou l&apos;autre des autres options.

Pour ce tutoriel, nous avons inclus des étapes pour les trois options susmentionnées, afin que vous puissiez commencer avec votre première condition comme vous le souhaitez. Sélectionnez simplement le chemin que vous souhaitez suivre dans la première étape, et à la fin de la procédure, vous aurez réussi à créer votre première condition d&apos;alerte à l&apos;aide de New Relic !

<Callout variant="tip">
  Si vous avez besoin de plus d&apos;informations sur l&apos;une des options ou des conditions abordées dans les étapes ci-dessous, consultez nos [documents sur les alertes](/docs/alerts-applied-intelligence/overview/#concepts-terms).
</Callout>

          Bien que l&apos;utilisation de graphiques soit un excellent moyen de créer une condition d&apos;alerte à partir de requêtes prédéfinies, vous devez parfois les créer à partir de zéro. Le processus de création d&apos;une nouvelle condition d&apos;alerte à partir d&apos;une requête NRQL est presque exactement le même que sa création à partir d&apos;un graphique : la principale différence est que vous écrivez la requête NRQL manuellement au lieu d&apos;en utiliser une pré-générée.

          <Callout variant="tip">
            NRQL signifie New Relic&apos;s Requête Language, une méthode de requête standardisée qui alimente tous nos graphiques et que vous pouvez utiliser pour analyser et interroger vos données. Si c&apos;est la première fois que vous travaillez avec NRQL, vous souhaiterez peut-être consulter notre [documentation NRQL](/docs/query-your-data/nrql-new-relic-query-language/get-started/introduction-nrql-new-relics-query-language/) pour en savoir plus sur NRQL, notamment [son fonctionnement](/docs/query-your-data/nrql-new-relic-query-language/get-started/introduction-nrql-how-nrql-works/) et un [didacticiel d&apos;introduction](/docs/query-your-data/nrql-new-relic-query-language/get-started/introduction-nrql-tutorial/).
          </Callout>

          2. Sélectionnez <DNT>**New alert condition**</DNT>.

             ```sql
             SELECT average(duration) from Transaction facet name
             ```

    ## Ajuster le comportement du signal

    <img title="Tuning your signal" alt="A screenshot displaying how to set options for a signal in an alert condition" src="/images/alerts_screenshot-crop_signal-tune.webp" />

    1. <DNT>**Set the data aggregation options**</DNT>:

       * <DNT>**Window duration**</DNT>:Utilisez le paramètre <DNT>**Window duration**</DNT> pour déterminer la fréquence à laquelle nous agrégeons vos données, par exemple des regroupements toutes les 5 minutes ou toutes les heures. Le paramètre de cette fonctionnalité dépend du type de données que vous monitoring. Si vous n&apos;êtes pas sûr de ce que cela devrait être, vous pouvez le laisser au paramètre par défaut. Vous pouvez en apprendre davantage dans notre [document Créer votre première alerte](/docs/alerts-applied-intelligence/new-relic-alerts/get-started/your-first-nrql-condition/#window-duration).
       * <DNT>**Sliding window aggregation**</DNT>: Choisissez si vous souhaitez ou non utiliser [l&apos;agrégation de fenêtres glissantes](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/#sliding-window-aggregation). Cette fonctionnalité crée des graphiques plus fluides lors du traitement de données erratiques en créant des fenêtres d&apos;agrégation qui se chevauchent. Cette fonctionnalité est désactivée par défaut et nous vous recommandons de la garder désactivée pour votre première alerte.
       * <DNT>**Streaming method**</DNT>:Les options les plus couramment utilisées sont <DNT>**Event flow**</DNT> et <DNT>**Event timer**</DNT>. <DNT>**Event flow**</DNT> est le mieux adapté aux rapports de données fréquents et cohérents, tandis que <DNT>**Event timer**</DNT> est le mieux adapté aux données qui sont rapportées de manière incohérente, peu fréquente ou ingérées par lots. Si vous souhaitez en savoir plus sur les paramètres individuels pour voir quelle méthode doit être utilisée dans chaque scénario, consultez [notre documentation](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/understand-technical-concepts/streaming-alerts-key-terms-concepts/#aggregation-methods)

    2. <DNT>**Set the timing option**</DNT>:Cela détermine combien de temps nous attendons l&apos;événement dans chaque fenêtre avant de les évaluer. Un délai plus long signifiera moins de fausses alertes, mais cela signifiera également un temps d’arrêt potentiel plus long avant qu’un incident ne soit ouvert. Cette capacité est également affichée différemment selon l&apos;option sélectionnée : elle est appelée <DNT>**Delay**</DNT> dans les options <DNT>**Event flow**</DNT> et <DNT>**Cadence**</DNT> et <DNT>**Timer**</DNT> dans <DNT>**Event timer**</DNT>. Pour votre première alerte, nous vous recommandons de conserver le paramètre d&apos;heure par défaut.

    3. <DNT>**Select the gap filling strategy**</DNT>: Gap-filling est une fonctionnalité réactive qui remplit les périodes vides dans le reporting des données avec une valeur Synthétique définie par l&apos;utilisateur. Si vous pensez que la remontée des données sera sporadique, vous pouvez éviter les fausses alertes en remplissant les fenêtres d&apos;agrégation vides avec ces données. Pour votre première alerte, sélectionnez <DNT>**None**</DNT> sous <DNT>**Gap-filling strategy**</DNT>.

    4. <DNT>**Select the evaluation delay**</DNT>:Si elle est utilisée, cette option active un délai pendant lequel New Relic doit attendre avant d&apos;évaluer un nouveau signal par rapport au seuil que vous avez défini. Cela est utile pour éviter les faux incidents positifs lorsqu&apos;un flux de données commence à générer un rapport. Pour votre première alerte, vous pouvez la laisser sur la bascule désactivée, sauf si vous savez que cette option s&apos;applique à vos données.

    5. Sélectionnez <DNT>**Next**</DNT> une fois que vous avez terminé pour enregistrer votre progression et passer à l&apos;étape suivante.
  </Step>

  <Step>
    ## Définir le seuil statique

    <img title="Set static thresholds" alt="A screenshot displaying how to set static thresholds for an alert condition" src="/images/alerts_screenshot-crop_set-thresholds.webp" />

    1. Commencez par sélectionner si le seuil aura un statut de <DNT>**Critical**</DNT> ou <DNT>**Warning**</DNT> dans la section <DNT>**Security level**</DNT> .

    2. Définissez la limite qui, une fois atteinte, ouvrira un incident. Ces valeurs changeront en fonction de ce que vous souhaitez que la condition fasse. Par exemple, si vous souhaitez créer une condition pour un message d&apos;erreur, vous pouvez définir le seuil sur <DNT>**equal to 1 at least once in 5 minutes**</DNT>, mais si vous souhaitez créer une condition pour des périodes plus longues de problèmes de latence, vous pouvez définir le seuil sur <DNT>**above or equal to 50 for at least 15 minutes**</DNT>.

    3. Si vous souhaitez ajouter un autre seuil, sélectionnez <DNT>**Add threshold**</DNT> ou <DNT>**Add lost signal threshold**</DNT>, selon vos besoins. Les seuils de signal perdu sont importants pour monitoring des entités qui peuvent cesser de signaler ou des requêtes qui peuvent [renvoyer des valeurs nulles](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/#query-order).

    4. Sélectionnez <DNT>**Next**</DNT> une fois que vous avez terminé pour enregistrer votre progression et passer à l&apos;étape suivante.
  </Step>

  <Step>
    ## Ajouter des détails sur condition d&apos;alerte

    <img title="Alert condition details" alt="A screenshot displaying how to create the details for an alert" src="/images/alerts_screenshot-crop_condition-details.webp" />

    1. Ajoutez un nom pour votre alerte dans le champ <DNT>**Name**</DNT> . Cela vous aidera à identifier cette condition, ainsi qu&apos;à la retrouver à l&apos;avenir au cas où vous auriez besoin de la modifier ou de la supprimer.

    2. Choisissez la politique à laquelle cette condition doit se connecter. Si vous avez un <DNT>**Existing policy**</DNT>, sélectionnez-le dans la liste déroulante. Sinon, vous devrez [créer une politique](/docs/alerts-applied-intelligence/new-relic-alerts/alert-policies/create-edit-or-find-alert-policy/#alert-policy-name) en sélectionnant <DNT>**New policy**</DNT> et en la créant à partir d’ici.

    3. Définissez l’un des paramètres supplémentaires pour l’alerte. Cela inclut la définition du délai avant lequel les incidents ouverts sont automatiquement fermés et la saisie [de descriptions personnalisées](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/advanced-techniques/alert-custom-incident-descriptions) pour l&apos;incident.

    4. Sélectionnez <DNT>**Save condition**</DNT> pour terminer.
  </Step>
</Steps>

## Créer une alerte à l&apos;aide du générateur d&apos;alertes classique

<Callout variant="tip">
  Nous vous recommandons d&apos;utiliser les méthodes ci-dessus pour créer une alerte NRQL. Si vous souhaitez créer une alerte classique, vous pouvez utiliser le formulaire de création d&apos;alerte d&apos;origine.
</Callout>

<img title="Create an alert with builder" alt="A screenshot showing the alert builder" src="/images/alerts_screenshot-crop_alert-builder.webp" />

1. À partir de <DNT>**one.newrelic.com**</DNT>, sélectionnez [<DNT>**Alerts**</DNT>](https://one.newrelic.com/alerts-ai), puis [<DNT>**Alert policies**</DNT>](https://one.newrelic.com/alerts-ai/condition-builder/policy-list).

2. Cliquez sur la politique pour laquelle vous souhaitez créer la condition d’alerte classique ou créez une nouvelle politique.

3. Sélectionnez <DNT>**New alert condition**</DNT>, puis sélectionnez <DNT>**Build a classic alert**</DNT>.

4. Construisez votre condition à partir de l’une des catégories répertoriées. Cela détermine ce dont la condition d&apos;alerte vous informe.

5. Sélectionnez un type de condition pour affiner les options de l’étape suivante. Vous devrez peut-être également sélectionner une portée.

6. Sélectionnez l&apos;entité ou l&apos;entité que vous souhaitez monitorer. Utilisez la barre de recherche pour affiner la liste des entités. Si vous avez sélectionné une condition de type métrique d&apos;application APM, vous pouvez choisir de cibler dynamiquement les applications qui partagent certaines balises.

7. Définissez votre seuil d&apos;alerte et tout autre paramètre de condition. Nommez votre condition, puis enregistrez.

Vous avez maintenant créé une condition d’alerte ! Nous proposons différentes options configuration pour les alertes, afin que vous puissiez affiner vos alertes en fonction de vos besoins monitoring . Vous pouvez également créer des conditions à partir d’une requête NRQL , que nous aborderons ensuite dans cette série de didacticiels.

## Tirez le meilleur parti des alertes [#more-alerting]

Nous disposons de fonctionnalités d&apos;alerte avancées qui peuvent vous aider à rendre les alertes plus efficaces et plus faciles à gérer pour votre équipe :

* Si vous souhaitez mieux contrôler où et quand vous recevez les alertes et vous assurer que les bonnes personnes sont informées des problèmes, découvrez comment utiliser [le workflow](/docs/alerts-applied-intelligence/applied-intelligence/incident-workflows/incident-workflows/).
* Si vous souhaitez activer notification quasi instantanée des critiques d&apos;alerte, vous aidant ainsi à réduire votre Délai moyen de résolution (MTTR) (MTTR), découvrez comment utiliser notre [détection d&apos;anomalies](/docs/alerts-applied-intelligence/applied-intelligence/anomaly-detection/anomaly-detection-applied-intelligence/).
* Si vous souhaitez utiliser des API pour gérer les alertes New Relic, vous pouvez le faire avec [l&apos;API REST ou GraphQL](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/alerts-api/intro-alerts-api/).
