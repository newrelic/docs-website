## Le quoi et le pourquoi des transactions [#why]

L&apos;activité des clients dans votre application Web crée des itinéraires à travers votre code qui commencent lorsqu&apos;une application reçoit une demande Web et se terminent lorsqu&apos;une réponse est renvoyée. Ces itinéraires sont des transactions, ou une unité de travail composée d’appels de fonctions et d’appels de méthodes.

Pensez à un moment commercial critique, comme le paiement d’un achat par un client. Si un client signale que son paiement n&apos;a pas été traité ou a été traité après un retard important, vous ne pouvez que deviner la cause de cet échec. Une API a-t-elle provoqué l&apos;échec ou s&apos;agissait-il d&apos;un débogage récent déclenché plus tôt dans la journée ?

## Observer les transactions pour améliorer les performances [#observe]

Notre interface utilisateur de transactions vous aide à diagnostiquer la source d&apos;un problème afin que vous puissiez améliorer les performances de transactions spécifiques. Les étapes suivantes vous guident pour identifier les transactions lentes et améliorer les performances de vos applications :

<Steps>
  <Step>
    Accédez à l&apos;interface utilisateur des transactions : Accédez à <DNT>**[one.newrelic.com](https://one.newrelic.com/nr1-core?filters=\(domain%3D'APM'ANDtype%3D'APPLICATION'\)) &gt; (select an app) &gt; Monitor &gt; Transactions**</DNT>
  </Step>

  <Step>
    ### Trouvez vos transactions lentes : [#slow-transactions]

    <SideBySide>
      <Side>
        1. Triez vos transactions par ordre de temps le plus long en haut de la page.
        2. Jetez un œil au graphique <DNT>**Top 20 transactions**</DNT> . Cette liste représente la meilleure opportunité de réduire la charge de votre application. En réduisant le temps consacré à ces transactions fréquemment appelées, vous réduirez l&apos;empreinte de votre application, car il s&apos;agit requests à volume élevé. Prenez note des quelques transactions les plus importantes dans ce graphique.
        3. Triez vos transactions par réponse moyenne la plus lente en haut de la page.
        4. Jetez à nouveau un œil au graphique <DNT>**Top 20 transactions**</DNT> . Ce graphique montre désormais le temps moyen par requête à partir du volume total d&apos;une requête particulière. Prenez note des quelques transactions les plus importantes dans ce graphique.
      </Side>

      <Side>
        <img title="Most consuming transactions chart" alt="Most consuming transactions chart" src="/images/journey_screenshot-consuming.webp" />

        <figcaption>
          Ces graphiques montrent vos transactions les plus chronophages.
        </figcaption>

        <img title="Slowest transactions chart" alt="Slowest transactions chart" src="/images/journey_screenshot-slow.webp" />

        <figcaption>
          Ces graphiques montrent vos transactions les plus lentes.
        </figcaption>
      </Side>
    </SideBySide>
  </Step>

  <Step>
    ### Améliorez vos performances [#improve-performance]

    Vous avez identifié les transactions susceptibles de provoquer des retards dans votre code, mais connaître un problème n&apos;est pas la même chose que trouver une solution pour vos clients. Grâce aux transactions identifiées ci-dessus, vous pouvez trouver une répartition de l&apos;endroit dans votre code où cet itinéraire particulier passe le plus de temps.

    <SideBySide>
      <Side>
        Pour chacune de vos transactions, cliquez sur leur nom dans le top 20 du graphique et procédez comme suit :

        1. Consultez le tableau de répartition des segments. Cela montre où votre demande passe le plus clair de son temps. Si, en moyenne, un itinéraire bloque dans une partie de votre code, concentrez-vous sur l&apos;optimisation de ce segment.
        2. Vérifiez la table de trace de transaction et récupérez votre trace la plus lente. Bien que vos traces les plus lentes soient des valeurs hors norme, elles constituent de bons indicateurs des méthodes, de la base de données ou des appels HTTP qui ont le plus d&apos;impact sur les performances de votre application.
      </Side>

      <Side>
        <img title="The transaction segment view" alt="The transactions segment view" src="/images/journey_screenshot-segment.webp" />
      </Side>
    </SideBySide>
  </Step>

  <Step>
    Vous devriez maintenant avoir une liste de méthodes lentes, requests ou d’autres parties de votre code qui ne fonctionnent pas bien. Vient maintenant la partie amusante : le codage proprement dit. Testez un correctif dans votre code, puis passez à la section suivante pour vérifier votre travail.
  </Step>
</Steps>

## Vérifiez votre travail [#check]

Vous avez identifié le problème et trouvé une solution potentielle. Envoyez votre correctif au développement, puis exécutez un test de charge typique pour avoir une idée de la manière dont votre application fonctionnera en production.

Pendant que vous monitorez votre temps de réponse, gardez un œil attentif sur les transactions de votre application.

* Vos transactions sont-elles normales ? C&apos;est fait !
* Est-ce qu&apos;ils se sont améliorés ? Utilisez ce que vous avez appris pour comprendre pourquoi ils se sont améliorés au-delà de la normale.
* Constatez-vous toujours des temps de réponse lents ? Il y a peut-être un [problème de base de données](/docs/journey-app-slow/slow-database-queries) ou peut-être que vos [services externes]() fonctionnent lentement :
