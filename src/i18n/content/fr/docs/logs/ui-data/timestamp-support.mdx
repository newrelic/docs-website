Un événement de log laisse derrière lui un attribut d&apos;horodatage, qui vous indique le moment exact où un certain événement s&apos;est produit. Les horaires sont essentiels pour commander un événement car ils vous permettent de rechercher dans le log des périodes spécifiques, vous permettant de découvrir quoi et quand un événement s&apos;est produit.

Les horodatages ont de nombreux formats sans norme unique. Ils apparaissent au début de l&apos;événement de log dans la plupart des cas, mais peuvent parfois apparaître beaucoup plus tard dans le log, voire pas du tout.

* Si un log est envoyé avec un attribut `timestamp`, ou si un attribut `timestamp` est analysé à partir de celui-ci, New Relic utilise cet horodatage pour le log.
* Si un log JSON est envoyé avec un champ `timestamp`, New Relic utilise cet horaire lors de l&apos;extraction des champs JSON en tant qu&apos;attribut, et il a la priorité sur tout attribut `timestamp` existant.
* Si `timestamp` n&apos;est pas envoyé, New Relic utilise l&apos;heure d&apos;ingestion du log comme horodatage.

La plupart des redirecteurs de logtransmettent les informations d&apos;horaire à partir de la source log , vous n&apos;aurez donc peut-être pas besoin de spécifier l&apos;horaire vous-même.

<Callout variant="important">
  Certains redirecteurs de logutilisent des champs d&apos;horaire internes, tels que`@timestamp` et `@realtime_timestamp`. New Relic reconnaît ces champs, mais ils ne doivent pas être utilisés par les clients. Pour envoyer l&apos;horaire log , veuillez utiliser l&apos;attribut `timestamp`.
</Callout>

## Format d&apos;horodatage du log pris en charge [#format]

Il existe deux options permettant de remplacer manuellement le comportement par défaut afin d&apos;attribuer la valeur `timestamp` contenue dans le log.

1. Si nous recevons un [logau format JSON](#timestamps_in_json_logs) contenant un attribut `timestamp` dans l’un de nos formats d’heure pris en charge, l’horaire d’ingestion sera remplacé. Au lieu de cela, la valeur d’horodatage de l’attribut JSON sera utilisée.
2. Vous pouvez utiliser [des règles d’analyse](#timestamp_parsing_rules) pour remplacer la valeur d’ingestion `timestamp` . Les règles d&apos;analyse vous permettent d&apos;extraire des données pour formater votre horodatage avec l&apos;un de nos formats pris en charge. Vous pouvez également utiliser le type Grok `datetime` et un `pattern` pour définir le format d&apos;horodatage personnalisé dans votre log.

Nous prenons en charge deux formats d&apos;horodatage :

<CollapserGroup>
  <Collapser id="unix" title="Format d'horodatage d'époque Unix">
    Un horodatage Unix <DNT>**epoch**</DNT> est basé sur le nombre de secondes (ou de millisecondes) qui se sont écoulées depuis 00:00:00 UTC le 1er janvier 1970. Deux exemples de formats pour `Fri, 03 Nov 2023 16:06:32 GMT` seraient :

    * Secondes de l&apos;époque à 10 chiffres : `1699027592`
    * Millisecondes de temps d&apos;époque à 13 chiffres : `1699027592000`
  </Collapser>

  <Collapser id="ISO8601" title="Format d'horodatage ISO8601">
    Le format ISO8601 suit le modèle `yyyy-MM-ddTHH:mm:ss.SSSTZD` (par exemple 07/11/2023 14:20:35+04:00). Par exemple:

    * `yyyy-MM-ddTHH:mm:ss`: 2023-11-03T15:28:05
    * `yyyy-MM-ddTHH:mm:ss.SSS`: 2023-11-03T15:28:05.250
    * `yyyy-MM-ddTHH:mm:ssZ`: 03/11/2023 à 15:28:05
    * `yyyy-MM-ddTHH:mm:ss+HH:mm`: 2023-11-03T15:28:05+07:00

## Horodatage pris en charge dans le log JSON [#timestamps-json]

Si nous recevons un log au format JSON avec un attribut `timestamp`, nous remplaçons la valeur d&apos;ingestion par défaut `timestamp` par la valeur JSON d&apos;origine. S&apos;il n&apos;y a pas de champ d&apos;horaire ou si le champ est mal formaté, le log sera stocké avec l&apos;horaire d&apos;ingestion. Gardez à l&apos;esprit que les frais dont l&apos;horodatage est supérieur à 48 heures sont abandonnés.

Les horodatages sont convertis en millisecondes d&apos;époque Unix et stockés en interne sous forme de type de données long, avec le mot-clé réservé `timestamp`. Selon le format du corps JSON, le champ d&apos;horodatage peut être alloué à quatre emplacements différents :

* À l&apos;intérieur de l&apos; [ensemble simplifié d&apos;attributs](/docs/logs/log-api/introduction-log-api/#simple-json) du message du corps JSON lors de l&apos;envoi d&apos;un seul objet JSON.
* À l&apos;intérieur de l&apos;objet [commun](/docs/logs/log-api/introduction-log-api/#json-common) dans l&apos;ensemble détaillé d&apos;attributs du message du corps JSON lors de l&apos;envoi d&apos;un ou plusieurs objets JSON. L&apos;horodatage s&apos;applique à tous les messages du log de ce JSON.
* À l&apos;intérieur de chaque message, log l&apos;objet [log](/docs/logs/log-api/introduction-log-api/#json-logs) dans l&apos;ensemble détaillé des attributs du corps du message JSON lors de l&apos;envoi d&apos;un ou plusieurs objets JSON. L&apos;horodatage s&apos;applique uniquement à ce message de log.
* À l’intérieur du champ JSON « message » lorsqu’il s’agit d’un message JSON valide. Nos capacités <InlinePopover type="logs" />analyseront n’importe quel attribut de message au format JSON. L&apos;attribut JSON résultant dans le message analysé sera ajouté au log.

Voici quelques exemples de logs JSON avec un attribut `timestamp` valide qui remplace l&apos;ingestion `timestamp`:

<CollapserGroup>
  <Collapser id="json-epoch" title="Log JSON au format d'époque (pas de précision en ms)">
    Si nous recevons un log JSON comme celui-ci avec `timestamp` au format <DNT>**epoch**</DNT> (sans précision en ms) :

    ```json
    { 
      "timestamp":1701445572,
      "message":"This is my sample JSON log message with timestamp attribute in epoch format"
    }
    ```

    Nous remplacerons l&apos;ingestion `timestamp` par l&apos;époque `timestamp` contenue dans le log, et le résultat ressemblera à ceci :

    ```json
    "timestamp" : 1701445572000,
    "message":"This is my sample JSON log message with timestamp attribute in epoch format"
    ```
  </Collapser>

  <Collapser id="json-ISO8601" title="Log JSON au format ISO8601">
    Si nous recevons un log JSON comme celui-ci avec un horodatage au format <DNT>**ISO8601**</DNT> comme ceci :

    ```json
    {
      "timestamp":"2023-12-01T15:46:26.607Z",
      "message":"This is my sample JSON log message with timestamp attribute in ISO8601 format"
    }
    ```

    Nous remplacerons l&apos;ingestion `timestamp` par l&apos;ISO8601 `timestamp` (converti en époque) contenu dans le log, et le résultat ressemblera à ceci :

    ```json
    "timestamp": 1701445586607,
    "message":"This is my sample JSON log message with timestamp attribute in ISO8601 format"
    ```
  </Collapser>
</CollapserGroup>

## Définir l&apos;horaire log avec des règles d&apos;analyse [#timestamp-parsing-rules]

Vous pouvez utiliser des règles d&apos;analyse lors de l&apos;ingestion pour remplacer l&apos;ingestion `timestamp` par la valeur contenue dans le log. analyser les règles extraire nos formats d&apos;heure pris en charge avec des expressions Grok simples, ou en utilisant le type Grok `datetime` et `pattern` pour définir le format d&apos;horodatage personnalisé dans votre log.

Le nom de l&apos;attribut extrait par l&apos;expression doit être `timestamp` et son type de données doit être chaîne ou long. Si le type de données n&apos;est pas fourni, l&apos;horodatage par défaut est une chaîne. Pour un horodatage d&apos;époque avec une précision de l&apos;ordre de la microseconde ou de la nanoseconde, vous devez inclure le type de données `long` même si la précision de l&apos;horodatage est réduite à quelques millisecondes.

Si l&apos;horodatage de votre log est un horodatage au format Unix ou ISO8601, vous pouvez utiliser les modèles Grok par défaut pour les faire correspondre. Par exemple:

* Pour extraire un horodatage d&apos;époque Unix, vous pouvez utiliser n&apos;importe quelle expression correspondant à la valeur, comme `NUMBER`, `NOTSPACE`, `DATA` ou `GREEDYDATA`.
* Pour extraire un horodatage au format ISO8601, vous devez utiliser l&apos;expression Grok `TIMESTAMP_ISO8601` .

Vous pouvez utiliser le type Grok `datetime` pour analyser l&apos;horodatage dans des formats non pris en charge. Pour ce faire, vous devez inclure le type Grok `datetime` avec le `pattern` correspondant à votre format d&apos;horodatage du log. Le `pattern` doit utiliser les modèles de date et d&apos;heure simples Java trouvés [ici](https://docs.oracle.com/javase/7/docs/api/java/text/SimpleDateFormat.html). Vous trouverez ci-dessous des exemples qui montrent comment les expressions Grok transforment les formats d’horodatage non pris en charge en formats pris en charge :

Vous trouverez ci-dessous quelques exemples sur la façon d&apos;analyser le `timestamp` de votre log à l&apos;aide de règles d&apos;analyse d&apos;ingestion :

<CollapserGroup>
  <Collapser id="parse-ISO8601" title="Règle d'analyse pour l'horodatage au format ISO8601">
    Ajoutez cette règle d’analyse :

    ```
    %{TIMESTAMP_ISO8601:timestamp}
    ```

    A ce message de log avec horodatage au format <DNT>**ISO8601**</DNT> :

    ```
    2023-12-05T16:50:48.421Z 146.190.212.184: This is my sample non-JSON log message with ISO8601 timestamp.
    ```

    Pour analyser votre log comme suit :

    ```json
    "ISO8601_TIMEZONE": "Z",
    "message": "2023-12-05T16:50:48.421Z 146.190.212.184: This is my sample non-JSON log message with ISO8601 timestamp.",
    "timestamp": 1701795048421
    ```
  </Collapser>

  <Collapser id="parse-epoch" title="Règle d'analyse pour l'horodatage au format d'époque">
    Ajoutez la règle d’analyse suivante :

    ```
    %{NUMBER:timestamp:long}
    ```

    Au message suivant de log avec horodatage au format <DNT>**epoch**</DNT> :

    ```
    1701795958504 146.190.212.184: This is my sample non-JSON log message with epoch (with ms) timestamp.
    ```

    Pour analyser votre log comme suit :

    ```json
    "message": "1701795958504 146.190.212.184: This is my sample non-JSON log message with epoch (with ms) timestamp.",
    "timestamp": 1701795958504
    ```
  </Collapser>

  <Collapser id="parse-unsupported" title="Règle d'analyse pour l'horodatage dans un format non pris en charge">
    Ajoutez cette règle d’analyse :

    ```
    ^%{DATA:timestamp:datetime;MM/dd/yyyy HH:mm:ss.SSS} %{IP:ip}
    ```

    À ce message de log avec un format d&apos;horaire non pris en charge :

    ```
    12/05/2023 17:12:45.347 146.190.212.184: This is my sample non-JSON log message with un-supported timestamp.
    ```

    Pour analyser votre log comme suit :

    ```json
    "ip": "146.190.212.184",
    "message": "12/05/2023 17:12:45.347 146.190.212.184: This is my sample non-JSON log message with DATE_US timestamp.",
    "timestamp": 1701796365347
    ```
  </Collapser>
</CollapserGroup>
