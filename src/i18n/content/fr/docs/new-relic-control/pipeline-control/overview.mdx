Pipeline Control est le composant de gestion des données de New Relic Control. Il vous aide à filtrer, transformer et gérer les données de télémétrie pour réduire les coûts, améliorer la qualité du signal et maintenir la conformité.

Pipeline Control offre deux couches de traitement que vous pouvez utiliser indépendamment ou conjointement :

## Gateway de contrôle de pipeline

Les **règles du gateway** s&apos;exécutent dans votre infrastructure :

* Traitez les données **avant qu&apos;elles ne quittent votre réseau** (réduit les frais de sortie)
* Configuration à l&apos;échelle de l&apos;organisation (les règles s&apos;appliquent à tous les comptes de votre organisation)
* Capacités complètes de transformation, de filtrage et d&apos;échantillonnage
* Vous gérez l&apos;infrastructure (nécessite le déploiement d&apos;un cluster Kubernetes)
* Facturé en fonction du volume de données reçu au niveau du gateway

Le gateway est idéal lorsque vous avez besoin de :

* Pour réduire les coûts de sortie réseau en filtrant les données avant qu&apos;elles ne quittent votre infrastructure
* Capacités de transformation de données plus riches pour tous les types MELT (par ex. analyser les logs, normaliser les attributs, enrichir avec des métadonnées)
* Pour supprimer ou masquer les données sensibles avant qu&apos;elles ne quittent votre réseau afin de répondre aux exigences de conformité
* Pour traiter des volumes de données plus importants, le gateway étant moins coûteux que les règles cloud

En savoir plus sur le gateway dans [Présentation du gateway](/docs/new-relic-control/pipeline-control/gateway/overview).

## Règles cloud Pipeline Control

Les **règles cloud** s&apos;exécutent dans l&apos;infrastructure de New Relic une fois que les données atteignent les points de terminaison de New Relic :

* Traitez les données **après leur arrivée dans New Relic** (aucune gestion d&apos;infrastructure requise)
* Règles spécifiques au compte (chaque règle s&apos;applique à un seul compte New Relic)
* Règles de suppression basées sur le NRQL pour filtrer les données et les attributs
* Aucune configuration ni gestion d&apos;infrastructure requise
* Facturé sur la base des données rejetées (7,5 Go par aCCU) plus les milliards d&apos;événements analysés (2,237 BSE par aCCU)

Les règles cloud sont les plus efficaces lorsque :

* Vous souhaitez des règles de rejet simples sans gérer l&apos;infrastructure de gateway
* Vos besoins de traitement sont modestes et ne justifient pas un déploiement de gateway à grande échelle
* Filtrage de volumes plus faibles en pourcentage de votre ingestion totale, là où la tarification des règles cloud peut être plus rentable

En savoir plus : [Créer des règles cloud](/docs/new-relic-control/pipeline-control/cloud-rules/create-pipeline-rules)

## Utiliser les deux ensemble

Vous pouvez également utiliser simultanément les règles de gateway et de cloud :

* Sources de télémétrie (p. ex. agents) peuvent être configurés différemment pour passer par votre gateway ou non
* La stratégie de défense en profondeur garantit que les données sensibles sont absolument supprimées
* Pratique pour gérer certaines règles via l&apos;API/l&apos;interface utilisateur cloud tout en utilisant le gateway pour le traitement principal

## Tableau comparatif

| Fonctionnalité                      | Règles de gateway                                                                      | Règles Cloud                                                                       |
| ----------------------------------- | -------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| **Lieu de traitement**              | Votre infrastructure (Kubernetes)                                                      | Cloud New Relic                                                                    |
| **Lorsqu&apos;il traite**           | Avant que les données ne quittent votre réseau                                         | Une fois que les données atteignent New Relic                                      |
| **Configuration requise**           | Oui (cluster Kubernetes, chart Helm)                                                   | Non                                                                                |
| **Méthode de configuration**        | YAML ou UI                                                                             | Interface utilisateur ou API NerdGraph (basée sur NRQL)                            |
| **Portée de la règle**              | À l&apos;échelle de l&apos;organisation                                                | Spécifique au compte                                                               |
| **Capacités**                       | Transformer, supprimer des données/attributs, échantillonner par taux                  | Supprimer les données/attributs                                                    |
| **Types de données pris en charge** | Métriques, Événements, Logs, Traces                                                    | Métriques, Événements, Logs, Traces                                                |
| **Idéal pour**                      | Traitement de gros volumes, transformations, réduction des coûts de sortie, conformité | Règles de suppression simples, faible volume, aucune gestion d&apos;infrastructure |

Pour des informations détaillées sur la tarification et le calcul des coûts, consultez [les coûts de Pipeline Control](/docs/new-relic-control/pipeline-control/costs).

## Prérequis

### Pour les règles cloud

* Un compte New Relic disposant d&apos;un rôle avec des permissions RBAC pour gérer les règles cloud de Pipeline Control
* Compréhension du NRQL pour créer des règles de filtrage

### Pour le gateway

* Un compte New Relic avec les autorisations Administrateur produit de l&apos;organisation ou Gestionnaire de l&apos;organisation

<Callout variant="important" title="important">
  Les rôles utilisateur Administrateur produit de l&apos;organisation ou Gestionnaire de l&apos;organisation sont requis pour :

  * Installer le gateway
  * Ajouter un cluster à un gateway
  * Dèployer une configuration de gateway
  * Afficher les règles du gateway
</Callout>

* Familiarité avec les opérations et la gestion de Kubernetes
* Accès administrateur du cluster Kubernetes pour déployer et gérer les ressources au sein de votre cluster
* Autorisations pour créer et gérer un nom d&apos;hôte DNS privé au sein de votre réseau
* Autorisations pour créer et gérer les enregistrements DNS
* Helm installé localement

<Callout variant="important">
  * Les règles de contrôle de pipeline s&apos;appliquent uniquement aux données qui arrivent à partir du moment où vous créez la règle, elles ne suppriment pas les données qui [ont déjà été ingérées](/docs/telemetry-data-platform/ingest-manage-data/manage-data/manage-data-retention#data-deletion).
  * Lors de la création de règles, vous êtes responsable de vous assurer que les règles identifient et suppriment avec précision les données qui répondent aux conditions que vous avez établies. Vous êtes également responsable du monitoring de la règle, ainsi que des données que vous divulguez à New Relic.
</Callout>
