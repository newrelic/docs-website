<Callout variant="important">
  Agent Control et New Relic Control sont **en disponibilité générale** pour Kubernetes. La prise en charge des hôtes Linux et Windows fait partie du programme de **version préliminaire publique**, conformément à nos [politiques de version préliminaire](/docs/licenses/license-information/referenced-policies/new-relic-pre-release-policy).
</Callout>

## Présentation

Agent Control simplifie la gestion de vos agents d&apos;instrumentation. Ce guide vous guidera dans le processus d&apos;installation et de désinstallation d&apos;Agent Control sur vos clusters Kubernetes, hôtes Linux ou hôtes Windows à l&apos;aide de différentes méthodes.

## Installer Agent Control

<Callout title="conseil">
  Pour automatiser la configuration du contrôle des agents sur une infrastructure à grande échelle, consultez [Configurer le contrôle des agents avec Terraform](/docs/infrastructure-as-code/terraform/agent-control).
</Callout>

### Installation guidée

1. Connectez-vous à New Relic.
2. Assurez-vous que le bon compte est sélectionné.

<Callout title="important">
  Pour installer Agent Control, il est obligatoire d&apos;avoir une flotte. Si vous n&apos;avez pas encore créé de flotte pour cette entité gérée, vous pouvez créer une flotte pendant l&apos;installation dans le Guided Install ou terminer le processus de création de flotte dans **New Relic Control**, puis revenir à cette étape d&apos;installation guidée.
</Callout>

3. Dans Intégration et agents, cliquez sur **Installer Agent Control** ou recherchez **Agent Control**.

   <img title="Agent Control guided install" alt="Screenshot of the guided install tasks for Agent Control" src="/images/agent-control-guided-install.webp" />

4. Suivez les étapes pour terminer le processus d&apos;installation et de configuration.

<Callout title="conseil">
  Pour installer Agent Control, il est obligatoire d&apos;avoir une flotte. Si vous n&apos;avez pas encore créé de flotte pour cette entité gérée, vous pouvez en créer une durant l&apos;installation dans l&apos;installation guidée ou terminer le processus de création de flotte dans **Fleet Control**, puis revenir à cette étape de l&apos;installation guidée.
</Callout>

5. Téléchargez la configuration générée sur votre ordinateur et exécutez la commande fournie dans votre terminal pour installer Agent Control. Après l&apos;installation, cliquez sur **Continue**.
6. Testez la connexion pour confirmer que l&apos;installation a réussi. Cette étape peut prendre 5 à 10 minutes.

<Callout variant="important" title="Notes d'installation Windows">
  **Méthode d&apos;installation :** Agent Control pour Windows s&apos;installe à l&apos;aide de scripts PowerShell. Les programmes d&apos;installation MSI ou MSIX ne sont pas disponibles actuellement. Vous devez exécuter le script d&apos;installation PowerShell fourni avec des privilèges d&apos;administrateur.

  **Type de flotte requis :** Pour les hôtes Windows, vous devez sélectionner ou créer une flotte de type **&quot;Host - Windows&quot;** lors de l&apos;installation guidée. L&apos;utilisation d&apos;une flotte d&apos;un type différent (tel que &quot;Host - Linux&quot;) entraînera des problèmes d&apos;installation ou de fonctionnement.

  **Exigences de version de l&apos;Agent :** Assurez-vous de sélectionner des versions d&apos;Agent compatibles lors de l&apos;installation :

  * Agent d&apos;infrastructure : version 1.71.0 ou supérieure
  * NRDOT (Collecteur OpenTelemetry) : version 1.5.0 ou supérieure

  Les modèles et configurations des environnements Linux peuvent nécessiter des ajustements pour la compatibilité avec Windows.

  **Agents existants :** Si l&apos;agent Infrastructure New Relic est déjà installé sur votre hôte Windows, vous devez le désinstaller avant d&apos;installer Agent Control. Agent Control ne prend pas en charge la migration automatique des agents existants sous Windows. Après avoir installé Agent Control, vous pouvez gérer l&apos;Agent d&apos;infrastructure via Fleet Control. Les agents APM (qui ne sont pas actuellement gérés par Agent Control sous Windows) peuvent rester installés et continueront de fonctionner indépendamment.
</Callout>

### Remarque sur l&apos;authentification

New Relic Control nécessite l&apos;utilisation d&apos;identités système, qui sont des identités non humaines utilisées pour authentifier et établir la confiance entre les services et les applications.

Lors du processus d&apos;installation guidée d&apos;Agent Control, la première identité système est créée à l&apos;aide des identifiants client, qui sont inclus dans les valeurs du chart Helm. **Les identifiants de cette identité système expirent après 12 heures.** Lorsqu&apos;ils expirent, le déploiement du chart Helm Agent Control ne parviendra pas à s&apos;authentifier auprès du service Fleet Control, entraînant l&apos;erreur suivante :

```
Error getting system identity auth token. The API endpoint returned 400: Expired client secret.
```

Dans ce cas, le graphique Helm doit être mis à jour avec les nouvelles informations d&apos;identification d&apos;identité du système.

```yaml
global:
  cluster: "cluster-name"
  licenseKey: "*************************"
agentControlDeployment:
  chartValues:
    systemIdentity:
      organizationId: "00000000-0000-0000-0000-000000000000"
      parentIdentity:
        clientId: "CLIENT_ID"
        clientSecret: "CLIENT_SECRET"
    config:
      fleet_control:
        fleet_id: "SAMPLE_FLEET_ID"
      agents:
        ...
```

## Vérifier l&apos;installation

Maintenant qu&apos;Agent Control est installé et en cours d&apos;exécution, vous êtes prêt à [configurer et à gérer vos agents](/docs/new-relic-control/agent-control/configuration) ou à déployer des modifications sur vos agents à l&apos;aide du contrôle de la flotte]\(/docs/new-relic-control/fleet-control/overview).

### Kubernetes

1. Exécutez les commandes suivantes pour vérifier l’état de votre pod :

Agent Control installe des sous-agents dans un espace de noms différent pour des raisons de sécurité. Pour vérifier que tout fonctionne, vérifiez que les pods Agent Control s&apos;exécutent dans le namespace `newrelic-agent-control` et que les pods de sous-agent s&apos;exécutent dans un namespace différent, tel que `newrelic`.

```shell
kubectl get pods -n newrelic-agent-control    # Check Agent Control pods
kubectl get pods -n newrelic                  # Check subagent pods
```

2. connectez-vous à New Relic et accédez au **Fleet Control**.
3. Accédez à la page Flottes et sélectionnez la flotte que vous avez choisie lors de l&apos;installation.
4. Dans la table **Entities** , confirmez que votre cluster Kubernetes apparaît dans la liste.
5. Vérifiez que l’état de l’instrumentation de votre cluster est **sain**.

#### Prise en charge des installations Flux existantes

Par défaut, le chart Helm Agent Control utilise une instance intégrée de [Flux CD](https://fluxcd.io/) pour gérer le cycle de vie de vos agents dans Kubernetes. Cependant, si votre organisation utilise déjà **Flux v2** pour GitOps, vous pouvez configurer Agent Control pour tirer parti de votre installation existante.

Cette approche découple Agent Control du moteur de livraison continue intégré, vous permettant de maintenir une instance Flux unique pour vos opérations de cluster tout en bénéficiant des capacités de gestion d&apos;Agent Control.

##### Exigences et compatibilité

Pour utiliser une installation externe de Flux avec Agent Control, votre environnement doit répondre aux exigences suivantes. Les configurations qui s&apos;écartent de ces spécifications ne sont pas validées.

* **Version Flux :** Flux v2 ou supérieur.

* **Composants requis :** Votre installation Flux doit inclure :

  * **Contrôleur Helm :** Avec la CRD HelmRelease (helm.toolkit.fluxcd.io/v2).
  * **Contrôleur de source :** Avec la CRD HelmRepository (source.toolkit.fluxcd.io/v1).

* **Portée de l&apos;espace de noms :** Votre instance Flux doit être configurée pour surveiller l&apos;espace de noms dans lequel Agent Control sera installé (ou configurée pour surveiller tous les espaces de noms).

##### Configuration

Pour activer ce mode, vous devez désactiver explicitement les composants Flux inclus dans la configuration du chart Helm Agent Control.

Dans votre fichier `values.yaml`, définissez `agentControlCd.enabled` sur `false`:

```
global:
  cluster: "<YOUR_CLUSTER_NAME>"
  licenseKey: "<YOUR_LICENSE_KEY>"

# Disable the embedded Flux instance
agentControlCd:
  enabled: false

agentControlDeployment:
  chartValues:
    # ... other configurations ...
```

##### Permissions pour Flux externe

Lorsque vous utilisez votre propre installation Flux, le compte de service Flux de votre cluster est responsable de l&apos;application des configurations générées par Agent Control. Par conséquent, votre instance Flux existante nécessite des autorisations spécifiques pour déployer des ressources New Relic. Nous recommandons vivement l&apos;une des approches ci-dessous :

* **Administrateur de cluster (recommandé) :** La configuration la plus simple consiste à s&apos;assurer que votre instance Flux s&apos;exécute avec les privilèges `cluster-admin`. Il s&apos;agit de la configuration standard pour le chart communautaire Flux et elle garantit qu&apos;il peut gérer toutes les ressources nécessaires (Deployments, DaemonSets, Services, etc.) requises par les agents New Relic.
* **Configuration du moindre privilège :** Si vos politiques de sécurité restreignent l&apos;utilisation de `cluster-admin`, vous devez créer un `ClusterRole` spécifique garantissant que votre compte de service Flux dispose des autorisations requises par le Source Controller, le Helm Controller, l&apos;Agent Control et chaque agent spécifique que vous prévoyez d&apos;installer.

<Callout variant="important">
  Remarque : les autorisations de l&apos;Agent peuvent changer à mesure que de nouvelles fonctionnalités ou agents sont ajoutés. Vous êtes responsable de la maintenance de ces permissions dans votre rôle personnalisé.
</Callout>

Voici un exemple `ClusterRole` illustrant les permissions minimales requises pour que les composants Agent Control et Flux interagissent :

```

kind: ClusterRole

  - apiGroups: [""]
    resources: ["configmaps"]

  - apiGroups: [""]
    resources: ["events"]

  - apiGroups: [""]

    verbs: ["create"]
  - apiGroups: [""]

  - apiGroups: [""]

  - apiGroups: ["apps"]

  # Permissions required by Agent Control logic
```

##### Limites du support

Lors de l&apos;utilisation d&apos;une installation Flux externe, le support New Relic se limite au logiciel Agent Control (la génération de manifestes de configuration valides). Dans ce mode, vous devez prendre en compte un schéma de responsabilité partagée comme suit :

* **Responsabilité de New Relic :** Nous nous assurons qu&apos;Agent Control interagit correctement avec le backend New Relic et génère des définitions `HelmRelease` et `HelmRepository` valides.
* **Responsabilité du client :** Vous êtes responsable de l&apos;état de santé, de la maintenance des versions, de la mise en réseau et du dépannage de votre propre installation Flux. Les problèmes découlant spécifiquement de la configuration ou de la défaillance des contrôleurs Flux externes ne sont pas couverts par le support d&apos;Agent Control.

### Linux

* Vérifiez l&apos;état du service `newrelic-agent-control` :

  ```bash
    sudo systemctl status newrelic-agent-control
  ```

  Si le service apparaît à l&apos;état `Failed` ou `Stopped`, cela signifie que l&apos;agent a été installé mais qu&apos;un problème empêche son fonctionnement normal. Vérifiez les logs des services de l&apos;Agent avec `journalctl` (ou tout autre outil Linux similaire) :

  ```bash
    journalctl -u newrelic-agent-control
  ```

  Si aucune information n&apos;est disponible, consultez la procédure pour [exécuter l&apos;agent en mode débogage](/docs/new-relic-control/agent-control/troubleshooting/#debug) afin d&apos;accéder aux logs détaillés et de mieux comprendre pourquoi le service ne peut pas être démarré.

* Si le service n&apos;est pas installé, essayez d&apos;ajouter `--debug` à la fin de la commande d&apos;installation CLI de l&apos;installation guidée et relancez-la. Cela active le logs détaillé pour le script d&apos;installation et peut fournir un contexte supplémentaire expliquant l&apos;erreur.

* Facultativement, répondez `yes` lorsqu&apos;il vous est demandé d&apos;envoyer des logs à New Relic pour aider au dépannage de l&apos;installation. Une fois soumis, les logs sont accessibles avec la requête NRQL suivante :

  ```sql
    SELECT * FROM Log WHERE hostname = `your-host-name`
  ```

### Windows

#### Liste de contrôle de pré-installation

Avant d&apos;installer Agent Control sur Windows, vérifiez ces prérequis :

* [ ] **Privilèges d&apos;administrateur** - Cliquez avec le bouton droit sur PowerShell et sélectionnez &quot;Run as Administrator&quot;
* [ ] **Version de PowerShell** - Vérifier PowerShell 5.0 ou version ultérieure :
  ```powershell
  $PSVersionTable.PSVersion
  ```
* [ ] **Disponibilité du port** - Assurez-vous que le port 51200 n&apos;est pas utilisé :
  ```powershell
  netstat -ano | findstr :51200
  ```
  Si la commande ne renvoie aucune sortie, le port est disponible.
* [ ] **Agent d&apos;infrastructure supprimé** - Désinstallez l&apos;agent d&apos;infrastructure s&apos;il est déjà installé :
  ```powershell
  Get-Service -Name newrelic-infra -ErrorAction SilentlyContinue
  ```
  Si cela renvoie un service, désinstallez d&apos;abord l&apos;agent Infrastructure. Les agents APM peuvent rester installés.
* [ ] **Parc créé** - Créez un parc de type **&quot;Host - Windows&quot;** dans Fleet Control
* [ ] **Connectivité réseau** - Tester l&apos;accès aux services New Relic :
  ```powershell
  Test-NetConnection -ComputerName download.newrelic.com -Port 443
  ```
* [ ] **Configuration de l&apos;antivirus** - Ajouter des exclusions (voir les notes sur les logiciels de sécurité ci-dessous)

<Callout variant="caution" title="Antivirus et logiciels de sécurité">
  Windows Defender ou un logiciel antivirus tiers peut bloquer l&apos;exécution d&apos;Agent Control en tant que service. Avant l&apos;installation, ajoutez ces répertoires aux exclusions de votre antivirus :

  * `C:\Program Files\New Relic\newrelic-agent-control\`
  * `C:\ProgramData\New Relic\newrelic-agent-control\`

  Si Agent Control ne démarre pas après l&apos;installation et s&apos;exécute correctement depuis la ligne de commande, cela indique une interférence de l&apos;antivirus. Collaborez avec votre équipe de sécurité pour configurer les exceptions appropriées pour les exécutables New Relic.
</Callout>

#### Vérifier l&apos;installation

1. Vérifiez l&apos;état du service `newrelic-agent-control` :

   Ouvrez PowerShell avec les privilèges d&apos;administrateur et exécutez :

   ```powershell
   Get-Service -Name newrelic-agent-control
   ```

   Le service doit afficher le statut `Running`. Si le service apparaît à l&apos;état `Stopped` ou n&apos;existe pas, procédez au dépannage ci-dessous.

2. Vérifier l&apos;endpoint de santé de l&apos;Agent Control :

   ```powershell
   Invoke-WebRequest -Uri "http://localhost:51200/status" -UseBasicParsing
   ```

   Un Agent Control sain doit renvoyer une réponse JSON avec `"healthy": true`.

3. connectez-vous à New Relic et accédez au **Fleet Control**.

4. Accédez à la page Flottes et sélectionnez la flotte que vous avez choisie lors de l&apos;installation.

5. Dans le tableau **Entities**, confirmez que votre hôte Windows apparaît dans la liste.

6. Vérifiez que le statut de l&apos;instrumentation de votre hôte est **sain**.

#### À quoi s&apos;attendre après l&apos;installation

Après avoir exécuté le script d&apos;installation, Agent Control configure uniquement le service de supervision. Aucun agent d&apos;instrumentation n&apos;est déployé automatiquement.

**Immédiat (0-2 minutes) :**

* **0-30 secondes :** Le service Agent Control s&apos;enregistre et démarre
* **30-60 secondes :** Première connexion à Fleet Control établie
* **1 à 2 minutes :** La synchronisation de la configuration se termine et l&apos;hôte apparaît dans Fleet Control

**Prochaines étapes requises :** Une fois Agent Control installé et connecté à Fleet Control, vous devez déployer manuellement des agents pour commencer à collecter la télémétrie :

1. Connectez-vous à New Relic et accédez à **Fleet Control**
2. Sélectionnez votre flotte et localisez votre hôte Windows dans la table **Entities**
3. Déployez des agents (agent Infrastructure, NRDOT, etc.) sur votre hôte à l&apos;aide de Fleet Control
4. Attendez 5 à 10 minutes que les agents déployés démarrent et envoient la télémétrie à New Relic

<Callout variant="important">
  **Pas de télémétrie par défaut :** Agent Control est un service de supervision qui gère les agents. Il ne collecte ni n&apos;envoie de données de télémétrie lui-même. Pour consulter les métriques d&apos;infrastructure, les logs ou toute autre télémétrie dans New Relic, vous devez déployer et configurer des agents (tels que l&apos;agent Infrastructure) via Fleet Control après l&apos;installation.
</Callout>

Si le service Agent Control ne se connecte pas à Fleet Control dans les 2 à 3 minutes, consultez [Dépannage des hôtes Windows](/docs/new-relic-control/agent-control/troubleshooting/#windows-hosts-troubleshooting).

#### Commandes de gestion des services

Gérez le service Windows Agent Control avec ces commandes PowerShell (exécuter en tant qu&apos;administrateur) :

**Démarrer le service :**

```powershell
Start-Service -Name newrelic-agent-control
```

**Arrêtez le service :**

```powershell
Stop-Service -Name newrelic-agent-control
```

**Redémarrez le service :**

```powershell
Restart-Service -Name newrelic-agent-control
```

**Vérifiez l&apos;état du service :**

```powershell
Get-Service -Name newrelic-agent-control | Format-List Status, StartType
```

Sortie attendue lorsque l&apos;état est sain :

```
Status    : Running
StartType : Automatic
```

**Afficher la configuration du service :**

```powershell
Get-WmiObject -Class Win32_Service -Filter "Name='newrelic-agent-control'" | Format-List Name, State, StartMode, StartName
```

#### Structure des répertoires

Agent Control crée la structure de répertoires suivante sous Windows :

**Fichiers de programme (exécutables) :**

```
C:\Program Files\New Relic\newrelic-agent-control\
├── newrelic-agent-control.exe       # Main service executable
├── uninstall.ps1                     # Uninstallation script
└── local-data\
    └── agent-control\
        └── local_config.yaml         # Local configuration file
```

**Données du programme (données d&apos;exécution) :**

```
C:\ProgramData\New Relic\newrelic-agent-control\
├── fleet-data\
│   └── agent-control\
│       └── remote_config.yaml        # Configuration synced from Fleet Control
├── logs\
│   └── newrelic-agent-control.log    # Service logs
├── filesystem\                       # Managed agent configurations and data
└── packages\                         # Downloaded agent packages (cached)
```

<Callout variant="tip" title="Emplacement des logs">
  **Retrouvez les logs de l&apos;Agent Control sous :** `C:\ProgramData\New Relic\newrelic-agent-control\logs\`

  Les fichiers logs incluent des suffixes de date (par ex. `newrelic-agent-control.log.2026-02-09-11`).

  Afficher les logs récents du dernier fichier :

  ```powershell
  Get-ChildItem "C:\ProgramData\New Relic\newrelic-agent-control\logs\" -Filter "newrelic-agent-control.log.*" | Sort-Object LastWriteTime -Descending | Select-Object -First 1 | Get-Content -Tail 100
  ```
</Callout>

#### Fonctionnalités spécifiques à Windows

**Collecte de logs :** Les hôtes Windows peuvent collecter des logs via le Fluent Bit intégré à l&apos;agent d&apos;infrastructure. Le package de l&apos;agent d&apos;infrastructure inclut les binaires Fluent Bit pour Windows, vous permettant de configurer le transfert de logs à l&apos;aide du paramètre `config_logging` dans la configuration de votre agent. Les configurations de logs fonctionnent de manière similaire aux hôtes Linux, mais utilisent des chemins et des formats spécifiques à Windows.

**Notes de configuration :**

* Les chemins de fichiers dans les configurations Windows utilisent des barres obliques inverses (`\\`) et des chemins de type Windows (par ex. `C:\\ProgramData\\...`)
* Les variables d&apos;environnement sont définies selon les conventions Windows.
* Les binaires de l&apos;Agent utilisent des extensions `.exe`
* Les fichiers de configuration sont stockés aux emplacements indiqués dans la structure de répertoires ci-dessus.
