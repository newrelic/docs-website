<Callout variant="important">
  Agent Control et New Relic Control sont **en disponibilité générale** pour Kubernetes. La prise en charge des hôtes Linux et Windows fait partie du programme de **version préliminaire publique**, conformément à nos [politiques de version préliminaire](/docs/licenses/license-information/referenced-policies/new-relic-pre-release-policy).
</Callout>

## Présentation

Agent Control fournit une plate-forme unique et unifiée pour gérer une grande variété d&apos;agents New Relic et OpenTelemetry dans vos flottes. En centralisant la gestion, vous pouvez déployer, configurer et mettre à jour votre instrumentation en toute simplicité.

Bien qu&apos;Agent Control soit conçu pour prendre en charge à la fois les environnements Kubernetes et les environnements basés sur des hôtes, la prise en charge de types d&apos;agents spécifiques varie. Le tableau suivant fournit une vue d&apos;ensemble complète des agents pouvant être gérés par Agent Control et de leur statut de prise en charge dans les différents environnements.

<Callout variant="important">
  Autorisations spécifiques à l&apos;agent : Agent Control est conçu pour vous fournir une gestion flexible des autorisations. Bien que le contrôle des agents lui-même nécessite un certain niveau d&apos;accès pour fonctionner, les autorisations qu&apos;il accorde aux agents individuels sont adaptées à leurs besoins spécifiques. Ci-dessous, vous trouverez une répartition des autorisations requises pour chaque type d’agent.
</Callout>

## Autorisations requises par type d&apos;agent

| Type d&apos;agent                               | Autorisations clés requises                                                                                                                                       | Environnement                          |
| :---------------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------- |
| **Agent d&apos;infrastructure New Relic**       | Accès au niveau de l&apos;hôte pour le système métrique et accès API Kubernetes pour les données cluster .                                                        | Kubernetes / basé sur l&apos;hôte      |
| **Collector OpenTelemetry New Relic (NRDOT)**   | Les autorisations dépendent du Récepteur et de l&apos;exportateur spécifiques. Nécessite souvent un accès à l&apos;API Kubernetes pour la découverte de services. | Kubernetes / basé sur l&apos;hôte      |
| **Fluent Bit**                                  | Accès en lecture aux logs de pod et du conteneur.                                                                                                                 | Kubernetes                             |
| **Agent Prometheus de New Relic**               | Autorisations pour découvrir et accéder au point de terminaison de service au sein du cluster pour récupérer les métriques.                                       | Kubernetes                             |
| **Agent eBPF New Relic**                        | Privilèges élevés (par exemple, `CAP_SYS_ADMIN`) pour charger des programmes eBPF sur le noyau hôte.                                                              | Kubernetes, hôtes Linux (expérimental) |
| **Agents APM (.NET, Java, Node, Python, Ruby)** | *Non pris en charge actuellement par Agent Control.*                                                                                                              | N/A                                    |

<Callout variant="caution" title="NRDOT sur les hôtes Windows est expérimental">
  Le New Relic OpenTelemetry Collector (NRDOT) sous Windows est disponible, mais n&apos;est pas officiellement testé ni documenté par l&apos;équipe NRDOT. La configuration groupée par défaut est conçue pour Linux et peut générer des avertissements ou des erreurs sous Windows (par exemple, à partir des chemins `filelogreceiver`). Aucune configuration par défaut n&apos;est fournie pour Windows — vous devez fournir votre propre configuration de collecteur. Utilisez NRDOT sur Windows uniquement dans des environnements non critiques ou de test jusqu&apos;à ce que la prise en charge complète de Windows soit disponible.
</Callout>

<Callout variant="caution" title="eBPF sur les hôtes Linux est expérimental">
  L&apos;agent eBPF New Relic nécessite des dépendances au niveau du noyau (telles que `linux-headers` correspondant à la version du noyau en cours d&apos;exécution) que Agent Control ne peut pas résoudre automatiquement sur les hôtes Linux. Si ces dépendances sont manquantes ou incompatibles, le déploiement peut échouer sans erreur explicite. La prise en charge d&apos;eBPF sur les hôtes Linux est disponible pour les environnements Kubernetes uniquement en production. Utilisez eBPF sur les hôtes Linux uniquement dans des environnements non critiques ou de test.

  eBPF n&apos;est **pas pris en charge** sur les hôtes Windows.
</Callout>
