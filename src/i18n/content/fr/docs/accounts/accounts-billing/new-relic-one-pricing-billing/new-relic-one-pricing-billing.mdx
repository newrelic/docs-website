Que vous envisagiez d&apos;utiliser New Relic ou que vous l&apos;utilisiez déjà, les informations suivantes vous seront utiles pour comprendre comment New Relic fixe le prix de ses produits et facture votre utilisation.

Pour en savoir plus, consultez les options de facturation ci-dessous et sur notre [site web](https://newrelic.com/pricing).

## Modèle de tarification [#pricing-model]

Notre modèle de tarification décrit les capacités existantes du produit et indique leurs options de facturation possibles, telles que Données + utilisateur ou Données + Core Compute pour une observabilitéstack et Advanced Compute pour une observabilité intelligente. Comprendre ces modèles de tarification vous aide à choisir ce qui convient le mieux aux besoins de votre entreprise.

<Callout title="Note">
  Le modèle de tarification à la consommation [Core Compute](/docs/licenses/license-information/product-definitions/new-relic-one-pricing-definitions/#compute-capacity-unit) offre à l’ensemble de votre organisation d’ingénierie un accès illimité à la plateforme et des capacités d’observabilité full-stack sans barrières de licence par utilisateur. Étant donné que Core Compute est actuellement en version préliminaire, si vous souhaitez tester le modèle de tarification, vous devriez contacter votre responsable de compte New Relic.
</Callout>

  <Collapser id="data" title="Données">
    Désigne la quantité de données ingérées par New Relic, mesurée en [Go ingérés](/docs/licenses/license-information/product-definitions/new-relic-one-pricing-definitions/#gb-ingested). Nous proposons deux options pour GB Ingested, comme suit :

    * <b>Données originales :</b> l&apos;option de données originales de New Relic offre une durée par défaut d&apos;au moins 8 jours de rétention des données et une limite de requête maximale de 20 milliards de points de données par minute.

    * <b>Data Plus :</b> L’option Data Plus de New Relic offre des fonctionnalités avancées telles qu’une conservation étendue des données jusqu’à 90 jours et une limite de requêtes maximale accrue de 60 milliards de points de données par minute.

    Nous facturons les données en fonction de la quantité de Go ingérée, quel que soit le type, ce qui rend votre facture facile à comprendre. Vous bénéficiez de 100 Go de données ingérées par mois gratuitement. Pour suivre votre consommation de Go, vous pouvez utiliser notre page [de résumé d&apos;utilisation](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/usage-ui) ou [exécuter une requête](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/usage-queries-alerts).

    ### Quelles données sont considérées comme facturables ? [#usage-calculation]

    De manière générale, les « données ingérées » désignent les données stockées dans New Relic par votre organisation après application de diverses règles de nettoyage et de transformation des données. Autrement dit, ce n&apos;est pas la quantité de données brutes envoyées à New Relic qui importe, mais la taille des données qui sont effectivement stockées.

    En règle générale, les données utilisées par notre fonctionnalité ne sont pas comptabilisées dans les Go ingérés. Exemples de données relatives aux fonctionnalités qui ne sont pas prises en compte dans le calcul des Go ingérés :

    * [Fonctionnalité d&apos;alerte de base](/docs/alerts-applied-intelligence/new-relic-alerts/learn-alerts/alerts-concepts-workflow). Par exemple, `NrAiIncident`, un événement utilisé pour générer ces fonctionnalités, ne compte pas dans les Go ingérés.
    * Données créées à partir de données brutes agrégées (par exemple, [des données infrastructure agrégées](/docs/data-apis/manage-data/manage-data-retention/#infrastructure-data) ou [des données métriques dimensionnelles](/docs/data-apis/understand-data/new-relic-data-types/#dimensional-metrics)).
    * [Métriques dorées](/docs/apis/nerdgraph/examples/golden-metrics-entities-nerdgraph-api-tutorial) (pour en savoir plus sur les métriques dorées, reportez-vous aux [définitions des entités](https://github.com/newrelic/entity-definitions)).
    * Contrôles de monitoring synthétiques. Pour plus de détails sur la facturation, [les limites et la tarificationSynthetics ](docs/synthetics/synthetic-monitoring/getting-started/monitor-limits/).
    * Données de suivi d’utilisation et de facturation (par exemple, `NrConsumption`).
    * Données relatives à l’administration du compte (par exemple, `NrIntegrationError`, `NrAuditEvent`).

    Le nombre mensuel de Go ingérés est arrondi à l&apos;entier inférieur le plus proche. Par exemple, si votre organisation utilise 100,9 Go au cours d&apos;un mois, cela est comptabilisé comme 100 Go.
  </Collapser>

  <Collapser id="user" title="Utilisateur">
    Un utilisateur de New Relic est une personne qui peut accéder à la plateforme et l&apos;utiliser. Le nombre d&apos;utilisateurs facturables est un facteur de facturation.

    ### Utilisateurs : Facturation et règles [#billable-users]

    Les utilisateurs sont facturables dès leur ajout en tant qu&apos;utilisateurs facturables à une organisation New Relic. Voici les types d&apos;utilisateurs de New Relic :

    * **Utilisateur de base :** En tant qu’utilisateur de base, vous ne serez pas facturé et vous pourrez configurer nos outils d’observabilité, exécuter des requêtes sur les données et utiliser un dashboard personnalisé (dashboard quickstart pendant 7 jours maximum). Vous aurez également accès à certaines fonctionnalités d&apos;alerte de base. Cependant, vous ne pouvez pas utiliser nos expériences sélectionnées, telles que l&apos;interface utilisateur APM, l&apos;interface utilisateur monitoring des navigateurs ou l&apos;interface utilisateur mobile. Bien que votre accès n&apos;entraîne pas de frais directs, vous pouvez facturer les fonctionnalités basées sur l&apos;utilisation, y compris l&apos;option CCU.

    * **Core utilisateur :** en tant que Core utilisateur, vous bénéficiez de toutes les fonctionnalités utilisateur de base ainsi que d&apos;un accès à de puissantes fonctionnalités centrées sur les développeurs, telles que New Relic CodeStream, la boîte de réception des erreurs et l&apos;interface utilisateur de notre gestion des log.

    * **Plateforme utilisateur complète :** En tant que plateforme utilisateur complète, vous avez accès à de nombreuses fonctionnalités de notre plateforme. Cela inclut nos expériences d&apos;interface utilisateur d&apos;observabilité plus soigneusement sélectionnées, telles que l&apos;APM, monitoring de infrastructure, monitoring des navigateurs, monitoring des applications mobiles et monitoring synthétique.

    Pour en savoir plus, consultez [les types d&apos;utilisateurs](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type/).

    <CollapserGroup>
      <Collapser id="user-count" title="Comment les utilisateurs facturables sont comptabilisés">
        La facturation des utilisateurs facturables s&apos;effectue par mois civil. Pour déterminer le nombre d&apos;utilisateurs facturables d&apos;une organisation au cours d&apos;un mois civil, nous comptons le nombre d&apos;utilisateurs au cours de ce mois qui avaient un <DNT>**billable user type**</DNT> d&apos;utilisateur de la plateforme complète ou d&apos;utilisateur de base. Le <DNT>**billable user type**</DNT> d&apos;un utilisateur est défini comme le type d&apos;utilisateur le plus élevé auquel un utilisateur a été affecté au cours d&apos;un mois calendaire. Nous utilisons le fuseau horaire UTC pour définir le début et la fin d&apos;un mois calendaire.

        Par exemple, si un utilisateur est défini comme utilisateur de la plateforme complète à un moment donné au cours d&apos;un mois civil, son type d&apos;utilisateur facturable pour ce mois est `full platform user` et ne changera pas, même s&apos;il passe à une version inférieure plus tard dans le mois. Cela reste vrai même si cet utilisateur n&apos;est que brièvement transformé en utilisateur de la plateforme complète. La modification sera prise en compte le mois calendaire suivant et cet utilisateur ne sera pas comptabilisé comme utilisateur de la plateforme complète, sauf si elle est modifiée ultérieurement.

        Si vous prévoyez d&apos;ajouter un utilisateur facturable ou de modifier le type d&apos;utilisateur de votre utilisateur, vous devriez garder ces conseils à l&apos;esprit :

        * Si vous souhaitez ajouter un utilisateur facturable ou mettre à niveau un utilisateur, il est préférable de le faire en début de mois.
        * Si vous souhaitez rétrograder un utilisateur, il est préférable de le faire à la fin du mois.
        * Nous utilisons le temps UTC pour déterminer le début et la fin d&apos;un mois. Cela signifie que si, par exemple, vous étiez sur le fuseau horaire du Pacifique et que vous souhaitiez rétrograder le statut utilisateur le dernier jour du mois, vous deviez effectuer ces modifications avant 17h, heure du Pacifique.

        Les utilisateurs de votre organisation ayant la même adresse électronique ne sont comptabilisés que comme un seul utilisateur. Pour plus d&apos;informations, consultez la section [relative au suivi des utilisateurs](/docs/accounts/accounts-billing/account-setup/multiple-logins-found#user-records) dans notre documentation.

        Si vous utilisez notre [modèle utilisateur le plus récent](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models), les utilisateurs sont facturables lorsqu&apos;ils sont ajoutés par les administrateurs en tant qu&apos;utilisateur facturable. Même si un utilisateur facturable ne s&apos;est jamais connecté à New Relic et possède une tag `Pending invite` dans l&apos;interface utilisateur, il est toujours facturable (si vous utilisez notre [modèle d&apos;utilisateur d&apos;origine](/docs/accounts/original-accounts-billing/original-users-roles/users-roles-original-user-model), les utilisateurs `Pending invite` ne sont pas facturables).

        Pour en savoir plus sur l&apos;ajout et l&apos;invitation d&apos;utilisateurs, consultez la section [« Ajouter un utilisateur »](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-ui-and-tasks#add-users) de notre documentation.

        Le coût de votre utilisateur facturable dépend de [l&apos;édition tarifaire](https://newrelic.com/pricing) de votre organisation (Standard, Pro ou Enterprise), ou de toute tarification personnalisée que vous pourriez avoir avec New Relic.

        Lors de la première facturation d&apos;une organisation New Relic, nous calculerons au prorata le nombre d&apos;utilisateurs facturables en fonction de votre date de début au cours de ce mois. Nous vous rembourserons également votre dernier mois au prorata si vous annulez votre abonnement.
      </Collapser>

      <Collapser id="user-downgrade-rules" title="Qègles de rétrogradation des utilisateurs">
        Les règles relatives au nombre de fois où vous pouvez rétrograder en tant qu&apos;utilisateur de la plateforme complète diffèrent selon que votre [plan d&apos;utilisation](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing#usage-plans) est à la carte (PAYG) ou un contrat avec engagement.

        ### Règles de rétrogradation à la carte

        Pour le forfait à la carte (PAYG), il n&apos;existe aucune règle limitant la rétrogradation de l&apos;utilisateur. Veuillez noter que le [calcul de la facturation](#user-count) peut avoir une incidence sur vos décisions concernant la mise à niveau ou la rétrogradation du forfait utilisateur.

        ### Règles de déclassement des contrats d&apos;engagement

        Avant de mettre à niveau ou de rétrograder un utilisateur, vous devez comprendre [comment fonctionne la facturation des utilisateurs](#user-count).

        Pour les organisations ayant des contrats d&apos;engagement, la règle régissant les mises à niveau et les rétrogradations des utilisateurs de la plateforme complète est la suivante :

        <DNT>**If a Full Platform user is downgraded to a lower user type in a later month and then returned to a Full Platform user in a later month, and this downgrade/upgrade cycle happens twice in a contract year, that user will be billed as a Full Platform user for the remainder of the year, regardless of edits to their user type.**</DNT> Il n&apos;existe aucune règle régissant les changements de type d&apos;utilisateur qui se produisent au cours d&apos;un mois en raison du [fonctionnement de la facturation des utilisateurs](#user-count): le type d&apos;utilisateur facturable d&apos;un utilisateur pour un mois calendaire est le type d&apos;utilisateur le plus élevé auquel il est défini au cours de ce mois.
      </Collapser>
    </CollapserGroup>

    ### Gérer les utilisateurs et les coûts [#manage]

    Pour gérer l&apos;utilisateur, reportez-vous à la section [gestion des utilisateurs](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-ui-and-tasks) dans notre documentation. Vous pouvez utiliser l&apos; [interface utilisateur d&apos;utilisation](/docs/accounts/accounts-billing/general-account-settings/introduction-account-settings/#pricing) pour obtenir un aperçu du nombre d&apos;utilisateurs facturables. Si vous avez besoin de plus de détails que ceux fournis par l&apos;interface utilisateur, vous pouvez également exécuter [une requête NRQL relative à l&apos;utilisation](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/usage-queries-alerts).
  </Collapser>

  <Collapser id="core-compute-capacity-unit" title="Core Compute">
    Core Compute est une tarification à la consommation pour des capacités d&apos;observabilité full-stack avec un nombre illimité d&apos;utilisateurs. Votre facture est basée sur [les Core unités de capacité de calcul](/docs/licenses/license-information/product-definitions/new-relic-one-pricing-definitions/#compute-capacity-unit) (Core CCU) consommées pour exécuter avec succès des actions telles que des requêtes, des alertes, des appels d&apos;API et des chargements de pages. Le tableau des modèles de tarification indique pour quelles fonctionnalités vous pouvez choisir Core Compute comme modèle de tarification alternatif pour l&apos;utilisateur.
  </Collapser>

  <Collapser id="advance-compute-capacity-unit" title="Compute avancé">
    Advanced Compute est un module complémentaire à votre forfait Data + utilisateur ou Data + Core Compute qui fournit une tarification à la consommation pour la fonctionnalité d&apos;observabilité intelligente. Ce module complémentaire est mesuré en [unité de capacité de calcul](/docs/licenses/license-information/product-definitions/new-relic-one-pricing-definitions/#compute-capacity-unit) avancée (aCCU). Vous êtes facturé en fonction de votre utilisation des unités de contrôle avancées.

    Si les capacités full-stack d’observabilité sont combinées à une fonctionnalité d’observabilité intelligente, l’unité de capacité de calcul avancée appropriée vous sera facturée.

    <Callout variant="tip">
      Votre utilisation de CodeStream dans le cadre Advanced Compute entraînera des frais CCU avancés, quel que soit le type d&apos;utilisateur. Cela signifie que l&apos;utilisateur de base peut accéder aux fonctionnalités Advanced Compute et encourir des frais pour les unités de calcul avancées.
    </Callout>
  </Collapser>

### Facturation des fonctionnalités complémentaires [#add-on-billing]

En plus des principaux facteurs de facturation que sont les données et l&apos;utilisateur facturable, vous pouvez également utiliser des modules complémentaires facturables optionnels pour enrichir votre expérience avec New Relic.

* **Advanced Compute**: Ce module complémentaire à votre forfait Data + utilisateur ou Data + Core Compute vous permet d’accéder à la fonctionnalité d’observabilité intelligente.
* **Centre de données UE pour les données originales ou les données Plus**: ce module complémentaire s’applique à votre option de données (données originales ou données Plus) lorsque vous sélectionnez l’Union européenne comme région de données, selon les disponibilités.
* **Conservation prolongée des données d&apos;origine ou des données plus** : ce module complémentaire s&apos;applique si vous dépassez la durée par défaut pendant laquelle vos données sont conservées. Cela s&apos;applique à toutes vos données, pas seulement aux logs, et constitue une bonne option si vous devez effectuer de nombreuses petites requêtes ou effectuer des requêtes sur de gros volumes de données.
* **Live Archives** : Prolongez la durée de stockage de vos logs jusqu&apos;à sept ans. Live Archives nécessite également Advanced Compute.
* **New Relic Synthetics Checks** : Ce module complémentaire s&apos;applique si vos contrôles dépassent le nombre par défaut de contrôles de monitoring synthétiques.
