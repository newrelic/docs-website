[OpenTelemetry Protocol](https://github.com/open-telemetry/opentelemetry-proto/blob/main/docs/specification.md) (OTLP) est un protocole de livraison de données télémétriques à usage général conçu pour le projet OpenTelemetry . Chaque OpenTelemetry SDK de langage fournit un exportateur OTLP, et le OpenTelemetry Collector dispose d&amp;apos;un récepteur et d&amp;apos;un exportateur OTLP. De plus, divers outils extérieurs au projet OpenTelemetry ont ajouté la prise en charge de l&amp;apos;exportation OTLP.

New Relic prend en charge l’ingestion OTLP native et la recommande comme méthode préférée pour l’envoi de données OpenTelemetry à la plateforme New Relic. Cette page couvre le support OTLP de New Relic, y compris la configuration, les exigences et les recommandations.

## Avant de commencer [#before-you-begin]

* Si vous ne l&apos;avez pas déjà fait, créez un [compte New Relic](https://newrelic.com/signup) gratuit.

* Obtenez votre <InlinePopover type="licenseKey" />pour le compte New Relic auquel vous souhaitez signaler des données. Cette clé de licence sera utilisée lors de [la configuration de l&apos;en-tête `api-key` ](#api-key).

## Configuration : point de terminaison OTLP, port et protocole [#configure-endpoint-port-protocol]

Niveau d&apos;exigence : **Obligatoire**

Afin de configurer l&apos;envoi de données OTLP à New Relic, vous devez configurer votre exportateur OTLP pour utiliser le point de terminaison et le port appropriés du tableau ci-dessous en fonction de votre environnement.

Le mécanisme de configuration du point de terminaison varie, mais les SDK de langage OpenTelemetry prennent généralement en charge la définition de la variable d&apos;environnement `OTEL_EXPORTER_OTLP_ENDPOINT=<INSERT_ENDPOINT>` (consultez [la documentation OpenTelemetry](https://opentelemetry.io/docs/specs/otel/protocol/exporter/) pour plus d&amp;apos;informations).

De plus, vous devez configurer votre exportateur OTLP pour utiliser la [version protobuf binaire OTLP/HTTP](https://opentelemetry.io/docs/specs/otlp/#binary-protobuf-encoding) du protocole si disponible. Bien que New Relic prenne en charge toutes les versions d&amp;apos;OTLP, le protocole binaire protobuf OTLP/HTTP s&amp;apos;est avéré plus robuste que gRPC sans aucune réduction apparente des performances.

Le mécanisme de configuration du point de terminaison varie, mais les SDK de langage OpenTelemetry prennent généralement en charge la définition de la variable d&apos;environnement `OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf` (consultez [la documentation OpenTelemetry](https://opentelemetry.io/docs/specs/otel/protocol/exporter/) pour plus d&amp;apos;informations).

Si vous utilisez un Collector, nous vous recommandons d&apos;utiliser [otlphttpexporter](https://github.com/open-telemetry/opentelemetry-collector/tree/main/exporter/otlphttpexporter).

<CollapserGroup>
  <Collapser className="freq-link" id="note-ports" title="Pourquoi trois ports ?">
    Les normes OTLP désignent le trafic gRPC vers le port `4317` et le trafic HTTP vers le port `4318`. Le point de terminaison OTLP natif New Relic US FedRamp adhère à ces spécifications. Il autorise également le trafic gRPC et HTTP sur le port `443`. Pour plus de simplicité, vous pouvez omettre le port puisque le schéma `https` sélectionne automatiquement le port `443`.
  </Collapser>

  <Collapser className="freq-link" id="note-endpoints" title="Détails supplémentaires du point de terminaison">
    Conformément à la documentation OpenTelemetry sur [les URL de point de terminaison pour OTLP/HTTP](https://opentelemetry.io/docs/specs/otel/protocol/exporter/#endpoint-urls-for-otlphttp), si vous envoyez du trafic OTLP/HTTP et utilisez la variable d&amp;apos;environnement indépendante du signal (`OTEL_EXPORTER_OTLP_ENDPOINT`), vous pouvez simplement définir `OTEL_EXPORTER_OTLP_ENDPOINT` sur `https://otlp.nr-data.net:{port}` et l&amp;apos;exportateur doit ajouter le chemin approprié pour le type de signal (tel que `v1/traces` ou `v1/metrics`).

    Si vous utilisez une variable d&apos;environnement spécifique au signal (telle que `OTEL_EXPORTER_OTLP_TRACES_ENDPOINT` et `OTEL_EXPORTER_OTLP_METRICS_ENDPOINT`), vous devez inclure le chemin approprié. Par exemple:

    ```
    OTEL_EXPORTER_OTLP_TRACES_ENDPOINT=https://otlp.nr-data.net:4318/v1/traces
    ```

    Si le chemin ne suit pas la syntaxe ci-dessus, vous recevrez une erreur `404` . Notez que les variables d’environnement spécifiques au signal ont priorité sur les variables d’environnement indépendantes du signal.
  </Collapser>
</CollapserGroup>

## Configuration : chiffrement TLS [#tls]

Niveau d&apos;exigence : **Obligatoire**

Afin d&apos;envoyer des données OTLP à New Relic, vous devez configurer votre exportateur OTLP pour utiliser TLS 1.2 (voir [Chiffrement TLS](/docs/new-relic-solutions/get-started/networks/#tls) pour plus d&amp;apos;informations). Généralement, SDK et Collector exportateur répondent à cette exigence par défaut.

Alors que de nombreux exportateurs OTLP déduisent les paramètres TLS du schéma de point de terminaison `https`, certains exportateurs gRPC peuvent nécessiter que vous activiez explicitement TLS. Le mécanisme de configuration de gRPC TLS varie, mais les SDK de langage OpenTelemetry prennent généralement en charge la définition de la variable d&amp;apos;environnement `OTEL_EXPORTER_OTLP_INSECURE=false` (voir [la documentation OpenTelemetry](https://opentelemetry.io/docs/specs/otel/protocol/exporter/) pour plus d&amp;apos;informations).

## Configuration : Définition de la clé API [#api-key]

Niveau d&apos;exigence : **Obligatoire**

Pour envoyer des données OTLP à New Relic, vous devez configurer votre exportateur OTLP pour inclure un en-tête nommé `api-key` avec la valeur définie sur votre [clé de licence](#prereqs). Le non-respect de cette consigne entraînera des erreurs d’authentification.

Le mécanisme de configuration des en-têtes varie, mais les SDK de langage OpenTelemetry prennent généralement en charge la définition de la variable d&apos;environnement `OTEL_EXPORTER_OTLP_HEADERS=api-key=<INSERT_LICENSE_KEY>` (consultez [la documentation OpenTelemetry](https://opentelemetry.io/docs/specs/otel/protocol/exporter/) pour plus d&amp;apos;informations).

## Configuration : limites d&apos;attribut [#attribute-limits]

Niveau d&apos;exigence : **Obligatoire**

Afin d&apos;envoyer des données OTLP à New Relic, vous devez configurer votre source de télémétrie pour qu&apos;elle soit conforme aux limites d&apos;attributs de New Relic. Le non-respect de cette consigne peut entraîner un événement [`NrIntegrationError`](/docs/data-apis/manage-data/nrintegrationerror/) lors de la validation asynchrone des données.

Les limites d&apos;attribut sont les suivantes :

* Longueur maximale du nom de l&apos;attribut : 255 caractères
* Longueur maximale de la valeur de l&apos;attribut : 4 095 caractères
* Taille maximale de la valeur de l&apos;attribut éventail : 64 entrées

Voir [limites d&apos;attributs métriques](/docs/data-apis/ingest-apis/metric-api/metric-api-limits-restricted-attributes/) et [limites d&apos;attributs d&apos;événement](/docs/data-apis/ingest-apis/event-api/introduction-event-api/#limits) pour d&amp;apos;autres limites.

Le mécanisme de configuration des limites d&apos;attribut varie, mais les SDK de langage OpenTelemetry prennent généralement en charge la définition des variables d&apos;environnement `OTEL_ATTRIBUTE_VALUE_LENGTH_LIMIT=4095` et `OTEL_ATTRIBUTE_COUNT_LIMIT=64` (consultez [la documentation OpenTelemetry](https://opentelemetry.io/docs/specs/otel/configuration/sdk-environment-variables/) pour plus d&amp;apos;informations).

Lorsque vous utilisez le collecteur, vous pouvez configurer le [processeur de transformation](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/transformprocessor) pour tronquer l&amp;apos;attribut aux limites de New Relic.

Remarques :

* Les attributs de ressources sont soumis à des limites d&apos;attributs, mais il n&apos;existe pas de variables d&apos;environnement standard pour les limiter. Si un attribut de ressource dépasse la limite autorisée, envisagez de le tronquer à l&apos;aide du processeur Collector [de transformation ou](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/transformprocessor) d&amp;apos;écraser l&amp;apos;attribut de ressource avec une autre valeur.
* Il n’existe pas de mécanisme standard pour limiter les noms d’attributs. Cependant, l&apos;instrumentation ne produit généralement pas de noms d&apos;attributs qui dépassent les limites de New Relic. Si vous rencontrez des limites de longueur de nom, supprimez l&apos;attribut à l&apos;aide du Collector.

## Configuration : traitement par lots de la charge utile, délai d&apos;expiration, compression et limites de débit [#payload-limits]

Niveau d&apos;exigence : **Obligatoire**

Afin d&apos;envoyer des données OTLP à New Relic, votre charge doit être inférieure à la taille de charge maximale de 1 Mo (10^6 octets). Les frais plus élevés seront rejetés avec un code d&apos;état d&apos;erreur. Des frais plus élevés peuvent également ne pas être exportés en raison d&apos;un délai d&apos;attente avant qu&apos;un code d&apos;état d&apos;erreur ne soit renvoyé.

De plus, New Relic impose [des limites de débit](/docs/data-apis/manage-data/view-system-limits/#all_products). Lorsque la limite de débit est dépassée, requests seront rejetées avec un code d&amp;apos;état d&amp;apos;erreur.

Pour éviter les limites de taille de charge utile et de débit, vous devez configurer votre exportateur OTLP pour utiliser une taille de lot appropriée qui entraîne l&apos;exportation des données à un intervalle approprié.

Le mécanisme de configuration du traitement par lots varie. Les SDK OpenTelemetry prennent généralement en charge la définition des variables d&apos;environnement suivantes (consultez [la documentation OpenTelemetry](https://opentelemetry.io/docs/specs/otel/configuration/sdk-environment-variables/) pour plus d&amp;apos;informations) :

* `OTEL_BSP_*` pour les travées
* `OTEL_METRIC_EXPORT_*` pour les métriques
* `OTEL_BLRP_*` pour les logs

Si vous utilisez le Collector, le [processeur de lots](https://github.com/open-telemetry/opentelemetry-collector/tree/main/processor/batchprocessor) contrôle la taille du lot.

De plus, vous devez prêter attention aux paramètres de délai d’expiration de l’exportateur. En général, requests d&apos;exportation prennent plus de temps lorsque les frais sont plus élevés et lorsque les réseaux sont plus lents (latence plus élevée, bande passante plus faible). Si votre application génère des charges importantes parce que le volume de télémétrie est élevé ou que l&apos;intervalle d&apos;exportation est élevé, vous devrez peut-être augmenter les paramètres de délai d&apos;expiration par défaut pour éviter les erreurs d&apos;exportation.

Le mécanisme de configuration du délai d&apos;expiration varie, mais les SDK de langage OpenTelemetry prennent généralement en charge la définition de la variable d&apos;environnement `OTEL_EXPORTER_OTLP_TIMEOUT` (consultez [la documentation OpenTelemetry](https://opentelemetry.io/docs/specs/otel/protocol/exporter/) pour plus d&amp;apos;informations).

De plus, vous devez activer la compression pour réduire la taille de la charge utile et limiter la probabilité de rencontrer des limites de taille de charge utile. New Relic prend en charge les compressions `gzip` et `zstd` . La compression `zstd` offre des performances supérieures et est recommandée si votre exportateur la prend en charge. Voir [la comparaison de compression](https://github.com/open-telemetry/opentelemetry-collector/blob/main/config/configgrpc/README.md#compression-comparison) pour plus de détails sur les informations d&amp;apos;évaluation.

Le mécanisme de configuration du point de terminaison varie, mais les SDK de langage OpenTelemetry prennent généralement en charge la définition de la variable d&apos;environnement `OTEL_EXPORTER_OTLP_COMPRESSION=gzip` (consultez [la documentation OpenTelemetry](https://opentelemetry.io/docs/specs/otel/protocol/exporter/) pour plus d&amp;apos;informations).

Si vous utilisez le Collector, `gzip` est la compression par défaut, mais vous pouvez éventuellement configurer `zstd`.

## Configuration : Réessayer [#retry]

Niveau requis : **Recommandé**

Afin d&apos;envoyer des données OTLP à New Relic, vous devez configurer votre exportateur OTLP pour réessayer lorsque des erreurs transitoires se produisent. Internet n’est pas fiable et l’échec d’une nouvelle tentative augmente le risque de perte de données.

Le mécanisme de configuration de la nouvelle tentative varie. Certains SDK OpenTelemetry peuvent avoir des variables d&apos;environnement spécifiques à la langue (par exemple, [Java prend en charge](https://github.com/open-telemetry/opentelemetry-java/tree/main/sdk-extensions/autoconfigure) le paramètre `OTEL_EXPERIMENTAL_EXPORTER_OTLP_RETRY_ENABLED=true`), mais il n&amp;apos;existe pas de mécanisme général. Une configuration programmatique peut être requise.

Si vous utilisez le Collector, `otlphttpexporter` et `otlpexporter` réessayent par défaut. Voir [`exporterhelper`](https://github.com/open-telemetry/opentelemetry-collector/blob/main/exporter/exporterhelper/README.md) pour plus de détails.

## Config : temporalité d&apos;agrégation métrique [#metric-aggregation-temporality]

Niveau requis : **Recommandé**

Afin d&apos;envoyer des données métriques OTLP à New Relic, vous devez configurer votre exportateur de métriques OTLP pour préférer [la temporalité d&apos;agrégation](https://opentelemetry.io/docs/specs/otel/metrics/data-model/#temporality) delta. Bien que New Relic prenne en charge la temporalité d&amp;apos;agrégation cumulative, l&amp;apos;architecture des métriques New Relic est généralement un système de métriques delta. L&amp;apos;utilisation du paramètre cumulatif par défaut entraînera généralement une utilisation plus importante de la mémoire des SDK et entraînera une ingestion élevée de données.

Le mécanisme de configuration du point de terminaison varie, mais les SDK de langage OpenTelemetry prennent généralement en charge la définition de la variable d&apos;environnement `OTEL_EXPORTER_OTLP_METRICS_TEMPORALITY_PREFERENCE=delta` (consultez [la documentation OpenTelemetry](https://opentelemetry.io/docs/specs/otel/metrics/sdk_exporters/otlp/) pour plus d&amp;apos;informations). Si vous définissez manuellement la temporalité, configurez par type d&amp;apos;instrument comme suit :

* Compteur asynchrone Compteur histogramme : Delta
* Compteur asynchrone, compteur asynchrone, jauge, jauge asynchrone : cumulatif

La temporalité cumulative est utilisée pour les instruments qui correspondent aux [types de jauge New Relic](/docs/data-apis/understand-data/metric-data/metric-data-type) et qui sont généralement analysés à l&amp;apos;aide de la valeur cumulative.

## Config : agrégation métrique de histogrammes [#metric-histogram-aggregation]

Niveau requis : **Recommandé**

Afin d&apos;envoyer des données de métriques OTLP à New Relic, vous devez configurer votre exportateur de métriques OTLP pour agréger les mesures des instruments d&apos;histogramme en [histogrammes exponentiels](https://opentelemetry.io/docs/specs/otel/metrics/data-model/#exponentialhistogram). Contrairement aux compartiments statiques utilisés avec l&amp;apos;histogramme de compartiment explicite par défaut, l&amp;apos;histogramme exponentiel ajuste automatiquement ses compartiments pour refléter la plage de mesures enregistrées. De plus, ils utilisent une représentation hautement compressée pour envoyer via le réseau. L&amp;apos;histogramme exponentiel fournit des données de distribution plus utiles sur la plateforme New Relic.

Le mécanisme de configuration du point de terminaison varie, mais les SDK de langage OpenTelemetry prennent généralement en charge la définition de la variable d&apos;environnement `OTEL_EXPORTER_OTLP_METRICS_DEFAULT_HISTOGRAM_AGGREGATION=base2_exponential_bucket_histogram` (consultez [la documentation OpenTelemetry](https://opentelemetry.io/docs/specs/otel/metrics/sdk_exporters/otlp/) pour plus d&amp;apos;informations).

## Version du protocole OTLP [#otlp-version-support]

New Relic utilise la sortie OTLP [v1.4.0](https://github.com/open-telemetry/opentelemetry-proto/releases/tag/v1.4.0). Les versions ultérieures et antérieures sont prises en charge, mais les nouvelles fonctionnalités ne sont pas encore implémentées. Les fonctionnalités expérimentales qui ont été supprimées après la version 0.18.0 ne sont pas prises en charge.

Consultez [trace](/docs/opentelemetry/best-practices/opentelemetry-best-practices-traces/), [Métriques](/docs/opentelemetry/best-practices/opentelemetry-best-practices-metrics/) et [log](/docs/opentelemetry/best-practices/opentelemetry-best-practices-logs/) pour obtenir des détails spécifiques sur la manière dont les données sont mappées et sur les fonctionnalités implémentées.

## Types d&apos;attributs OTLP [#otlp-attribute-types]

Les attributs sont un concept récurrent dans OpenTelemetry et OTLP. OpenTelemetry dispose d&apos;une définition [d&apos;attribut standard](https://opentelemetry.io/docs/specs/otel/common/#standard-attribute) , indiquant que les valeurs d&amp;apos;attribut sont des primitives (chaîne, booléen, double virgule flottante, entier 64 bits) ou un éventail homogène de primitives. Cependant, au niveau du protocole OTLP, les attributs sont représentés à l&amp;apos;aide d&amp;apos;une définition [`AnyValue`](https://github.com/open-telemetry/opentelemetry-proto/blob/4f69356d853029975649c3f38b06fc77d77975fc/opentelemetry/proto/common/v1/common.proto#L28-L40) plus étendue. De ce fait, il est possible pour les clients OTLP d&amp;apos;envoyer des attributs qui ne sont pas conformes à la définition standard OpenTelemetry .

Le point de terminaison New Relic OTLP prend en charge la définition d’attribut standard. Les types complexes tels que les cartes de cartes, les éventails d&apos;objets et les éventails hétérogènes ne sont pas pris en charge. Les SDK OpenTelemetry ne doivent produire que des données conformes à la définition d&apos;attribut standard.

<Callout variant="important">
  Bien que la définition d&apos;attribut standard soit généralement utilisée, les attributs d&apos;enregistrement de log sont exceptionnels et prennent en charge des valeurs complexes (par exemple, le type d&apos;attribut d&apos;enregistrement log est [`map<string, any>`](https://opentelemetry.io/docs/specs/otel/logs/data-model/#field-attributes)). Malgré cela, New Relic ne prend actuellement en charge que les attributs d&amp;apos;enregistrement de log conformes à la définition d&amp;apos;attribut standard.
</Callout>

## Frais de réponse OTLP [#payloads]

Veuillez noter les détails suivants concernant les frais de réponse au point de terminaison OTLP de New Relic :

* Les réponses réussies de New Relic n&apos;ont pas de corps de réponse, au lieu d&apos;une [réponse codée Protobuf](https://github.com/open-telemetry/opentelemetry-specification/blob/v1.8.0/specification/protocol/otlp.md#success) basée sur le type de données.
* New Relic répond après validation de l&apos;authentification, de la taille de la charge utile et de la limitation du débit. La validation du contenu de la charge utile se produit de manière asynchrone. Par conséquent, New Relic peut renvoyer des codes d&apos;état de réussite même si l&apos;ingestion des données échoue finalement et entraîne l&apos;événement [`NrIntegrationError`](/docs/data-apis/manage-data/nrintegrationerror/) .
* [Les réponses d’échec](https://github.com/open-telemetry/opentelemetry-specification/blob/v1.8.0/specification/protocol/otlp.md#failures) de New Relic n’incluent pas `Status.message` ou `Status.details`.
