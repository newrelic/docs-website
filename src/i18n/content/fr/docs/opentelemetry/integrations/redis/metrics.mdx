## Référence complète des métriques

L&apos;intégration collecte plus de 30 métriques d&apos;instance à travers les commandes, la mémoire, l&apos;espace clé, les connexions et la réplication — plus 9 métriques spécifiques au cluster pour les déploiements Redis Cluster. Développez les sections ci-dessous pour des spécifications de métriques détaillées.

<Callout variant="tip">
  Pour le catalogue complet en amont, consultez la [documentation redisreceiver](https://github.com/open-telemetry/opentelemetry-collector-contrib/blob/main/receiver/redisreceiver/documentation.md) d&apos;OpenTelemetry.
</Callout>

## Métriques Cluster

Ces métriques ne sont disponibles que lors du monitoring de Redis en mode Cluster. Elles nécessitent l&apos;activation des métriques de cluster dans le Récepteur Redis (toutes les métriques de cluster sont désactivées par défaut) ou l&apos;utilisation de l&apos;approche du Récepteur Prometheus avec `redis_exporter --is-cluster`.

<table>
  <thead>
    <tr>
      <th style={{ width: "260px" }}>
        métrique
      </th>

      <th>
        Description
      </th>

      <th style={{ width: "100px" }}>
        Type
      </th>
    </tr>
  </thead>

  <tbody>
    <tr>
      <td>
        `redis.cluster.state`
      </td>

      <td>
        État de santé global du cluster. 1 = OK (tous les slots servis), 0 = FAIL (un ou plusieurs slots indisponibles).
      </td>

      <td>
        Jauge
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.slots_assigned`
      </td>

      <td>
        Nombre de slots de hachage mappés aux nœuds. Doit être de 16 384 pour un cluster pleinement opérationnel.
      </td>

      <td>
        Jauge
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.slots_ok`
      </td>

      <td>
        Nombre d&apos;emplacements servant activement des requests sans problème.
      </td>

      <td>
        Jauge
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.slots_pfail`
      </td>

      <td>
        Nombre de slots à l’état PFAIL (défaillance probable) — un nœud est suspecté d’être injoignable mais ce n’est pas encore confirmé.
      </td>

      <td>
        Jauge
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.slots_fail`
      </td>

      <td>
        Nombre d&apos;emplacements dans un état FAIL confirmé — NE servant PAS de trafic.
      </td>

      <td>
        Jauge
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.known_nodes`
      </td>

      <td>
        Nombre total de nœuds (maîtres + réplicas) visibles dans le cluster.
      </td>

      <td>
        Jauge
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.node.count`
      </td>

      <td>
        Nombre de nœuds maîtres servant actuellement des slots de hachage.
      </td>

      <td>
        Jauge
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.stats_messages_sent`
      </td>

      <td>
        Total des messages de bus de cluster envoyés (ping, pong, meet, fail). Un trafic régulier indique un protocole de gossip sain.
      </td>

      <td>
        Cumulatif
      </td>
    </tr>

    <tr>
      <td>
        `redis.cluster.stats_messages_received`
      </td>

      <td>
        Nombre total de messages du bus de cluster reçus des nœuds pairs. Une baisse soudaine indique une possible partition du réseau.
      </td>

      <td>
        Cumulatif
      </td>
    </tr>
  </tbody>
</table>

<Callout variant="tip">
  Tous les nœuds d&apos;un Cluster Redis signalent des métriques de cluster identiques (`CLUSTER INFO` renvoie les mêmes données à partir de n&apos;importe quel nœud). Vous pouvez activer le monitoring de cluster sur tous les nœuds pour des raisons de résilience ou sur un seul nœud pour minimiser les données redondantes. Pour l&apos;approche du Récepteur Prometheus, utilisez `redis_exporter --is-cluster` qui collecte automatiquement ces métriques.
</Callout>
