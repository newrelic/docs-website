Monitorez RabbitMQ s&apos;exécutant dans des clusters Kubernetes à l&apos;aide du [collecteur NRDOT](https://github.com/newrelic/nrdot-collector-releases/releases/latest) (recommandé) ou du [collecteur OpenTelemetry](https://github.com/open-telemetry/opentelemetry-collector-contrib) avec découverte automatique des pods. Ce guide vous accompagne dans le déploiement du collecteur via Helm avec la détection dynamique des brokers assurée par l&apos;observateur Kubernetes.

<Callout variant="tip">
  Vous exécutez RabbitMQ sur des hôtes Linux ? Consultez le [guide d&apos;installation auto-hébergée](/docs/opentelemetry/integrations/rabbitmq/self-hosted) pour les déploiements sur machine virtuelle ou bare metal.
</Callout>

## Fonctionnement de l&apos;intégration [#architecture]

Ce déploiement natif Kubernetes fournit une découverte et un monitoring automatiques :

**Composants clés :**

* **Observateur Kubernetes**: surveille en continu les pods RabbitMQ basés sur des sélecteurs de labels

* **Créateur de récepteurs**: Crée dynamiquement des récepteurs RabbitMQ pour chaque pod découvert

* **Attribution des ressources**: enrichit automatiquement les métriques avec les métadonnées Kubernetes :

  * `k8s.cluster.name` - Votre identifiant de cluster
  * `k8s.namespace.name` - Espace de noms du pod
  * `k8s.pod.name` - Nom du pod individuel
  * `rabbitmq.deployment.name` - Dérivé du nom du pod

**Modèle de déploiement :** Le collecteur s&apos;exécute en tant que déploiement Kubernetes (réplica unique) avec des permissions RBAC pour lister et surveiller les pods dans les espaces de noms. Lorsque de nouveaux pods RabbitMQ sont créés ou que des pods existants sont supprimés, l&apos;observateur met automatiquement à jour la configuration du collecteur.

<Callout variant="important">
  Le collecteur nécessite les permissions `get`, `list` et `watch` sur les pods pour découvrir automatiquement les instances RabbitMQ.
</Callout>

          2. Mettez à jour le [`deployment-configmap.yaml`](https://github.com/newrelic/helm-charts/blob/master/charts/nr-k8s-otel-collector/examples/k8s/rendered/deployment-configmap.yaml) pour le monitoring de RabbitMQ.

             ```yaml
                           enabled: true

                           enabled: true

                           enabled: true

                           enabled: true

                           enabled: true
                         rabbitmq.node.mem_alarm:
                           enabled: true

                           enabled: true

                           enabled: true

                           enabled: true
                         rabbitmq.node.sockets_total:
                           enabled: true
                         rabbitmq.node.proc_used:
                           enabled: true
                         rabbitmq.node.proc_total:
                           enabled: true
                         rabbitmq.node.uptime:
                           enabled: true

                       rabbitmq.server.endpoint: 'http://`endpoint`:15672'
                       rabbitmq.port: '15672'

             processors:
               batch:
                 send_batch_max_size: 1000
                 timeout: 30s
                 send_batch_size: 800

               memory_limiter:
                 check_interval: 1s
                 limit_percentage: 80
                 spike_limit_percentage: 25

               resource/cluster:
                 attributes:
                   - key: k8s.cluster.name

                     action: insert

               transform/rabbitmq:

                   - context: resource
             ```

        <Step>
          ### Déployer avec Helm [#deploy-helm]

          Ajouter le dépôt Helm OpenTelemetry :

          ```bash
          helm repo add open-telemetry https://open-telemetry.github.io/opentelemetry-helm-charts
          helm repo update
          ```

          Installer ou mettre à jour le collecteur :

          ```bash
          helm upgrade --install rabbitmq-otel-collector \
            open-telemetry/opentelemetry-collector \
            --namespace newrelic \
            --create-namespace \
            --values otel-collector-values.yaml
          ```

          Cela crée :

          * Un déploiement nommé `rabbitmq-otel-collector` dans l&apos;espace de noms `newrelic`
          * Un compte de service avec des permissions RBAC pour surveiller les pods
          * ConfigMaps contenant la configuration du collecteur
        </Step>

        <Step>
          ### Vérifier le déploiement et le flux de données [#verify]

          Vérifiez que le pod du collecteur est en cours d&apos;exécution :

          ```bash
          kubectl get pods -n newrelic -l app.kubernetes.io/name=opentelemetry-collector
          ```

          Résultat attendu :

          ```
          NAME                                 READY   STATUS    RESTARTS   AGE
          rabbitmq-otel-collector-6d8c5c5d8d-abc12   1/1     Running   0          2m
          ```

          Consultez les logs du collecteur pour vérifier la découverte des pods :

          ```bash
          kubectl logs deploy/rabbitmq-otel-collector -n newrelic --tail=50
          ```

          Recherchez des messages indiquant la réussite de la découverte des pods et de la collecte de métriques :

          ```
          INFO    RabbitmqReceiver                Successfully scraped rabbitmq metrics from pod rabbitmq-0
          ```

          Vérifiez vos données dans New Relic :

          Attendez 2 à 3 minutes que les données apparaissent, puis exécutez cette requête dans le [générateur de requêtes](/docs/query-your-data/explore-query-data/query-builder/introduction-query-builder):

          ```sql
          SELECT count(*)
          FROM Metric
          WHERE metricName LIKE 'rabbitmq.%'
            AND instrumentation.provider = 'opentelemetry'
            AND k8s.cluster.name = 'my-rabbitmq-cluster'
          FACET k8s.pod.name, metricName
          SINCE 10 minutes ago
          ```

          Vous devriez voir des métriques de chaque pod RabbitMQ avec des attributs Kubernetes :

          * `k8s.cluster.name` - Votre identifiant de cluster
          * `k8s.namespace.name` - Espace de noms du pod
          * `k8s.pod.name` - Nom du pod individuel
          * `rabbitmq.deployment.name` - Dérivé du nom du pod

## Dépannage [#troubleshoot]

<CollapserGroup>
  <Collapser id="no-data" title="Aucune donnée n'apparaît dans New Relic">
    Si les métriques n&apos;apparaissent pas dans New Relic après le déploiement :

    **1. Vérifier que le plugin de gestion RabbitMQ est activé :**

    ```bash
    kubectl exec <rabbitmq-pod> -n <namespace> -- rabbitmq-plugins list | grep management
    ```

    Vous devriez voir `[E*] rabbitmq_management` indiquant qu&apos;il est activé.

    **2. Vérifier les logs du pod du collecteur :**

    ```bash
    kubectl logs deploy/rabbitmq-otel-collector -n newrelic --tail=100
    ```

    Rechercher :

    * Messages de découverte de pods
    * Erreurs d&apos;authentification (identifiants incorrects)
    * Erreurs de connexion (problèmes réseau)
    * Erreurs d&apos;analyse (problèmes de configuration)

    **3. Vérifier que le service RabbitMQ est accessible :**

    ```bash
    kubectl get svc -n <rabbitmq-namespace>
    ```

    Assurez-vous qu&apos;un service expose le port 15672.

    **4. Tester l&apos;API de gestion depuis le cluster :**

    ```bash
    kubectl run test-pod --image=curlimages/curl:latest --rm -it --restart=Never -- \
      curl -I -u admin:password http://<rabbitmq-service>.<namespace>.svc.cluster.local:15672/api/overview
    ```

    Devrait renvoyer `HTTP/1.1 200 OK`.

    **5. Vérifier les permissions RBAC :**

    ```bash
    kubectl auth can-i list pods --as=system:serviceaccount:newrelic:rabbitmq-otel-collector -n <rabbitmq-namespace>
    ```

    Doit renvoyer `yes`. Sinon, vérifiez le ClusterRole et le ClusterRoleBinding.
  </Collapser>

  <Collapser id="pod-discovery" title="La découverte des pods ne fonctionne pas">
    Si le collecteur ne découvre pas automatiquement les pods RabbitMQ :

    **1. Vérifiez que le sélecteur de label correspond à vos pods :**

    ```bash
    kubectl get pods --show-labels -n <rabbitmq-namespace>
    ```

    Comparez les étiquettes avec le `rule` dans votre fichier de valeurs Helm.

    **2. Vérifier les logs de k8s\_observer :**

    ```bash
    kubectl logs deploy/rabbitmq-otel-collector -n newrelic | grep k8sobserver
    ```

    Recherchez des messages concernant les pods découverts.

    **3. Tester le sélecteur d&apos;étiquettes :**

    ```bash
    kubectl get pods -l app=rabbitmq --all-namespaces
    ```

    Cela devrait lister vos pods RabbitMQ. S&apos;il est vide, votre sélecteur d&apos;étiquettes est incorrect.

    **4. Vérifier les permissions du ClusterRole :**

    ```bash
    kubectl describe clusterrole rabbitmq-otel-collector
    ```

    Assurez-vous qu&apos;il dispose des permissions `get`, `list` et `watch` sur les pods.

    **Motifs d&apos;étiquettes courants :**

    * Déploiements standards : `app=rabbitmq`
    * StatefulSets : `app.kubernetes.io/name=rabbitmq`
    * Opérateurs : `app.kubernetes.io/component=rabbitmq`
  </Collapser>

  <Collapser id="connection-errors" title="Erreurs de connexion">
    Si vous constatez des erreurs de connexion dans les logs du collecteur :

    **Vérifier les points de terminaison du service :**

    ```bash
    kubectl get endpoints -n <rabbitmq-namespace>
    ```

    Vérifiez que des points de terminaison existent pour votre service RabbitMQ.

    **Vérifiez les stratégies réseau :** Assurez-vous que les stratégies réseau autorisent le trafic depuis l&apos;espace de noms `newrelic` vers votre espace de noms RabbitMQ.

    **Tester la connectivité :**

    ```bash
    kubectl run test-pod --image=curlimages/curl:latest --rm -it --restart=Never -n newrelic -- \
      curl http://<rabbitmq-service>.<namespace>.svc.cluster.local:15672/api/overview
    ```

    **Vérifier la résolution DNS :**

    ```bash
    kubectl run test-pod --image=busybox:latest --rm -it --restart=Never -n newrelic -- \
      nslookup <rabbitmq-service>.<namespace>.svc.cluster.local
    ```
  </Collapser>

  <Collapser id="high-cardinality" title="Avertissements de haute cardinalité">
    Si vous voyez des avertissements de cardinalité élevée dans New Relic :

    **Filtrer sur des files d&apos;attente spécifiques :** Si vous monitorez de nombreuses files d&apos;attente, envisagez de filtrer uniquement les critiques.

    **Augmenter l&apos;intervalle de collecte :**

    ```yaml
    receivers:
      receiver_creator/rabbitmq:
        receivers:
          rabbitmq:
            config:
              collection_interval: 60s # Increased from 30s
    ```

    **Utilisez l&apos;agrégation de métriques :** Configurez le processeur de lots pour agréger les métriques avant l&apos;exportation.

    **Désactiver les métriques moins critiques :** Commentez les métriques dont vous n&apos;avez pas besoin dans la configuration du récepteur.

    **Conseils d&apos;optimisation supplémentaires :**

    * Utilisez des sélecteurs de labels plus spécifiques pour monitorer uniquement les pods de production
    * Filtrer des files d&apos;attente spécifiques par nom au lieu de monitorer toutes les files d&apos;attente
    * Augmentez le `collection_interval` pour réduire les points de données
    * Envisagez d&apos;utiliser l&apos;agrégation de métriques dans le collecteur
  </Collapser>

  <Collapser id="collector-crashes" title="Pod du Collecteur qui plante ou redémarre">
    Si le pod du collecteur plante :

    **Vérifier les limites de ressources :**

    ```bash
    kubectl describe pod -n newrelic -l app.kubernetes.io/name=opentelemetry-collector
    ```

    Recherchez le statut OOMKilled. Si c&apos;est le cas, augmentez les limites de mémoire :

    ```yaml
    resources:
      limits:
        memory: 512Mi # Increased from 300Mi
    ```

    **Examiner les logs de plantage :**

    ```bash
    kubectl logs -n newrelic -l app.kubernetes.io/name=opentelemetry-collector --previous
    ```

    **Vérifiez la syntaxe de la configuration :** Assurez-vous que votre configuration YAML est valide et correctement indentée.
  </Collapser>

  <Collapser id="auth-failures" title="Échecs d'authentification">
    Si vous voyez `401 Unauthorized` erreurs :

    **Vérifiez que les identifiants dans les valeurs Helm correspondent à votre configuration RabbitMQ :**

    ```bash
    kubectl exec <rabbitmq-pod> -n <namespace> -- rabbitmqctl list_users
    ```

    **Vérifiez que l&apos;utilisateur RabbitMQ dispose des privilèges d&apos;administrateur** et assurez-vous que les identifiants ne contiennent pas de caractères spéciaux nécessitant un échappement.
  </Collapser>
</CollapserGroup>
