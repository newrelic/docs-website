Le monitoring de RabbitMQ offre une visibilité en temps réel sur votre infrastructure de courtiers de messages pour garantir un traitement fiable des messages et éviter des temps d&apos;arrêt coûteux dans les systèmes distribués. En utilisant le [NRDOT Collector](https://github.com/newrelic/nrdot-collector-releases/releases/latest) (recommandé) ou l&apos;[OpenTelemetry Collector Contrib](https://github.com/open-telemetry/opentelemetry-collector-contrib), vous bénéficiez d&apos;une supervision complète grâce à une approche flexible et neutre vis-à-vis des fournisseurs qui fonctionne dans les environnements auto-hébergés et Kubernetes.

## Pourquoi le monitoring de RabbitMQ ?

* Évitez les pannes : recevez des alertes sur les accumulations dans les files d&apos;attente, les défaillances de consommateurs et les alarmes mémoire/disque avant qu&apos;elles ne causent des temps d&apos;arrêt
* Optimiser les performances - Identifier les consommateurs lents, les goulots d&apos;étranglement de messages et les contraintes de ressources qui affectent la vitesse de traitement
* Planifier la capacité - Suivre l&apos;utilisation des ressources, les débits de messages et le nombre de connexions pour évoluer de manière proactive
* Assurez la remise des messages - Monitorez la profondeur des files d&apos;attente, les taux d&apos;acquittement et la persistance des messages pour éviter la perte de données

### Cas d&apos;utilisation courant

Que vous traitiez des messages de traitement de commandes, gériez la communication entre microservices orientée événements ou administriez des files d&apos;attente de tâches en arrière-plan, le monitoring RabbitMQ vous aide à détecter les problèmes avant qu&apos;ils n&apos;impactent votre activité. Soyez alerté lorsque des pics de profondeur de file d&apos;attente menacent les SLA de traitement des messages, lorsque des défaillances de consommateurs risquent d&apos;entraîner une accumulation de messages, ou lorsque des alarmes de mémoire menacent la stabilité du broker. Ce monitoring est essentiel pour les plateformes d&apos;e-commerce, les systèmes financiers et toute application où les retards ou les échecs de livraison de messages peuvent affecter l&apos;expérience utilisateur ou les opérations commerciales.

## Comment ça marche [#how-it-works]

Le NRDOT Collector (recommandé) ou l&apos;OpenTelemetry Collector Contrib se connecte à votre API de gestion RabbitMQ pour collecter des métriques complètes :

**Collecte de données :**

* **[Le récepteur RabbitMQ](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/receiver/rabbitmqreceiver)** se connecte à l&apos;API de gestion RabbitMQ (port 15672) pour les métriques de file d&apos;attente, la santé des nœuds, l&apos;activité des consommateurs et l&apos;état de la connexion

**Flux de données :**

1. Le Collecteur collecte des métriques depuis l&apos;API de gestion à intervalles réguliers.
2. Les données sont traitées, enrichies avec les informations de l&apos;hôte et mises en lots pour une transmission efficace.
3. Les métriques sont exportées vers New Relic via l&apos;exportateur OTLP
4. New Relic crée automatiquement des entités et remplit les dashboards

**Ce que vous obtenez :** Les métriques clés incluent la profondeur des files d&apos;attente, les taux de messages, le nombre de consommateurs, l&apos;utilisation des ressources des nœuds, les alarmes mémoire/disque et l&apos;activité des connexions.

Pour obtenir les noms complets des métriques, les descriptions et les recommandations d&apos;alerte, consultez la [référence des métriques RabbitMQ](/docs/opentelemetry/integrations/rabbitmq/metrics-reference).

### Facultatif : Ajouter la collecte de logs

La configuration du monitoring ci-dessus suit la santé et les performances de votre broker RabbitMQ. Pour obtenir une vue d&apos;ensemble de ce qui se passe dans votre courtier de messages, vous pouvez également collecter et transmettre les logs RabbitMQ.

**La collecte de logs ajoute :**

* Messages d&apos;erreur et avertissements des processus RabbitMQ
* Événements de connexion et d&apos;authentification
* Opérations sur les files d&apos;attente et les échangeurs
* Visibilité complète de l&apos;ingestion des messages à la livraison

**Configuration :** La collecte de logs RabbitMQ est disponible pour les déploiements auto-hébergés. Consultez le [guide d&apos;installation auto-hébergée](/docs/opentelemetry/integrations/rabbitmq/self-hosted) pour les détails de configuration.
