Monitorez RabbitMQ sur des hôtes Linux à l&apos;aide du [collecteur NRDOT](https://github.com/newrelic/nrdot-collector-releases/releases/latest) (recommandé) ou du [collecteur OpenTelemetry](https://github.com/open-telemetry/opentelemetry-collector-contrib) pour envoyer des métriques et des données de télémétrie à New Relic. Ce guide complet vous accompagne dans l&apos;installation, la configuration et la vérification des systèmes Debian, Ubuntu, RHEL et CentOS.

## Étapes d&apos;installation [#install]

Suivez ces étapes pour configurer le monitoring de votre instance RabbitMQ auto-hébergée.

        <Step>
          ### Avant de commencer [#requirements-nrdot]

          Assurez-vous que votre environnement respecte ces prérequis :

          <CollapserGroup>
            <Collapser id="rabbitmq-reqs" title="Prérequis RabbitMQ">
              * **Version**: RabbitMQ 3.8 ou version ultérieure
              * **Plugin de gestion**: Doit être activé pour exposer l&apos;API de métriques
              * **Compte utilisateur**: Utilisateur administrateur avec les permissions pour accéder à l&apos;API de gestion
            </Collapser>

            <Collapser id="system-reqs" title="exigences système">
              * **Système d&apos;exploitation**: Debian, Ubuntu, RHEL ou CentOS
              * **Accès shell**: Utilisateur avec privilèges sudo
              * **Réseau**: Connectivité HTTPS sortante vers le [point de terminaison OTLP](https://docs.newrelic.com/docs/opentelemetry/best-practices/opentelemetry-otlp/#configure-endpoint-port-protocol)de New Relic
              * **Ressources**: Minimum 512 Mo de RAM et 1 cœur de CPU disponibles pour le collecteur
            </Collapser>

            <Collapser id="newrelic-reqs" title="Exigences New Relic">
              * **Compte**: Compte New Relic actif
              * **Clé de licence**: [Clé de licence New Relic](/docs/apis/intro-apis/new-relic-api-keys/#license-key) pour votre compte
            </Collapser>

            <Collapser id="software-reqs" title="Prérequis logiciels">
              * **Collecteur NRDOT**: [Dernière version](https://github.com/newrelic/nrdot-collector-releases/releases/latest) (distribution OpenTelemetry de New Relic)
              * **Paquets système**: `wget`, `systemd` (généralement préinstallés)
            </Collapser>
          </CollapserGroup>

Consultez la [section d&apos;installation](https://github.com/newrelic/nrdot-collector-releases/blob/main/distributions/README.md#installation) du dépôt nrdot-collector-releases.

        <Step>
          ### Activer le plugin de gestion RabbitMQ [#enable-plugin]

          Le plugin de gestion expose l&apos;API que l&apos;OpenTelemetry Collector scrape :

          ```bash
          sudo rabbitmq-plugins enable rabbitmq_management
          ```

          Vérifier que cela fonctionne :

          ```bash
          curl -I -u admin:password http://localhost:15672/api/overview
          ```

          Vous devriez voir `HTTP/1.1 200 OK` dans la réponse.
        </Step>

        <Step>
          ### Installer le collecteur NRDOT [#install-nrdot-collector]

          Téléchargez et installez le collecteur NRDOT pour votre système d&apos;exploitation hôte à partir des [versions du collecteur NRDOT](https://github.com/newrelic/nrdot-collector-releases/releases/latest). Consultez le [guide d&apos;installation](https://github.com/newrelic/nrdot-collector-releases/blob/main/distributions/README.md#installation) pour obtenir des instructions spécifiques à la distribution.

          Remarque : Lors de l&apos;installation, utilisez `export collector_distro="nrdot-collector"`
        </Step>

        <Step>
          ### Configurer le collecteur [#configure-nrdot]

          Créez votre fichier de configuration du collecteur à `/etc/nrdot-collector/rabbitmq-config.yaml`:

          ```yaml
          receivers:
            rabbitmq:
              endpoint: http://localhost:15672
              username: admin
              password: password
              collection_interval: 30s
              metrics:
                # Queue Metrics (essential for message flow and backlog)
                rabbitmq.consumer.count:
                  enabled: true
                rabbitmq.message.delivered:
                  enabled: true
                rabbitmq.message.published:
                  enabled: true
                rabbitmq.message.acknowledged:
                  enabled: true
                rabbitmq.message.dropped:
                  enabled: true
                rabbitmq.message.current:
                  enabled: true # Crucial for monitoring queue backlog, includes 'ready' and 'unacknowledged' states

                # Node Health Metrics (critical for server resource monitoring)
                rabbitmq.node.disk_free:
                  enabled: true
                rabbitmq.node.disk_free_limit:
                  enabled: true
                rabbitmq.node.disk_free_alarm:
                  enabled: true
                rabbitmq.node.mem_used:
                  enabled: true
                rabbitmq.node.mem_limit:
                  enabled: true
                rabbitmq.node.mem_alarm:
                  enabled: true
                rabbitmq.node.mem_used_details.rate:
                  enabled: true
                rabbitmq.node.fd_used:
                  enabled: true
                rabbitmq.node.fd_total:
                  enabled: true
                rabbitmq.node.sockets_used:
                  enabled: true
                rabbitmq.node.sockets_total:
                  enabled: true
                rabbitmq.node.proc_used:
                  enabled: true
                rabbitmq.node.proc_total:
                  enabled: true
                rabbitmq.node.uptime:
                  enabled: true
                rabbitmq.node.run_queue:
                  enabled: true
                rabbitmq.node.processors:
                  enabled: true
                rabbitmq.node.context_switches_details.rate:
                  enabled: true
                rabbitmq.node.gc_num_details.rate:
                  enabled: true
                rabbitmq.node.gc_bytes_reclaimed_details.rate:
                  enabled: true

                # I/O Metrics (important for understanding disk and network activity)
                rabbitmq.node.io_read_count_details.rate:
                  enabled: true
                rabbitmq.node.io_read_bytes_details.rate:
                  enabled: true
                rabbitmq.node.io_read_avg_time_details.rate:
                  enabled: true
                rabbitmq.node.io_write_count_details.rate:
                  enabled: true
                rabbitmq.node.io_write_bytes_details.rate:
                  enabled: true
                rabbitmq.node.io_write_avg_time_details.rate:
                  enabled: true
                rabbitmq.node.io_sync_count_details.rate:
                  enabled: true
                rabbitmq.node.io_sync_avg_time_details.rate:
                  enabled: true
                rabbitmq.node.io_seek_count_details.rate:
                  enabled: true
                rabbitmq.node.io_seek_avg_time_details.rate:
                  enabled: true
                rabbitmq.node.io_reopen_count_details.rate:
                  enabled: true

                # Mnesia and Store Metrics (for internal database and message storage)
                rabbitmq.node.mnesia_ram_tx_count_details.rate:
                  enabled: true
                rabbitmq.node.mnesia_disk_tx_count_details.rate:
                  enabled: true
                rabbitmq.node.msg_store_read_count_details.rate:
                  enabled: true
                rabbitmq.node.msg_store_write_count_details.rate:
                  enabled: true
                rabbitmq.node.queue_index_write_count_details.rate:
                  enabled: true
                rabbitmq.node.queue_index_read_count_details.rate:
                  enabled: true

                # Connection/Channel/Queue Lifecycle Metrics
                rabbitmq.node.connection_created_details.rate:
                  enabled: true
                rabbitmq.node.connection_closed_details.rate:
                  enabled: true
                rabbitmq.node.channel_created_details.rate:
                  enabled: true
                rabbitmq.node.channel_closed_details.rate:
                  enabled: true
                rabbitmq.node.queue_declared_details.rate:
                  enabled: true
                rabbitmq.node.queue_created_details.rate:
                  enabled: true
                rabbitmq.node.queue_deleted_details.rate:
                  enabled: true

          # processors: Process data before exporting.
          processors:
            resourcedetection:
              detectors: [system]
              system:
                resource_attributes:
                  host.name:
                    enabled: true
                  host.id:
                    enabled: true
            resource:
              attributes:
                - key: instrumentation.provider
                  value: opentelemetry
                  action: upsert
                - key: rabbitmq.deployment.name
                  value: my-rabbitmq-server  # Replace with your server name
                  action: upsert
            batch:
              send_batch_size: 1024
              timeout: 30s
            transform/metadata_nullify:
              metric_statements:
                - context: metric
                  statements:
                    - set(description, "")
                    - set(unit, "")

          exporters:
            otlp_http:
              endpoint: ${env:OTEL_EXPORTER_OTLP_ENDPOINT:-https://otlp.nr-data.net}
              headers:
                api-key: ${env:NEW_RELIC_LICENSE_KEY}

          service:
            pipelines:
              metrics:
                receivers: [rabbitmq]
                processors: [resourcedetection, resource, batch, transform/metadata_nullify]
                exporters: [otlp_http]
          ```

          Mettez à jour ces valeurs dans la configuration :

          * `endpoint`: Votre URL d&apos;API de gestion RabbitMQ (par défaut : `http://localhost:15672`)
          * `username` et `password`: Vos identifiants RabbitMQ
          * `rabbitmq.deployment.name`: Un nom unique pour cette instance RabbitMQ
          * `collection_interval`: Fréquence de collecte des métriques (par défaut : 30 secondes)

        <Step>
          ### Configurer l&apos;environnement et mettre à jour le chemin de configuration [#set-environment-nrdot]

          Mettez à jour la configuration du collecteur NRDOT pour utiliser votre fichier de configuration RabbitMQ et définissez le point de terminaison OTLP.

          <Callout variant="important">
            Votre clé de licence a déjà été configurée dans `/etc/nrdot-collector/nrdot-collector.conf` lors de l&apos;installation. Il vous suffit de mettre à jour le chemin du fichier de configuration et le point de terminaison OTLP.
          </Callout>

          Déterminez votre point de terminaison OTLP en fonction de votre région New Relic. Consultez [Configurer le point de terminaison, le port et le protocole](/docs/opentelemetry/best-practices/opentelemetry-otlp/#configure-endpoint-port-protocol) pour la liste complète des points de terminaison.

          ```bash
          export collector_distro="nrdot-collector"
          export otlp_endpoint="https://otlp.nr-data.net"  # Replace with your region's endpoint

          # Update the config file path to point to rabbitmq-config.yaml
          sudo sed -i 's|OTELCOL_OPTIONS="--config=/etc/nrdot-collector/config.yaml"|OTELCOL_OPTIONS="--config=/etc/nrdot-collector/rabbitmq-config.yaml"|' /etc/${collector_distro}/${collector_distro}.conf

          # Add the OTLP endpoint
          echo "OTEL_EXPORTER_OTLP_ENDPOINT=${otlp_endpoint}" | sudo tee -a /etc/${collector_distro}/${collector_distro}.conf > /dev/null
          ```

          Rechargez systemd et redémarrez le collecteur :

          ```bash
          sudo systemctl daemon-reload
          sudo systemctl restart nrdot-collector.service
          ```
        </Step>

        <Step>
          ### Vérifiez vos données dans New Relic [#verify-nrdot]

          Vérifiez l&apos;état du service :

          ```bash
          sudo systemctl status nrdot-collector.service
          ```

          Vous devriez voir `active (running)` dans la sortie.

          Voir les logs récents :

          ```bash
          sudo journalctl -u nrdot-collector.service -n 50 --no-pager
          ```

          Recherchez des messages indiquant une collecte de métriques réussie :

          ```
          INFO    RabbitmqReceiver        Successfully scraped rabbitmq metrics
          ```

          Vérifier les métriques dans New Relic :

          Attendez 2 à 3 minutes que les données apparaissent, puis exécutez cette requête dans le [générateur de requêtes](/docs/query-your-data/explore-query-data/query-builder/introduction-query-builder):

          ```sql
          SELECT count(*)
          FROM Metric
          WHERE metricName LIKE 'rabbitmq.%'
            AND instrumentation.provider = 'opentelemetry'
            AND rabbitmq.deployment.name = 'my-rabbitmq-server'
          FACET metricName
          SINCE 10 minutes ago
          ```

          Vous devriez voir des noms de métriques tels que :

          * `rabbitmq.queue.count` - Nombre de files d&apos;attente
          * `rabbitmq.queue.message.count` - Total des messages dans les files d&apos;attente
          * `rabbitmq.connection.count` - Connexions actives
          * `rabbitmq.consumer.count` - Consommateurs actifs

          <Callout variant="tip">
            Si vous ne voyez pas de données après 5 minutes, consultez la [section de dépannage](#troubleshoot) ci-dessous.
          </Callout>
        </Step>

        <Step>
          ### (Facultatif) Transférer les logs RabbitMQ vers New Relic [#logs]

          En plus des métriques, vous pouvez transférer les logs d&apos;application RabbitMQ vers New Relic pour une observabilité complète. Cela permet de corréler les problèmes entre les métriques et les logs.

          <Callout variant="important">
            Le transfert de logs nécessite une configuration supplémentaire et des permissions système. Assurez-vous de disposer des accès nécessaires avant de continuer.
          </Callout>

          **Mettez à jour `/etc/nrdot-collector/rabbitmq-config.yaml`** pour ajouter le récepteur filelog :

          ```yaml
          receivers:
            rabbitmq:
              # ... existing rabbitmq config ...

            # Add filelog receiver for RabbitMQ logs
            filelog/rabbitmq:
              include:
                - /var/log/rabbitmq/*.log
                - /var/log/rabbitmq/**/*.log
              include_file_path: true
              include_file_name: false
              operators:
                - type: regex_parser
                  regex: '^(?P<timestamp>\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2}\.\d{3}) \[(?P<level>\w+)\] <(?P<pid>[^>]+)> (?P<message>.*)$'
                  timestamp:
                    parse_from: attributes.timestamp
                    layout: '%Y-%m-%d %H:%M:%S.%L'
                  severity:
                    parse_from: attributes.severity

          processors:
            # ... existing processors ...

            transform/rabbitmq_logs:
              log_statements:
                - context: resource
                  statements:
                    - set(attributes["rabbitmq.display.name"], Concat(["server", attributes["rabbitmq.deployment.name"]], ":"))

          service:
            pipelines:
              metrics:
                receivers: [rabbitmq]
                processors: [resourcedetection, resource, batch, transform/metadata_nullify]
                exporters: [otlp_http]

              logs/rabbitmq:
                receivers: [filelog]
                processors: [batch, resource, transform/rabbitmq_logs]
                exporters: [otlp_http]
          ```

          Accordez au collecteur l&apos;autorisation de lire les logs. Choisissez une méthode :

          Ajouter au groupe rabbitmq :

          ```bash
          sudo usermod -a -G rabbitmq nrdot-collector
          ```

          Ou définissez les permissions de fichier :

          ```bash
          sudo chmod 644 /var/log/rabbitmq/*.log
          ```

          Redémarrez le collecteur :

          ```bash
          sudo systemctl restart nrdot-collector.service
          ```

          Vérifiez que les logs sont collectés dans New Relic :

          ```sql
          SELECT count(*)
          FROM Log
          WHERE service.name = 'rabbitmq'
            AND rabbitmq.deployment.name = 'my-rabbitmq-server'
          SINCE 10 minutes ago
          ```
        </Step>

          ```sql
          WHERE metricName LIKE 'rabbitmq.%'
          ```

Après quelques minutes, vos métriques RabbitMQ devraient apparaître dans New Relic.
```sql
FROM Metric SELECT * WHERE rabbitmq.deployment.name = 'my-rabbitmq-server'
```

## Dépannage [#troubleshoot]

<CollapserGroup>
  <Collapser id="no-data" title="Aucune donnée n'apparaît dans New Relic">
    Si vous ne voyez pas de métriques dans New Relic après l&apos;installation :

    **1. Vérifiez que le plugin de gestion est activé :**

    ```bash
    sudo rabbitmq-plugins list | grep management
    ```

    Vous devriez voir `[E*] rabbitmq_management` indiquant qu&apos;il est activé et en cours d&apos;exécution.

    **2. Tester l&apos;API de gestion :**

    ```bash
    curl -u admin:password http://localhost:15672/api/overview
    ```

    Cela devrait renvoyer des données JSON concernant votre instance RabbitMQ. Sinon, le plugin de gestion ne fonctionne pas correctement.

    **3. Vérifier les logs du collecteur pour détecter des erreurs :**

    ```bash
    sudo journalctl -u nrdot-collector.service -f
    ```

    Rechercher :

    * Erreurs d&apos;authentification (nom d&apos;utilisateur/mot de passe incorrect)
    * Délais de connexion dépassés (problèmes réseau)
    * Erreurs d&apos;analyse (problèmes de syntaxe de configuration)

    **4. Vérifiez que le service du collecteur est en cours d&apos;exécution :**

    ```bash
    sudo systemctl status nrdot-collector.service
    ```

    Devrait afficher `active (running)`. Sinon, consultez les journaux pour savoir pourquoi le démarrage a échoué.

    **5. Vérifiez votre clé de licence :**

    ```bash
    sudo cat /etc/nrdot-collector/nrdot-collector.conf
    ```

    Assurez-vous que `NEW_RELIC_LICENSE_KEY` est correctement défini et ne contient pas d&apos;espaces ou de guillemets supplémentaires.
  </Collapser>

  <Collapser id="connection-refused" title="Erreurs de connexion refusée">
    Si vous voyez `connection refused` dans les logs du collecteur :

    **Vérifiez que RabbitMQ écoute sur le port 15672 :**

    ```bash
    sudo netstat -tlnp | grep 15672
    ```

    Vous devriez voir RabbitMQ en écoute. Sinon, le plugin de gestion n&apos;est pas activé ou ne se lie pas au port attendu.

    **Vérifier les règles de pare-feu :**

    ```bash
    sudo iptables -L -n | grep 15672
    ```

    Assurez-vous qu&apos;aucune règle de pare-feu ne bloque le port 15672.

    **Vérifiez la configuration du point de terminaison :**\
    Assurez-vous que `endpoint` dans `/etc/nrdot-collector/rabbitmq-config.yaml` correspond à votre configuration RabbitMQ (nom d&apos;hôte, port).
  </Collapser>

  <Collapser id="auth-failures" title="Échecs d'authentification (401 Non autorisé)">
    Si vous voyez `401 Unauthorized` erreurs :

    **Vérifier les identifiants :**\
    Vérifiez que le nom d&apos;utilisateur et le mot de passe dans `/etc/nrdot-collector/rabbitmq-config.yaml` correspondent à votre utilisateur RabbitMQ.

    **Assurez-vous de disposer des privilèges d&apos;administrateur :**

    ```bash
    sudo rabbitmqctl list_users
    ```

    Votre utilisateur doit posséder le tag `[administrator]`. Sinon, accordez les privilèges d&apos;administrateur :

    ```bash
    sudo rabbitmqctl set_user_tags your-username administrator
    ```
  </Collapser>

  <Collapser id="high-memory" title="Utilisation élevée de la mémoire ou du CPU">
    Si le collecteur consomme des ressources excessives :

    **Réduire la taille du lot :**\
    Dans le processeur `batch`, réduisez `send_batch_size` de 512 à 256 ou moins.

    **Augmenter l&apos;intervalle de collecte :**\
    Passez `collection_interval` de 30 s à 60 s ou plus pour scraper moins fréquemment.

    **Désactiver les métriques inutiles :**\
    Commentez les métriques dont vous n&apos;avez pas besoin dans la configuration du récepteur `rabbitmq`.

    Exemple:

    ```yaml
    receivers:
      rabbitmq:
        collection_interval: 60s  # Increased
        metrics:
          rabbitmq.queue.message.count:
            enabled: true
          # Disable less critical metrics
          # rabbitmq.queue.count:
          #   enabled: false
    ```
  </Collapser>

  <Collapser id="log-permissions" title="Permission refusée pour les logs">
    Si vous rencontrez des erreurs d&apos;autorisation lors de la collecte de logs :

    **Tester l&apos;accès au fichier :**

    ```bash
    sudo -u nrdot-collector cat /var/log/rabbitmq/rabbit@$(hostname).log
    ```

    Si cela échoue, l&apos;utilisateur du collecteur ne peut pas lire les fichiers de log.

    **Vérifier les permissions des fichiers :**

    ```bash
    ls -la /var/log/rabbitmq/
    ```

    **Corriger les permissions :**\
    Ajoutez l&apos;utilisateur collector au groupe rabbitmq :

    ```bash
    sudo usermod -a -G rabbitmq nrdot-collector
    sudo systemctl restart nrdot-collector.service
    ```
  </Collapser>
</CollapserGroup>
