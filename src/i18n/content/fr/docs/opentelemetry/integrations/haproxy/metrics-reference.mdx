Cette page fournit une référence complète de toutes les métriques HAProxy collectées via OpenTelemetry, y compris les chemins du haproxyreceiver et du récepteur Prometheus.

## Présentation [#overview]

L’intégration HAProxy OpenTelemetry collecte des métriques à partir du point de terminaison de statistiques CSV HAProxy via le [haproxyreceiver](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/receiver/haproxyreceiver). Les métriques sont organisées en deux niveaux :

* **Métriques par défaut (17)**: collectées automatiquement avec la configuration par défaut — suffisantes pour la plupart des cas d’utilisation de monitoring
* **Métriques optionnelles (16)**: désactivées par défaut, activées individuellement dans la section `metrics` du récepteur — utiles pour le débogage détaillé et la planification de la capacité

Lors de l’utilisation de l’installation guidée NRDOT, vous pouvez choisir entre deux configurations :

* **Optimisé** (ingestion plus faible) : 17 métriques par défaut, intervalle de collecte de 60 s, processeurs d&apos;optimisation appliqués
* **Complet** (détails complets) : 27 métriques (17 par défaut + 10 facultatives), intervalle de collecte de 15 s

<Callout variant="tip">
  Pour activer les métriques facultatives, ajoutez-les à la section `metrics` de votre configuration de Récepteur haproxy avec `enabled: true`. Consultez [Monitorer HAProxy auto-hébergé avec OpenTelemetry](/docs/opentelemetry/integrations/haproxy/self-hosted/) pour les détails de configuration.
</Callout>

<Callout variant="important">
  **Limites connues du chemin Prometheus :**

  * **Aucune FACET par code d&apos;état**: le haproxyreceiver analyse les champs CSV `hrsp_*` en un attribut `status_code` sur `haproxy.requests.total`. Le chemin Prometheus ne prend pas cela en charge — la répartition des codes de réponse HTTP est disponible uniquement sous forme de métriques distinctes (`haproxy_frontend_http_responses_total{code="2xx"}`) non incluses dans la configuration de transformation.
  * **Les métriques de taux sont des maximums à vie**: `haproxy.connections.rate` correspond à `haproxy_frontend_connections_rate_max` et `haproxy.requests.rate` correspond à `haproxy_frontend_http_requests_rate_max`. Ce sont des valeurs maximales à vie, pas des taux instantanés. Pour les taux actuels réels, utilisez la fonction NRQL `rate()` sur les métriques de compteur cumulatif.
  * **-136 métriques Prometheus n&apos;ont pas d&apos;équivalent OTel**: le point de terminaison Prometheus de HAProxy expose -204 métriques. Seules 36 sont mappées aux 17 métriques OTel par défaut (sur les niveaux frontend/backend/serveur). Le reste (éléments internes du processus, détails des vérifications d&apos;état, statistiques du résolveur DNS) n&apos;a pas d&apos;équivalent haproxyreceiver.
</Callout>

## Exemples de requêtes NRQL [#sample-queries]

Utilisez ces requêtes pour explorer vos métriques HAProxy dans New Relic :

```sql
-- Current sessions across all frontends
FROM Metric SELECT latest(haproxy.sessions.count)
FACET haproxy.proxy_name
WHERE haproxy.service_name = 'FRONTEND'

-- Request rate per frontend over time
FROM Metric SELECT average(haproxy.requests.rate)
FACET haproxy.proxy_name
WHERE haproxy.service_name = 'FRONTEND'
TIMESERIES

-- Backend error rate (responses.errors as rate)
FROM Metric SELECT rate(sum(haproxy.responses.errors), 1 second)
FACET haproxy.proxy_name
WHERE haproxy.service_name = 'BACKEND'
TIMESERIES

-- Bytes throughput per backend
FROM Metric SELECT rate(sum(haproxy.bytes.input), 1 minute) AS 'Bytes In/min',
  rate(sum(haproxy.bytes.output), 1 minute) AS 'Bytes Out/min'
FACET haproxy.proxy_name
WHERE haproxy.service_name = 'BACKEND'

-- Average session time per backend (latency indicator)
FROM Metric SELECT average(haproxy.sessions.average)
FACET haproxy.proxy_name
WHERE haproxy.service_name = 'BACKEND'

-- Per-server session count (individual backend servers)
FROM Metric SELECT latest(haproxy.sessions.count)
FACET haproxy.proxy_name, haproxy.service_name
WHERE haproxy.service_name NOT IN ('FRONTEND', 'BACKEND')

-- Queued requests (indicates backend saturation)
FROM Metric SELECT max(haproxy.requests.queued)
FACET haproxy.proxy_name
WHERE haproxy.service_name = 'BACKEND'
TIMESERIES
```
