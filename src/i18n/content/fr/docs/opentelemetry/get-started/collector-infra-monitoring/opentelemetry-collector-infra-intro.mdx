[OpenTelemetry Collector](https://opentelemetry.io/docs/collector/) est un outil indépendant du fournisseur pour la réception, le traitement et l&apos;exportation de données télémétriques. Bien que les Collector exigences et configuration varient, il est livré avec une variété de récepteurs et de processeurs qui le rendent populaire pour monitoring d&apos;infrastructure. Il est également courant d&apos;utiliser le Collector pour le traitement des données, mais cette documentation se concentre sur les cas d&apos;utilisation de monitoring d&apos;infrastructure . Consultez [OpenTelemetry Collector pour le traitement des données](/docs/opentelemetry/get-started/collector-processing/opentelemetry-collector-processing-intro) pour plus d&apos;informations.

## Corrélation de l&apos;infrastructure avec OpenTelemetry APM [#infrastructure-correlation]

L&apos;exemple suivant illustre l&apos;utilisation du Collector pour corréler [APM OpenTelemetry](/docs/opentelemetry/get-started/apm-monitoring/opentelemetry-apm-intro) avec les données infrastructure :

* [Exemple de référentiel Hôtes (repository)](https://github.com/newrelic/newrelic-opentelemetry-examples/tree/main/other-examples/collector/host-monitoring)

Le modèle général consiste à configurer le Collector avec un Processeur qui détecte et enrichit la télémétrie APM avec un contexte d&apos;environnement supplémentaire sous la forme d&apos;un attribut de ressource, avant d&apos;exporter les données vers New Relic via OTLP. New Relic détecte ces données de corrélation et construit des relations entre l&apos;APM et l&apos;entité infrastructure via [les ressources](/docs/opentelemetry/best-practices/opentelemetry-best-practices-resources).

## OpenTelemetry Collector pour le monitoring de Kubernetes [#k8s-monitoring]

Bien que les exigences Collector et configuration varient, il est livré avec une variété de récepteurs et de processeurs qui le rendent populaire pour Kubernetes monitoring. Obtenir toute la télémétrie requise nécessite de récupérer diverses API Kubernetes à l&apos;aide de plusieurs récepteurs avec des exigences d&apos;autorisation différentes. Par conséquent, pour vous simplifier la tâche, nous avons créé un graphique Helm qui utilise notre distribution personnalisée du collecteur [nrdot-collecteur-k8s](https://github.com/newrelic/nrdot-collector-releases/tree/main/distributions/nrdot-collector-k8s) sous le capot. Pour plus de détails sur la façon de démarrer, veuillez vous référer à [OpenTelemetry pour Kubernetes](/docs/kubernetes-pixie/k8s-otel/intro/).
