ATP collecte des métriques de processus via le `hostmetricsreceiver` OpenTelemetry et les enrichit de métadonnées de filtrage intelligent. Vous pouvez interroger ces données pour :

* Monitorer les états des processus et identifier les processus `stuck` ou problématiques
* Suivez comment les seuils dynamiques d&apos;ATP s&apos;adaptent à vos modèles de workload
* Mesurez l&apos;efficacité du filtrage pour comprendre l&apos;efficacité de la réduction des données.
* Détecter les anomalies et les comportements inhabituels des processus
* Configurer des alertes pour le monitoring des processus critiques

Toutes les métriques de processus sont disponibles sous l&apos;espace de noms `process.*` dans New Relic, avec les métadonnées spécifiques à ATP stockées dans le champ `process.atp`.

## Monitoring de l&apos;état des processus [#process-state]

<CollapserGroup>
  <Collapser id="process-count-by-name" title="Répartition des processus par état">
    Cette requête affiche une répartition de tous les processus en cours d&apos;exécution sur votre hôte, classés par état d&apos;exécution. C&apos;est essentiel pour identifier les processus `stuck` ou l&apos;épuisement des ressources.

    ```sql
    FROM Metric
    SELECT uniqueCount(process.pid) as 'Process Count'
    FACET process.state, CASES(
      WHERE process.state = 'R' AS 'Running',
      WHERE process.state = 'S' AS 'Sleeping (Interruptible)',
      WHERE process.state = 'D' AS 'Uninterruptible Sleep (Disk I/O)',
      WHERE process.state = 'Z' AS 'Zombie',
      WHERE process.state = 'T' AS 'Stopped',
      WHERE process.state = 'I' AS 'Idle Kernel Thread',
      WHERE process.state = 'X' AS 'Dead',
      WHERE process.state = 'W' AS 'Paging'
    )
    ```
  </Collapser>
</CollapserGroup>

## Monitoring de l&apos;intelligence ATP [#atp-intelligence]

<CollapserGroup>
  <Collapser id="dynamic-thresholds" title="Suivi des seuils dynamiques">
    Cette requête affiche les valeurs actuelles des seuils dynamiques comparées aux métriques réelles des processus, vous aidant à comprendre à quel point les processus sont proches de déclencher des actions ATP.

    ```sql
    FROM Metric
    SELECT
      latest(numeric(jparse(`process.atp`)[threshold_details][`process.cpu.utilization`][threshold])) * 100 AS 'Process CPU Utilization',
      latest(numeric(jparse(`process.atp`)[threshold_details][`process.memory.usage`][threshold])) AS 'Memory Usage'
    WHERE host.name = 'YOUR_HOST_NAME'
    TIMESERIES MAX
    ```

    **Comment utiliser cette requête :**

    * Remplacez `YOUR_HOST_NAME` par votre véritable nom d&apos;hôte
    * Vous pouvez modifier les noms de métriques dans la fonction `jparse` pour suivre les seuils d&apos;autres métriques
    * Surveillez les motifs où les valeurs réelles s&apos;approchent systématiquement des valeurs seuils
  </Collapser>

  <Collapser id="filtering-efficiency" title="Efficacité du filtrage ATP">
    Cette requête mesure l&apos;efficacité avec laquelle ATP réduit le volume de télémétrie tout en conservant les données critiques.

    ```sql
    FROM Metric
    SELECT latest(numeric(jparse(`process.atp`)['filtering_summary']['efficiency_ratio'])) * 100 as 'Filtering Efficiency'
    WHERE metricName like 'process.%'
    AND `process.atp` IS NOT NULL
    ```
  </Collapser>
</CollapserGroup>

## Requêtes de détection d&apos;anomalies [#anomaly-detection]

<CollapserGroup>
  <Collapser id="anomaly-overview" title="Nombre d'anomalies au fil du temps">
    Cette requête visualise la fréquence à laquelle ATP détecte des anomalies dans différents exécutables, vous aidant à identifier les processus les plus sujets à un comportement inhabituel.

    ```sql
    FROM Metric
    SELECT latest(numeric(jparse(`process.atp`, 'filtering_summary.stage_hits.anomaly_detection'))) AS 'Anomaly Count'
    WHERE metricName like 'process.%'
    FACET process.executable.name
    TIMESERIES MAX
    ```
  </Collapser>

  <Collapser id="memory-usuage" title="Anomalies d'utilisation de la mémoire">
    Cette requête met en évidence des processus spécifiques ayant subi des anomalies d&apos;utilisation de la mémoire détectées par ATP.

    ```sql
    FROM Metric
    SELECT max(process.memory.usage) as 'Memory Usage'
    WHERE metricName = 'process.memory.usage'
    AND numeric(jparse(`process.atp`)['filtering_summary']['stage_hits']['anomaly_detection']) > 0
    AND process.atp IS NOT NULL
    FACET process.pid, process.executable.name, host.name
    TIMESERIES MAX
    ```
  </Collapser>

  <Collapser id="cpu-usage" title="Anomalies d'utilisation du CPU">
    Cette requête affiche les processus ayant connu des anomalies d&apos;utilisation du CPU.

    ```sql
    FROM Metric
    SELECT max(process.cpu.utilization) * 100 as 'CPU Utilization'
    WHERE metricName like 'process.cpu.utilization'
    AND numeric(jparse(`process.atp`)['filtering_summary']['stage_hits']['anomaly_detection']) > 0
    AND process.atp IS NOT NULL
    FACET process.pid, process.executable.name, host.name
    TIMESERIES MAX
    ```

    **Remarque**: vous pouvez modifier ces requêtes pour monitorer les anomalies de toute autre métrique de processus en modifiant le filtre `metricName`.
  </Collapser>
</CollapserGroup>

## Monitoring des performances [#performance-monitoring]

<CollapserGroup>
  <Collapser id="context-switches" title="Changements de contexte par minute">
    Cette requête suit le taux de changement de contexte par processus. Des taux élevés de changement de contexte indiquent souvent une contention du processeur ou un multi-threading inefficace.

    ```sql
    SELECT rate(sum(process.context_switches), 1 minute) AS 'Context Switches/min'
    FROM Metric
    WHERE metricName = 'process.context_switches'
    FACET process.executable.name, process.pid
    TIMESERIES MAX
    ```
  </Collapser>
</CollapserGroup>

## Configuration des alertes

<CollapserGroup>
  <Collapser id="data-alert" title="Alerte de manque de données">
    Utilisez cette requête pour créer une alerte qui se déclenche lorsque la télémétrie des processus s&apos;arrête complètement, indiquant une défaillance potentielle de l&apos;agent ou des problèmes réseau. Pour plus d&apos;informations sur la configuration des alertes, consultez [Créer des conditions d&apos;alerte](/docs/alerts/create-alert/create-alert-condition/alert-conditions/).

    ```sql
    FROM Metric
    SELECT count(*)
    WHERE metricName LIKE 'process.%'
    ```
  </Collapser>
</CollapserGroup>
