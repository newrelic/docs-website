La distribution New Relic de l&apos;OpenTelemetry Collector (NRDOT) est une distribution préconstruite de l&apos;OpenTelemetry [Collector](https://opentelemetry.io/docs/collector/) prise en charge par New Relic. NRDOT est conçu pour fonctionner de manière transparente avec les produits et services New Relic, et c&apos;est le moyen recommandé pour collecter, traiter, et exporter des données télémétriques OpenTelemetry pour une utilisation avec New Relic. New Relic fournit une distribution unique appelée `nrdot-collector` qui est destinée à fonctionner avec de nombreuses configurations, chacune s&apos;intégrant à l&apos;ingestion de New Relic et pilotant des cas d&apos;utilisation spécifiques au domaine.

De plus, les utilisateurs avancés peuvent créer des distributions personnalisées avec des composants supplémentaires à l&apos;aide d&apos;[OCB](https://opentelemetry.io/docs/collector/custom-collector/) si les composants [inclus dans NRDOT](https://github.com/newrelic/nrdot-collector-releases/blob/main/distributions/nrdot-collector/manifest.yaml) ne répondent pas à leurs besoins.

<Callout variant="tip">
  Il existe des scénarios dans lesquels vous souhaiterez peut-être déployer le collecteur de manière moins subjective. Nous vous recommandons de lire notre documentation sur [le traitement du collecteur](/docs/opentelemetry/get-started/collector-processing/opentelemetry-collector-processing-intro) pour comprendre comment configurer le collecteur pour votre cas d&apos;utilisation spécifique.
</Callout>

Avant de vous plonger dans l&apos;un de nos cas d&apos;utilisation, nous vous recommandons de vous familiariser avec les concepts de base d&apos;OpenTelemetry Collector pour comprendre comment nos configurations sont structurées et comment les personnaliser selon vos besoins.

<Callout variant="tip">
  Le Collector NRDOT prend en charge la conformité [FIPS 140-3](https://csrc.nist.gov/pubs/fips/140-3/final). Cependant, veuillez noter que ceci s&apos;applique uniquement aux images Docker avec le suffixe `-fips` dans le tag de version (par exemple, `newrelic/nrdot-collector:1.12.0-fips`). Cette version utilise BoringCrypto ([certificat NIST](https://csrc.nist.gov/projects/cryptographic-module-validation-program/certificate/4735)) et garantit que toutes les communications TLS utilisent des suites de chiffrement approuvées. Pour plus d&apos;informations, consultez la [documentation NRDOT FIPS](https://github.com/newrelic/nrdot-collector-releases/blob/main/fips/README.md).
</Callout>

## déploierons [#deployments]

Le collecteur lui-même est un binaire unique qui peut être déployé de diverses manières. Les stratégies de déploiement typiques sont :

* En tant qu&apos;agent : déployer sur le même hôte que l&apos;application.
* En tant que gateway : se déployer comme point central de collecte de télémétrie.

Chaque cas d&apos;utilisation du NRDOT est prévu pour un déploiement spécifique, et bien que les configurations puissent être personnalisées pour répondre à vos besoins, nous vous recommandons de vous en tenir aux recommandations de la documentation correspondante, à moins que vous n&apos;ayez une bonne raison de vous en écarter.

Des informations supplémentaires sur les modèles de déploiement sont disponibles dans la [documentation d&apos;OpenTelemetry Collector](https://opentelemetry.io/docs/collector/deployment/).
