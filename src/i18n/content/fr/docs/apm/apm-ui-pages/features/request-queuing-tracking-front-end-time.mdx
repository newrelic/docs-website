APM suit le temps écoulé entre l&apos;entrée d&apos;une demande dans votre système de production et son arrivée dans votre application. Nous appelons cette partie du cycle de vie de votre demande <DNT>**request queuing**</DNT>. Selon les spécificités de votre système de production, cette mesure du temps peut ou non inclure une file d&amp;apos;attente réelle dans laquelle requests entrent. Il peut également représenter d’autres fonctions (telles que l’équilibrage de charge ou la latence du réseau interne).

## Utiliser la mise en file d&apos;attente des demandes pour identifier les problèmes de mise à l&apos;échelle [#performance-scaling]

Le suivi du temps passé dans la file d&apos;attente des demandes est utile pour identifier certains types de problèmes de performances et de mise à l&apos;échelle ; par exemple :

* Lorsque votre serveur Web frontal perd du temps à attendre que les travailleurs de l&apos;application soient disponibles
* Lorsque du temps supplémentaire est consacré au réchauffement des travailleurs de l&apos;application après un réveil ou un redémarrage

Vous devez configurer votre [agent](/docs/features/configuring-request-queue-reporting) et votre [serveur](/docs/features/request-queue-server-configuration-examples) New Relic pour signaler la mise en file d&amp;apos;attente des demandes. Ensuite, les informations apparaîtront dans le graphique <DNT>**Requests time**</DNT> de l&amp;apos;application sélectionnée pour le Web des transactions (dans la liste <DNT>**Applications**</DNT> de l&amp;apos;APM, sélectionnez l&amp;apos;application), ainsi qu&amp;apos;à d&amp;apos;autres endroits de l&amp;apos;interface utilisateur. La légende du graphique indique quelle couleur représente la mise en file d&amp;apos;attente des demandes.

## Calculs Apdex [#apdex]

La mise en file d&apos;attente des requêtes correspond au temps écoulé entre le moment où le navigateur requests du contenu et le moment où il le reçoit. Étant donné que votre score Apdex reflétera ces calculs, vous pouvez choisir de signaler le temps d&apos;attente des demandes séparément ou non. Pour plus d&apos;informations, voir [configuration de l&apos;agent](/docs/features/configuring-request-queue-reporting).

## Décalage de l&apos;horloge

Si le serveur Web frontal (tel que Nginx) et votre application ne résident pas sur le même serveur physique, la mise en file d&apos;attente des demandes signalées peut être affectée par un décalage d&apos;horloge. [NTP](http://www.ntp.org/ "Le lien s'ouvre dans une nouvelle fenêtre") fournit un excellent moyen de maintenir les horloges des serveurs synchronisées. Cependant, ils continueront à dériver l’un par rapport à l’autre. Étant donné que les agents New Relic s&amp;apos;appuient sur un horodatage défini par le serveur frontal, ils peuvent sur- ou sous-évaluer la mise en file d&amp;apos;attente des demandes si l&amp;apos;horloge de ce serveur n&amp;apos;est pas étroitement synchronisée avec l&amp;apos;horloge du serveur d&amp;apos;applications.

Cela peut sembler être un problème majeur avec cette fonctionnalité ; cependant, il est peu probable que le décalage d’horloge entraîne des pics soudains dans la file d’attente des demandes signalées. Des pics soudains se produisent généralement lorsqu&apos;une application est redémarrée ou est surchargée de requests. Notre expérience montre que les rapports sur les files d&apos;attente de demandes peuvent être utiles pour identifier les problèmes de performances réels, mais assurez-vous de prendre en compte le décalage d&apos;horloge lors de l&apos;interprétation de ces données.
