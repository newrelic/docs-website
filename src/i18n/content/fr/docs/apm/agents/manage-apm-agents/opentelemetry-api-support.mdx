Les agents APM New Relic prennent désormais en charge les API OpenTelemetry (OTel), vous permettant d&apos;instrumenter vos applications à l&apos;aide d&apos;API OpenTelemetry indépendantes des fournisseurs tout en conservant un accès complet aux puissantes fonctionnalités APM de New Relic. Cette fonctionnalité vous offre la flexibilité d&apos;écrire du code d&apos;instrumentation portable sans sacrifier la profondeur des informations et les fonctionnalités offertes par New Relic.

## Avantages [#benefits]

L&apos;utilisation de la prise en charge de l&apos;API OpenTelemetry avec les agents New Relic offre plusieurs avantages :

* **Instrumentation indépendante du fournisseur**: Écrivez une instrumentation personnalisée à l&apos;aide des API OpenTelemetry standards qui fonctionnent sur les plateformes d&apos;observabilité
* **Aucune migration requise**: Activez la prise en charge de l&apos;API OTel dans votre agent New Relic existant sans remplacer votre instrumentation actuelle
* **Adoption progressive**: Utilisez les API OpenTelemetry pour la nouvelle instrumentation personnalisée tandis que l&apos;instrumentation New Relic existante continue de fonctionner
* **Compatibilité en mode mixte**: tracez les requêtes de manière transparente à travers les services instrumentés avec des agents New Relic, des API OpenTelemetry ou des agents OTel purs
* **Instrumentation communautaire**: Tirez parti de l&apos;instrumentation OpenTelemetry native pour les frameworks et les bibliothèques

## Ce que cette fonctionnalité apporte [#what-it-provides]

Lorsque vous activez la prise en charge de l&apos;API OpenTelemetry dans un agent New Relic, l&apos;agent peut :

* Capturez les spans, les traces, les métriques et les logs créés à l&apos;aide des API OpenTelemetry
* Intégrez de manière transparente la télémétrie générée par OTel dans les traces New Relic APM
* Prise en charge de l&apos;instrumentation OpenTelemetry native qui n&apos;entre pas en conflit avec l&apos;instrumentation intégrée de New Relic
* Maintenir le contexte de trace à travers des services instrumentés mixtes New Relic et OpenTelemetry

<Callout variant="important">
  Cette fonctionnalité est **désactivée par défaut**. Vous devez l&apos;activer explicitement via la configuration de l&apos;Agent.
</Callout>

### Échantillonnage dans des environnements mixtes

Tous les agents prennent désormais en charge l&apos;**échantillonnage par ratio de traces**, s&apos;alignant sur les standards OpenTelemetry et garantissant une logique d&apos;échantillonnage cohérente dans les environnements mixtes.

**Principaux avantages**:

* Décisions d&apos;échantillonnage cohérentes entre New Relic et les services instrumentés par OTel
* Préservation des décisions de trace en amont issues des SDK OTel

Pour plus de détails sur le fonctionnement de l&apos;échantillonnage adaptatif, les cibles d&apos;échantillonnage par agent et les options de configuration, consultez [Échantillonnage du traçage distribué](/docs/distributed-tracing/concepts/how-new-relic-distributed-tracing-works/#trace-origin-sampling).

## Types d&apos;instrumentation pris en charge [#instrumentation-types]

### Instrumentation native (automatique)

L&apos;instrumentation OpenTelemetry native est prise en charge prête à l&apos;emploi. Ces instrumentations sont directement intégrées à la bibliothèque ou au framework par son mainteneur et fonctionnent automatiquement lorsque vous activez la prise en charge de l&apos;API OpenTelemetry.

Les instrumentations natives spécifiques disponibles varient selon l&apos;agent de langage. Consultez la [section configuration](#configuration) pour plus de détails sur les instrumentations prises en charge pour chaque langage.

<Callout variant="important">
  **Limitation connue**: certaines instrumentations natives peuvent générer des noms de spans ou de métriques légèrement différents de ceux de l&apos;instrumentation intégrée de New Relic en raison de différences dans les conventions sémantiques d&apos;OpenTelemetry (par exemple, Elasticsearch, RabbitMQ).
</Callout>

### Instrumentation personnalisée avec les API OpenTelemetry

Vous pouvez écrire une instrumentation personnalisée à l&apos;aide des API OpenTelemetry standard :

* **Traces**: Créez des spans personnalisés et ajoutez des attributs à l&apos;aide de l&apos;API de traçage OpenTelemetry
* **Métriques**: Enregistrez des métriques personnalisées à l&apos;aide de l&apos;API OpenTelemetry Metrics (si prise en charge)
* **Logs**: Écrivez des logs à l&apos;aide de l&apos;API OpenTelemetry Logs (si pris en charge)

Consultez la [documentation OpenTelemetry](https://opentelemetry.io/docs/instrumentation/) pour plus de détails sur l&apos;utilisation de ces API dans votre langage.

## Comment activer la prise en charge de l&apos;API OpenTelemetry [#enable]

La prise en charge de l&apos;API OpenTelemetry est désactivée par défaut. Activez-le via la configuration de votre agent :

<CollapserGroup>
  <Collapser id="java-config" title="Configuration de l'agent Java">
    ### Prérequis

    Ajoutez les dépendances OpenTelemetry requises à votre application :

    ```groovy
    implementation(platform("io.opentelemetry:opentelemetry-bom:1.44.1"))
    implementation("io.opentelemetry:opentelemetry-sdk-extension-autoconfigure")
    implementation("io.opentelemetry:opentelemetry-exporter-otlp")
    ```

    ### Activer la prise en charge de l&apos;API OpenTelemetry

    **1. Activer l&apos;autoconfiguration du SDK OpenTelemetry**

    Utilisez l&apos;une des méthodes suivantes :

    Propriété du système :

    ```bash
    -Dotel.java.global-autoconfigure.enabled=true
    ```

    Variable d&apos;environnement :

    ```bash
    export OTEL_JAVA_GLOBAL_AUTOCONFIGURE_ENABLED=true
    ```

    **2. Activer la prise en charge de New Relic OpenTelemetry**

    Propriété du système :

    ```bash
    -Dnewrelic.config.opentelemetry.enabled=true
    ```

    Variable d&apos;environnement :

    ```bash
    NEW_RELIC_OPENTELEMETRY_ENABLED=true
    ```

    Configuration YAML (`newrelic.yml`) :

    ```yaml
    opentelemetry:
      enabled: true
    ```

    <Callout variant="important">
      L&apos;agent Java instrumente les API de traces, métriques et logs OpenTelemetry pour capturer les signaux émis par votre application. Il n&apos;inclut **pas** et n&apos;active pas automatiquement d&apos;instrumentations OpenTelemetry. L&apos;auto-instrumentation OpenTelemetry n&apos;est pas prise en charge.
    </Callout>

    ### Types d&apos;instrumentation pris en charge

    #### Instrumentation manuelle

    L&apos;instrumentation personnalisée utilisant les API de traces, de métriques ou de logs OpenTelemetry sera automatiquement capturée par l&apos;agent.

    #### Instrumentation native et de bibliothèque

    Les instrumentations de bibliothèques autonomes OpenTelemetry et l&apos;instrumentation native intégrée directement dans les bibliothèques fonctionneront avec l&apos;agent New Relic. Cependant, l&apos;auto-instrumentation OpenTelemetry basée sur le bytecode ne sera pas détectée par l&apos;agent New Relic.
  </Collapser>

  <Collapser id="nodejs-config" title="Configuration de l'agent Node.js">
    ### Activer la prise en charge de l&apos;API OpenTelemetry

    Activez la prise en charge de l&apos;API OpenTelemetry dans votre fichier de configuration `newrelic.js` :

    ```javascript
    exports.config = {
      // ... other config
      opentelemetry: {
        enabled: true,
        traces: {
          enabled: true
        }
      }
      // ... other config
    }
    ```

    <Callout variant="important">
      L&apos;agent Node.js n&apos;active **pas** automatiquement d&apos;instrumentations OpenTelemetry. Il instrumente uniquement les API OpenTelemetry pour capturer la télémétrie émise par votre application.
    </Callout>

    ### API OpenTelemetry prises en charge

    Lorsque la prise en charge d&apos;OpenTelemetry est activée, l&apos;agent Node.js prend entièrement en charge :

    #### API Traces

    Créer des spans personnalisés et ajouter des attributs à l&apos;aide de l&apos;API de traçage OpenTelemetry. Consultez l&apos;[exemple de traçage OpenTelemetry Node.js](https://github.com/newrelic/newrelic-node-examples/tree/4a3332d55e4455dca0f8eeba34a87e4a513a679a/opentelemetry-example).

    #### API métriques

    L&apos;**API Metrics est la fonctionnalité la plus utile** pour les clients Node.js. Cela étend la capacité de collecte de métriques pour inclure les métriques dimensionnelles, une fonctionnalité fréquemment demandée.

    Enregistrez des métriques personnalisées à l&apos;aide de l&apos;API OpenTelemetry Metrics. Consultez l&apos;[exemple de métriques OpenTelemetry Node.js](https://github.com/newrelic/newrelic-node-examples/tree/4a3332d55e4455dca0f8eeba34a87e4a513a679a/opentelemetry-example).

    #### API Logs

    L&apos;API OpenTelemetry Logs JavaScript est encore en développement, mais l&apos;agent Node.js la prend en charge. Vous pouvez utiliser l&apos;API pour émettre des logs qui seront envoyés à New Relic.

    Consultez l&apos;[exemple de logs OpenTelemetry Node.js](https://github.com/newrelic/newrelic-node-examples/tree/4a3332d55e4455dca0f8eeba34a87e4a513a679a/opentelemetry-example).

    ### Types d&apos;instrumentation pris en charge

    #### Instrumentation manuelle (recommandée)

    Les clients qui instrumentent leurs applications avec l&apos;API OpenTelemetry constateront que l&apos;agent traite et transmet ces données correctement.

    #### Instrumentation native et de bibliothèque (limitée)

    La liste des instrumentations OpenTelemetry recoupe largement les modules pris en charge par l&apos;agent Node.js. Les instrumentations qui ne sont pas déjà couvertes par l&apos;agent New Relic peuvent :

    * Conflit avec les éléments internes de l&apos;agent
    * Ne pas avoir une demande client suffisante

    <Callout variant="caution">
      Nous n&apos;avons pas testé les instrumentations OpenTelemetry en dehors de la liste officielle OpenTelemetry et ne pouvons en recommander aucune spécifique pour le moment.
    </Callout>

    ### Limitations connues

    * **Aucune instrumentation automatique**: l&apos;agent n&apos;active pas automatiquement les instrumentations OpenTelemetry. Les clients doivent instrumenter manuellement leur code à l&apos;aide des API OpenTelemetry.
  </Collapser>

  <Collapser id="dotnet-config" title="Configuration de l'agent .NET">
    ### Activer la prise en charge de l&apos;API OpenTelemetry

    Activez la prise en charge de l&apos;API OpenTelemetry à l&apos;aide des variables d&apos;environnement ou du fichier de configuration XML `newrelic.config`.

    **Utilisation des variables d&apos;environnement :**

    ```bash
    NEW_RELIC_OPENTELEMETRY_ENABLED=true
    ```

    **Utilisation de newrelic.config :**

    ```xml
    <configuration>
      <openTelemetry enabled="true" />
    </configuration>
    ```

    <Callout variant="tip">
      Les **Traces** et les **Métriques** sont activées par défaut lorsque la prise en charge d&apos;OpenTelemetry est activée.
    </Callout>

    ### Instrumentations natives prises en charge

    Lorsque vous activez la prise en charge de l&apos;API OpenTelemetry, l&apos;agent .NET peut capturer la télémétrie provenant des bibliothèques instrumentées par OpenTelemetry.

    #### Instrumentations des traces

    L&apos;Agent prend en charge les bibliothèques avec une instrumentation de tracing OpenTelemetry native. L&apos;instrumentation personnalisée écrite à l&apos;aide de l&apos;API de traçage OpenTelemetry est également prise en charge.

    #### Instrumentations des métriques

    Lorsque les métriques sont activées, l&apos;agent prend en charge les instrumentations de métriques OpenTelemetry provenant de bibliothèques et de frameworks. L&apos;instrumentation personnalisée écrite à l&apos;aide de l&apos;API OpenTelemetry Metrics est également prise en charge.

    #### Instrumentations des logs

    L&apos;agent .NET instrumente déjà la bibliothèque de logging qui est devenue l&apos;API OpenTelemetry pour les données de logs. **Aucune modification n&apos;est requise** pour les logs lors de l&apos;activation de la prise en charge de l&apos;API OpenTelemetry. Pour assurer la rétrocompatibilité, l&apos;instrumentation des logs continue de fonctionner comme auparavant.
  </Collapser>

  <Collapser id="python-config" title="Python agent configuration">
    ### Activer la prise en charge de l&apos;API OpenTelemetry

    Activez la prise en charge de l&apos;API OpenTelemetry dans votre fichier de configuration `newrelic.ini` ou via des variables d&apos;environnement :

    **Utilisation de newrelic.ini :**

    ```ini
    [newrelic]
    opentelemetry.enabled = true
    ```

    **Utilisation de la variable d&apos;environnement :**

    ```bash
    NEW_RELIC_OPENTELEMETRY_ENABLED=true
    ```

    <Callout variant="important">
      Lorsque la prise en charge d&apos;OpenTelemetry est activée, l&apos;agent détecte automatiquement si `opentelemetry-api` est installé et bascule vers l&apos;instrumentation OpenTelemetry pour les bibliothèques prises en charge.
    </Callout>

    ### Comment ça marche

    Lorsque la prise en charge d&apos;OpenTelemetry est activée et que l&apos;agent détecte que `opentelemetry-api` est installé :

    1. **Instrumentation native**: Pour les bibliothèques avec une instrumentation OpenTelemetry native intégrée, l&apos;agent désactive automatiquement les hooks de New Relic et utilise l&apos;instrumentation OpenTelemetry native à la place.

    2. **Instrumentation de bibliothèque**: Si une bibliothèque d&apos;instrumentation OpenTelemetry est détectée, l&apos;agent utilise automatiquement l&apos;instrumentation OpenTelemetry au lieu des hooks d&apos;instrumentation de New Relic.

    Pour la liste actuelle des instrumentations Python disponibles, consultez la [liste officielle des instrumentations Python OpenTelemetry](https://opentelemetry.io/ecosystem/registry/?language=python&component=instrumentation).

    <Callout variant="caution">
      Certaines instrumentations (telles que les bibliothèques AWS et gRPC) continueront d&apos;utiliser l&apos;instrumentation de New Relic. Les bibliothèques d&apos;IA avec instrumentation native sont explicitement désactivées pour l&apos;instant et seront prises en charge dans une future version.
    </Callout>

    ### Instrumentation manuelle

    Toute instrumentation personnalisée écrite à l&apos;aide de l&apos;API de traçage OpenTelemetry sera capturée par l&apos;agent.

    **Exemple:**

    ```python
    from opentelemetry import trace

    tracer = trace.get_tracer(__name__)

    with tracer.start_as_current_span("my-custom-span") as span:
        span.set_attribute("custom.attribute", "value")
        # Your code here
    ```

    ### Fonctionnalités prises en charge

    #### API Traces

    ✅ **Entièrement pris en charge**: Créer des spans personnalisés et ajouter des attributs à l&apos;aide de l&apos;API de tracing OpenTelemetry

    #### API métriques

    ❌ **Pas encore pris en charge**: l&apos;API OpenTelemetry Metrics n&apos;est pas activée dans l&apos;agent Python

    #### API Logs

    ✅ **Fonctionne déjà**: L&apos;agent prend déjà en charge le module de logging. Il n&apos;existe pas d&apos;API OpenTelemetry pour le logging en Python (seul un SDK existe). Ce qu&apos;OpenTelemetry configure pour le logging est si c&apos;est activé et vers quels points de terminaison envoyer les données.

    ### Comment ça marche

    Lorsque la prise en charge d&apos;OpenTelemetry est activée :

    1. **L&apos;Agent recherche** dans le système pour déterminer si les bibliothèques d&apos;instrumentation OpenTelemetry sont installées
    2. **Basculement automatique**: si elle est détectée, l&apos;agent utilise l&apos;instrumentation de la bibliothèque OpenTelemetry au lieu des hooks de New Relic
    3. **Intégration transparente**: les spans OpenTelemetry sont intégrés aux traces New Relic

    ### Limitations connues

    * **Métriques non prises en charge**: l&apos;API OpenTelemetry Metrics n&apos;est pas encore activée
    * **Certaines bibliothèques AWS et gRPC exclues**: continueront d&apos;utiliser l&apos;instrumentation New Relic
    * **Bibliothèques d&apos;IA désactivées**: L&apos;instrumentation OpenTelemetry native pour les bibliothèques d&apos;IA est désactivée pour le moment
    * **Paramètres d&apos;inclusion/exclusion**: Des options d&apos;affinement sont prévues pour les futures versions, mais seule l&apos;activation/désactivation est actuellement fonctionnelle.
  </Collapser>

  <Collapser id="ruby-config" title="Configuration de l'agent Ruby">
    ### Activer la prise en charge de l&apos;API OpenTelemetry

    Pour activer la prise en charge de l&apos;API OpenTelemetry dans l&apos;agent Ruby, ajoutez ce qui suit à votre fichier de configuration `newrelic.yml` :

    ```yaml
    opentelemetry:
      enabled: true
    ```

    Ou définissez la variable d&apos;environnement :

    ```bash
    NEW_RELIC_OPENTELEMETRY_ENABLED=true
    ```

    ### Fonctionnalités prises en charge

    #### API Traces

    ✅ **Pris en charge**: Créer des spans personnalisées et ajouter des attributs à l&apos;aide de l&apos;API de traçage OpenTelemetry

    #### API métriques

    ❌ **Non pris en charge**: OpenTelemetry Metrics API

    #### API Logs

    ❌ **Non pris en charge**: OpenTelemetry Logs API

    #### Liens de span

    ❌ **Non pris en charge**: Liens de span

    #### Événements sur les spans

    ❌ **Non pris en charge**: Événements sur les Spans

    ### Comment ça marche

    L&apos;agent Ruby détecte la présence de l&apos;installation de l&apos;API OpenTelemetry et instrumente les API OpenTelemetry pour le traçage afin de générer de la télémétrie. Lorsqu&apos;une bibliothèque ou le code de votre application appelle les API OpenTelemetry, l&apos;agent Ruby New Relic traduit ces appels en données New Relic si la prise en charge d&apos;OpenTelemetry est activée.

    * **Prérequis**: L&apos;API OpenTelemetry doit être présente dans votre application pour que la fonctionnalité de l&apos;agent hybride fonctionne.

    ### Instrumentations prises en charge

    L&apos;agent Ruby fonctionne avec trois types d&apos;instrumentation OpenTelemetry :

    #### 1. Instrumentation native

    Les bibliothèques avec des appels d&apos;instrumentation OpenTelemetry intégrés sont prises en charge. Lorsque ces bibliothèques effectuent des appels à l&apos;API OpenTelemetry, l&apos;agent Ruby les capture et les traduit en télémétrie New Relic.

    #### 2. Instrumentation tierce (opentelemetry-ruby-contrib)

    Si vous installez des gems à partir du [dépôt opentelemetry-ruby-contrib](https://github.com/open-telemetry/opentelemetry-ruby-contrib), leurs appels aux API OpenTelemetry seront traités et transformés en données New Relic.

    Pour une liste complète des instrumentations Ruby disponibles, consultez le [registre OpenTelemetry](https://opentelemetry.io/ecosystem/registry/).

    #### 3. Instrumentation personnalisée/manuelle

    Lorsque la prise en charge d&apos;OpenTelemetry est activée, l&apos;appel des API OpenTelemetry dans le code de votre application génère automatiquement des données de télémétrie New Relic.

    ### Limitations connues

    * **Prise en charge limitée de l&apos;API**: seule l&apos;API Traces est prise en charge dans cette version
    * **Métriques et logs**: Non disponible
    * **Liste d&apos;exclusion**: Des travaux sont en cours pour automatiser la prévention de la télémétrie en double lorsque OpenTelemetry et New Relic ont tous deux instrumenté les mêmes méthodes. Pour l&apos;instant, vous devrez peut-être désactiver manuellement une instrumentation pour éviter les doublons.
    * **Liens de span et événements**: Non pris en charge actuellement
  </Collapser>
</CollapserGroup>

## Vérifier que cela fonctionne [#verify]

Après avoir activé la prise en charge de l&apos;API OpenTelemetry, vérifiez que cela fonctionne :

1. Déployez votre application avec la configuration de l&apos;agent mise à jour.

2. Si vous utilisez une instrumentation personnalisée, assurez-vous que votre code génère des spans/métriques à l&apos;aide des API OpenTelemetry.

3. Générez du trafic vers votre application.

4. Vérifiez votre interface New Relic APM :

   * Allez sur **[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &gt; APM &amp; services &gt; (sélectionnez votre application)**
   * Affichez les traces distribuées pour voir les spans créés par les API OpenTelemetry intégrés à vos traces

### Différences des conventions sémantiques

Certaines instrumentations OpenTelemetry utilisent des conventions de nommage différentes pour les spans et les métriques par rapport à l&apos;instrumentation intégrée de New Relic :

* **Elasticsearch**: Les noms de span et les noms d&apos;opération peuvent différer par la casse et la longueur
* **RabbitMQ**: Le traçage distribué fonctionne différemment (liens de span vs relations parent/enfant)

<Callout variant="tip">
  Ces différences peuvent affecter les dashboards et les conditions d&apos;alerte existants qui reposent sur des noms de spans ou de métriques spécifiques. Vérifiez et mettez à jour vos requêtes si vous passez de l&apos;instrumentation New Relic à l&apos;instrumentation OpenTelemetry pour ces services.
</Callout>

### Limitations spécifiques à l&apos;Agent

* **Python**: Seule l&apos;API Traces est prise en charge ; les API Metrics et Logs ne sont pas disponibles
* **Ruby**: Seule l&apos;API Traces est prise en charge ; les API Metrics et Logs ne sont pas disponibles
* **.NET**: L&apos;activation des métriques OpenTelemetry peut augmenter les coûts d&apos;ingestion des métriques dimensionnelles
