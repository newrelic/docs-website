## Problème

Selon la situation, vous pouvez trouver [une trace d'erreur](/docs/apm/applications-menu/error-analytics/error-analytics-manage-error-traces) dans l&amp;apos;UI APM qui n&amp;apos;inclut pas de trace d&amp;apos;appels pour votre application .NET.

## Solution

Selon la situation, suivez ces procédures de dépannage.

<CollapserGroup>
  <Collapser id="500-errors" title="Aucune trace d'appels pour 500 erreurs">
    Il s’agit d’ [un comportement normal](#cause) de la manière dont l’agent .NET gère les erreurs 500. Pour forcer le signalement des traces d&amp;apos;appels, appelez API l&amp;apos;API .NET de New Relic à partir de votre propre code. L&amp;apos;exécution de la méthode `NewRelic.NoticeError(Exception)` entraînera le signalement d&amp;apos;une erreur ainsi que la trace stack représentée par `Exception`. Pour plus d&amp;apos;informations sur cette méthode et ses surcharges, consultez l&amp;apos;[API d'agent .NET sur GitHub](/docs/agents/net-agent/net-agent-api/noticeerror-net-agent-api/) de New Relic.
  </Collapser>
</CollapserGroup>

## Cause

Le renvoi d'une erreur `500` signifie que le serveur d&amp;apos;application lui-même a détecté une erreur et défini le code d&amp;apos;état HTTPS `500` .

* Si la condition d'erreur a été détectée et gérée par la logique application , il n'y avait aucun objet d'exception et donc aucune stack.
* S'il y avait un objet d'exception à un moment donné, mais qu'il était géré en interne par le code d'application qui définissait le statut `500` sur la réponse, alors l&amp;apos;exception n&amp;apos;est jamais devenue visible pour l&amp;apos;agent .NET. Aucune stack n&amp;apos;est disponible pour que l&amp;apos;agent .NET puisse la signaler.

Lorsque des traces d'appels sont signalées, l'erreur résulte d'une exception qui n'a pas été détectée et gérée dans la logique du serveur application . L'agent .NET voit l'exception non gérée lors d'une transaction de monitoring, il signale donc la trace des appels.

Cependant, aucune trace d'appel n'apparaît pour les erreurs `500` car le serveur application gère les erreurs et définit ensuite le code d&amp;apos;état. Le code application lui-même ne conserve aucune trace d&amp;apos;appels.
