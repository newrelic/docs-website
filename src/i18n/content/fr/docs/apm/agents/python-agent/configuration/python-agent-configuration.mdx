[Notre agent Python](/docs/agents/python-agent/getting-started/introduction-new-relic-python) vous permet de modifier le comportement par défaut de l&apos;agent à l&apos;aide d&apos;options de configuration.

Le seul paramètre de configuration de l&apos;agent Python requis est le <InlinePopover type="licenseKey" />. La clé de licence identifie le compte sur lequel l&apos;agent signale les données d&apos;application. Selon la manière dont vous hébergez votre application, la clé de licence peut être fournie via un fichier de configuration ou une variable d&apos;environnement.

## Méthodes de configuration et priorité [#options]

Le principal moyen de configurer l&apos;agent Python est via le [fichier de configuration](#agent-configuration-file), qui est généré dans le cadre du [processus d&apos;installation](/docs/agents/python-agent/installation-configuration/python-agent-installation) standard. Il est également possible de définir un nombre limité d&apos;options configuration en utilisant [configuration côté serveur dans l&apos; UI](#server-side-configuration) ou en utilisant [des variables d&apos;environnement](#environment-variables). Vous pouvez également spécifier certains paramètres pour chaque demande en transmettant des paramètres avec le [dictionnaire d&apos;environnement de demande WSGI](#per-request-configuration).

L&apos;agent Python suit cet ordre de priorité pour la configuration :

<img title="diagram-python-config-precedence.png" alt="diagram-python-config-precedence.png" src="/images/apm_diagram_config-settings.webp" />

<figcaption>
  Avec l&apos;agent Python, les options par requête remplacent la configuration côté serveur. Si cette option est activée, la configuration côté serveur remplace <DNT>**all**</DNT> valeurs correspondantes dans le fichier de configuration de l&apos;agent, même si les valeurs côté serveur sont laissées vides. Le fichier de configuration de l&apos;agent remplace les variables d&apos;environnement. Les variables d’environnement remplacent les valeurs par défaut de l’agent.
</figcaption>

Voici des descriptions détaillées de chaque méthode de configuration :

<CollapserGroup>
  <Collapser className="freq-link" id="agent-configuration-file" title="Fichier de configuration de l'agent">
    En règle générale, vous configurez votre agent Python à partir d&apos;un fichier de configuration local sur le système hôte de l&apos;agent. Fournissez le chemin d’accès au fichier de configuration au démarrage en utilisant l’une de ces méthodes :

    * Lorsque vous appelez [`newrelic.agent.initialize()`](/docs/agents/python-agent/customization-extension/python-management-api#mgmt), indiquez le chemin d’accès au fichier de configuration comme premier argument.

      OU

    * Définissez la variable d’environnement `NEW_RELIC_CONFIG_FILE` . Si vous utilisez le script wrapper `newrelic-admin`, vous devez utiliser la variable d&apos;environnement car le script wrapper appelle l&apos;agent automatiquement.

      L&apos; agent prend en charge deux types de fichiers configuration, l&apos;agent v10.2.0 et les versions antérieures doivent utiliser la syntaxe `.ini` tandis que les versions agent v10.3.0 et supérieures exécutées sur les versions Python 3.11 et supérieures prennent également en charge les fichiers `.toml`.

      <Collapser className="freq-link" id="agent-configuration-file-ini" title="Format du fichier de configuration « .ini »">
        Le fichier de configuration utilise une structure similaire aux fichiers `.ini` de Microsoft Windows. Pour plus d&apos;informations, consultez la [documentation sur le format de fichier](https://docs.python.org/3/library/configparser.html) du module Python ConfigParser.

        La structure de base devrait ressembler à ce qui suit :

        ```ini
        [newrelic]
        license_key = <license key>
        app_name = Python Application
        ```
      </Collapser>

      <Collapser className="freq-link" id="agent-configuration-file-toml" title="Format du fichier de configuration `.toml`">
        <Callout variant="tip">
          À partir de Python 3.11, la nouvelle syntaxe `.toml` pour la configuration a été ajoutée à la [bibliothèque standard](https://docs.python.org/3/library/tomllib.html). La version de l&apos;agent `10.3.0` a ajouté la prise en charge de l&apos;utilisation de ce nouveau format.
        </Callout>

        Le fichier de configuration utilise les fichiers `.toml` du langage minimal évident de Tom. Pour plus d&apos;informations, consultez la [documentation officielle sur le format de fichier](https://toml.io/en/).

        Pour utiliser la syntaxe `.toml`, le nom du fichier de configuration doit se terminer par `.toml` pour indiquer le type de syntaxe. Le fichier standard `pyproject.toml` peut être utilisé avec configuration d&apos;autres bibliothèques, ou vous pouvez utiliser un fichier plus spécifique comme `newrelic.toml`.

        La structure de base devrait ressembler à ce qui suit :

        ```ini
        [tool.newrelic]
        license_key = <license key>
        app_name = Python Application
        ```
      </Collapser>

      <Callout variant="tip">
        Un exemple de fichier de configuration est inclus avec l&apos;agent Python sous le nom `newrelic/newrelic.ini`. Vous pouvez également en générer un à partir du script `newrelic-admin` en utilisant la commande `generate-config`, ou télécharger une copie à partir de [notre référentiel de téléchargement (repository)](https://download.newrelic.com/python_agent/release/).
      </Callout>
  </Collapser>

  <Collapser className="freq-link" id="server-side-configuration" title="Configuration côté serveur">
    [configuration côté serveur vous](/docs/agents/manage-apm-agents/configuration/server-side-agent-configuration) permet de configurer certains paramètres dans l&apos;UI New Relic. Cela applique automatiquement vos modifications à tous les agents, même s&apos;ils s&apos;exécutent sur plusieurs hôtes. Le cas échéant, ce document inclut les étiquettes UI pour la configuration côté serveur sous les options de configuration individuelles comme <DNT>**Server-side label**</DNT>.

    <Callout variant="important">
      Si la configuration côté serveur est activée, l&apos; agent ignore toute valeur dans le fichier de configuration qui <DNT>**could**</DNT> doit être définie dans l&apos; UI. Même si la valeur UI est vide, l&apos; agent la traite comme une chaîne vide et n&apos;utilise pas le fichier de configuration agent .
    </Callout>
  </Collapser>

  <Collapser className="freq-link" id="environment-variables" title="Variables d'environnement">
    Les variables d’environnement vous permettent de remplacer les valeurs par défaut de certains paramètres de base. Si le paramètre équivalent est explicitement répertorié dans le fichier de configuration de l&apos;agent, les paramètres du fichier de configuration ont priorité sur la variable d&apos;environnement. Lorsque disponibles, les variables d&apos;environnement sont documentées ci-dessous sous les options de configuration individuelles sous la forme <DNT>**Environ variable**</DNT>.

    Pour les configurations simples, vous pouvez utiliser les variables d&apos;environnement conjointement avec [la configuration côté serveur](#server-side-configuration) et éviter complètement le fichier de configuration de l&apos;agent. Il s&apos;agit de la configuration par défaut avec [Heroku](/docs/agents/python-agent/hosting-services/python-agent-heroku), où l&apos;installation du module complémentaire New Relic remplit automatiquement les variables d&apos;environnement nécessaires.

    Si vous utilisez New Relic CodeStream pour monitorer les performances de votre IDE, vous souhaiterez peut-être également [associer un référentiel à vos services](/docs/codestream/observability/repo-association) et [associer des SHA de build ou une balise de sortie à des erreurs](/docs/codestream/observability/error-investigation/#buildsha).

  <Collapser className="freq-link" id="per-request-configuration" title="Configuration par requête">
    Pour certains serveurs WSGI, vous pouvez remplacer le [nom de l&apos;application](#app_name) et [capturer les paramètres d&apos;attribut](#attributes) pour chaque demande. Ceci est possible avec les serveurs WSGI où vous pouvez définir des paires valeur-clé supplémentaires qui sont transmises au dictionnaire d&apos;environnement WSGI par demande.

    Définissez ces valeurs avec les chaînes `on`, `off`, `true`, `false`, `1` et `0`. Si défini à partir d&apos;un mécanisme de configuration implémenté à l&apos;aide de code Python, les objets Python évaluant à True ou False seront également acceptés.

    <CollapserGroup>
      <Collapser id="example-apache-mod-wsgi" title="Exemple : nom de l'application Apache/mod_wsgi">
        Dans le serveur Apache/mod\_wsgi, vous pouvez utiliser la directive `SetEnv` pour remplacer les paramètres de configuration (éventuellement à l&apos;intérieur d&apos;un bloc `Location` ou `Directory` ). Par exemple, vous pouvez remplacer le [nom de l’application](#app_name) pour un hôte virtuel complet ou pour un sous-ensemble d’URL gérées par l’application WSGI pour cet hôte virtuel.
      </Collapser>
    </CollapserGroup>

    En plus de pouvoir remplacer certains paramètres de configuration de l&apos;agent, vous pouvez définir d&apos;autres paramètres de configuration par demande avec leur clé d&apos;environnement WSGI :

    <CollapserGroup>
      <Collapser id="set_background_task" title="newrelic.set_background_task">
        Si la valeur est `true`, cette transaction Web sera signalée comme une [transaction non Web](/docs/apm/transactions/intro-transactions/monitor-background-processes-other-non-web-transactions).
      </Collapser>

      <Collapser id="ignore_transaction" title="newrelic.ignore_transaction">
        Si la valeur est `true`, cette transaction Web ne sera pas signalée.
      </Collapser>

      <Collapser id="suppress_apdex_metric" title="newrelic.suppress_apdex_metric">
        Si défini sur `true`, aucune métrique Apdex ne sera générée pour cette transaction Web.
      </Collapser>

      <Collapser id="suppress_transaction_trace" title="newrelic.suppress_transaction_trace">
        Si défini sur `true`, cette transaction Web ne peut pas être enregistrée dans une [trace de transaction](/docs/apm/transactions/transaction-traces/transaction-traces).
      </Collapser>

      <Collapser id="disable_browser_autorum" title="newrelic.disable_browser_autorum">
        Si défini sur `true`, cela désactive l&apos;insertion automatique de l&apos;en-tête/pied de page JavaScript pour le temps de chargement de la page (parfois appelé monitoring des utilisateurs réels ou RUM). Applicable uniquement si l&apos;insertion automatique est [disponible pour votre frameworkWeb](/docs/agents/python-agent/supported-features/page-load-timing-python#restrictions_on_instrumentation).
      </Collapser>
    </CollapserGroup>

    <Callout variant="important">
      L&apos;utilisation d&apos;un middleware WSGI pour définir ces valeurs ne fonctionnera pas lorsque le wrapper d&apos;application WSGI de l&apos;agent Python a été appliqué à une portée externe. Dans ces cas, vous devez effectuer des appels à l&apos; API d&apos; agent pour obtenir le même résultat.
    </Callout>
  </Collapser>
</CollapserGroup>

## Configuration d&apos;environnements multiples [#config-file-deployment-environments]

L&apos;agent lit sa configuration principale à partir d&apos;une section de configuration d&apos;agent appelée `[newrelic]` pour les fichiers `ini` ou `[tool.newrelic]` pour les fichiers `.toml` . Vous pouvez fournir des remplacements pour des environnements de déploiement spécifiques (par exemple, développement, simulation, production) dans des sections supplémentaires. Préfacez ces sections avec `[newrelic:environment]` pour les fichiers `.ini` ou `[tool.newrelic.env.environment]` pour les fichiers `.toml` (où `environment` est remplacé par le nom de votre environnement).

Pour spécifier que l&apos;agent doit utiliser une configuration basée sur l&apos;environnement, utilisez l&apos;une de ces méthodes :

* Lorsque vous appelez [`newrelic.agent.initialize()`](/docs/agents/python-agent/customization-extension/python-management-api#mgmt), indiquez le nom de l’environnement comme deuxième argument.

  OU

* Définissez la variable d’environnement `NEW_RELIC_ENVIRONMENT` sur le nom de l’environnement.

Si aucun environnement n&apos;est spécifié, l&apos;agent utilisera les paramètres par défaut spécifiés dans la section de configuration de l&apos;agent `newrelic` .

La structure de base du fichier de configuration est :

<Collapser className="freq-link" id="config-file-format-example-ini" title="Format du fichier de configuration « .ini »">
  ```ini
  [newrelic]
  ... default settings

  [newrelic:development]
  ... override settings

  [newrelic:staging]
  ... override settings

  [newrelic:production]
  ... override settings
  ```
</Collapser>

<Collapser className="freq-link" id="config-file-format-example-toml" title="Format du fichier de configuration `.toml`">
  ```ini
  [tool.newrelic]
  ... default settings

  [tool.newrelic.env.development]
  ... override settings

  [tool.newrelic.env.staging]
  ... override settings

  [tool.newrelic.env.production]
  ... override settings
  ```
</Collapser>

    Lorsque le traceur de transaction est [activé](#txn-tracer-enabled), l&apos;agent peut enregistrer des instructions SQL. L&apos;enregistreur dispose de trois modes : `off` (n&apos;envoie aucun SQL), `raw` (envoie l&apos;instruction SQL dans sa forme d&apos;origine) et `obfuscated` (supprime les littéraux numériques et de chaîne).

    La plupart des frameworks Web (y compris Django) paramétrent les requêtes SQL afin qu&apos;elles ne contiennent pas réellement les valeurs utilisées pour remplir la requête. Si vous utilisez `raw` le mode avec l’un de ces frameworks, l’ Python agent ne verra que le SQL avant l’insertion des valeurs. Le SQL paramétré ressemblera beaucoup au mode `obfuscated` .

    <Callout variant="important">

    Pour les fonctions ou méthodes spécifiées, l&apos;agent capturera une instrumentation de synchronisation de fonction supplémentaire. Spécifiez ces noms sous la forme `module:function` ou `module:class.function`.

    Le globbing (caractères génériques) pour les noms de fonctions et de classes est possible à l&apos;aide de modèles pris en charge par le module [fnmatch](https://docs.python.org/3/library/fnmatch.html) . Les chemins de module ne sont pas pris en charge par les caractères génériques. Spécifiez les modèles sous la forme `module:function*` ou `module:class.*`.

    Par exemple, si vous souhaitez ajouter le tracing de fonction à toutes les fonctions de validation dans le fichier ci-dessous :

    <b>
      my-app/common/utils.py
    </b>

    ```py
    def validate_credentials():
    …
    def validate_status():
    …
    def format_message():
    …
    ```

    Ajoutez la ligne suivante au fichier de configuration de l&apos;agent pour inclure le tracing des fonctions dans toutes les fonctions de validation dans `my-app/common/utils.py` en utilisant des caractères génériques.

    <b>
      my-app/newrelic.ini
    </b>

    ```ini
    [newrelic]
    ...
    transaction_tracer.function_trace = common.utils:validate*
    ```

    <Callout variant="important">
      Le wilcarding nécessite [la version 6.4.4.161 ou supérieure de l&apos;agent Python](/docs/agents/python-agent/installation-configuration/upgrade-python-agent).
    </Callout>
  </Collapser>

## Paramètres de tracing distribué [#dt-main]

## Configuration de récolte d&apos;événement [#event-harvest-config]

Les paramètres de récolte d&apos;événements limitent la quantité de données de type d&apos;événement envoyées à New Relic. Lorsque vous utilisez ces paramètres, tenez compte de ces points importants :

* Les paramètres de récolte d&apos;événements affectent les limites d&apos;une seule instance de l&apos; agent, et non de l&apos;ensemble application. Consultez l’exemple d’utilisation ci-dessous pour savoir comment définir des limites pour l’ensemble d’une application.
* [Le streaming temps réel](/docs/agents/manage-apm-agents/agent-data/real-time-streaming) envoie des données toutes les cinq secondes (12 fois par minute), mais les paramètres de récolte d&apos;événements affectent toujours le débit en événement par <DNT>**minute**</DNT>. L&apos;activation ou la désactivation de la diffusion en temps réel ne nécessite pas de modifier ces paramètres.
* Avec le streaming en temps réel (activé par défaut), New Relic affichera les limites de récolte d&apos;événements pour l&apos;entité par intervalles de cinq secondes. Cela signifie, par exemple, que lorsque vous définissez une valeur limite de 1 200 dans le fichier de configuration, vous la verrez comme 100 dans New Relic.

### Exemple d&apos;utilisation

Disons qu&apos;une application est déployée sur 10 hôtes, chacun exécutant quatre processus par hôte. Pour limiter le nombre d&apos;événements SPAN à 10 000 événements par minute pour l&apos;ensemble application, divisez ce nombre par 10 hôtes. Divisez ensuite à nouveau par quatre processus par hôte.

10000 / (10 \* 4) = 250

Sur la base de ce calcul, le réglage final est :

span\_events.max\_échantillons\_stockés = 250

<Callout variant="tip">
  En raison de la manière dont New Relic collecte les données (12 fois par minute), si le nombre de données d&apos;événement est inférieur à 12, il s&apos;affichera comme 0 dans New Relic.
</Callout>

## Erreurs configuration de la boîte de réception [#errors-inbox-configuration]

La définition de l’une des balises suivantes vous aidera à identifier les versions de votre logiciel qui génèrent les erreurs.

* `NEW_RELIC_METADATA_SERVICE_VERSION` créera tag.service.version sur les données d&apos;événement contenant la version de votre code qui est déployée, dans de nombreux cas une version sémantique telle que 1.2.3, mais pas toujours.
* `NEW_RELIC_METADATA_RELEASE_TAG `créera tag.releaseTag sur les données d&apos;événement contenant la tag de sortie (comme v0.1.209 ou sortie-209).
* `NEW_RELIC_METADATA_COMMIT` créera tag.commit sur les données d&apos;événement contenant le commit sha. Le SHA entier peut être utilisé ou seulement les sept premiers caractères (par exemple, 734713b).

Une prochaine version de la boîte de réception des erreurs suivra automatiquement les versions de votre logiciel qui génèrent des erreurs. Toutes les données de version seront également affichées dans [CodeStream](/docs/codestream/how-use-codestream/performance-monitoring/#buildsha).

    Par défaut, l&apos;agent démarre lorsqu&apos;il reçoit la première transaction (Web ou [non Web](/docs/apm/transactions/intro-transactions/monitor-background-processes-other-non-web-transactions)). L&apos;agent mettra ensuite en pause la transaction initiale et attendra que l&apos;enregistrement soit terminé.     <Callout variant="important">

## Instrumentation intégrée [#builtin-instrumentation]

L&apos;agent Python instrumente une gamme de Python packages/modules . Cette instrumentation se produit uniquement lorsque le package/module Python cible est importé par une application.

Pour désactiver l&apos;instrumentation par défaut, fournissez une section `import-hook` spéciale correspondant au nom du module qui a déclenché l&apos;instrumentation. Réglez ensuite le paramètre `enabled` sur `false` pour désactiver l’instrumentation de ce module.

<CollapserGroup>
  <Collapser id="example-disable-mysqldb" title="Exemple : Désactivation MySQLdb de l' de la requête de base de données instrumentation">
    Ajoutez ce qui suit au fichier de configuration `.ini` :

    ```ini
    [import-hook:MySQLdb]
    enabled = false
    ```

    Ou pour `.toml` fichiers de configuration, ajoutez ce qui suit :

    ```ini
    [tool.newrelic.import-hook.MySQLdb]
    enabled = false
    ```
  </Collapser>
</CollapserGroup>
