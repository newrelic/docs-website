## Problème

Vous pouvez configurer l'agent Python pour log la sortie. Cette configuration permet à l'agent de vérifier s'il se connecte correctement à New Relic et si des erreurs se produisent. Ces informations seront utiles au support New Relic si vous rencontrez des problèmes.

## Solution

Un logging `debug` détaillé peut vous aider à résoudre les problèmes d’installation de votre agent Python standard.

<Callout variant="important">
  Certains partenaires utilisent des procédures différentes :

  * [Débogage de l'agent Python avec Heroku](/docs/agents/python-agent/hosting-services/python-agent-and-heroku#debugging-the-python-agent)
  * [Débogage de l'agent Python avec OpenShift](/docs/agents/python-agent/hosting-services/python-agent-openshift#debugging-the-python-agent)
</Callout>

Pour activer le logging `debug` :

1. Ouvrez votre <DNT>**newrelic.ini**</DNT>, généralement situé dans la hiérarchie de votre application.

2. Décommentez `#log_file = /tmp/newrelic-python-agent.log`. Assurez-vous que vous disposez des autorisations d&amp;apos;écriture sur l&amp;apos;emplacement log , en modifiant le chemin et le nom du fichier si nécessaire. S&amp;apos;il n&amp;apos;y a pas d&amp;apos;emplacement de fichier approprié, vous pouvez définir `log_file` sur `stderr`.

3. Remplacez `log_level` par `debug` (au lieu de `info`).

   <Callout variant="caution">
     Le logging à `debug` peut générer beaucoup de données très rapidement. Monitorez attentivement la taille de votre fichier log , en remplaçant `log_level` par `info` une fois le dépannage terminé.
   </Callout>

4. Enregistrez et fermez le fichier. Redémarrez votre application pour que les paramètres prennent effet.

5. Générez quelques minutes de trafic vers votre application.

6. Si vous envoyez votre fichier de log au support New Relic, joignez votre <DNT>**newrelic.ini**</DNT> à votre ticket d&amp;apos;assistance et indiquez au support votre fuseau horaire.

<Callout variant="important">
  Lorsque vous dépannagez votre agent Python New Relic, assurez-vous qu'il a été configuré pour générer un fichier log de niveau de débogage et monitorez de près la taille de votre fichier log . L'utilisation de `log_level = debug` génère beaucoup de données très rapidement. Après avoir reproduit votre problème, renvoyez le fichier de log à `log_level = info`.
</Callout>

### se connecter à un fichier [#logging-to-a-file]

L'agent utilise le module de logging Python pour générer le message de log. La sortie de l'agent doit donc être prise en compte dans la stratégie de logging globale de votre application.

Si vous n'utilisez pas le module de logging, l'agent fournit un moyen simplifié d'activer un fichier de log pour l'agent Python . Pour l'utiliser, définissez les options `log_file` et `log_level` dans le [fichier de configuration de l'agent](/docs/agents/python-agent/installation-and-configuration/python-agent-configuration).

<DNT>
  **Example:**
</DNT>

```ini
log_file = /tmp/newrelic-python-agent.log
log_level = info
```

Le chemin fourni pour `log_file` doit être accessible en écriture pour l&amp;apos;utilisateur sous lequel votre application s&amp;apos;exécute.

<Callout variant="tip">
  Si vous utilisez `Apache/mod_wsgi` l&amp;apos;utilisateur Apache a un accès restreint au système de fichiers. Créez un répertoire spécial dans lequel le fichier de log peut être placé et qui est accessible en écriture à l&amp;apos;utilisateur Apache . Il est recommandé d&amp;apos;utiliser un chemin absolu et non un chemin relatif.
</Callout>

Le niveau de log utilisé peut être l'un des suivants : `error`, `warning`, `info` ou `debug`. Dans des circonstances normales, utilisez `info`. Des options de débogage plus détaillées sont utilisées pour `debug`. N&amp;apos;utilisez pas ces options de débogage détaillées pendant une période prolongée. Ils peuvent générer une sortie excessive et le gestionnaire de fichiers log du module de logging est le gestionnaire de fichiers standard et n&amp;apos;effectue aucune rotation de fichiers log .

### logarithme de l'erreur standard [#logging-to-standard-error]

Pour certains fournisseurs d'hébergement, il peut ne pas être possible d'utiliser un fichier de log distinct pour l'agent. Configurez le module de logging pour log sur la sortie d'erreur standard. Cette sortie est capturée dans le fichier log d'erreur normal de votre mécanisme d'hébergement.

Pour ce faire dans le fichier de configuration de l’agent :

```ini
log_file = stderr
log_level = info
```

La valeur `stdout` peut également être utilisée à la place de `stderr`.

### enregistrer toutes les données ( log d'audit) [#audit-log]

Si vous devez enregistrer et afficher des informations sur [les données<DNT>**all**</DNT> transmises](/docs/agents/manage-apm-agents/configuration/log-audit-all-data-your-new-relic-agent-transmits) entre le processus de monitoring et le collecteur New Relic, vous pouvez activer [le logging d'audit](/docs/apm/agents/python-agent/configuration/python-agent-configuration/#audit-log-file) pendant de courtes périodes. Cela est utile, par exemple, pour le débogage ou l&amp;apos;audit, lorsque vous avez besoin d&amp;apos;informations détaillées sur ce qui est exactement transmis.

### Résoudre les conflits du module log [#logging-module-conflicts]

Si aucun logging n'est capturé, il peut y avoir un conflit avec la manière dont le module de logging Python est initialisé et/ou configuré. Des problèmes peuvent survenir lors de l’utilisation des fonctions suivantes :

<CollapserGroup>
  <Collapser

      logging.config.fileConfig()

  >
    Un problème qui peut survenir est lorsque l’application de monitoring utilise la fonction `logging.config.fileConfig()` pour initialiser le module de logging Python . Si le module de logging Python est initialisé avec cet appel, cela désactivera par défaut le logging de l&amp;apos;agent si l&amp;apos;agent avait déjà été initialisé avant l&amp;apos;appel.

    Si vous utilisez Python 2.6 ou supérieur et que l'appel à `logging.config.fileConfig()` est sous votre contrôle, une option consiste à modifier l&amp;apos;appel pour passer l&amp;apos;argument `disable_existing_loggers` avec la valeur de `False`. Cela empêchera que l&amp;apos;enregistreur existant soit désactivé et ainsi l&amp;apos;agent enregistreur sera laissé seul.

    ```py
    logging.config.fileConfig('logging.cfg', disable_existing_loggers=False)
    ```

    Si vous ne parvenez pas à modifier l'appel, ou qu'il ne serait pas approprié de le faire, l'alternative est de rétablir l'enregistreur installé par l'agent. Pour ce faire, une section enregistreur doit être ajoutée au fichier configuration de logging transmis à `logging.config.fileConfig()`.

    ```ini
    [loggers]
    keys = root,newrelic

    [logger_newrelic]
    qualname = newrelic
    level = INFO
    handlers =
    ```

    Dans ce cas, avec le paramètre `handlers` laissé non défini, la manière dont les messages seront enregistrés sera dictée par le gestionnaire associé à l&amp;apos;enregistreur racine. Vous pouvez également vous référer directement au gestionnaire approprié qui doit être utilisé pour log les messages.

    Un problème similaire à celui ci-dessus peut également survenir lorsque l'application de monitoring utilise `logging.config.dictConfig()`.
  </Collapser>

  <Collapser
    id="existing.loggers"
    title={<><InlineCode>
      disable_existing_loggers
    </InlineCode> argument de fonction</>
    }
  >
    Lors de l'utilisation d'un dictionnaire pour configurer le module de logging, il n'y a pas d'argument de fonction `disable_existing_loggers`. Spécifiez-le plutôt comme clé dans le dictionnaire.

    ```py
    LOGGING = {
      'disable_existing_loggers': False,
      ...
    }
    ```

    Comme précédemment, pour éviter l'héritage d'un enregistreur existant, une définition explicite d'enregistreur doit être ajoutée au dictionnaire utilisé pour configurer le module de logging.

    ```py
    LOGGING = {
      ...,

      'loggers': {
        'newrelic': {
          'level': 'INFO',
        },
        ...
      }
    }
    ```

    Pour plus d'informations, consultez la documentation [du module de logging Python](http://docs.python.org/library/logging.config.html) .
  </Collapser>
</CollapserGroup>

Reportez-vous à la documentation de tout framework ou application Web utilisé. Il peut fournir un mécanisme spécialisé pour configurer le module de logging Python. Par exemple, lors de l’utilisation de Django, l’approche du dictionnaire pour la configuration du module de logging Python est prise en charge automatiquement, les définitions étant définies dans l’attribut `LOGGING` dans le module de paramètres Django.

Pour plus d'informations, consultez la documentation [de configuration du logging Django](https://docs.djangoproject.com/en/dev/topics/logging/#configuring-logging) .

### Fichier de logde l'agent de rotation [#rotation-of-agent-log-file]

Lors de l'utilisation de l'option `log_file` dans la configuration de l&amp;apos;agent, le gestionnaire de fichiers standard du module de logging est utilisé. Cela n&amp;apos;effectue aucune rotation de fichier de log . La rotation des fichiers logs ne se fait pas automatiquement car nous ne saurons pas quel type de gestionnaire de fichiers log rotatifs vous souhaitez utiliser, de plus les gestionnaires de fichiers log rotatifs standard fournis avec Python ne sont pas nécessairement sûrs pour une configuration multi-processus. Il peut donc être nécessaire de télécharger et d&amp;apos;utiliser l&amp;apos;un des gestionnaires de fichiers log rotatifs tiers ou d&amp;apos;utiliser un système de logging dédié.

Si votre application s'exécute dans un processus unique, vous pouvez utiliser en toute sécurité l'un des gestionnaires `RotatingFileHandler` ou `TimedRotatingFileHandler` fournis avec le module de logging. Pour l&amp;apos;utiliser uniquement pour la sortie de l&amp;apos;agent Python, incluez au début de votre fichier de script ou module WSGI, mais avant l&amp;apos;importation du module `newrelic` , les éléments suivants :

```py
_LOG_FORMAT = (
    "%(asctime)s (%(process)d/%(threadName)s)" " %(name)s %(levelname)s - %(message)s"
)

_logger = logging.getLogger("newrelic")
_handler = logging.handlers.TimedRotatingFileHandler(
    "agent.log", when="midnight", backupCount=7
)
_formatter = logging.Formatter(_LOG_FORMAT)
_handler.setFormatter(_formatter)
_logger.addHandler(_handler)
_logger.setLevel(logging.INFO)
```

Ce code accède à l'instance d'enregistreur racine `newrelic`, y attache le gestionnaire de fichier de log rotatif et définit le niveau de log pour les messages à envoyer à ce fichier de log. Nous montrons également comment configurer le format du message de log , mais cela est facultatif.

Si le module de logging dans son ensemble est initialisé, toute sortie log sera propagée et enregistrée par tout gestionnaire associé à l'enregistreur racine, y compris éventuellement vers l'erreur standard. Pour éviter les doublons, configurez le gestionnaire log pour l'enregistreur racine.

Le code ci-dessus pourrait également être simplifié en utilisant la fonction `logging.fileConfig()` et un fichier de configuration. Pour plus de détails sur l&amp;apos;utilisation d&amp;apos;un fichier configuration consultez la documentation du module de logging Python sur [configuration de logging](http://docs.python.org/howto/logging.html#configuring-logging).

### Faire pivoter les logs dans une configuration multiprocessus [#multi-process-safe-log-rotation]

Les gestionnaires de fichiers log rotatifs fournis dans le module de logging standard ne sont pas entièrement sûrs à utiliser dans une configuration de serveur multi-processus. Les problèmes qui peuvent survenir sont le mélange de messages provenant de plusieurs processus et les tentatives de plusieurs processus d'effectuer une rotation du fichier log en même temps.

Pour un mécanisme de rotation de fichier de log plus robuste, utilisez un gestionnaire log tiers en conjonction avec le module de logging Python .

Une telle implémentation disponible sur PyPi est :

* [Gestionnaire de logs simultané](http://pypi.python.org/pypi/ConcurrentLogHandler/)

Pour une solution plus complexe, vous pouvez également envisager un service de logging tel que :

* [Serveur de logging Python](http://code.google.com/p/python-loggingserver/)

Consultez également la liste des [autres gestionnaires](http://docs.python.org/library/logging.handlers.html#module-logging.handlers) fournis en standard dans le module de logging Python, car ceux destinés à l&amp;apos;envoi vers un socket ou à la publication vers une URL HTTP peuvent également être des solutions raisonnables dans certaines circonstances.
