## Syntaxe

```py
newrelic.agent.function_trace(name=None, group=None, label=None, params=None, terminal=False)
```

Utilisé pour instrumenter des fonctions, des méthodes, des générateurs et des coroutines qui ne sont pas instrumentés par défaut.

## Description

`function_trace` est un décorateur permettant d&amp;apos;ajouter des fonctions, des méthodes, des générateurs et des coroutines. L&amp;apos;ajout de ce décorateur vous permet de collecter des informations [de transaction](/docs/accounts-partnerships/education/getting-started-new-relic/glossary#transaction) supplémentaires (y compris des informations trace de transaction). (Une autre façon d&amp;apos;instrumenter les fonctions sans avoir à toucher au code de votre application est de [les répertorier dans le fichier de configuration](/docs/agents/python-agent/custom-instrumentation/python-custom-instrumentation-config-file).)

`function_trace` ne vous donne pas un profil complet de toutes les fonctions qui se produisent dans une fonction décorée. Ce qu&amp;apos;il fait, c&amp;apos;est ajouter la fonction décorée en tant que nœud dans l&amp;apos;UI New Relic et capturer le temps passé dans cette fonction. Si vous avez besoin de plus de détails sur ce qui se passe dans cette fonction, vous devrez également appliquer la trace de fonction aux fonctions enfants de la fonction.

Le décorateur `function_trace` peut être utilisé sur les générateurs et les coroutines avec la version d&amp;apos;agent 2.102.0.85 ou supérieure. La synchronisation de ces objets commence lorsque la consommation commence et se termine lorsque l&amp;apos;objet est épuisé ou sort du champ d&amp;apos;application. Il s’agit d’un changement par rapport aux versions précédentes où la métrique représentait le temps nécessaire à la création du générateur ou de l’objet coroutine lui-même.

Vous pouvez utiliser le décorateur en conjonction avec les décorateurs existants, y compris ceux pour les méthodes statiques et de classe. Lors de l&apos;imbrication de plusieurs décorateurs, choisissez le décorateur `function_trace` comme décorateur le plus externe.

<Callout variant="important">
  Pour tous les décorateurs qui sont encapsulés, utilisez `functools.wraps()` de la bibliothèque standard Python dans leur implémentation. (Ou, sinon, assurez-vous que le décorateur propage l&amp;apos;attribut de nom correct de l&amp;apos;objet enveloppé le plus à l&amp;apos;intérieur requis pour permettre une introspection de nom correcte.) Si cela n&amp;apos;est pas fait, lorsque la métrique est signalée, le nom de la fonction décoratrice imbriquée (et non la fonction encapsulée la plus interne) sera utilisé.
</Callout>

## Formes d&apos;appel alternatives [#alternate]

Pour les configurations où vous ne pouvez pas utiliser le décorateur, ces formulaires d&apos;appel alternatifs sont disponibles :

### Le gestionnaire de contexte [#context-manager]

Le gestionnaire de contexte `FunctionTrace` est utilisé lorsque le paramètre à transmettre ne peut pas être déterminé avant l&amp;apos;exécution. (Par exemple, vous ne connaissez peut-être pas le nom d’une fonction au moment de l’importation.) Lorsque vous utilisez le gestionnaire de contexte pour nommer une métrique, vous devez toujours fournir le nom et le préfixe du chemin de la métrique.

La classe `FunctionTrace` implémente le gestionnaire de contexte et est utilisée conjointement avec l&amp;apos;instruction `with` . La classe `FunctionTrace` est la primitive de niveau le plus bas disponible pour tracer le temps par rapport à une transaction.

L&apos;instruction `with` définit les limites de ce qui est chronométré et non une fonction unique. Ainsi, le gestionnaire de contexte pourrait être appliqué à un bloc de code arbitraire. Le bloc de code peut donc contenir des appels à plusieurs fonctions, ou il peut s&amp;apos;agir d&amp;apos;un bloc de code autonome implémentant un algorithme orienté temporel que vous souhaitez suivre.

<Callout variant="important">
  Évitez de tracer des blocs de code appelés un nombre excessif de fois. Par exemple, ne l’utilisez pas dans le contexte d’une boucle qui s’exécute plusieurs fois. Les données générées entraîneront une perte de performances. Une meilleure solution est de fermer la boucle.
</Callout>

### L&apos;emballage

Si vous savez à l&apos;avance où se trouvent les fonctions spécifiques que vous souhaitez trace, vous pouvez utiliser le décorateur `function_trace`. Cependant, si vous <DNT>**don&apos;t**</DNT> connaissez toutes les fonctions qui doivent être tracées (par exemple, si elles sont recherchées dynamiquement dans le cadre d&amp;apos;un système de routage), vous devez alors utiliser `FunctionTraceWrapper` pour encapsuler la fonction au moment de l&amp;apos;enregistrement ou au moment de l&amp;apos;appel.

### Enveloppement basé sur le chemin [#path-based]

`wrap_function_trace` est utilisé pour encapsuler des fonctions en dehors du code dans lequel elles sont déclarées. Par exemple : vous pouvez l&amp;apos;utiliser pour instrumenter le code de la bibliothèque que vous ne souhaitez pas modifier.

Pour en savoir plus sur les différences entre ces formats d’appel, voir [Différents formats d’appel](/docs/python-agent-api-different-call-forms).

## Exemples

### `function_trace` [#trace\_example][#trace_example]

Un exemple d&apos;utilisation du décorateur `function_trace` :

```py
import newrelic.agent

class _Database(UserDict.DictMixin):

    ...

    @newrelic.agent.function_trace()
    def _commit(self):
        ...

@newrelic.agent.function_trace()
def open(file, flag=None, mode=0666):
...
```

Un exemple d&apos;utilisation du décorateur `function_trace` avec des coroutines natives :

```py
import newrelic.agent

class _Database(UserDict.DictMixin):

    ...

    @newrelic.agent.function_trace()
    async def _commit(self):
        ...

@newrelic.agent.function_trace()
async def open(file, flag=None, mode=0666):
...
```

### Exemple de gestionnaire de contexte [#context-mgr-example]

Un exemple d&apos;utilisation du gestionnaire de contexte `FunctionTrace` :

```py
import newrelic.agent

def dispatch_request(self, request):
    adapter = self.url_map.bind_to_environ(request.environ)
    try:
        endpoint, values = adapter.match()
        function = getattr(self, 'on_' + endpoint)
        with newrelic.agent.FunctionTrace(
                endpoint, 'Python/EndPoint'):
            return function(request, <DNT>**values)
    except HTTPException as e:
        return e
```

### Exemple d&apos;enveloppe [#trace-wrap-example]

Un exemple d&apos;utilisation du `FunctionTraceWrapper`:

Le routage d&apos;URL avec Werkzeug génère un nom qui est d&apos;abord utilisé pour rechercher dynamiquement une méthode d&apos;une classe à l&apos;aide de `getattr()` et le résultat est ensuite invoqué :

```py
def dispatch_request(self, request):
    adapter = self.url_map.bind_to_environ(request.environ)
    try:
        endpoint, values = adapter.match()
        return getattr(self, 'on_' + endpoint)(request, **</DNT>values)
    except HTTPException as e:
        return e
```

Si vous souhaitez trace la fonction point de terminaison, vous pouvez la réécrire comme suit :

```py
import newrelic.agent

def dispatch_request(self, request):
    adapter = self.url_map.bind_to_environ(request.environ)
    try:
        endpoint, values = adapter.match()
        function = getattr(self, 'on_' + endpoint)
        function = newrelic.agent.FunctionTraceWrapper(function)
        return function(request, <DNT>**values)
    except HTTPException as e:
        return e
```

Si vous souhaitez nommer la métrique d&apos;après le nom du point de terminaison (plutôt que de nommer la métrique en fonction de l&apos;identifiant de la fonction appelée), vous pouvez fournir le nom à utiliser ainsi qu&apos;un préfixe de chemin de métrique alternatif lors de la création de l&apos;objet `FunctionTraceWrapper` .

```py
import newrelic.agent

def dispatch_request(self, request):
    adapter = self.url_map.bind_to_environ(request.environ)
    try:
        endpoint, values = adapter.match()
        function = getattr(self, 'on_' + endpoint)
        function = newrelic.agent.FunctionTraceWrapper(
                function, name=endpoint, group='Python/EndPoint')
        return function(request, **</DNT>values)
    except HTTPException as e:
        return e
```

Pour cet exemple, si le point de terminaison était appelé `help`, la métrique finale serait :

```
Python/EndPoint/help
```

Dans la répartition des performances d&apos;une transaction, la catégorie serait `Python` et le nom du segment `EndPoint/help`. Ce nom de segment apparaîtra également dans la trace lente de la transaction comme nom de nœud.
