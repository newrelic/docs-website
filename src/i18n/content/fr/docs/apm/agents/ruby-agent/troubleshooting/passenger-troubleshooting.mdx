Lisez les conseils de dépannage suivants pour les problèmes courants lors de l’utilisation de l’agent Ruby et de Passenger.

<CollapserGroup>
  <Collapser id="agent" title="Redémarrage de l'agent">
    Si vous ne faites que modifier le fichier passager `tmp/restart.txt` , il ne récupérera pas le nouvel agent. Lorsque vous mettez à jour l&apos;agent et redémarrez Passenger, vous devez effectuer un redémarrage matériel d&apos;Apache pour que la modification prenne effet.
  </Collapser>

  <Collapser id="logs" title="Fichier de logmanquant">
    Pour Passenger, le fichier de log est `newrelic_agent.passenger.log`. Recommandation : pour l&apos;utilisateur Apache , fournissez des autorisations d&apos;écriture pour l&apos;ensemble du répertoire `/log`.
  </Collapser>

  <Collapser id="developer_mode" title="mode développeur">
    Lorsque vous utilisez Passenger en mode développeurs, vous remarquerez parfois que la liste des requests peut disparaître ou qu&apos;il semble manquer certaines requests. De plus, lors de la sélection d&apos;une demande, vous pouvez voir un message d&apos;erreur indiquant que la trace n&apos;est pas disponible, suggérant peut-être que vous avez redémarré.

    Cela se produit lorsque Passenger génère plus d&apos;un processus de gestion. La trace de transaction est placée dans l&apos;un des processus, et si vous vous connectez à l&apos;autre, vous ne la verrez pas.

    La solution de contournement consiste à définir le nombre maximal d&apos;instances sur 1 en utilisant le paramètre [PassengerMaxPoolSize](http://www.modrails.com/documentation/Users%20guide%20Apache.html#_passengermaxpoolsize_lt_integer_gt) . De plus, si vous actualisez plusieurs fois, vous avez de bonnes chances de récupérer les requests .
  </Collapser>
</CollapserGroup>
