Le C SDK est conçu pour prendre en charge la nature souvent complexe et multithread des applications C/C++. Vous pouvez obtenir un nouveau niveau de visibilité pour vous aider à identifier et à résoudre les problèmes de performances. Vous pouvez également collecter et analyser des données pour vous aider à améliorer l&apos;expérience client et à prendre des décisions commerciales data-driven .

Le C SDK peut être utilisé pour instrument une large gamme d&apos;applications au-delà de C ou C++. Si votre application n&apos;utilise pas [d&apos;autres langages pris en charge par New Relic](/docs/using-new-relic/welcome-new-relic/getting-started/glossary#agent) et peut importer la bibliothèque C, vous pouvez utiliser le C SDK New Relic pour profiter de nos capacités et fonctionnalités monitoring .

## architecture : C bibliothèque et démon [#architecture]

Le C SDK s&apos;appuie sur deux composants pour envoyer des données de votre application à New Relic :

* Les appels du SDK lC : vous téléchargez cette bibliothèque, puis ajoutez les appels et l&apos;instrumentation au code de votre application. Cela vous permet d’identifier et de personnaliser les types de données qui comptent le plus pour vous.
* Le démon C SDK : il s&apos;agit d&apos;un binaire distinct qui accumule les données des appels C SDK et les envoie à New Relic. Cela agit comme un proxy entre le SDK et New Relic.

<img title="C SDK architecture" alt="C SDK architecture" src="/images/apm_diagram_C-SDK-architecture.webp" />

<figcaption>
  <DNT>**C SDK architecture:**</DNT> Pour envoyer des données de votre application à New Relic, le démon doit être appelé <DNT>**before**</DNT> en appelant la bibliothèque d&amp;apos;instrumentation C SDK de votre application.
</figcaption>

En travaillant ensemble, l&apos;instrumentation du C SDK et le démon transmettent les données à New Relic où vous pouvez afficher et interroger les données sur les transactions. Le workflow entre votre application et New Relic doit se dérouler dans cet ordre :

1. Un lien HTTPS est établi entre le démon et New Relic. Le démon doit être invoqué en premier, avant que votre application instrumentée ne soit invoquée.
2. Ensuite, une communication socket est établie entre votre application instrumentée et le démon. Cela se produit après des appels réussis à [`newrelic_new_app_config()`](https://newrelic.github.io/c-sdk/libnewrelic_8h.html#af067b3ff89a40b6fa417e18be5e847aa) et [`newrelic_create_app()`](https://newrelic.github.io/c-sdk/libnewrelic_8h.html#a61dd90439ae3cc5060021f6ab4701132).

L&apos;appel à [`newrelic_create_app()`](https://newrelic.github.io/c-sdk/libnewrelic_8h.html#a61dd90439ae3cc5060021f6ab4701132) est non bloquant. Son deuxième paramètre vous permet de spécifier un délai d&amp;apos;attente pour que votre application instrumentée soit correctement établie. Par exemple:

```c
newrelic_app_t* app = newrelic_create_app(config, 10000);
```

Si votre application instrumentée envoie des transactions avant que la connexion démon et la communication socket de votre application ne soient établies, les données signalées par votre application seront perdues.
