Informations sur les exigences en matière de données [de l&apos;API Trace](/docs/apm/distributed-tracing/trace-api/introduction-new-relic-trace-api), notamment :

* Spécifications des données et limites maximales
* Métadonnées obligatoires (en-têtes, paramètres de requête)
* Détails de validation de la réponse

Ce document s’applique à l’ensemble de l’API de trace. Pour les règles concernant les formats de données spécifiques, voir :

* [Données de trace au format New Relic](/docs/apm/distributed-tracing/trace-api/report-new-relic-format-traces-trace-api)
* [Données de trace au format Zipkin](/docs/apm/distributed-tracing/trace-api/report-zipkin-format-traces-trace-api)

## point de terminaison [#requirements]

Toutes les données trace sont envoyées via HTTPS POST à un point de terminaison d&apos;API de trace. Nous avons quelques points de terminaison, en fonction de votre configuration :

* Points de trace par défaut de terminaison d&apos;API : `https://trace-api.newrelic.com/trace/v1`
* Centre de données de l&apos;UE : `https://trace-api.eu.newrelic.com/trace/v1` (voir autre [point de terminaison de l&apos;UE](/docs/using-new-relic/welcome-new-relic/getting-started/introduction-eu-region-data-center#endpoints)).
* [Infinite Tracing](/docs/understand-dependencies/distributed-tracing/infinite-tracing/introduction-infinite-tracing): lorsque vous terminez la [configuration de l’observateur de trace](/docs/understand-dependencies/distributed-tracing/infinite-tracing/set-trace-observer), vous obtenez une valeur YOUR\_TRACE\_OBSERVER\_URL personnalisée à utiliser comme point de terminaison. Si vous utilisez une intégration qui utilise l&amp;apos;API de trace (par exemple, [le reporter Kamon](/docs/more-integrations/open-source-telemetry-integrations/kamon/kamon-reporter/#install)), vous devez configurer cette intégration avec ce point de terminaison. Vous souhaiterez également ajuster l&amp;apos; [échantillonnage de votre service de tracing](/docs/understand-dependencies/distributed-tracing/get-started/how-new-relic-distributed-tracing-works#tail-based) pour nous envoyer 100 % des portées.
* FedRAMP : Voir [point de terminaison FedRAMP](/docs/security/security-privacy/compliance/fedramp-compliant-endpoints/#trace-api).

## Formats de données [#data-formats]

Actuellement, l&apos;API de trace accepte deux types de formats de données :

* [`zipkin`](/docs/apm/distributed-tracing/trace-api/report-zipkin-format-traces-trace-api): pour signaler les données de trace Zipkin. Les données Zipkin doivent être [Zipkin JSON v2](https://zipkin.io/zipkin-api/#/default/post_spans).
* [`newrelic`](/docs/apm/distributed-tracing/trace-api/report-new-relic-format-traces-trace-api): pour signaler toutes les autres données de trace.

## Validation des réponses


* `NrIntegrationError` événement (asynchrone). Les erreurs liées à la charge JSON ou d&amp;apos;autres erreurs sémantiques sont signalées de manière asynchrone via [l&apos;événement`NrIntegrationError` ](/docs/telemetry-data-platform/manage-data/nrintegrationerror)qui est stocké dans le compte dont <InlinePopover type="licenseKey" />est associé à la requête. Pour toutes les erreurs de ce type, l&amp;apos;attribut `newRelicFeature` sera `Distributed Tracing` et `requestId` sera le `requestId` de la réponse du point de terminaison.

Si vous recevez une réponse `202` et ne voyez pas d&amp;apos;événement `NrIntegrationError`, vos données devraient être visibles dans notre [UI de tracing distribué](https://one.newrelic.com/launcher/distributed-tracing-nerdlets.distributed-tracing#launcher=eyJ0aW1lUmFuZ2UiOnsiYmVnaW5fdGltZSI6bnVsbCwiZW5kX3RpbWUiOm51bGwsImR1cmF0aW9uIjoxODAwMDAwfX0=&pane=eyJuZXJkbGV0SWQiOiJkaXN0cmlidXRlZC10cmFjaW5nLW5lcmRsZXRzLmRpc3RyaWJ1dGVkLXRyYWNpbmctbGF1bmNoZXIiLCJzb3J0SW5kZXgiOjAsInNvcnREaXJlY3Rpb24iOiJERVNDIn0=) globale dans environ une minute. Vous devriez pouvoir trouver la trace en utilisant une [recherche de trace](/docs/apm/distributed-tracing/ui-data/additional-distributed-tracing-features-new-relic-one#find-data) standard comme :

```
traceId = TRACE_ID_SENT
```
