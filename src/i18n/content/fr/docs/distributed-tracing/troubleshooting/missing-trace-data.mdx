## Problème

Vous avez activé [le tracing distribué,](/docs/apm/distributed-tracing/getting-started/introduction-distributed-tracing) mais les données que vous vous attendiez à voir n&amp;apos;apparaissent pas dans UI du tracing distribué de New Relic.

## Avant de commencer [#before]

Avant de procéder au dépannage, il peut être utile de :

* Consultez les détails techniques du [fonctionnement du tracing distribué](/docs/apm/distributed-tracing/getting-started/how-new-relic-distributed-tracing-works).
* Utilisez notre [application d'état d'instrumentation](https://one.newrelic.com/dt-setup-assistant) pour voir un aperçu de votre instrumentation de trace, y compris les taux d&amp;apos;échantillonnage. Cela peut vous aider à comprendre les étendues manquantes et les traces fragmentées. Pour le trouver, allez à : <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &amp;gt; Apps &amp;gt; Distributed tracing instrumentation status**</DNT>.

## Solutions

Voici quelques causes et solutions pour les données de trace manquantes :

### Problèmes d'activation ou d'instrumentation [#enable]

<CollapserGroup>
  <Collapser id="not-enabled" title="Tracés manquants en raison du fait que le service n'a pas activé le tracing distribué">
    Pour que le tracing distribué signale les détails de tous les nœuds d'une trace, chaque application doit être monitorée par un agent New Relic sur lequel [le tracing distribué est activé](/docs/apm/distributed-tracing/enable-configure/enable-distributed-tracing).

    Si le tracing distribué n'est pas activé sur le compte New Relic d'une application, les problèmes suivants se produiront :

    * Sa page d' UI de tracing distribué n'aura pas de données.
    * Il ne signalera pas de données aux traces distribuées d'autres comptes.
  </Collapser>

  <Collapser id="manual-instrumentation" title="Les applications/services manquants peuvent nécessiter une instrumentation manuelle">
    Lorsque vous activez le tracing distribué pour les applications et les services que New Relic instrumente automatiquement, vous verrez généralement des données complètes et détaillées pour ces nœuds dans l'UI de tracing distribué.

    Cependant, vous remarquerez peut-être que certains services ou applications sont manquants dans la trace, ou que certaines étendues internes que vous vous attendez à voir sont manquantes. Si tel est le cas, vous souhaiterez peut-être implémenter [instrumentation personnalisée](/docs/apm/distributed-tracing/enable-configure/enable-distributed-tracing#agent-apis) des applications ou des transactions spécifiques pour voir plus de détails dans la trace. Voici quelques exemples de situations dans lesquelles vous pourriez avoir besoin de procéder ainsi :

    * <DNT>**Transactions not automatically instrumented**</DNT>Pour vous assurer que votre application est automatiquement instrumentée, lisez la [documentation de compatibilité et des exigences](/docs/apm/new-relic-apm/getting-started/introduction-apm) de l&amp;apos;agent New Relic que vous utilisez. Si une application n&amp;apos;est pas automatiquement instrumentée, ou si vous souhaitez ajouter instrumentation d&amp;apos;activité spécifique, consultez [instrumentation personnalisée](/docs/apm/agents/manage-apm-agents/agent-data/custom-instrumentation/#implementing).
    * <DNT>**All Go applications**</DNT>L&amp;apos;agent Go, contrairement à d&amp;apos;autres agents, nécessite une instrumentation manuelle de votre code. Pour obtenir des instructions, consultez [Instrumenter une application Go](/docs/agents/go-agent/instrumentation/instrument-go-transactions).
    * <DNT>**A service doesn't use HTTP**</DNT>Si un service ne communique pas via HTTP, l&amp;apos;agent New Relic n&amp;apos;enverra pas d&amp;apos;en-têtes de tracing distribué. Cela peut être le cas pour certaines applications non Web ou fichiers d&amp;apos;attente de messages. Pour remédier à cela, utilisez les [API de tracing distribué](/docs/apm/distributed-tracing/enable-configure/enable-distributed-tracing#agent-apis) pour instrument l’application appelante ou appelée.
  </Collapser>
</CollapserGroup>

  <Collapser id="export" title="La plage manquante n'est pas exportée">
    Parfois, la propagation de l'en-tête réussit, mais les informations de portée ne sont pas envoyées à New Relic. Par exemple, si OpenTelemetry n’est pas instrumenté avec un exportateur New Relic, les détails de la plage ne parviennent jamais à New Relic.

    Dans ce diagramme, notez que la propagation de l'en-tête est réussie, mais aucun exportateur n'est configuré dans le service 2 pour envoyer le span à New Relic :

    <img title="missing-exporter.png" alt="Diagram showing missing exporter." src="/images/distributed-tracing_diagram_missing-exporter.webp" />

    Le diagramme suivant montre également une propagation d'en-tête réussie, mais il inclut un exportateur dans le service 2 qui envoie les détails de l'étendue à New Relic (voir [API Trace](/docs/understand-dependencies/distributed-tracing/trace-api/introduction-trace-api)) :

    <img title="exporter-deployed.png" alt="Diagram showing a deployed exporter." src="/images/distributed-tracing_diagram_exporter-deployed.webp" />
  </Collapser>

  <Collapser id="sampling" title="Portées manquantes en raison du processus d'échantillonnage">
    Le tracing distribué standard pour <InlinePopover type="apm" />utilise [l'échantillonnage adaptatif](/docs/apm/distributed-tracing/getting-started/how-new-relic-distributed-tracing-works#sampling). Cela signifie qu&amp;apos;un pourcentage d&amp;apos;appels à un service sera signalé dans le cadre d&amp;apos;une trace distribuée. Il est possible que des appels spécifiques à votre service n&amp;apos;aient pas été sélectionnés pour être échantillonnés.
  </Collapser>

  <Collapser id="agent-limits" title="Travées manquantes en raison de limites de portée dépassées">
    Il existe [des limites quant au nombre de plages pouvant être collectées et affichées](/docs/apm/distributed-tracing/ui-data/understand-use-distributed-tracing-data#1k-limit). Si une application génère un très grand nombre d&amp;apos;étendues pour un seul appel, elle peut dépasser la limite de collecte d&amp;apos;étendues de l&amp;apos;agent APM pour ce cycle de collecte. Cela pourrait entraîner des pertes de données et limiter considérablement le nombre de traces que l&amp;apos;agent peut entièrement échantillonner et signaler.

    Nous n'affichons actuellement que 10 000 spans à la fois.
  </Collapser>

  <Collapser id="late-spans" title="Travées manquantes en raison de travées envoyées en retard">
    Les intervalles doivent être envoyés dans les soixante dernières minutes pour être capturés dans un index de trace. Si vous envoyez des données datant de plus de soixante minutes mais de plus d'un jour, les données seront toujours écrites. Cependant, il ne sera pas intégré à l'index trace, qui contrôle la liste trace dans l'UI de tracing distribué. Si une plage a un horodatage plus ancien qu'un jour, elle sera supprimée. Cela se produit souvent lorsqu'il y a un décalage d'horloge (différences de synchronisation) entre les tâches système ou les tâches d'arrière-plan de longue durée.

    <Callout variant="important">
      Toutes les périodes envoyées via le [protocole OpenTelemetry](https://github.com/open-telemetry/opentelemetry-specification/blob/main/specification/protocol/otlp.md) (OTLP en abrégé) d&amp;apos;une durée supérieure à soixante minutes ne seront pas écrites dans [NRDB](/docs/data-apis/get-started/nrdb-horsepower-under-hood/), ce qui produira l&amp;apos;erreur NrIntegrationError suivante :

      ```
      The span timestamp cannot be older than 60 minutes.
      ```
    </Callout>
  </Collapser>
</CollapserGroup>

### Problèmes avec les détails de la trace

<CollapserGroup>
  <Collapser id="middleware-failure" title="Le middleware ne reconnaît pas l'en-tête propriétaire New Relic">
    Si vos transactions n'envoient que l'en-tête propriétaire New Relic, certains intergiciels peuvent ne pas reconnaître le format et supprimer les informations comme indiqué dans ce diagramme :

    <img title="broken-trace.png" alt="Diagram showing dropped header information." src="/images/distributed-tracing_diagram_broken-trace.webp" />

    Une solution consiste à mettre à niveau votre agent New Relic vers une version prenant en charge le contexte de trace W3C. Dans le diagramme ci-dessous, l'agent New Relic conforme au W3C transmet l'en-tête précédent ainsi que deux en-têtes standardisés :

    <img title="upgraded-agent.png" alt="Diagram showing a W3C compatible New Relic agent." src="/images/distributed-tracing_diagram_upgraded-agent.webp" />
  </Collapser>

  <Collapser id="proxy" title="Un intermédiaire est manquant ou ne transmet pas le contexte de trace">
    Quelques problèmes potentiels avec les proxys et autres intermédiaires :

    * <DNT>**Incomplete trace.**</DNT> Certains intermédiaires ne propageront pas automatiquement l&amp;apos; [en-tête de tracing distribué](/docs/apm/distributed-tracing/getting-started/how-new-relic-distributed-tracing-works#headers). Dans ce cas, vous devez configurer ce composant pour permettre à l’en-tête d’être transmis de la source à la destination. Pour obtenir des instructions, consultez la documentation de ce composant intermédiaire.

    * <DNT>**Missing intermediary in trace.**</DNT> Si l&amp;apos;intermédiaire est New Relic-monitorer, assurez-vous qu&amp;apos;il propage l&amp;apos;en-tête `newrelic` généré ou mis à jour par l&amp;apos;agent New Relic exécuté sur cet intermédiaire. Cela peut se manifester lorsqu&amp;apos;un intermédiaire était auparavant visible dans la trace, mais a disparu après l&amp;apos;activation du tracing distribué pour une entité en amont (par exemple, une application de monitoring de navigateur).

      <Callout variant="tip">
        Si une entité signale des données trace à un autre système de tracing, vous pouvez utiliser l'ID trace de l'UI de New Relic pour rechercher les étendues manquantes dans d'autres systèmes de tracing.
      </Callout>
  </Collapser>

  <Collapser id="mixed-sourcees" title="Coudre ensemble des travées provenant de sources mixtes">
    Si chaque agent d'une chaîne prend en charge W3C Trace Context, nous pouvons alors assembler les segments dans une trace complète. Si une partie de la chaîne provient d'un agent, tel que Zipkin, qui ne prend pas en charge W3C Trace Context, les étendues provenant de cet agent peuvent ne pas être incluses dans la trace.
  </Collapser>

  <Collapser id="account-access" title="Les détails de la trace sont obscurcis">
    Si une trace contient des données provenant du moniteur d'applications de plusieurs comptes New Relic et que vos autorisations d'utilisateur ne vous permettent pas d'accéder à ces comptes, certains détails de l'étendue et du service seront [masqués dans l'UI](/docs/apm/distributed-tracing/ui-data/understand-use-distributed-tracing-data#account-access).

    Par exemple, vous pouvez voir une série d'astérisques (\*\*\*\*\*) au lieu du nom du service dans votre liste de tracing distribué si vous n'avez pas accès au compte lié au service.
  </Collapser>

  <Collapser id="mismatched-trace-details" title="les informations de la liste de suivi et les détails trace ne correspondent pas">
    La liste trace est générée par des index trace, qui sont capturés dans une fenêtre de vingt minutes à compter de la réception des premières traces.

    En général, cela est dû à [des retards](#late-spans).
  </Collapser>

  <Collapser id="long-traces-short-backend" title="Trace longue avec des temps backend courts">
    Si vous constatez des temps backend inhabituellement courts pour une trace longue, il s'agit probablement d'un problème lié à l'horodatage envoyé.

    Par exemple, la portée racine peut consister à republier des microsecondes sous forme de millisecondes. Cela peut également se produire si la racine est une application de navigateur. Lorsque vous utilisez un client externe comme un navigateur Web, vous risquez de rencontrer plus souvent des décalages d'horloge (différences de synchronisation).
  </Collapser>
</CollapserGroup>

### Problèmes avec les applications de navigateur [#browser]

<CollapserGroup>
  <Collapser id="spans-browser" title="Travées et transactions manquantes après l'activation d'une application de navigateur">
    Les anciennes versions de certains agents <InlinePopover type="apm" />sont incompatibles avec le tracing distribué pour les applications de navigateur. Si l&amp;apos;application de navigateur envoie une requête AJAX à une application APM exécutant un agent incompatible, l&amp;apos;agent APM peut ne pas enregistrer les données de transaction et d&amp;apos;étendue pour cette requête.

    Si le tracing distribué est activé pour une application de navigateur et que vous ne voyez pas de données de transaction ou d'étendue pour requests APM en aval, examinez les [données de navigateur dans les exigences de tracing distribué](/docs/browser/new-relic-browser/browser-pro-features/browser-data-distributed-tracing#requirements) et confirmez que toutes les applications exécutent des versions prises en charge de l&amp;apos;agent APM.
  </Collapser>

  <Collapser id="browser-spans-missing" title="Ne pas voir les étendues finales de l'utilisateur de l'application du navigateur">
    Si la trace semble manquer de portées finales d'utilisateur, assurez-vous d'avoir lu et compris les [exigences](/docs/browser/new-relic-browser/browser-pro-features/browser-data-distributed-tracing#requirements) de tracing distribué du navigateur et [les procédures d'activation](/docs/browser/new-relic-browser/browser-pro-features/browser-data-distributed-tracing#enable).

    Sur la [page UI <DNT>**AJAX**</DNT> ](/docs/browser/new-relic-browser/browser-pro-features/ajax-page-identifying-time-consuming-calls), il existe des liens vers l&amp;apos;UI de tracing distribué, indépendamment du fait qu&amp;apos;il existe ou non des étendues finales d&amp;apos;utilisateur présentes dans cette trace. Pour plus de détails sur les données qui génèrent des étendues, voir [Exigences](/docs/browser/new-relic-browser/browser-pro-features/browser-data-distributed-tracing#requirements).
  </Collapser>

  <Collapser id="orphan-browser-spans" title="Les étendues Browser ne sont pas connectées à d'autres étendues">
    Les anciennes versions de certains agents <InlinePopover type="apm" />sont incompatibles avec le tracing distribué pour les applications de navigateur. Si les étendues APM manquent systématiquement dans la trace qui inclut les applications de navigateur, veuillez vous référer aux [données du navigateur dans le tracing distribué des exigences](/docs/browser/new-relic-browser/browser-pro-features/browser-data-distributed-tracing#requirements) et confirmer que toutes les applications exécutent des versions prises en charge de l&amp;apos;agent APM.

    Pour d'autres causes d'étendues de navigateur orphelines, voir [Rapport d'étendueBrowser ](/docs/understand-dependencies/distributed-tracing/get-started/how-new-relic-distributed-tracing-works#browser-spans).
  </Collapser>
</CollapserGroup>

### Autres problèmes [#other]

<CollapserGroup>
  <Collapser id="multiple-app-names" title="La recherche par entity.name ne trouve pas les noms d'applications associés">
    Cause potentielle : pour les applications qui ont [plusieurs noms d’application](/docs/agents/manage-apm-agents/app-naming/use-multiple-names-app), l’attribut `entity.name` sera associé uniquement au nom d’application principal. Pour rechercher par d’autres noms d’application, effectuez une recherche à l’aide de l’attribut `appName` .
  </Collapser>

  <Collapser id="supporting-open-telemetry" title="Prise en charge d'OpenTelemetry">
    Les questions concernant l'intégration avec OpenTelemetry doivent être adressées au [Forum d'assistance](https://discuss.newrelic.com).
  </Collapser>
</CollapserGroup>
