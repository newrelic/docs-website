CodeStream signale automatiquement les problèmes de performances dans votre code, vous permettant de résoudre de manière proactive les problèmes dès qu'ils surviennent.

<img title="Transaction performance section showing poorly performing transactions" alt="A screenshot of the Transaction performance section showing poorly performing transactions." src="/images/codestream_screenshot-crop_code-level-anomalies.webp" />

CodeStream identifie les problèmes de performances en commençant par les transactions et en recherchant une anomalie dans le taux d'erreur ou la durée moyenne. Si vous utilisez [le suivi des changements](/docs/change-tracking/change-tracking-introduction/) de New Relic pour envoyer des informations de déploiement, les anomalies sont calculées en comparant les performances depuis votre dernière sortie avec les trois semaines précédentes. Dans le cas contraire, la comparaison est effectuée sur les sept derniers jours. Pour toutes les transactions identifiées comme présentant des problèmes de performances, CodeStream analyse ensuite en profondeur les segments (par exemple, requête de base de données, appels externes, méthodes individuelles, etc.) qui composent la transaction pour voir si l&amp;apos;un d&amp;apos;entre eux fonctionne mal. Cela vous permet de cibler la source du problème.

Cliquez sur n'importe quelle transaction ou segment pour voir le problème de performance représenté sur un graphique au fil du temps. Pour un taux d'erreur anormal, les erreurs contribuant à l'augmentation du taux d'erreur sont répertoriées et vous pouvez cliquer sur CodeStream l'une d'entre elles pour accéder à [l'expérience d'investigation des erreurs de .](/docs/codestream/observability/error-investigation)

<img title="Chart of anomalous error rate over time" alt="A screenshot of a chart showing error rate over time." src="/images/codestream_screenshot-crop_anomalous-errorrate.webp" />

Pour une durée moyenne anormale, CodeStream effectue une analyse du chemin critique pour répertorier les opérations les plus lentes. Et pour les opérations de base de données anormales, les requêtes réelles sont répertoriées.
