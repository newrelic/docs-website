## v10.5.0

* **Fonctionnalité : ajouter la prise en charge de Dalli 5.0 et corriger l&apos;instrumentation du méta-protocole**

  L’agent prend désormais en charge Dalli 5.0+, qui a supprimé `Dalli::Protocol::Binary` au profit exclusif du méta-protocole. Pour Dalli 3.2.0+, L&apos;instrumentation `pipelined_get` cible désormais correctement `Dalli::Protocol::Base` (où la méthode est définie) plutôt que `Dalli::Protocol::Binary`, corrigeant une lacune où les appels `get_multi` n&apos;étaient pas instrumentés lors de l&apos;utilisation du méta-protocole. Pour Dalli 5.0+, l’agent instrumente également `Dalli::Protocol::Meta#read_multi_req`, qui est invoqué par l’optimisation `get_multi` à serveur unique de Dalli. [PR#3541](https://github.com/newrelic/newrelic-ruby-agent/pull/3541)

* **Fonctionnalité : ajouter une option de configuration active\_record\_use\_table\_name**

  Une nouvelle option de configuration, `active_record_use_table_name`, utilise le nom de table d&apos;un modèle Active Record au lieu de son nom de classe pour nommer les métriques, les spans et les segments de trace de transaction. Cela peut être particulièrement utile pour réduire la cardinalité dans les applications utilisant l’héritage à table unique. L&apos;option est définie par défaut sur `false` pour préserver le comportement existant. [PR#3540](https://github.com/newrelic/newrelic-ruby-agent/pull/3540)

* **Fonctionnalité : masquer partiellement les clés de licence dans les logs de l&apos;agent**

  Auparavant, l&apos;agent masquait complètement les clés de licence New Relic dans les logs de l&apos;agent. Maintenant, les 10 premiers caractères sont visibles tandis que le reste est remplacé par `*`. Cela préserve suffisamment d’informations pour résoudre les problèmes liés à la région sans exposer la partie secrète de la clé. [PR#3547](https://github.com/newrelic/newrelic-ruby-agent/pull/3547)

* **Correctif : résolution de l&apos;incompatibilité de l&apos;instrumentation de l&apos;enregistreur sémantique avec `rails_semantic_logger`**

  Auparavant, un `ArgumentError` était levé lorsqu&apos;une exception atteignait `ActionDispatch::DebugExceptions` lors de l&apos;utilisation de `rails_semantic_logger`. Ce problème est résolu. Merci à [@jdelStrother](https://github.com/jdelStrother) de l’avoir signalé ! [PR#3548](https://github.com/newrelic/newrelic-ruby-agent/pull/3548)
