Lorsque `NEW_RELIC_APM_LAMBDA_MODE` est `true`, la variable d&apos;environnement `NEW_RELIC_APP_NAME` contrôle la façon dont vos fonctions Lambda apparaissent en tant qu&apos;entités APM dans New Relic. Cette page explique les comportements clés à comprendre avant de définir cette variable.

## Comportement de l&apos;entité

Les fonctions partageant la même valeur `NEW_RELIC_APP_NAME` et le même compte New Relic sont regroupées sous une seule entité APM dans l&apos;UI de New Relic.

Si le nom que vous définissez correspond à une entité APM existante — par exemple, une entité déjà créée par un service non-Lambda —, la télémétrie de votre fonction Lambda fusionne automatiquement avec cette entité. Aucune entité distincte n&apos;est créée.

<Callout variant="important">
  La modification de `NEW_RELIC_APP_NAME` crée une nouvelle entité APM. Les données historiques associées au nom précédent ne sont pas migrées vers la nouvelle entité.
</Callout>

## Reporting multi-régions

Plusieurs fonctions Lambda déployées dans différentes régions AWS peuvent partager le même `NEW_RELIC_APP_NAME`. Toutes ces fonctions remontent à une seule entité APM, quelle que soit la région dans laquelle elles sont déployées.

Pour distinguer les fonctions individuelles ou les déploiements spécifiques à une région au sein de l’entité groupée, filtrez par l’attribut `tags.aws.arn` dans NRQL ou utilisez la barre de filtrage par tag dans la vue de l’entité.

## Reporting inter-runtime

Les fonctions Lambda utilisant différents environnements d&apos;exécution (par exemple, une fonction Python et une fonction Node.js) qui partagent le même `NEW_RELIC_APP_NAME` créent des entités APM **distinctes** — une par environnement d&apos;exécution. Le regroupement ne franchit pas les limites d&apos;exécution.

## Même nom d’application pour les services Lambda et non-Lambda

Si une fonction Lambda et un service APM non-Lambda (par exemple, une application web traditionnelle ou un microservices) sont tous deux configurés avec le même `NEW_RELIC_APP_NAME`, ils remontent dans la **même** entité APM.

Cela peut être intentionnel : si vous souhaitez une vue unifiée d&apos;un service qui s&apos;exécute à la fois sur Lambda et sur une infrastructure basée sur des serveurs, le partage du nom permet à toute la télémétrie — transactions, erreurs, débit — d&apos;apparaître à un seul endroit.

Cependant, si cela est fait involontairement, cela peut prêter à confusion. Les métriques, les taux d’erreur et le débit de la fonction Lambda et du service non-Lambda sont combinés, ce qui peut rendre plus difficile le diagnostic des problèmes ou la compréhension des performances de manière indépendante.
