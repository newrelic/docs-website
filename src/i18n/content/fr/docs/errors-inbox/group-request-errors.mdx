Les erreurs de demande réseau qui se produisent en raison de problèmes de serveur ou de pannes de réseau peuvent ralentir votre application mobile et avoir un impact négatif sur l&apos;expérience utilisateur. Utilisez la page des erreurs de demande pour comprendre la cause des erreurs HTTP et partagez des données exploitables avec votre équipe pour résoudre les problèmes sous-jacents.

## Afficher la page des erreurs de demande [#request-errors-page]

Pour afficher la page des erreurs de demande, accédez à **<DNT>[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &gt; Mobile &gt; (select an app) &gt; Request errors</DNT>**.

## workflowrecommandé [#suggested-workflow]

Nous recommandons l&apos;approche suivante pour enquêter sur les erreurs de requête HTTP et les échecs de réseau, vous permettant d&apos;acquérir une compréhension complète de l&apos;erreur, d&apos;identifier les facteurs contributifs et de mettre en œuvre des stratégies de débogage efficaces :

1. **Accédez à une seule erreur**: pour afficher les détails d’une erreur de demande ou d’une défaillance du réseau, cliquez sur la ligne du tableau du groupe d’erreurs. Ici, vous pouvez afficher les informations de la demande, l&apos;attribut de la demande et le corps de la réponse, ainsi qu&apos;obtenir plus d&apos;informations sur cette erreur.

2. **Interroger et partager les données d&apos;erreur**: Pour explorer les données derrière l&apos;un des graphiques ou des listes sur la page d&apos;erreurs HTTP :

   * Sur n’importe quel graphique, cliquez sur le menu **…**, puis sur **<DNT>View query</DNT>**.
   * Depuis le générateur de requêtes, vous pouvez ajouter les données d&apos;erreur à un dashboard et les partager via un permalien.

   Pour approfondir les données d&apos;erreur, interrogez vos données pour l&apos;événement et l&apos;attribut suivants :

   * `MobileRequestError` événement et attribut
   * `MobileRequest` événement et attribut

3. **Modifier la manière dont la page regroupe et trie les erreurs et les pannes de réseau**: effectuez des sélections à l&apos;aide des barres d&apos;attributs et de filtres en haut de la page. Par défaut, les erreurs sont regroupées par domaine de requête et chemin de requête.

4. **Filtrer pour des erreurs spécifiques et des échecs réseau**: sélectionnez une erreur ou un échec à l&apos;aide de plusieurs filtres dans la barre de filtres.

5. **Voir les filtres que vous avez appliqués ou supprimer des filtres**: les filtres que vous sélectionnez s&apos;affichent dans la barre de filtres. Pour effacer les filtres, sélectionnez le **X** à côté du filtre que vous souhaitez effacer.

6. **Changer la fenêtre horaire**: Sélectionnez une nouvelle période horaire dans la liste déroulante du sélecteur d&apos;intervalle de temps.

7. **Afficher les informations pour une version d&apos;application spécifique**: sélectionnez la version pour laquelle vous souhaitez voir les graphiques et les listes dans la liste déroulante des versions. Par défaut, toutes les versions disponibles sont affichées.

Les sections ci-dessous décrivent ce que vous pouvez faire sur la page des erreurs de demande.

## Onglet Triage [#triage-errors]

L&apos;onglet triage affiche un aperçu des erreurs non résolues et de leur corrélation avec le taux d&apos;erreur. Le tableau des groupes d&apos;erreurs répertorie les groupes d&apos;erreurs par occurrence et vous permet de les attribuer à un utilisateur qui peut les étudier et les corriger.

## Onglet Erreurs de demande de groupe [#group-errors]

Dans l&apos;onglet Erreurs de groupe, vous pouvez :

* **Évaluez les tendances globales des erreurs de requête HTTP et réseau**: consultez un aperçu des erreurs de requête et du taux d&apos;erreur sur plusieurs domaines de requête et types et versions d&apos;applications mobiles. Utilisez ces données pour identifier et corriger rapidement vos requests API.

* **Filtre pour une analyse plus approfondie**: utilisez des groupes et des filtres pour vous concentrer sur un attribut de demande spécifique, tel que le type de demande, le chemin de la demande, le type d&apos;erreur ou tout attribut personnalisé que vous avez défini.

* **Identifier les modèles**: examinez la table des erreurs de demande pour connaître les tendances liées au domaine de demande, aux chemins de demande et à la fréquence d&apos;occurrence.

* **Enquêter sur les erreurs de demande individuelles**: sélectionnez un rapport d&apos;erreur de demande pour afficher ses tracedistribuées, le suivi des événements, le corps de la réponse, l&apos;attribut et d&apos;autres détails pertinents nécessaires à un débogage efficace.

* **Mettre à jour le statut d&apos;erreur**: marquez l&apos;erreur de demande comme **résolue** ou attribuez-la à un membre de l&apos;équipe spécifique qui peut posséder et examiner le problème, puis déployer un correctif.

## Demander les détails de l&apos;erreur [#error-details]

Sur la page des erreurs de demande, cliquez sur une erreur spécifique pour voir :

* **Parcours utilisateur**: Affiche les différents chemins et actions effectués par un utilisateur qui ont conduit à l&apos;erreur.
* **Graphique de toutes les occurrences**: affiche la fréquence d&apos;une erreur de demande sur la période sélectionnée, à partir de sa détection initiale. Vous pouvez afficher soit les données agrégées de toutes les occurrences, soit une répartition par version d&apos;application.
* **Répartition des types d&apos;erreurs**: Affiche la répartition des erreurs de cette requête sur différentes versions du système d&apos;exploitation ou sur les appareils concernés.

Les erreurs résolues incluent une bannière contenant des détails sur l&apos;utilisateur qui a résolu l&apos;erreur de demande et l&apos;horodatage de résolution. Notez que [des politiques de conservation des données s&apos;appliquent àmonitoring des applications mobiles](/docs/data-apis/manage-data/manage-data-retention/), vous permettant de filtrer par erreurs résolues pour une analyse historique si nécessaire.

### Triage [#triage-errors]

Lorsque vous examinez une erreur spécifique, la section de triage associe l&apos;occurrence d&apos;erreur spécifique que vous visualisez à son [groupe d&apos;erreurs créé par le système](/docs/errors-inbox/errors-inbox/#groups). Ces groupes d’erreurs créés par le système sont identifiés par une empreinte digitale unique. C&apos;est cette empreinte digitale unique qui vous permet de trier les groupes d&apos;erreurs par mises à jour de statut ou affectations.

Pour plus d&apos;informations sur la création des groupes d&apos;erreurs, consultez [Fonctionnement des groupes d&apos;erreurs](/docs/errors-inbox/errors-inbox/#how-groups-work) et pour en savoir plus sur le statut et les affectations, consultez [suivi des erreurs](/docs/errors-inbox/errors-inbox/#assign).

### Réponse [#response]

Une réponse typique à une demande comprend un en-tête de réponse et un corps de réponse, qui transmettent ensemble des informations sur le succès ou l’échec de la demande. L&apos;en-tête de réponse contient des métadonnées sur le serveur, tandis que le corps de la réponse contient des informations sur la sortie, notamment un code de réussite ou d&apos;échec et le message correspondant.

Nous capturons le corps de la réponse aux requests chaque fois qu&apos;il est disponible, en l&apos;affichant sur la page de détails des erreurs de demande pour accélérer le débogage.

### suivi des événements [#event-trail]

Lorsque vous examinez une erreur spécifique, le suivi des événements fournit un log chronologique de tous les événements ayant conduit à l&apos;erreur demandée, ce qui aide à l&apos;analyse des causes profondes. Il peut s&apos;agir d&apos;un événement monitoré par New Relic par défaut ou d&apos;un événement personnalisé. Le suivi des événements est trié par ordre chronologique, en commençant par l&apos;événement le plus ancien, qui est généralement le lancement de l&apos;application, mais vous pouvez modifier les éléments suivants :

* Trier : basculer entre l&apos;ordre croissant et décroissant
* Filtrage d&apos;événements : filtrez par type d&apos;événement, comme `app launch`, `request`, `request error` ou `user actions`.
* Détails de l&apos;événement : développez un événement individuel pour inspecter son attribut, comme `errorType`, `responseTime` ou `requestUrl` pour un événement de demande.

Après avoir trié et filtré votre événement, vous pouvez approfondir un peu l&apos;événement qui a conduit à l&apos;erreur en examinant :

* **Fil d&apos;Ariane personnalisé**: utilisez le [SDK Record fil d&apos;Ariane](/docs/mobile-monitoring/new-relic-mobile/mobile-sdk/record-breadcrumb/) pour créer un événement `MobileBreadcrumb` personnalisé. Cela vous permet d&apos; log une interaction d&apos;application spécifique qui peut être pertinente pour enquêter sur votre erreur.
* **Exceptions gérées**: utilisez les méthodes [SDK Record Handled Exception](/docs/mobile-monitoring/new-relic-mobile/mobile-sdk/record-handled-exceptions/) pour iOS et Android pour annoter l’endroit où les exceptions sont gérées dans votre application. Ces annotations rempliront automatiquement le suivi des événements de crash.

Pour obtenir des conseils sur l&apos;amélioration du suivi des événements en cas d&apos;accident avec des données personnalisées, voir [Record fil d&apos;Ariane](/docs/mobile-monitoring/new-relic-mobile/mobile-sdk/record-breadcrumb/).

Pour tirer pleinement parti de nos outils d&apos;analyse des accidents, assurez-vous de :

1. Utilisez le SDK mobile pour créer un événement `MobileBreadcrumb` ou `MobileHandledException` personnalisé.
2. Activez `MobileRequest` événement pour capturer les données de requête réseau.

## Dépannage [#troubleshooting]

* Gardez à l’esprit que les profils sont désactivés lorsqu’aucun profil n’est disponible correspondant aux filtres appliqués.
* tracedistribuées pour requests ne contiendront probablement qu&apos;une seule occurrence car il s&apos;agit d&apos;un seul événement HTTP enregistré dans notre système.
* Les pannes de réseau n’ont pas de corps de réponse.
* Les agents mobiles conservent une liste de types d’exceptions. Dans certains cas, les exceptions personnalisées levées par les applications ne font pas partie de cette liste. Lorsque cela se produit, `Unknown` peut apparaître dans la page de la boîte de réception des erreurs mobiles. Si vous trouvez `Unknown` dans votre liste d&apos;erreurs et avez besoin d&apos;aide pour rechercher les types d&apos;exceptions manqués, obtenez de l&apos;aide sur [support.newrelic.com](https://support.newrelic.com).
