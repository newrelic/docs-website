Avec l&apos;API de dashboard de New Relic vous pouvez utiliser [NerdGraph](https://api.newrelic.com/graphiql) pour créer votre [dashboard .](/docs/query-your-data/explore-query-data/dashboards/introduction-dashboards) Ce document explique les différents types de widgets que vous pouvez ajouter à votre dashboard, et comment les créer et les obtenir à l&amp;apos;aide de l&amp;apos;API.

## schéma et types de widgets [#widget-schema]

Le schéma GraphQL du widget pour les sorties de requête et de mutation ressemble à ceci :

```graphql
type Widget {
  id 
  visualization
  layout
  title
  linkedEntities
  configuration
  rawConfiguration
}
```

Où:

* `id`: L&amp;apos;identifiant du widget.
* `visualization`: Le type de visualisation du widget, sous forme de chaîne. Par exemple : `viz.line`, `viz.area`. Voir les [exemples ci-dessous](#typed-widget-data).
* `layout`: La position et la taille du widget dans le dashboard. Le nombre maximum de colonnes est de 12.
* `title`: Le titre du widget.
* `linkedEntities`: entité du dashboard liée au widget. Actuellement, il ne prend en charge que l&amp;apos;entité dashboard . Il est utilisé par la [fonctionnalité de liaison de facettes](/docs/query-your-data/explore-query-data/dashboards/filter-new-relic-one-dashboards-facets).
* `configuration`: La [configuration typée](#widget-typed) du widget. Il s&amp;apos;agit d&amp;apos;une représentation GraphQL commodément typée des widgets les plus courants : [area](#area), [bar](#bar), [billboard](#billboard), [line](#line), [markdown](#markdown), [pie](#pie) et [table](#table).
* `rawConfiguration`: La [configuration non typée](#widget-untyped) du widget. Il s&amp;apos;agit d&amp;apos;un moyen libre (scalaire JSON) de stocker les propriétés widget non couvertes par la configuration typée telles que [carte thermique](#heatmap), [histogramme](#histogram), [bullet](#bullet), [entonnoirs](#funnel), etc. Dans ce cas, `configuration` est nul et `rawConfiguration` contient toutes les données requises par le widget.

### Types de widgets [#types-widget]

L&apos;API du dashboard prend en charge les widgets typés et non typés :

<CollapserGroup>
  <Collapser id="widget-typed" title="Widget typé">
    Les widgets typés sont [area](#area), [bar](#bar), [billboard](#billboard), [line](#line), [markdown](#markdown), [pie](#pie) et [table](#table). Ils peuvent être construits directement à l&amp;apos;aide de NerdGraph, qui fournit une assistance avec les champs obligatoires.

    En entrée :

    * Il vous suffit de fournir le `configuration`.

    * `visualization` peut être nul tant qu&amp;apos;il peut être déduit de la configuration fournie.

      En sortie, l&apos;API fournira la configuration que vous avez fournie ainsi que la représentation brute au format JSON à `rawConfiguration`. Par conséquent, `rawConfiguration` est toujours un sur-ensemble de toutes les configurations widget .
  </Collapser>

  <Collapser id="widget-untyped" title="Widget non typé">
    Les widgets non typés sont tous les autres widgets, tels que [puce](#bullet), [histogramme](#histogram), [inventaire](#inventory), etc.

    Pour les widgets non typés, nous connaissons et validons la forme du JSON fourni. Par exemple, une [carte thermique](#heatmap) a une configuration spécifique que nous connaissons à l’avance, nous pouvons donc la valider.

    En entrée :

    * `configuration` n&amp;apos;est pas pris en compte dans ce cas.

    * `rawConfiguration` doit être fourni. Voir les [exemples ci-dessous](#untyped-widget-data).

    * `visualization` ne peut pas être nul et doit être une chaîne fixe comme `viz.heatmap` ou `viz.bullet`.

      En sortie, l&apos;API répond exactement avec ce qui a été fourni en entrée.
  </Collapser>
</CollapserGroup>

## Définitions widget non typées [#untyped-widget-data]

### Surface (métriques prédéfinies) [#areap]

<table>
  <thead>
    <tr>
      <th width="25%">
        Élément de données du widget
      </th>

      <th>
        Description
      </th>
    </tr>
  </thead>

  <tbody>
    <tr>
      <td>
        `duration`

        *Long*
      </td>

      <td>
        Durée de la fenêtre temporelle demandée, en millisecondes.

        * Lorsque `endTime` est fourni, la fenêtre temporelle est définie sur les `x` dernières millisecondes se terminant à l&amp;apos;heure spécifiée.
        * Si `endTime` est nul, la fenêtre temporelle est définie sur les `x` dernières millisecondes se terminant maintenant.
      </td>
    </tr>

    <tr>
      <td>
        `endTime`

        *Long*
      </td>

      <td>
        Facultatif. Fin de la fenêtre temporelle, en millisecondes.
      </td>
    </tr>

    <tr>
      <td>
        `entityIds`

        *\[Integer]*
      </td>

      <td>
        éventail d&apos;identifiants agent sources à partir desquels récupérer des données.
      </td>
    </tr>

    <tr>
      <td>
        `metrics`

        *\[Object]*
      </td>

      <td>
        Pour `type` `SCOPE_BREAKDOWN` seulement. Liste des métriques à récupérer.
      </td>
    </tr>

    <tr id="linep-presentation">
      <td>
        `type`

        *Chaîne*
      </td>

      <td>
        Type du graphique prédéfini. Cela peut être : `APPLICATION_BREAKDOWN`, `BACKGROUND_BREAKDOWN`, `BROWSER_BREAKDOWN`, `GC_RUNS_BREAKDOWN`, `SCOPE_BREAKDOWN`, `SOLR_BREAKDOWN.`
      </td>
    </tr>
  </tbody>
</table>
