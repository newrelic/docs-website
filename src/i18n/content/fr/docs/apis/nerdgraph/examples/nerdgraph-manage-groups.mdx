Vous pouvez utiliser notre [APINerdGraph](/docs/apis/nerdgraph/get-started/introduction-new-relic-nerdgraph) pour afficher et gérer les groupes d&amp;apos;utilisateurs et ce à quoi ces groupes peuvent accéder. Pour savoir comment procéder dans l&amp;apos;UI, consultez la [documentation UI de gestion des utilisateurs](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-ui-and-tasks).

Pour utiliser NerdGraph pour créer un utilisateur et afficher ses informations, voir [Gérer l&apos;utilisateur avec NerdGraph](/docs/apis/nerdgraph/examples/nerdgraph-manage-users).

## Exigences [#requirements]

Quelques exigences pour la gestion des utilisateurs et des groupes via NerdGraph :

* [L&apos;édition Pro ou Enterprise](/docs/accounts/accounts-billing/new-relic-one-pricing-billing/new-relic-one-pricing-billing/#editions) est requise pour personnaliser les groupes d&amp;apos;utilisateurs et les rôles

* Si vous utilisez [le provisionnement SCIM](/docs/accounts/accounts/automated-user-management/automated-user-provisioning-single-sign): pour ce domaine d&amp;apos;authentification, vous ne pouvez pas créer de groupes ni ajouter d&amp;apos;utilisateurs à des groupes, car vos groupes et utilisateurs sont gérés via SCIM.

* Vous devez être un utilisateur de [notre nouveau modèle d&apos;utilisateur](/docs/accounts/original-accounts-billing/original-users-roles/overview-user-models#determine-user-model). Autres exigences liées aux autorisations :

  * [Type d&apos;utilisateur](/docs/accounts/accounts-billing/new-relic-one-user-management/user-type) requis : utilisateur principal ou utilisateur de la plateforme complète
  * [Paramètres d&apos;administration](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts#admin-settings) requis : <DNT>**Organization settings**</DNT> ou <DNT>**Authentication domain settings**</DNT>

## Avant de commencer [#concepts]

Avant d&apos;utiliser NerdGraph pour gérer l&apos;utilisateur :

* Assurez-vous d&apos;avoir une bonne compréhension de nos [concepts de gestion des utilisateurs](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts#understand-concepts)
* Si vous ne l&apos;avez pas déjà fait, nous vous suggérons de consulter l&apos;UI <DNT>**Access management**</DNT> pour améliorer votre compréhension du fonctionnement des groupes et de l&amp;apos;accès des utilisateurs, et pour comprendre les groupes qui existent déjà. Avant de faire cela, nous vous recommandons de créer un plan pour l&amp;apos;accès au groupe que vous devez créer : voici [un exemple de feuille de calcul de planification](https://docs.google.com/spreadsheets/d/1FnguDXRUX9FGY14oV4Gx6O08v4vNC2Pv0GGCsU7Pxuw/edit?usp=sharing).
* Notez que l&apos;[explorateur NerdGraph](/docs/apis/nerdgraph/get-started/introduction-new-relic-nerdgraph/#explorer) dispose de documents intégrés qui définissent les champs utilisés dans ces requests.
* Notez que vous pouvez [suivre les modifications apportées à votre compte New Relic](/docs/data-apis/understand-data/event-data/query-account-audit-logs-nrauditevent).

## workflow suggéré pour la création de groupes [#flow]

Vous pouvez utiliser ces requêtes et mutations de différentes manières et dans différents ordres, mais voici un workflow courant pour la configuration de groupes :

1. [interrogez les informations de votre utilisateur](#query-users) et [les rôles disponibles](#query-roles): cela peut être un premier point de départ utile pour vous assurer de comprendre quel utilisateur vous avez dans New Relic et les rôles disponibles. Si vous débutez, vous n&amp;apos;avez peut-être pas encore ajouté d&amp;apos;utilisateur et vous n&amp;apos;avez peut-être que nos rôles standard.
2. [Facultatif : Créer un nouveau groupe](#create-group): <DNT>**Not available if using SCIM provisioning.**</DNT> Vous pouvez soit utiliser des groupes existants, soit créer un nouveau groupe. Après avoir créé un groupe, vous devez lui accorder l&amp;apos;accès aux rôles et aux comptes. Notez qu&amp;apos;un groupe, à lui seul, n&amp;apos;accorde aucun accès à l&amp;apos;utilisateur de ce groupe : c&amp;apos;est seulement lorsqu&amp;apos;un rôle et un compte lui sont attribués que l&amp;apos;utilisateur peut réellement accéder à New Relic.
3. [Accorder l&apos;accès à un groupe](#grant-access): c&amp;apos;est ce qui attribue aux groupes l&amp;apos;accès aux rôles et aux comptes.

Une fois que vous avez terminé, s&apos;il y a déjà des utilisateurs dans le groupe que vous avez créé et que ce groupe a accès à au moins un rôle et un compte, ils devraient avoir accès dans quelques minutes (bien que pour [les comptes New Relic de la région UE](/docs/using-new-relic/welcome-new-relic/get-started/our-eu-us-region-data-centers), cela puisse prendre jusqu&amp;apos;à 20 minutes environ). Si votre utilisateur n&amp;apos;est pas encore dans ce groupe (ce qui serait vrai si vous venez de créer un nouveau groupe), vous pouvez [ajouter un utilisateur à ce groupe](/docs/accounts/accounts-billing/new-relic-one-user-management/account-user-mgmt-tutorial#add-users).

## Create a role [#create-role]

Here&apos;s an example of creating a [role](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts/#roles):

```graphql
mutation {
  customRoleCreate(
    container: {
      id: "$YOUR_ORGANIZATION_ID"
      type: "ORGANIZATION"
    }
    name: "MY CUSTOM ROLE"
    permissionIds: [1,2,3] 
    scope: "ACCOUNT" 
  ) {
    id
  }
}
```

### Parameters

* `container`:

  * `id`: (String) The unique identifier of your organization. Replace $YOUR\_ORGANIZATION\_ID with your actual organization ID.
  * `type`: (String) The type of container. Currently, the only supported type is &amp;quot;ORGANIZATION&amp;quot;.
  * `name`: (String) The name assigned to the custom role. Example: &amp;quot;MY CUSTOM ROLE&amp;quot;.

* `permissionIds`: (Array) A list of permission IDs representing the capabilities assigned to the custom role. Ensure these IDs correspond to valid permissions in your system.

* `scope`: (String) The level at which the role&amp;apos;s permissions apply. In this instance, the scope is &amp;quot;ACCOUNT&amp;quot;.

### Response

* `id`: Returns the unique ID of the newly created custom role.

  <Callout variant="important">
    * Replace `$YOUR_ORGANIZATION_ID` with your specific organization ID before executing the mutation.
    * Ensure that the `permissionIds` provided are valid and align with the permissions you want to grant.
  </Callout>

Before you create a custom role, you have to identify the permissions you want to assign to it.

Use the following query to retrieve the list of permissions:

```graphql
mutation {
  customerAdministration {
    permissions {
      items {
        category
        feature
        id
        product
        subsetIds
      }
      nextCursor
    }
  }
}
```

This returns account-scoped permissions. For permissions scoped to an org, run the following query instead:

```graphql
{
  customerAdministration {
    permissions(filter: {scope: {eq: "organization"}}) {
      items {
        category
        feature
        id
        product
        subsetIds
      }
      nextCursor
    }
  }
}
```

Note the following fields:

* `items`: An array of permission objects, each containing the following attributes:

  * `category`: (String) The category or grouping the permission belongs to.
  * `feature`: (String) The specific feature the permission is associated with.
  * `id`: (String) A unique identifier for each permission.
  * `product`: (String) The product that the permission applies to.
  * `subsetIds`: (Array) A list of IDs representing subsets or related permissions.

After you have the unique identifier for each permission you want to assign to the new role, create that role.

## Update role [#update-role]

Here&apos;s an example of updating a [role](/docs/accounts/accounts-billing/new-relic-one-user-management/user-management-concepts/#roles).

```graphql
mutation {
  customRoleUpdate(
    id: $ROLE_ID
    name: "MY NEW CUSTOM ROLE NAME"
    permissionIds: [4,5,6] 
  ) {
    id
  }
}
```

### Parameters

* `id`: The unique identifier of the custom role you wish to modify. Replace `$ROLE_ID` with the actual ID of the role.
* `name`: The new name you want to assign to the custom role. In this example, it&amp;apos;s `MY NEW CUSTOM ROLE NAME`.
* `permissionIds`: An array of permission IDs you want to assign to this role. Ensure these IDs are valid and correspond to the permissions you intend to implement.

```graphql
mutation {
  customRoleDelete(

  ) {
    id
  }
}
```

### Parameters

* `id`: The unique identifier of the role you wish to delete. Replace `$ROLE_ID` with the actual ID of the role you want to remove.

Pour certains cas d&apos;utilisation, comme l&apos;octroi de l&apos;accès à un groupe, vous pouvez avoir besoin de l&apos;ID d&apos;un rôle : l&apos;ID numérique représentant ce rôle dans New Relic.

* <DNT>**All product admin**</DNT>: `1254`.

* <DNT>**Standard user**</DNT>: `1253`.

* <DNT>**Read only**</DNT>: `1252`.

* <DNT>**Organization manager setting**</DNT> `1994`
  * <DNT>**Read only**</DNT>: `1995`

* <DNT>**Authentication domain setting**</DNT>:
