Cette page est un endroit où vous et votre équipe pouvez apprendre à interroger vos données sur les problèmes et les incidents, puis à configurer les fonctionnalités à l&apos;aide de notre API NerdGraph.

Nos deux principaux types de données sont `Issue Type` et `Incident Type`.

## Cycle de vie et état du problème [#issue-lifecycle]

Problèmes de ralenti :

* Un problème peut devenir inactif s&apos;il est trop important, c&apos;est-à-dire s&apos;il contient 5 000 incidents ou plus.
* Une question inactive n&apos;accepte plus aucun incident.
* Un problème d’inactivité ne sera pas lié à d’autres problèmes.

  <Collapser id="filter" title="Problèmes de filtre">
    Filtrez les problèmes par GUID d&apos;entité, types d&apos;entité, priorité des problèmes ou états des problèmes ou toute combinaison de ces attributs.

    ```graphql
    "Filter issues"
    input FilterIssues {
      "entityGuids"
      entityGuids: [EntityGuid!]
      "entityTypes"
      entityTypes: [String!]
      "ids"
      ids: [ID!]
      "priority"
      priority: [String!]
      "states"
      states: [IssueState!]
    }
    ```

    Ici, `entityGuides` et `entityTypes` sont les attributs d&apos;entité définis dans la page [En savoir plus sur l&apos;entité New Relic ](/docs/new-relic-solutions/new-relic-one/core-concepts/what-entity-new-relic/).

    * La priorité du problème peut être l&apos;une des suivantes : `LOW`, `MEDIUM`, `HIGH`, `CRITICAL`
    * Les états des problèmes peuvent être l&apos;un des suivants : `CREATED`, `ACTIVATED`, `DEACTIVATED`, `CLOSED`
    * `timeWindow` avec les heures de début et de fin au format EpochMilliseconds. Si elle n&apos;est pas fournie, la valeur par défaut est les dernières 24 heures
    * Et `cursor` comme nous l&apos;avons déjà vu

    ```graphql
              issues {
                issueId

                state
                title
              }
              nextCursor
            }
          }
        }
      }
    }
    ```
