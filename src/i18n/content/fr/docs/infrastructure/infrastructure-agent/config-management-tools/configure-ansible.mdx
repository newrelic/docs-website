Vous pouvez utiliser le rôle Ansible de New Relic pour vous aider à faire évoluer vos efforts d'observabilité New Relic à l'aide de la [CLI New Relic](https://github.com/newrelic/newrelic-cli) et [du référentiel d'installation ouverte New Relic](https://github.com/newrelic/open-install-library). Pour obtenir des instructions sur l&amp;apos;utilisation d&amp;apos;Ansible, consultez la [documentation d'Ansible](https://docs.ansible.com/). Il s’agit d’un effort [soutenu par la communauté](https://github.com/newrelic/ansible-install) .

Vous souhaitez configurer l’agent d’infrastructure à l’aide d’Ansible ? [Créez un compte New Relic](https://newrelic.com/signup) gratuitement ! Aucune carte de crédit requise.

<Callout variant="important">
  Ce rôle installera toujours la dernière version de sortie de l'agent New Relic et ne prend pas en charge l'installation de versions plus anciennes.
</Callout>

## Exemple de code [#role-description]

New Relic fournit un exemple de code de gestion configuration pour vous aider à installer notre agent infrastructure à l'aide d'un workflow et d'outils communs à de nombreuses équipes d'opérations. Il s’agit d’un rôle Ansible de base et est destiné à être utilisé comme point de départ pour créer votre propre workflow personnalisé.

La configuration dépend de votre configuration et de vos normes spécifiques. Pour afficher un exemple de rôle Ansible et plus d'informations sur l'intégration, consultez la [documentation Ansible Galaxy](https://galaxy.ansible.com/newrelic/newrelic_install).

## Compatibilité et exigences [#role-compatibility]

Le rôle Ansible avec l'agent infrastructure de New Relic nécessite un [système d'exploitation pris en charge](/docs/infrastructure/new-relic-infrastructure/getting-started/compatibility-requirements-new-relic-infrastructure#operating-systems).

Si vous avez besoin d’aide avec Ansible pour le système d’exploitation Windows , consultez [Configuration d’un hôte Windows ](https://docs.ansible.com/ansible/latest/os_guide/windows_setup.html), sur le site [de documentationAnsible ](https://docs.ansible.com/ansible/latest/).

Ce rôle est testé avec Ansible Core 2.13 et 2.14 et Python 3.10. Les versions d'Ansible Core antérieures à 2.10 ne sont pas prises en charge.

## Configurer New Relic avec Ansible [#role-setup]

### installation

#### Ansible Galaxy

[Lien vers Galaxy](https://galaxy.ansible.com/newrelic/newrelic_install)

```shell
ansible-galaxy install newrelic.newrelic_install
```

Assurez-vous d'avoir `ansible.windows` et `ansible.utils` s&amp;apos;ils ne sont pas déjà installés :

```shell
ansible-galaxy collection install ansible.windows ansible.utils
```

#### Manuel

Si vous souhaitez utiliser une copie locale du rôle, clonez le référentiel et exécutez `make` dans la racine du projet pour copier ce référentiel vers `~/.ansible/roles/newrelic.newrelic_install`, permettant au rôle de se comporter comme s&amp;apos;il était installé depuis Galaxy.

### Bien démarrer

Pour commencer à utiliser le rôle `newrelic.newrelic_install`:

1. Incluez le rôle dans votre manuel de jeu.
2. Personnalisez les variables requises.

Après l’installation, incluez le rôle `newrelic.newrelic_install` dans un playbook nouveau ou existant. Voici un exemple de configuration de votre rôle pour installer l&amp;apos;agent d&amp;apos;infrastructure :

```yml
---
- name: Install New Relic
  hosts: all
  roles:
    - role: newrelic.newrelic_install
      vars:
        targets:
          - infrastructure
          - logs
        tags:
          foo: bar
  environment:
    NEW_RELIC_API_KEY: <API key>
    NEW_RELIC_ACCOUNT_ID: <Account ID>
    NEW_RELIC_REGION: <Region>
```

### Valeurs par défaut

Définir dans [defaults/main.yml](https://github.com/newrelic/ansible-install/blob/main/defaults/main.yml):

* `cli_install_url`
* `cli_install_download_location`
* `target_name_map`
* `verbosity_on_log_file_path_linux`
* `verbosity_on_log_file_path_windows`
* `default_install_timeout_seconds`

### Variables d'environnement

Les valeurs sont définies sous le mot-clé [`environment`](https://docs.ansible.com/ansible/latest/playbook_guide/playbooks_environment.html) dans votre playbook :

* `NEW_RELIC_API_KEY` (requis)
* `NEW_RELIC_ACCOUNT_ID` (requis)
* `NEW_RELIC_REGION` (facultatif : « US » ou « EU », par défaut « US »)

De plus, vous pouvez définir une variable `HTTPS_PROXY` facultative pour activer un proxy pour votre installation.

<CollapserGroup>
  <Collapser id="environment-vars" title="apm-php">
    * `NEW_RELIC_APPLICATION_NAME` (facultatif) Le nom de l&amp;apos;application PHP à instrumenter. Ce nom sera répertorié sous `APM & Services` de New Relic. Si elle est omise, la valeur par défaut est `PHP Application`.
  </Collapser>

  <Collapser id="environment-vars" title="Apache">
    * `NEW_RELIC_APACHE_STATUS_URL` (facultatif) L&amp;apos;URL pour vérifier l&amp;apos;état du serveur Web Apache. Ceci est utilisé pour s&amp;apos;assurer qu&amp;apos;un serveur Web Apache est en cours d&amp;apos;exécution sur l&amp;apos;hôte et qu&amp;apos;il est dans un état sain avant de tenter l&amp;apos;installation de l&amp;apos;intégration Apache sur l&amp;apos;hôte. Par défaut : `http://127.0.0.1/server-status?auto`
  </Collapser>

  <Collapser id="environment-vars" title="mssql">
    * `NEW_RELIC_MSSQL_DB_HOSTNAME` (facultatif) nom d&amp;apos;hôte ou IP où le serveur MS SQL est exécuté. La valeur par défaut est le nom d&amp;apos;hôte découvert s&amp;apos;il n&amp;apos;est pas spécifié.
    * `NEW_RELIC_MSSQL_DB_PORT` (facultatif) Port sur lequel le serveur MS SQL écoute. Par défaut `1433`
    * `NEW_RELIC_MSSQL_DB_USERNAME` (facultatif) Nom d&amp;apos;utilisateur pour accéder au serveur MS SQL. La valeur par défaut est `newrelic`. Si vous utilisez un utilisateur de domaine, utilisez la syntaxe `domain\user`
    * `NEW_RELIC_MSSQL_DB_PASSWORD` (facultatif) Mot de passe pour l&amp;apos;utilisateur SQL ou de domaine donné. Si aucun mot de passe n&amp;apos;est fourni, un mot de passe aléatoire sera généré.
    * `NEW_RELIC_MSSQL_SQL_USERNAME` (facultatif) Remplacement facultatif des informations d&amp;apos;identification transmis à `sqlcmd` lors de la création de l&amp;apos;utilisateur SQL spécifié par `NEW_RELIC_MSSQL_DB_USERNAME`. Si omis, le nom d&amp;apos;utilisateur de connexion par défaut sera utilisé.
    * `NEW_RELIC_MSSQL_SQL_PASSWORD` (facultatif) Remplacement facultatif des informations d&amp;apos;identification transmis à `sqlcmd` lors de la création de l&amp;apos;utilisateur SQL spécifié par `NEW_RELIC_MSSQL_DB_USERNAME`. Si omis, le mot de passe de connexion par défaut sera utilisé.
    * `NEW_RELIC_MSSQL_ENABLE_BUFFER_METRICS` (facultatif) Activer la collecte des mesures de tampons de piscine. La valeur par défaut est true
    * `NEW_RELIC_MSSQL_ENABLE_RESERVE_METRICS` (facultatif) Activer la collecte de l&amp;apos;espace de réserve de partition de base de données. La valeur par défaut est true
  </Collapser>

  <Collapser id="environment-vars" title="MySQL">
    * `NEW_RELIC_MYSQL_PORT` (facultatif) La valeur par défaut est `3306` si non spécifiée.
    * `NEW_RELIC_MYSQL_USERNAME` (facultatif) La valeur par défaut est `newrelic` si aucune autre valeur n&amp;apos;est spécifiée. Il s&amp;apos;agit du nom d&amp;apos;utilisateur que l&amp;apos;intégration `mysql` configurera et qui sera également défini dans le fichier configuration de l&amp;apos;intégration (par exemple : `mysql-config.yml`) à des fins de reporting des données. En savoir plus sur [l'intégration MySQL](/install/mysql/).
    * `NEW_RELIC_MYSQL_PASSWORD` (facultatif) Le mot de passe de l&amp;apos;utilisateur spécifié dans `NEW_RELIC_MYSQL_USERNAME`. En savoir plus sur [l'intégration MySQL](/install/mysql/).
    * `NEW_RELIC_MYSQL_ROOT_PASSWORD` (obligatoire) L&amp;apos;intégration `mysql` doit se connecter à `mysql` pour créer les informations d&amp;apos;identification appropriées.
  </Collapser>
</CollapserGroup>

Consultez [l'environnement distant d'Ansible](https://docs.ansible.com/ansible/latest/playbook_guide/playbooks_environment.html) pour plus d&amp;apos;informations.

<Callout variant="important" title="Remarque importante">
  * La cible `logs` nécessite `infrastructure` et une erreur sera générée si `logs` est spécifié sans `infrastructure`.

  * L'installation de l'agent `apm-nodejs` est prise en charge uniquement pour les applications gérées par [PM2](https://pm2.keymetrics.io/). Pour installer l&amp;apos;agent à l&amp;apos;aide d&amp;apos;un gestionnaire de paquets tel que `npm` ou `yarn` ou via d&amp;apos;autres chemins d&amp;apos;installation, veuillez vous référer à notre [documentation](/docs/apm/agents/nodejs-agent/installation-configuration/install-nodejs-agent/).

  * L'installation de l'agent `apm-dotnet` pour Windows est prise en charge uniquement pour les applications hébergées par [IIS](https://www.iis.net/). Les installations Linux ne sont prises en charge que pour les applications .NET qui s&amp;apos;exécutent en tant que service `systemd` .

  * L'installation de l'agent `apm-java` prend en charge Java exécuté dans Tomcat, Wildfly/Jboss et Jetty (autonome).

  * L'intégration suivante nécessite l'installation de l'agent infrastructure :

    * apm-java
    * Apache
    * mssql
    * MySQL
    * nginx
</Callout>

## Manuel d'exemples supplémentaires [#examples]

```yml
- name: Install New Relic
  hosts: all
  roles:
    - role: newrelic.newrelic_install
      vars:
        targets:
          - infrastructure
          - logs
          - apm-php
        tags:
          foo: bar
        install_timeout_seconds: 1000
        verbosity: debug
  environment:
    NEW_RELIC_API_KEY: <API key>
    NEW_RELIC_ACCOUNT_ID: <Account ID>
    NEW_RELIC_REGION: <Region>
    NEW_RELIC_APPLICATION_NAME: "My Application"
    HTTPS_PROXY: "http://my.proxy:8888"
```
