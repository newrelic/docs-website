L&apos;agent d&apos;infrastructure est un logiciel léger, conçu pour minimiser son impact sur les performances de vos hôtes. Cependant, la charge exacte varie en fonction de workload de votre hôte, notamment du nombre de processus exécutés sur l&apos;hôte. Cela est dû au fait que l’agent collecte [des données](/docs/data-analysis/metrics/analyze-your-metrics/data-collection-metric-timeslice-event-data#event-data) détaillées sur chaque processus individuel.

À titre indicatif, New Relic a collecté des évaluations pour certains types d&apos;hôtes courants :

<CollapserGroup>
  <Collapser id="single-task" title="Hôte Linux monotâche">
    L&apos;agent présente une surcharge de performances très faible sur un hôte classique à tâche unique. Par exemple, un serveur exécutant Apache, Unicorn ou une seule application Java.

    Notre évaluation pour ce type d&apos;hôte est basée sur un [Amazon EC2 t3.2xlarge](https://aws.amazon.com/ec2/instance-types/):

    * <DNT>**vCPUs**</DNT>: 8 vCPU

    * <DNT>**Memory**</DNT>: 32,0 Go

    * <DNT>**Storage**</DNT>: 160,0 Go

    * <DNT>**Operating system**</DNT>: CentOS 7

      Pour ce type d&apos;hôte classique à tâche unique, l&apos;utilisation typique est :

    * <DNT>**CPU**</DNT>: environ 0,3%

    * <DNT>**Virtual memory**</DNT>: environ 1 Go

    * <DNT>**Resident memory**</DNT>: 25 à 35 Mo

    * <DNT>**Storage on disk**</DNT>: environ 50 Mo
  </Collapser>

  <Collapser id="docker-host" title="Hôte Linux Docker">
    L&apos;agent a une surcharge de performances très faible sur un hôte exécutant Docker, avec une utilisation exacte dépendant du nombre de conteneurs Docker sur votre machine hôtes, et si ces processus sont de longue ou de courte durée.

    Notre évaluation pour ce type d&apos;hôte est basée sur un [Amazon EC2 t3.2xlarge](https://aws.amazon.com/ec2/instance-types/):

    * <DNT>**vCPUs**</DNT>: 8 vCPU

    * <DNT>**Memory**</DNT>: 32,0 Go

    * <DNT>**Storage**</DNT>: 160,0 Go

    * <DNT>**Number of containers**</DNT>:25 conteneurs, environ 100 processus de longue durée exécutés dans un conteneur

    * <DNT>**Operating system**</DNT>: CentOS 7

      Pour ce type d&apos;hôte Docker, l&apos;utilisation typique est :

    * <DNT>**CPU**</DNT>: environ 0,8%

    * <DNT>**Virtual memory**</DNT>: environ 1 Go

    * <DNT>**Resident memory**</DNT>: 25 à 35 Mo

    * <DNT>**Storage on disk**</DNT>: environ 50 Mo
  </Collapser>

  <Collapser id="windows-host" title="Hôte Windows">
    L&apos;agent a une surcharge de performances très faible sur un Windows hôte typique servant des applications Web et exécutant la Windows/IIS.stack

    Notre évaluation pour ce type d&apos;hôte est basée sur un [Amazon EC2 t2.small](https://aws.amazon.com/ec2/instance-types/):

    * <DNT>**vCPUs**</DNT>: 1

    * <DNT>**Memory**</DNT>: 2,0 Go

    * <DNT>**Storage**</DNT>: 30,0 Go

    * <DNT>**Operating system**</DNT>: Windows Server 2012 R2

      Pour ce type d&apos;hôte Windows, l&apos;utilisation typique est :

    * <DNT>**CPU**</DNT>: 2 à 3%

    * <DNT>**Resident Memory**</DNT>: 30 Mo

    * <DNT>**Storage on disk**</DNT>: environ 50 Mo
  </Collapser>

  <Collapser id="arm64" title="Hôte Linux ARM64">
    L&apos;agent a une surcharge de performances similaire sur un hôte ARM64 (Graviton 2) sur EC2 par rapport aux machines AMD64.

    L&apos;évaluation est basée sur l&apos;instance [Amazon EC2 t3.2xlarge vs. t4g.2xlarge](https://aws.amazon.com/ec2/instance-types/) .

    instanceAmazon Linux 2 EC2 avec paramètres par défaut de l&apos;agent infrastructure :

    * <DNT>**CPU**</DNT>: environ 0,1 % sur ARM contre 0,13 % sur AMD
    * <DNT>**Virtual memory**</DNT>: environ 0,75 Go ARM contre 1 Go AMD
    * <DNT>**Resident memory**</DNT>: 20 Mo ARM contre 22 Mo AMD
  </Collapser>
</CollapserGroup>

Nous améliorons constamment les performances de l&apos;agent d&apos;infrastructure. Si vous constatez une surcharge de performances d&apos;agent inhabituellement élevée, obtenez de l&apos;aide sur [support.newrelic.com](https://support.newrelic.com).

## Gérer les données [#manage-data]

Pour savoir comment ajuster la quantité de données que notre monitoring deinfrastructure ingère et rapporte, consultez [Gérer les infrastructure données](/docs/infrastructure/manage-your-data/data-instrumentation/manage-infrastructure-data-reporting).

## Utilisation des ressources [#resource-utilization]

Sur le système Linux, infrastructure est installée avec les paramètres par défaut pour chaque gestionnaire de services pris en charge. Une limite de mémoire de 1 gigaoctet est appliquée. Veuillez envisager de revoir et d&apos;ajuster la configuration par défaut en fonction de vos exigences système.
