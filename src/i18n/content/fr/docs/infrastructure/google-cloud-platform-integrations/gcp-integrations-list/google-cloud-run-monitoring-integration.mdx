[L&apos;intégration de New Relic](/docs/infrastructure/introduction-infra-monitoring) inclut une intégration permettant de signaler vos données GCP Run à nos produits. Nous expliquons ici comment activer l&apos;intégration et quelles données elle collecte.

## Activer l&apos;intégration [#activate]

Pour activer l&apos;intégration, suivez les procédures standard pour [connecter votre service GCP à New Relic](/docs/connect-google-cloud-platform-services-infrastructure).

## configuration et sondage [#polling]

Vous pouvez modifier la fréquence d&apos;interrogation et filtrer les données à l&apos;aide [des options de configuration](/docs/integrations/new-relic-integrations/getting-started/configure-polling-frequency-data-collection-cloud-integrations).

Informations [d&apos;interrogation](/docs/integrations/google-cloud-platform-integrations/getting-started/polling-intervals-gcp-integrations) par défaut pour l&apos;intégration GCP Run :

* Intervalle d&apos;interrogation de New Relic : 5 minutes

## Rechercher et utiliser des données [#find-data]

Pour trouver vos données d’intégration, accédez à <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &gt; Infrastructure &gt; GCP**</DNT> et sélectionnez une intégration.

Les données sont jointes au [type d’événement](/docs/data-apis/understand-data/new-relic-data-types/#event-data) suivant :

## données métriques [#metrics]

Cette intégration collecte les données GCP Run pour la révision.

### Exécuter les données de révision

<table>
  <thead>
    <tr>
      <th style={{ width: "275px" }}>
        métrique
      </th>

      <th style={{ width: "150px" }}>
        Unité
      </th>

      <th>
        Description
      </th>
    </tr>
  </thead>

  <tbody>
    <tr>
      <td>
        `container.BillableInstanceTime`
      </td>

      <td>
        Secondes
      </td>

      <td>
        Temps facturable agrégé à partir de toutes les instances de conteneur de la révision. Pour une instance de conteneur donnée, le temps facturable se produit lorsque l&apos;instance de conteneur démarre ou qu&apos;au moins une demande est en cours de traitement. Le temps facturable est arrondi aux 100 millisecondes les plus proches. Exemples : si une révision avec 2 instances de conteneur a diffusé du trafic en continu au cours de la dernière minute, la valeur est de 2 s/s avec l&apos;aligneur « rate » par défaut. Si une seule requête d&apos;une durée de 30 ms a été reçue par une révision au cours de la dernière minute, elle est arrondie à 100 ms et ramenée en moyenne à 1,7 ms/s sur la minute avec l&apos;aligneur « taux » par défaut.
      </td>
    </tr>

    <tr>
      <td>
        `container.cpu.AllocationTime`
      </td>

      <td>
        Secondes
      </td>

      <td>
        Allocation du processeur du conteneur de la révision en secondes.
      </td>
    </tr>

    <tr>
      <td>
        `container.memory.AllocationTime`
      </td>

      <td>
        Autre
      </td>

      <td>
        Allocation de mémoire du conteneur de la révision en gigaoctets-secondes.
      </td>
    </tr>

    <tr>
      <td>
        `Request`
      </td>

      <td>
        Compter
      </td>

      <td>
        Nombre de requests atteignant la révision. Exclut requests qui n&apos;atteignent pas votre instance de conteneur (par exemple, requests non autorisées ou lorsque le nombre maximal d&apos;instances est atteint).
      </td>
    </tr>

    <tr>
      <td>
        `RequestLatencies`
      </td>

      <td>
        Millisecondes
      </td>

      <td>
        Répartition de la latence des requêtes en millisecondes atteignant la révision.
      </td>
    </tr>
  </tbody>
</table>
