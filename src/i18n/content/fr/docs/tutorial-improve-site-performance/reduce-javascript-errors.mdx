Les erreurs JavaScript affectent la façon dont les clients utilisent votre site et leur perception de votre marque. Vous devez donner la priorité à la correction des erreurs JavaScript dès qu&apos;elles se produisent pour aider à maintenir les performances du site et à satisfaire vos clients.

## Objectifs [#objectives]

Ce didacticiel vous guide dans la recherche et la résolution des erreurs JavaScript. À la fin du didacticiel, vous aurez accompli ces tâches :

* Recherche d&apos;informations sur les erreurs JavaScript dans New Relic
* Appliquer nos recommandations lors de la priorisation des erreurs
* Utiliser le suivi des événements pour résoudre les erreurs JavaScript

## Trouver les erreurs JavaScript [#find-errors]

Depuis la page monitoring des navigateurs <DNT>**Summary**</DNT>, deux graphiques peuvent indiquer que vous avez des erreurs JavaScript : <DNT>**Error Rate**</DNT> et <DNT>**Top 10 error messages**</DNT>. La sélection de l&apos;un ou l&apos;autre graphique vous amène à la page <DNT>**JS errors**</DNT> , qui vous donne une vue plus détaillée de vos erreurs JavaScript.

* Si un client ou un utilisateur interne signale quelque chose qui pourrait être une erreur JavaScript, nous vous recommandons de commencer votre recherche sur les pages <DNT>**Summary**</DNT> ou <DNT>**JS errors**</DNT> .
* Si vous souhaitez monitorer de manière proactive votre site pour détecter les erreurs, la vérification de la page <DNT>**Summary**</DNT> est la plupart du temps suffisante.

## Déterminer quelles erreurs comptent [#determine-errors]

Une fois que vous avez découvert certaines erreurs JavaScript, vous devez hiérarchiser vos correctifs. Vous pouvez hiérarchiser les erreurs en fonction de leur fréquence, de leur gravité, des régions affectées et d’autres facteurs. Prenons un exemple pour vous expliquer comment hiérarchiser les erreurs JavaScript.

<img title="JS error charts" alt="A screenshot of JavaScript error charts in the browser summary page" src="/images/browser_screenshot-crop_error-overview-1.webp" />

En utilisant le graphique <DNT>**Top 10 error messages**</DNT> ci-dessus, vous pouvez voir qu&apos;une erreur comporte 6 460 instances. La fréquence de cette erreur pourrait en faire un candidat de choix à résoudre en premier, mais ce n’est pas toujours le cas. Vous aurez peut-être besoin d’une vue plus large de toutes vos erreurs pour déterminer l’ordre dans lequel vous devez les corriger. À partir de la page <DNT>**JS errors**</DNT> , vous pouvez regrouper les erreurs par `errorMessage`, `pageURL` ou `deviceType`. Vous pouvez vous poser ces questions lors de la planification de votre travail :

* Cela affecte-t-il les parcours utilisateurs critiques pour l’entreprise ? Si une erreur n&apos;a pas d&apos;impact sur la capacité de vos clients à s&apos;inscrire, log connecter ou à effectuer un achat, vous pourrez peut-être déprioriser le problème.
* Cette erreur s&apos;est-elle produite récemment ? Se concentrer sur les erreurs les plus récentes peut éviter qu’elles ne deviennent des problèmes plus importants plus tard.
* Cette erreur entraîne-t-elle une latence élevée ?

## Corrigez les erreurs rapidement [#fix-errors]

<img title="JS error overview" alt="A screenshot of a single error overview page" src="/images/browser_screenshot-crop_error-eventlog-stacktrace-overview-1.webp" />

Une fois que vous avez choisi une erreur, sélectionnez-la sur la page <DNT>**JS errors**</DNT> . À partir de là, vous pouvez ouvrir différentes vues pour enquêter sur une erreur spécifique :

* <DNT>**Error instances**</DNT> vous amène aux pages <DNT>**Event Trail**</DNT> et <DNT>**Stack trace**</DNT> . Vous pouvez trouver plus de détails sur l&apos;erreur spécifique, par exemple si un fichier JavaScript a provoqué l&apos;erreur ou si vous devez corriger des parties du code en ligne.
* Faites défiler la section <DNT>**Event Trail**</DNT> pour vous assurer que vous avez trouvé tout ce qui doit être corrigé. S&apos;il y a davantage de problèmes à résoudre, votre suivi des événements vous le fera savoir.
