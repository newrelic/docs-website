[Les métriques dimensionnelles](/docs/using-new-relic/data/understand-data/new-relic-data-types#dimensional-metrics) sont une norme industrielle pour le stockage et l&amp;apos;interrogation de données métriques. New Relic stocke toutes les métriques infrastructure [sous forme de données d&apos;événement](/docs/data-apis/understand-data/new-relic-data-types/#metric-event-attributes), mais vous pouvez également les requêter sous forme de métriques dimensionnelles.

Chez New Relic, nous rapportons [les métriques](/docs/using-new-relic/data/understand-data/new-relic-data-types#metrics) de plusieurs manières, y compris [les métriques dimensionnelles](/docs/using-new-relic/data/understand-data/new-relic-data-types#dimensional-metrics), que notre API métrique, notre SDK télémétrie, certaines intégrations open source et nos services infrastructure peuvent tous utiliser.

Ce type de métrique vous permet de :

* Bénéficiez d&apos;une expérience de requête améliorée pour les données d&apos;infrastructure.
* Découvrez toutes vos métriques en un seul endroit.
* Utilisez des sources plus métriques, comme [Prometheus](/docs/integrations/prometheus-integrations/get-started/new-relic-prometheus-openmetrics-integration-kubernetes).

Par exemple, vous pouvez utiliser la requête suivante pour obtenir la durée maximale de votre fonction Lambda:

## Démarrer [#compatibility-reqs]

Vous n’avez besoin d’aucune mise à jour d’agent ou d’intégration pour utiliser ces métriques. Nous prenons en charge [les alertes NRQL ](/docs/alerts/new-relic-alerts/defining-conditions/create-alert-conditions-nrql-queries)basées sur des métriques dimensionnelles, à l&amp;apos;exception des données provenant du cloud d&amp;apos;intégration (telles que les métriques d&amp;apos;[intégration de sondage AWS](/docs/infrastructure/amazon-integrations/aws-integrations-list), [GCP](/docs/integrations/google-cloud-platform-integrations) et [Azure](/docs/integrations/microsoft-azure-integrations/azure-integrations-list)). New Relic ingère les métriques AWS CloudWatch Metric Streams en tant que métriques dimensionnelles, et nous vous recommandons d&amp;apos;utiliser NRQL <InlinePopover type="alerts" />dans ces cas.

## Où et comment requêter des métriques dimensionnelles [#where]

New Relic prend en charge toutes [les fonctionnalités de requête NRQL](/docs/query-data/nrql-new-relic-query-language/getting-started/nrql-syntax-clauses-functions) actuelles. Vous pouvez utiliser `WHERE`, `FACET` et les fonctions de sélection de temps telles que `SINCE`, `UNTIL` et `COMPARE WITH` pour l&amp;apos;interrogation.

## Conventions de dénomination pour les métriques et les attributs [#naming-conventions]

Tous les noms et attributs métriques pour les métriques dimensionnelles suivent la même convention de dénomination pour les rendre faciles à trouver et à utiliser. Les noms de métrique et d&apos;attribut sont séparés par des points : par exemple, le préfixe `host.` est utilisé pour les métriques de l&amp;apos;hôte, le préfixe `k8s.` est utilisé pour les métriques Kubernetes et `aws.` est utilisé pour les métriques AWS.

Le graphique ci-dessous montre comment un `ProcessSample` contenant trois métriques (`cpuPercent`, `ioTotalReadBytes` et `ioTotalWriteBytes`) est divisé en trois métriques distinctes. Notez la dénomination mise à jour des métriques et de l&amp;apos;attribut.

## Différences dans l&apos;interrogation des mesures dimensionnelles et des événements [#known-limitations]

Les métriques dimensionnelles sont un type de données fondamentalement différent des données d’événement. Ci-dessous, vous trouverez quelques différences notables lors de l&apos;interrogation de métriques dimensionnelles :

* la requête métrique avec `*` ne renvoie pas [d&apos;échantillons de donnéesinfrastructure ](/docs/infrastructure/manage-your-data/data-instrumentation/default-infrastructure-monitoring-data/#infrastructure-events). Par exemple:

  ```sql
  SELECT * FROM Metric
  ```

* La requête métrique avec `metricName LIKE` ne renvoie pas d&amp;apos;échantillons de données infrastructure . Par exemple:

  ```sql
  SELECT uniques(metricName) FROM Metric where metricName like 'k8%'
  ```

* Afin de sélectionner un attribut commençant par `tags.` , un nom métrique doit être fourni. Par exemple, cela ne fonctionne pas sans la clause `WHERE` :

  ```sql
  SELECT uniques(tags.environment) FROM Metric WHERE metricName='aws.lambda.function.duration'
  ```

* Les résultats peuvent ne pas être complets si les critères de sélection correspondent à trop d’échantillons. Par exemple, cette requête correspond à tous les exemples d’infrastructure et peut renvoyer des résultats incomplets :

  ```sql
  SELECT uniqueCount(entity.guid) FROM Metric
  ```

* Nous ne prenons pas en charge la nouvelle fonctionnalité de métrique générique, par exemple :

  ```sql
  SELECT average(host.swap%Bytes) FROM Metric
  ```

* [Les fonctions](/docs/query-your-data/nrql-new-relic-query-language/get-started/nrql-syntax-clauses-functions/#functions) utilisées sur plusieurs métriques peuvent échouer ou renvoyer des résultats incorrects, par exemple :

  ```sql
  FROM Metric SELECT latest(metricNameA + metricNameB)
  ```

* Lorsque vous incluez `RAW` dans une requête, la demande se transforme en interne et imprime des données d&amp;apos;événement agrégées équivalentes. Il n&amp;apos;imprimera pas de données brutes. Reportez-vous à l&amp;apos;exemple de requête pour voir ce comportement :

  ```sql
  SELECT max(host.cpuPercent) FROM Metric TIMESERIES 1 MINUTE SINCE 60 MINUTES AGO RAW
  ```

* Nous ne prenons pas en charge l&apos;utilisation de `keyset` avec `TIMESERIES`, car cela renverrait une erreur. Voir l&amp;apos;exemple ci-dessous :

  ```sql
  FROM Metric SELECT keyset() WHERE instrumentation.provider = 'infrastructure' TIMESERIES
  ```

Pour un aperçu des différences de types de données, voir [Types de données New Relic](/docs/data-apis/understand-data/new-relic-data-types).
