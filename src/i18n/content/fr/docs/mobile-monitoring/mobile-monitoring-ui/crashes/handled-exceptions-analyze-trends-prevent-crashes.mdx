Vous pouvez également utiliser l&apos; [API des exceptions gérées](#hex-api) pour personnaliser les données que vous envoyez et utiliser NRQL pour interroger et partager les données.

## Caractéristiques

La gestion des exceptions au fur et à mesure qu&apos;elles se produisent peut contribuer à améliorer l&apos;expérience utilisateur de votre application mobile, mais elle ne suffit pas à détecter les exceptions. Il faut aussi savoir comment les prévenir. Notre fonctionnalité Exceptions traitées vous aide à répondre à ces types de questions :

* Combien de types différents d’exceptions traitées se produisent ? Un taux d&apos;occurrence élevé peut indiquer que vous devez apporter des modifications au système backend .
* Pourquoi l&apos;utilisation de l&apos;application par l&apos;utilisateur entraîne-t-elle un try/catch ? Quel est le contexte des exceptions ?
* Quand les réponses d’un environnement de test aux exceptions traitées peuvent-elles indiquer des problèmes supplémentaires plus graves ?
* Qu&apos;est-ce qui aurait pu provoquer un crash si l&apos;exception n&apos;avait pas été détectée en production ? Qu&apos;est-ce qui d&apos;autre (dans le code ou l&apos;API backend ) affecte encore l&apos;expérience de l&apos;utilisateur ?

En utilisant des exceptions gérées, vous pouvez identifier et résoudre ces types de problèmes plus rapidement.

## Exigences

Pour afficher les données d’exception traitées, votre application doit répondre aux exigences suivantes :

* <DNT>**Android:**</DNT> [Agent Android version 5.15.0 ou supérieure](/docs/release-notes/mobile-release-notes/android-release-notes)
* <DNT>**iOS:**</DNT> [Agent iOS version 5.15.0 ou supérieure](/docs/release-notes/mobile-release-notes/ios-release-notes)

## Graphiques de pourcentage d&apos;exceptions [#percentage-charts]

Commencez par les graphiques de pourcentage <DNT>**Users affected**</DNT> et <DNT>**Sessions affected**</DNT> pour voir s’il y a des pics, des creux ou des modèles inattendus avec des exceptions en général. Si le graphique <DNT>**Users affected**</DNT> est vide, cela signifie qu&apos;il n&apos;y a eu aucune session utilisateur pendant la période sélectionnée.

Utilisez ceci pour répondre à des questions telles que :

* Y a-t-il des pics à proximité d&apos;une sortie de version récente ?
* Existe-t-il une période pendant laquelle le pourcentage d&apos;utilisateurs a été affecté de manière significative par l&apos;exception ?
* Y a-t-il des périodes sans incident ?

Pour examiner les données plus en détail : sous n’importe quel graphique, sélectionnez [<DNT>**Expand chart**</DNT>](#top5-image).

## Groupes et filtres [#groups-filters]

Utilisez les groupes et les filtres pour examiner plus en détail les attributs des pannes, des appareils, des emplacements ou d&apos;autres attributs personnalisés. Vous pouvez sélectionner un groupe, puis filtrer sur des données spécifiques. Utilisez ceci pour faire des choses comme :

* Regroupez la liste par emplacement d&apos;exception (par défaut), cause, build ou version d&apos;application, appareils, connexions ou autre attribut personnalisé. Cela vous permet de découvrir des modèles dans vos exceptions pour déterminer la cause première.
* Utilisez le [sélecteur d&apos;intervalle de temps](/docs/query-your-data/explore-query-data/dashboards/manage-your-dashboard/#dash-time-picker) pour régler la période de temps actuellement sélectionnée.
* Filtrez par un <DNT>**Version**</DNT> spécifique ou par un ou plusieurs attributs <DNT>**Filter**</DNT>, tels que `appVersion`, `exceptionLocationMethod`, `lastInteraction`, ou l&apos;un des attributs standard et personnalisés de la [liste plus longue](/docs/insights/insights-data-sources/default-data/mobile-events-attributes) .

Les filtres actuellement sélectionnés apparaissent en haut de la page de l’interface utilisateur. Vous pouvez les fermer, ajouter d’autres filtres ou sélectionner d’autres groupes et filtres.

## Les meilleurs emplacements d&apos;exception [#top5-exceptions]

Utilisez le tableau <DNT>**Top exception locations**</DNT> pour rechercher ou trier des modèles dans le type d&apos;exception que vous avez sélectionné parmi les [groupes et les filtres](#groups-filters). Cela comprend :

* Lieux récurrents dans la trace d&apos;appels
* Version de l&apos;application mobile
* Nombre d&apos;occurrences
* Nombre d&apos;utilisateurs concernés pendant la [période sélectionnée](/docs/query-your-data/explore-query-data/dashboards/manage-your-dashboard/#dash-time-picker)

Par exemple, vous pouvez regrouper par `Exception Message`, filtrer sur `timeout` message, puis sélectionner des emplacements de délai d&apos;expiration individuels dans le tableau pour consulter le fil de trace des appels et les détails de chaque occurrence.

* Pour filtrer ou regrouper par un autre attribut, utilisez la fenêtre de recherche du tableau ou sélectionnez l&apos;un des filtres disponibles. Par exemple, filtrez par type d&apos;événement, appareil, emplacement spécifique ou tout attribut personnalisé.
* Pour rechercher d’autres modèles historiques, modifiez la période sélectionnée.

## Liens vers le générateur de requêtes [#insights]

<DNT>**Handled exceptions**</DNT> les graphiques utilisent l&apos;attribut par défaut pour l&apos;événement mobile (y compris [`MobileHandledException`](/docs/insights/insights-data-sources/default-data/mobile-events-attributes#mobilehandledexception-attributes)), ainsi que tout attribut personnalisé que vous avez ajouté à ce type d&apos;événement. Lorsque vous passez la souris sur les graphiques, [des liens directs](#top5-image) apparaissent en dessous. Ces liens vers le générateur de requêtes vous permettent d&apos;analyser encore plus en profondeur les données de votre application mobile.

* <DNT>**View query**</DNT> lien : Afficher la [requête NRQL](/docs/insights/new-relic-insights/using-new-relic-query-language/using-nrql) utilisée pour calculer les données du graphique.

## Tableau des emplacements d&apos;exception [#locations-table]

Le tableau <DNT>**Exception locations**</DNT> complète les graphiques. Il répertorie où les cinq principales exceptions traitées apparaissent dans leur fil de trace des appels et les relie aux détails pertinents. Chaque ligne vous aide à trouver des réponses à des questions telles que :

* Combien de ces exceptions se sont produites au cours de la [période](#time-period) sélectionnée ?
* Une version d&apos;application spécifique a-t-elle un nombre plus élevé (ou plus faible) d&apos;utilisateurs affectés ?
* Quelle exception a le moins d’occurrences ?

Vous pouvez modifier l&apos;ordre de tri ou les options de filtrage pour vous concentrer uniquement sur les types d&apos;exceptions qui comptent le plus pour vous et vos équipes. Pour afficher [des détails de thread supplémentaires pour chaque occurrence](/docs/mobile-monitoring/mobile-monitoring-ui/crashes/handled-exceptions-occurrences) de l&apos;exception, sélectionnez un enregistrement dans la table <DNT>**Top exceptions location**</DNT> .

## Trace d&apos;appels non symbolisée [#unsymbolicated]

New Relic ne stocke pas les traces d&apos;appels non symbolisées pour les exceptions traitées. Ceci est différent du [suivi des événements de crash](/docs/mobile-monitoring/mobile-monitoring-ui/crashes/mobile-crash-event-trail). Vous pouvez [télécharger un dSYM](/docs/mobile-monitoring/new-relic-mobile-ios/install-configure/upload-dsyms-bitcode-apps) ultérieurement si nécessaire, car les nouvelles exceptions gérées seront symbolisées.

## API des exceptions gérées [#hex-api]

Notre fonctionnalité <DNT>**Handled exceptions**</DNT> inclut automatiquement un attribut par défaut que vous pouvez utiliser pour explorer vos données d&apos;exceptions gérées dans le générateur de requêtes et obtenir des détails spécifiques :

* Utilisez la [méthode`recordHandledExceptions()` ](/docs/mobile-monitoring/new-relic-mobile/mobile-sdk/record-handled-exceptions/).
* [Requête](/docs/insights/nrql-new-relic-query-language/using-nrql/query-page-create-edit-nrql-queries) le type d&apos;événement [`MobileHandledException`](/docs/insights/insights-data-sources/default-data/mobile-events-attributes#mobilehandledexception-attributes) . Pour plus d&apos;informations, consultez les [exemples NRQL](/docs/insights/nrql-new-relic-query-language/nrql-query-examples/insights-query-examples-new-relic-mobile#mobilehandledexception-examples) pour <InlinePopover type="mobile" />.
