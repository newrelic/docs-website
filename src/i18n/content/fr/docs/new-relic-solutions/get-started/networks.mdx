## Chiffrement TLS [#tls]

Afin d&apos;assurer la sécurité des données de nos clients et d&apos;être en conformité avec [FedRAMP](https://marketplace.fedramp.gov/#/product/new-relic?sort=productName&productNameSearch=new%20relic) et d&apos;autres normes de [chiffrement des données](/docs/security/new-relic-security/compliance/data-encryption), toutes les connexions entrantes pour tous les domaines nécessitent Transport Layer Security (TLS) 1.2. Pour plus de détails, consultez [notre article du forum d&apos;assistance sur TLS 1.2](https://forum.newrelic.com/s/hubtopic/aAX8W0000008dM6WAI/tls-1011-has-been-disabled-for-all-inbound-connections-on-feb-2nd-2023).

Pour les futures mises à jour des versions de protocole requises et prises en charge, suivez la [tag `Security Notifications` dans le forum d&apos;assistance de New Relic](https://discuss.newrelic.com/t/security-notifications/45862).

### Ports [#ports]

Pour toutes les applications d&apos;ingestion de données, à l&apos;[exception d&apos;OpenTelemetry](#otel-ports), utilisez le port 443, un canal sécurisé pour le trafic HTTPS crypté et notre valeur par défaut.

Si vous disposez d&apos;une configuration existante qui utilise le port 80, nous vous recommandons de la mettre à jour pour utiliser le port 443.

#### Ports OpenTelemetry [#otel-ports]

Les ports utilisés pour `otlp.nr-data.net` et `otlp.eu01.nr-data.net` sont :

* 443
* 4317 (HTTP/2)
* 4318 (HTTP/1.1)

## domaine orienté utilisateur [#user-facing-domains]

Votre navigateur doit pouvoir communiquer avec un certain nombre de domaines pour que New Relic fonctionne correctement. Mettez à jour votre liste de domaines autorisés pour garantir que New Relic puisse communiquer avec un certain nombre de domaines intégrés répertoriés dans cette section. Le blocage d&apos;un domaine peut entraîner des problèmes avec les fonctionnalités individuelles d&apos;un produit ou empêcher le chargement complet des pages.

Cette liste ne couvre pas les domaines auxquels New Relic se connecte et qui peuvent être bloqués sans affecter votre utilisation du produit. Il ne couvre pas non plus les Nerdpacks ou autres fonctionnalités qui communiquent avec des services externes ayant des exigences de domaine supplémentaires.

Si votre organisation utilise un pare-feu qui restreint le trafic sortant, suivez les procédures spécifiques au système d&apos;exploitation et au pare-feu que vous utilisez pour ajouter le domaine suivant à la liste de domaines autorisés.

## Détails de l&apos;infrastructure [#infrastructure]

Afin de signaler des données à New Relic, notre [monitoring deinfrastructure](/docs/infrastructure/new-relic-infrastructure/getting-started/welcome-new-relic-infrastructure) a besoin d&apos;un accès [sortant au point de terminaison dans la table des points de terminaison .](#new-relic-endpoints) [TLS](#tls) est requis pour tous les domaines.

Si votre système a besoin d&apos;un proxy pour se connecter à New Relic, utilisez le [paramètre Infrastructure `proxy` ](/docs/infrastructure/new-relic-infrastructure/configuration/configure-infrastructure-agent#proxy).

Notre monitoring deinfrastructure utilise plusieurs autres points de terminaison d&apos;ingestion, notamment les points de terminaison métriques d&apos;API et les points de terminaison d&apos;API log (inclus dans [la table des points de terminaison](#new-relic-endpoints)).

Détails sur le point de terminaison non lié à l&apos;ingestion :

* `identity-api.newrelic.com` | `identity-api.eu.newrelic.com`: Obligatoire pour l’enregistrement d’une entité (par exemple, une entité `host` ).
* `infrastructure-command-api.newrelic.com` | `infrastructure-command-api.eu.newrelic.com`: Utilisé par l&apos;agent pour contrôler les aspects du comportement de l&apos;agent (par exemple, l&apos;utilisation d&apos;indicateurs de fonctionnalité).

## Détails de l&apos;agent APM [#apm]

Pour améliorer les performances du réseau et [la sécurité des données](/docs/security/new-relic-security/data-privacy/new-relic-security-protecting-your-data-content), New Relic utilise un service de prévention CDN et DDoS avec une large plage d&apos;adresses IP. Les agents New Relic nécessitent votre pare-feu pour autoriser les connexions sortantes vers le point de terminaison lié à l&apos;APM dans la [table des points de terminaison d&apos;ingestion](#new-relic-endpoints). Pour les ajouter à votre liste de domaines autorisés, suivez les procédures spécifiques au système d&apos;exploitation et au pare-feu que vous utilisez.

[TLS](#tls) est requis pour tous les domaines.

## Intégration de Pixie [#pixie-integration]

L&apos; [intégration Pixie](https://github.com/newrelic/newrelic-pixie-integration) s&apos;exécute dans votre cluster Kubernetes et extrait un ensemble de données d&apos;observabilité organisées de Pixie pour les envoyer à New Relic à l&apos;aide du protocole OpenTelemetry (OTLP).

L&apos;intégration de Pixie nécessite un accès réseau sortant aux éléments suivants :

* `work.withpixie.ai:443`
* `withpixie.ai:443`
* `otlp.nr-data.net:4317` ( data center américain)
* `otlp.eu01.nr-data.net:4317` ( data center de l&apos;UE)

<Callout variant="tip">
  Si le port `4317` ne fonctionne pas, vous pouvez utiliser le port `443`.
</Callout>

Le projet communautaire Pixie utilise des images de conteneurs hébergées dans [le registre de conteneurs de Google](https://cloud.google.com/container-registry). Assurez-vous que votre cluster peut extraire des images de `gcr.io`.
