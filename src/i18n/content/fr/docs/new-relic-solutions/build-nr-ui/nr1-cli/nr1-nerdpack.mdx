## `nr1 nerdpack:build` [#nr1-nerdpackbuild]

Construit un nerdpack.

### Usage [#build-usage]

```sh
nr1 nerdpack:build
```

### Options [#build-options]

<table>
  <tbody>
    <tr>
      <td>
        `--extra-metadata-path=extra-metadata-path`
      </td>

      <td>
        `default: extra-metadata.json` spécifiez un chemin de fichier json avec des métadonnées supplémentaires.
      </td>
    </tr>

    <tr>
      <td>
        `--prerelease=prerelease`
      </td>

      <td>
        Si spécifié, la valeur sera ajoutée à la version actuelle des fichiers générés. Par exemple `--prerelease=abc`. Alors la version sera `1.2.3-abc`.
      </td>
    </tr>

    <tr>
      <td>
        `--profile=profile`
      </td>

      <td>
        Le profil d’authentification que vous souhaitez utiliser.
      </td>
    </tr>

    <tr>
      <td>
        `--verbose`
      </td>

      <td>
        Ajoute des informations supplémentaires à la sortie.
      </td>
    </tr>
  </tbody>
</table>

### Description [#build-description]

Exécute un processus webpack pour assembler votre Nerdpack en bundles javascript et CSS . Comme beaucoup d&apos;autres commandes CLI, elle doit être exécutée au niveau `package.json` de votre Nerdpack. Il recherchera `nr1.json` fichiers dans vos sous-répertoires, les validera et essaiera de les créer. Ceci exécute également `nr1 nerdpack:validate` avant de démarrer le processus de construction.

## `nr1 nerdpack:clone` [#nr1-nerdpackclone]

Clone un nerdpack à partir d&apos;un référentiel git.

### Usage [#clone-usage]

```sh
nr1 nerdpack:clone
```

### Options [#clone-options]

<table>
  <tbody>
    <tr>
      <td>
        `-f`, `--force`
      </td>

      <td>
        Remplacez le dossier de destination s&apos;il existe.
      </td>
    </tr>

    <tr>
      <td>
        `-p`, `--path=path`
      </td>

      <td>
        Répertoire où cloner (par défaut le nom du référentiel).
      </td>
    </tr>

    <tr>
      <td>
        `-r`, `--repo=repo`
      </td>

      <td>
        Ceci est **obligatoire**. Emplacement du référentiel, soit un chemin HTTPS ou SSH.
      </td>
    </tr>

    <tr>
      <td>
        `--profile=profile`
      </td>

      <td>
        Le profil d’authentification que vous souhaitez utiliser.
      </td>
    </tr>

    <tr>
      <td>
        `--verbose`
      </td>

      <td>
        Ajoute des informations supplémentaires à la sortie.
      </td>
    </tr>
  </tbody>
</table>

### Description [#clone-description]

Étant donné un référentiel git, il exécute toutes les actions pour que vous puissiez commencer à l&apos;utiliser. Cela comprend, dans l’ordre :

1. Cloner le référentiel.
2. Définir le référentiel comme étant distant en amont.
3. Installez toutes ses dépendances en utilisant `npm`.
4. Générez un nouvel UUID en utilisant votre profil et validez-le.

      <td>
        `default: extra-metadata.json` spécifiez un chemin de fichier json avec des métadonnées supplémentaires.
      </td>

    <tr>

      <td>
        Le profil d’authentification que vous souhaitez utiliser.
      </td>
    </tr>

    <tr>
      <td>
        `--verbose`
      </td>

      <td>
        Ajoute des informations supplémentaires à la sortie.
      </td>
    </tr>
  </tbody>
</table>

### Description [#publish-description]

Cette commande va créer et télécharger votre Nerdpack dans le registre.

Si aucun paramètre supplémentaire n&apos;est transmis, cette commande tag automatiquement la version nerdpack comme `STABLE`. Si vous souhaitez désactiver ce comportement, transmettez l&amp;apos;indicateur `--skip-tag` à la commande.

Voir `nr1 nerdpack:tag --help` pour plus d&amp;apos;informations sur la balise.

## `nr1 nerdpack:tag` [#nr1-nerdpacktag]

taguer une version spécifique de nerdpack.

### Usage [#tag-usage]

```sh
nr1 nerdpack:tag
```

### Options [#tag-options]

<table>
  <tbody>
    <tr>
      <td>
        `-V`, `--from-version=from-version`
      </td>

      <td>
        Version à tag. S&apos;il n&apos;est pas fourni, celui de `package.json` sera utilisé.
      </td>
    </tr>

    <tr>
      <td>
        `-i`, `--nerdpack-id=nerdpack-id`
      </td>

      <td>
        UUID Nerdpack à déployer. S&apos;il n&apos;est pas fourni, celui de `nr1.json` sera utilisé.
      </td>
    </tr>

    <tr>
      <td>
        `-t`, `--tag=tag`
      </td>

      <td>
        `default: STABLE` nom tag
      </td>
    </tr>

    <tr>
      <td>
        `--profile=profile`
      </td>

      <td>
        Le profil d’authentification que vous souhaitez utiliser.
      </td>
    </tr>

    <tr>
      <td>
        `--verbose`
      </td>

      <td>
        Ajoute des informations supplémentaires à la sortie.
      </td>
    </tr>
  </tbody>
</table>

### Description [#tag-description]

Une seule version peut avoir plusieurs balises, mais chaque tag ne peut être appliquée qu&apos;à une seule version. marquer une version différente avec une tag existante supprimera la balise précédente.

Nous vous recommandons d&apos;utiliser `STABLE` pour tag les versions que vous souhaitez montrer à vos abonnés. Voir la commande `nr1 nerdpack:subscribe` pour plus d&amp;apos;informations.

### Alias [#tag-aliases]

```sh
$ nr1 nerdpack:deploy
```
