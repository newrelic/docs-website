En éliminant les plantages et en augmentant la vitesse sur la stack, vous pouvez créer de meilleures performances dans chaque sortie d&apos;application mobile. Voici quelques bonnes pratiques pour profiter pleinement du monitoring des applications mobiles avec New Relic.

## 1. Commencez à collecter des données [#start-collecting]

<DNT>
  **Start collecting data in production.**
</DNT>

Vous verrez une valeur immédiate en ajoutant notre SDK monitoring des applications mobiles à votre application pour sa prochaine sortie sur l&apos;App Store. Nous vous recommandons d&apos;installer l&apos;agent iOS ou Android sur votre sortie de production, même si vous testez simplement les capacités de New Relic. Cela vous permettra d&apos;obtenir une quantité suffisante de données pour vraiment comprendre toutes les capacités de nos applications de monitoring mobiles et votre expérience d&apos;utilisateur final.

### Comment faire [#1-how-to]

<CollapserGroup>
  <Collapser id="1-add-more" title="Ajoutez votre application mobile à New Relic">
    Tout d’abord, ajoutez une nouvelle application dans l’interface utilisateur de New Relic ([étapes complètes](/docs/mobile-monitoring/new-relic-mobile/getting-started/introduction-new-relic-mobile#new_users)) :

    1. Allez à <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &amp;gt; Mobile**</DNT>.

    2. Dans la liste des applications de monitoring, sélectionnez <DNT>**Add more**</DNT>.

    3. Sélectionnez la plateforme concernée.

    4. Saisissez un nom pour votre application mobile et sélectionnez <DNT>**Continue**</DNT>.

       Utilisez une application et le jeton d&apos;application associé pour la sortie de développement de votre application iOS, une autre application et son jeton d&apos;application associé pour la sortie de production de cette application iOS, et une autre paire de jetons d&apos;application pour la sortie de développement et de production de votre application Android . Vous pourrez comparer les données des quatre applications à l&apos;aide de notre générateur de requêtes.
  </Collapser>

  <Collapser id="1-install" title="Installer et configurer notre SDK mobile">
    Ensuite, installez et configurez l’agent mobile :

    1. Téléchargez la dernière version du SDK de New Relic pour iOS ou Android et ajoutez-la à votre application mobile. (Étapes complètes pour [iOS](/docs/mobile-monitoring/new-relic-mobile-ios/get-started/ios-installation-configuration#configuration) et [Android](/docs/mobile-monitoring/new-relic-mobile-android/get-started/android-installation-configuration#installation))
    2. Configurez votre installation selon vos besoins. Par exemple:
    3. Activez l&apos;indicateur de fonctionnalité pour capturer l&apos;événement `MobileRequest` afin d&apos;utiliser nos capacités d&apos;analyse de réseau complètes pour [iOS](/docs/mobile-monitoring/new-relic-mobile-ios/api-guides/ios-agent-configuration-feature-flags#networkRequest) ou [Android](/docs/mobile-monitoring/new-relic-mobile-android/api-guides/android-agent-configuration-feature-flags#feature-settings).
    4. Automatisez les téléchargements dSYM pour vos versions de production.
    5. Pour éviter les surprises, suivez le modèle standard par lequel vous démarrez notre agent mobile. De plus, démarrez notre agent avant de lancer tout autre SDK. Pour plus d&apos;informations, consultez notre documentation pour [l&apos;installation et la configuration d&apos;Android](/docs/mobile-monitoring/new-relic-mobile-android/install-configure/installing-android-apps-gradle-android-studio) et pour [l&apos;installation et la configuration d&apos;iOS](/docs/mobile-monitoring/new-relic-mobile-ios/get-started/ios-installation-configuration).
    6. présenter une nouvelle sortie de votre application.
  </Collapser>
</CollapserGroup>

## 2. Explorez vos données [#key-features]

<DNT>
  **Immediately be able to explore your data.**
</DNT>

Dès que New Relic commence à générer des rapports de données, prenez le temps d&apos;explorer la richesse des informations sur vos applications mobiles. Pour commencer, accédez à <DNT>**[one.newrelic.com](https://one.newrelic.com/all-capabilities)**</DNT> et sélectionnez <DNT>**Mobile**</DNT>.

### Comment faire [#2-howto]

<CollapserGroup>
  <Collapser id="2-howto" title="Analyser, résoudre et prévenir les pannes">
    Trouvez la cause des plantages afin de pouvoir les résoudre. Ensuite, découvrez les tendances derrière les crashs afin de pouvoir les prévenir :

    1. À partir de la page de résumé de l’interface utilisateur d’une application mobile de monitoring, cliquez sur <DNT>**Crash analysis**</DNT>.

    2. [Analysez les données de crash](/docs/mobile-monitoring/mobile-monitoring-ui/crashes/crash-analysis-group-filter-your-crashes) afin de pouvoir voir et corriger les crashs les plus impactants.

       Suivez le fil d&apos;Ariane dans le [suivi des événements crash](/docs/mobile-monitoring/mobile-monitoring-ui/crashes/mobile-crash-event-trail) pour corriger les crashs et résoudre les erreurs critiques plus facilement :

    3. Depuis la page <DNT>**Crash analysis**</DNT> de votre application mobile dans l’interface utilisateur, sélectionnez un <DNT>**Crash type**</DNT>.

    4. À partir de la page <DNT>**Summary**</DNT> du type de crash, sélectionnez l’onglet <DNT>**Crash trail**</DNT> .
  </Collapser>

  <Collapser id="workflows" title="Monitorez vos workflows clé">
    Créez [un fil d&apos;Ariane personnalisé](/docs/mobile-monitoring/new-relic-mobile/maintenance/add-custom-data-new-relic-mobile#custom-breadcrumbs) pour monitorer les workflows clé (comme le processus de connexion ou de paiement de votre application), suivre les clics des utilisateurs, optimiser le timing et identifier la région de plantation pour MTTR plus rapide.
  </Collapser>

  <Collapser id="sessions" title="Obtenez des données de session exploitables">
    Empruntez à notre [exemple de requête](/docs/insights/nrql-new-relic-query-language/nrql-examples/insights-query-examples-mobile) pour :

    * Découvrez dans quelle mesure votre base d&apos;utilisateurs a effectué une mise à niveau vers la dernière version de votre application.
    * Suivre l’adoption au fil du temps.
    * Obtenez un décompte des nouvelles installations par rapport aux mises à niveau d&apos;une version antérieure.
    * Consultez la répartition de l&apos;utilisation entre les emplacements géographiques, les opérateurs, les propriétaires d&apos;ASN (fournisseurs Wi-Fi), les appareils, les types de connexion, etc.
  </Collapser>
</CollapserGroup>

## 3. Prendre des mesures préventives [#crashes-performance]

<DNT>
  **Use alert notifications to help uncover the causes behind crashes and errors.**
</DNT>

Une fois que vous avez exécuté nos capacités monitoring des applications mobiles en production et que vous avez exploré certaines des fonctionnalités clés, vous pouvez utiliser New Relic pour découvrir les problèmes critiques de votre application mobile. Concentrez-vous particulièrement sur les plantages et les erreurs.

1. Utilisez <InlinePopover type="alerts" />basé sur des métriques comme point de départ pour obtenir une visibilité de haut niveau sur les problèmes.
2. Ensuite, à mesure que de nouveaux services ou fonctionnalités sont ajoutés, utilisez les alertes plus ciblées de New Relic, basées sur les événements (NRQL), pour vous assurer que les problèmes les plus critiques sont traités rapidement.

Utilisez ces alertes pour déterminer si un crash affecte un grand nombre d&apos;utilisateurs. Si tel est le cas, les personnes concernées (par exemple, le membre d&apos;astreinte de l&apos;équipe d&apos;application mobile et le propriétaire du service backend associé) seront alertées.

Assurez-vous que votre équipe sait comment réagir aux changements de taux de plantage (plantages par session) et au pourcentage d&apos;utilisateurs impactés par les plantages de votre application mobile.

### Comment faire [#3-howto-alerts]

<CollapserGroup>
  <Collapser id="3-howto-metric-alerts" title="alertes basées sur les métriques">
    Utilisez notre interface utilisateur d&apos;alertes et monitoring métriques des applications mobiles :

    * Assurez-vous que les membres clés de votre équipe mobile ont choisi de recevoir [des e-mails](/docs/mobile-monitoring/mobile-monitoring-ui/crashes/receive-crash-notifications-email) pour les nouveaux crashs. Ces e-mails vous redirigeront vers le type de crash dans la page d&apos;interface utilisateur de votre application mobile.

    * Créez une alerte pour votre [seuil de taux de plantage](/docs/alerts/new-relic-alerts/defining-conditions/define-alert-conditions#create-condition) que les membres clés de votre équipe peuvent consulter dans Slack ou tout autre canal de notification d&apos;alertes de votre choix.

      Votre notification d&apos;alerte inclura un lien vers notre interface utilisateur, où vous pourrez examiner un graphique avec le seuil dépassé. À partir de là, un lien vous mènera à la page monitoring des applications mobiles concernée, où vous pourrez obtenir des détails supplémentaires.
  </Collapser>

  <Collapser id="3-howto-nrql" title="alertes basées sur des événements">
    Combinez le workflow basé sur des métriques avec des alertes basées sur des événements et <InlinePopover type="dashboards" />spécifiques à votre application mobile et à votre système.

    1. Créez votre alerte NRQL qui se concentre sur les plantages de votre version de production la plus récemment sortie.
    2. Créez une alerte NRQL supplémentaire qui se concentre sur le pourcentage d&apos;utilisateurs impactés par des plantages dans la dernière version de votre application (pour vérifier l&apos;impact global sur votre base d&apos;utilisateurs).
  </Collapser>
</CollapserGroup>

## 4. Approfondissez vos recherches sur les accidents [#crash-reporting]

<DNT>
  **Take your crash reporting to the next level.**
</DNT>

Il est maintenant temps de s&apos;attaquer à certains de ces crashs plus difficiles à reproduire en :

1. Application d&apos;une instrumentation supplémentaire à votre application.
2. Tirer parti du suivi des événements en cas de crash, de la requête NRQL et du tableau de bord pour visualiser vos données.

### Comment faire [#4-howto]

<CollapserGroup>
  <Collapser id="4-howto-analysis" title="Analyse des accidents">
    Les pannes font inévitablement partie de l’exécution d’applications mobiles à grande échelle. Nous fournissons des outils pour vous aider à comprendre vos accidents les plus impactants. Utilisez l&apos;interface utilisateur <DNT>**Crash analysis**</DNT> pour voir quels plantages se produisent le plus souvent, le pourcentage d&apos;utilisateurs sans plantage, quels fichiers et méthodes sont associés à peut-être 80 % de vos plantages, quels plantages un utilisateur particulier vient de subir, et plus encore.

    Recherchez les accidents à fort impact grâce à l&apos;analyse des accidents :

    1. Ajoutez [des attributs personnalisés](/docs/insights/insights-data-sources/custom-events/insert-custom-events-attributes-mobile-data) pour catégoriser les clients les plus précieux, puis utilisez-les pour segmenter les crashs en fonction de vos publics les plus précieux.
    2. Ajoutez le fil d&apos;Ariane et l&apos;événement personnalisé pour produire une piste détaillée des événements menant à chaque occurrence de crash (au cours des 90 derniers jours). Ensuite, visualisez ce crash suivi des événements dans notre UI monitoring des applications mobiles.
    3. Utilisez notre page monitoring des applications mobiles [<DNT>**Versions trends**</DNT> ](/docs/mobile-monitoring/mobile-monitoring-ui/mobile-app-pages/version-trends-compare-user-adoption-metrics-performance)pour vous assurer d&apos;améliorer le taux de plantage au fil des sorties suivantes et d&apos;éviter les régressions.
  </Collapser>

  <Collapser id="4-howto-breadcrumbs" title="Fil d'Ariane">
    Utilisez le fil d&apos;Ariane et l&apos;attribut personnalisé pour mieux reproduire et déboguer les crashs. Par exemple, utilisez notre [API`recordBreadrumb` ](/docs/mobile-monitoring/new-relic-mobile/mobile-sdk/record-breadcrumb/)pour suivre les changements d&apos;état en cours de session qui peuvent aider à déboguer les plantages, tels que :

    * Changement de type de connexion
    * Changement d&apos;orientation
    * État du processeur et de la mémoire à des points clés du code de l&apos;application
  </Collapser>

  <Collapser id="4-howto-attributes" title="Les attributs personnalisés">
    Ajoutez un attribut personnalisé à l&apos;événement existant, afin de pouvoir poser des questions encore plus précises à votre application mobile. Ajoutez des dimensions telles que :

    * ID utilisateur : permet de réagir aux préoccupations spécifiques des utilisateurs et de mieux comprendre quels clients et segments sont les plus impactés.

    * ID du magasin : utilisé pour résoudre les problèmes liés aux appareils obsolètes, au mauvais Wi-Fi, etc.

    * Segments d&apos;utilisateurs : utilisez-les pour mieux comprendre quels clients et quels segments sont les plus impactés ; par exemple, connexion et non-connexion.

    * Argent dans le panier à la fin de la séance

    * Origine ou attribution de l&apos;entrée dans l&apos;application

      Normalisez les noms d&apos;attributs personnalisés sur vos applications mobiles et alignez-les sur votre site Web (si vous utilisez notre <InlinePopover type="browser" />), afin de pouvoir comparer et contraster les performances sur les différents canaux finaux des utilisateurs. Les agents mobiles suivent la dernière interaction des sessions en panne et non en panne pour fournir une visibilité sur ce que l&apos;utilisateur a vu avant de quitter votre application (intentionnellement ou en raison d&apos;un crash).
  </Collapser>
</CollapserGroup>

## 5. Concentrez-vous sur les indicateurs clés de performance [#key-performance]

<DNT>
  **Bring your key performance indicators to life.**
</DNT>

Créez des interactions et des événements personnalisés pour vous concentrer sur les signaux les plus importants, y compris l&apos;activité des utilisateurs pour les aspects critiques de votre application. Grâce à ces puissantes API, vous pouvez ajouter une instrumentation personnalisée à presque tout. Voici quelques exemples de cas d’utilisation.

### Comment faire [#5-howto]

<CollapserGroup>
  <Collapser id="5-howto-feature" title="Piste et requête d'utilisation des fonctionnalités">
    Utilisez événement personnalisé pour suivre l’utilisation des fonctionnalités. Par exemple:

    * Utilisez événement personnalisé pour suivre le flux utilisateur entre l&apos;application et le site Web.

    * Utilisez événement personnalisé ou mobile fil d&apos;Ariane pour optimiser les performances des flux utilisateurs clés (connexion , panier, etc.).

    * utilisez l&apos;événement personnalisé pour fournir des expériences de support plus rapides et contextualisées à vos clients numériques.

    * Utilisez l&apos;API `recordCustomEvent` . Comparez ensuite l’utilisation de différentes fonctionnalités en utilisant :

      ```sql
      SELECT count(*) FROM MyNewMobileEventType FACET name
      ```
  </Collapser>

  <Collapser id="5-howto-pageload" title="Mesurer le temps de chargement de la page">
    Utilisez événement personnalisé pour mesurer le temps de chargement de la page :

    1. Créez un minuteur de lancement d&apos;application pour mesurer le temps qui s&apos;écoule entre un appui de l&apos;utilisateur pour lancer votre application et le point dans le code de votre application où le premier écran est disponible. Enregistrez la valeur du minuteur dans New Relic en utilisant quelque chose comme ce qui suit :

       ```java
       recordCustomEvent(<myMobileTimers>, <appLaunch>, {attributes})
       ```

    2. Définissez des minuteries de démarrage et d&apos;arrêt pour mesurer la durée des spinners, puis enregistrez ce temps dans New Relic en utilisant quelque chose comme ce qui suit :

       ```java
       recordCustomEvent(<myMobileTimers>, <spinnerOnScreenA>, {attributes})
       ```

    3. Suivez les actions spécifiques des utilisateurs ou les étapes entonnoirs dans l&apos;application (comme « ajouter au panier ») et incluez le prix comme attribut sur cet événement.

    4. Mesurez les flux à travers l&apos;application en visualisant les étapes d&apos;entonnoirs liées à l&apos;événement personnalisé. Par exemple, créez un minuteur pour suivre les heures de début et de fin d&apos;un sous-flux ou d&apos;un flux entier sur plusieurs étapes pour comprendre combien de temps il a fallu à l&apos;utilisateur pour terminer le processus.
  </Collapser>

  <Collapser id="5-howto-interactions" title="Examiner les données sur l’interaction avec les pages Web">
    Utilisez une interaction personnalisée pour voir la trace associée aux mêmes pages Web.

    1. Utilisez l&apos;API d&apos;agent pour désactiver l&apos;interaction par défaut et concentrez-vous sur l&apos;interaction personnalisée afin de pouvoir examiner les chargements de pages les plus cruciaux pour vos entonnoirs principaux.

    2. Définissez les heures de début et de fin de ces interactions personnalisées pour trace les activités associées aux étapes individuelles de ces entonnoirs (par exemple, une action utilisateur spécifique ou un chargement de page).

    3. Suivez la dernière interaction des sessions plantées et non plantées afin de pouvoir voir ce que l&apos;utilisateur a consulté avant de quitter votre application (intentionnellement ou à cause d&apos;un crash).

       <DNT>
         **Example query for non-crashed sessions:**
       </DNT>

       ```sql
       SELECT count(*) FROM Mobile WHERE sessionCrashed IS NULL FACET lastInteraction SINCE 90 days ago
       ```

       <DNT>
         **Example query for crashed sessions:**
       </DNT>

       ```sql
       SELECT count(*) FROM Mobile WHERE sessionCrashed = true FACET lastInteraction SINCE 90 days ago
       ```
  </Collapser>
</CollapserGroup>
