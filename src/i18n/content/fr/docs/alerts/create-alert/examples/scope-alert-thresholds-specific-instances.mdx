Vous pouvez définir [un seuil](/docs/using-new-relic/welcome-new-relic/get-started/glossary#alert-threshold) d&apos;alerte qui se déclenche lorsqu&apos;il est dépassé par l&apos;une des instances de votre application Java . L&apos;application d&apos;une condition à l&apos;instance de votre application est utile pour détecter les anomalies qui se produisent uniquement dans un sous-ensemble de l&apos;instance de votre application.

Ces types d&apos;anomalies sont faciles à manquer pour les applications qui sont agrégées sur un grand nombre d&apos;instances. En examinant chaque instance, vous pouvez identifier plus rapidement l’origine des problèmes potentiels.

## Exemple [#example]

Dans cet exemple, vous configurez une politique pour une application Java avec trois instances. Vous souhaitez ouvrir un événement d&apos;alerte lorsque le seuil critique de la condition pour le taux d&apos;erreur de n&apos;importe quelle instance est supérieur à 0,02 % <DNT>**for at least**</DNT> cinq minutes.

## Créer une condition d&apos;alerte basée sur instance [#procedure]

<DNT>
  /* &lt;img title=&quot;Create alerts based on threshold instances&quot; alt=&quot;A screenshot showing what to select to create alerts based on threshold instances&quot; src=&quot;/images/alerts_screenshot-crop_thresholds-instances.webp&quot; /&gt; */
</DNT>

Pour créer une politique qui déclenche des notifications pour les événements d&apos;alerte par les instances individuelles de votre application :

1. Suivez le [processus de workflows de base](/docs/alerts/new-relic-alerts/configuring-alert-policies/alert-policy-workflow#alert-policy-process) pour configurer une politique.
2. Lors de la création d’une [condition](/docs/alerts/create-alert/create-alert-condition/alert-conditions) (étape 2), sélectionnez [<DNT>**APM**</DNT>](/docs/alerts/new-relic-alerts/configuring-alert-policies/select-product-targets-alert-condition).
3. Sélectionnez <DNT>**Application metric**</DNT> comme type de condition.
4. Pour évaluer les événements d&apos;alerte de seuil d&apos;alerte <DNT>**individually**</DNT> pour chacune des instances sélectionnées de l&apos;application, sélectionnez <DNT>**Scope to Java application instances**</DNT>.
5. Sélectionnez <DNT>**Next, select entities**</DNT>, puis identifiez une ou plusieurs applications pour cette condition.
6. Facultatif : Modifiez le délai après lequel <InlinePopover type="alerts" />forcera la fermeture des événements d&apos;alerte (la valeur par défaut est de 24 heures).
7. Utilisez la [préférence d&apos;événement d&apos;alerte](/docs/alerts/organize-alerts/specify-when-alerts-create-events) <DNT>**By condition**</DNT> ou <DNT>**By condition and signal**</DNT>.
8. Continuez le reste du [processus de workflow des politiques (étape 3)](/docs/alerts/new-relic-alerts/configuring-alert-policies/alert-policy-workflow#alert-policy-process).

<Callout variant="tip">
  Pour ouvrir des événements d&apos;alerte basés sur <DNT>**average**</DNT> de toutes les instances de vos applications, sélectionnez <DNT>**Scope to the application**</DNT> au lieu de <DNT>**Scope to Java application instances**</DNT>.
</Callout>

## Utiliser la préférence d&apos;événement d&apos;alerte &quot;Par condition&quot; [#targets]

Lors de la configuration de la [préférence d&apos;événement d&apos;alerte](/docs/alerts/organize-alerts/specify-when-alerts-create-events) pour les politiques contenant des conditions basées sur des instances, nous vous recommandons de sélectionner <DNT>**By condition**</DNT> plutôt que <DNT>**By condition and signal**</DNT>. Bien que les applications soient les entités sélectionnées pour cette condition, nous évaluons chaque JVM comme une entité distincte.

Si vous définissez votre <DNT>**Alert event preference**</DNT> sur <DNT>**By condition and signal**</DNT>, un événement d&apos;alerte distinct sera ouvert pour chaque JVM qui dépasse le seuil critique. Lorsqu&apos;une application subit une défaillance sur plusieurs JVM, cela peut entraîner une « fatigue » des alertes et de la frustration.

## Utiliser l&apos;API REST pour les alertes d&apos;instance

Pour créer instanceune [condition d&apos;alerte basée sur avec l&apos;API REST de New Relic](/docs/alerts/new-relic-alerts/rest-api-alerts/manage-entities-alerts-conditions), incluez ces éléments avec votre appel d&apos;API REST :

* Votre [clé API](/docs/apis/rest-api-v2/getting-started/introduction-new-relic-rest-api-v2#api_key).
* Le [`entity_id`](/docs/using-new-relic/welcome-new-relic/get-started/glossary#entity) numérique pour l&apos;[entité](/docs/using-new-relic/welcome-new-relic/get-started/glossary#entity) monitorée
* Le [`condition_id`](/docs/using-new-relic/welcome-new-relic/get-started/glossary#alert-condition) (disponible depuis l&apos;API Explorer : [<DNT>**Alerts Conditions &gt; GET &gt; List**</DNT>](https://rpm.newrelic.com/api/explore/alerts_entity_conditions/list))
* Le `entity_type` (défini sur `"application"`)
* Le `condition_scope` (défini sur `"instance"` pour une instance d&apos;application Java ou `"application"` pour une application Java)

Voici un exemple du format de demande d&apos;API et de la réponse JSON.
