Lorsque vous créez une condition, vous définissez des <DNT>**thresholds**</DNT> personnalisés qui déterminent ce qui ouvrira un événement d&apos;alerte. Ce document explique ce que sont les seuils et comment les définir.

## Qu&apos;est-ce qu&apos;un seuil ? [#threshold-definition]

Pour une [condition](/docs/using-new-relic/welcome-new-relic/get-started/glossary#alert-condition), les seuils sont les paramètres qui déterminent ce qui déclenche un [événement d&apos;alerte](/docs/new-relic-solutions/get-started/glossary#alert-event). En fonction de la [préférence de création de problèmes](/docs/alerts/organize-alerts/specify-when-alerts-create-events) d&apos;une politique et des éventuels [workflows](/docs/alerts/get-notified/alert-event-workflows/) que vous avez configurés, un événement d&apos;alerte peut entraîner :

* La création d&apos;un problème
* Notification envoyée

### Il existe deux types de seuil : [#threshold-types]

* <DNT>**Static**</DNT>:Une valeur définie par vous.
* <DNT>**Anomaly**</DNT>:Un seuil [d&apos;anomalie](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/other-condition-types/create-anomaly-alert-conditions/#set-anomaly-thresholds) utilise les données passées pour prédire de manière dynamique le comportement des données dans un futur proche. Cela s&apos;ajustera au fil du temps à mesure qu&apos;il apprendra les modèles de vos données.

### Exemples de seuil : [#threshold-examples]

* Le temps de réponse Web moyen d’une application est supérieur à 5 secondes pendant 15 minutes.
* Le taux d&apos;erreur par minute d&apos;une application atteint 10 % ou plus au moins une fois par heure.
* Le temps de réponse AJAX d&apos;une application s&apos;écarte d&apos;une certaine manière de son comportement attendu.

Outre un niveau de seuil critique, vous pouvez également définir un seuil pour un [niveau d&apos;avertissement](#threshold-levels) moins grave.

  <Collapser id="signal-loss" title="Perte de signal (NRQL uniquement)">
    Une perte de signal est une période pendant laquelle aucune donnée n&apos;est reçue par New Relic. Cela pourrait être le résultat d&apos;une entité ou d&apos;un service hors ligne, d&apos;un problème avec un agent ou un collecteur, ou de problèmes réseau dans un centre de données ou sur Internet. Vous pouvez utiliser la détection de perte de signal pour créer un nouvel événement d&apos;alerte lorsqu&apos;un signal s&apos;arrête de manière inattendue. Vous pouvez également l&apos;utiliser pour déterminer quand un service éphémère s&apos;arrête et définir l&apos;action pour fermer tous les événements d&apos;alerte ouverts existants pour cette condition. Une autre option des paramètres de perte de signal consiste à ignorer l&apos;ouverture d&apos;un événement d&apos;alerte lorsqu&apos;il est prévu qu&apos;un signal s&apos;interrompe. Consultez la [documentation sur la perte de signal](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/#signal-loss) pour plus d&apos;informations.

    Vous devrez peut-être également configurer la perte de signal si vous utilisez `count(*)` ou `uniqueCount(*)` dans votre requête. Consultez [les conditions NRQL et l’ordre des opérations de requête](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/#query-order) pour plus de détails.
  </Collapser>

  <Collapser id="time-intervals" title="Définir des intervalles de temps">
