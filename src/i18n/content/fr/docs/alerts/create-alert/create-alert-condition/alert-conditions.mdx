Une condition d’alerte est l’élément central qui définit quand un [incident](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/understand-technical-concepts/incident-event-attributes/#definition) est créé. Il constitue le point de départ essentiel pour créer toute alerte significative. la condition d&amp;apos;alerte contient le paramètre ou le seuil atteint avant que vous en soyez informé. Ils peuvent atténuer les alertes excessives ou informer votre équipe lorsqu&amp;apos;un comportement nouveau ou inhabituel apparaît.

## Créer une nouvelle condition d&apos;alerte [#create-alert-condition]

Une condition d&apos;alerte est une requête exécutée en continu qui mesure un ensemble donné d&apos;événements par rapport à un seuil défini et ouvre un [incident](/docs/alerts-applied-intelligence/new-relic-alerts/alert-policies/specify-when-alerts-create-incidents/) lorsque le seuil est atteint pendant une fenêtre de temps spécifiée.

Cet exemple montre la création manuelle d’une nouvelle condition d’alerte à l’aide de la page <DNT>**Alert condition details**</DNT> . Mais il existe de nombreuses façons de créer une condition d’alerte. Vous pouvez créer une condition d&amp;apos;alerte à partir de :

* [Un graphique](/docs/tutorial-create-alerts/create-an-alert/)
* [Une page de politique](https://one.newrelic.com/nr1-core/condition-builder/policy-entity)
* [La page <DNT>**Alert coverage gaps**</DNT>](https://one.newrelic.com/alerts-ai/detection-gaps/)

Vous pouvez également utiliser l&apos;un de nos générateurs d&apos;alertes :

* Utilisez <DNT>**Write your own query**</DNT> pour créer des alertes à partir de zéro

* <DNT>
    **Use guided mode**
  </DNT>

Pour toutes les méthodes à l’exception de notre mode guidé, le processus de création d’une condition d’alerte sera *exactement* le même que celui décrit dans les étapes ci-dessous.

<Steps>
  <Step>
    ### Définissez le comportement de votre signal

    Dans cet exemple, imaginez que votre équipe gère l’application `WebPortal` pour un site de commerce électronique. Vous souhaitez être alerté de tout problème de latence.

    <CollapserGroup>
      <Collapser id="set-your-query" title="Définissez le comportement de votre signal">
        Vous pouvez utiliser une requête NRQL pour définir les signaux que vous souhaitez qu&apos;une condition d&apos;alerte utilise comme base pour votre alerte. Pour cet exemple, vous utiliserez cette requête :

        ```sql
        SELECT average(duration)
        FROM PageView
        WHERE appName = 'WebPortal'
        ```

        L&apos;utilisation de cette requête pour votre condition d&apos;alerte indique New Relic que vous souhaitez connaître la latence moyenne, ou la durée, de chargement des pages dans votre application `WebPortal`. L&amp;apos;alerte proactive sur la latence, un élément central des signaux dorés, fournit des alertes précoces sur une dégradation potentielle.

        Vous pouvez en apprendre plus sur l&apos;utilisation de NRQL, le langage de requête de New Relic, consultez notre [documentation NRQL](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions/).
      </Collapser>
    </CollapserGroup>
  </Step>

  <Step>
    ### Ajuster les paramètres avancés du signal [#advanced-signal-settings]

    Après avoir défini votre signal, cliquez sur <DNT>**Run**</DNT>. Un graphique apparaîtra et affichera le paramètre que vous avez défini.

    <Callout variant="tip">
      Pour configurer des alertes inter-comptes, sélectionnez un **compte de données** dans la liste déroulante. Et vous ne pouvez interroger les données que d&amp;apos;un seul compte à la fois pour les alertes inter-comptes.
    </Callout>

    <img title="set a query" alt="A screenshot showing a user how to set the signal behavior for an alert condition." src="/images/alerts_screenshot-crop_set-a-query.webp" />

    Pour cet exemple, le graphique montrera la durée moyenne d&apos;une transaction. Cliquez sur <DNT>**Next**</DNT> et commencez à configurer votre condition d’alerte.

    Pour cet exemple, vous allez personnaliser ces paramètres de signal avancés pour la condition que vous avez créée pour monitorer la latence dans votre application `WebPortal`.

    <CollapserGroup>
      <Collapser id="window-duration" title="Durée de la fenêtre">
        <img title="fine tune alert settings" alt="A screenshot depicting advanced signal settings." src="/images/alerts_screenshot-crop_fine-tune-alert-signals.webp" />

        La durée de la fenêtre définit la manière dont New Relic regroupe vos données pour analyse dans une condition d&apos;alerte. Le choix du bon paramètre dépend de la fréquence de vos données et du niveau de détail souhaité :

        * <DNT>**High-frequency data (for example, pageviews every minute)**</DNT>: Définissez la durée de la fenêtre pour qu&amp;apos;elle corresponde à la fréquence des données (1 minute dans ce cas) pour les informations en temps réel détaillées sur les fluctuations et les tendances.
        * <DNT>**Low-frequency data (for example, hourly signals)**</DNT>:Choisissez une durée de fenêtre qui capture suffisamment de données pour révéler des modèles et des anomalies (par exemple, 60 minutes pour les signaux horaires).

        N&apos;oubliez pas que vous pouvez personnaliser la durée de la fenêtre en fonction de vos besoins et de votre expérience. Nous vous recommandons d&apos;utiliser les valeurs par défaut lors du démarrage et d&apos;expérimenter à mesure que vous vous sentez plus à l&apos;aise avec la création de conditions d&apos;alerte.
      </Collapser>

      <Collapser id="sliding-window" title="Utiliser l'agrégation de fenêtres glissantes">
        Les méthodes d’agrégation traditionnelles peuvent s’avérer insuffisantes lorsqu’il s’agit de traiter des données peu peuplées ou présentant des fluctuations importantes entre les intervalles. Voici comment utiliser l&apos;agrégation de fenêtres glissantes pour analyser ces données et déclencher efficacement des alertes opportunes :

        1. <DNT>**Smooth out the noise**</DNT>:Commencez par créer une grande fenêtre d’agrégation. Cette fenêtre (par exemple, 5 minutes) agit comme un tampon, atténuant le « bruit » ou la variabilité inhérente à vos données. Cela permet d’éviter les alertes intempestives déclenchées par des pics ou des creux isolés.

        2. <DNT>**Avoid lag with a sliding window**</DNT>:Bien qu&amp;apos;une grande fenêtre facilite l&amp;apos;analyse des données, si vous attendez que l&amp;apos;intervalle entier soit écoulé avant de vérifier le seuil, vous pouvez rencontrer des retards importants dans la notification des alertes. Nous recommandons des fenêtres coulissantes plus petites (par exemple, une minute). Imaginez cette fenêtre coulissante comme un cadre mobile analysant vos données dans la fenêtre d’agrégation plus grande. Chaque fois que le cadre avance de son intervalle le plus petit, il calcule une valeur globale (par exemple, une moyenne).

        3. <DNT>**Set your threshold duration**</DNT>:Vous pouvez désormais définir votre seuil d’alerte dans le contexte de la fenêtre glissante plus petite. Cela vous permet de déclencher rapidement des alertes lorsque la valeur globale de la trame actuelle s&amp;apos;écarte considérablement de la plage souhaitée sans sacrifier l&amp;apos;effet de lissage de la fenêtre plus grande.

           Vous pouvez en apprendre davantage sur l’agrégation de fenêtres glissantes dans [ce didacticiel NRQL](/docs/query-your-data/nrql-new-relic-query-language/nrql-query-tutorials/create-smoother-charts-with-sliding-windows/).
      </Collapser>

      <Collapser id="streaming-method" title="Méthode de diffusion en continu">
        En général, nous recommandons d&apos;utiliser la méthode de streaming <DNT>**event flow**</DNT> . Cette option est idéale pour les données qui arrivent fréquemment et régulièrement dans votre système. Il existe des cas spécifiques où <DNT>**event timer**</DNT> pourrait être une meilleure méthode à choisir, mais pour votre première alerte, nous recommandons notre valeur par défaut, <DNT>**event flow**</DNT>. Nous vous recommandons de regarder cette courte vidéo (environ 5:31 minutes) pour mieux comprendre quelle méthode de streaming choisir.

        <Video type="wistia" id="n6nei987ln" />
      </Collapser>

      <Collapser id="delay" title="Retard">
        La fonctionnalité de retard dans la condition d&apos;alerte protège contre les problèmes potentiels découlant d&apos;une collecte de données incohérente. Il agit comme un tampon, laissant du temps supplémentaire aux données pour arriver et être traitées avant de déclencher une alerte. Cela permet d’éviter les faux positifs et garantit une création incident plus précise.

        Comment ça marche :

        Le paramètre de délai approprié est déterminé en évaluant la cohérence de vos données entrantes :

        * Données cohérentes : un paramètre de délai inférieur est suffisant si les points de données arrivent systématiquement avec un horodatage dans un délai d&apos;une minute.
        * Données incohérentes : si les points de données arrivent avec un horodatage s&apos;étendant sur plusieurs minutes dans le passé ou le futur, un paramètre de délai plus élevé est nécessaire pour tenir compte de l&apos;incohérence.

        Créer un tampon :

        * Le paramètre de délai sélectionné introduit une période d&apos;attente avant que la condition d&apos;alerte évalue les données par rapport au seuil défini.
        * Cette mémoire tampon laisse le temps aux divergences de données de se résorber, réduisant ainsi le risque d&apos;alertes trompeuses.
      </Collapser>

      <Collapser id="gap-filling-strategy" title="Stratégie de comblement des lacunes">
        Vous créez une condition d&apos;alerte pour informer votre équipe de tout problème de latence avec l&apos;application `WebPortal`. Dans cet exemple, votre application envoie systématiquement des données New Relic. Un flux constant de signaux est envoyé depuis votre application vers New Relic, et il n&amp;apos;y a pas d&amp;apos;écart prévu dans le signal, vous n&amp;apos;aurez donc pas besoin de sélectionner une stratégie de comblement des écarts.

        Les stratégies de comblement des lacunes s’adressent aux scénarios dans lesquels la collecte de données peut être intermittente ou incomplète. Ils fournissent une méthode permettant de remplacer les points de données manquants par des valeurs estimées, garantissant que la condition d&apos;alerte peut toujours fonctionner efficacement même avec des lacunes dans le flux de données.

        Quand ne pas utiliser le remplissage des lacunes :

        * <DNT>**Consistent data flow**</DNT>:Si votre application envoie systématiquement des données à New Relic sans interruptions attendues, comme dans le cas de l&amp;apos;application WebPortal, le remplissage des interruptions est généralement inutile. Dans de tels cas, laisser la stratégie de comblement des lacunes définie sur « aucune » est souvent l’approche la plus appropriée.

        Considérations clés :

        * <DNT>**Popular use case**</DNT>:Une utilisation courante du remplissage d’espaces vides consiste à insérer une valeur de 0 pour les fenêtres sans données reçues.

        * <DNT>**Anomaly thresholds**</DNT>:La valeur de remplissage des lacunes est interprétée comme le nombre d&amp;apos;écarts types par rapport à la dernière valeur observée lors de l&amp;apos;utilisation du seuil d&amp;apos;anomalie. Par exemple, combler les lacunes avec une valeur de 0 reproduirait la dernière valeur observée, supposant ainsi qu&amp;apos;il n&amp;apos;y ait aucun changement.

          Apprenez-en davantage sur les stratégies de comblement des lacunes dans nos [documents sur les signaux perdus](/docs/apis/nerdgraph/examples/nerdgraph-api-loss-signal-gap-filling/).
      </Collapser>

      <Collapser id="cross-account-alerts" title="Alertes inter-comptes">
        Les alertes inter-comptes dans New Relic vous permettent de configurer une condition d&apos;alerte qui monitore les données d&apos;un compte différent de celui sur lequel l&apos;alerte est configurée. Cette fonctionnalité permet une plus grande flexibilité dans monitoring et la gestion des dépendances sur plusieurs comptes au sein de New Relic.

        Pour plus d&apos;informations, reportez-vous à [Alerte inter-comptes](/docs/alerts/create-alert/create-alert-condition/cross-account-alert/).
      </Collapser>
    </CollapserGroup>
  </Step>

  <Step>
    ### Fixer le seuil de condition d&apos;alerte [#thresholds]

    Si une condition d&apos;alerte est un conteneur, alors les seuils sont les règles que chaque condition d&apos;alerte doit suivre. Au fur et à mesure que les données circulent dans votre système, la condition d&apos;alerte recherche tout incident lié à ces règles. Si la condition d&apos;alerte voit des données de votre système qui ont rempli toutes les conditions que vous avez définies, cela créera un incident. Un incident signale que quelque chose ne va pas dans votre système et que vous devriez y prêter attention.

    <img title="set a threshold" alt="A screenshot depicting how to set the threshold for an alert condition." src="/images/alerts_screenshot-crop_set-a-threshold-for-an-alert-condition.webp" />

    <CollapserGroup>
      <Collapser id="anomaly-threshold" title="seuil d'anomalie">
        Les seuils d&apos;anomalie sont idéaux lorsque vous êtes plus préoccupé par l&apos;écart par rapport aux modèles attendus que par des valeurs numériques spécifiques. Ils vous permettent de monitorer les activités inhabituelles sans avoir à définir des limites prédéfinies. La détection d&apos;anomalies de New Relic analyse dynamiquement vos données au fil du temps, en adaptant le seuil pour refléter l&apos;évolution du comportement du système.

        Configuration de la détection d&apos;anomalies :

        1. Choisissez supérieur ou inférieur :

           * Sélectionnez supérieur et inférieur pour être alerté de tout écart supérieur ou inférieur aux prévisions.
           * Sélectionnez « supérieur uniquement » pour vous concentrer uniquement sur les valeurs inhabituellement élevées.

        2. Attribuer une priorité :
           * Définissez le niveau de priorité sur critique pour votre alerte initiale afin de garantir une attention prompt aux problèmes potentiels.

        3. Définir les critères de violation :

           * Commencez avec les paramètres par défaut : ouvrez un incident lorsqu&apos;une requête renvoie une valeur qui s&apos;écarte de la valeur prédite pendant au moins cinq minutes de trois écarts types.
           * Personnalisez ces paramètres selon vos besoins pour les adapter à votre application spécifique et à vos exigences d&apos;alerte.

        Vous pouvez en apprendre davantage sur les niveaux de priorité dans nos [documents sur les conditions d&apos;alerte](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/advanced-techniques/set-thresholds-alert-condition#threshold-levels).

        Apprenez-en plus sur le seuil d&apos;anomalie et les comportements du modèle dans notre [documentation](/docs/alerts-applied-intelligence/applied-intelligence/anomaly-detection/anomaly-detection-applied-intelligence/) sur les anomalies.
      </Collapser>

      <Collapser id="static-threshold" title="Seuil statique">
        Contrairement au seuil d&apos;anomalie, un seuil statique ne considère pas votre ensemble de données dans son ensemble et détermine quel comportement est inhabituel en fonction de l&apos;historique de votre système. Au lieu de cela, un seuil statique ouvrira un incident chaque fois que votre système se comporte différemment des critères que vous avez définis.

        Vous devez définir le niveau de priorité pour l&apos;anomalie et le seuil statique. Voir la section ci-dessus pour plus de détails.
      </Collapser>

      <Collapser id="lost-signal" title="Seuil de perte de signal (optionnel)">
        Le [seuil de signal perdu](/docs/alerts/create-alert/create-alert-condition/create-nrql-alert-conditions/#signal-loss) détermine combien de temps attendre avant de considérer un signal manquant comme perdu. Si le signal ne revient pas dans ce délai, vous pouvez choisir d&amp;apos;ouvrir un nouvel incident ou de fermer ceux qui y sont liés. Vous pouvez également choisir d’ignorer l’ouverture d’un incident lorsqu’un signal doit se terminer. Définissez le seuil en fonction du comportement attendu de votre système et de la fréquence de collecte des données. Par exemple, si un site Web subit une perte totale de trafic, ou de débit, les données télémétriques correspondantes envoyées à New Relic cesseront également. la monitoring de cette perte de signal peut servir de système d&amp;apos;alerte précoce pour de telles pannes.
      </Collapser>
    </CollapserGroup>
  </Step>

  <Step>
    ### Ajouter des détails sur condition d&apos;alerte [#add-details]

    À ce stade du processus, vous disposez désormais d’une condition entièrement définie et définissez toutes les règles pour garantir qu’un incident est ouvert lorsque vous le souhaitez. En fonction des paramètres ci-dessus, si votre condition d&apos;alerte reconnaît ce comportement dans votre système qui dépasse le seuil que vous avez défini, cela créera un incident. Il ne vous reste plus qu’à nommer cette condition et à l’associer à une politique.

    La politique est le système de tri des incident. Lorsque vous créez une politique, vous créez l’outil qui organise tous vos incidents entrants. Vous pouvez connecter des politiques à <DNT>**[workflows](/docs/alerts-applied-intelligence/applied-intelligence/incident-workflows/incident-workflows/)**</DNT> qui indiquent à New Relic où vous souhaitez que toutes ces informations entrantes soient envoyées, à quelle fréquence vous souhaitez qu&amp;apos;elles soient envoyées et où.

    <img title="name an alert condition " alt="A screenshot demonstrating how you can new alert condition." src="/images/alerts_screenshot-crop_name-your-alert-condition.webp" />

    <CollapserGroup>
      <Collapser id="name-your-condition" title="Nommez votre état">
        Les bonnes pratiques pour la dénomination des conditions impliquent un format structuré qui transmet les informations essentielles en un coup d&apos;œil. Incluez les éléments suivants dans les noms de vos conditions :

        * **Priorité**: Indique la gravité ou l&amp;apos;urgence de l&amp;apos;alerte, comme P1, P2, P3.

        * **Signal**: Spécifiez la métrique ou la condition monitorée, comme latence moyenne élevée ou débit faible.

        * **entité**: identifiez le système, application ou le composant concerné, comme l&amp;apos;application WebPortal ou le serveur de base de données.

          Un exemple de nom de condition bien formé suivant cette structure serait `P2 | High Avg Latency | WebPortal App`.
      </Collapser>

      <Collapser id="select-an-existing-policy" title="Sélectionnez une politique existante">
        Si vous disposez déjà d’une politique que vous souhaitez connecter à une condition d’alerte, sélectionnez la politique existante.

        Apprenez-en davantage sur la création de politiques [ici](/docs/alerts-applied-intelligence/new-relic-alerts/alert-policies/specify-when-alerts-create-incidents/).
      </Collapser>

      <Collapser id="create-a-new-policy" title="Créer une nouvelle politique">
        Il est essentiel d&apos;équilibrer la réactivité et la fatigue dans votre stratégie d&apos;alerte, et vous avez défini les considérations clés concernant monitoring `pageview` de votre application `WebPortal`. Explorons les options politiques :

        1. Un problème par politique (par défaut) :

           * Avantages : Réduit le bruit et assure une action immédiate.
           * Inconvénients : regroupe tous les incidents sous un seul problème, même s&apos;ils sont déclenchés par des conditions différentes. Ce n&apos;est pas idéal pour les problèmes de pages vues multiples.

        2. Un problème par condition :

           * Avantages : crée des problèmes distincts pour chaque condition, idéal pour isoler et résoudre des problèmes de latence spécifiques.
           * Inconvénients : peut générer plus d’alertes, ce qui peut potentiellement entraîner de la fatigue.

        3. À chaque incident son problème :

           * Avantages : Fournit des détails granulaires pour le système externe, mais n&apos;est pas optimal pour la consommation interne en raison d&apos;une surcharge potentielle.
           * Inconvénients : c’est l’option la plus bruyante et il est difficile de suivre les tendances plus larges et d’établir des priorités efficacement.

           Apprenez-en davantage sur la création de politiques [ici](/docs/alerts-applied-intelligence/new-relic-alerts/alert-policies/specify-when-alerts-create-incidents/).
      </Collapser>

      <Collapser id="close-open-incidents" title="Fermer l'incident ouvert">
        Un incident se ferme automatiquement lorsque le signal cible revient à un état de non-violation pendant la période indiquée dans le seuil de la condition. Ce temps d’attente est appelé période de récupération.

        Par exemple, si vous mesurez la latence et que le comportement de violation est que `duration` pour charger des pages dans votre application `WebPortal` a augmenté de plus de 3 secondes, l&amp;apos;incident se fermera automatiquement lorsque `duration` sera égal ou inférieur à 3 secondes pendant 5 minutes consécutives.

        Lorsqu&apos;un incident se ferme automatiquement :

        1. L&apos;horodatage de clôture est rétroactif au début de la période de récupération.

        2. L&apos;évaluation se réinitialise et redémarre à partir du moment où l&apos;incident précédent s&apos;est terminé.

           Toutes les conditions disposent d&apos;un paramètre de limite de temps d&apos;incident qui force automatiquement la fermeture d&apos;un incident de longue durée.

           New Relic définit automatiquement une durée de 3 jours par défaut et vous recommande d&apos;utiliser nos paramètres par défaut pour votre première alerte.

           Une autre façon de fermer un incident ouvert lorsque le signal ne renvoie pas de données est de configurer un seuil `loss of signal` . Reportez-vous à la section sur le seuil de perte de signal ci-dessus pour plus de détails.
      </Collapser>

      <Collapser id="custom-incident-description" title="Définir une description d'incident personnalisée">
        Étant donné que vous créez une condition d&apos;alerte qui vous permet de savoir s&apos;il existe des problèmes de latence avec votre application `WebPortal`, vous souhaitez vous assurer que vos développeurs disposent de toutes les informations dont ils ont besoin lorsqu&amp;apos;ils sont informés de cet incident. Vous utiliserez le flux de travail pour informer un canal Slack d&amp;apos;équipe lorsqu&amp;apos;un incident est créé.

        Apprenez-en plus sur les descriptions d’incidents personnalisées dans notre [documentation](/docs/alerts-applied-intelligence/new-relic-alerts/advanced-alerts/advanced-techniques/alert-custom-incident-descriptions).
      </Collapser>

      <Collapser id="title-template" title="Utiliser le modèle de titre">
        <img title="alert details page" alt="A screenshot depicting the final step of creating an alert condition- the alert details page" src="/images/alerts_screenshot-crop_adding-alert-details-to-an-alert-condition.webp" />

        L&apos;utilisation du modèle de titre est facultative mais nous le recommandons. Une condition d&apos;alerte définit un ensemble de seuil que vous souhaitez monitorer. Si l’un de ces seuils est dépassé, un incident est créé. Des modèles de titres significatifs vous aident à identifier les problèmes et à résoudre les pannes plus rapidement.

        Apprenez-en plus sur les modèles de titres dans notre [documentation](//docs/alerts/create-alert/condition-details/title-template/).
      </Collapser>

      <Collapser id="runbook" title="Ajouter l'URL runbook">
        Un runbook d&apos;exploitation détaillant les étapes d&apos;enquête, de triage ou de correction est souvent lié dans ce domaine.
      </Collapser>
    </CollapserGroup>
  </Step>
</Steps>

## Modifier une condition d&apos;alerte existante [#edit-existing-alert-condition]

Si vous souhaitez modifier une condition d&apos;alerte que vous avez déjà créée, vous pouvez :

1. Accédez à [one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &amp;gt; <DNT>**Alerts**</DNT>.
2. Sélectionnez <DNT>**Alert Conditions**</DNT> dans la navigation de gauche.
3. Cliquez sur la condition d’alerte que vous souhaitez modifier.

De là, vous verrez la page <DNT>**Alert conditions details**</DNT> . Cette page contient tous les éléments que vous avez définis lors de la création de votre condition. Vous pouvez modifier des aspects spécifiques de la condition d&amp;apos;alerte en cliquant sur le crayon en haut à droite de chaque section.

### Historique du signal [#signal-coverage]

Sous <DNT>**Signal history**</DNT>, vous pouvez voir les résultats les plus récents de la requête NRQL que vous avez utilisée pour créer votre condition d&amp;apos;alerte. Pour cet exemple, vous verrez la latence moyenne sur l&amp;apos;application `WebPortal` pour la période spécifique que vous avez définie.

Pour toutes les conditions d&apos;alerte construites avec une requête NRQL , le <DNT>**Signal history**</DNT> sera présenté avec un graphique linéaire.

Toute condition d&apos;alerte construite avec un moniteur Synthétique sera un peu différente. Cela est dû au fait que le moniteur Synthétique vous permet de pinger votre application à partir de plusieurs emplacements, ce qui peut produire des résultats positifs ou négatifs à chaque exécution du moniteur. Ces données ne peuvent être présentées que sous forme de tableau.

## Types de conditions [#condition-types]

Le type de condition principal et recommandé est une condition d’alerte NRQL, mais il existe d’autres types de conditions. Nous avons inclus une liste complète de nos types de conditions ci-dessous.

<CollapserGroup>
  <Collapser id="nrql-conditions" title="Conditions de requête NRQL">
    Utilisez l&apos;UI ou [APINerdGraph](/docs/alerts/alerts-nerdgraph/nerdgraph-examples/nerdgraph-api-alerts-nrql-conditions) pour créer des conditions NRQL pour [les requêtes NRQL de base qui renvoient un nombre](/docs/new-relic-alerts-alert-nrql-queries).
  </Collapser>

  <Collapser id="apm-conditions" title="APM métrique condition d'alerte">
    Consultez [APM métrique condition d&apos;alerte](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/apm-metric-alert-conditions) pour obtenir des conseils sur la création de <InlinePopover type="apm" />conditions à l&amp;apos;aide de NRQL.
  </Collapser>

  <Collapser id="anomaly-conditions" title="Conditions d'anomalie">
    [Les alertes d&apos;anomalie](/docs/alerts/new-relic-alerts/configuring-alert-policies/create-anomaly-alert-conditions) vous permettent de créer des conditions qui s&amp;apos;ajustent de manière dynamique aux données et aux tendances changeantes, telles que les modèles hebdomadaires ou saisonniers. Cette fonctionnalité est disponible pour les applications [<InlinePopover type="apm" />](/docs/apm/new-relic-apm/getting-started/welcome-new-relic-apm)et [<InlinePopover type="browser" />](/docs/browser/new-relic-browser/getting-started/new-relic-browser), ainsi que pour [les requêtesNRQL ](/docs/alerts/new-relic-alerts/defining-conditions/create-alert-conditions-nrql-queries).
  </Collapser>

  <Collapser id="synthetics-multi-location" title="monitoring synthétique des conditions multi-sites">
    Avec [les conditions monitoring Synthétique multi-emplacements](/docs/alerts/new-relic-alerts/defining-conditions/multi-location-synthetics-alert-conditions), vous pouvez configurer un moniteur pour vous avertir lorsqu&amp;apos;un nombre spécifique d&amp;apos;emplacements tombent en panne en même temps.
  </Collapser>

  <Collapser id="key-transactions" title="transaction clé métriques conditions">
    Pour APM, vous pouvez définir des conditions pour [la clé de transaction](/docs/apm/transactions/key-transactions/introduction-key-transactions).
  </Collapser>

  <Collapser id="instance-incidents" title="Conditions d'instance Java">
    Vous pouvez définir des seuils qui ouvrent un incident lorsqu&apos;ils sont violés par Java instance l&apos;une des métriques de votre application .

    En [limitant le seuil à un cas spécifique](/docs/alerts/new-relic-alerts/configuring-alert-policies/scope-alert-thresholds-specific-instances), vous pouvez identifier plus rapidement l&amp;apos;origine des problèmes potentiels. Cela est utile, par exemple, pour détecter des anomalies qui se produisent uniquement dans un sous-ensemble de l&amp;apos;instance de votre application. Ces types d&amp;apos;anomalies sont faciles à manquer pour les applications qui sont agrégées sur un grand nombre d&amp;apos;instances.
  </Collapser>

  <Collapser id="jvm-metrics" title="Conditions de santé JVM (applications Java )">
    Pour le moniteur d&apos;applications Java d&apos;APM, vous pouvez définir [un seuil](/docs/using-new-relic/welcome-new-relic/get-started/glossary#alert-threshold) qui ouvre un incident lorsque la taille du tas ou le nombre de threads pour une seule JVM est hors de la plage de fonctionnement prévue.

    Nous évaluons les violations du seuil d&apos;alerte individuellement pour chacune des [instances sélectionnées](#instance-incidents) de l&amp;apos;application. Lors de la création de votre condition, sélectionnez <DNT>**JVM health metric**</DNT> comme type de condition pour la règle d&amp;apos;alerte de votre application Java , puis sélectionnez l&amp;apos;un des seuils disponibles :

    * Fils bloqués

    * Utilisation de la mémoire du tas

    * Temps d&apos;utilisation du processeur

    * Temps CPU de collecte des déchets

      l&apos;incident se fermera automatiquement lorsque l&apos;inverse du seuil sera atteint, mais en utilisant l&apos;UI vous pouvez également modifier le moment où un [incident](/docs/new-relic-solutions/get-started/glossary/#alert-incident) se ferme de force pour une métrique de santé JVM . La valeur par défaut est 24 heures.
  </Collapser>

  <Collapser id="app-response-percentiles" title="conditions percentile de transaction Web">
    Nous incluons l&apos;option permettant de définir un percentile comme [seuil](/docs/using-new-relic/welcome-new-relic/get-started/glossary#alert-threshold) de votre condition lorsque le temps de réponse de votre application Web est supérieur, inférieur ou égal à cette valeur. Cela est utile, par exemple, lorsque le personnel des opérations souhaite alerter sur un percentile pour le temps de réponse [Web de transaction](/docs/using-new-relic/welcome-new-relic/getting-started/glossary#transaction) <DNT>**overall**</DNT> d&amp;apos;un serveur d&amp;apos;applications plutôt que sur le temps de réponse Web <DNT>**average**</DNT>.

    <Callout variant="tip">
      Si vous souhaitez définir un seuil arbitraire dans une condition pour une [transaction d&apos;application non Web](/docs/using-new-relic/welcome-new-relic/getting-started/glossary#non-web-transaction), utilisez la fonctionnalité [de requêteNRQL ](#nrql-conditions).
    </Callout>

    Pour définir le seuil percentile :

    1. Sélectionnez <DNT>**Web transactions percentiles**</DNT> comme type de condition pour la condition de votre application <InlinePopover type="apm" />, puis sélectionnez une seule application. (Pour alerter sur plusieurs applications, créez une condition <DNT>**Web transactions percentiles**</DNT> individuelle pour chacune.)

    2. Pour définir le seuil d&apos;ouverture de l&apos;incident, saisissez la valeur <DNT>**Percentile nth**</DNT> temps de réponse, puis sélectionnez sa fréquence (supérieure, inférieure ou égale à cette valeur).

       Nous stockons le temps de transaction en millisecondes, bien que l&apos;interface utilisateur affiche les valeurs critiques et d&apos;avertissement en secondes. Si vous souhaitez définir des millisecondes, assurez-vous d&apos;inclure le point décimal dans votre valeur.
  </Collapser>

  <Collapser id="dynamic-targeting-labels" title="Ciblage dynamique avec des étiquettes pour les applications">
    En appliquant [des étiquettes](/docs/data-analysis/user-interface-functions/organize-your-data/labels-categories-organize-apps-servers-monitors#labels) à l&amp;apos;application, vous pouvez automatiquement lier ces [entités](/docs/using-new-relic/welcome-new-relic/get-started/glossary#alert-entity) à votre condition. Cela facilite la gestion de toutes les applications dans un environnement dynamique. Nous vous recommandons d&amp;apos;utiliser le [agent configuration fichier](/docs/data-analysis/user-interface-functions/organize-your-data/labels-categories-organize-apps-servers-monitors#config) pour mieux gérer les étiquettes d&amp;apos;entité.

    Une seule étiquette identifie <DNT>**all**</DNT> entité associée à cette étiquette (maximum 10 000 entités). Plusieurs étiquettes identifient uniquement les entités qui partagent toutes les étiquettes sélectionnées.

    L&apos;utilisation du ciblage dynamique avec votre condition nécessite également que vous définissiez un [minuteur de fermeture d&apos;incident](/docs/alerts-applied-intelligence/new-relic-alerts/alert-incidents/how-alert-condition-incidents-are-closed#time-limit).

    Pour ajouter, modifier ou supprimer jusqu&apos;à dix étiquettes pour une condition :

    1. Sélectionnez <DNT>**APM &gt; Application metric**</DNT> comme type de produit.

    2. Lors de l&apos;identification de l&apos;entité, sélectionnez l&apos;onglet <DNT>**Labels**</DNT> . Recherchez une étiquette par nom ou sélectionnez une étiquette dans la liste des catégories.

       Vous pouvez également créer des conditions directement dans le contexte de ce que vous monitoring avec [l&apos;infrastructure](/docs/infrastructure/new-relic-infrastructure/configuration/infrastructure-alerts-add-edit-or-view-host-alert-information).
  </Collapser>

  <Collapser id="infrastructure" title="État des infrastructures">
    Vous pouvez [créer des conditions pour vos ressources](/docs/infrastructure/new-relic-infrastructure/configuration/infrastructure-alerts-add-edit-or-view-host-alert-information) directement depuis notre monitoring d&amp;apos;infrastructure UI.

    Par exemple, pour obtenir une notification lorsqu&apos;un agent d&apos;infrastructure cesse de recevoir des données, utilisez le type de condition [hôte ne signalant pas](/docs/infrastructure/new-relic-infrastructure/configuration/create-infrastructure-host-not-reporting-condition) . Cela vous permet d&amp;apos;alerter dynamiquement sur des groupes d&amp;apos;hôtes filtrés et de configurer la fenêtre temporelle de 5 à 60 minutes.
  </Collapser>
</CollapserGroup>

## Ajouter des conditions à une politique [#add-conditions]

Pour [ajouter d’autres conditions à une politique](/docs/alerts/new-relic-alerts-beta/configuring-alert-policies/define-alert-conditions):

1. Aller à <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &amp;gt; Alerts &amp;gt; Alert Policies**</DNT>
2. Détecter une politique.
3. Cliquez sur <DNT>**Add a condition**</DNT>.

Pour [créer une nouvelle condition NRQL](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-nrql-alert-conditions):

1. Aller à <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &amp;gt; Alerts &amp;gt; Alert Conditions**</DNT>
2. Cliquez sur <DNT>**Add a condition**</DNT>.

## Copier une condition [#copy-condition]

Pour copier une condition existante, y compris sa [cible](/docs/apm/new-relic-apm/getting-started/glossary#alert-target) et [son seuil](/docs/apm/new-relic-apm/getting-started/glossary#alert-threshold), et l&amp;apos;ajouter à une autre politique pour le compte sélectionné :

1. Allez à <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &amp;gt; Alerts &amp;gt; Alert conditions**</DNT>.

2. Dans la liste des conditions d&apos;alerte, cliquez sur l&apos;icône à trois points de l&apos;alerte que vous souhaitez copier et sélectionnez <DNT>**Duplicate condition**</DNT>.

3. À partir de <DNT>**Copy alert condition**</DNT>, recherchez ou faites défiler la liste pour sélectionner la politique à laquelle vous souhaitez ajouter cette condition.

4. Facultatif : modifiez le nom de la condition si nécessaire.

5. Facultatif : cliquez sur l&apos;interrupteur à bascule pour <DNT>**Enable on save**</DNT>

6. Sélectionnez <DNT>**Copy condition**</DNT>.

   Par défaut, la règle d&apos;alerte sélectionnée ajoutera la condition copiée dans l&apos;état **Désactivé** . Suivez les procédures standard pour [ajouter](/docs/alerts/new-relic-alerts-beta/configuring-alert-policies/define-alert-conditions) ou copier davantage de conditions à la règle d&amp;apos;alerte, puis [**activez** la condition](/docs/alerts/new-relic-alerts-beta/updating-alert-policies/disable-or-delete-alert-policies-conditions#disable-conditions) selon vos besoins. De plus, la nouvelle condition ne copiera aucune [balise](/docs/new-relic-solutions/new-relic-one/core-concepts/use-tags-help-organize-find-your-data/#add-via-ui-api) ajoutée à la condition d&amp;apos;origine.

## Activer/désactiver une condition [#disable-conditions]

Pour désactiver ou réactiver une condition :

1. Allez à <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &amp;gt; Alerts &amp;gt; Alert Conditions**</DNT>. Ensuite, dans la liste de <DNT>**Alert conditions**</DNT>, utilisez la bascule pour activer ou désactiver la condition.
2. Cliquez sur le commutateur <DNT>**On/Off**</DNT> pour l&amp;apos;activer.

Si vous [copiez une condition](/docs/alerts/new-relic-alerts-beta/updating-alert-policies/copy-alert-conditions), elle est automatiquement enregistrée dans la nouvelle politique comme désactivée (<DNT>**Off**</DNT>), même si la condition était activée (<DNT>**On**</DNT>) dans la politique d&amp;apos;origine.

## Supprimer une condition [#delete-conditions]

Pour désactiver une condition tout en la conservant dans la politique, [désactivez](/docs/alerts/new-relic-alerts-beta/updating-alert-policies/disable-or-delete-alert-policies-conditions#condition-on-off) -la. Pour supprimer une ou plusieurs conditions :

1. Allez à <DNT>**[one.newrelic.com &gt; All capabilities](https://one.newrelic.com/all-capabilities) &amp;gt; Alerts &amp;gt; Alert Conditions**</DNT>.

2. Dans la liste de <DNT>**Alert conditions**</DNT>, sélectionnez une condition, puis cliquez sur <DNT>**Delete**</DNT> dans le menu à ellipses (...).

   <Callout variant="tip">
     Si vous ne voyez pas le bouton Supprimer, il se peut que l&apos;administrateur de votre compte ait désactivé la suppression des conditions pour votre organisation.
   </Callout>

## Dépanner la page de condition d&apos;alerte [#troubleshoot]

Si vous voyez un signal vide dans le graphique historique sur la page **Alert Condition**, envisagez l&amp;apos;une des solutions suivantes :

* <DNT>**Review the condition&apos;s settings**</DNT>:Vérifiez que tous les éléments sont correctement configurés.
* <DNT>**Inspect NRQL queries**</DNT>: Assurez-vous qu&amp;apos;ils ciblent des métriques ou des événements valides et renvoient des données.
* <DNT>**Examine entity configuration**</DNT>: Confirmez que l’entité est correctement configurée pour envoyer des signaux.
* <DNT>**Consult New Relic documentation**</DNT>:Reportez-vous aux guides pertinents pour obtenir de l’aide sur des problèmes spécifiques.
