## Problème [#problem]

Après l&apos;installation de l&apos;agent monitoring réseau `ktranslate`, vous rencontrez des problèmes lors de la collecte de la télémétrie du flux réseau.

## Arrière-plan [#background]

`ktranslate` renvoie la télémétrie de flux brute collectée, sans modifier aucune partie de la charge utile du paquet. Il existe [plusieurs types de flux](/docs/network-performance-monitoring/setup-performance-monitoring/network-flow-monitoring/#supported-network-flow-data-types) pris en charge dès la sortie de la boîte, les plus importants étant [NetFlow v5](https://www.cisco.com/c/en/us/td/docs/net_mgmt/netflow_collection_engine/3-6/user/guide/format.html), [NetFlow v9](https://datatracker.ietf.org/doc/html/rfc3954), [sFlow](https://sflow.org/sflow_version_5.txt) et [IPFIX](https://datatracker.ietf.org/doc/html/rfc7011).

<SideBySide>
  <Side>
    Toutes les données de télémétrie de flux réseau sont stockées dans le [type d&apos;événement KFlow](/docs/network-performance-monitoring/setup-performance-monitoring/network-flow-monitoring/#find-your-metrics). Vous pouvez interroger ceci directement dans NRQL. Si ce type d’événement est absent, cela indique que votre compte ne reçoit pas les données.
  </Side>

  <Side>
    ```sql
    FROM KFlow 
    SELECT *
    ```
  </Side>
</SideBySide>

## Solution [#solution]

<CollapserGroup>
  <Collapser id="ktranslate-configuration" title="configuration du flux sur votre agent réseau monitoring">
    L&apos;agent `ktranslate` ne peut collecter qu&amp;apos;un seul type de modèle de flux configuré par l&amp;apos;argument [-nf.source](/docs/network-performance-monitoring/advanced/ktranslate-container-management/#container-runtime-options) lors de l&amp;apos;exécution, dont la valeur par défaut est `auto`. Cela indique `ktranslate` d&amp;apos;attendre tous les modèles de `NetFlow v5` | `NetFlow v9` | `sFlow` | `IPFIX` afin qu&amp;apos;il puisse traduire les paquets. Un problème courant consiste à configurer `ktranslate` pour écouter un type spécifique de modèle de flux, puis à exporter un autre type dans l&amp;apos;agent. Vous devez exécuter un conteneur séparé pour tous les modèles non couverts par `auto`.

    Une autre erreur courante ici consiste à exporter votre télémétrie de flux vers un agent `ktranslate` , tout en ciblant plusieurs ports comme destination. Dans ce scénario, vous devez exécuter plusieurs agents `ktranslate` , chacun étant défini sur une valeur différente pour [-nf.port](/docs/network-performance-monitoring/advanced/ktranslate-container-management/#container-runtime-options) au moment de l&amp;apos;exécution (la valeur par défaut est `9995`). Vous devrez peut-être également mettre à jour la configuration de l&amp;apos;exportateur de flux sur vos périphériques réseau source pour cibler leur port spécifique.
  </Collapser>

  <Collapser id="device-configuration" title="configuration du flux sur votre périphérique réseau">
    Chaque fournisseur aura sa propre documentation sur la configuration correcte de ses appareils pour l&apos;exportation des flux réseau. Les versions plus avancées comme `NetFlow v9`, `IPFIX` et `sFlow` disposent d&amp;apos;options qui permettent à l&amp;apos;administrateur de personnaliser les champs collectés et exportés. La modification de ces éléments peut effectivement désactiver la capacité de traiter correctement les enregistrements de flux par `ktranslate`.

    Les champs suivants sont <DNT>**required**</DNT>:

    <CollapserGroup>
      <Collapser id="required-record-group" title="Obligatoire pour tous les enregistrements">
        * Protocole (Numéro de type de champ : `4`) - Protocole de couche 4
        * Adresse source (Numéro de type de champ : `8`, `27`) - Adresse IPv4 ou IPv6 source
        * Port source (Numéro de type de champ : `7`) - Port TCP/UDP source
        * Adresse de destination (Numéro de type de champ : `12`, `28`) - Adresse IPv4 ou IPv6 de destination
        * Port de destination (Numéro de type de champ : `11`) - Port TCP/UDP de destination
      </Collapser>

      <Collapser id="required-interfaces-group" title="Champs d'interfaces obligatoires (au moins un)">
        * Interface de réception (numéro de type de champ : `10`) - index SNMP pour l&amp;apos;interface d&amp;apos;entrée
        * Transmission d&apos;interface (Numéro de type de champ : `14`) - Index SNMP pour l&amp;apos;interface de sortie
      </Collapser>

      <Collapser id="required-bytes-group" title="Champs d'octets obligatoires (au moins un)">
        * Octets Delta (Numéro de type de champ : `1`) - Octets Delta
        * Nombre total d&apos;octets (type de champ : `85`) - Nombre total d&amp;apos;octets
        * Octets sortants (type de champ numéro : `23`) - Octets sortants
        * Octets initiateurs (Numéro de type de champ : `231`) - Octets initiateurs
        * Octets du répondeur (Numéro de type de champ : `232`) - Octets du répondeur
      </Collapser>

      <Collapser id="required-packets-group" title="Champs de paquets obligatoires (au moins un)">
        * Paquets Delta (Numéro de type de champ : `2`) - Paquets Delta
        * Nombre total de paquets (type de champ : `86`) - Nombre total de paquets
        * Paquets sortants (Numéro de type de champ : `24`) - Paquets sortants
        * Paquets initiateurs (Numéro de type de champ : `298`) - Paquets initiateurs
        * Paquets de réponse (Numéro de type de champ : `299`) - Paquets de réponse
      </Collapser>

      <Collapser id="required-routing-group" title="Champs de routage obligatoires (au moins un)">
        * ToS (Numéro de type de champ : `5`) - Type de service
        * Source AS (Numéro de type de champ : `16`) - Numéro du système autonome BGP source
        * Destination AS (Numéro de type de champ : `17`) - Numéro de système autonome BGP de destination
        * AS source homologue (numéro de type de champ : `129`) - Numéro de système autonome BGP source homologue
        * AS de destination homologue (Numéro de type de champ : `128`) - Numéro de système autonome BGP de destination homologue
      </Collapser>

      <Callout variant="tip">
        `ktranslate` définit par défaut tous les enregistrements de flux sur `Direction: ingress` , sauf si l&amp;apos;enregistrement utilise explicitement une valeur de `egress`. Cela couvre diverses situations dans lesquelles les enregistrements de flux sont envoyés sans le champ `Direction` .
      </Callout>
    </CollapserGroup>
  </Collapser>

  <Collapser id="tcpdump-confirmation" title="Confirmation de la réception du flux">
    Chaque fournisseur aura sa propre documentation sur le monitoring des compteurs d&apos;exportation de flux via l&apos;interface CLI/UI de son appareil. L&apos;absence de croissance du compteur sur l&apos;appareil indique que l&apos;exportation du flux n&apos;est pas configurée correctement sur l&apos;appareil.

    Les options de déploiement du conteneur Docker et du service Linux pour `ktranslate` utilisent le réseau de l&amp;apos;hôte pour recevoir des données sur le port mappé. Afin de valider que les enregistrements de flux sont reçus par l&amp;apos;hôte, vous pouvez utiliser l&amp;apos;utilitaire [tcpdump](https://www.tcpdump.org) pour créer un fichier de capture de paquets (`.pcap`) que vous pourrez ensuite examiner dans [Wireshark](https://www.wireshark.org/).

    L&apos;exécution de cette commande définira `tcpdump` pour capturer chaque paquet entrant sur toutes les interfaces de l&amp;apos;hôte et écrire la sortie dans un fichier dans le répertoire actuel :

    ```shell
    sudo tcpdump -s 0 -i any -w dump_capture.pcap
    ```

    Notez que vous pouvez ajouter plusieurs arguments à [tcpdump](https://www.tcpdump.org/manpages/tcpdump.1.html), l&amp;apos;élément le plus important ici est le fichier de sortie que vous pouvez utiliser pour une analyse ultérieure. La sortie des résultats dans `STDOUT` crée une valeur limitée pour les objectifs actuels.

    Une fois que vous avez ce fichier, la section suivante vous montrera comment analyser les résultats.

    <Callout variant="tip">
      L’un des problèmes les plus courants rencontrés est une règle de configuration réseau/pare-feu qui bloque les paquets des périphériques réseau sources vers l’hôte `ktranslate` cible. Si vous n&amp;apos;obtenez aucun résultat avec l&amp;apos;utilitaire `tcpdump` , le meilleur endroit pour commencer le dépannage est de confirmer vos règles réseau et votre configuration `iptables` .
    </Callout>
  </Collapser>

  <Collapser id="improperly-formatted-flow-records" title="Identifier les modèles d'enregistrement de flux avec Wireshark">
    Suivez ces étapes pour utiliser [Wireshark](https://www.wireshark.org/) pour inspecter le fichier de capture de paquets.

    <Steps>
      <Step>
        Démarrez l&apos;application Wireshark et ouvrez le fichier de capture de paquets
      </Step>

      <Step>
        La vue initiale affiche tous les paquets capturés, mais pour l&apos;analyse du flux, vous devrez configurer l&apos;application pour les décoder correctement. À l’aide du menu, accédez à `Analyze > Decode As...`, ce qui ouvre une nouvelle fenêtre contextuelle.
      </Step>

      <Step>
        Dans la fenêtre contextuelle, cliquez sur l’icône plus (`+`) en bas à gauche, ce qui ajoutera une nouvelle ligne au panneau. L&amp;apos;option initiale dans la colonne `Current` est `(none)`. Cliquez ici pour ouvrir un menu déroulant, puis sélectionnez `CFLOW` pour `NetFlow` et `IPFIX`, ou `sFlow` pour les paquets `sFlow` . Cliquez sur `OK` en bas à droite pour revenir à l&amp;apos;interface utilisateur principale.

        <Callout variant="tip">
          Ce menu est classé par ordre alphabétique en respectant la casse. L&apos;option `sFlow` se trouve tout en bas de la liste.
        </Callout>
      </Step>

      <Step>
        Dans l&apos;interface utilisateur principale, vous devriez maintenant pouvoir voir les paquets `CFLOW` | `sFlow` en les identifiant dans la colonne `Protocol` . L&amp;apos;application du filtre d&amp;apos;affichage `(cflow or sflow)` isolera automatiquement les paquets dont vous avez besoin de tout ce qui pourrait se trouver dans le fichier de capture.

        Les sections suivantes décrivent comment inspecter chaque type de paquet.

        <CollapserGroup>
          <Collapser id="netflow-ipfix-analysis" title="Analyse des paquets NetFlow et IPFIX">
            `NetFlow` et les protocoles `IPFIX` utilisent une approche de modèle où l&amp;apos;administrateur peut identifier les champs à collecter, à partir d&amp;apos;une liste d&amp;apos;options standard. L&amp;apos;analyse de ces paquets est effectuée pour garantir que les [champs obligatoires](#device-configuration) pour `ktranslate` sont capturés.

            Dans l&apos;interface utilisateur principale de Wireshark, cliquez pour sélectionner un seul paquet `CFLOW` , puis développez la section intitulée `FlowSet n`, où `n` est un entier qui identifie un enregistrement de flux singulier dans un paquet. Vous développerez ensuite le sous-groupe `Flow n` pour analyser les champs de cet enregistrement de flux.

            <Callout variant="tip">
              Vous pouvez également cliquer sur le lien `Template Frame` dans le paquet pour accéder à un paquet capturé contenant le modèle de tous les flux de cet appareil.
            </Callout>
          </Collapser>
