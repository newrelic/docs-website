Si vous avez [configuré le monitoring des logs de flux Azure VNet](/docs/network-performance-monitoring/setup-performance-monitoring/cloud-flow-logs/azure-vnet-flow-log-monitoring) et que vous rencontrez des problèmes, utilisez les sections suivantes pour diagnostiquer les problèmes courants avec le redirecteur de logs de VNet Flow.

## Aucune donnée de log de flux dans New Relic [#no-data]

Si les données de log de flux n’apparaissent pas dans New Relic, vérifiez les éléments suivants :

* Confirmez que `NR_LICENSE_KEY` est correctement configuré.
* Vérifiez que le log de VNet Flow est activé et écrit activement des blobs PT1H.json sur votre compte de stockage.
* Vérifiez que l’abonnement Event Grid est actif et achemine les événements `BlobCreated` vers l’Event Hub.
* Confirmez que les paramètres `EVENTHUB_CONSUMER_CONNECTION` et `EVENTHUB_NAME` correspondent à l&apos;Event Hub recevant les événements.
* Vérifiez les logs de la Function App dans Azure pour les erreurs. Pour voir les logs détaillés, définissez le paramètre d&apos;application `AzureFunctionsJobHost__logging__logLevel__Function` sur `Debug`.

## Enregistrements de log de flux en double [#duplicate-data]

Le mécanisme de curseur empêche le traitement des doublons dans des conditions normales. Si vous voyez des doublons, vérifiez les éléments suivants :

* Vérifiez que `CURSOR_STORAGE_CONNECTION` est correctement configuré et que la table est accessible.
* Vérifiez qu&apos;il n&apos;y a qu&apos;une seule instance active du consommateur traitant chaque partition Event Hub.

## Événements ignorés (protection contre les événements empoisonnés) [#poison-events]

Après `MAX_CONSECUTIVE_FAILURES` échecs consécutifs (5 par défaut) lors du traitement du même blob, le forwarder le marque comme un événement empoisonné et l&apos;ignore pour éviter des tentatives infinies. Vérifiez les logs de la Function App pour trouver la cause première des échecs initiaux, corrigez le problème et réinitialisez le curseur en supprimant la ligne correspondante de la table `nrvnetflowlogscursors`. Le forwarder réessaie d&apos;abord les erreurs de livraison New Relic (429, 5xx et les erreurs réseau pouvant être réessayées). Le compteur d&apos;échecs ne s&apos;incrémente qu&apos;une fois ces tentatives épuisées ou si un autre échec matériel se produit.
