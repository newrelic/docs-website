Cette page décrit les options de gestion du conteneur <DNT>**ktranslate**</DNT> utilisé par le réseau monitoring de New Relic.

## exigences relatives aux conteneurs [#container-requirements]

Nous recommandons les ressources suivantes pour l&apos;image du conteneur <DNT>**ktranslate**</DNT> :

<DNT>
  **Disk**
</DNT>

* 100 Mo d&apos;espace disque disponible

<DNT>
  **CPU**
</DNT>

* Interrogation/collecte d&apos;interruptions SNMP : 1 cœur de processeur dédié pour environ 1 000 appareils
* Collecte de flux de périphériques : 1 cœur de processeur dédié pour environ 2 000 flux par seconde (ips)
* Collecte de messages Syslog : 1 cœur de processeur dédié pour environ 2 000 messages par seconde

<DNT>
  **Memory**
</DNT>

* <DNT>**ktranslate**</DNT> n&amp;apos;est généralement pas limité par les ressources mémoire. La quantité de mémoire sur votre hôte doit être déterminée par les types d&amp;apos;applications/conteneurs que vous prévoyez d&amp;apos;exécuter. Pour une idée générale, nous constatons généralement du succès avec des tailles d’image aussi petites que l’ [AWS t2.micro](https://aws.amazon.com/ec2/instance-types/t2/) qui dispose de 1 vCPU et de 1,0 Go de RAM disponible.

<Callout variant="tip">
  L&apos;image conteneur KTranslate exécute un seul « type de travail » à la fois. Par instance, un conteneur déployé pour l&apos;interrogation SNMP et la collecte d&apos;interruptions ne sera pas utilisé pour la collecte de flux. De plus, les conteneurs déployés pour la collecte de flux sont limités à un seul type `-nf.source` par conteneur. Cela signifie qu&amp;apos;il est courant d&amp;apos;avoir plusieurs conteneurs déployés sur un seul hôte Docker à un moment donné. Ils peuvent également partager un fichier de configuration commun, mais ce n’est pas obligatoire.
</Callout>

## Mise à jour du conteneur [#updating-the-container]

Maintenir l&apos;image du conteneur <DNT>**ktranslate**</DNT> à jour est une bonne pratique pour recevoir les dernières mises à jour et résoudre les problèmes courants via divers débogages appliqués au cours du cycle de développement. Il est recommandé de toujours récupérer la dernière image disponible lors du redéploiement de votre conteneur.

1. Extrayez la dernière image de conteneur disponible en exécutant l’une des opérations suivantes :

   * [Docker Hub](https://hub.docker.com/)
     ```shell
     docker pull kentik/ktranslate:v2
     ```
   * [Quay.io](https://quay.io/)
     ```shell
     docker pull quay.io/kentik/ktranslate:v2
     ```

2. Collectez les identifiants et les noms de tous les conteneurs existants :

   ```shell
   docker ps -a --filter ancestor=kentik/ktranslate:v2 --format "{{.ID}} - {{.Names}}"
   ```

   Exemple de sortie :

   ```
   3297b134a352 - ktranslate-snmp
   4962a854b386 - ktranslate-sflow
   ```

3. Retirer tout conteneur préexistant

   ```shell
   docker rm -f $CONTAINER_ID
   ```

4. Redéployez votre conteneur <DNT>**ktranslate**</DNT> en utilisant les paramètres d&amp;apos;origine que vous avez déployés à partir de [SNMP](/docs/network-performance-monitoring/setup-performance-monitoring/snmp-performance-monitoring), [de données de flux](/docs/network-performance-monitoring/setup-performance-monitoring/network-flow-monitoring) ou [de collectesyslog ](/docs/network-performance-monitoring/setup-performance-monitoring/network-syslog-monitoring).

<Callout variant="important">
  Le fichier de configuration utilisé par <DNT>**ktranslate**</DNT> est appliqué au conteneur lors de l&amp;apos;exécution. Les modifications apportées à ce fichier nécessitent que vous supprimiez et redémarriez vos conteneurs en cours d&amp;apos;exécution pour appliquer les modifications, à l&amp;apos;exception de l&amp;apos;utilisation de tâches de découverte intégrées.
</Callout>
