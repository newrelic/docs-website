Les logs de Azure Virtual Network (VNet) Flow vous permettent d&apos;enregistrer dans un log des informations sur le traffic IP circulant dans vos réseaux virtuels. Avec le redirecteur de logs de New Relic VNet Flow, vous pouvez transférer automatiquement ces logs vers New Relic pour l&apos;analyse des performances et le dépannage de la connectivité réseau.

## Comment ça marche [#how-it-works]

Le redirecteur de logs de VNet Flow est une application Azure Function pilotée par les événements :

<img title="Azure VNet Flow Logs Architecture" alt="Azure VNet Flow Logs Architecture" src="/images/azure-arch.webp" />

1. **Event Grid → Event Hub**: un abonnement Event Grid sur votre compte de stockage de log de flux écoute les événements `BlobCreated` sur les blobs PT1H.json et les transmet directement à un Event Hub. L&apos;abonnement utilise le chemin du blob (sujet de l&apos;événement) comme clé de partition pour garantir un ordre chronologique par fichier. Il n&apos;y a pas de fonction de relais distincte.

2. **VNetFlowLogsConsumer**: une fonction déclenchée par Event Hub qui traite chaque événement blob en :

   * Lecture d&apos;un curseur à partir d&apos;Azure Table Storage pour déterminer ce qui a déjà été traité.
   * Téléchargeant uniquement les blocs nouvellement ajoutés à partir du blob PT1H.json (lectures delta).
   * Analyse des enregistrements de log de VNet flow (tuples de flux) en entrées de log structurées.
   * Envoi des enregistrements analysés à New Relic via l&apos;API de log.
   * Validant le curseur mis à jour lors d’une livraison réussie.

3. **VNetFlowLogsCleanup**: une fonction déclenchée par un minuteur qui supprime périodiquement les entrées de curseur obsolètes d&apos;Azure Table Storage (par défaut, tous les jours à 03:00 UTC).

Cette architecture garantit un traitement efficace et incrémentiel — seules les nouvelles données sont lues et transférées, évitant le traitement en double des enregistrements déjà vus.

## Prérequis [#prerequisites]

### Prérequis pour New Relic [#prerequisites-NR]

Vous avez besoin des éléments suivants dans New Relic :

* Un compte New Relic. Vous n&apos;en avez pas ? [Inscrivez-vous gratuitement !](https://newrelic.com/signup) Aucune carte de crédit requise.
* Une New Relic <InlinePopover type="licenseKey" />.

### Prérequis Azure [#prerequisites-azure]

Vous avez besoin des éléments suivants dans Azure :

* Un abonnement Azure avec VNet Flow Logs activé sur les réseaux virtuels cibles. Consultez la [documentation de Microsoft sur l’activation de VNet Flow Logs](https://learn.microsoft.com/en-us/azure/network-watcher/vnet-flow-logs-manage?tabs=portal).
* Un compte de stockage où vos logs de VNet Flow sont écrits (blobs PT1H.json).
* Un espace de nommage Event Hub et un Event Hub pour la livraison d&apos;événements (peuvent être créés automatiquement par le modèle ARM/Bicep).
* Un compte de stockage Azure pour le runtime de la Function App et le stockage de la table de curseurs (peut être créé automatiquement par le modèle ARM/Bicep).

      Le modèle va automatiquement :

      * Créer un espace de nommage Event Hub, un Event Hub et un groupe de consommateurs
      * Déployer la Function App (plan de consommation Flex, Node.js 22) avec les fonctions de logs de VNet Flow
      * Créer un compte de stockage pour le runtime de la Function App et le suivi du curseur
      * Créez un topic système et un abonnement Event Grid qui acheminent les événements de création de blob PT1H.json directement vers l&apos;Event Hub
      * Attribuez une identité gérée avec les attributions de rôles requises
      * Configurez tous les paramètres d&apos;application nécessaires
      * Activer en option la mise en réseau privée avec l&apos;intégration VNet et des points de terminaison privés

      Suivez ces étapes :

      1. Allez à <DNT>**[one.newrelic.com](https://one.newrelic.com/) &gt; Integrations &amp; Agents**</DNT>.

      2. Sous <DNT>**Collections**</DNT>, sélectionnez <DNT>**Logging**</DNT>, puis cliquez sur la tuile <DNT>**Microsoft Azure VNet Flow Logs**</DNT>.

      3. Dans la liste déroulante <DNT>**Select an account**</DNT>, choisissez le compte vers lequel envoyer vos logs, puis cliquez sur <DNT>**Continue**</DNT>.

      4. À l&apos;étape <DNT>**Enter your license key**</DNT>, cliquez sur <DNT>**Create a new key**</DNT>, cliquez sur <DNT>**Copy key**</DNT> et enregistrez-le en lieu sûr, puis cliquez sur <DNT>**Continue**</DNT>.

      5. Cliquez sur <DNT>**Deploy to Azure**</DNT>. Un nouvel onglet s’ouvre dans Azure avec le modèle ARM chargé.

      6. Sélectionnez le <DNT>**Resource group**</DNT> et un <DNT>**Region**</DNT> pour le déploiement. Nous recommandons un nouveau groupe de ressources, pour éviter de supprimer accidentellement les composants qu&apos;il crée.

      7. Dans le champ <DNT>**New Relic license key**</DNT>, collez la clé de licence que vous avez copiée précédemment.

      8. Assurez-vous que le [point de terminaison New Relic](/docs/logs/log-api/introduction-log-api/#endpoint) est défini sur celui correspondant à votre compte (US, EU ou JP).

      9. (Facultatif) Entrez le <DNT>**Source Storage Account Name**</DNT> où vos logs de VNet Flow sont stockés. Il doit se trouver dans le même groupe de ressources que ce déploiement. Laissez ce champ vide pour créer automatiquement un nouveau compte de stockage source.

      10. (Facultatif) Sélectionnez <DNT>**Disable Public Access To Storage Account**</DNT> pour activer le mode réseau privé avec l&apos;intégration VNet et les points de terminaison privés.

      11. (Facultatif) Configurez les paramètres avancés :

          * <DNT>**New Relic Tags**</DNT>: tags séparés par des points-virgules (format : `key1:value1;key2:value2`)
          * <DNT>**Max Retries**</DNT>: nombre maximum de tentatives de livraison (par défaut : 3)
          * <DNT>**Retry Interval**</DNT>: millisecondes entre les tentatives (par défaut : 2000)
          * <DNT>**Function Log Level**</DNT>: verbosité des logs de l&apos;application Function — `Trace`, `Debug`, `Information`, `Warning` ou `Error` (par défaut : `Information`)
          * <DNT>**Max Event Batch Size**</DNT>: nombre maximum d&apos;événements Event Hub traités par lot (par défaut : 10)
          * <DNT>**Min Event Batch Size**</DNT>: nombre minimum d&apos;événements Event Hub par lot (par défaut : 5)
          * <DNT>**Max Wait Time**</DNT>: temps maximum que le déclencheur attend pour remplir un lot avant le traitement, sous la forme `hh:mm:ss` (par défaut : `00:00:30`)
          * <DNT>**Event Hub Scaling Mode**</DNT>: `Basic` (1 unité de débit, 4 partitions, sans auto-inflate — trafic faible à moyen) ou `Enterprise` (auto-inflate jusqu&apos;à 40 unités de débit, 32 partitions — volumes à haut débit) (par défaut : `Basic`)

      12. Cliquez sur <DNT>**Review + create**</DNT>, passez en revue la configuration et cliquez sur <DNT>**Create**</DNT>.

      13. De retour dans New Relic, à l’étape <DNT>**Create log partition**</DNT>, cliquez sur <DNT>**Create partition**</DNT>. Cela crée une partition de données `Log_VNET_Flows_Azure` (règle : `instrumentation.name = 'vnet-app'`, rétention STANDARD) afin que vos VNet Flow Logs soient stockés et fassent l’objet d’une requête séparément de vos autres logs. Si une partition `Log_VNET_Flows_Azure` existe déjà sur le compte, cliquez sur <DNT>**Continue**</DNT> pour ignorer cette étape.

      14. À l&apos;étape <DNT>**Test your logs**</DNT>, confirmez que les données arrivent en exécutant `SELECT * FROM Log_VNET_Flows_Azure`, puis cliquez sur <DNT>**See your data**</DNT>.

      ### Architectures de déploiement [#deployment-architectures]

      Le modèle ARM/Bicep prend en charge deux modes de déploiement :

      **Déploiement standard** (par défaut) :

      * Utilise Event Grid pour monitorer la création de blobs dans le compte de stockage source
      * Achemine les événements directement vers l&apos;Event Hub pour un traitement ordonné, en utilisant le sujet comme clé de partition
      * Traite les logs de VNet Flow dans la Function App et les transfère à New Relic
      * Active l&apos;accès au réseau public sur la Function App, son compte de stockage et le hub d&apos;événements

      **Déploiement de réseau privé** (facultatif, `disablePublicAccessToStorageAccount=true`) :

      * Comprend toutes les fonctionnalités du déploiement standard
      * Isole la Function App, son compte de stockage et l&apos;Event Hub au sein d&apos;un réseau virtuel privé, avec l&apos;accès au réseau public désactivé pour ceux-ci
      * Ajoute l&apos;intégration VNet avec des points de terminaison privés pour les services de stockage (blob, file, queue et table)
      * Crée des zones DNS privées pour la résolution de noms
      * Ne modifie pas le compte de stockage source (contenant les logs de VNet Flow), qui doit conserver l&apos;accès au réseau public activé
    </TabsPageItem>

    <TabsPageItem id="manual-deploy">

      1. Déployez l&apos;application de fonction [azure-vnet-flow-logs](https://github.com/newrelic/azure-vnet-flow-logs) sur votre abonnement Azure.
      2. Assurez-vous que les fonctions `VNetFlowForwarder` (`VNetFlowLogsConsumer` et `VNetFlowLogsCleanup`) sont incluses dans le déploiement.
      3. Configurez les variables d&apos;environnement requises (paramètres de l&apos;application) listées ci-dessous.
      4. Configurez un abonnement Event Grid sur votre compte de stockage de logs de flux pour router les événements `BlobCreated` directement vers l&apos;Event Hub, en utilisant le sujet de l&apos;événement comme clé de partition.
    </TabsPageItem>
