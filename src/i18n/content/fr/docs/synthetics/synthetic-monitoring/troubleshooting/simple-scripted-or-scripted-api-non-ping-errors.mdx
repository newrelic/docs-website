## Problème [#problems]

Le moniteur Synthétique [Simple, Scripted ou Scripted API (non-ping)](/docs/synthetics/new-relic-synthetics/using-monitors/add-edit-monitors#monitor-type-ping) de New Relic a signalé une erreur, mais l&amp;apos;application semble s&amp;apos;être chargée correctement. Pour les erreurs de ping et de moniteur simples, voir [les erreurs de moniteur non scriptées](/docs/synthetics/new-relic-synthetics/troubleshooting/non-scripted-monitor-errors).

## Solutions

Vous trouverez ci-dessous quelques-uns des messages d'erreur de moniteur non ping les plus courants.

### Erreurs de navigateur simples ou scriptées [#simple-browser-errors]

<CollapserGroup>
  <Collapser id="element-not-clickable" title="L'élément A n'est pas cliquable au point (X, Y). Un autre élément recevrait le clic : l'élément B">
    ### Problème

    Le script Synthétique tente de `.click()` un élément (Élément A) au point (X,Y), mais un autre élément (Élément B) masque l&amp;apos;élément cible.

    ### Solution

    Définissez un temps d’attente personnalisé, laissant le temps nécessaire pour qu’une condition spécifique soit remplie. Dans ce cas, jusqu'à ce que l'animation de chargement ne soit plus visible :

    ```js
    .then(function () {
      return $browser.wait(
        $driver.until.elementIsNotVisible($browser.findElement($driver.By.id("LOADING"))),
        10000
      );
    });
    ```

    Vous pouvez également définir un [délai de veille](/docs/synthetics/new-relic-synthetics/scripting-monitors/synthetics-scripted-browser-reference-monitor-versions-04x-or-lower#browser-sleep) personnalisé à l&amp;apos;aide de `$browser.sleep(sleeptime_ms)`, bloquant ainsi l&amp;apos;exécution du script pendant une durée spécifiée. Comme il s&amp;apos;agit d&amp;apos;un temps d&amp;apos;attente fixe, qui ne tient pas compte de l&amp;apos;augmentation de la latence du réseau ou de la dégradation des performances du site, nous vous recommandons d&amp;apos;utiliser plutôt la fonction `.wait()` .

    <Callout variant="tip">
      Cela ne corrigera pas les problèmes `.click()` causés par les en-têtes ou pieds de page collants. Dans ces cas, vous devrez peut-être faire défiler manuellement pour afficher la cible.
    </Callout>

    ### Cause

    Cela se produit si l'élément cible, au moment de la fonction `.click()` , est masqué par :

    * Une superposition de chargement, une fenêtre modale ou une fenêtre contextuelle
    * Une animation qui révèle l'élément cible
    * Un en-tête ou un pied de page collant
  </Collapser>

  <Collapser id="element-not-visable" title="Erreur : élément non visible">
    ### Problème

    L'élément cible n'est pas visible par le sélénium Webdriver.

    ### Solution

    Vérifiez que l'élément cible n'a pas les propriétés CSS de `display: none` ou `visibility: hidden` appliquées.

    ### Cause

    Tout élément ayant une propriété CSS de `display: none` ou `visibility: hidden` ne sera pas trouvé par le Webdriver sélénium, car le script ne recherchera que les éléments réellement visibles par un utilisateur.
  </Collapser>

  <Collapser id="no-such-element" title="Erreur : aucun élément de ce type : Impossible de localiser l'élément : <LOCATOR>">
    ### Problème

    Le Webdriver sélénium n'a pas pu trouver cet élément dans le DOM visible.

    ### Solution

    Pour résoudre ce problème :

    * Confirmez que le localisateur d’élément utilisé pour l’élément cible est précis. Évitez d'utiliser `By.XPath` dans la mesure du possible, car il est étroitement lié à la structure DOM de la page et peut facilement devenir obsolète lorsque des mises à jour sont effectuées sur la page.

    * Si l'élément est dans un iframe, utilisez `$browser.switchTo().frame(<index or element reference>`.

      <Callout variant="tip">
        Consultez la documentation de sélénium pour plus d'informations sur les fonctions [`switchTo()`](https://seleniumhq.github.io/selenium/docs/api/javascript/module/selenium-webdriver/lib/webdriver_exports_WebDriver.html#switchTo) et [`TargetLocator()`](https://seleniumhq.github.io/selenium/docs/api/javascript/module/selenium-webdriver/lib/webdriver_exports_TargetLocator.html) .
      </Callout>

      ### Cause

      Les raisons courantes de cette erreur incluent :

    * L'élément cible ne peut pas être [localisé](/docs/synthetics/new-relic-synthetics/scripting-monitors/synthetics-scripted-browser-reference-monitor-versions-050#locators) par des fonctions telles que :

      * `$browser.findElement(locator: $driver.Locator)` ou
      * `$browser.waitForAndFindElement(locator: $driver.Locator [, timeout: number]`

      Cela peut être dû à un problème de timing. Par exemple, le Webdriver tente de localiser l'élément avant que la page ne soit chargée.

    * L'élément se trouve dans un iframe, qui est un contexte de document distinct.
  </Collapser>

  <Collapser id="job-timed-out" title="JobTimeoutError : le travail a expiré après 180 s">
    ### Problème

    L'exécution du moniteur scripté a atteint le seuil de délai d'expiration non configurable de 180 secondes et l'exécution a été interrompue.

    ### Solution

    S’il s’agit d’une erreur fréquente, envisagez de diviser les tâches scriptées en un moniteur scripté distinct.

    S'il semble qu'une tâche spécifique entraîne une attente inacceptable du travail, envisagez de modifier la méthode par laquelle vous accomplissez cette tâche. Par exemple, changer `$browser.findElement(locator: $driver.Locator)` en `$browser.waitForAndFindElement(locator: $driver.Locator [, timeout: number)` attribuerait à la tâche son propre délai d&amp;apos;expiration configurable.

    Si vous avez plusieurs étapes où la fonction `$browser.waitForAndFindElement(locator, timeout)` est appelée, assurez-vous que la somme des délais d&amp;apos;attente fournis pour ces étapes ne dépasse pas 180 secondes. Si vous trouvez difficile d&amp;apos;accomplir cela, c&amp;apos;est un signe que le moniteur doit probablement être divisé en un script de moniteur distinct.

    ### Cause

    Tous les moniteurs scriptés Synthétique ont un délai d'expiration global maximal non configurable de 180 s pour l'exécution d'un script.

    Si un script n'est pas terminé après 180 secondes, le travail est terminé. Si cela se produit systématiquement, cela peut être le signe d'un script qui prend trop de temps à se terminer ou que le travail attend une période prolongée lors de la tentative d'exécution d'une tâche scriptée.
  </Collapser>

  <Collapser id="network-error-browser" title="NetworkError : le moniteur n'a généré aucun trafic">
    ### Problème

    Le test API ou le moniteur de navigateur scripté semble être en cours d'exécution mais renvoie cette erreur.

    ### Solution

    Assurez-vous que `$http.get()` ou `$browser.get()` sont appelés correctement et génèrent du trafic.

    Pour le moniteur d'API scripté, si vous utilisez une [option de demande](https://github.com/request/request#requestoptions-callback) pour lancer un agent HTTP non instrumenté en arrière-plan, spécifiez l&amp;apos;un de nos agents HTTP instrumentés à l&amp;apos;aide de l&amp;apos;une des options de demande d&amp;apos;agent ci-dessous :

    * `$globalAgents.http`

    * `$globalAgents.https`

      Exemple:

      ```js
      var options = {
        uri: "https://www.newrelic.com",
        agent: $globalAgents.https,
        agentOptions: {
          rejectUnauthorized: false,
        },
        strictSSL: false,
      };

      function callback(err, res, body) {
        //...
      }

      $http.get(options, callback);
      ```

      ### Cause

      Cela se produit dans les exécutions du moniteur scripté lorsque le client HTTP ($http dans le moniteur d'API scripté) ou le navigateur Chrome ($browser dans le moniteur de navigateur scripté) n'est pas utilisé pour générer du trafic HTTP.

      Dans certains cas, certaines [options de requête](https://github.com/request/request#requestoptions-callback) dans API Monitorer peuvent forcer un nouvel agent HTTP, qui n&amp;apos;est pas instrumenté par Synthétique monitoring, à être utilisé pour collecter le trafic HTTP.
  </Collapser>

  <Collapser id="network-not-defined" title="ReferenceError : $network n'est pas défini">
    ### Problème

    L'objet `$network` utilisé pour définir les proxys [du moniteur](/docs/synthetics/new-relic-synthetics/scripting-monitors/set-proxy-settings-scripted-monitors) n&amp;apos;est pas disponible pour [l'exécution](/docs/synthetics/new-relic-synthetics/scripting-monitors/scripted-monitor-runtime-environment#runtime-table) de ce moniteur.

    ### Solution

    Si votre moniteur a été créé avant la sortie du runtime 0.4.0, créez un nouveau moniteur pour profiter du dernier runtime. La version d'exécution actuelle de votre moniteur est affichée en haut des paramètres du moniteur.

    Pour plus d'informations, consultez [Environnements d'exécution de la version du moniteur scripté](/docs/synthetics/new-relic-synthetics/scripting-monitors/scripted-monitor-version-runtime-environments).

    ### Cause

    Cette erreur se produit lorsque vous tentez d'utiliser `$network` sur un moniteur avec un runtime égal ou inférieur à 0.2.2. Le trafic proxy du moniteur a été introduit dans la version d&amp;apos;exécution du moniteur 0.4.0, provoquant l&amp;apos;évaluation de cette méthode comme non définie lors des exécutions de monitoring précédentes.
  </Collapser>

  <Collapser id="script-timeout-error-browser" title="Erreur de délai d'attente du script">

  <Collapser id="stale-element-reference" title="StaleElementReferenceError : l'élément n'est pas attaché au document de page">
    ### Problème

    La page cible a été chargée, mais une modification a été apportée à un élément entre l'exécution d'un localisateur d'élément et une action en cours d'exécution sur l'élément.

    ### Solution

    Configurez votre navigateur scripté pour attendre que la page soit configurée avant d'effectuer une action `findElement()` . Cela peut être accompli en définissant un [temps d'attente](/docs/synthetics/new-relic-synthetics/scripting-monitors/synthetics-scripted-browser-reference-monitor-versions-04x-or-lower#browser-wait) personnalisé, en utilisant la fonction `$browser.wait(fn, timeout)` avant l&amp;apos;appel findElement, pour attendre une condition indiquant un état de page défini. Cela réduira le risque que la manipulation du DOM entraîne l&amp;apos;obsolescence d&amp;apos;une référence.

    Vous pouvez également définir un [délai de veille](/docs/synthetics/new-relic-synthetics/scripting-monitors/synthetics-scripted-browser-reference-monitor-versions-04x-or-lower#browser-sleep) personnalisé à l&amp;apos;aide de `$browser.sleep(sleeptime_ms)`, bloquant ainsi l&amp;apos;exécution du script pendant une durée spécifiée. Comme il s&amp;apos;agit d&amp;apos;un temps d&amp;apos;attente fixe, qui ne tient pas compte de l&amp;apos;augmentation de la latence du réseau ou de la dégradation des performances du site, nous vous recommandons d&amp;apos;utiliser plutôt la fonction `.wait()` .

    ### Cause

    Cette erreur se produit généralement lorsque le script tente de `.click()` un élément après avoir utilisé la fonction `findElement()` ou `waitForAndFindElement()` .

    Si le DOM a changé entre le moment où le localisateur d'élément a été généré et le moment où l'action a été exécutée sur l'élément, cette erreur se produira car l'élément réel a changé.

    Par exemple : la fonction `findElement()` est utilisée pour générer une référence d&amp;apos;élément pendant que le script de la page manipule activement le DOM. Le DOM est ensuite modifié, ce qui rend obsolète la référence précédemment générée. La référence désormais obsolète est utilisée pour tenter d&amp;apos;effectuer une action `.click()` , ce qui entraîne l&amp;apos;échec de ce moniteur.

    <Callout variant="tip">
      Pour plus d'informations, consultez la documentation Sélénium sur [les exceptions de référence d'éléments obsolètes](https://www.seleniumhq.org/exceptions/stale_element_reference.jsp).
    </Callout>
  </Collapser>

  <Collapser id="task-timed-out" title="TaskTimedOut : la tâche a expiré en attendant que l'élément soit localisé à l'aide de : <LOCATOR>">
    ### Problème

    La fonction `waitForAndFindElement(<locator>, <timeout>)` n&amp;apos;a pas réussi à localiser un élément dans le délai imparti.

    ### Solution

    Confirmez que le localisateur d’élément utilisé pour l’élément cible est précis. Évitez d'utiliser `By.XPath()` dans la mesure du possible, car il est étroitement lié à la structure DOM de la page et peut facilement devenir obsolète lorsque des mises à jour sont effectuées sur la page.

    ### Cause

    L'élément cible n'existait pas sur la page lorsque la fonction `waitForAndFindElement(<locator>, <timeout>)` a été appelée. Cela peut être dû au fait que la page cible n’est pas dans l’état attendu.

    Les raisons courantes de cette erreur incluent :

    * Il y a un problème légitime avec le site cible.
    * Le localisateur d'élément utilisé est incorrect.
    * Le site cible a changé, nécessitant la révision du script Synthetics.
    * L'action précédente dans le script n'a pas été exécutée avec succès, ce qui a entraîné un état inattendu de la page lorsque l'appel waitForAndFindElement() suivant a été lancé.
  </Collapser>

  <Collapser id="timeouterror-page-load" title="TimeoutError : le chargement de la page a expiré (impossible de terminer toutes requests réseau à temps)">
    ### Problème

    La page cible a été chargée avec succès, mais a renvoyé l'erreur :

    TimeoutError : le chargement de la page a expiré (impossible de terminer toutes requests réseau à temps)

    ### Solution

    Si les échecs ont commencé soudainement, examinez toutes les requests qui pourraient bloquer ou retarder l'événement de chargement de la page. Si vous n'êtes pas sûr de la requête à l'origine de l'erreur, utilisez la [vue chronologique](/docs/synthetics/new-relic-synthetics/using-monitors/view-monitor-results#understanding) pour identifier requests HTTP de longue durée.

    Si la page ne parvient souvent pas à se charger complètement dans le délai d'expiration actuel, définissez un [délai d'expiration de chargement de page](/docs/synthetics/new-relic-synthetics/scripting-monitors/synthetics-scripted-browser-reference-monitor-versions-04x-or-lower#options) personnalisé à l&amp;apos;aide de la fonction `$browser.manage().timeouts().pageLoadTimeout(ms: number)` .

    ### Cause

    La page cible a été chargée avec succès, mais l'événement de chargement de la page n'a pas été déclenché dans le délai de chargement de la page défini dans la fonction `.pageLoadTimeout()` .

    Il existe un certain nombre de raisons pour lesquelles vous pourriez voir ce message d'erreur, notamment :

    * Une demande de ressource bloquée sur la page a retardé le chargement de la page.
    * Une demande de ressource a été traitée plus lentement que la normale en raison d'un problème réseau sous-jacent.
    * Une ressource dépendante sur la page a bloqué l'événement de chargement de l'iframe.
  </Collapser>

  <Collapser id="is-element-present" title="TypeError : $browser.isElementPresent n'est pas une fonction">
    ### Problème

    La fonction `isElementPresent()`, utilisée par le monitorer Synthétique avec un runtime &amp;gt;= 0.5.0, est obsolète dans sélénium 3.

    ### Solution

    Pour continuer à utiliser cette fonction après l'amortissement, vous devrez créer une version personnalisée de cette fonction, telle que :

    ```js
    return $browser.findElements(ele).then(function (found) {
      return found.length > 0;
    });
    ```

    Exemple d'utilisation, qui renverrait vrai :

    ```js
    $browser
      .get("https://www.newrelic.com")
      .then(function () {
        return isElementPresent($driver.By.id("nav_signup"));
      })
      .then(function (found) {
        return console.log(found);
      });
    ```

    ### Cause

    Cela peut se produire lorsque vous essayez d'utiliser un script moniteur de navigateur scripté provenant d'un ancien moniteur (runtime &lt;= 0.4.1) avec un moniteur plus récent (&gt;= 0.5.0) Durée d'exécution.
  </Collapser>
</CollapserGroup>

  <Collapser id="unexpected-token-arrow" title="SyntaxError : jeton inattendu <">
    ### Problème

    `JSON.parse()` une chaîne qui commence par le caractère &amp;lt; et qui est probablement du HTML, au lieu de JSON, lui a été transmise.

    ### Solution

    Assurez-vous que le point de terminaison cible renvoie le corps de réponse attendu. Vous pouvez le faire en enregistrant le corps de la réponse dans le rappel, avant de tenter l'analyse.

    Exemple:

    ```js
    $http.get("http://www.newrelic.com", function (error, response, body) {
      if (error) {
        throw new Error(error);
      }

      console.log(body); // Log HTML response body, don't parse as JSON
    });
    ```

    Selon les points de terminaison cibles de l'API, vous devrez peut-être inclure des en-têtes de requête spécifiques pour garantir que JSON est renvoyé.

    ### Cause

    Le script tente d'utiliser `JSON.parse()` sur un corps de réponse après qu&amp;apos;une demande a été effectuée et s&amp;apos;attend à ce que le point de terminaison renvoie JSON, mais HTML a été renvoyé à la place.
  </Collapser>

  <Collapser id="unexpected-token-u-api" title="SyntaxError : jeton u inattendu dans JSON à la position 0">
    ### Problème

    `JSON.parse()` un paramètre indéfini a été passé, mais on attendait une chaîne JSON.

    ### Solution

    Résolvez la cause de l’erreur de demande. Les détails sur les causes des erreurs de demande peuvent être trouvés dans l'objet d'erreur transmis à la fonction de rappel de demande.

    Exemple:

    ```js
    $http.get("http://www.newrelic.com", function (error, response, body) {
      if (error) {
        throw new Error(error);
      }

      var bodyJson = JSON.parse(body);
      console.log(bodyJson); // Log response body
    });
    ```

    ### Cause

    Cela peut se produire dans le moniteur d'API scripté lors de l'exécution d'une requête API, puis de la tentative d'analyse de la réponse à la requête dans la fonction rappel. Le corps de la réponse est transmis à `JSON.parse()` sans vérifier au préalable si le corps de la réponse n&amp;apos;est pas défini.

    Un corps de réponse indéfini est souvent causé par une erreur de demande. S'il n'existe aucune gestion des erreurs pour empêcher le code d'analyser le corps de la réponse, cette défaillance du moniteur se produira.
  </Collapser>

  <Collapser id="cannot-read-property" title="TypeError : Impossible de lire la propriété « statusCode » de undefined">
    ### Problème

    L'objet de réponse (et donc response.statusCode) dans un rappel de demande d'API n'est pas défini.

    ### Solution

    Résolvez la cause de l’erreur de demande. Les détails sur les causes des erreurs de demande peuvent être trouvés dans l'objet d'erreur transmis à la fonction de rappel de demande.

    Exemple:

    ```js
    $http.get("http://www.newrelic.com", function (error, response, body) {
      if (error) {
        throw new Error(error);
      }
      console.log(response.statusCode);
    });
    ```

    ### Cause

    Cette erreur se produit lorsqu'une erreur s'est produite lors de la finalisation de la demande d'API (par exemple, impossible d'atteindre le serveur, impossible de résoudre le DNS). Dans ces cas, la demande n'a pas été complétée, donc l'objet de réponse dans les arguments de la fonction rappel n'est pas défini.

    S'il n'existe aucune gestion des erreurs pour empêcher le code de vérifier le code d'état de réponse, cette défaillance du moniteur se produira.
  </Collapser>
</CollapserGroup>
