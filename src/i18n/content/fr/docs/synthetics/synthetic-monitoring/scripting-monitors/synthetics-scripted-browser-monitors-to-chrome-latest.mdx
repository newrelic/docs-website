Nous présentons notre dernier runtime de moniteur synthétique, intégrant la **Latest** version de Chrome et Node.js 22. Le runtime **Latest** utilise automatiquement la version la plus récente de Chrome, garantissant que vos moniteurs scriptés s&apos;exécutent dans un environnement conforme aux standards des navigateurs modernes, offrant ainsi une sécurité, une stabilité et des performances accrues.

<Callout variant="important">
  **Mise à jour de la gestion des versions**: New Relic effectue une transition pour maintenir uniquement la **dernière** version de Chrome. Des versions spécifiques de Chrome (telles que 134 et 140) arriveront en fin de vie. Le **18 août 2026**, les outils pour monitorer encore sur d&apos;anciennes versions de Chrome seront mis à niveau de force vers la dernière version stable (Chrome 147 ou supérieure). Nous recommandons de mettre à niveau tous les outils pour monitorer afin d&apos;utiliser **Latest** avant cette date pour garantir un support continu et des mises à jour automatiques.
</Callout>

## Action requise : impact de la mise à niveau de Node.js 22

Dans le cadre de la mise à niveau de Chrome 134 vers la **Latest** version de Chrome, la version backend de Node passe de Node.js 16 à Node.js 22. Ce runtime plus récent est plus strict concernant les cycles de vie des processus. Si vos scripts de moniteur contiennent des **handles ouverts non traités** (tels que des promesses non résolues, des timers persistants ou des connexions réseau non fermées), vos moniteurs échoueront.

**Pour résoudre ce problème :** Validez vos scripts de monitoring avec le runtime **Latest** pour vous assurer que tous les handles sont correctement fermés avant que le script ne termine son exécution. Les validations en échec apparaissent dans l&apos;interface utilisateur de Runtime Upgrades. Consultez [Résoudre les erreurs de mise à niveau du runtime](/docs/synthetics/synthetic-monitoring/troubleshooting/runtime-upgrade-troubleshooting).

## Qui est affecté

New Relic gère automatiquement le processus de mise à niveau pour :

* Moniteurs de navigateur scriptés sur des emplacements publics
* Monitors d&apos;API sur des emplacements publics
* Moniteurs sur des emplacements à la fois publics et privés (hybride)

New Relic applique la validation et la mise à niveau automatisées à toutes les configurations d&apos;emplacement.

<Callout variant="important">
  Les moniteurs s&apos;exécutant exclusivement sur des sites privés nécessitent une mise à niveau manuelle. Consultez le [guide de transition du runtime de site privé](/docs/synthetics/synthetic-monitoring/private-locations/runtime-transition-guide) pour les stratégies de migration et les étapes de déploiement de SJM.
</Callout>

## Qu&apos;est-ce que Latest ?

L&apos;option d&apos;environnement d&apos;exécution **Latest** dans le menu déroulant de création/mise à niveau du moniteur utilise automatiquement la version la plus récente de Chrome prise en charge par New Relic. Au lieu de fixer une version spécifique de Chrome (comme Chrome 134 ou Chrome 145), **Latest** garantit que vos moniteurs s&apos;exécutent toujours sur la version de Chrome la plus récente disponible, avec les dernières fonctionnalités et mises à jour de sécurité.

Lors de la sortie d’une nouvelle version de Chrome, celle-ci devient disponible pour monitorer dans New Relic Synthetics en tant que **Latest** sous une semaine, garantissant de monitorer en restant à jour avec les dernières capacités du navigateur.

## Mettre à niveau les moniteurs via NerdGraph

Pour mettre à niveau vos runtimes de moniteur par programmation, utilisez les mutations NerdGraph ci-dessous. Vous avez besoin du GUID de l&apos;entité de votre moniteur (disponible dans les paramètres du moniteur).

### moniteur de navigateur scripté

```graphql
mutation {
  syntheticsUpdateScriptBrowserMonitor(
    guid: "YOUR_MONITOR_GUID",
    monitor: {
      runtime: {
        runtimeType: "CHROME_BROWSER",
        runtimeTypeVersion: "LATEST",
      }
    }
  ) {
    errors {
      description
      type
    }
  }
}
```

### Moniteur d&apos;API scripté

```graphql
mutation {
  syntheticsUpdateScriptApiMonitor(
    guid: "YOUR_MONITOR_GUID",
    monitor: {
      runtime: {
        runtimeType: "NODE_API",
        runtimeTypeVersion: "LATEST",
      }
    }
  ) {
    errors {
      description
      type
    }
  }
}
```

## Le processus de mise à niveau automatisé

Pour garantir une transition fluide vers le dernier runtime Chrome, New Relic teste proactivement vos moniteurs existants. Le processus comprend :

<Steps>
  <Step>
    ## Validation backend

    New Relic teste automatiquement vos moniteurs existants sur le dernier runtime Chrome / Node.js 22 dans le backend.

    <Callout variant="tip">
      Cette validation ne consomme pas vos tests synthétiques et n&apos;affecte pas vos résultats de production.
    </Callout>
  </Step>

  <Step>
    ## Mise à niveau automatique

    **Si la validation réussit :** New Relic met à niveau le moniteur vers le runtime le plus récent en votre nom.
  </Step>

  <Step>
    ## Examen manuel des échecs

    **Si la validation échoue :** New Relic ne force pas la mise à niveau. Cela se produit souvent en raison de la gestion stricte des handles ouverts par Node.js 22. Au lieu de cela, New Relic signale les moniteurs dont la validation a échoué et les affiche dans la fonctionnalité **Runtime Upgrades** de l&apos;interface utilisateur de votre Synthetics Nerdlet.

    Vous pouvez y consulter les détails de l&apos;échec, résoudre les problèmes de syntaxe de votre script, et valider et mettre à niveau manuellement le monitor après avoir corrigé le script.
  </Step>
</Steps>

## Dépannage des problèmes de mise à niveau

Le problème le plus courant lors de la mise à niveau vers Node.js 22 est la présence de handles ouverts non gérés, y compris les promesses non résolues, les timers persistants, les connexions non fermées et les connexions HTTP/HTTPS. Pour obtenir des solutions détaillées et des exemples de code pour chaque type de problème, consultez [Dépanner les erreurs de mise à niveau du runtime](/docs/synthetics/synthetic-monitoring/troubleshooting/runtime-upgrade-troubleshooting).

## Questions fréquemment posées

**Quels types de moniteurs sont concernés ?** Moniteurs de navigateur scriptés et moniteurs d&apos;API scriptés. Les moniteurs de navigateur simple et les moniteurs de ping ne sont pas affectés.

**Puis-je tester mes scripts localement avant la mise à niveau ?** Oui. Installez Node.js 22 localement et testez vos scripts de monitoring pour identifier les problèmes avant la mise à niveau.

**Puis-je revenir en arrière si mon monitor échoue après la mise à niveau ?** Oui. Vous pouvez rétablir la version d&apos;exécution dans les paramètres du moniteur ou via NerdGraph.

**Quand mes moniteurs seront-ils mis à niveau automatiquement ?** New Relic effectue d&apos;abord la validation. Si la validation réussit, New Relic met automatiquement à niveau les moniteurs. Si la validation échoue, vous voyez l&apos;échec dans l&apos;interface utilisateur des mises à niveau du runtime et devez corriger et mettre à niveau manuellement.

## Besoin d&apos;aide ?

Si vous rencontrez des problèmes lors du processus de mise à niveau :

1. Consultez l&apos;**interface utilisateur de Runtime Upgrades** dans votre Nerdlet Synthetics pour obtenir des messages d&apos;erreur spécifiques.
2. Passez en revue vos scripts de monitoring pour détecter les promesses non gérées et les connexions ouvertes.
3. Testez vos scripts localement avec Node.js 22 avant la mise à niveau
4. Contactez le support New Relic pour obtenir de l&apos;aide pour résoudre les échecs de validation.
